# Docker 与 K8s 之间的关系是什么？

> 注：文件名中的“之前”应理解为“之间”。

## 核心概念

Docker 主要解决 **容器构建、运行和分发** 问题；Kubernetes（K8s）主要解决 **容器编排和集群管理** 问题。

可以简单理解为：Docker 让应用以容器形式运行，K8s 负责在多台机器上调度、扩缩容、滚动发布、服务发现和故障自愈。

## 面试官想考什么

- 是否能区分容器运行时和容器编排平台；
- 是否知道 K8s 的基本对象，如 Pod、Deployment、Service；
- 是否理解为什么单机 Docker 不够用；
- 是否知道滚动发布、扩缩容和自愈的意义。

## 标准回答

> Docker 更偏向单机容器运行和镜像管理，K8s 更偏向集群级容器编排。生产环境中，一个服务通常会有多个副本分布在不同节点上，K8s 可以根据资源和策略调度 Pod，通过 Deployment 管理副本和滚动发布，通过 Service 提供稳定访问入口，并在容器异常时自动拉起新实例。

## 深挖追问

### Pod 是什么？

Pod 是 K8s 中最小调度单位，里面可以包含一个或多个容器。通常一个业务应用容器对应一个 Pod，也可以搭配 sidecar 容器实现日志采集、代理等能力。

### 为什么需要 Service？

Pod IP 会随着重建变化，Service 提供稳定访问入口和负载均衡能力，让调用方不需要关心具体 Pod 地址。

## 实战场景

一个订单服务打包成 Docker 镜像后，通过 K8s Deployment 部署 3 个副本。发布新版本时 K8s 按策略逐步替换旧 Pod；如果某个 Pod 崩溃，K8s 会自动重启或重新调度。

## 易错点/总结

- Docker 和 K8s 不是替代关系，而是不同层面的能力；
- K8s 不能修复应用代码 bug，只能做调度和自愈；
- 上 K8s 前要准备镜像规范、配置管理、日志监控和资源限制；
- 不理解基础对象就直接写 YAML，容易排障困难。
