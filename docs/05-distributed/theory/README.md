# 分布式理论

## 核心概念

### CAP 定理

一个分布式系统最多只能同时满足三项中的两项：

| 特性 | 说明 |
|------|------|
| **C**onsistency | 一致性：所有节点同一时间看到相同数据 |
| **A**vailability | 可用性：每个请求都能获得响应（成功或失败） |
| **P**artition tolerance | 分区容错：网络分区时系统仍能运行 |

**组合选择**：
- **CP**：保证一致性和分区容错（如 ZooKeeper、HBase）
- **AP**：保证可用性和分区容错（如 Cassandra、Eureka）
- **CA**：理论上不存在（分布式环境无法避免网络分区）

### BASE 理论

CAP 的补充，追求可用性和最终一致性：

| 特性 | 说明 |
|------|------|
| **B**asically **A**vailable | 基本可用：允许部分响应延迟或降级 |
| **S**oft state | 软状态：允许中间状态 |
| **E**ventually consistent | 最终一致性：一段时间后达到一致 |

### 一致性模型

| 模型 | 说明 |
|------|------|
| 强一致性 | 写入后立即可见 |
| 最终一致性 | 一段时间后可见 |
| 因果一致性 | 有因果关系的事件有序 |
| 会话一致性 | 同一会话内读己之写 |

## 面试高频问题

### 1. 为什么分布式系统不能同时满足 CAP？

**回答要点**：
- 网络分区是必然存在的（P 必须满足）
- 分区发生时，要么拒绝写入（牺牲 A），要么接受写入但数据不一致（牺牲 C）
- 因此只能在 CP 和 AP 之间选择

### 2. 最终一致性如何保证？

**回答要点**：
- 读修复（Read Repair）
- 写修复（Write Repair）
- 反熵（Anti-Entropy）
- 向量时钟（Vector Clock）

### 3. Raft 协议如何保证一致性？

**回答要点**：
- Leader 选举：获得多数票成为 Leader
- 日志复制：Leader 将日志复制到 Follower
- 安全性：已提交的日志不会被覆盖
- 心跳机制：维持 Leader 地位

### 4. 两阶段提交（2PC）有什么问题？

**回答要点**：
- **同步阻塞**：参与者阻塞等待协调者
- **单点故障**：协调者宕机导致阻塞
- **数据不一致**：部分参与者提交失败

## 实战场景

### 分布式事务方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 中 | 传统数据库 |
| TCC | 最终一致 | 高 | 高 | 高并发金融 |
| Saga | 最终一致 | 中 | 高 | 长事务 |
| 本地消息表 | 最终一致 | 高 | 低 | 简单业务 |
| 事务消息 | 最终一致 | 高 | 低 | 异步场景 |

### 分布式锁实现对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| Redis | 简单、高性能 | 主从切换可能丢锁 |
| ZooKeeper | 可靠性强 | 性能较低、依赖重 |
| etcd | 可靠、支持租约 | 依赖重 |
| 数据库 | 简单、可靠 | 性能最低 |
