# 常见的分布式协议有哪些？

## 核心概念

分布式协议是一组规则，让分散在不同机器、可能故障、通过网络通信的节点协同完成某项任务。不同协议解决不同问题，不能混为一谈：**共识协议**（Paxos/Raft/Zab）解决"多个节点对同一个值达成一致"；**事务提交协议**（2PC/3PC）解决"多个资源参与者一起提交或回滚"；**复制协议**（Gossip/Quorum）解决"数据如何在副本间传播和仲裁"；**协调协议**（Lease/心跳/选主）解决"谁说了算、谁还活着"。

理解分布式协议的关键是先看它属于哪一类问题，再看它如何权衡 CAP——是优先一致性强一致，还是优先可用性最终一致。面试中常考的协议包括 2PC/3PC、Paxos、Raft、Zab、Gossip、Quorum NWR，以及业务层的事务模式 TCC/Saga（详见《事务补偿机制是什么？》）。

## 标准回答

> 常见分布式协议按解决的问题分类：事务提交类有 2PC/3PC，2PC 分准备和提交两阶段，强一致但协调者故障会阻塞，3PC 引入超时缓解阻塞但引入了不一致风险；共识类有 Paxos/Raft/Zab，Paxos 是理论基础但难实现，Raft 把共识拆成 Leader 选举、日志复制、安全性三个子问题更易理解，etcd/Consul 都用它，Zab 是 ZooKeeper 专用协议，类似 Raft 但更强调主备而非对等；传播类有 Gossip，去中心化最终一致，Cassandra/Redis Cluster 用它；仲裁思想有 Quorum/NWR，写多数派读多数派必有交集。选型看一致性强弱：强一致选共识协议，最终一致选 Gossip，业务事务选 TCC/Saga。

## 实现原理

### 2PC（两阶段提交）

```
协调者                    参与者1   参与者2
   |--- prepare --------->|         |
   |<-------- yes ---------|         |
   |--- prepare ------------------> |
   |<-------- yes -------------------|
   |--- commit --------->|         |
   |--- commit ------------------> |
```

- **准备阶段**：协调者问所有参与者"能否提交"，参与者锁定资源并写 undo/redo log，回复 yes/no。
- **提交阶段**：所有 yes → 协调者发 commit；任一 no → 协调者发 rollback。

2PC 的核心问题：

1. **同步阻塞**：参与者锁定资源后一直阻塞等待协调者决议，期间其他事务无法访问。
2. **协调者单点**：协调者在第二阶段前宕机，参与者锁着资源等待，进入阻塞态。
3. **数据不一致**：第二阶段只有部分参与者收到 commit 时，部分提交部分未提交，全局不一致。

### 3PC（三阶段提交）

3PC 在 2PC 的 prepare 前加一个 CanCommit 阶段，并给参与者引入超时：

- **CanCommit**：协调者问"能否执行事务"，参与者不锁资源只做校验。
- **PreCommit**：相当于 2PC 的 prepare，参与者锁资源。
- **DoCommit**：协调者发提交；参与者在超时未收到时**自动提交**。

3PC 解决了 2PC 的阻塞问题（参与者超时可自行决定），但引入新风险：网络分区下协调者发了 rollback 但部分参与者超时自动提交，导致不一致。3PC 实际工程中很少用，XA 仍是 2PC 的实现。

### Paxos

Paxos 由 Lamport 提出，是分布式共识的理论基础。核心角色：

- **Proposer**：提议值。
- **Acceptor**：对提议投票。
- **Learner**：学习已决出的值。

基本流程（Basic Paxos）：

1. Proposer 选一个提案编号 N，向多数派 Acceptor 发送 `Prepare(N)`。
2. Acceptor 收到 `Prepare(N)`，若 N 大于已承诺的所有编号，承诺不再接受小于 N 的提案，并返回已接受过的最大编号提案（若有）。
3. Proposer 收到多数派响应后，发送 `Accept(N, value)`，value 取响应中编号最大的已接受值（若无则用自己的值）。
4. Acceptor 收到 `Accept`，若 N 仍大于等于已承诺编号，就接受该提案。
5. Learner 从 Acceptor 学习已决出的值。

Paxos 的关键性质：**多数派交集**保证两个不同提案不可能都被多数派接受，因此一旦值被选定，后续选定的值必然相同。

Multi-Paxos 在 Basic Paxos 基础上优化：选定一个稳定的 Leader 跳过 Prepare 阶段，直接对日志的每个位置跑 Accept，性能大幅提升。但 Paxos 工程实现复杂，Google Chubby、PaxosStore 等少数系统真正落地。

### Raft

Raft 是 Stanford 2014 年提出的"可理解的共识算法"，把共识拆成三个子问题：

1. **Leader 选举**：节点三种角色 Follower/Candidate/Leader；Follower 超时未收到心跳就转 Candidate，自增 term 投自己一票，向其他节点拉票，拿到多数票成为 Leader。
2. **日志复制**：Leader 收到客户端写请求，追加到本地日志，并行发 `AppendEntries` 给所有 Follower；多数派复制成功后 Leader 提交，再通知 Follower 提交。
3. **安全性**：通过"选举限制"（只有日志最完整的节点能当选 Leader）和"提交限制"（Leader 只提交当前 term 的日志）保证已提交日志不丢失、不覆盖。

Raft 比 Paxos 易实现，是 etcd、Consul、TiKV、Kafka KRaft 的共识基础。详细机制见《Raft 共识算法如何工作？》。

### Zab（ZooKeeper Atomic Broadcast）

Zab 是 ZooKeeper 的原子广播协议，分两个阶段：

- **崩溃恢复阶段**：Leader 选举，选 zxid 最大的节点为新 Leader。
- **原子广播阶段**：类似 Raft 的日志复制，Leader 把写请求转为 Proposal 写入队列，多数派 Follower 确认后 Leader 发 Commit。

Zab 和 Raft 的核心差异：Raft 是对等复制（任何节点可参选），Zab 强调主备（Leader 单点写，Follower 转发）；Raft 用 term 编号，Zab 用 zxid（epoch + counter）。两者性能和一致性相当，工程取舍不同。

### Gossip

Gossip 是去中心化的最终一致传播协议，每个节点定期随机选几个邻居交换状态，信息像传染病一样扩散。三次模式：

- **Push**：节点把已知状态推给随机邻居。
- **Pull**：节点向随机邻居拉取未知状态。
- **Push-Pull**：双向交换，收敛最快。

Gossip 优势是去中心化、可扩展（O(log N) 收敛）、容忍节点故障；代价是最终一致、有传播延迟、中间态可能不一致。Cassandra、Redis Cluster、Consul 服务发现都用 Gossip。

### Quorum / NWR

NWR 是读写仲裁思想：N 个副本、写需要 W 个确认、读需要 R 个确认。只要 `W + R > N`，读和写必有交集，读到最新写入的副本。

- W = N, R = 1：强一致写，读快。
- W = 1, R = N：写快，强一致读。
- W = R = 多数派：常用平衡配置，例如 N=3, W=R=2。

Dynamo、Cassandra 用 NWR 思想让用户在一致性和性能间权衡。

### Lease（租约）

Lease 是带超时的授权：授予方在租约期内承诺不改变某项决策，承租方在租约内可自信地使用该决策。常用于：

- **主备切换**：Leader 持有租约，租约期内其他节点不挑战；租约到期才允许重新选举。
- **缓存一致性**：数据源给缓存发租约，租约内缓存可自信返回数据，租约到期前数据源修改数据要等租约失效。

Lease 比"心跳 + 集中超时"更稳健，因为依赖每个节点本地时钟推进，不依赖全局同步时钟。

## 代码示例

### 模拟 2PC 协调者

```java
public class TwoPhaseCoordinator {

    private List<Participant> participants;

    public boolean commit(Transaction tx) {
        // 阶段一：prepare
        List<Participant> prepared = new ArrayList<>();
        for (Participant p : participants) {
            if (p.prepare(tx)) {
                prepared.add(p);
            } else {
                // 任一拒绝，全员回滚
                for (Participant ok : prepared) ok.rollback(tx);
                return false;
            }
        }
        // 阶段二：commit
        for (Participant p : prepared) p.commit(tx);
        return true;
    }
}
```

这段代码体现 2PC 的两阶段语义，但没处理协调者宕机、参与者超时等异常。生产中需要 XA 规范的实现（如 Atomikos、Narayana）补齐这些细节。

### Gossip 状态传播示意

```java
// 节点定时向随机邻居同步状态
@Scheduled(fixedDelay = 1000)
public void gossip() {
    List<Node> peers = peerList.shuffle().subList(0, 3);
    for (Node peer : peers) {
        // 把本地已知版本号发过去，对方返回更新的状态
        Map<String, VersionedValue> delta = peer.exchange(myVersions);
        merge(delta);
    }
}
```

实际生产中用现成实现，例如 Netflix Pentaho、HashiCorp memberlist，不必手写。

## 实战场景

| 场景 | 协议 | 选型理由 |
|------|------|----------|
| etcd 配置中心、Kubernetes 元数据 | Raft | 强一致，少量关键数据 |
| ZooKeeper 分布式协调 | Zab | 强一致，临时节点支持会话感知 |
| Cassandra 副本同步 | Gossip + NWR | 大规模、可扩展、最终一致 |
| Redis Cluster 节点发现 | Gossip | 去中心化，节点增减自动感知 |
| MySQL XA 跨库事务 | 2PC | 数据库层标准，强一致但性能差 |
| Kafka 控制面（KRaft） | Raft | 替代 ZooKeeper，简化部署 |

## 深挖追问

### 2PC 和共识协议（Paxos/Raft）有什么本质区别？

2PC 是**事务提交协议**：协调者单点决定提交或回滚，参与者执行协调者的决议，参与方不参与决策。共识协议是**状态机复制协议**：所有节点共同对同一个值达成一致，没有"协调者单点决议"，多数派共同决定。2PC 协调者宕机会阻塞；共识协议通过多数派选举新 Leader 自动恢复。工程上用"共识协议 + 状态机复制"替代 2PC 来构建强一致的分布式存储（例如 Spanner、TiKV）。

### 为什么 3PC 比 2PC 多一阶段反而更少用？

3PC 引入 CanCommit 和参与者超时自动提交，目标是解决 2PC 的阻塞。但参与者超时自动提交在网络分区下可能造成数据不一致：协调者发了 rollback，部分参与者收到 rollback 回滚，另一部分参与者因网络分区超时自动提交，全局数据分裂。3PC 的代价（不一致风险）大于收益（缓解阻塞），所以工程中很少用，宁愿用 2PC + 协调者 HA（备份协调者）来规避单点。

### Paxos、Raft、Zab 怎么选？

理论上 Paxos 最通用，但实现复杂、调试困难，工程中很少直接用 Basic Paxos；Multi-Paxos 在 Google 内部大规模使用（Chubby、Spanner），但开源实现少。Raft 把共识拆成清晰的三子问题，可读性强，开源生态最活跃（etcd、Consul、TiKV），新项目首选。Zab 是 ZooKeeper 专用协议，类似 Raft 但细节不同，只在用 ZooKeeper 时遇到。面试和实际选型优先掌握 Raft。

### Gossip 会不会传播不一致？

会。Gossip 是最终一致，传播过程中不同节点看到的状态可能不同。例如节点 A 已知"节点 X 下线"，节点 B 还认为 X 在线，短时间内查询不同节点得到不同结果。Gossip 系统通常用版本号或时间戳判断新旧，并配合"读多数派"或"读 Leader"在关键路径上提供强一致读。

### Lease 怎么避免脑裂？

脑裂是网络分区下两个节点都认为自己是 Leader。Lease 的核心约束是"租约期内其他节点不挑战 Leader"，依赖各节点本地时钟超时。即使网络分区，备节点在原 Leader 租约到期前不会晋升，原 Leader 在租约内仍能服务。代价是租约期内原 Leader 真宕机时，集群要等租约到期才能切主，存在短暂的不可用窗口。这是用 CAP 中"A"换"C"的典型权衡。

## 易错点

- 把 2PC 当成共识协议 → 2PC 是事务提交协议，协调者单点决议，不是多数派共识。
- 把 Paxos 等同于 Multi-Paxos → Basic Paxos 是单值共识，Multi-Paxos 才是日志复制，工程实现差异巨大。
- 以为 Raft 比 Paxos 弱 → 两者等价强，Raft 只是把 Paxos 重新组织得更易理解。
- 把 Zab 当成 Raft 的实现 → 两者独立设计，Zab 主备思路、zxid 编号都和 Raft 不同。
- 把 Gossip 用于强一致决策 → Gossip 最终一致，决策类场景必须用共识协议。
- 漏掉 NWR 的 `W+R>N` 约束 → 配置不当会读到旧数据。
- 把 3PC 当成 2PC 的"升级版"就一定更好 → 3PC 用不一致风险换阻塞缓解，实际工程很少用。

## 总结

分布式协议按解决的问题分类：事务提交（2PC/3PC）、共识（Paxos/Raft/Zab）、传播（Gossip）、仲裁（NWR）、协调（Lease）。理解每类协议解决的本质问题、权衡的 CAP 维度、典型应用场景，比死记流程图更重要。工程中新项目优先选 Raft（生态成熟、可读性强），大规模最终一致场景选 Gossip，业务层分布式事务选 TCC/Saga。CAP 决定协议的"基因"，业务场景决定选型，监控和兜底决定生产可用性。
