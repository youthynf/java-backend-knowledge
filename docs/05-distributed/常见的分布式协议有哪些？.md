# 常见的分布式协议有哪些？

## 核心概念

分布式协议用于在多节点、不可靠网络、节点可能故障的环境下，协调状态、复制数据或达成一致。常见协议和思想包括：

- 2PC / 3PC：分布式事务提交协议；
- Paxos / Raft / Zab：共识协议，用于多副本日志一致；
- Gossip：去中心化信息传播协议；
- Quorum / NWR：读写仲裁思想；
- Lease / 心跳 / 选主：常见协调机制；
- TCC / Saga：业务层分布式事务模式。

这些协议解决的问题不同，不能混为一谈。共识协议解决“多个节点对同一序列决策达成一致”，事务协议解决“多个资源参与者一起提交或回滚”。

## 面试官想考什么

- 是否能区分共识、复制、事务、协调协议；
- 是否知道 CAP、可用性和一致性的取舍；
- 是否理解 2PC、Raft、Gossip 的典型应用场景；
- 是否能把协议和中间件联系起来，例如 ZooKeeper、etcd、Kafka。

## 标准回答

> 常见分布式协议包括 2PC/3PC、Paxos、Raft、Zab、Gossip、Quorum 等。2PC 主要用于分布式事务，但协调者故障可能阻塞；Raft/Paxos 用于共识，让多副本对日志顺序达成一致，常见于 etcd、ZooKeeper 类系统；Gossip 用于节点状态传播，适合大规模弱一致场景；Quorum 通过读写多数派提高一致性。实际选型要看业务需要强一致、最终一致还是高可用优先。

## 深挖追问

### 2PC 有什么问题？

2PC 分准备和提交两个阶段。参与者在准备成功后会锁定资源，如果协调者故障，参与者可能不知道该提交还是回滚，造成阻塞。它适合强一致要求高但参与方数量可控的场景，不适合高并发长链路业务。

### Raft 解决什么？

Raft 通过 Leader、日志复制和多数派提交，让多个节点维护一致的日志序列。只要多数节点存活，系统就能继续工作。它常用于配置中心、注册中心、元数据存储等需要强一致的小数据场景。

## 实战场景 / 示例

订单跨库存、支付、积分多个系统时，通常不直接用数据库 2PC，而采用 Saga：

```text
创建订单 -> 扣库存 -> 发起支付 -> 增加积分
   |          |          |          |
失败补偿: 取消订单 <- 释放库存 <- 退款 <- 扣回积分
```

如果是配置中心的配置变更，则更适合由 Raft/Zab 复制日志，保证多节点看到同一份配置版本。

## 易错点 / 总结

- 共识协议不等于分布式事务协议；
- 2PC 强一致但可能阻塞，性能和可用性成本高；
- Raft/Paxos 适合少量关键元数据，不适合直接承载所有业务大流量；
- Gossip 是最终一致传播，不适合强一致决策；
- 面试回答要先说明“协议解决什么问题”。
