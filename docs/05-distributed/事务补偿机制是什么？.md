# 事务补偿机制是什么？

## 核心概念

事务补偿机制用于处理分布式事务中某个步骤失败后的恢复问题。由于跨服务、跨数据库很难像本地事务一样强一致提交，系统通常通过 **正向操作 + 反向补偿 + 重试 + 对账** 实现最终一致性。

常见方案包括 TCC、Saga、本地消息表、可靠消息最终一致性等。

## 面试官想考什么

- 是否理解本地事务和分布式事务的区别；
- 是否知道补偿不是简单 rollback，而是业务反向操作；
- 是否能说明 TCC、Saga、本地消息表的适用场景；
- 是否考虑幂等、空补偿、悬挂、重试和对账。

## 标准回答

> 事务补偿是分布式事务的一种最终一致性处理方式。一个业务流程由多个本地事务组成，如果后续步骤失败，系统不能直接回滚已经提交的远程事务，只能执行对应的反向业务操作进行补偿，例如取消订单、退回库存、退款等。为了保证可靠性，补偿操作通常要支持幂等、可重试，并配合状态机、消息表和定时对账。

## 深挖追问

### 补偿和数据库 rollback 有什么区别？

数据库 rollback 是本地事务提交前的物理回滚；补偿是业务层面的反向操作，通常发生在部分步骤已经提交之后，例如“扣库存成功但支付失败，需要释放库存”。

### TCC 和 Saga 有什么区别？

- **TCC**：Try 预留资源，Confirm 确认，Cancel 取消，控制精细但侵入性强；
- **Saga**：把长事务拆成多个本地事务，每步都有补偿动作，适合流程较长、最终一致可接受的场景。

## 实战场景

下单流程中：创建订单 → 锁定库存 → 发起支付。如果支付失败，需要取消订单并释放库存。释放库存接口必须幂等，因为补偿任务可能因超时、消息重复而多次执行。

## 易错点/总结

- 补偿动作必须可重试、可观测、可人工介入；
- 不要把所有业务都强行做强一致，成本很高；
- 补偿本身也可能失败，需要任务表、告警和对账；
- 设计状态机时要防止重复补偿和状态乱跳。

## 代码示例

```java
// 伪代码：补偿任务必须先判断状态，再做幂等更新
@Transactional
public void releaseStock(String orderNo) {
    StockLog log = stockLogRepository.findByOrderNo(orderNo);
    if (log == null || log.isReleased()) {
        return; // 空补偿/重复补偿直接返回
    }
    stockRepository.addAvailable(log.getSkuId(), log.getCount());
    log.markReleased();
}
```

真实项目里还要记录补偿次数、最后错误、下一次重试时间，并对长期失败的记录触发告警或人工处理。

## 面试复习强化

### 核心概念

从面试角度看，**事务补偿机制是什么？** 可以放在“分布式系统”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 先定位它解决的一致性、可用性、容错或协作问题，再说明失败场景下如何恢复。 追问常围绕网络分区、节点宕机、重复请求、消息乱序、数据回滚和监控告警。 对于“事务补偿机制是什么？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- 补偿事务要保证可重试、幂等、可观测，并能处理补偿本身失败后的人工介入。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：落地要补充超时、重试、幂等、限流、降级、补偿任务和人工兜底。
