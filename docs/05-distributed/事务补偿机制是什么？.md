# 事务补偿机制是什么？

## 核心概念

跨服务的业务流程无法用本地事务的 `rollback` 回滚——库存服务的扣减已经提交、支付接口已经扣款、积分已经发放，下游失败时上游已经物理改库，不可能像数据库那样"撤销 redo log"。补偿机制的本质是：**用业务层面的反向操作，替代数据库的物理回滚**，通过"正向操作 + 反向补偿 + 重试 + 对账"实现最终一致性。

补偿不是"撤销已提交的事务"，而是"做一笔新的反向业务"。比如订单已创建，补偿不是删订单记录，而是把订单状态改成"已取消"；库存已扣，补偿不是回滚事务，而是加一笔"释放库存"的记录。常见的补偿模式有 TCC、Saga、本地消息表、可靠消息最终一致性。

## 标准回答

> 事务补偿是分布式事务最终一致性的核心实现思路。一个长流程拆成多个本地事务，每个本地事务都立即提交，每个正向操作都预先定义好对应的反向补偿操作；任一步骤失败时，按已完成步骤的逆序触发补偿。补偿操作必须幂等（可能被重试多次）、必须可观测（任务表记录重试次数和最后错误）、必须有兜底（长期失败告警转人工）。常见落地模式：TCC 适合短流程强一致、Saga 适合长流程最终一致、本地消息表适合异步解耦、RocketMQ 事务消息适合和消息队列集成。

## 实现原理

### TCC（Try-Confirm-Cancel）

TCC 把每个分支拆成三个接口：

- **Try**：预留资源，例如冻结库存、冻结余额，不真正扣减。
- **Confirm**：确认提交，把冻结转为真实扣减。
- **Cancel**：取消，释放 Try 阶段冻结的资源。

TCC 框架（Seata、ByteTCC、Hmily）充当协调者：所有分支 Try 成功 → 全局 Confirm；任一 Try 失败 → 已 Try 成功的分支 Cancel。

TCC 的三个核心难点：

1. **幂等**：Confirm 和 Cancel 可能因网络重试被调用多次，必须幂等。
2. **空补偿**：Try 未到达或失败，Cancel 先到，必须能识别"没 Try 就 Cancel"并记录空补偿标记。
3. **悬挂**：Cancel 先于 Try 执行，之后 Try 才到，导致资源被冻结后再也无人释放。解决：Cancel 执行后记录"已取消"标记，Try 检查到该标记直接拒绝。

### Saga

Saga 把长事务拆成 N 个本地事务 T1...Tn，每个 Ti 配一个补偿 Ci。执行顺序：

- 正向：T1 → T2 → ... → Tn
- 任一 Ti 失败：执行 C1...C(i-1) 的逆序补偿

Saga 不预留资源，每个 Ti 直接提交，失败时靠 Ci 业务回滚。相比 TCC，Saga 业务侵入小（每个分支只多写一个补偿方法）、并发好（不持有冻结资源），但中间状态可见——T1 提交后 T2 失败前的窗口期，外部能看到"已扣库存但未支付"的中间态。

Saga 分两种编排方式：

- **Choreography（编舞）**：各服务监听事件、自行决定下一步，去中心化但流程难追踪。
- **Orchestration（编排）**：中央协调器（Saga Orchestrator）统一调度，流程清晰但协调器要保证 HA。

### 本地消息表

把"执行本地业务 + 发送消息"拆成两步，用本地事务保证业务表和消息表一起提交：

```
本地事务 {
    update business_table ...;
    insert into local_message_table (msg) values (...);
}
```

后台任务扫描 `local_message_table` 中未发送的消息，投递到 MQ，投递成功后标记已发送。消费端幂等处理。这种方案把分布式事务降级为"本地事务 + 可靠消息"，是业界最常用的最终一致性方案。

### RocketMQ 事务消息

RocketMQ 把"本地事务 + 消息发送"原子化，提供类似两阶段的消息机制：

1. Producer 发送"半消息"到 Broker，半消息对消费者不可见。
2. 半消息发送成功后执行本地事务。
3. 本地事务成功 → Producer 发送 Commit 让半消息转可见；失败 → 发送 Rollback 删除半消息。
4. Broker 长时间未收到二次确认 → 回查 Producer 的本地事务状态。

相比本地消息表，事务消息省去消息表扫描的定时任务，由 Broker 主动回查，延迟更低。

## 代码示例

### Saga 补偿动作（幂等 + 状态机）

```java
@Service
public class OrderSaga {

    @Autowired private OrderMapper orderMapper;
    @Autowired private StockClient stockClient;
    @Autowired private PaymentClient paymentClient;
    @Autowired private PointClient pointClient;

    public void createOrder(Order order) {
        // T1: 创建订单
        orderMapper.insert(order);
        try {
            // T2: 扣库存
            stockClient.deduct(order.getSkus());
            try {
                // T3: 发起支付
                paymentClient.charge(order.getId(), order.getAmount());
                try {
                    // T4: 加积分
                    pointClient.add(order.getUserId(), order.getPoints());
                } catch (Exception e) {
                    // C3: 补偿积分（接口内部幂等）
                    pointClient.deduct(order.getUserId(), order.getPoints());
                    throw e;
                }
            } catch (Exception e) {
                // C2: 补偿支付（退款）
                paymentClient.refund(order.getId(), order.getAmount());
                throw e;
            }
        } catch (Exception e) {
            // C1: 补偿订单
            orderMapper.updateStatus(order.getId(), OrderStatus.CANCELLED);
            stockClient.release(order.getSkus());  // C1 的另一部分：释放库存
            throw e;
        }
    }
}
```

### 幂等补偿动作

```java
// 释放库存接口：必须先查"是否已释放"，避免重复释放
@Transactional
public void releaseStock(String orderNo) {
    StockLog log = stockLogMapper.findByOrderNo(orderNo);
    // 空补偿：Try 未成功，Cancel 先到
    if (log == null) {
        // 插入一条"已取消"标记，防止后续悬挂的 Try
        stockLogMapper.insert(StockLog.cancelled(orderNo));
        return;
    }
    // 已释放：幂等直接返回
    if (log.isReleased()) {
        return;
    }
    stockMapper.addAvailable(log.getSkuId(), log.getCount());
    log.markReleased();
    stockLogMapper.update(log);
}
```

### 本地消息表

```sql
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    biz_key VARCHAR(64) NOT NULL,
    topic VARCHAR(64) NOT NULL,
    body TEXT NOT NULL,
    status TINYINT NOT NULL DEFAULT 0,  -- 0 待发送 1 已发送 2 失败
    retry_count INT NOT NULL DEFAULT 0,
    next_retry_at DATETIME,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_biz (biz_key)
);
```

```java
// 业务 + 消息表同事务
@Transactional
public void payOrder(Long orderId) {
    orderMapper.updateStatus(orderId, OrderStatus.PAID);
    Message msg = Message.builder()
        .bizKey("order:" + orderId)
        .topic("order-paid")
        .body(JSON.toJSONString(orderId))
        .build();
    messageMapper.insert(msg);  // 同事务写入
}

// 定时扫描发送
@Scheduled(fixedDelay = 5000)
public void sendPendingMessages() {
    List<Message> pending = messageMapper.selectPending(100);
    for (Message msg : pending) {
        try {
            mqProducer.send(msg.getTopic(), msg.getBody());
            messageMapper.markSent(msg.getId());
        } catch (Exception e) {
            messageMapper.incrRetry(msg.getId(), nextRetry());
        }
    }
}
```

## 实战场景

| 场景 | 模式 | 选型理由 |
|------|------|----------|
| 跨行转账（冻结余额→扣款） | TCC | 资金强一致，需 Try 冻结 |
| 电商下单（订单/库存/支付/积分） | Saga | 流程长，中间态可接受 |
| 订单支付成功后通知物流 | 本地消息表 | 异步解耦，下游可重试 |
| 注册成功发优惠券 | RocketMQ 事务消息 | 已用 RocketMQ，省去本地表 |
| 跨系统对账 | 定时对账 + 人工补单 | 强一致成本高，对账兜底 |

## 深挖追问

### 补偿和数据库 rollback 有什么区别？

数据库 rollback 是事务提交前的物理回滚——redo log 撤销，数据回到事务开始前的状态，外部不可见。补偿是业务层面的反向操作，发生在本地事务已经提交之后，外部已经能看到中间状态；补偿动作是一笔新的业务记录（如一笔退款单），不是删除原记录。因此补偿必然暴露中间态，只能做到最终一致而非原子一致。

### TCC 和 Saga 怎么选？

TCC 适合短流程、强一致需求高、能改造接口为三段式的场景，例如资金冻结扣减。Saga 适合长流程、参与方多、中间态业务可接受、不愿改造接口为 Try/Confirm/Cancel 的场景，例如电商下单全链路。TCC 业务侵入大但一致性好，Saga 业务侵入小但中间态可见。

### 补偿失败了怎么办？

补偿本身也可能失败（下游服务挂了、网络抖动）。标准做法：

1. 补偿任务表记录每次尝试的结果、错误原因、下次重试时间。
2. 退避重试（指数退避 + 抖动），避免下游刚恢复就被打挂。
3. 重试 N 次仍失败 → 升级到人工处理队列，触发告警。
4. 长期靠对账兜底：每天比对上下游数据，发现不一致人工补单。

### 什么是空补偿和悬挂？

- **空补偿**：Try 没成功（或网络丢包），Cancel 先到达。Cancel 必须能识别"没有 Try 记录"并安全返回，同时写入"已取消"标记。
- **悬挂**：Cancel 先于 Try 执行完，之后延迟到达的 Try 又冻结了资源，但永远不会被 Cancel。防止悬挂：Try 前检查"是否已 Cancel"，是则拒绝 Try。

这两个问题是 TCC 协议在网络乱序下的必然产物，任何 TCC 框架都必须处理。

### 本地消息表和 RocketMQ 事务消息哪个好？

本地消息表通用，不依赖具体 MQ，但需要维护消息表和定时扫描任务，延迟较高（秒级）。RocketMQ 事务消息由 Broker 主动回查，延迟低，但绑定 RocketMQ。如果已经用 RocketMQ，优先事务消息；多 MQ 或想解耦中间件，用本地消息表。

## 易错点

- 补偿动作没做幂等 → 重试时重复扣减/重复退款，资金事故。
- 漏处理空补偿和悬挂 → 网络乱序下资源被锁死或被误释放。
- 补偿动作里再次抛异常但不记录 → 任务表无错误信息，难以排查。
- 把 Saga 当 TCC 用，在 Ti 里冻结资源 → 丢失 Saga 的并发优势，复杂度等于 TCC 但没拿到 TCC 的一致性。
- 用 `@Transactional` 包裹整个 Saga 流程 → 本地事务被跨服务 RPC 拉长，连接池打满、长事务锁表。
- 补偿顺序写错 → 例如先退款再取消订单，可能触发"已取消订单不能退款"的业务校验。
- 不做对账兜底 → 长期累积不一致，最终数据漂移不可恢复。

## 总结

事务补偿是把"分布式回滚"转化为"业务反向操作 + 重试 + 对账"，是最终一致性的工程化实现。TCC 强一致但侵入大、Saga 长流程友好但中间态可见、本地消息表通用、RocketMQ 事务消息低延迟。无论选哪种，幂等、空补偿、悬挂、重试退避、对账兜底都是必须处理的生产问题。设计补偿链路时，先把"失败时谁来补偿、补偿失败怎么办、最终靠什么兜底"三个问题想清楚，再开始写代码。
