# 服务熔断是什么？

## 核心概念

服务熔断是微服务容错机制，用来在下游服务持续失败或响应过慢时，快速失败并返回降级结果，避免调用方线程被拖死，防止故障级联扩散。它类似电路保险丝：异常达到阈值后打开熔断器，暂时不再访问下游。

熔断器通常有三种状态：关闭（正常调用）、打开（直接降级）、半开（放少量请求探测恢复）。常与超时、限流、隔离、重试、降级一起使用。

## 面试官想考什么

- 是否能区分熔断、限流、降级、隔离；
- 是否理解熔断状态机和触发条件；
- 是否知道重试风暴、超时设置和线程池隔离；
- 是否能结合 Sentinel、Hystrix、Resilience4j 等组件说明。

## 标准回答

> 服务熔断是在下游错误率、慢调用比例或异常数超过阈值时，临时切断对下游的调用，直接返回兜底结果或错误。经过一段恢复窗口后进入半开状态，放少量请求探测，如果成功再关闭熔断，否则继续打开。它的目标是保护调用方和整体系统，避免一个慢服务拖垮整条链路。

## 深挖追问

### 熔断和限流有什么区别？

限流是限制进入系统的流量，防止超过承载能力；熔断是下游已经异常时切断调用，防止故障扩散。限流面向流量，熔断面向依赖健康状态。

### 为什么重试要谨慎？

下游已经慢或失败时，大量重试会放大流量，形成重试风暴。重试应有次数限制、退避策略、只对幂等操作开启，并与熔断和超时配合。

## 实战场景 / 代码示例

Resilience4j 示例：

```java
CircuitBreaker cb = CircuitBreaker.ofDefaults("inventory");
Supplier<StockDTO> decorated = CircuitBreaker.decorateSupplier(cb, () -> inventoryClient.getStock(skuId));
StockDTO stock = Try.ofSupplier(decorated)
        .recover(ex -> StockDTO.unknown())
        .get();
```

库存服务异常时，商品详情页可以返回“库存紧张/稍后刷新”，而不是让整个页面超时。

## 易错点 / 总结

- 熔断不是解决所有错误，而是防止故障扩散；
- 必须有合理降级结果，否则用户体验仍然很差；
- 超时时间要小于上游可接受时间，避免线程堆积；
- 重试只适合幂等操作，并要限制次数；
- 熔断指标要结合错误率、慢调用比例和最小请求数。

## 面试复习强化

### 核心概念

从面试角度看，**服务熔断是什么？** 可以放在“分布式系统”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 先定位它解决的一致性、可用性、容错或协作问题，再说明失败场景下如何恢复。 追问常围绕网络分区、节点宕机、重复请求、消息乱序、数据回滚和监控告警。 对于“服务熔断是什么？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- 熔断器有关闭、打开、半开状态，必须配合超时、隔离、降级结果，避免重试风暴。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：落地要补充超时、重试、幂等、限流、降级、补偿任务和人工兜底。
