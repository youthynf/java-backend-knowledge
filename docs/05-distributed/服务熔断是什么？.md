# 服务熔断是什么？

## 标准概念

服务熔断是一种**故障隔离机制**：当下游服务的错误率、慢调用比例或异常数超过阈值时，调用方临时切断对该下游的调用，直接返回降级结果，避免调用方线程被慢下游拖死、防止故障在调用链中级联扩散。它借鉴电路保险丝的思路——异常达到阈值后"跳闸"打开熔断器，经过恢复窗口后"试送电"半开探测，探测成功再"合闸"恢复正常调用。

熔断要解决的核心问题是**雪崩**：服务 A 依赖 B，B 变慢导致 A 调 B 的线程堆积，A 也变慢，依赖 A 的服务继续堆积，最终整条链路被一个慢节点拖垮。熔断在调用方主动切断，把"等待超时"换成"立即失败"，让 A 在 B 故障时仍能对外提供降级服务。

## 标准回答

> 服务熔断在下游错误率或慢调用比例超阈值时，临时切断调用直接返回兜底结果，防止调用方线程堆积引发雪崩。熔断器三状态：关闭（正常调用并统计）、打开（直接失败不调下游）、半开（放少量请求探测下游是否恢复）。熔断触发条件常用滑动窗口错误率（如 50%）、慢调用比例、最小请求数（避免少量样本误判）。它和限流不同——限流防流量过载，熔断防依赖故障扩散；和降级不同——降级是兜底策略，熔断是触发降级的开关。主流实现有 Resilience4j、Sentinel、Hystrix（已停止维护）。

## 实现原理

### 熔断状态机

```
       错误率超阈值
  关闭 ─────────────► 打开
   ▲                    │
   │ 探测成功           │ 等待恢复窗口（如 10s）
   │                    ▼
   └──── 半开 ◄─────────
          │
          │ 探测失败
          ▼
        打开
```

- **关闭（Closed）**：正常调用下游，统计窗口内的成功/失败/慢调用。
- **打开（Open）**：直接返回降级，不发起对下游的调用，等待恢复窗口。
- **半开（Half-Open）**：恢复窗口到期后进入，放行少量探测请求（如 1 个或 10 个），全部成功 → 关闭；任一失败 → 重新打开。

半开状态是关键：它避免"恢复窗口一过就全量放行"，防止下游其实没恢复就被打回打开态。

### 触发条件

主流框架的核心指标：

| 指标 | 含义 | 典型阈值 |
|------|------|----------|
| 错误率 | 失败请求 / 总请求 | 50% |
| 慢调用比例 | 慢于阈值的请求 / 总请求 | 50% |
| 慢调用阈值 | 单请求 RT 超过该值算慢调用 | 1s |
| 最小请求数 | 窗口内至少多少请求才统计 | 10~20 |
| 时间窗口 | 统计区间 | 10s 滑动窗口 |
| 等待窗口 | 打开后多久进入半开 | 10s |

最小请求数很关键——窗口内只有 2 个请求且都失败，错误率 100% 但样本太小，贸然熔断会误伤。

### 与限流、降级、隔离的关系

| 机制 | 解决问题 | 触发 | 典型实现 |
|------|----------|------|----------|
| 熔断 | 依赖故障扩散 | 下游错误率/慢调用超阈值 | 切断调用 |
| 限流 | 自身被流量打垮 | QPS 超过承载能力 | 拒绝/排队 |
| 降级 | 非核心功能拖累主流程 | 主动开关或熔断触发 | 返回兜底 |
| 隔仓 | 单依赖拖垮整个应用 | 线程池/信号量隔离 | 限制资源占用 |

四者常组合使用：熔断触发后调降级逻辑；线程池隔离保证 A 依赖 B 出问题不会耗光 A 的所有线程；入口限流防止上游洪水把整个系统冲垮。

### 主流框架对比

| 框架 | 状态 | 特点 |
|------|------|------|
| Hystrix | 停止维护 | Netflix 出品，线程池隔离为主 |
| Resilience4j | 活跃 | Hystrix 接班者，函数式风格，轻量 |
| Sentinel | 活跃 | 阿里出品，熔断+限流+系统自适应一体，控制台可视化 |

## 代码示例

### Resilience4j 熔断

```java
// 配置熔断器
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)              // 错误率 50% 触发
    .slowCallRateThreshold(50)             // 慢调用比例 50% 触发
    .slowCallDurationThreshold(Duration.ofSeconds(1))  // 慢调用阈值 1s
    .minimumNumberOfCalls(20)              // 至少 20 个请求才统计
    .slidingWindowSize(10)                 // 滑动窗口 10 次
    .waitDurationInOpenState(Duration.ofSeconds(10))   // 打开后 10s 进入半开
    .permittedNumberOfCallsInHalfOpenState(5)          // 半开放 5 个探测
    .build();

CircuitBreaker cb = CircuitBreaker.of("inventory", config);

// 装饰调用
Supplier<StockDTO> supplier = CircuitBreaker.decorateSupplier(
    cb, () -> inventoryClient.getStock(skuId));

// 降级处理
StockDTO stock = Try.ofSupplier(supplier)
    .recover(CallNotPermittedException.class, e -> StockDTO.fallback())  // 熔断打开时降级
    .recover(ex -> StockDTO.fallback())  // 业务异常时降级
    .get();
```

### Sentinel 注解方式

```java
@SentinelResource(
    value = "queryStock",
    blockHandler = "blockHandler",       // 限流/熔断降级
    fallback = "fallback"                // 业务异常降级
)
public StockDTO queryStock(Long skuId) {
    return inventoryClient.getStock(skuId);
}

public StockDTO blockHandler(Long skuId, BlockException ex) {
    return StockDTO.fallback();  // 熔断打开时返回兜底
}

public StockDTO fallback(Long skuId, Throwable ex) {
    return StockDTO.fallback();  // 业务异常时返回兜底
}
```

### Spring Cloud Circuit Breaker 抽象

```java
@Service
public class OrderService {

    @Autowired
    private CircuitBreakerFactory<?, ?> cbf;

    public OrderDTO getOrder(Long id) {
        CircuitBreaker cb = cbf.create("order");
        return cb.run(
            () -> orderClient.getOrder(id),       // 正常调用
            ex -> OrderDTO.fallback()             // 降级
        );
    }
}
```

这套抽象屏蔽底层实现，配置切换 Resilience4j / Sentinel 只需换依赖。

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 商品详情页查库存 | 库存服务异常时熔断返回"库存紧张" | 降级文案要友好，不能直接抛 500 |
| 下单查优惠券 | 优惠券服务慢，熔断后按"无优惠"下单 | 区分核心和非核心，下单仍能进行 |
| 网关聚合多下游接口 | 单下游熔断不影响其他聚合字段 | 部分字段降级，整体页面可用 |
| 第三方支付通道 | 错误率高时熔断切到备用通道 | 熔断 + 自动故障转移 |

## 深挖追问

### 熔断和限流有什么区别？

限流面向**流量**：控制进入系统的请求速率，防止流量超过自身承载能力，是入口保护。熔断面向**依赖健康**：当下游已经异常时切断调用，防止故障扩散，是出口保护。限流是"我自己受不了了先拒绝一部分"，熔断是"下游不行了我别去送命"。两者都触发降级，但触发原因不同——限流降级因为"自己忙不过来"，熔断降级因为"下游病了"。

### 半开状态为什么放少量请求而不是直接全量恢复？

如果下游只是临时抖动后已经恢复，全量放行没问题；但如果是持续故障，恢复窗口一过就全量放行会让所有请求瞬间涌入下游，下游可能本来快恢复了又被冲垮（"重试风暴"）。半开放少量请求（1~10 个）做探测，全部成功才认为真恢复，任一失败立即重新打开。这把"恢复"做成渐进式而非开关式。

### 重试和熔断怎么配合？

重试放大流量，下游越慢重试越多，重试越多下游越慢，形成恶性循环。配合原则：

1. **重试有上限**：最多 1~2 次，不能无限重试。
2. **退避策略**：指数退避 + 抖动，避免同步重试压垮下游。
3. **只对幂等操作重试**：非幂等接口重试可能造成重复扣款。
4. **熔断优先**：一旦熔断打开，重试逻辑直接跳过——熔断器已判定下游不健康，再重试就是雪上加霜。
5. **超时小于熔断窗口**：避免重试占满熔断统计窗口导致误判。

### 线程池隔离和信号量隔离怎么选？

- **线程池隔离**：每次调用下游用一个独立线程池，下游变慢不会耗光调用方主线程。优势：强隔离、可超时；劣势：线程切换开销、线程池数量管理复杂。适合调用外部服务、网络 IO 慢的场景。
- **信号量隔离**：用计数器限制并发，调用仍在主线程。优势：轻量、无线程切换；劣势：不能异步超时、下游慢会拖住主线程。适合内部高速调用、纯内存操作。

Hystrix 默认线程池隔离，Sentinel 默认信号量隔离，按场景调整。

### 熔断指标怎么设置才合理？

错误率阈值过低（如 5%）会频繁误熔断，过高（如 90%）会保护不及时，常用 50%。慢调用阈值要基于业务 SLA——下单 RT 99 分位 500ms，慢调用阈值设 1s 留余量。最小请求数要保证统计有意义——QPS 1000 的服务设 20 没问题，QPS 5 的服务要降到 5 或不用熔断。等待恢复窗口要长于下游典型恢复时间——数据库重启要 30s，等 10s 进半开必然失败。

## 易错点

- 熔断打开时业务异常被吞 → 降级方法必须有，且降级返回必须可被业务接受，不能返回 null 让上层 NPE。
- 最小请求数设太小 → 低峰期少量失败就误熔断。
- 慢调用阈值设得比超时还大 → 永远不会触发慢调用熔断。
- 半开放行请求数设太大 → 探测变成全量恢复，丢失半开意义。
- 熔断器和重试没配合 → 熔断打开后还在底层重试，浪费资源。
- 降级逻辑里继续调用其他下游 → 降级本身可能再触发熔断，降级要尽量本地或缓存兜底。
- 给非幂等接口配重试 → 重试 + 熔断恢复时可能造成重复扣款。

## 总结

服务熔断是出口侧的故障隔离机制，通过三状态机在下游异常时主动切断调用，防止雪崩。它和限流（入口流量保护）、降级（兜底策略）、隔仓（资源隔离）共同构成微服务容错体系。生产落地要重点关注触发条件（错误率 + 慢调用 + 最小请求数）、半开探测、与重试/超时的配合，以及降级逻辑的可用性。框架上 Resilience4j 适合新项目函数式风格、Sentinel 适合需要可视化和一体化的场景。
