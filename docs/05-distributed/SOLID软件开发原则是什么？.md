# SOLID 软件开发原则是什么？

## 核心概念

SOLID 是面向对象设计中的五个常见原则，用于提高代码的可维护性、可扩展性和可测试性：

- **S：单一职责原则**，一个类只负责一个相对独立的职责；
- **O：开闭原则**，对扩展开放，对修改关闭；
- **L：里氏替换原则**，子类应该能替换父类且不破坏行为；
- **I：接口隔离原则**，接口应小而专，不强迫实现方依赖不需要的方法；
- **D：依赖倒置原则**，高层模块不依赖低层细节，二者都依赖抽象。

## 面试官想考什么

- 是否理解设计原则背后的目标，而不是只背缩写；
- 是否能结合项目代码说明如何落地；
- 是否知道原则之间可能存在取舍；
- 是否能识别过度设计。

## 标准回答

> SOLID 是一组指导面向对象设计的原则，核心目标是降低耦合、提高内聚，让系统面对需求变化时更容易扩展。比如策略模式体现了开闭原则，面向接口编程体现了依赖倒置，拆分臃肿接口体现了接口隔离。实际使用时要结合业务复杂度，不能为了原则而制造过多抽象。

## 深挖追问

### 开闭原则怎么落地？

当业务规则经常变化时，可以把变化点抽象成接口，通过策略、工厂、配置或插件方式扩展新实现，尽量避免反复修改稳定的主流程代码。

### 单一职责是不是类越小越好？

不是。单一职责强调变化原因单一，不是机械拆小。过度拆分会导致类数量爆炸、调用链变长，反而降低可读性。

## 实战场景

支付系统支持微信、支付宝、银行卡。如果把所有支付逻辑写在一个大 `if-else` 中，后续新增渠道会不断修改主流程。更好的方式是定义 `PayStrategy` 接口，每种渠道一个实现类，再通过工厂或 Spring 容器选择策略。

## 易错点/总结

- 设计原则服务于可维护性，不是越抽象越好；
- 开闭原则通常需要稳定抽象作为前提；
- 依赖倒置不是“不依赖任何东西”，而是依赖抽象；
- 面试中最好结合设计模式或项目重构案例回答。

## 代码示例

```java
public interface DiscountStrategy {
    BigDecimal discount(Order order);
}

@Service("vipDiscount")
class VipDiscountStrategy implements DiscountStrategy {
    public BigDecimal discount(Order order) {
        return order.amount().multiply(new BigDecimal("0.90"));
    }
}
```

这段代码把折扣规则抽象成策略接口，新增会员等级或活动规则时只增加实现类，主流程依赖 `DiscountStrategy` 抽象，体现开闭原则和依赖倒置原则。
