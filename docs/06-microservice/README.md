# 微服务

微服务不是“把单体拆小”这么简单，它是一套围绕 **服务拆分、独立部署、服务治理、故障隔离和可观测性** 的架构方法。

## 核心概念

- **服务拆分**：按业务边界拆分，避免按技术层简单拆分。
- **注册发现**：服务实例动态上下线后，调用方能发现可用实例。
- **配置中心**：集中管理不同环境、不同服务的配置。
- **网关**：统一入口，处理路由、鉴权、限流、灰度等能力。
- **服务治理**：限流、熔断、降级、重试、超时、负载均衡。
- **可观测性**：日志、指标、链路追踪，帮助定位跨服务问题。

## 面试官想考什么

- 为什么要拆微服务？拆了会带来什么代价？
- 服务拆分的边界怎么确定？
- 服务之间如何通信？RPC 和 HTTP 怎么选？
- 如何处理分布式事务和数据一致性？
- 如何防止一个服务故障拖垮整条链路？

## 标准回答

> 微服务适合业务复杂、团队协作和独立迭代诉求强的系统。它的收益是独立部署、技术异构、弹性扩缩容和故障隔离；代价是分布式复杂度上升，包括服务治理、链路追踪、数据一致性、部署运维成本。因此是否采用微服务，要看业务规模、团队能力和基础设施成熟度。

## 深挖追问

### 1. 单体一定不好吗？

不是。早期业务不复杂、团队较小、访问量不高时，模块化单体往往更简单、更稳定。微服务是为了解决复杂度和协作问题，不是架构炫技。

### 2. 服务拆分按什么原则？

优先按业务能力和领域边界拆分，保持高内聚、低耦合；避免一个业务流程跨太多服务，否则会增加调用链路、事务和排障成本。

### 3. 微服务如何保证稳定性？

设置合理的超时、重试、熔断、限流和降级策略；同时建设日志、指标、链路追踪和告警，避免故障发生后无法定位。

## 实战场景

电商下单链路通常涉及用户、商品、库存、订单、支付、优惠券等服务。下单主链路应尽量短，非核心动作如发短信、发通知、积分发放可以通过 MQ 异步化，减少主流程耗时。

## 易错点

- 为了微服务而微服务，业务还没复杂就过度拆分；
- 服务边界不清，导致循环调用和数据重复；
- 重试没有幂等保障，造成重复扣库存或重复支付；
- 没有链路追踪，线上问题只能靠猜。
