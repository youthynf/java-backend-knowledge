# 微服务核心治理与面试题

## 核心概念

微服务是一种把复杂系统按业务能力拆成多个自治服务的架构方式。每个服务可以独立开发、测试、部署和扩容，通常通过 HTTP/RPC/MQ 通信，并拥有相对独立的数据模型。微服务不是“服务越小越好”，而是在业务边界、团队协作、发布频率、稳定性和运维成本之间做权衡。

常见治理能力包括：

- **服务注册与发现**：服务实例上下线自动感知，调用方通过注册中心或服务发现机制找到可用实例。
- **配置中心**：集中管理配置，支持动态刷新、环境隔离、灰度配置和审计。
- **API 网关**：统一入口，处理路由、鉴权、限流、灰度、协议转换和日志。
- **负载均衡**：在多个实例之间分配请求，常见有轮询、权重、随机、一致性哈希、最少连接。
- **熔断/限流/降级/隔离**：防止故障扩散，保障核心链路。
- **分布式事务/最终一致**：解决跨服务数据一致性，常用 TCC、Saga、可靠消息、本地消息表、Seata AT 等。
- **可观测性**：日志、指标、链路追踪、告警和诊断工具。

## 面试官想考什么

1. **架构判断力**：知道什么时候该拆，什么时候不该拆。
2. **治理体系**：能把注册发现、配置、网关、限流熔断、事务、观测串成完整链路。
3. **稳定性意识**：知道超时、重试、幂等、隔离、降级是基础能力。
4. **落地经验**：能结合 Spring Cloud、Spring Cloud Alibaba、Dubbo、Kubernetes 等生态讲实际方案。
5. **故障处理能力**：能解释雪崩、级联故障、注册中心异常、配置误推、消息积压、数据库瓶颈如何处理。

## 标准回答

微服务的核心价值是按业务边界拆分系统，让不同团队可以独立迭代、独立部署和按需扩容。但拆分后会引入远程调用、网络失败、数据一致性、链路追踪和运维复杂度，所以必须配套服务治理。

以 Spring Cloud Alibaba 为例，服务通过 Nacos 注册实例并订阅服务列表，调用端通过负载均衡选择实例；配置由 Nacos Config 管理；入口流量走 Spring Cloud Gateway，统一鉴权、限流和路由；服务间调用使用 OpenFeign/Dubbo，并配置连接超时、读取超时、重试和熔断；Sentinel 负责限流、熔断、热点参数和降级；跨服务一致性优先采用可靠消息最终一致，强一致场景再考虑 TCC/Seata；全链路通过 traceId、日志、指标和链路追踪定位问题。

微服务不是银弹。如果系统规模小、团队小、业务变化不复杂，单体或模块化单体可能更合适。微服务落地的前提是自动化部署、监控告警、容量规划、服务治理和团队边界都比较成熟。

## 深挖追问

### 1. 微服务如何拆分？

优先按业务能力和领域边界拆分，而不是按技术层拆分。常见方法是 DDD 的限界上下文，例如订单、支付、库存、会员、营销。拆分时要关注数据归属、事务边界、调用频率和团队归属。强耦合、高频同步调用的模块不宜过早拆开。

### 2. 服务注册发现怎么保证可用？

注册中心需要多节点部署，客户端本地缓存服务列表。注册中心短暂不可用时，客户端仍可使用缓存继续调用；实例通过心跳续约，异常实例超时摘除。生产中要关注心跳间隔、摘除延迟、保护阈值和网络分区带来的误摘除。

### 3. 熔断、限流、降级、隔离有什么区别？

- **限流**：控制进入系统的流量，防止超过承载能力。
- **熔断**：下游错误率或慢调用过高时，短时间停止调用，避免故障扩散。
- **降级**：返回兜底数据或关闭非核心能力，保障核心链路。
- **隔离**：用线程池、信号量、舱壁模式隔离资源，避免一个依赖拖垮全局。

### 4. 重试会带来什么问题？

重试可以提高短暂故障下的成功率，但会放大流量，造成重试风暴。非幂等接口重试还可能导致重复扣款、重复下单。因此重试必须配合超时、退避、最大次数、熔断和幂等键，并且核心写接口要谨慎自动重试。

### 5. 分布式事务怎么选？

- **可靠消息最终一致**：最常用，适合订单后发库存/积分/通知，依赖本地消息表、MQ、幂等消费和补偿。
- **TCC**：Try/Confirm/Cancel，适合强业务控制但开发成本高的场景。
- **Saga**：长事务拆成多个本地事务，每步有补偿动作，适合流程长的业务。
- **Seata AT**：对业务侵入低，但依赖 undo log 和数据库能力，适合特定场景，不应无脑套用。

### 6. 网关有哪些职责？

网关是流量入口，负责路由、鉴权、限流、跨域、灰度、协议转换、日志和统一错误处理。网关不应承载复杂业务逻辑，否则会变成新的巨石。高可用上要多实例部署，前面可接 Nginx/LB，配置熔断限流和监控。

### 7. 如何做灰度发布？

常见方案包括按用户 ID、租户、地域、Header、Cookie 或权重路由到新版本。灰度前要保证接口兼容、数据库变更向前/向后兼容、配置可回滚。灰度过程中观察错误率、RT、核心业务指标和日志，异常时快速回滚。

## 示例/实战场景

### 场景一：订单服务调用库存服务超时

处理思路：

1. 订单服务设置合理连接超时和读取超时，避免线程长时间阻塞。
2. 库存服务慢调用比例升高时触发熔断，订单服务返回“稍后重试”或进入排队。
3. 对扣库存接口设置幂等键，避免重试导致重复扣减。
4. 通过 traceId 追踪订单 → 库存 → 数据库链路，定位是库存接口慢、DB 锁冲突还是网络问题。
5. 大促前对库存服务单独压测，设置限流和热点商品隔离。

### 场景二：支付成功后通知订单、积分、消息中心

不建议支付服务同步调用多个下游并放在一个大事务里。更稳妥的方案是支付本地事务提交成功后写本地消息表，由可靠投递任务发送 MQ；订单、积分、消息中心分别消费并做幂等处理。失败消息进入重试和死信队列，最终通过对账补偿保证一致性。

### 场景三：配置中心误推导致服务异常

配置中心必须有环境隔离、权限控制、变更审计、灰度发布和快速回滚。关键配置变更前应校验格式和范围；客户端收到配置后要有默认值和保护逻辑，避免空配置、非法阈值直接拖垮服务。

## 易错点/总结

- 微服务不是为了“技术先进”，而是为了应对复杂业务和团队协作；没有治理能力时，拆分会放大问题。
- 服务间调用必须默认会失败，所以超时、重试、熔断、限流、降级、幂等是标配。
- 分布式事务优先考虑最终一致和补偿，强一致方案成本高、复杂度高。
- 网关、注册中心、配置中心都属于关键基础设施，必须高可用、可观测、可回滚。
- 可观测性不是上线后补的，日志 traceId、指标、告警和链路追踪要随服务一起设计。
