# 消息队列如何防止消息重复消费？

消息队列如何防止消息重复消费？
使用消息队列时，我们经常会遇到一个可能对业务产生影响的问题，消息重复。在订单、扣款、对账等对幂等有要求的场景，消息重复的问题必须解决。
什么情况会发生消息重复？
生产者发送消息后，Broker 保存成功，但是没有成功给生产者返回 ACK，生产者以为消息发送失败，重试，再次给 Broker 发送。Broker 保存了重复消息，导致 Consumer 多次消费。
消费者消费消息后，给 Broker 返回 ACK 失败，导致 Broker 没有修改偏移量，同一条消息再次发送给消费者，或者被消费者拉取到。

如何防止消息重复？
生产者防重（以 Kafka 为例）
有的消息中间件是支持生产者幂等的。比如 Kafka 从 0.11.0 版本开始引入了幂等 Producer。Kafka 实现生产者幂等的原理是在生产者引入了 Producer ID（PID）和 Sequence Number 这两个参数，通过这两个参数，Broker 单分区就可以唯一标识一个生产者发送的唯一消息。
PID：Producer 拥有的 ID，唯一标识一个 Producer。
Sequence Number：自增的数值，唯一标识同一个 Producer 发送到指定分区的消息 ID。
但是幂等 Producer 只能在单分区下生效，多分区情况下是不生效的。因为多个分区之间并不能相互访问对方的<PID,SequenceNumber>。

Broker 防重（以 Pulsar 为例）
Broker 通过参数 BrokerDeduplicationEnabled 开启防重功能。对于 Producer 发送的重复消息，Broker 返回响应 -1:-1。Producer 发送消息时，会带一个 sequenceId 字段，Broker 会按照 ProducerName 维度记录当前生产者最大的 sequenceId(highestSequenceId)。Broker 收到消息时，首先会判断消息中的 sequenceId 是否大于自己保存的当前生产者的 highestSequenceId，如果是则保存消息并更新 highestSequenceId，否则丢弃消息，并且给 Producer 返回 -1:-1。
下面是三个极端情况：
Producer 断开连接：这种情况下，跟 Broker 重新建立连接后，本地保存的 sequenceId 还在，只要使用 sequenceId 递增后发送消息即可；
Producer 宕机：Producer 重启后，缓存的 sequenceId 肯定不存在了，这时跟 Broker 重新建立连接后，Broker 会根据 ProducerName 找出 highestSequenceId 发给 Producer，Producer 使用这个 sequenceId 来发送消息；
Producer 和 Broker 都宕机：Broker 重启后，可以从宕机前保存的快照中恢复各 Producer 对应的 highestSequenceId 发送给各 Producer。但这个 highestSequenceId 不一定准确，因为 Broker 宕机瞬间很有可能最新的 sequenceId 没有来得及保存快照。
需要注意的是，跟 Kafka 的幂等 Producer 类似，Pulsar 的 Broker 幂等也只能保证 Topic/Partition 级别，

消费者防重
从上面的分析可以看出，靠生产者防重和 Broker 防重，只能在 Topic/Partition 级别生效，这通常并不能满足我们的需求。而为了避免消费者重复消费对业务造成影响，消息防重还是必要的。这就要求我们做最后一道防线，在消费端进行防重或幂等处理。

消费端做防重，就不再考虑消息中间件层面的配置（比如 sequenceId），而是从消息体进行下手。生产者发送消息时，给消息体赋值一个全局唯一的 ID，消费者处理消息时，根据全局唯一 ID 做防重。比如消费端的逻辑是保存一条订单消息，那把唯一 ID 保存到数据库并且加一个唯一索引，这样根据唯一索引就可以做消息去重。
不过使用唯一索引也有缺点：
如果使用 MySQL 数据库，不能使用 Change Buffer；
非插入的场景（比如更新库存）不能去重。
对于唯一索引的缺点，我们可以引入 Redis 对唯一 ID 做保存，利用 setNx 判断消息是否已经处理过。

## 面试复习补充

### 核心概念

消息重复消费指同一业务消息被处理一次以上。常见原因是消费者处理成功但 ack/offset 提交失败、生产者重试、Broker 重投递或消费端超时。核心思路不是要求 MQ 永不重复，而是在业务侧接受“至少一次”语义并实现幂等。

### 面试官想考什么

面试官想看你是否知道 MQ 的可靠投递通常会带来重复风险，以及能否用业务唯一键、状态机、唯一索引、去重表、本地事务等手段把重复影响消掉。

### 标准回答

MQ 通常保证至少一次投递，重复消费是正常边界条件。消费端应基于消息 ID 或业务单号做幂等；消费前检查处理状态，处理和幂等标记尽量放在同一事务；对扣库存、发券、转账等非天然幂等操作，要用状态机、乐观锁或唯一约束兜底。

### 深挖追问

- 如果消息处理成功但确认失败会怎样？
- 如何设计幂等键和补偿任务？
- 该方案在高并发或故障恢复时有什么边界？

### 实战场景/示例

发放优惠券时对 `user_id + coupon_id + activity_id` 建唯一索引，重复消息再次插入失败即可视为已处理。

### 易错点/总结

MQ 不是银弹。不要只说“加 MQ 解耦”，还要说明可靠投递、重复消费、顺序性、延迟、监控和补偿。

## 面试复习强化

### 核心概念

重复消费是 MQ 的常态风险，来源包括生产者重试、Broker 重投、消费者处理成功但 ack/offset 提交失败、Rebalance、网络抖动等。工程上通常追求 **At-Least-Once + 消费幂等**，而不是幻想所有环节绝不重复。

### 面试官想考什么

- 是否知道“重复不可完全避免”，关键在业务幂等。
- 是否能区分生产端重复、Broker 重投和消费端重复。
- 是否会设计幂等键、状态机、唯一约束、去重表和补偿任务。

### 标准回答

防重复消费的核心是消费端幂等。首先为每条消息设计全局唯一业务键，例如订单号、支付流水号、业务事件 ID；然后在消费端用数据库唯一索引、Redis `SETNX`、状态机版本号或去重表控制只处理一次。消息确认必须放在业务处理成功之后；失败时允许重试，超过阈值进入死信队列并告警。对于强一致场景，还要配合本地消息表、Outbox、对账和补偿。

### 深挖追问

- **Redis 去重有什么坑？** 先 set 后处理，如果处理失败可能导致消息被误判已处理；需要处理中状态、过期时间、补偿扫描，或以数据库最终状态为准。
- **唯一索引幂等怎么设计？** 以业务唯一键建唯一索引，重复插入捕获 duplicate key 后查询最终状态，不能直接当成功。
- **幂等和分布式锁谁更重要？** 幂等更基础。锁只能降低并发冲突，不能解决重放、补偿、超时后的重复。

### 示例/实战场景

支付成功消息消费时，以 `pay_order_no` 建唯一索引写入支付流水；如果重复消息到达，插入失败后读取流水状态，已成功则直接 ack。若业务处理失败不 ack，让 MQ 重试；重试多次仍失败则进入死信队列，由补偿任务或人工处理。

### 易错点/总结

- 不要把“消费者 ack”当作业务幂等，它只代表本次投递确认。
- 不要用无过期的 Redis key 无限堆积，也不要只用本地内存去重。
- 消息体里必须带业务唯一键，否则很难可靠去重。
