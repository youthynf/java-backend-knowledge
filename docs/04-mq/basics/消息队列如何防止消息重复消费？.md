# 消息队列如何防止消息重复消费

## 核心概念

消息重复消费是指同一条业务消息被消费者处理了一次以上。在订单、扣款、对账等对幂等有严格要求的场景里，重复消费会直接造成业务脏数据，比如重复扣款、重复发券、重复发货。

**关键认知是：在分布式消息系统里，重复消费几乎无法完全避免。** 原因是 MQ 为了保证不丢消息，普遍采用"至少一次"投递语义（At-Least-Once），这意味着网络抖动、消费者崩溃、ack 失败时，MQ 会重投消息。工程上的正解不是要求 MQ 永不重复，而是接受"至少一次"语义，在业务侧实现幂等，让重复消息被处理多次也不会改变最终结果。

## 标准回答

防重复消费的核心是**消费端幂等**，分三步设计：

1. **消息体携带全局唯一业务键**，例如订单号、支付流水号、事件 ID，由生产者生成并写入消息属性或消息体。
2. **消费端按业务键去重**，可选方案包括数据库唯一索引、Redis SETNX、去重表、状态机版本号。
3. **幂等判定与业务处理绑定为原子操作**，处理成功后再提交 MQ 的 offset/ack，失败则不提交，让 MQ 重试；超过阈值进入死信队列。

兜底手段是本地消息表、Outbox、对账补偿，用于覆盖重放、补偿、超时等极端边界。

## 重复来源

要正确防重，先要弄清楚重复是从哪里产生的，三个环节都可能产生。

### 生产者端重复

生产者发送消息后，Broker 保存成功但 ack 网络丢失，生产者以为失败而重试，Broker 因此保存了两份相同消息。这种重复靠生产者幂等机制兜底，例如 Kafka 的 `enable.idempotence=true` 通过 PID + SequenceNumber 让 Broker 单分区去重。

### Broker 端重投

消费者处理完消息但 ack/offset 提交失败，Broker 没收到确认，认为消息没被消费，下一次 poll/推送时再次发送。这是最常见的重复来源，必须靠消费端幂等。

### 消费者端 rebalance

消费者组 rebalance 期间，已处理但未提交 offset 的消息会被重新分配给其他消费者再次消费。Kafka 重平衡、RocketMQ 队列重分配都会触发。

## 实现机制

### 生产者幂等（以 Kafka 为例）

Kafka 从 0.11.0 引入幂等 Producer，原理是引入两个参数：

- **PID（Producer ID）**：每个生产者启动时由 Broker 分配，唯一标识一个生产者会话。
- **SequenceNumber**：生产者向每个分区发送消息时携带的单调递增序列号。

Broker 在每个分区内维护 `<PID, SequenceNumber>`，收到消息时检查：序列号与已记录的相同则丢弃（去重）；序列号不连续则拒绝后续消息（防丢）。**局限性：仅保证单生产者单分区的幂等，跨分区、跨会话不生效。** 跨会话场景需要 Kafka 事务机制（`transactional.id`）。

### Broker 防重（以 Pulsar 为例）

Pulsar 通过 `brokerDeduplicationEnabled=true` 开启 Broker 防重。生产者发送时带 `sequenceId`，Broker 按 ProducerName 维度记录 `highestSequenceId`。收到消息时若 `sequenceId <= highestSequenceId` 则丢弃并返回 `-1:-1`。生产者重启后 Broker 会把 `highestSequenceId` 返回给它，避免回滚。

### 消费端幂等（最关键的一层）

生产者和 Broker 防重只能在 Topic/Partition 级别生效，业务场景下完全不够。**消费端幂等是最后一道防线，也是必须做好的一层。** 主流方案：

#### 方案一：数据库唯一索引

消费端处理消息时，把业务唯一键写入数据库并加唯一索引，重复消息再次插入会因 duplicate key 失败，捕获异常后查询最终状态即可视为已处理。

```java
public void consume(OrderMessage msg) {
    try {
        // 插入流水，pay_order_no 建唯一索引
        payOrderDao.insert(new PayOrderDO(msg.getPayOrderNo(), msg.getAmount(), ...));
        // 业务处理：更新订单状态、扣库存等
        orderService.markPaid(msg.getOrderId());
    } catch (DuplicateKeyException e) {
        // 重复消息，查询最终状态后直接返回
        PayOrderDO exist = payOrderDao.findByOrderNo(msg.getPayOrderNo());
        if (exist.getStatus() == PayStatus.SUCCESS) {
            return; // 已处理，直接 ack
        }
        // 状态不一致则进入补偿
    }
}
```

适用：插入场景、有自然唯一键的业务。局限：不能用于 Change Buffer；非插入场景（如更新库存）不适用。

#### 方案二：Redis SETNX 去重

把唯一键写入 Redis，利用 SETNX 原子性判定是否首次处理。需要设置过期时间，避免无限堆积。

```java
public void consume(OrderMessage msg) {
    String key = "mq:dedup:" + msg.getPayOrderNo();
    // SETNX + 过期时间，10 分钟内重复消息直接丢弃
    Boolean first = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", 10, TimeUnit.MINUTES);
    if (Boolean.FALSE.equals(first)) {
        return; // 已处理过，直接 ack
    }
    try {
        orderService.markPaid(msg.getOrderId());
    } catch (Exception e) {
        // 处理失败必须删除 key，否则重试时被误判已处理
        redisTemplate.delete(key);
        throw e; // 触发 MQ 重试
    }
}
```

坑：先 set 后处理的方式，如果处理失败必须回滚 key，否则重试时会被误判已处理，造成消息丢失。

#### 方案三：状态机版本号

对于状态流转类业务（订单状态、库存扣减），用版本号或状态前置条件保证幂等。

```java
// UPDATE order SET status='PAID', version=version+1
//   WHERE id=? AND status='PENDING' AND version=?
int updated = orderDao.updateStatusCAS(orderId, Status.PAID, expectedVersion);
if (updated == 0) {
    // 状态已变更（可能是重复消息或并发竞争），查询当前状态决定是否跳过
}
```

#### 方案四：去重表

独立建一张 `msg_consume_log(msg_id, consumer_group, status, create_time)` 表，`msg_id + consumer_group` 建唯一索引。消费前先查表，处理成功后更新状态。

## 代码示例

完整的消费端幂等示例（Spring Boot + Kafka + MySQL 唯一索引）。

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class OrderPayConsumer {

    private final PayOrderMapper payOrderMapper;
    private final OrderService orderService;

    @KafkaListener(topics = "order-pay-event", groupId = "order-pay-group")
    public void onMessage(ConsumerRecord<String, String> record, Acknowledgment ack) {
        OrderPayEvent event = JSON.parseObject(record.value(), OrderPayEvent.class);
        String bizKey = event.getPayOrderNo();
        try {
            // 1. 幂等插入流水，pay_order_no 唯一索引
            payOrderMapper.insertIgnore(new PayOrderDO(bizKey, event.getAmount(),
                    event.getPayTime(), PayStatus.PROCESSING));
            // 2. 业务处理
            orderService.markPaid(event.getOrderId(), event.getAmount());
            // 3. 更新流水为成功
            payOrderMapper.updateStatus(bizKey, PayStatus.SUCCESS);
            // 4. 业务成功后再提交 offset
            ack.acknowledge();
        } catch (DuplicateKeyException e) {
            // 重复消息：流水已存在，检查最终状态
            PayOrderDO exist = payOrderMapper.selectByOrderNo(bizKey);
            if (exist.getStatus() == PayStatus.SUCCESS) {
                ack.acknowledge(); // 已成功处理，跳过
                return;
            }
            // 中间态：可能是上次处理一半崩溃，重新走业务流程
            orderService.markPaid(event.getOrderId(), event.getAmount());
            payOrderMapper.updateStatus(bizKey, PayStatus.SUCCESS);
            ack.acknowledge();
        } catch (Exception e) {
            log.error("consume order pay failed, bizKey={}", bizKey, e);
            // 不 ack，让 Kafka 重试；超过阈值由重试 Topic 接管
            throw e;
        }
    }
}
```

`insertIgnore` 配合唯一索引是关键，避免并发下重复插入抛异常打断流程。

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 支付成功消息消费 | 以 `pay_order_no` 建唯一索引写入支付流水，重复消息插入失败后查状态 | 处理失败必须不 ack，让 MQ 重试 |
| 优惠券发放 | 以 `user_id + coupon_id + activity_id` 建联合唯一索引 | 高并发下 INSERT IGNORE 比 try-catch DuplicateKey 更稳 |
| 库存扣减 | 状态机版本号 + CAS 更新，避免重复扣减 | 必须把扣减与幂等标记放在同一事务 |
| 风控事件写入 | Redis SETNX 短期去重 + 异步落库 | Redis 故障时降级到数据库唯一索引 |
| 跨系统数据同步 | 本地消息表 + Outbox，定时扫描重发 | 必须有对账任务兜底 |

## 深挖追问

### Redis 去重有什么坑？

先 set 后处理，处理失败则 key 还在，重试时被误判已处理，造成消息丢失。规避方案：设置"处理中"中间态、加过期时间、补偿任务扫描"处理中"超时记录，或以数据库最终状态为准。Redis 集群故障时也要降级到数据库去重。

### 唯一索引幂等怎么设计？

以业务唯一键（订单号、流水号、事件 ID）建唯一索引，重复插入捕获 `DuplicateKeyException` 后查询最终状态决定下一步。注意不能直接当成功处理，因为可能是上次处理到一半崩溃。建议用 `INSERT IGNORE` 或 `ON DUPLICATE KEY UPDATE` 减少异常开销。

### 幂等和分布式锁谁更重要？

幂等更基础。锁只能降低并发冲突，无法解决重放、补偿、超时后的重复。一个消费端即使加了分布式锁，处理成功但 ack 失败后仍会被重投递。所以幂等是必须项，锁是优化项。

### Kafka 幂等生产者能跨会话去重吗？

不能。PID 是会话级的，生产者重启后 PID 变化，去重失效。跨会话需要事务机制，配置稳定的 `transactional.id`，由事务协调器维护跨会话状态。

### 如何设计补偿任务？

补偿任务扫描"处理中"超时、死信队列中的消息，按业务键查最终状态：已成功则标记完成；未处理则重新执行；处理失败多次后转人工。补偿任务自身也要幂等。

## 易错点

- **把消费者 ack 当作业务幂等**：ack 只代表本次投递确认，不代表业务处理完成。处理失败但 ack 已提交，消息就丢了。
- **用无过期的 Redis key 无限堆积**：不设 TTL 会导致 Redis 内存膨胀。设 TTL 又要覆盖业务最长处理时间，通常 10 分钟到 1 小时。
- **消息体不带业务唯一键**：只能靠 MQ 自身的 `msgId` 去重，但 `msgId` 对业务无意义，重投递时 `msgId` 可能变化，无法可靠去重。
- **幂等判定和业务处理不在同一事务**：先标记已处理再处理业务，业务失败后状态不一致；先处理业务再标记，重复消息会重复处理。
- **只靠数据库唯一索引处理更新场景**：更新库存、修改状态这类非插入场景，唯一索引不生效，必须用状态机或版本号。

## 总结

防止消息重复消费的正解是消费端幂等，而不是依赖 MQ 永不重复。MQ 普遍采用"至少一次"投递，重复是正常边界条件。设计要点是消息体携带全局唯一业务键、消费端用唯一索引/Redis SETNX/状态机做原子去重、处理成功后再提交 ack、失败转死信队列并由补偿任务兜底。生产者幂等和 Broker 防重只能解决 Topic/Partition 级别重复，业务级去重必须由消费端负责。
