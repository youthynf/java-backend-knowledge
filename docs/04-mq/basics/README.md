# MQ 基础

消息队列（Message Queue，MQ）所有产品共性的部分：消息模型、投递语义、重复消费、顺序性、消息丢失、消息积压、延迟消息、死信处理。掌握这些共性问题后，再学具体产品会事半功倍。

## 高频考点

1. **MQ 会不会丢消息？** 分别从生产者、Broker、消费者三个阶段讲，每个阶段都有对应的可靠配置。
2. **如何防止重复消费？** MQ 通常保证至少一次投递，重复是常态，核心在业务幂等而不是幻想 MQ 绝不重复。
3. **如何保证顺序？** 同一业务键进入同一分区/队列，消费端单线程或有序处理。
4. **消息积压怎么办？** 定位瓶颈、扩容消费、限流下游、临时旁路、复盘治理。
5. **延迟消息怎么做？** RocketMQ 原生支持，RabbitMQ 用 TTL + DLX 或插件，Kafka 需自建。
6. **死信队列处理失败消息？** 多次重试失败的消息转入死信队列，由补偿任务或人工处理。

## 通用可靠方案

- **本地消息表 / Outbox**：业务库写本地消息表，后台扫描发送，保证业务与消息最终一致。
- **唯一业务键**：消息体携带全局唯一 ID，消费端基于唯一索引、Redis SETNX、状态机版本号做去重。
- **重试退避**：失败后按指数退避重试，避免风暴。
- **死信队列**：超过重试阈值的消息进入死信队列，避免阻塞分区。
- **监控告警**：lag、消费耗时、错误率、rebalance 次数、下游 RT 都要监控。
- **人工补偿**：对账、回放、修复数据。
