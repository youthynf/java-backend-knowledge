# Kafka 百万消息积压如何处理？

## 核心概念

Kafka 消息积压，本质是**生产速度长期大于消费速度**，导致消费者组的 committed offset 落后于分区最新 offset。观察指标通常是 consumer lag。

处理积压不能只说“加消费者”。正确思路是：**先定位瓶颈，再止血保护系统，然后扩容消费能力，最后复盘并建立容量水位。**

## 面试官想考什么

这个问题考察的是线上故障处理能力，而不是 Kafka API 背诵。面试官通常关注：

1. 你是否知道 lag/offset/partition/consumer group 的关系；
2. 你是否能区分 Kafka 慢、消费者慢、下游慢；
3. 你是否知道分区数限制消费者并发度；
4. 你是否能兼顾恢复速度、消息顺序、幂等和下游保护。

## 标准回答

我会按五步处理。

### 一、定位：积压在哪里，为什么积压

先看消费者组 lag：

```bash
kafka-consumer-groups.sh \
  --bootstrap-server broker:9092 \
  --describe \
  --group order-consumer-group
```

重点看：

- `CURRENT-OFFSET`：消费者已提交的位置；
- `LOG-END-OFFSET`：分区最新位置；
- `LAG`：积压量；
- 是否某几个 partition lag 特别高。

常见原因：

| 瓶颈 | 表现 | 处理方向 |
|---|---|---|
| 消费者实例少 | 所有分区 lag 均匀上涨 | 增加消费者实例，但不能超过分区数 |
| 单条处理慢 | 消费 QPS 低、线程阻塞 | 优化业务逻辑、批量处理、异步化 |
| 下游慢 | DB/Redis/HTTP 调用延迟高 | 限流、批量写、降级、扩容下游 |
| 分区倾斜 | 少数分区 lag 很高 | 检查消息 key、热点用户/商户 |
| Rebalance 频繁 | 消费抖动、重复消费 | 调整心跳、poll 参数、消费者稳定性 |
| 消息异常 | 某条消息反复失败 | 死信队列、跳过/修复毒丸消息 |

### 二、止血：防止问题扩大

如果积压还在持续增长，要先控制入口或保护下游：

1. 对生产端限流，避免继续无限写入；
2. 暂停非核心业务消费者，优先核心链路；
3. 对下游数据库、三方接口加限流和熔断；
4. 对失败消息进入重试 Topic 或死信队列，避免卡住整个分区；
5. 如果是非关键消息，评估是否可以跳过过期消息。

止血的目标不是马上清空积压，而是让系统进入可控状态。

### 三、扩容消费：提升吞吐量

#### 1. 增加消费者实例，但不能超过分区数

Kafka 中同一个消费者组里，一个 partition 同一时刻只能被一个 consumer 消费。因此：

```text
有效并发度 <= Topic 分区数
```

如果 Topic 有 12 个分区，消费者扩到 12 个实例基本到顶；再加实例也只能闲置。

#### 2. 增加分区数

如果分区数太少，可以增加分区：

```bash
kafka-topics.sh \
  --bootstrap-server broker:9092 \
  --alter \
  --topic order-event \
  --partitions 48
```

注意：增加分区不会重新分布旧消息；如果依赖 key 保序，扩分区可能改变新消息的分区映射；分区过多也会增加 broker 元数据和 leader 选举压力。

#### 3. 批量处理和异步化

把逐条处理改成批量处理通常收益很大：批量插入数据库、Redis pipeline、批量调用内部服务、聚合后写入 ClickHouse/ES。

```java
@KafkaListener(topics = "order-event", containerFactory = "batchFactory")
public void consume(List<ConsumerRecord<String, String>> records) {
    // 批量解析、批量写入、成功后再提交 offset
}
```

Kafka poll 线程不要做过长阻塞。可以把消息提交到业务线程池，但必须处理好 offset 提交、失败重试、分区内顺序和线程池满时的反压。

### 四、保护下游：不要把积压转移成数据库事故

清积压时最容易犯的错是把消费者猛增，结果数据库被打挂。

原则：

1. 消费速度要小于下游可承受速度；
2. 数据库写入尽量批量化；
3. 热点更新先聚合，例如计数先写 Redis 再异步落库；
4. 对三方接口加限流和重试退避；
5. 消费端必须幂等，避免重复消息造成脏数据。

常见幂等方案：业务唯一键 + 数据库唯一索引、消息 ID 去重表、Redis `SETNX` 短期去重、状态机版本号、下游接口幂等 key。

### 五、恢复复盘：避免再次积压

补齐这些机制：

- lag 监控和持续增长告警；
- 消费耗时 P95/P99 监控；
- 下游依赖耗时和错误率监控；
- rebalance 次数监控；
- Topic 分区容量规划；
- 失败消息进入重试 Topic/死信队列；
- 压测确认峰值生产速率和最大消费速率。

## 深挖追问

### 如果必须快速清空历史积压怎么办？

可以临时新增一个高并发消费者组消费旧 Topic，处理后写入新 Topic 或直接落库。但前提是业务允许乱序，且幂等已经做好。对有严格顺序要求的订单状态流，不能简单并行乱消费。

### 可以直接 reset offset 跳过消息吗？

只有在业务明确允许丢弃历史消息时才可以，例如过期埋点、临时统计。订单、支付、库存类消息不能随意跳过。执行前要备份 offset，并确认补偿方案。

```bash
kafka-consumer-groups.sh \
  --bootstrap-server broker:9092 \
  --group order-consumer-group \
  --topic order-event \
  --reset-offsets --to-latest --execute
```

## 总结

Kafka 百万积压的面试回答不要停留在“加机器”。完整回答应该是：先看 lag 定位瓶颈，再限流止血，然后在分区数允许范围内扩消费者、批量处理、异步化，同时保护下游并保证幂等，最后补监控、压测和死信机制。

## 面试总结
### 核心概念

消息积压本质是生产速度长期大于消费速度，或者消费链路被下游卡住。处理积压要先定位瓶颈，再止血扩容，最后复盘治理。

### 面试官想考什么

- 是否会看 consumer lag、消费 TPS、错误率、下游 RT、Rebalance 频率。
- 是否知道消费者实例数受分区数限制。
- 是否能区分临时突发、下游故障、毒丸消息和容量不足。

### 标准回答

先确认积压范围：哪些 topic、partition、consumer group lag 增长，消费端是否报错或频繁 rebalance，下游 DB/接口是否变慢。止血阶段可以限流生产、降级非核心消息、隔离毒丸消息。扩容阶段增加消费者实例但不能超过分区数，必要时增加分区、批量消费、异步处理、优化下游写入。恢复阶段控制回放速度，防止压垮下游，并补充监控、容量评估和压测。

### 深挖追问

- **消费者加到很多仍没效果？** 可能分区数不足、单条消息处理慢、下游限流或锁冲突。
- **能不能直接 reset offset？** 只有在消息可丢或可离线补偿时才能跳过；核心业务必须评估影响并保留审计。
- **如何处理毒丸消息？** 捕获异常，按消息 ID 记录失败次数，超过阈值转入死信队列，避免阻塞整个分区。

### 示例/实战场景

大促期间订单消息 lag 飙升，先关闭非核心推荐消息消费，订单消费者按分区扩容到上限；数据库写入改为批量 upsert，并开启下游限流；积压清完后根据峰值 TPS 重新评估分区数、消费者池和告警阈值。

### 易错点/总结

- 积压处理不是简单“加机器”，还要保护下游。
- 临时扩分区会影响按 key 的顺序和分区分布，需要评估。
- 跳过消息必须有业务授权、影响面记录和补偿方案。
