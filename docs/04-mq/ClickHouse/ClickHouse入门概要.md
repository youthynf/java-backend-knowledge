# ClickHouse入门概要

ClickHouse入门概要
一、概述
ClickHouse 作为高性能列式 OLAP 数据库，其底层原理围绕 “高效处理海量数据分析场景” 设计，核心可总结为 “存储层极致优化 + 计算层高效执行 + 分布式协同” 三大支柱。

二、存储原理
列式存储（核心特性）
数据组织形式：同一列所有数据连续存储（如 id 列单独为 id.bin，name 列存为 name.bin），每个列单独文件存放；
核心优势：
查询时仅读取涉及的列，也叫列裁剪，从而减少 IO 量（例如 select id from t 仅读 id.bin）；
同列数据类型一致，压缩率极高，通常比行式高 5-10 倍，默认使用 LZ4 算法，支持 ZSTD 等；
适配向量化计算，即 CPU 可批量处理同列数据，利用 SIMD 指令。

表引擎：以 MergeTree 为核心
ClickHouse 的存储逻辑依赖表引擎（类似 MySQL 存储引擎），其中 MergeTree 系列（如 MergeTree、ReplacingMergeTree、SummingMergeTree）是支撑海量数据存储的核心，其设计包含三大关键机制：
分区（Partition）：按指定字段（通常是时间，如 event_date）将数据逻辑划分为独立目录（如 event_date=2023-10-09），查询时通过分区裁剪（如 WHERE event_date='2023-10-09'）直接跳过无关分区，减少扫描范围。
数据片段（Part）：每个分区内的数据以 “不可变数据片段” 为物理存储单元（命名如 1-3_0），包含列数据文件（.bin）、标记文件（.mrk，记录数据块偏移量）、主键索引（primary.idx）等。数据写入时生成新 Part，不修改旧 Part（避免随机写，提升写入性能）。
后台合并（Merge）：频繁写入会产生大量小 Part，后台线程自动将小 Part 合并为大 Part（类似 LSM 树），减少文件数量、保持主键排序，优化查询效率（合并时可触发聚合，如 SummingMergeTree 自动求和）。

不可变数据模型
数据写入后生成的 Part 不可修改，更新 / 删除操作（Mutation）通过 “生成新 Part 替换旧 Part” 实现（异步执行），避免磁盘随机写，保障高写入吞吐量（支持每秒数百万行写入）。

二、索引体系：快速定位数据，减少扫描范围
ClickHouse 索引设计的核心是 “通过稀疏索引 + 辅助索引，快速排除无关数据块”，而非行式数据库的 “精确到行的索引”。
主键索引（必选）
定义：建表时通过 PRIMARY KEY (col1, col2) 指定，基于排序后的主键列构建稀疏索引（默认每 8192 行记录一个索引值，index_granularity 控制）。
作用：快速定位数据块范围（如 WHERE id > 1000 时，通过主键索引找到 id > 1000 的起始数据块，跳过前面的块）。

跳数索引（Skip Index，可选）
定义：显式创建（如 ALTER TABLE t ADD INDEX col_minmax col TYPE minmax GRANULARITY 8192），记录每个数据块的统计信息（如 min/max、set、bloom_filter）。
作用：过滤无匹配数据的块（如 WHERE score > 90 时，若某块 max(score) ≤ 90，直接跳过该块），减少需加载的数据块数量。

二级索引（Secondary Index，可选）
定义：针对非主键列的索引（如低基数列），记录值与数据块的映射关系。
作用：提升非主键列过滤效率（如 WHERE status='error' 时，直接定位包含 'error' 的数据块）。

三、计算层：向量化执行与流水线优化
ClickHouse 的查询执行效率依赖 “批量处理 + 流水线” 机制，核心是 向量化计算 和 算子流水线。
向量化计算
以 “数据块（Block）” 为处理单位（默认 65536 行 / 块），而非单行。例如：
过滤 WHERE score > 90 时，一次性对整个 Block 的 score 列（65536 个值）执行比较，利用 CPU SIMD 指令批量处理，效率远高于单行循环。
聚合函数（如 sum、count）也基于 Block 批量计算，减少函数调用开销。

查询优化与执行计划
优化阶段：通过规则式优化缩减处理范围，包括：
分区裁剪：仅保留符合条件的分区；
列裁剪：仅读取查询涉及的列；
谓词下推：将过滤条件下推到数据读取阶段（如先过滤再关联）；
索引过滤：利用主键 / 跳数索引排除无关数据块。
执行计划：转换为物理算子流水线（如 “读取→过滤→聚合→投影”），前一个算子处理完一个 Block 后立即传给下一个算子（流式处理，减少内存占用）。

四、分布式架构：集群协同与分片并行
针对超大规模数据（PB 级），ClickHouse 通过 分片（Shard）+ 副本（Replica） 实现分布式扩展：
分片：数据按分片键（sharding_key）分散到多个节点（如按 user_id % 4 分为 4 个分片），每个分片存储部分数据。查询时，协调节点将查询拆分为子查询，并行下发到各分片，分片独立执行后返回中间结果，协调节点合并得到最终结果（利用集群算力并行处理）。
副本：每个分片可配置多个副本（如 2 副本），通过 ZooKeeper 同步数据，实现高可用（某节点故障时，自动切换到副本节点）。

五、核心优势总结
ClickHouse 的高性能源于各层机制的协同：
存储层：列式存储减少 IO、分区 + 索引缩小扫描范围、不可变 Part 提升写入性能；
计算层：向量化计算提升 CPU 效率、算子流水线减少内存开销；
分布式层：分片并行处理利用集群算力，支撑 PB 级数据查询。
