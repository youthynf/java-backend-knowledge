# 如何保证接口幂等？

## 核心概念

接口幂等是指同一个请求被执行一次或多次，最终业务结果保持一致。它常见于支付、下单、扣库存、发券、消息消费等场景。

幂等不是“不执行”，而是 **重复执行不会产生重复副作用**。

## 面试官想考什么

- 是否知道哪些场景必须做幂等；
- 是否能区分防重提交和业务幂等；
- 是否能给出多种实现方案及适用场景；
- 是否考虑并发、过期、重试和异常恢复。

## 标准回答

> 保证接口幂等通常从唯一请求标识、业务唯一约束和状态机控制入手。客户端或服务端生成幂等号，请求进入时先检查是否处理过；数据库层可以通过唯一索引防止重复插入；状态流转类业务可以限制合法状态转换；消息消费场景可以用消息 ID 或业务 ID 做消费记录。核心是识别重复请求并确保重复请求返回相同或兼容结果。

## 常见方案

1. **唯一索引**：订单号、支付流水号、业务单号加唯一约束。
2. **Token 防重**：提交前获取 token，提交时校验并删除，适合表单防重复提交。
3. **幂等表/请求记录表**：记录 requestId、状态、响应结果，重复请求直接返回。
4. **状态机**：只允许从指定状态流转，例如订单只能从“待支付”变为“已支付”。
5. **分布式锁**：控制短时间并发，但不能替代最终幂等，需要和数据库约束配合。

## 深挖追问

### 分布式锁能保证幂等吗？

只能降低并发重复执行概率，不能作为唯一保障。锁可能超时、释放失败或业务执行时间超过租约，因此最终还要依赖唯一索引、状态机或幂等记录兜底。

### MQ 重复消费怎么处理？

消费者根据消息 ID 或业务 ID 建立消费记录，处理前先查是否已成功处理；业务写入要有唯一约束；处理成功后再标记完成，失败则允许重试。

## 实战场景

支付回调可能被支付平台多次通知。系统应根据支付流水号建立唯一约束，只有订单处于“待支付”状态时才更新为“已支付”，重复回调直接返回成功，避免重复加积分或重复发货。

## 易错点/总结

- 幂等要靠服务端保证，不能只依赖前端按钮置灰；
- 唯一索引是非常重要的兜底；
- 幂等记录需要处理处理中、成功、失败等状态；
- 重复请求的返回结果要设计清楚，不能让调用方误判失败。

## 面试复习强化

### 核心概念

从面试角度看，**如何保证接口幂等？** 可以放在“系统设计与排障”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 按“现象确认→指标定位→根因分析→临时止血→长期治理”回答。 追问会覆盖流量、线程/连接池、缓存、数据库、MQ、下游依赖、机器资源和发布变更。 对于“如何保证接口幂等？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- 幂等实现常见方案有唯一业务号、去重表、乐观锁、状态机流转和 Redis 原子占位。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：给结论前先给证据，实战动作要包括监控、日志、压测复现、灰度回滚、限流降级和容量评估。
