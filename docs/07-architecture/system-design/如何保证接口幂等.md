# 如何保证接口幂等？

## 核心概念

接口幂等是指同一个请求被执行一次或多次，最终业务结果保持一致。它常见于支付、下单、扣库存、发券、消息消费等场景。

幂等不是“不执行”，而是 **重复执行不会产生重复副作用**。

## 面试官想考什么

- 是否知道哪些场景必须做幂等；
- 是否能区分防重提交和业务幂等；
- 是否能给出多种实现方案及适用场景；
- 是否考虑并发、过期、重试和异常恢复。

## 标准回答

> 保证接口幂等通常从唯一请求标识、业务唯一约束和状态机控制入手。客户端或服务端生成幂等号，请求进入时先检查是否处理过；数据库层可以通过唯一索引防止重复插入；状态流转类业务可以限制合法状态转换；消息消费场景可以用消息 ID 或业务 ID 做消费记录。核心是识别重复请求并确保重复请求返回相同或兼容结果。

## 常见方案

1. **唯一索引**：订单号、支付流水号、业务单号加唯一约束。
2. **Token 防重**：提交前获取 token，提交时校验并删除，适合表单防重复提交。
3. **幂等表/请求记录表**：记录 requestId、状态、响应结果，重复请求直接返回。
4. **状态机**：只允许从指定状态流转，例如订单只能从“待支付”变为“已支付”。
5. **分布式锁**：控制短时间并发，但不能替代最终幂等，需要和数据库约束配合。

## 深挖追问

### 分布式锁能保证幂等吗？

只能降低并发重复执行概率，不能作为唯一保障。锁可能超时、释放失败或业务执行时间超过租约，因此最终还要依赖唯一索引、状态机或幂等记录兜底。

### MQ 重复消费怎么处理？

消费者根据消息 ID 或业务 ID 建立消费记录，处理前先查是否已成功处理；业务写入要有唯一约束；处理成功后再标记完成，失败则允许重试。

## 实战场景

支付回调可能被支付平台多次通知。系统应根据支付流水号建立唯一约束，只有订单处于“待支付”状态时才更新为“已支付”，重复回调直接返回成功，避免重复加积分或重复发货。

## 易错点/总结

- 幂等要靠服务端保证，不能只依赖前端按钮置灰；
- 唯一索引是非常重要的兜底；
- 幂等记录需要处理处理中、成功、失败等状态；
- 重复请求的返回结果要设计清楚，不能让调用方误判失败。
