# CPU 100% 问题怎么排查

## 核心概念

CPU 100% 表示机器或容器的 CPU 资源被持续打满，可能来自业务线程忙循环、频繁 GC、锁竞争、正则灾难回溯、加解密/压缩计算、流量突增等。排查目标是定位到**哪个进程、哪个线程、哪段代码**消耗 CPU，再判断是代码问题、JVM 问题、还是容量问题。

排查思路："进程定位 → 线程定位 → 代码定位 → 根因分类 → 止血 → 复盘"。

## 标准回答

线上 CPU 高按"四步定位法"：`top` 找 CPU 最高的 Java 进程，`top -Hp <pid>` 找进程内 CPU 最高的线程，`printf '%x\n' <tid>` 把线程 ID 转 16 进制，`jstack <pid> | grep <nid>` 看对应线程栈定位代码。如果同时伴随频繁 GC，用 `jstat -gcutil <pid> 1000` 和 GC 日志确认。止血优先级：摘流量、限流、扩容、回滚，最后才是重启。重启会丢失现场，能不重启就不重启。

## 排查步骤

```bash
# 1. 找到 CPU 高的进程
top -c
# 关注 %CPU 列，假设找到 java 进程 PID = 12345

# 2. 查看进程内线程 CPU
top -Hp 12345
# -H 显示线程，找到 CPU 高的线程 TID = 12367

# 3. 线程 ID 转 16 进制（jstack 输出里 nid 是 16 进制）
printf '%x\n' 12367
# 输出: 304f

# 4. 查看 Java 线程栈
jstack 12345 | grep -A 30 'nid=0x304f'
# 或者导出完整栈
jstack 12345 > jstack.log

# 5. 查看 GC 情况
jstat -gcutil 12345 1000 10
# 每 1 秒一次，共 10 次，看 FGC、GCT 列

# 6. 看 GC 日志（启动参数里要有 -Xlog:gc*）
# JDK 8: -XX:+PrintGCDetails -Xloggc:gc.log
# JDK 9+: -Xlog:gc*=info:file=gc.log
```

## 实现原理

### top -Hp 怎么对应到 Java 线程

Linux 线程 ID（`top -Hp` 看到的 TID）是十进制，Java 线程在 `jstack` 输出里的 `nid` 是 16 进制。两者数值相等，只是进制不同。`printf '%x\n' <tid>` 转成 16 进制后，去 `jstack` 输出里搜 `nid=0x<hex>`，就能定位到具体 Java 线程栈。

### 容器场景的注意点

容器里 `/proc` 看到的是宿主机视角，`top` 可能显示异常。Docker 内用 `top` 通常 OK（因为容器有独立 PID namespace），但要确认 CPU limit 是 cgroup 限制，`top` 显示的 100% 可能是 limit 的 100% 而不是宿主机 CPU 的 100%。K8s 环境用 `kubectl top pod` 看资源使用。

### 频繁 GC 导致 CPU 高的判断

`jstat -gcutil` 输出几列关键值：

- `FGC`：Full GC 次数，短时间内激增说明堆压力异常。
- `GCT`：GC 总耗时，占比高说明 CPU 被 GC 吃掉。
- `YGC`/`YGCT`：Young GC 次数和耗时。

如果 FGC 频繁、堆利用率高、CPU 又被 GC 线程吃掉，根因是内存问题（泄漏或堆太小），不是业务代码。

## 代码示例

采样多次线程栈对比（找出持续占用 CPU 的代码）：

```bash
# 连续抓 5 次栈，间隔 2 秒
for i in 1 2 3 4 5; do
  jstack 12345 > jstack_$i.log
  sleep 2
done

# 找出多次出现在同一位置的线程
grep -A 5 'nid=0x304f' jstack_*.log
```

火焰图（更直观）：

```bash
# 用 async-profiler 采样
./profiler.sh -d 30 -f cpu.html 12345
# 生成火焰图，看哪个方法栈最宽
```

容器内无 JDK 时用 arthas：

```bash
# arthas 不需要 JDK，用 attach API
java -jar arthas-boot.jar 12345
# 直接看 CPU 高的线程栈
thread -n 5
```

## 实战场景

| 场景 | 表现 | 排查手段 |
|------|------|----------|
| 死循环 | 业务线程 100% 卡同一栈 | jstack 多次采样定位 |
| 频繁 Full GC | GC 线程高、堆利用率高、响应变慢 | jstat、堆 dump 分析 |
| 锁竞争 | 多线程 BLOCKED 状态、上下文切换高 | jstack 看 deadlock、监控 cs |
| 正则回溯 | 解析输入时 CPU 飙升 | jstack 看是否卡 Pattern 匹配 |
| 流量突增 | 多线程都活跃、QPS 暴涨 | 看入口监控、限流 |
| 序列化/加解密 | 大对象 JSON 序列化、AES 加密 | jstack 看栈、改用流式 |
| 容器 CPU limit 过低 | 容器内 top 接近 100%、宿主机低 | 调整 limit 或扩容 |

## 深挖追问

### jstack 看不到明显业务栈怎么办？

多抓几次对比，确认线程是否一直停留在同一位置。同时检查：GC 日志、JNI 调用、系统调用（`strace`）、容器 CPU 限额、磁盘 IO wait。有时候 CPU 高不是业务线程，而是 GC 线程或 JIT 编译线程，jstack 看 GC 线程栈意义不大，要结合 `jstat`。

### 线上能直接 kill 进程吗？

不建议第一时间 kill。应先评估：是否有多副本（能不能摘一个）、是否能摘流量、是否会影响交易。kill 前先抓 jstack、jmap、GC 日志、监控截图保留现场。如果业务影响面扩大且没有其他止血手段，再考虑重启。

### top 显示 CPU 100% 但 jstack 看不到业务线程？

可能原因：CPU 被 GC 线程、JIT 编译线程、外部进程（如 sidecar）吃掉；容器 limit 设置过低；线程在 native 代码里（JNI、Zip 压缩），jstack 看不到完整栈。用 `top -Hp` 配合 `printf` 多次定位，必要时用 `perf` 或 `async-profiler`。

### 容器里 jstack 没权限怎么办？

容器没装 JDK 或非 root 用户无法 jstack。解决办法：

- 用 `arthas` 的 `thread -n 5` 直接看 CPU 高的线程栈（arthas 不需要 JDK，用 attach API）。
- 启动时加 JMX 远程端口，从外部连。
- 镜像里装 JDK 或预装 arthas。

### 怎么预防 CPU 飙升？

- 监控告警：CPU、GC、线程数、接口 RT 设阈值告警。
- 限流：网关或 Sentinel 对热点接口限流。
- 压测：上线前压测确认容量。
- 代码审查：关注死循环、正则、大对象序列化、深递归。
- JVM 参数：合理设置堆大小、GC 算法，避免 Full GC 频繁。

## 易错点

- 只看进程 CPU 不够，要定位到线程和代码。
- 线程 ID 必须**转 16 进制**才能匹配 `nid`。
- CPU 高不一定是死循环，也可能是 GC、锁、正则、流量。
- 排障前**先保留现场**（jstack、jmap、GC 日志），不要先重启。
- `top -Hp` 在容器里可能因为 cgroup 显示异常，要用容器内 top。
- 多次采样比单次 jstack 更可靠，单次可能错过瞬时栈。

## 总结

CPU 100% 排查的核心是"进程 → 线程 → 代码"三级定位，工具链是 `top` + `top -Hp` + `printf` + `jstack` + `jstat`。重点掌握线程 ID 16 进制转换、GC 与业务 CPU 高的区分、容器场景的特殊性、止血优先级。和"内存高"排查相比，CPU 排查更关注线程栈，内存排查更关注堆 dump。线上处理要先止血再根因，重启永远是最后手段。
