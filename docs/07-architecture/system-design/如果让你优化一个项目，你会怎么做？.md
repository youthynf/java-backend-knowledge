# 如果让你优化一个项目，你会怎么做

## 核心概念

项目优化不是"哪里慢改哪里"，而是基于目标和数据的系统性改进。常见方向：性能、稳定性、可维护性、成本、安全、研发效率。一句话点题：优化的核心是"先量化、再定位、再优化、再验证"，没有指标的优化都是耍流氓。

优化要先明确目标（降 RT、提 QPS、降成本、提可用性），用监控数据定位瓶颈，按 ROI 排序动手，最后用压测和监控验证效果。避免"我觉得这里慢就改这里"。

## 标准回答

项目优化按"目标 → 现状 → 定位 → 方案 → 验证 → 沉淀"六步闭环。先明确优化目标（如 P95 从 500ms 降到 200ms），再通过监控、链路追踪、慢 SQL、GC 日志、压测报告定位瓶颈，按收益和风险排序。具体手段：SQL/索引优化、缓存、异步化、批处理、限流降级、JVM 调优、代码重构、拆分热点。最后通过灰度发布、压测、监控验证效果，沉淀告警和文档。优化要避免过早优化、避免引入新风险、避免只优化不验证。

## 实现原理

### 优化的五个方向

1. **性能优化**：降低接口耗时、提升吞吐。关注 P95/P99、QPS、错误率。
2. **稳定性优化**：减少故障、提升可用性。关注 SLA、告警、降级、容灾。
3. **可维护性优化**：降低代码复杂度、补充测试、补充文档。关注圈复杂度、覆盖率。
4. **成本优化**：降低机器、带宽、存储成本。关注资源利用率、扩缩容。
5. **研发效率优化**：降低迭代周期、降低故障恢复时间。关注 CI/CD、可观测性。

### 定位瓶颈的工具链

- **接口耗时**：APM（SkyWalking、Pinpoint、Zipkin）看链路。
- **数据库**：慢查询日志、`EXPLAIN`、`SHOW PROCESSLIST`、连接池监控。
- **缓存**：命中率、大 key、热 key、QPS。
- **JVM**：GC 日志、`jstat`、`jmap`、堆 dump。
- **线程**：`jstack`、线程池监控、锁竞争。
- **机器**：CPU、内存、网络 IO、磁盘 IO。
- **MQ**：堆积、消费速率、死信。

### 优化的常见手段及收益

| 手段 | 适用场景 | 收益 | 风险 |
|------|----------|------|------|
| SQL/索引优化 | 慢查询 | 高 | 低 |
| 缓存 | 读多写少 | 极高 | 一致性 |
| 异步化 | 写多、可等待 | 高 | 用户体验、补偿 |
| 批处理 | N+1、循环调用 | 高 | 实时性 |
| 限流降级 | 流量突增 | 高 | 业务有损 |
| JVM 调优 | GC 频繁 | 中 | 测试不充分易引入问题 |
| 代码重构 | 复杂代码 | 中 | 引入 bug |
| 分库分表 | 数据量大 | 高 | 改造成本高 |
| 读写分离 | 读压力大 | 中 | 主从延迟 |
| 微服务拆分 | 单体过大 | 中 | 分布式复杂性 |

## 代码示例

N+1 查询优化（订单列表）：

```java
// 原代码：N+1 查询，100 个订单发 101 次 SQL
List<Order> orders = orderMapper.selectByUserId(userId);
for (Order order : orders) {
    User user = userMapper.selectById(order.getUserId()); // N 次
    order.setUserName(user.getName());
}

// 优化后：1 次批量查询
List<Order> orders = orderMapper.selectByUserId(userId);
Set<Long> userIds = orders.stream().map(Order::getUserId).collect(Collectors.toSet());
Map<Long, User> userMap = userMapper.selectByIds(userIds).stream()
        .collect(Collectors.toMap(User::getId, Function.identity()));
orders.forEach(o -> o.setUserName(userMap.get(o.getUserId()).getName()));
```

深分页优化：

```java
// 原代码：深分页慢，OFFSET 越大越慢
SELECT * FROM `order` ORDER BY id LIMIT 1000000, 20;

// 优化：游标分页，记住上一页最后一条 id
SELECT * FROM `order` WHERE id > ? ORDER BY id LIMIT 20;
```

索引优化：

```sql
-- 原查询：按 user_id 过滤 + create_time 排序
SELECT * FROM `order` WHERE user_id = ? ORDER BY create_time DESC LIMIT 20;

-- 加联合索引，避免回表 + 排序
CREATE INDEX idx_user_time ON `order`(user_id, create_time DESC);
```

缓存优化：

```java
// 读多写少的字典数据加本地 + 远程二级缓存
@Cacheable(value = "dict", key = "#type", unless = "#result == null")
public List<Dict> getByType(String type) {
    return dictMapper.selectByType(type);
}
```

异步化优化：

```java
// 原代码：同步调用三个外部接口，总耗时 = t1 + t2 + t3
UserInfo info = userService.getUser(uid);
OrderStat stat = orderService.stat(uid);
CouponList coupons = couponService.list(uid);
return buildProfile(info, stat, coupons);

// 优化：CompletableFuture 并行，总耗时 = max(t1, t2, t3)
CompletableFuture<UserInfo> infoF = CompletableFuture.supplyAsync(() -> userService.getUser(uid), executor);
CompletableFuture<OrderStat> statF = CompletableFuture.supplyAsync(() -> orderService.stat(uid), executor);
CompletableFuture<CouponList> couponF = CompletableFuture.supplyAsync(() -> couponService.list(uid), executor);
CompletableFuture.allOf(infoF, statF, couponF).join();
return buildProfile(infoF.get(), statF.get(), couponF.get());
```

## 实战场景

| 场景 | 现象 | 优化手段 |
|------|------|----------|
| 订单列表慢 | N+1 查询、缺索引 | 批量查询 + 联合索引 |
| 接口超时 | 调用第三方慢 | 异步化 + 超时熔断 |
| DB 压力大 | 慢 SQL、连接池满 | 索引 + 缓存 + 读写分离 |
| Full GC 频繁 | 堆压力大 | 调大堆、定位内存泄漏 |
| CPU 高 | 死循环、正则 | jstack 定位 + 修复 |
| 雪崩 | 缓存集体失效 | 加随机过期、降级、互斥重建 |
| 链路长 | 多次 RPC | 合并接口、批量化 |
| 大对象序列化 | JSON 慢 | 改 Protobuf、流式 |

## 深挖追问

### 性能优化从哪里开始？

先看数据，不要凭感觉。优先看影响面大且证据明确的瓶颈：

- 接口 P95/P99 超标 → 看 APM 链路找慢在哪一段
- 数据库慢 → 看慢查询日志、EXPLAIN
- 缓存命中率低 → 看缓存 key 设计、TTL
- GC 频繁 → 看 GC 日志、堆 dump
- 线程池满 → 看线程池监控、jstack

按"80/20 原则"：80% 的性能问题来自 20% 的代码，先解决影响最大的瓶颈。

### 如何避免优化引入风险？

- **小步灰度**：分批上线，灰度环境先跑。
- **保留回滚**：每次优化都能独立回滚。
- **关键链路加监控**：RT、错误率、QPS 加告警。
- **压测验证**：上线前压测，确认收益和稳定性。
- **一致性设计**：缓存、异步、并发改动要补幂等和补偿。
- **代码审查**：复杂改动至少两人 review。

### 优化的 ROI 怎么算？

ROI = 收益 / 成本。收益要量化（P95 降多少、QPS 提多少、成本降多少），成本包括开发工时、风险、维护复杂度。优先做高 ROI 的优化，低 ROI 的延后或不做。例如：

- 加索引：收益高（SQL 快 10 倍）、成本低（一条 DDL）→ 高 ROI
- 分库分表：收益高、成本高（改造成本大）→ 中 ROI，看必要性
- JVM 调参：收益中（GC 改善）、成本低 → 中 ROI

### 过早优化有什么问题？

- 浪费时间：优化了用不到的代码。
- 增加复杂度：抽象层、配置项让代码难懂。
- 引入 bug：优化改动多，bug 概率高。
- 锁死方案：过早优化让架构难以调整。

Knuth 名言："过早优化是万恶之源"。但要区分"刻意优化"和"避免明显的低效代码"。后者是基本功，前者要等有数据支撑再做。

### 优化完成后怎么沉淀？

- **监控告警**：核心指标加告警，防止回退。
- **防回退测试**：压测基准加进 CI，性能回退自动报警。
- **文档**：记录瓶颈、方案、收益，方便后续回顾。
- **代码规范**：把踩过的坑写进规范，避免重复犯错。
- **复盘**：上线一段时间后复盘效果，验证预期。

### 怎么衡量优化效果？

要量化对比：

- 优化前 P95/P99、QPS、错误率、资源利用率
- 优化后同指标对比
- 业务影响（用户投诉、订单转化）
- 成本变化（机器数、带宽）

只优化不衡量 = 没优化。监控和压测数据是优化的唯一证据。

### 缓存引入的问题怎么处理？

缓存是双刃剑，性能提升同时带来：

- **数据一致性**：写 DB 后缓存未及时更新。方案：延迟双删、Canal 监听 binlog、TTL 兜底。
- **缓存穿透**：查不存在的 key。方案：空值缓存、布隆过滤器。
- **缓存击穿**：热 key 过期瞬间大量请求穿透。方案：互斥重建、热 key 永不过期。
- **缓存雪崩**：大量 key 同时过期。方案：过期时间加随机偏移、多级缓存、限流降级。

## 易错点

- 没有指标的优化是主观猜测，必须先量化。
- 只看平均值不看 P95/P99，长尾问题被掩盖。
- 缓存引入一致性问题，异步引入补偿问题，要同步设计。
- 过早优化浪费时间，先做高 ROI 的。
- 优化不验证 = 没做，要有压测和监控数据。
- 优化不沉淀 = 下次还会犯，加监控和防回退测试。
- 一次改太多变量，出问题不知道哪个改动导致的。
- 优化只考虑正常流量，不考虑异常和故障场景。

## 总结

项目优化的核心是"目标 → 现状 → 定位 → 方案 → 验证 → 沉淀"闭环。重点掌握：五个优化方向、工具链、常见手段及收益、ROI 计算、避免风险。优化要数据驱动，避免过早优化和主观猜测。缓存、异步、批处理是性能优化三板斧，但都引入新问题（一致性、补偿、实时性），必须配套设计。优化完成后要沉淀监控、文档、防回退测试，否则下次还会犯。
