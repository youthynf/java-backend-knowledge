# 如果让你优化一个项目，你会怎么做？

## 核心概念

项目优化不是“哪里慢改哪里”，而是基于目标和数据的系统性改进。常见方向包括性能、稳定性、可维护性、成本、安全和研发效率。

## 面试官想考什么

- 是否能从业务目标出发，而不是直接堆技术；
- 是否有定位瓶颈、量化收益、灰度验证的意识；
- 是否知道不同优化方向之间可能冲突；
- 是否有项目复盘和长期治理能力。

## 标准回答

> 我会先明确优化目标，例如降低接口耗时、提升吞吐、减少故障、降低成本或改善代码可维护性。然后通过监控、日志、链路追踪、慢 SQL、压测报告定位瓶颈，按收益和风险排序。具体手段可能包括 SQL 和索引优化、缓存、异步化、批处理、限流降级、JVM 参数调整、代码重构、拆分热点模块。最后通过灰度发布、压测和监控验证效果，并补充文档和告警。

## 深挖追问

### 性能优化从哪里开始？

先看数据：接口 P95/P99、QPS、错误率、数据库慢查询、缓存命中率、线程池队列、GC 指标。优先优化影响面大且证据明确的瓶颈。

### 如何避免优化引入风险？

小步灰度，保留回滚方案；关键链路加监控和告警；涉及缓存、异步和并发的改动要补充一致性和幂等设计。

## 实战场景

一个订单列表接口响应慢，排查发现存在 N+1 查询和缺失索引。优化方案包括补充联合索引、批量查询关联数据、增加分页限制、对热点维度加缓存，并通过压测确认 P95 明显下降。

## 易错点/总结

- 没有指标的优化容易变成主观猜测；
- 缓存会引入一致性问题，异步会引入补偿问题；
- 过早优化可能浪费时间；
- 优化完成后要沉淀监控、文档和防回退测试。

## 代码示例

```sql
-- 原查询按 user_id 和 create_time 过滤、排序
CREATE INDEX idx_order_user_time ON t_order(user_id, create_time DESC);
```

优化时不能只看单条 SQL 执行计划，还要看数据分布、索引选择性、分页深度和是否回表。对于深分页可以改成游标分页；对于写多读少的场景，缓存收益可能不如索引和批量化明显。

## 深挖补充

回答项目优化题时，最好按“目标-现状-方案-验证-沉淀”闭环表达：先给出量化指标，再说明定位证据，最后说明上线后 P95、错误率、资源成本等指标变化。这样比单纯罗列缓存、MQ、分库分表更像真实项目经验。
