## 系统设计方法论

### 1. 明确需求（5分钟）

**功能需求：**
- 系统要做什么？
- 核心功能有哪些？
- 用户是谁？

**非功能需求：**
- 性能：QPS、响应时间
- 可用性：SLA 目标
- 扩展性：用户增长预期
- 数据一致性要求

### 2. 容量估算（5分钟）

**QPS 估算：**
```
日活用户 × 每人操作次数 ÷ 峰值系数
示例：100万日活 × 10次操作 ÷ 86400秒 × 峰值系数3 ≈ 350 QPS
峰值 QPS = 平均 QPS × 3~5
```

**存储估算：**
```
每日数据量 = QPS × 每条数据大小 × 86400
示例：1000 QPS × 1KB × 86400 ≈ 82GB/天
月存储 = 82GB × 30 ≈ 2.4TB/月
```

### 3. 系统架构（10分钟）

**画图要素：**
```
客户端 → 负载均衡 → 应用服务器 → 缓存
                            ↓
                         数据库
                            ↓
                        消息队列
```

**常用组件：**
- 负载均衡：Nginx、HAProxy
- 缓存：Redis、Memcached
- 数据库：MySQL、PostgreSQL
- 消息队列：Kafka、RocketMQ
- 搜索引擎：Elasticsearch
- 配置中心：Nacos、Apollo

### 4. 核心难点（10分钟）

分析系统核心问题：
- 高并发 → 缓存、限流、削峰
- 高可用 → 集群、熔断、降级
- 数据一致性 → 分布式事务
- 数据量大 → 分库分表

### 5. 扩展讨论（10分钟）

- 如何应对用户增长？
- 单点故障如何处理？
- 如何监控告警？
- 安全问题？

## 面试高频问题

### Q1: 如何设计一个高并发系统？

**答案要点：**

1. **缓存**
   - 多级缓存：本地缓存 + 分布式缓存
   - 缓存预热、缓存穿透、缓存击穿、缓存雪崩

2. **异步**
   - 消息队列解耦
   - 削峰填谷

3. **分治**
   - 分库分表
   - 微服务拆分

4. **限流**
   - 令牌桶、漏桶
   - 应用层限流、网关限流

5. **降级熔断**
   - Hystrix、Sentinel
   - 服务降级策略

### Q2: 如何保证系统高可用？

**答案要点：**

1. **冗余设计**
   - 多机房部署
   - 主从切换

2. **故障隔离**
   - 熔断器
   - 线程池隔离

3. **快速恢复**
   - 限流降级
   - 灰度发布
   - 回滚机制

4. **监控告警**
   - Prometheus + Grafana
   - 日志中心 ELK

### Q3: 如何做容量规划？

**答案要点：**

```
1. 估算 QPS
   日活用户 × 人均操作 ÷ 86400 × 峰值系数

2. 计算机器数量
   单机 QPS = 1000（经验值）
   机器数 = 总 QPS ÷ 单机 QPS × 冗余系数(1.5)

3. 存储规划
   数据保留周期 × 每日增量

4. 带宽估算
   QPS × 平均响应大小
```

## 系统设计模板

### 问题陈述

> 设计一个 XXX 系统

### 1. 需求澄清

**面试官：** 我需要设计一个秒杀系统。
**候选人：** 我先确认一下需求：
- 日活用户大概是多少？
- 秒杀商品的库存量级？
- 是否允许超卖？
- 秒杀时间窗口多长？

### 2. 容量估算

```
假设：
- 日活用户：100万
- 秒杀参与率：50%
- 秒杀时间：10分钟
- 峰值系数：10

QPS = 100万 × 50% ÷ (10 × 60) × 10 ≈ 8000 QPS
```

### 3. 系统架构

```
┌─────────┐    ┌─────────┐    ┌──────────────┐
│ 客户端  │───→│ CDN/静态│───→│ 负载均衡     │
└─────────┘    └─────────┘    └──────┬───────┘
                                      │
                    ┌─────────────────┼─────────────────┐
                    ↓                 ↓                 ↓
              ┌──────────┐     ┌──────────┐     ┌──────────┐
              │ 应用服务 │     │ 应用服务 │     │ 应用服务 │
              └────┬─────┘     └────┬─────┘     └────┬─────┘
                   │                │                │
                   └────────────────┼────────────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ↓               ↓               ↓
              ┌──────────┐   ┌──────────┐   ┌──────────┐
              │  Redis   │   │  MySQL   │   │  MQ      │
              │ (库存)   │   │ (订单)   │   │ (异步)   │
              └──────────┘   └──────────┘   └──────────┘
```

### 4. 核心难点与解决方案

**难点1：防止超卖**
- Redis 原子操作扣减库存
- 数据库乐观锁兜底

**难点2：高并发压力**
- 多级缓存
- 消息队列削峰
- 限流

**难点3：数据一致性**
- 分布式事务
- 最终一致性

### 5. 扩展讨论

- 如何防止黄牛？（风控、验证码）
- 如何保证公平？（随机排队）
- 如何应对更大流量？（分桶、排队）

## 常见错误

### 1. 一上来就画图

❌ 错误：直接画架构图
✅ 正确：先问需求，再画图

### 2. 忽略约束条件

❌ 错误：不考虑成本、时间
✅ 正确：权衡取舍，说明原因

### 3. 过度设计

❌ 错误：上来就微服务、分库分表
✅ 正确：从简单架构开始，逐步演进

### 4. 没有数据支撑

❌ 错误："需要很多服务器"
✅ 正确："按 10000 QPS 估算，需要 15 台应用服务器"
