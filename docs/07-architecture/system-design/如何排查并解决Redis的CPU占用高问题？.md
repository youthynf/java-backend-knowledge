# 如何排查并解决 Redis 的 CPU 占用高问题？

## 核心概念

Redis 单线程执行命令为主，CPU 高通常意味着某些命令执行成本高、请求量过大、数据结构使用不当，或后台任务/持久化/网络开销造成压力。

## 面试官想考什么

- 是否知道 Redis CPU 高的常见原因；
- 是否会用 `slowlog`、`monitor`、`info` 等命令定位问题；
- 是否理解大 key、热 key、复杂命令的影响；
- 是否能给出止血和长期优化方案。

## 标准回答

> 我会先看 Redis 所在机器的 CPU、QPS、连接数、慢日志和命令统计，判断是流量突增还是慢命令导致。常见原因包括大 key 操作、`keys`/`hgetall`/大范围 `zrange` 等复杂命令、热 key 访问集中、Lua 脚本耗时、持久化或 AOF rewrite 压力。解决时可以先限流、拆分热 key、禁用高危命令、优化数据结构和访问模式，必要时做读写分离或分片。

## 排查命令

```bash
# 查看整体状态
redis-cli info stats
redis-cli info commandstats

# 查看慢查询
redis-cli slowlog get 20

# 谨慎使用，生产高峰不建议长时间 monitor
redis-cli monitor
```

## 深挖追问

### 大 key 为什么会导致 CPU 高？

大 key 操作一次处理大量元素，会阻塞 Redis 主线程，导致其他请求排队。例如对超大 hash 执行 `hgetall`，或删除大 key 时同步释放内存，都可能造成抖动。

### 热 key 怎么处理？

可以本地缓存、拆分 key、增加副本读、请求合并、限流，或把热点数据提前推送到多级缓存。具体方案取决于读写比例和一致性要求。

## 易错点/总结

- 不要在生产高峰长时间使用 `monitor`；
- 禁止或谨慎使用 `keys *` 这类全量扫描命令；
- Redis CPU 高要结合慢日志和命令统计，不要只看机器指标；
- 删除大 key 可考虑异步删除或分批处理。

## 面试复习强化

### 核心概念

从面试角度看，**如何排查并解决 Redis 的 CPU 占用高问题？** 可以放在“系统设计与排障”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 按“现象确认→指标定位→根因分析→临时止血→长期治理”回答。 追问会覆盖流量、线程/连接池、缓存、数据库、MQ、下游依赖、机器资源和发布变更。 对于“如何排查并解决 Redis 的 CPU 占用高问题？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- CPU 排查要落到 top -H、jstack、火焰图、GC 日志和热点代码，区分用户态、内核态和 IO wait。
- Redis CPU 高常见原因包括大 key、热 key、慢命令、Lua 脚本、持久化/过期风暴和网络包量过大。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：给结论前先给证据，实战动作要包括监控、日志、压测复现、灰度回滚、限流降级和容量评估。
