# 如何设计秒杀场景处理高并发以及超卖现象

## 核心概念

秒杀系统的核心矛盾是**瞬时高并发 + 库存有限 + 不能超卖 + 用户体验**。设计目标不是让所有请求都打到数据库，而是在入口削峰、缓存层快速判断、消息队列异步排队、数据库最终一致扣减。一句话点题：秒杀系统的核心是"分层过滤 + 异步削峰 + 原子扣减 + 兜底对账"，把 99% 的请求挡在数据库之前。

典型链路：活动预热 → CDN 静态化 → 网关限流 → 风控验证 → Redis 预扣库存 → MQ 削峰 → 异步创建订单 → 数据库最终扣减 → 超时回补。

## 标准回答

秒杀系统按"四层过滤"设计：**入口层**做 CDN 静态化、验证码、限流、风控，挡掉大部分无效请求；**缓存层**用 Redis Lua 原子扣减库存并记录用户购买，扣减失败直接返回；**队列层**扣减成功后写 MQ，订单服务异步消费创建订单；**数据库层**用 `UPDATE ... WHERE stock > 0` 条件更新做最终防线，配合唯一索引防重复下单。防超卖靠"Redis 原子扣减 + DB 条件更新"双重保证；防重复下单靠"用户维度 Redis set + 唯一索引"；可靠性靠"幂等消费 + 死信队列 + 对账补偿"。

## 实现原理

### 全链路设计

```
用户 → CDN(静态) → 网关(限流/风控) → 应用层(库存预热) → Redis(Lua 预扣) → MQ(削峰) → 订单服务(异步) → DB(最终扣减)
                                                                    ↓
                                                                用户查询"排队中"
```

1. **活动预热**：开始前把库存、活动规则、用户白名单加载到 Redis。
2. **入口限流**：网关层用 Sentinel/令牌桶限制 QPS，挡住突发流量。
3. **风控验证**：验证码、设备指纹、黑名单、用户频率控制，挡住机器人。
4. **Redis 预扣**：Lua 脚本原子执行"判断是否已买 + 判断库存 + 扣减 + 记录用户"。
5. **MQ 削峰**：扣减成功后发送消息，订单服务按消费能力处理。
6. **异步创建订单**：消费者创建订单、扣减数据库库存。
7. **数据库兜底**：`WHERE stock > 0` 条件更新，保证不超卖。
8. **超时回补**：未支付订单超时后取消，回补 Redis 和数据库库存。
9. **兜底对账**：定时对账 Redis 库存、数据库库存、订单数量。

### 防超卖的双层保障

**Redis 层**用 Lua 脚本保证"判断库存 + 扣减 + 记录用户"原子执行：

```lua
local stockKey = KEYS[1]
local userKey = KEYS[2]
local userId = ARGV[1]

-- 1. 判断是否已购买
if redis.call('SISMEMBER', userKey, userId) == 1 then
    return -2  -- 已购买
end

-- 2. 判断库存
local stock = tonumber(redis.call('GET', stockKey) or '0')
if stock <= 0 then
    return -1  -- 售罄
end

-- 3. 扣减库存 + 记录用户
redis.call('DECR', stockKey)
redis.call('SADD', userKey, userId)
return 1  -- 成功
```

**数据库层**用条件更新兜底，避免 Redis 异常导致超卖：

```sql
UPDATE seckill_stock
SET stock = stock - 1
WHERE sku_id = ? AND stock > 0;
```

返回影响行数 0 表示库存不足，订单创建失败，回补 Redis 库存。

### 防重复下单

- **Redis 用户维度 set**：记录已购买用户，Lua 脚本里检查。
- **数据库唯一索引**：`UNIQUE(user_id, activity_id, sku_id)`，重复插入直接报错。
- **MQ 消费幂等**：消息 ID 或业务 ID 建消费记录，重复消息跳过。

## 代码示例

Redis 预扣库存：

```java
@Service
public class SeckillService {
    private final StringRedisTemplate redis;
    private final RocketMQTemplate mq;

    private static final DefaultRedisScript<Long> SECKILL_SCRIPT;
    static {
        SECKILL_SCRIPT = new DefaultRedisScript<>();
        SECKILL_SCRIPT.setLocation(new ClassPathResource("seckill.lua"));
        SECKILL_SCRIPT.setResultType(Long.class);
    }

    public SeckillResult seckill(Long activityId, Long skuId, Long userId) {
        String stockKey = "seckill:stock:" + skuId;
        String userKey = "seckill:users:" + activityId + ":" + skuId;
        Long result = redis.execute(SECKILL_SCRIPT,
                Arrays.asList(stockKey, userKey), userId.toString());
        if (result == -2L) return SeckillResult.duplicate();
        if (result == -1L) return SeckillResult.soldOut();
        // 扣减成功，发 MQ 异步创建订单
        mq.asyncSend("seckill-order", new SeckillMessage(activityId, skuId, userId));
        return SeckillResult.queuing();
    }
}
```

订单消费者幂等处理：

```java
@RocketMQMessageListener(topic = "seckill-order", consumerGroup = "order-consumer")
public class OrderConsumer implements RocketMQListener<SeckillMessage> {
    @Override
    @Transactional
    public void onMessage(SeckillMessage msg) {
        // 1. 幂等检查
        if (orderMapper.existsByUserAndActivity(msg.getUserId(), msg.getActivityId())) {
            return;
        }
        // 2. 数据库扣减库存（条件更新）
        int rows = stockMapper.decrStock(msg.getSkuId());
        if (rows == 0) {
            // Redis 库存和 DB 不一致，回补 Redis
            redis.opsForValue().increment("seckill:stock:" + msg.getSkuId());
            return;
        }
        // 3. 创建订单
        orderMapper.insert(buildOrder(msg));
    }
}
```

```sql
-- 库存条件扣减
UPDATE seckill_stock SET stock = stock - 1
WHERE sku_id = #{skuId} AND stock > 0;

-- 唯一索引防重复
CREATE UNIQUE INDEX uk_user_activity_sku
ON seckill_order(user_id, activity_id, sku_id);
```

## 实战场景

| 环节 | 方案 | 注意点 |
|------|------|--------|
| 入口限流 | 网关 Sentinel 令牌桶 | 限流值要压测确认 |
| 静态资源 | CDN + 页面静态化 | 商品详情页动态部分用接口 |
| 风控 | 验证码 + 设备指纹 + 黑名单 | 别误伤正常用户 |
| 库存预热 | 活动开始前加载到 Redis | 预热要校验和 DB 一致 |
| Redis 扣减 | Lua 原子脚本 | 集群模式注意跨 slot |
| MQ 削峰 | Kafka/RocketMQ | 消费者按消费能力扩容 |
| 异步下单 | 消费者创建订单 | 幂等 + 死信队列 |
| DB 扣减 | 条件更新 `stock > 0` | 高并发下注意行锁 |
| 超时回补 | 延迟队列/定时任务 | 回补 Redis + DB |
| 兜底对账 | 定时全量对账 | 不一致时告警人工介入 |

## 深挖追问

### 如何防止超卖？

双重保障：Redis Lua 脚本原子扣减（前置），数据库 `WHERE stock > 0` 条件更新（兜底）。任何一层单独都不够：Redis 异常会导致 Redis 库存和 DB 不一致；DB 直接扛流量会拖垮数据库。两层叠加保证即使 Redis 异常，DB 也不会超卖。

### 如何防止重复下单？

三层防护：Redis 用户 set（前置，挡住同一用户重复请求）、数据库唯一索引（兜底，挡住重复插入）、MQ 消费幂等（中间，挡住消息重复消费）。三层任何一层都能独立阻止重复，叠加保证健壮。

### MQ 异步下单会不会丢单？

会，但有兜底。可靠消息机制：

- **发送确认**：生产者等 MQ 确认收到，失败重试。
- **消费重试**：消费失败自动重试 N 次。
- **死信队列**：N 次重试失败进死信队列，人工处理。
- **本地消息表**：业务和消息发送在一个事务里，定时扫描未发送的消息补偿。
- **订单状态查询接口**：用户查询"排队中"，最终通过查询接口确认是否成功。

### Redis 库存和数据库库存怎么对账？

定时任务（如每 5 分钟）对账：

- Redis 库存 = 数据库库存 + 待处理 MQ 消息数 + 处理中订单数
- 不一致时告警，人工介入或自动补偿
- 大型活动后做全量对账

### 集群模式 Lua 脚本跨 slot 怎么办？

Redis Cluster 下 Lua 脚本里所有 key 必须在同一 slot。stockKey 和 userKey 用 `{skuId}:stock` 和 `{skuId}:users` 这种 hash tag 强制同 slot。

### 秒杀流量是平时的多少倍？

典型秒杀活动瞬时 QPS 可能是日常的 100-1000 倍。日常 1 万 QPS 的系统，秒杀时可能瞬间到 100 万 QPS。这是为什么必须分层过滤：网关限流挡掉 90%、Redis 售罄挡掉 80%、MQ 削峰剩余流量，最终到 DB 的可能只有几百 QPS。

### 库存回补怎么处理？

- **超时未支付**：延迟队列（如 RocketMQ 延迟消息、Redis ZSet），到期检查订单状态，未支付则取消订单 + 回补 Redis 和 DB 库存。
- **支付失败**：支付回调失败时取消订单 + 回补。
- **消费失败**：MQ 消费失败回补 Redis。
- **回补要幂等**：避免重复回补导致库存虚高。

### 为什么不让请求直接打到数据库？

数据库单机 QPS 上限通常几千（写），秒杀瞬时 100 万 QPS 直接打过去：连接池打满、行锁排队、TPS 雪崩、整个库可能挂掉。Redis 单机 QPS 上限 10 万+，配合 MQ 削峰，把 DB 写入控制在它能承受的范围内（如几百 QPS），系统才稳。

## 易错点

- 不要让所有请求直接打到数据库，否则秒杀一开始数据库就挂了。
- Redis 扣减必须用 Lua 保证原子性，不要"先 GET 再 DECR"。
- 数据库扣减必须用 `WHERE stock > 0`，不要"先 SELECT 再 UPDATE"。
- MQ 消费必须幂等，重复消息不能创建重复订单。
- 库存回补要处理超时、取消、失败多种场景，且要幂等。
- 热点活动要提前压测，准备限流、降级、扩容预案。
- 活动结束、库存售罄要快速关闭入口，避免无效流量继续冲击。
- 集群模式 Lua 注意 hash tag。

## 总结

秒杀系统设计的核心是"分层过滤 + 异步削峰 + 原子扣减 + 兜底对账"。重点掌握：四层过滤链路、Redis Lua 原子扣减、DB 条件更新兜底、防超卖防重复、MQ 异步下单、库存回补、对账机制。和普通高并发系统相比，秒杀系统对"防超卖"和"削峰"的要求更极致，必须做到 99% 请求不到 DB。监控、压测、降级、对账是上线前必须准备好的配套。
