# 导致 CPU 飙升到 100% 有什么原因？

## 核心概念

CPU 飙升通常意味着计算资源被大量消耗，原因可能来自应用代码、JVM、系统资源或外部流量。面试回答时要按类别展开，而不是只说“死循环”。

## 面试官想考什么

- 是否能系统性列举原因；
- 是否知道 Java 应用中 GC、线程、锁、算法都可能导致 CPU 高；
- 是否能结合排查手段验证猜想；
- 是否知道临时止血和长期优化的区别。

## 标准回答

> 常见原因包括：业务代码死循环或递归；流量突增导致请求线程大量计算；频繁 Full GC；锁竞争和自旋；复杂 SQL 或大量数据在应用层处理；正则表达式灾难回溯；日志量过大；加解密、压缩、序列化等 CPU 密集操作；容器 CPU limit 设置过低。排查时要通过 `top`、`top -Hp`、`jstack`、`jstat`、日志和监控来定位具体原因。

## 深挖追问

### CPU 高和内存有什么关系？

内存不足可能导致频繁 GC，GC 线程会消耗大量 CPU，表现为 CPU 高、吞吐下降、响应变慢。因此 CPU 排查要同时关注 GC 次数、停顿时间和堆使用率。

### 锁竞争为什么会导致 CPU 高？

大量线程竞争锁时，可能产生上下文切换、自旋等待或阻塞唤醒开销。如果代码中存在高频同步块、热点锁或并发容器使用不当，就可能导致 CPU 被消耗在等待和调度上。

## 实战场景

某接口上线后 CPU 突然升高，线程栈显示大量线程停留在正则匹配方法。排查发现用户输入触发了复杂正则的回溯，优化正则并限制输入长度后恢复。

## 易错点/总结

- CPU 高不是原因，是现象；
- 不要只凭经验改代码，要用线程栈和监控验证；
- 先止血再根因，例如限流、扩容、回滚；
- 复盘时要补充压测、告警和代码审查规则。

## 代码示例

```bash
top                         # 找到 CPU 高的 Java 进程
top -Hp <pid>               # 找到最耗 CPU 的线程 nid
printf "%x\n" <tid>         # 线程 id 转十六进制
jstack <pid> | grep -A30 nid # 定位线程栈
jstat -gcutil <pid> 1s 10    # 判断是否频繁 GC
```

如果线程栈多次采样都指向同一段业务代码，通常是算法、循环、正则或序列化问题；如果主要是 GC 线程，则要继续分析对象分配、内存泄漏和堆参数。

## 深挖补充

线上处理要先控制影响面：限流、降级、扩容或回滚；再保留现场信息，例如线程栈、GC 日志、火焰图、容器指标。不要重启后才开始排查，否则关键证据可能丢失。
