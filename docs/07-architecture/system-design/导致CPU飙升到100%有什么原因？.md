# 导致 CPU 飙升到 100% 有什么原因

## 核心概念

CPU 飙升是现象不是原因。Java 应用 CPU 高，可能来自业务代码、JVM、并发、外部流量、容器配置多个层面。一句话点题：CPU 高的根因无非"算得多、GC 频繁、锁竞争、流量大、配置错"五类，面试时要按类别展开，不能只说"死循环"。

排查思路：先分类（业务/GC/锁/流量/系统），再用对应工具验证。

## 标准回答

Java 应用 CPU 飙升的常见原因按类分：**业务代码**（死循环、深递归、正则回溯、大对象序列化、加解密压缩）；**JVM**（频繁 Full GC、JIT 编译）；**并发**（锁竞争、自旋、上下文切换）；**流量**（QPS 突增、热点接口）；**系统/容器**（CPU limit 过低、JNI、内核线程）。排查时用 `top`/`top -Hp`/`jstack` 定位业务线程，`jstat`/GC 日志判断 GC，`vmstat` 看上下文切换，监控看流量趋势。临时止血优先级：摘流量、限流、扩容、回滚，重启最后。

## 实现原理

### 五类原因详解

#### 1. 业务代码问题

- **死循环/无限递归**：`while(true)` 没退出条件、递归没终止条件。线程栈会一直停在同一处。
- **正则灾难回溯**：复杂正则匹配特定输入时回溯指数级增长，如 `(a+)+` 匹配 `aaaaaaaaaaaaaaaa!`。线程栈停在 `Pattern.matcher`。
- **大对象序列化**：JSON 序列化嵌套深的对象，CPU 长时间占用。
- **加解密/压缩**：AES、RSA、GZIP 大文件压缩，本身 CPU 密集。
- **复杂算法**：O(n²)、O(n³) 算法在大数据集上跑，如嵌套循环、冒泡排序。

#### 2. JVM 问题

- **频繁 Full GC**：堆太小、内存泄漏、大对象进老年代。`jstat -gcutil` 看 FGC 列激增，GC 线程吃满 CPU。
- **Young GC 频繁**：Eden 区太小，对象频繁进 Survivor。YGC 列高，每次 GC 暂停应用线程。
- **JIT 编译**：热点代码 C2 编译时占用 CPU，通常启动阶段明显，运行中偶发。

#### 3. 并发问题

- **锁竞争**：高频同步块、热点锁。线程 BLOCKED，上下文切换高。
- **自旋锁**：CAS 自旋失败重试，CPU 空转。
- **上下文切换**：线程数过多（如几千个），调度开销大。`vmstat 1` 看 `cs` 列。

#### 4. 流量问题

- **QPS 突增**：营销活动、上游异常重试、爬虫。多线程都活跃。
- **热点接口**：单个接口被打爆，相关线程池满。
- **重试风暴**：上游超时重试放大流量（如熔断未生效）。

#### 5. 系统/容器问题

- **CPU limit 过低**：容器 cgroup 限制 CPU，单核 limit 下 Java 看到 100% 实际是 1 核 100%。
- **JNI 调用**：native 代码死循环，jstack 看不到完整栈。
- **内核线程**：IO 等待、内存回收、网络软中断。

## 代码示例

正则回溯（典型 CPU 杀手）：

```java
// 灾难正则：匹配 aaaa...! 时回溯指数增长
String pattern = "(a+)+$";
Pattern.compile(pattern).matcher("aaaaaaaaaaaaaaaaaaaaaaaaaaaaa!").matches();
// 这一行可能跑几十秒，CPU 100%
```

死循环：

```java
// 错误：break 条件永远不满足
while (true) {
    if (someFlag) {  // someFlag 在另一线程改了，但这里没 volatile 看不到
        break;
    }
}
```

锁竞争：

```java
// 热点锁：所有请求串行化
public class Counter {
    private final Object lock = new Object();
    private long count = 0;
    public void incr() {
        synchronized (lock) {  // 高并发下成为瓶颈
            count++;
        }
    }
}

// 优化：用 LongAdder 或 AtomicLong
private final LongAdder count = new LongAdder();
public void incr() { count.increment(); }
```

GC 频繁：

```bash
# jstat 看 GC 频率
jstat -gcutil <pid> 1000 10
#  S0     S1     E      O      M     CCS    YGC   YGCT    FGC   FGCT    GCT
#  0.00  90.00  80.00  95.00  95.0  90.0    100  10.5     50   20.3    30.8
# FGC 50 次，FGCT 20 秒，GC 占 CPU 严重

# 解决：调大堆、定位泄漏
-Xmx4g -Xms4g -XX:+UseG1GC
```

上下文切换过高：

```bash
# vmstat 看 cs（context switch）列
vmstat 1
#  r  b   swpd   free   buff  cache  si  so  bi  bo  in  cs us sy id wa st
#  2  0      0 100000  20000 500000   0   0   0   0 5000 50000 80 20  0  0  0
# cs = 50000，上下文切换过高，可能是线程数过多或锁竞争
```

## 实战场景

| 原因类型 | 典型现象 | 排查手段 |
|----------|----------|----------|
| 死循环 | 单线程 100%，栈停同一处 | jstack 多次采样 |
| 正则回溯 | 解析输入时 CPU 飙升 | jstack 看 Pattern 栈 |
| 频繁 Full GC | GC 线程高，响应慢 | jstat、堆 dump |
| 锁竞争 | 多线程 BLOCKED，cs 高 | jstack 看 deadlock |
| 流量突增 | 多线程都活跃，QPS 暴涨 | 入口监控 |
| 序列化 | JSON 序列化栈 | jstack 看 ObjectMapper |
| CPU limit 低 | 容器内 100%，宿主机低 | 调整 limit |
| 自旋 CAS | 高并发 AtomicXxx | jstack、监控 |

## 深挖追问

### CPU 高和内存有什么关系？

内存不足导致频繁 GC，GC 线程吃 CPU，表现为 CPU 高、吞吐下降、响应变慢。所以 CPU 排查要同时关注 GC 次数、停顿时间、堆使用率。如果 FGC 激增且堆接近上限，根因是内存问题，扩堆或定位内存泄漏。

### 锁竞争为什么会导致 CPU 高？

锁竞争有两种极端：

- **大量线程 BLOCKED**：CPU 不高（线程都睡了），但响应慢，看 `cs` 上下文切换。
- **大量线程自旋（CAS）**：CPU 高，线程都在跑但拿不到锁，看 `RUNNABLE` 状态。

实际上锁竞争高发的是上下文切换开销，而不是 CPU 算力消耗。`vmstat 1` 看 `cs` 列，正常几百到几千，几万就异常。

### 容器 CPU 100% 和宿主机 CPU 100% 有什么区别？

容器有 cgroup CPU limit，如 `--cpu=1` 限制 1 核。容器内 `top` 显示 100% 是这 1 核的 100%，对应宿主机 1/8（8 核机器）。如果宿主机 CPU 100%，说明所有容器加起来打满了物理 CPU。K8s 看 `kubectl top pod` 和 `kubectl top node` 区分。

### JIT 编译为什么会让 CPU 高？

JIT 编译器（C1/C2）在热点代码达到阈值时编译成本地代码，编译过程 CPU 密集。通常启动阶段明显（预热），运行中偶发（新热点代码触发编译）。`-XX:+PrintCompilation` 可以看到编译事件。生产环境一般不在意，但启动慢的服务要注意。

### 怎么预防 CPU 飙升？

- **监控告警**：CPU、GC、线程数、RT、QPS 加阈值告警。
- **限流**：网关或 Sentinel 对热点接口限流。
- **压测**：上线前压测，确认容量。
- **代码审查**：关注死循环、正则、大对象、深递归。
- **JVM 参数**：合理设置堆、GC 算法。
- **资源隔离**：容器 CPU limit 合理，避免相互影响。

### CPU 高一定有 bug 吗？

不一定。流量真的很大、业务确实是 CPU 密集（如视频转码、加密计算），CPU 高是正常现象。判断标准：

- 业务 RT 正常、错误率正常 → CPU 高是容量问题，扩容即可
- RT 异常、错误率上升 → CPU 高影响业务，要优化

### 正则回溯怎么避免？

- 用简单的字符类替代复杂分组：`[a-z]+` 比 `(a|b|c|...)+` 安全。
- 避免嵌套量词：`(a+)+`、`(a*)*` 这类组合容易引发指数回溯。
- 用原子组或占有量词（Java 不直接支持，可用 `(?>...)` 模拟）。
- 限制输入长度。
- 用 RE2J 这类线性时间正则引擎。

## 易错点

- CPU 高是现象不是原因，要找根因。
- 不要只凭经验改代码，要用 jstack、jstat、监控验证。
- 先止血再根因：限流、扩容、回滚，重启最后。
- 复盘要补充压测、告警、代码审查规则。
- 容器场景注意 CPU limit 和 cgroup。
- 单次 jstack 不够，要多次采样对比。
- 别忽略 GC 线程、JIT 线程、JNI 这些非业务线程。

## 总结

CPU 飙升原因按五类分：业务代码、JVM、并发、流量、系统。重点掌握每类的典型现象和排查工具（jstack、jstat、vmstat、监控）。和 CPU 排查的"四步定位法"配合：top → top -Hp → printf → jstack。线上处理要先止血（摘流量、限流、扩容）再根因，重启永远最后。预防靠监控、限流、压测、代码审查。CPU 高不一定有 bug，可能只是容量到顶，要看业务影响判断。
