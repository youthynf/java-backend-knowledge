# 单例模式

## 核心概念

单例模式用于保证某个类只有一个实例，常用于配置管理、工具类、线程池管理等需要共享的对象。

## 面试官想考什么

- 单例有哪些实现方式？
- 双重检查锁为什么要用 `volatile`？
- 单例是否天然线程安全？
- Spring 单例和 GoF 单例有什么区别？

## 标准回答

> 单例模式的重点是控制实例创建。常见实现有饿汉式、懒汉式、双重检查锁、静态内部类和枚举。实际项目中更推荐使用 Spring 容器管理对象生命周期。需要注意，单例只表示实例唯一，如果内部有可变状态，仍然要处理并发安全。

## 示例

```java
public enum EnumSingleton {
    INSTANCE;
}
```

枚举单例写法简单，并且天然处理序列化和反射破坏单例的问题。

## 深挖追问

### 为什么不建议到处使用单例？

全局状态会增加耦合，影响单元测试和扩展。如果对象有复杂依赖关系，交给 IOC 容器管理通常更合适。

## 易错点/总结

- 单例不等于线程安全；
- 单例不适合保存用户级上下文；
- 不要为了少创建对象滥用单例；
- 面试中可以结合 Spring Bean 生命周期回答。

## 实战场景/代码示例

例如系统启动时加载一份本地规则表，可以用容器单例缓存不可变快照：

```java
@Component
public class RuleHolder {
    private volatile Map<String, Rule> rules = Map.of();

    public Rule get(String code) {
        return rules.get(code);
    }

    public void refresh(Map<String, Rule> newRules) {
        this.rules = Map.copyOf(newRules);
    }
}
```

这里单例对象本身唯一，但内部状态刷新仍要考虑可见性和线程安全。使用不可变 Map 加 `volatile` 引用替换，比在共享 Map 上原地修改更容易保证并发读安全。

## 深挖补充

如果面试官追问“枚举单例为什么更安全”，可以回答：枚举由 JVM 保证实例唯一，天然支持序列化时维持同一个实例，并且常规反射无法创建枚举对象。不过在 Spring 项目中，很多对象需要依赖注入、生命周期回调和配置管理，直接交给容器通常比手写单例更合适。
