# 单例模式

## 核心概念

单例模式用于保证某个类只有一个实例，常用于配置管理、工具类、线程池管理等需要共享的对象。

## 面试官想考什么

- 单例有哪些实现方式？
- 双重检查锁为什么要用 `volatile`？
- 单例是否天然线程安全？
- Spring 单例和 GoF 单例有什么区别？

## 标准回答

> 单例模式的重点是控制实例创建。常见实现有饿汉式、懒汉式、双重检查锁、静态内部类和枚举。实际项目中更推荐使用 Spring 容器管理对象生命周期。需要注意，单例只表示实例唯一，如果内部有可变状态，仍然要处理并发安全。

## 示例

```java
public enum EnumSingleton {
    INSTANCE;
}
```

枚举单例写法简单，并且天然处理序列化和反射破坏单例的问题。

## 深挖追问

### 为什么不建议到处使用单例？

全局状态会增加耦合，影响单元测试和扩展。如果对象有复杂依赖关系，交给 IOC 容器管理通常更合适。

## 易错点/总结

- 单例不等于线程安全；
- 单例不适合保存用户级上下文；
- 不要为了少创建对象滥用单例；
- 面试中可以结合 Spring Bean 生命周期回答。

## 实战场景/代码示例

例如系统启动时加载一份本地规则表，可以用容器单例缓存不可变快照：

```java
@Component
public class RuleHolder {
    private volatile Map<String, Rule> rules = Map.of();

    public Rule get(String code) {
        return rules.get(code);
    }

    public void refresh(Map<String, Rule> newRules) {
        this.rules = Map.copyOf(newRules);
    }
}
```

这里单例对象本身唯一，但内部状态刷新仍要考虑可见性和线程安全。使用不可变 Map 加 `volatile` 引用替换，比在共享 Map 上原地修改更容易保证并发读安全。

## 深挖补充

如果面试官追问“枚举单例为什么更安全”，可以回答：枚举由 JVM 保证实例唯一，天然支持序列化时维持同一个实例，并且常规反射无法创建枚举对象。不过在 Spring 项目中，很多对象需要依赖注入、生命周期回调和配置管理，直接交给容器通常比手写单例更合适。

## 面试复习强化

### 核心概念

从面试角度看，**单例模式** 可以放在“设计模式”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 先说明变化点与稳定点，再讲类之间职责边界以及调用方如何依赖抽象。 追问常考适用场景、优缺点、与相似模式差异，以及在 Spring/JDK 中的真实应用。 对于“单例模式”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：避免为用模式而用模式；抽象带来的扩展收益要大于复杂度成本。
