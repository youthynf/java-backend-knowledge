# 代理模式

## 核心概念

代理模式为目标对象提供一个代理对象，由代理对象控制对目标对象的访问，并在调用前后增强逻辑。常见形式包括静态代理、JDK 动态代理和 CGLIB 动态代理。

## 面试官想考什么

- 是否理解代理模式解决什么问题；
- 是否能区分静态代理和动态代理；
- 是否知道 JDK 动态代理和 CGLIB 的区别；
- 是否能结合 Spring AOP、事务、RPC 举例。

## 标准回答

> 代理模式用于在不修改目标类代码的情况下增强功能，比如日志、鉴权、事务、监控。静态代理需要手写代理类；JDK 动态代理基于接口生成代理对象；CGLIB 通过继承目标类生成子类代理。Spring AOP 会根据目标类是否有接口等条件选择代理方式。

## 深挖追问

### JDK 动态代理和 CGLIB 区别？

JDK 动态代理要求目标类实现接口，代理对象实现同样接口；CGLIB 通过生成目标类子类实现代理，因此不能代理 `final` 类或 `final` 方法。

### Spring 事务为什么会失效？

事务依赖代理拦截方法调用。同类内部方法调用绕过代理，`private` 方法、`final` 方法、异常被吞掉、传播行为设置不当等都可能导致事务不生效。

## 实战场景

在订单服务中，希望统一记录方法耗时，不应在每个业务方法里手写日志，可以通过 AOP 代理在方法调用前后记录耗时和异常。

## 易错点/总结

- 代理增强的是“通过代理对象发起的调用”；
- 动态代理不是只能做 AOP，也常用于 RPC 客户端；
- CGLIB 不能代理 final；
- 代理会增加调用链复杂度，排查问题时要注意真实执行对象。

## 代码示例

```java
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    long start = System.currentTimeMillis();
    try {
        return method.invoke(target, args);
    } finally {
        log.info("method={}, cost={}ms", method.getName(), System.currentTimeMillis() - start);
    }
}
```

这类统一增强适合横切逻辑，但不适合把复杂业务规则藏进代理层，否则调用方很难从代码表面看出真实行为。

## 面试复习强化

### 核心概念

从面试角度看，**代理模式** 可以放在“设计模式”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 先说明变化点与稳定点，再讲类之间职责边界以及调用方如何依赖抽象。 追问常考适用场景、优缺点、与相似模式差异，以及在 Spring/JDK 中的真实应用。 对于“代理模式”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：避免为用模式而用模式；抽象带来的扩展收益要大于复杂度成本。
