# 模板方法模式

## 核心概念

模板方法模式在父类中定义算法骨架，把某些步骤延迟到子类实现。它适合流程稳定、部分步骤可变的场景。

## 面试官想考什么

- 是否理解“固定流程 + 可变步骤”的设计思想；
- 是否能区分模板方法和策略模式；
- 是否知道模板方法可能带来的继承耦合；
- 是否能在项目中找到类似场景。

## 标准回答

> 模板方法模式把通用流程放在抽象父类中，例如参数校验、执行前置逻辑、核心处理、结果封装、异常处理等，子类只实现变化的步骤。这样可以复用流程代码，避免多个子类重复实现相同骨架。但它基于继承，层级过深会增加维护成本。

## 示例

```java
public abstract class ImportTemplate {
    public final void importData() {
        checkFile();
        parse();
        save();
        afterImport();
    }

    protected abstract void parse();
    protected abstract void save();
    protected void afterImport() {}
}
```

不同文件类型只实现解析和保存逻辑，整体导入流程保持一致。

## 深挖追问

### 为什么模板方法常把主流程方法设为 final？

为了防止子类随意改动流程骨架，保证核心流程稳定。如果流程本身也经常变化，模板方法可能不适合。

## 易错点/总结

- 模板方法适合流程稳定的业务；
- 继承层级不要太深；
- 可选步骤可以用钩子方法；
- 如果需要运行时灵活切换算法，策略模式更合适。
