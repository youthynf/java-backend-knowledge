# 设计模式补充

本目录补充常见创建型设计模式，适合和 `patterns` 目录一起复习。重点不是背 UML，而是说明模式解决了什么变化点。

## 文档导航

- [工厂模式](factory.md)
- [单例模式](singleton.md)

## 面试官想考什么

- 创建对象的逻辑为什么要封装？
- 简单工厂、工厂方法、抽象工厂有什么区别？
- 单例如何保证线程安全？
- Spring Bean 默认单例和设计模式里的单例有什么差异？

## 标准回答

> 创建型模式主要解决对象创建复杂、调用方依赖具体类、扩展成本高的问题。实际项目中如果创建逻辑很简单，不一定需要引入复杂模式；如果对象族、配置、插件化扩展较多，工厂模式会更合适。

## 易错点

- 误以为用了 Spring 就完全不需要理解设计模式；
- 工厂模式层级设计过深，反而降低可读性；
- 单例持有可变状态，引发并发问题。
