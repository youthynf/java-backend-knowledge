# 如何理解DIP/IOC/DI？

如何理解DIP/IOC/DI？
依赖倒置、依赖注入和控制反转是软件开发中重要的设计原则与设计模式相关概念，在提升代码的可维护性、可测试性和可扩展性方面发挥关键作用。
依赖倒置（Dependency Inversion Principle, DIP）
•  定义：一种设计原则，核心思想是高层模块不应该依赖低层模块（具体实现），二者都应该依赖其抽象；抽象不应该依赖细节，细节应该依赖抽象。
•  目的：通过依赖于抽象而不是具体实现，降低模块间的耦合度，使得系统更具灵活性和可维护性。当需求变化时，只需调整具体的实现类，而不会影响到高层模块的代码。
•  示例：在一个电商系统中，高层模块 “订单处理” 不应该直接依赖低层模块 “支付宝支付” 或 “微信支付” 的具体实现，而是依赖 “支付接口” 这个抽象。这样，无论添加新的支付方式（如银联支付），还是修改现有支付方式的实现，都不会影响 “订单处理” 模块的代码。

依赖注入（Dependency Injection, DI）
•  定义：一种设计模式，是实现依赖倒置原则的一种具体方式。在这种模式中，将对象所依赖的外部资源（依赖）通过构造函数、方法参数或属性等方式传递（注入）给该对象，而不是由对象自己去创建或查找依赖。
•  目的：分离对象的创建和使用，使代码更加松耦合，便于测试和维护。通过依赖注入，可以在不修改对象内部代码的情况下，替换其依赖的对象，从而实现不同的行为。
•  示例：有一个 “用户服务” 类依赖 “邮件服务” 来发送注册通知邮件。可以通过构造函数将 “邮件服务” 的实例注入到 “用户服务” 中。这样，在测试 “用户服务” 时，可以轻松地注入一个模拟的 “邮件服务” 实例，而无需真正发送邮件。

控制反转（Inversion of Control, IoC）
•  定义：是一种设计思想，并非某种具体的技术实现。它将传统上由程序代码直接控制的对象创建、依赖管理等职责，转交给一个外部的容器（IoC 容器）来管理。即控制权由应用程序本身转移到了外部容器，实现了控制的反转。
•  目的：通过 IoC，使得对象之间的依赖关系变得更加清晰和易于管理，提高了代码的可维护性和可测试性。同时，IoC 容器可以根据配置动态地创建对象、组装对象之间的依赖关系，使得系统更加灵活和可扩展。
•  示例：在一个 Java Web 应用中，使用 Spring 框架作为 IoC 容器。Spring 容器负责创建和管理应用中的各种 Bean（对象），并根据配置文件或注解来注入它们之间的依赖关系。开发人员只需要关注业务逻辑的实现，而无需手动管理对象的创建和依赖关系。
