# 如何解决哈希冲突问题

## 核心概念

哈希冲突指的是：不同 key 经过哈希函数计算后，落到了同一个哈希桶位置。只要哈希空间有限、输入数据可能无限，冲突就**不可避免**。所以哈希表设计的重点不是"完全避免冲突"，而是"让冲突尽量少，冲突发生后能高效处理"。

主流解决方案有 5 种：**链地址法、开放寻址法、再哈希法、公共溢出区、扩容 + 优化哈希函数**。Java HashMap 用链地址法，JDK 8 后链表过长时转红黑树优化极端冲突。

## 标准回答

一句话结论：**哈希冲突不可彻底避免，主流解决方案有链地址法（拉链法）、开放寻址法（线性/二次/双重散列探测）、再哈希法、公共溢出区；Java HashMap 采用链地址法，JDK 8 后链表长度 ≥ 8 且容量 ≥ 64 时转红黑树，把极端冲突下的查询从 `O(n)` 优化到 `O(log n)`。**

5 种方案速查：

| 方案 | 原理 | 代表实现 | 优点 | 缺点 |
|------|------|---------|------|------|
| 链地址法 | 桶后挂链表/树 | HashMap、LinkedHashMap | 实现简单，装载因子容忍度高 | 链表过长退化 `O(n)` |
| 开放寻址法 | 冲突后探测下一个空位 | ThreadLocalMap、IdentityHashMap | 缓存友好，无指针开销 | 删除复杂，易聚集 |
| 再哈希法 | 用另一个哈希函数重算 | 双重哈希 | 减少聚集 | 需多个好哈希函数 |
| 公共溢出区 | 冲突元素放溢出表 | 老式数据库索引 | 思路简单 | 溢出表过大时性能差 |
| 扩容 + 优化哈希 | 增加桶数 + 改进 hash | 所有 HashMap | 治本 | 扩容成本高 |

## 实现原理

### 1. 链地址法（拉链法）

每个桶后挂一个链表（或树）。冲突元素都放在同一个桶的链上：

```text
bucket[0] -> null
bucket[1] -> A -> B -> C
bucket[2] -> D
bucket[3] -> null
```

- **优点**：实现简单，对装载因子容忍度高（可以超过 1），删除方便。
- **缺点**：链表过长时查询退化 `O(n)`；链表节点内存不连续，缓存不友好。
- **代表**：Java HashMap、LinkedHashMap、ConcurrentHashMap、C++ STL unordered_map。

HashMap 在链表长度 ≥ 8 且容量 ≥ 64 时把链表转红黑树，退化阈值 6（`UNTREEIFY_THRESHOLD`）。

### 2. 开放寻址法

所有元素都存在数组内部，冲突后按规则找下一个空位。

#### 线性探测

```text
hash(k) = 5, 但 bucket[5] 已占 → 尝试 6 → 7 → 8 ...
```

简单，但容易产生"一次聚集"——连续占用块越来越大。

#### 二次探测

```text
尝试 hash(k) + 1, hash(k) + 4, hash(k) + 9, hash(k) + 16, ...
```

步长是平方，减少连续聚集，但仍可能有"二次聚集"。

#### 双重散列

```text
step = hash2(k)
尝试 hash(k), hash(k) + step, hash(k) + 2*step, ...
```

用第二个哈希函数算步长，分布最均匀。

- **优点**：所有数据紧凑存在数组里，缓存友好；无指针开销。
- **缺点**：删除复杂（不能直接置空，否则破坏探测链，要用"墓碑"标记）；装载因子接近 1 时性能急剧下降。
- **代表**：ThreadLocal.ThreadLocalMap、IdentityHashMap、Python dict。

### 3. 再哈希法

准备多个哈希函数 `h1, h2, h3, ...`，冲突时换下一个：

```text
idx = h1(k)
if occupied: idx = h2(k)
if occupied: idx = h3(k)
...
```

- **优点**：减少聚集。
- **缺点**：计算成本高；需要设计多个质量好的哈希函数。
- **代表**：Bloom Filter（布隆过滤器）、Cuckoo Hashing（布谷鸟哈希）。

### 4. 公共溢出区

主表 + 溢出表。冲突元素统一放溢出表，查找时先查主表，没找到再查溢出表。

- **优点**：思路简单。
- **缺点**：溢出表过大时查询退化 `O(n)`。
- **代表**：早期数据库索引、教学示例。现代通用 HashMap 不用。

### 5. 扩容 + 优化哈希函数

冲突过多时，要么扩容增加桶数，要么改进哈希函数让分布更均匀。

HashMap 的两层防御：

- **扩容**：`size > capacity * 0.75` 时翻倍扩容，把冲突元素拆开。
- **扰动 hash**：`hash = hashCode ^ (hashCode >>> 16)`，让高位参与低位计算，减少低位规律性导致的集中冲突。

## 代码示例

### 错误的 hashCode 导致性能退化

```java
import java.util.HashMap;
import java.util.Map;

public class BadHashCodeDemo {
    static class BadKey {
        private final Long id;
        BadKey(Long id) { this.id = id; }

        @Override
        public int hashCode() { return 1; }   // 所有 key 落到同一个桶

        @Override
        public boolean equals(Object o) {
            if (!(o instanceof BadKey)) return false;
            return java.util.Objects.equals(id, ((BadKey) o).id);
        }
    }

    public static void main(String[] args) {
        Map<BadKey, String> map = new HashMap<>();
        long s = System.currentTimeMillis();
        for (long i = 0; i < 10_000; i++) {
            map.put(new BadKey(i), "v" + i);   // 全部冲突，链表长度 10000
        }
        long e = System.currentTimeMillis();
        System.out.println("插入 10000 个元素耗时：" + (e - s) + " ms");
        // 由于全冲突，put 退化 O(n)，整体 O(n^2)
    }
}
```

### 正确的 hashCode 实现

```java
public final class GoodKey {
    private final Long id;
    private final String tenant;

    public GoodKey(Long id, String tenant) {
        this.id = id;
        this.tenant = tenant;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof GoodKey)) return false;
        GoodKey g = (GoodKey) o;
        return java.util.Objects.equals(id, g.id)
            && java.util.Objects.equals(tenant, g.tenant);
    }

    @Override
    public int hashCode() {
        return java.util.Objects.hash(id, tenant);   // 均匀分布
    }
}
```

### Java record 自动生成

```java
public record OrderKey(Long orderId, String tenant) {}

Map<OrderKey, String> map = new HashMap<>();
// record 自动生成 hashCode + equals，分布均匀
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 自定义 key | 正确重写 `equals` + `hashCode` | equals 相等则 hashCode 必相等 |
| 批量导入数据 | `new HashMap<>(expected / 0.75f + 1)` | 预估容量减少扩容 |
| 高冲突风险 key | 改进 hashCode 实现 | 不要返回常量 |
| 安全防护 | 限制外部 key 大小、限流 | 警惕恶意构造 hash 冲突攻击 |
| 高并发场景 | 用 `ConcurrentHashMap` | HashMap 并发不安全 |
| 需要有序 | 用 `TreeMap` | `O(log n)`，hash 冲突无关 |

## 深挖追问

### HashMap 用的是哪种冲突解决方案？

链地址法（拉链法）。每个桶是一个单向链表，冲突元素追加到链表尾部。JDK 8 后链表长度 ≥ 8 且容量 ≥ 64 时转红黑树，退化阈值 6。

关键常量：

```java
TREEIFY_THRESHOLD = 8         // 链表树化阈值
UNTREEIFY_THRESHOLD = 6       // 树退化回链表阈值
MIN_TREEIFY_CAPACITY = 64     // 树化要求的最小容量
```

### 为什么 HashMap 不用开放寻址法？

- HashMap 的装载因子默认 0.75，开放寻址法在这个装载因子下性能下降明显（需要频繁探测）。
- 开放寻址法删除元素复杂（要标记"墓碑"），HashMap 的 remove 是高频操作。
- HashMap 需要支持任意对象 key，链地址法对 key 类型无要求。
- JDK 8 引入红黑树后，链地址法的"链表过长退化"问题已被 `O(log n)` 解决。

开放寻址法更适合"key 类型固定、删除少、装载因子低"的场景，比如 ThreadLocalMap（每个线程的 ThreadLocal 数量有限，几乎不删除）。

### 链表长度到 8 一定树化吗？

不一定。链表长度 ≥ 8 触发 `treeifyBin`，但 `treeifyBin` 内部先检查容量：

```java
if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY) {
    resize();   // 容量 < 64，优先扩容
} else {
    // 容量 ≥ 64，才真正树化
}
```

容量 < 64 时扩容比树化更划算——冲突主要来自数组太小，扩容能把链表拆开。

### 冲突过多会有什么影响？

- **链表**：查询、插入、删除都退化 `O(n)`。
- **红黑树**：仍 `O(log n)`，但有维护成本（旋转、变色）。
- **整体**：HashMap 性能从 `O(1)` 退化到 `O(log n)` 或 `O(n)`。

如果大量 key 的 hashCode 都返回固定值，HashMap 会变成"挂在一个桶上的红黑树"，性能接近 TreeMap。

### 如何降低哈希冲突概率？

1. **设计均匀分布的 hashCode**：用所有关键字段参与计算，避免返回常量。
2. **遵守 equals/hashCode 约定**：equals 相等则 hashCode 必相等。
3. **预估容量**：避免小容量下反复冲突。
4. **合理负载因子**：0.75 是 HashMap 默认值，不要乱调。
5. **对热点 key 防护**：限制外部输入 key 数量，防止恶意构造冲突。

### HashMap 链表是单向还是双向？

单向。`Node.next` 指向下一个节点。所以 HashMap 的链表不能反向遍历，删除节点需要先找到前驱。`LinkedHashMap` 才用双向链表维护插入顺序。

## 易错点

- 哈希冲突无法彻底避免，只能降低概率并优化处理方式。
- "链地址法"不只是链表，桶内也可以用红黑树（HashMap 就是这样）。
- 链表长度到 8 不一定树化，容量 < 64 时优先扩容。
- 开放寻址法删除元素不能直接置空，要用墓碑标记。
- `hashCode` 相同不等于 `equals` 相等，HashMap 在桶内还要调 `equals` 判断。
- HashMap 的链表是单向的，`LinkedHashMap` 才是双向。

## 总结

主流方案是链地址法和开放寻址法。Java HashMap 选链地址法的理由：装载因子容忍度高、删除简单、key 类型无限制；JDK 8 引入红黑树把极端冲突从 `O(n)` 优化到 `O(log n)`。面试时建议先说"冲突不可避免"，再对比 5 种方案，最后结合 HashMap 的链表 + 红黑树 + 扩容 + hash 扰动讲工程实践。
