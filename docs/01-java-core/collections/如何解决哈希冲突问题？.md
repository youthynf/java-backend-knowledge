# 如何解决哈希冲突问题？

如何解决哈希冲突问题？
解决哈希冲突方法：
链地址法：通过链表或者其他数据结构来存储发生冲突的键值对，并将他们链在同一个哈希桶上；
开放寻址法：在哈希表中找到另一个可用的位置，而不是存储在链表中，常见的有线性探测、二次探测、双重散列。
重哈希：发生哈希冲突时，通过调用另一个哈希算法重新计算新的哈希值，直到找到一个可用的哈希桶为止；
哈希桶扩容：当哈希冲突过多时，采用动态扩容的方式，增加哈希桶的数量，重新分配键值对，降低哈希冲突概率；

开放寻址法：
线性试探：当发生哈希冲突时，从冲突的位置开始，依次检查下一个位置，直到找到一个可用位置；如果已经到达哈希表末尾，则从表头继续试探，如果仍找不到，则进行扩容。
二次试探：解决线性试探的哈希 key 聚集问题，试探的步长通过二次函数计算；
双重散列：进一步解决哈希 key 聚集问题，试探步长采用另一个哈希函数计算，随机性更好；

## 面试复习版

### 核心概念
- 哈希冲突是指不同 key 经过哈希函数和数组下标计算后落到同一个桶位。冲突不可完全避免，只能通过良好的哈希函数、合理装载因子和冲突处理策略降低影响。
- 常见处理方式包括：链地址法、开放寻址法、再哈希/双重散列、扩容再散列。Java `HashMap` 主要采用链地址法，并在 JDK 8 后当单桶链表过长且数组容量足够时树化为红黑树。
- 冲突过多会让查询从理想 O(1) 退化。链表桶中最坏 O(n)，红黑树桶中查找约 O(log n)，开放寻址则受装载因子和探测序列影响很大。

### 面试官想考什么
- 是否能把“冲突是什么”和“解决冲突的几类方案”讲清楚。
- 是否知道 Java `HashMap` 的具体做法：数组 + 链表/红黑树、负载因子 0.75、扩容、扰动函数。
- 是否能分析不同方案的优缺点，而不是只罗列名词。

### 标准回答
解决哈希冲突通常有四类思路：第一是链地址法，把冲突元素放到同一个桶的链表、树或其他结构中，Java `HashMap` 就是这个思路；第二是开放寻址法，冲突后按一定探测规则寻找下一个空槽，例如线性探测、二次探测、双重散列；第三是再哈希，用另一个哈希函数重新计算位置；第四是扩容，通过增加桶数量并重新分布元素来降低冲突概率。实际工程中还要保证 key 的 `hashCode` 分布均匀，并正确实现 `equals`。

### 深挖追问
- 为什么 `HashMap` 负载因子默认 0.75？它在空间利用率和冲突概率之间折中，太高冲突多，太低浪费内存。
- 树化条件是什么？JDK 8 中桶内元素达到阈值且数组容量至少 64 时才树化，否则优先扩容。
- 开放寻址有什么问题？删除处理复杂，装载因子高时探测次数增加，线性探测容易产生聚集。
- 哈希函数很差会怎样？大量 key 落到少数桶，导致性能退化，甚至带来哈希碰撞攻击风险。

### 实战场景/代码示例
```java
class UserKey {
    private final Long id;
    private final String type;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof UserKey)) return false;
        UserKey other = (UserKey) o;
        return Objects.equals(id, other.id) && Objects.equals(type, other.type);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, type); // 分布尽量均匀，且与 equals 使用同一组字段
    }
}
```

### 易错点/总结
- 冲突不是异常情况，而是哈希表必须处理的正常情况。
- 重写 `equals` 必须同步重写 `hashCode`，否则 Map/Set 中可能出现查不到、去重失败等问题。
- 可变字段不适合参与 `hashCode`，对象放入 HashMap 后字段变化会导致定位不到原桶。
- 不要把所有集合问题都套成 ArrayList/通用集合模板；这道题重点是哈希表冲突策略和 HashMap 的具体实现。
