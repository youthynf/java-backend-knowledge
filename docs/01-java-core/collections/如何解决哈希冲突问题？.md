# 如何解决哈希冲突问题？

## 核心概念

哈希冲突指的是：不同的 key 经过哈希函数计算后，落到了同一个哈希桶位置。只要哈希空间有限、输入数据可能无限，冲突就不可避免，所以哈希表设计的重点不是“完全避免冲突”，而是 **让冲突尽量少，并在冲突发生后高效处理**。

常见解决方案包括：

- 链地址法
- 开放寻址法
- 再哈希法
- 公共溢出区
- 扩容与优化哈希函数

在 Java 面试里，通常会结合 HashMap 追问。HashMap 主要使用链地址法，JDK 1.8 后在链表过长时会转红黑树。

## 面试官想考什么

这道题不只是让你背“链地址法、开放寻址法”，而是想看你是否理解：

- 哈希冲突为什么不可避免。
- 每种冲突处理方式的原理和优缺点。
- Java HashMap 采用了什么方案。
- 冲突过多会带来什么性能问题。
- 工程中如何降低冲突概率。

## 标准回答

哈希冲突可以通过多种方式解决。

### 1. 链地址法

链地址法是在每个哈希桶后面挂一个链表或其他数据结构。多个 key 落到同一个桶时，就把这些元素放到同一个桶的链上。

```text
bucket[0] -> null
bucket[1] -> A -> B -> C
bucket[2] -> D
```

优点是实现简单，对装载因子的容忍度较高，删除元素也比较方便。缺点是如果大量元素集中到同一个桶，链表会变长，查询性能会从 O(1) 退化到 O(n)。

Java HashMap 就是典型的链地址法实现。JDK 1.8 后，当桶中链表过长且数组容量足够大时，会把链表转成红黑树，把极端冲突下的查询复杂度从 O(n) 优化到 O(log n)。

### 2. 开放寻址法

开放寻址法不额外挂链表，所有元素都存储在哈希表数组内部。发生冲突后，按照某种探测规则继续寻找下一个可用位置。

常见探测方式：

- **线性探测**：依次检查下一个位置。
- **二次探测**：按平方步长探测，减少连续聚集。
- **双重散列**：用另一个哈希函数计算探测步长。

开放寻址法的优点是空间局部性好，不需要额外链表指针；缺点是删除处理更复杂，对装载因子更敏感，容易出现聚集问题。

### 3. 再哈希法

再哈希法是发生冲突后，使用另一个哈希函数重新计算位置，直到找到可用桶位。

它可以减少冲突集中，但会增加计算成本，而且需要设计多个质量较好的哈希函数。

### 4. 公共溢出区

公共溢出区会把冲突元素统一放到一个额外区域中。查找时先查主表，主表没有再查溢出区。

这种方式思路简单，但如果溢出区过大，查询性能会明显下降，现代通用 HashMap 中并不常作为首选。

### 5. 扩容和优化哈希函数

当冲突过多时，说明桶数量可能不够，或者 hash 分布不均匀。可以通过扩容增加桶数量，并重新分布元素，降低冲突概率。

HashMap 默认负载因子是 0.75，当元素数量超过阈值时会扩容。

## 深挖追问

### 1. HashMap 用的是哪种冲突解决方案？

HashMap 使用链地址法。JDK 1.8 后，每个桶中先用链表存储冲突节点，当链表长度达到阈值且数组容量足够大时，会转成红黑树。

关键阈值：

```text
TREEIFY_THRESHOLD = 8
MIN_TREEIFY_CAPACITY = 64
```

也就是说，链表长度达到 8 不一定马上树化；如果数组容量小于 64，HashMap 会优先扩容。

### 2. 为什么 HashMap 不直接使用开放寻址法？

开放寻址法要求所有元素都放在数组内部，装载因子高时性能下降明显，删除元素也要用特殊标记处理，否则会破坏探测路径。

HashMap 需要支持通用对象 key、动态扩容、删除操作和较高装载因子，链地址法实现更灵活。JDK 1.8 引入红黑树后，也缓解了极端冲突下链表过长的问题。

### 3. 冲突过多会有什么影响？

冲突过多会让多个 key 集中到同一个桶，查询、插入、删除都要在桶内继续比较。链表情况下可能退化成 O(n)，红黑树情况下也会有额外维护成本。

如果大量 key 的 hashCode 质量很差，例如都返回固定值，即使 HashMap 有红黑树优化，性能也会明显变差。

### 4. 如何降低哈希冲突概率？

常见手段：

- 设计分布均匀的 `hashCode()`。
- 正确重写 `equals()` 和 `hashCode()`。
- 合理设置初始容量，减少频繁扩容。
- 控制负载因子，不让哈希表过满。
- 对热点 key 或恶意输入做额外保护。

## 实战场景

### 场景 1：错误的 hashCode 导致性能退化

```java
class BadKey {
    private final Long id;

    BadKey(Long id) {
        this.id = id;
    }

    @Override
    public int hashCode() {
        return 1; // 所有对象都落到同一个桶
    }

    @Override
    public boolean equals(Object obj) {
        if (!(obj instanceof BadKey)) return false;
        return java.util.Objects.equals(id, ((BadKey) obj).id);
    }
}
```

这种写法会让 HashMap 的大部分元素集中到一个桶中，查询性能严重下降。

### 场景 2：批量导入前预估容量

```java
int expectedSize = 100_000;
int capacity = (int) (expectedSize / 0.75f) + 1;
Map<Long, String> map = new HashMap<>(capacity);
```

提前设置容量可以减少扩容次数，也能降低一段时间内的冲突概率。

### 场景 3：面向安全的冲突防护

如果接口接收大量外部输入作为 key，需要警惕恶意构造 hash 冲突导致 CPU 飙高。可以限制请求大小、增加限流，或者选择更适合该场景的数据结构。

## 易错点

- 哈希冲突无法彻底避免，只能降低概率并优化处理方式。
- 链地址法不是只有链表，也可以在桶内使用树等结构。
- 开放寻址法删除元素更复杂，不能简单把位置置空。
- HashMap 链表长度达到 8 不一定树化，容量小于 64 时优先扩容。
- hashCode 相同不代表 equals 一定相等。

## 总结

解决哈希冲突的主流方式有链地址法和开放寻址法。Java HashMap 采用链地址法，并在 JDK 1.8 后用红黑树优化极端冲突。面试回答时建议先解释冲突不可避免，再对比几种方案，最后结合 HashMap 的树化、扩容和 hashCode 设计讲工程实践。
