# CopyOnWriteArrayList 底层实现原理是怎么样的？

## 核心概念

`CopyOnWriteArrayList` 是 Java 并发包里的线程安全 List。它的核心思想是：

> 读操作不加锁，写操作加锁并复制一份新数组，在新数组上修改完成后，再把数组引用切换过去。

这就是“写时复制”（Copy-On-Write）。它牺牲写入成本，换取读操作的无锁和稳定快照。

## 底层结构

它内部维护一个数组引用：

```java
private transient volatile Object[] array;
```

关键点有两个：

1. **数组引用用 volatile 修饰**：写线程替换新数组后，其他线程能及时看到最新数组引用。
2. **写操作使用锁保护**：同一时刻只允许一个写线程复制和替换数组，避免写写冲突。

## 写入流程

以 `add(E e)` 为例，整体流程是：

1. 获取锁，保证同时只有一个写线程执行。
2. 读取当前数组 `oldArray`。
3. 复制一个长度为 `oldLength + 1` 的新数组。
4. 把新增元素放到新数组末尾。
5. 将内部数组引用从旧数组切换到新数组。
6. 释放锁。

示意代码：

```java
public boolean add(E e) {
    synchronized (lock) {
        Object[] es = getArray();
        int len = es.length;
        es = Arrays.copyOf(es, len + 1);
        es[len] = e;
        setArray(es);
        return true;
    }
}
```

不同 JDK 版本内部锁实现细节可能不同，但核心机制都是“写加锁 + 复制数组 + volatile 发布”。

## 读取流程

读取时直接访问当前数组，不加锁：

```java
public E get(int index) {
    return elementAt(getArray(), index);
}
```

因此它的读性能很好，也不会被写线程阻塞。但读线程看到的是某一刻的数组快照，不保证马上看到正在进行的写入。

## 迭代器特点

`CopyOnWriteArrayList` 的迭代器是**快照迭代器**：创建迭代器时会持有当时的数组快照。

这意味着：

- 迭代过程中其他线程修改列表，不会抛出 `ConcurrentModificationException`。
- 迭代器不一定能看到创建之后的新数据。
- 迭代器不支持 `remove`、`set`、`add` 这类修改操作。

## 面试官想考什么

1. 是否理解“写时复制”而不是只背线程安全。
2. 能不能说清楚为什么读不加锁仍然线程安全。
3. 是否知道它适合读多写少，不适合频繁写入。
4. 是否知道迭代器是快照语义。

## 标准回答

可以这样答：

> CopyOnWriteArrayList 底层是一个 volatile 数组。读操作直接读数组，不加锁；写操作会先加锁，然后复制一份新数组，在新数组上完成新增、删除或修改，最后把数组引用切换到新数组。volatile 保证新数组引用对其他线程可见。它适合读多写少、集合规模不大的场景，比如配置列表、监听器列表。缺点是写入时会复制数组，内存和 CPU 成本较高，而且读到的是快照，实时一致性不强。

## 深挖追问

### 为什么读操作不用加锁？

因为写操作不会在原数组上原地修改，而是复制新数组后一次性替换引用。读线程要么读到旧数组，要么读到新数组，不会读到一个被改到一半的数组。

### volatile 在这里解决什么问题？

`volatile` 修饰的是数组引用，保证写线程调用 `setArray(newArray)` 后，其他线程能看到新的数组引用。它不是让数组内部元素都具备原子复合操作能力，而是用于发布新快照。

### 为什么不适合写多场景？

每次写入都要复制整个数组，时间复杂度是 `O(n)`，还会产生额外内存分配和 GC 压力。数据量越大、写越频繁，性能越差。

## 实战场景

适合：

- 黑白名单、配置项、路由规则这类读多写少数据。
- 监听器列表：注册/取消不频繁，但事件通知会频繁遍历。
- 需要遍历时不被并发修改打断的场景。

不适合：

- 高频写入的订单列表、消息队列、实时统计容器。
- 大集合频繁增删改。
- 要求读操作强实时看到最新写入的场景。

## 易错点

- 它线程安全，但不代表适合所有并发场景。
- 读到的是快照，不是强一致实时视图。
- 写操作成本很高，不能把它当普通 ArrayList 的并发替代品滥用。
- `volatile Object[] array` 保证的是引用可见性，不是让所有业务复合操作自动原子化。
