# GC 算法

> 垃圾收集算法是 GC 收集器的理论基础。理解算法思想，才能理解各收集器的设计取舍。

## 判断对象是否存活

### 1. 引用计数法（已废弃）

- 每个对象维护一个引用计数器，被引用时 +1，引用断开时 -1
- **致命缺陷**：无法解决循环引用

```java
// 循环引用导致内存泄漏
Object a = new Object();
Object b = new Object();
a.ref = b;
b.ref = a;
a = null; b = null; // 引用计数不为0，无法回收
```

### 2. 可达性分析算法（JVM 采用）

- 从 **GC Roots** 开始，沿引用链向下搜索
- 不可达的对象即为可回收对象

#### GC Roots 包括：

| 类型 | 示例 |
|------|------|
| 虚拟机栈中引用的对象 | 方法中的局部变量 |
| 方法区中类静态属性引用的对象 | static 变量 |
| 方法区中常量引用的对象 | 常量池引用 |
| 本地方法栈中 JNI 引用的对象 | Native 方法引用 |
| JVM 内部引用 | 基本数据类型对应的 Class 对象、常驻异常对象、类加载器 |
| 同步锁（synchronized）持有的对象 | 锁对象 |

```
GC Roots
   │
   ├──> Object A ──> Object C
   │
   ├──> Object B ──> Object D ──> Object E
   │
   └──> Object F (不可达，可回收)
   
         Object X ←──→ Object Y  (循环引用但不可达，可回收)
```

### 引用类型

| 引用类型 | 回收时机 | 用途 |
|---------|---------|------|
| 强引用（Strong） | 永不回收（只要引用存在） | 普通变量赋值 |
| 软引用（SoftReference） | 内存不足时回收 | 缓存 |
| 弱引用（WeakReference） | 下次 GC 时回收 | ThreadLocal、WeakHashMap |
| 虚引用（PhantomReference） | 随时可回收，仅用于跟踪回收 | 跟踪 GC、管理堆外内存 |

```java
// 引用类型示例
Object obj = new Object();                    // 强引用
SoftReference<Object> soft = new SoftReference<>(obj);  // 软引用
WeakReference<Object> weak = new WeakReference<>(obj);  // 弱引用
PhantomReference<Object> phantom = new PhantomReference<>(obj, queue); // 虚引用
```

## 四种基础 GC 算法

### 1. 标记-清除算法（Mark-Sweep）

```
标记前:  [A][B][C][D][E][F][G][H]
标记后:  [A][✗B][C][✗D][E][✗F][G][✗H]  (B/D/F/H 可回收)
清除后:  [A][  ][C][  ][E][  ][G][  ]
```

**过程**：
1. 标记阶段：从 GC Roots 遍历，标记所有可达对象
2. 清除阶段：统一回收未标记对象

**优点**：实现简单

**缺点**：
- **内存碎片**：清除后产生大量不连续空间
- **分配效率低**：大对象分配时需要找连续空间，触发 Full GC
- **STW**：标记和清除都要暂停用户线程

### 2. 复制算法（Copying）

```
复制前（From区）: [A][B][C][D][E]  To区: [空]
                   ↓ 存活对象复制
复制后（From区）: [空]              To区: [A][B][C][D][E]
```

**过程**：
1. 将内存分为两块，每次只使用其中一块
2. GC 时将存活对象复制到另一块
3. 清空当前使用的这块

**优点**：
- 无内存碎片
- 分配快（指针碰撞法）

**缺点**：
- **空间浪费**：可用内存减半
- **复制开销**：存活对象多时效率低

> **新生代优化**：Eden + 2个 Survivor（8:1:1），每次只浪费 10% 空间

### 3. 标记-整理算法（Mark-Compact）

```
标记前:  [A][✗B][C][✗D][E][✗F][G][✗H]
整理后:  [A][C][E][G][        空闲        ]
```

**过程**：
1. 标记阶段：同标记-清除
2. 整理阶段：将存活对象向一端移动，清理边界外内存

**优点**：
- 无内存碎片
- 不浪费空间

**缺点**：
- 移动对象需要更新引用，开销大
- STW 时间较长

### 4. 分代收集算法

> **核心思想**：根据对象存活时间将堆分为不同代，每代采用最适合的算法。

```
┌─────────────────────────────────────────────────────┐
│                     堆内存                          │
├─────────────────────────┬───────────────────────────┤
│       新生代             │         老年代            │
│  (对象存活率低)          │   (对象存活率高)          │
│  → 复制算法              │   → 标记-整理/标记-清除   │
│  → Minor GC 频繁但快    │   → Major GC 少但慢       │
└─────────────────────────┴───────────────────────────┘
```

### 对象晋升老年代的条件

1. **年龄达到阈值**（默认 15，-XX:MaxTenuringThreshold）
2. **大对象直接进入老年代**（-XX:PretenureSizeThreshold）
3. **Survivor 空间不足**，动态年龄判断

### GC 类型对比

| GC 类型 | 回收区域 | 频率 | 速度 | 触发条件 |
|---------|---------|------|------|---------|
| Minor GC / Young GC | 新生代 | 频繁 | 快 | Eden 区满 |
| Major GC / Old GC | 老年代 | 较少 | 慢 | 老年代空间不足 |
| Full GC | 整个堆+方法区 | 极少 | 最慢 | 多种条件（见下表） |

### Full GC 触发条件

1. 调用 `System.gc()`（建议但非强制）
2. 老年代空间不足
3. 方法区/元空间空间不足
4. Minor GC 后存活对象大于老年代剩余空间
5. CMS GC 时出现 Concurrent Mode Failure

## 算法对比总结

| 算法 | 适合场景 | 内存碎片 | 空间利用率 | STW 时间 |
|------|---------|---------|-----------|---------|
| 标记-清除 | 存活对象多 | 有碎片 | 高 | 中等 |
| 复制 | 存活对象少 | 无碎片 | 低（减半） | 短 |
| 标记-整理 | 存活对象多 | 无碎片 | 高 | 长 |
| 分代收集 | 通用 | - | 高 | 优化后最短 |

## 面试高频问题

### Q1: 为什么新生代用复制算法，老年代用标记-整理？

> 新生代对象存活率低（约 98% 会被回收），复制少量存活对象效率高，且使用 Eden+2 Survivor 只浪费 10%。老年代对象存活率高，复制算法需要复制大量对象，效率反而低，因此用标记-整理。

### Q2: 什么叫 "stop the world"？能避免吗？

> STW 指 GC 时暂停所有用户线程。目前所有 GC 算法都无法完全避免 STW，因为可达性分析必须在一个一致性的快照中进行。但可以通过并发收集来缩短 STW 时间（如 CMS、G1、ZGC）。

### Q3: 对象在什么情况下会进入老年代？

> 1. 年龄达到 MaxTenuringThreshold（默认15）；2. 大对象直接分配（超过 PretenureSizeThreshold）；3. Survivor 区相同年龄对象大小超过 Survivor 空间一半，年龄大于等于该年龄的直接晋升；4. Minor GC 后 Survivor 放不下，通过担保机制进入老年代。

### Q4: 说说你对分代收集理论的理解

> 分代收集基于两个假说：1）弱分代假说——绝大多数对象朝生夕灭；2）强分代假说——熬过多次GC的对象越难回收。因此将堆分为新生代和老年代，新生代用复制算法（效率高），老年代用标记-整理（避免碎片）。跨代引用通过记忆集（Remembered Set）解决。
