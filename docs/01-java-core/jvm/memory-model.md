# JVM 内存模型

> JVM 在执行 Java 程序时，会将其管理的内存划分为若干个运行时数据区域。理解内存模型是排查 OOM、优化 GC 的基础。

## 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        JVM 运行时数据区                         │

│                                                                 │

│  │       线程私有                 │  │      线程共享          │ │
│  │  ┌───────────┐               │  │  ┌──────────────────┐  │ │
│  │  │  程序计数器 │               │  │  │       堆          │  │ │
│  │  │ (PC Register)│             │  │  │  ┌────┬────┐     │  │ │

│  │  ┌───────────┐               │  │  │  │ 代 │      │    │  │ │
│  │  │  虚拟机栈  │               │  │  │  └────┴────┘     │  │ │
```

## 1. 程序计数器（PC Register）

- **作用**：记录当前线程正在执行的字节码指令地址
- **特点**：
  - 唯一一个不会发生 OOM 的区域
  - 线程私有，生命周期与线程一致
  - 执行 Native 方法时，计数器值为空（Undefined）
- **为什么需要**：CPU 在多线程间切换后，需要知道下一条该执行哪条指令

## 2. 虚拟机栈（VM Stack）

- **作用**：描述 Java 方法执行的内存模型，每个方法执行时创建一个**栈帧**
- **线程私有**，生命周期与线程一致

### 栈帧结构

```
┌─────────────────────────────────────┐
│              栈帧                    │
├─────────────────────────────────────┤
│  ┌─────────────────────────────┐   │
│  │     局部变量表               │   │
│  │  [0] this                   │   │
│  │  [1] 参数1                  │   │
│  │  [2] 参数2                  │   │
│  │  [3] 局部变量1              │   │
│  └─────────────────────────────┘   │
│  ┌─────────────────────────────┐   │
│  │     操作数栈                 │   │
│  │  (计算过程的中间结果)        │   │
│  └─────────────────────────────┘   │
│  ┌─────────────────────────────┐   │
│  │     动态链接                 │   │
│  │  (指向运行时常量池的方法引用)│   │
│  └─────────────────────────────┘   │
│  ┌─────────────────────────────┐   │
│  │     方法返回地址             │   │
│  │  (方法退出后返回的位置)      │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

### 局部变量表

- 以**变量槽（Slot）** 为单位，long/double 占 2 个 Slot，其余占 1 个
- 实例方法第 0 个 Slot 是 `this` 引用
- **不会自动初始化**（类变量有默认值，局部变量没有）

### 异常

| 异常 | 触发条件 |
|------|---------|
| StackOverflowError | 栈深度超过限制（递归调用过深） |
| OutOfMemoryError | 无法分配新栈（-Xss 设置栈大小） |

```java
// StackOverflowError 示例
public class StackOverflowDemo {
    private static int depth = 0;
    public static void recursion() {
        depth++;
        recursion(); // 无限递归
    }
    public static void main(String[] args) {
        try {
            recursion();
        } catch (StackOverflowError e) {
            System.out.println("栈深度: " + depth); // 默认约 5000-10000
        }
    }
}
```

**面试题**：-Xss 参数影响什么？设置过大/过小有什么影响？

> -Xss 设置每个线程的栈大小。设置过小→容易 StackOverflowError；设置过大→线程数受限（总内存有限）。一般 256K~1M 足够。

## 3. 本地方法栈（Native Method Stack）

- **作用**：为 Native 方法（如 `System.currentTimeMillis()` 的底层实现）服务
- HotSpot 将虚拟机栈和本地方法栈合二为一
- 同样可能抛出 StackOverflowError 和 OutOfMemoryError

## 4. 堆（Heap）

- **作用**：存放几乎所有**对象实例和数组**
- **线程共享**，JVM 中最大的一块内存
- **GC 的主要管理区域**

### 堆内存分代

```
┌─────────────────────────────────────────────────────────────┐
│                         堆内存                               │
├───────────────────────────┬─────────────────────────────────┤
│        新生代 (1/3)        │          老年代 (2/3)           │
│  ┌────────┬────────┬─────┐│                                 │
│  │  Eden  │ S0/From│S1/To││                                 │
│  │  (8/10)│ (1/10) │(1/10)││                                 │
│  └────────┴────────┴─────┘│                                 │
└───────────────────────────┴─────────────────────────────────┘
  比例：新生代:老年代 = 1:2（-XX:NewRatio=2）
  Eden:S0:S1 = 8:1:1（-XX:SurvivorRatio=8）
```

### 堆内存参数

| 参数 | 含义 | 示例 |
|------|------|------|
| -Xms | 堆初始大小 | -Xms256m |
| -Xmx | 堆最大大小 | -Xmx1g |
| -Xmn | 新生代大小 | -Xmn256m |
| -XX:NewRatio | 新生代与老年代比例 | -XX:NewRatio=2 |
| -XX:SurvivorRatio | Eden与Survivor比例 | -XX:SurvivorRatio=8 |

> **生产建议**：-Xms 和 -Xmx 设置相同，避免堆动态扩缩容的开销。

### 堆溢出示例

```java
// OOM: Java heap space
List<byte[]> list = new ArrayList<>();
while (true) {
    list.add(new byte[1024 * 1024]); // 不断分配 1MB
}
```

## 5. 方法区 / 元空间

### 演进历史

| JDK 版本 | 方法区实现 | 存储位置 | 大小限制 |
|----------|-----------|---------|---------|
| JDK 7 及以前 | 永久代（PermGen） | JVM 堆中 | 固定大小（-XX:MaxPermSize） |
| JDK 8 及以后 | 元空间（Metaspace） | 本地内存 | 默认无上限（受限于物理内存） |

### 存储内容

- **类的元信息**：类名、访问修饰符、字段描述、方法描述
- **运行时常量池**：编译期生成的各种字面量和符号引用
- **静态变量**（JDK 7 起移至堆中）
- **JIT 编译后的代码缓存**

### 字符串常量池的迁移

```
JDK 6:  字符串常量池在永久代
JDK 7:  字符串常量池移至堆中（避免 OOM: PermGen space）
JDK 8+: 永久代被元空间取代，字符串常量池仍在堆中
```

```java
// 字符串常量池示例
String s1 = "hello";           // 常量池
String s2 = "hello";           // 复用常量池中的引用
String s3 = new String("hello"); // 堆上新对象
String s4 = s3.intern();       // 尝试放入常量池

System.out.println(s1 == s2);  // true
System.out.println(s1 == s3);  // false
System.out.println(s1 == s4);  // true（JDK 7+）
```

### 元空间参数

| 参数 | 含义 |
|------|------|
| -XX:MetaspaceSize | 元空间初始大小（默认约 21M） |
| -XX:MaxMetaspaceSize | 元空间最大大小（默认无限制） |

```java
// OOM: Metaspace — 大量动态生成类（如 CGLIB 代理）
// -XX:MaxMetaspaceSize=32m
public class MetaspaceOOM {
    public static void main(String[] args) {
        while (true) {
            Enhancer enhancer = new Enhancer();
            enhancer.setSuperclass(MetaspaceOOM.class);
            enhancer.setUseCache(false);
            enhancer.setCallback((MethodInterceptor) (obj, method, args1, proxy) -> 
                proxy.invokeSuper(obj, args1));
            enhancer.create(); // 不断生成新类
        }
    }
}
```

## 6. 直接内存（Direct Memory）

- **不属于 JVM 规范中的运行时数据区**，但会被频繁使用
- NIO 的 DirectByteBuffer 分配堆外内存，避免 Java 堆和 Native 堆之间来回复制数据
- 大小不受 -Xmx 限制（但受物理内存限制），可能导致 OOM

```java
// DirectByteBuffer 示例
ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024 * 100); // 100MB 堆外内存

// 与 HeapByteBuffer 对比
ByteBuffer heapBuffer = ByteBuffer.allocate(1024 * 1024 * 100); // 100MB 堆内内存
```

### 直接内存 vs 堆内存

| 对比项 | 直接内存 | 堆内存 |
|--------|---------|-------|
| 分配方式 | Unsafe.allocateMemory / DirectByteBuffer | new / ByteBuffer.allocate |
| 访问速度 | 少一次数据拷贝，IO 场景更快 | 需要在 JVM 堆和 OS 之间拷贝 |
| 回收方式 | 依赖 Cleaner（PhantomReference）+ GC 触发 | GC 自动回收 |
| 适合场景 | 大文件 IO、网络通信（Netty 大量使用） | 普通对象存储 |

## 面试高频问题

### Q1: JVM 内存模型中有哪些是线程私有的？

> 程序计数器、虚拟机栈、本地方法栈。堆和方法区（元空间）是线程共享的。

### Q2: 栈和堆的区别？

| 维度 | 栈 | 堆 |
|------|---|---|
| 存储内容 | 局部变量、操作数栈、方法调用信息 | 对象实例、数组 |
| 生命周期 | 随方法调用创建/销毁 | 由 GC 管理 |
| 线程隔离 | 线程私有 | 线程共享 |
| 大小 | 固定（-Xss） | 动态（-Xms/-Xmx） |
| 异常 | StackOverflowError | OutOfMemoryError |

### Q3: 方法区和永久代、元空间的关系？

> 方法区是 JVM 规范定义的逻辑概念。永久代（JDK 7-）和元空间（JDK 8+）是 HotSpot 虚拟机对方法区的具体实现。元空间使用本地内存，解决了永久代容易 OOM 的问题。

### Q4: 什么情况下栈会溢出？堆会溢出？

- **栈溢出**：递归调用过深、方法嵌套层数过多
- **堆溢出**：大量创建对象且不被回收、内存泄漏

### Q5: 直接内存有什么用？为什么会 OOM？

> 直接内存用于 NIO 的零拷贝场景，减少数据在 JVM 堆和 OS 内核缓冲区之间的来回复制。直接内存不受 -Xmx 限制，但如果物理内存不足，仍会 OOM。需要通过 -XX:MaxDirectMemorySize 限制。

## 面试总结

围绕「JVM 内存模型」，面试官通常不只考概念定义，更关注你能否把机制、使用场景和线上问题串起来。

### 核心回答

1. JVM 相关问题要从运行时数据区、类加载、对象布局、执行引擎和 GC 协同理解。
2. 面试重点不是背名词，而是能解释对象如何创建、访问、回收，以及参数/工具如何验证判断。
3. 线上问题通常表现为内存异常、频繁 GC、类加载冲突、线程阻塞或性能抖动。

### 高频追问

- 堆、栈、方法区/元空间分别存什么？
- 类加载的加载、验证、准备、解析、初始化分别做什么？
- 如何用 jcmd、jmap、jstat、MAT 验证你的判断？

### 实战落地

- **排查类问题**：先收集监控、日志和 JVM 现场信息，再用工具验证假设，避免凭经验改参数。
- **调优类问题**：先明确目标是降低停顿、提升吞吐还是减少内存，再选择收集器、堆大小和业务代码优化。
- **面试表达**：用“现象 → 原理 → 工具验证 → 解决方案 → 风险边界”的链路回答。

### 易错点

- 不要把 JVM 内存结构和 Java 内存模型混为一谈。
- 不要脱离 JDK 版本讨论永久代/元空间、字符串常量池等细节。
