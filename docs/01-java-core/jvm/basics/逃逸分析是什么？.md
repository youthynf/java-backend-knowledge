# 逃逸分析是什么

## 核心概念

逃逸分析（Escape Analysis）是 JIT 编译器在编译期分析对象动态作用域的全局数据流分析算法，用于判断一个对象是否会"逃逸"出当前方法或线程。基于逃逸分析结果，JIT 可以做三种优化：标量替换、栈上分配（间接实现）、锁消除。

逃逸分析解决的问题是：减少堆分配压力、降低 GC 频率、消除不必要的锁。它本身不改变程序语义，只影响性能。

逃逸分析在 JDK 6u23+ 默认开启（`-XX:+DoEscapeAnalysis`）。HotSpot 中"栈上分配"实际上是通过标量替换间接实现的，不是真的把对象分配在栈帧上。

```text
new 一个对象 → JIT 编译期分析 →
   ├─ 全局逃逸 Global Escape：逃出方法或线程（赋给静态字段、返回、传入其他线程）
   │   → 走堆分配
   ├─ 方法逃逸 Arg Escape：作为参数传给其他方法但未逃出线程
   │   → 一般走堆分配
   └─ 未逃逸 No Escape：只在当前方法内使用
       → 标量替换 / 锁消除
```

## 标准回答

逃逸分析是 JIT 编译期对对象作用域的分析，识别"未逃逸"的对象后做三种优化：标量替换、栈上分配（标量替换的间接结果）、锁消除。一句话结论：JIT 通过逃逸分析让不逃逸的对象不进堆、不进 GC，让单线程对象不加锁。

要点：

1. **逃逸类型**：未逃逸（No Escape）、方法逃逸（Arg Escape）、全局逃逸（Global Escape）。
2. **优化 1 标量替换**：把不逃逸对象拆成基本类型字段直接放栈/寄存器，不创建对象。
3. **优化 2 栈上分配**：HotSpot 通过标量替换间接实现，对象不进堆。
4. **优化 3 锁消除**：JIT 发现锁对象不会逃逸到其他线程，自动消除同步代码块。
5. **默认开启**：JDK 6u23+ 默认 `-XX:+DoEscapeAnalysis`。

## 逃逸类型

### 1. 未逃逸（No Escape）

对象只在当前方法内使用，不传给其他方法、不返回、不赋给静态字段。这是最适合优化的场景。

```java
public int sum() {
    User u = new User();   // 未逃逸
    u.setId(1);
    return u.getId();
}
```

### 2. 方法逃逸（Arg Escape）

对象作为参数传给其他方法，但未逃出当前线程。

```java
public void process() {
    User u = new User();
    helper(u);   // 方法逃逸
}

private void helper(User u) {
    // 使用 u
}
```

### 3. 全局逃逸（Global Escape）

对象逃出当前方法或线程：作为返回值、赋给静态字段、被其他线程访问、注册为监听器。

```java
public User buildUser() {
    User u = new User();
    u.setId(1);
    return u;   // 全局逃逸
}

public static User CACHE;
public void cache() {
    User u = new User();
    CACHE = u;  // 全局逃逸
}
```

## 优化 1：标量替换

标量（Scalar）是不可再分解的值（int、long、reference）。聚合量（Aggregate）可分解（对象、数组）。JIT 通过逃逸分析确定对象不逃逸且可拆解时，不创建对象，直接把字段拆为标量。

```java
// 原始代码
class Point { int x; int y; }
public int demo() {
    Point p = new Point(1, 2);
    return p.x + p.y;
}

// JIT 标量替换后
public int demo() {
    int x = 1;       // Point 对象从未被创建
    int y = 2;
    return x + y;
}
```

控制参数：`-XX:+EliminateAllocations`（默认开启），同时控制标量替换和栈上分配。

## 优化 2：栈上分配（间接实现）

HotSpot 没有真正实现"对象直接分配在栈帧上"的代码路径，而是通过标量替换间接达到效果：JIT 分析对象不逃逸且可拆解时，直接把字段拆成标量，连对象都不创建。从结果看，等同于"栈上分配"。

```java
public void alloc() {
    User u = new User();   // 未逃逸
    u.id = 1;
    u.name = "a";
}
// JIT 优化后，等价于：
public void alloc() {
    int id = 1;            // User 对象从未创建
    String name = "a";
}
```

## 优化 3：锁消除

如果锁对象不会逃逸到其他线程，JIT 在编译时自动消除 synchronized 块，避免无谓的加锁开销。基于逃逸分析判断"该对象不会被其他线程访问"。

```java
// 原始代码
public void keep() {
    Object keeper = new Object();   // keeper 不逃逸
    synchronized (keeper) {         // 无意义的锁
        System.out.println(keeper);
    }
}

// JIT 锁消除后
public void keep() {
    Object keeper = new Object();
    System.out.println(keeper);
}
```

控制参数：`-XX:+EliminateLocks`（默认开启）。

## 代码示例

通过 GC 行为侧面验证逃逸分析：

```java
public class EscapeTest {
    static class User { int id; String name; }

    // 未逃逸：应被标量替换
    public static void noEscape() {
        User u = new User();
        u.id = 1;
        u.name = "a";
    }

    // 全局逃逸：返回对象
    public static User escape() {
        User u = new User();
        u.id = 1;
        return u;
    }

    public static void main(String[] args) {
        // 开启逃逸分析（默认）
        for (int i = 0; i < 1_000_000; i++) noEscape();
        System.gc();
        // 几乎无 GC，因为 User 对象从未被创建

        // 关闭逃逸分析：-XX:-DoEscapeAnalysis
        for (int i = 0; i < 1_000_000; i++) noEscape();
        System.gc();
        // 触发多次 Minor GC，因为每个 User 都进了堆
    }
}
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 高频临时对象 | 让对象不逃逸，享受标量替换 | 避免在循环里返回临时对象 |
| 锁优化 | 让锁对象不逃逸，触发锁消除 | StringBuffer 的 synchronized 在单线程被消除 |
| 字符串拼接 | JDK 9+ 的 `+` 用 invokedynamic，避免临时 StringBuilder | JIT 优化已经做了大量工作 |
| 调试逃逸分析 | `-XX:+PrintEscapeAnalysis` | 需要 FastDebug/SlowDebug 构建，生产 JVM 不可用 |

## 深挖追问

### 逃逸分析什么时候会失效？

对象逃逸的情况：

- 作为方法返回值。
- 被赋给静态字段。
- 被赋给实例字段（且实例本身可能逃逸）。
- 被传入其他方法且该方法可能保留引用。
- 被异步任务持有（提交到线程池）。
- 被注册为监听器。

只要对象引用可能离开当前方法或当前线程，就属于逃逸，不能标量替换。

### 逃逸分析有什么代价？

逃逸分析本身是耗时操作，JIT 需要做全局数据流分析。如果分析后发现"所有对象都逃逸"，分析过程就白费了。HotSpot 用分层编译降低开销：解释执行时不分析，C1 编译快速分析，C2 编译深度分析。

### 标量替换和栈上分配是同一个东西吗？

不是。栈上分配是把对象放在栈帧上，方法返回时销毁；标量替换是根本不创建对象，把字段拆为标量。HotSpot 实际没有实现"对象栈上分配"，所有"栈上分配"效果都是通过标量替换间接实现的。两者效果类似（都不进堆），机制不同。

### 锁消除对 StringBuffer 有什么影响？

`StringBuffer` 的方法用 synchronized 修饰，但在单线程使用时 JIT 通过逃逸分析发现 `StringBuffer` 对象不逃逸，会自动消除锁。这就是为什么单线程下 `StringBuffer` 性能接近 `StringBuilder`。但多线程场景仍需 `StringBuffer` 或显式锁。

## 易错点

- 把"栈上分配"当作独立的优化机制，实际是标量替换的间接结果。
- 以为关闭逃逸分析后程序会出错，只是性能略差，堆压力增加。
- 以为所有局部变量都能栈上分配，需要 JIT 分析确定不逃逸。
- 以为 `final` 字段一定不逃逸，final 只保证不可变，与逃逸无关。
- 在生产环境用 `-XX:+PrintEscapeAnalysis`，该参数需要 Debug 构建。

## 总结

逃逸分析是 JIT 编译期对对象作用域的分析，让"未逃逸"的对象享受标量替换、栈上分配（间接实现）、锁消除三种优化。HotSpot 中"栈上分配"通过标量替换间接实现，对象根本不被创建。逃逸分析默认开启，写代码时让对象不逃逸（短小、不返回、不传外部、不进异步任务），能让 JIT 自动优化。理解逃逸分析能帮助写出对 JIT 友好的代码，避免在循环里创建临时对象。
