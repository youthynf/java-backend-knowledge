# 如何区分堆和栈

## 核心概念

堆和栈是 JVM 运行时数据区中两个最常用的区域，分工完全不同：栈是"运行的单位"，存方法调用的上下文；堆是"存储的单位"，存对象实例。区分两者的关键在于看存储内容、生命周期、可见性、扩展方式、异常类型。

一句话总结：栈解决"方法怎么执行"，堆解决"对象怎么放"。

```text
┌─────────────────────────────────┐    ┌──────────────────────────────┐
│           栈（线程私有）        │    │     堆（线程共享）           │
│  ┌─────────────────────────┐    │    │  ┌─────────┬──────────────┐  │
│  │ 栈帧 main               │    │    │  │ 新生代  │  老年代      │  │
│  ├─────────────────────────┤    │    │  │Eden+S+S │              │  │
│  │ 栈帧 method1            │    │    │  └─────────┴──────────────┘  │
│  ├─────────────────────────┤    │    │                              │
│  │ 栈帧 method2            │    │    │  对象实例 / 数组 / 静态变量 │
│  └─────────────────────────┘    │    │                              │
│  局部变量 / 操作数栈 / 返回地址│    │  GC 管理                     │
└─────────────────────────────────┘    └──────────────────────────────┘
```

## 标准回答

栈和堆是 JVM 内存中两个完全不同的区域。一句话结论：栈是线程私有的方法调用上下文，存局部变量；堆是线程共享的对象存储区，存对象实例。

| 维度 | 栈 | 堆 |
|------|----|----|
| 存储内容 | 局部变量、操作数栈、方法返回地址 | 对象实例、数组 |
| 生命周期 | 随方法调用创建/销毁 | 由 GC 管理，不确定 |
| 线程隔离 | 线程私有 | 线程共享 |
| 大小 | 固定（`-Xss`，默认 1MB） | 动态（`-Xms` / `-Xmx`） |
| 异常 | `StackOverflowError` / `OutOfMemoryError` | `OutOfMemoryError: Java heap space` |
| 存取速度 | 快，指针移动即可 | 慢，受 GC 影响 |
| 回收方式 | 栈帧出栈自动回收 | GC 自动回收 |
| 是否有 GC | 否 | 是 |

## 栈的特点

### 用途

栈存储方法调用上下文。每调用一个方法，JVM 在当前线程栈顶创建一个栈帧，方法返回时弹出。栈帧包含：

- **局部变量表**：方法参数和方法内变量，以 Slot 为单位。
- **操作数栈**：计算过程的中间结果。
- **动态链接**：指向运行时常量池的方法引用。
- **返回地址**：方法退出后回到调用者的位置。

### 生命周期

栈帧随方法调用入栈、返回出栈，无需 GC。线程结束时整个栈销毁。

### 存取速度

栈的访问基于指针移动，速度快。栈帧内的局部变量在编译期已知，JVM 可以直接索引。

### 异常

| 异常 | 触发条件 |
|------|---------|
| `StackOverflowError` | 栈深度超过限制（递归过深） |
| `OutOfMemoryError` | 无法为新线程分配栈（`-Xss` 设置过大） |

```java
// StackOverflowError
public class Recursion {
    public static void recursion() { recursion(); }
    public static void main(String[] args) { recursion(); }
}
```

### 控制参数

`-Xss` 设置单线程栈大小：

- JDK 5+ 默认 1MB。
- 设置过大，单机能创建的线程数减少。
- 设置过小，递归容易溢出。

## 堆的特点

### 用途

堆存对象实例和数组，是 GC 主战场。几乎所有的 `new` 都在堆上分配（除非逃逸分析后栈上分配）。

### 生命周期

对象生命周期不确定，由 GC 管理。从 GC Roots 不可达的对象会被回收。

### 存取速度

堆的访问比栈慢，因为：

- 对象创建涉及内存分配。
- GC 需要扫描堆。
- 多线程访问需要同步。

### 异常

| 异常 | 触发条件 |
|------|---------|
| `OutOfMemoryError: Java heap space` | 堆空间不足 |
| `OutOfMemoryError: GC overhead limit exceeded` | GC 占用过多时间但仍回收甚少 |

```java
// Java heap space OOM
List<byte[]> list = new ArrayList<>();
while (true) list.add(new byte[1024 * 1024]);
```

### 控制参数

- `-Xms`：堆初始大小。
- `-Xmx`：堆最大大小。
- `-Xmn`：新生代大小。
- `-XX:NewRatio`：老年代:新生代比例。
- `-XX:SurvivorRatio`：Eden:Survivor 比例。

## 栈 vs 堆的存储示例

```java
public class Demo {
    public void method() {
        // 局部变量 i 在栈帧中
        int i = 10;
        // 引用 user 在栈帧中，对象 User 实例在堆中
        User user = new User();
        user.setName("a");
    }
}
```

`i` 是基本类型局部变量，存在栈帧局部变量表。`user` 是引用，本身存于栈帧，指向堆中的 User 对象。User 对象的实例字段也在堆中。

## 实战场景

| 场景 | 栈相关 | 堆相关 |
|------|--------|--------|
| 递归调用 | 控制 `-Xss`，避免 StackOverflow | - |
| 对象频繁创建 | 让对象不逃逸，享受栈上分配 | 减小临时对象，避免 GC 压力 |
| 线程数优化 | 减小 `-Xss` 提升并发数 | - |
| OOM 排查 | 看栈深度 | dump heap 分析对象分布 |

## 深挖追问

### 栈上分配对象时，对象还存在堆中吗？

不存在。如果 JIT 通过逃逸分析确认对象不逃逸并触发标量替换，对象根本不会被创建，字段直接拆为标量存在栈帧或寄存器中。如果只是"栈上分配"未触发标量替换，HotSpot 实际上没有真正实现，等价于标量替换。

### 栈帧之间可以共享数据吗？

不可以。不同线程的栈帧互不可见。同一线程内不同方法的栈帧也不直接共享，但可以通过参数传递和返回值传递引用。栈帧之间不存在引用关系，这是 JVM 规范的明确约束。

### 堆和栈哪个更大？

堆通常远大于栈。栈固定（默认 1MB），堆可达数 GB 甚至 TB。栈的内存上限受线程数和 `-Xss` 影响（如 1000 线程 × 1MB = 1GB），堆的上限受物理内存和 `-Xmx` 限制。

### 为什么堆要分代？

基于分代假说：绝大多数对象朝生夕灭，少数对象长期存活。分代后新生代用复制算法（存活少效率高），老年代用标记-整理（避免碎片）。详见 [常用的垃圾回收算法有哪些？](/01-java-core/jvm/gc/常用的垃圾回收算法有哪些？.md)。

## 易错点

- 把"栈帧"和"栈"混淆，栈帧是栈的一个元素。
- 以为引用存在堆中，引用本身在栈帧，引用指向的对象在堆。
- 把"递归过深"和"循环过深"混淆，循环不增加栈深度，递归会增加。
- 以为增大 `-Xss` 能提升性能，只能提升递归深度，对性能无直接帮助。
- 以为堆越大约好，堆越大 GC 停顿越长，需要平衡。

## 总结

栈和堆是 JVM 内存中分工最不同的两个区域。栈线程私有、存方法调用上下文、固定大小、随方法自动管理；堆线程共享、存对象实例、动态扩展、由 GC 管理。区分两者看存储内容、生命周期、可见性、异常类型、控制参数。理解它们的差异是排查 StackOverflow、OOM 和优化 GC 的基础。
