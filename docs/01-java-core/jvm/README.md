# JVM 深度解析

> 面试高频模块：内存模型、GC、类加载、性能调优

## 核心概念

### JVM 内存模型（JDK8）

```
┌─────────────────────────────────────────────────────────────────┐
│                         JVM 进程                                │
├─────────────────────────────────────────────────────────────────┤
│  线程私有                                                        │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │ 程序计数器  │  │  虚拟机栈   │  │ 本地方法栈  │             │
│  │ (PC)        │  │ (Stack)     │  │ (Native)    │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
├─────────────────────────────────────────────────────────────────┤
│  线程共享                                                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                      堆 (Heap)                            │  │
│  │  ┌─────────────────────────────────┬───────────────────┐  │  │
│  │  │           新生代                │      老年代       │  │  │
│  │  │  ┌───────┬───────┬───────┐    │                   │  │  │
│  │  │  │ Eden  │ S0    │ S1    │    │                   │  │  │
│  │  │  │  80%  │ 10%   │ 10%   │    │                   │  │  │
│  │  │  └───────┴───────┴───────┘    │                   │  │  │
│  │  └─────────────────────────────────┴───────────────────┘  │  │
│  └───────────────────────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │              元空间 (Metaspace) - 直接内存                │  │
│  │              类信息、常量、静态变量                       │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

#### 各区域详解

| 区域 | 线程 | 存储 | 异常 |
|------|------|------|------|
| 程序计数器 | 私有 | 当前执行的字节码指令位置 | 无（唯一不会 OOM） |
| 虚拟机栈 | 私有 | 栈帧（局部变量、操作数栈、动态链接、返回地址） | StackOverflowError / OOM |
| 本地方法栈 | 私有 | Native 方法调用 | StackOverflowError / OOM |
| 堆 | 共享 | 对象实例、数组 | OOM |
| 元空间 | 共享 | 类元数据、常量池、方法数据 | OOM（本地内存） |

#### 虚拟机栈栈帧结构

```
┌────────────────────────────────────────┐
│              栈帧 (Stack Frame)        │
├────────────────────────────────────────┤
│  局部变量表                             │
│  - 方法参数                             │
│  - 局部变量（基本类型 + 对象引用）       │
├────────────────────────────────────────┤
│  操作数栈                               │
│  - 计算中间结果                         │
│  - 方法调用参数                         │
├────────────────────────────────────────┤
│  动态链接                               │
│  - 指向运行时常量池的方法引用            │
├────────────────────────────────────────┤
│  返回地址                               │
│  - 正常退出：调用者的 PC + 指令长度      │
│  - 异常退出：异常表                     │
└────────────────────────────────────────┘
```

#### 为什么 JDK8 用元空间替代永久代？

| 对比项 | 永久代（JDK7） | 元空间（JDK8） |
|--------|---------------|---------------|
| 位置 | JVM 堆内存 | 本地内存 |
| 大小限制 | 受 MaxPermSize 限制 | 受系统内存限制 |
| OOM 风险 | 容易出现 | 降低但仍可能 |
| 内存回收 | 困难（类卸载条件苛刻） | 仍困难，但空间更大 |
| 调优 | `-XX:PermSize` / `-XX:MaxPermSize` | `-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize` |

### 对象创建与内存布局

#### 对象创建流程

```
Object obj = new Object();

1. 类加载检查
   ↓ new 指令是否已加载？符号引用转直接引用
2. 分配内存
   ↓ 指针碰撞（规整内存）or 空闲列表（不规整）
3. 初始化零值
   ↓ 除对象头外全部置零
4. 设置对象头
   ↓ 哈希码、GC 分代年龄、锁状态
5. 执行 <init>
   ↓ 构造函数
```

#### 内存分配的并发问题

**方案一：CAS + 失败重试**
- 原子操作更新指针
- 失败则重试

**方案二：TLAB（Thread Local Allocation Buffer）**
- 每个 Eden 区预分配一小块内存给线程
- 只有 TLAB 用完需要申请新 TLAB 时才同步
- `-XX:+UseTLAB`（默认开启）

#### 对象内存布局

```
┌────────────────────────────────────────┐
│              对象头 (Header)           │
│  ┌────────────────────────────────────┐│
│  │ Mark Word (32/64 bit)              ││
│  │ - 哈希码                            ││
│  │ - GC 分代年龄                       ││
│  │ - 锁状态标志                        ││
│  │ - 偏向线程 ID                       ││
│  └────────────────────────────────────┘│
│  ┌────────────────────────────────────┐│
│  │ 类型指针 (Class Pointer)           ││
│  │ - 指向类元数据                      ││
│  └────────────────────────────────────┘│
│  ┌────────────────────────────────────┐│
│  │ 数组长度（仅数组对象有）            ││
│  └────────────────────────────────────┘│
├────────────────────────────────────────┤
│           实例数据 (Instance Data)     │
│  - 父类继承的变量                       │
│  - 本类定义的变量                       │
├────────────────────────────────────────┤
│           对齐填充 (Padding)           │
│  - 保证对象大小是 8 字节的整数倍        │
└────────────────────────────────────────┘
```

#### Mark Word 状态变化（64 位 JVM）

| 锁状态 | 存储内容 | 说明 |
|--------|---------|------|
| 无锁 | hashcode(31) + age(4) + biased(1) + lock(2) | 未锁定状态 |
| 偏向锁 | thread(54) + epoch(2) + age(4) + biased(1) + lock(2) | 锁偏向某线程 |
| 轻量级锁 | ptr(62) + lock(2) | 指向栈中 Lock Record |
| 重量级锁 | ptr(62) + lock(2) | 指向 ObjectMonitor |
| GC 标记 | ptr(62) + lock(2) | 标记为垃圾 |

### 垃圾回收基础

#### 对象存活判定

**引用计数法**
- 对象被引用时计数 +1，引用失效时 -1
- 问题：循环引用无法回收

**可达性分析（JVM 使用）**
```
GC Roots:
├── 虚拟机栈中引用的对象（局部变量）
├── 方法区中类静态属性引用的对象
├── 方法区中常量引用的对象
├── 本地方法栈中 JNI 引用的对象
├── 同步锁持有的对象
├── JVM 内部引用（基本类型的 Class 对象、常驻异常对象等）
```

#### 引用类型

| 类型 | 类 | 回收时机 | 使用场景 |
|------|-----|---------|---------|
| 强引用 | - | 从不回收 | 普通对象 |
| 软引用 | SoftReference | 内存不足时 | 缓存 |
| 弱引用 | WeakReference | 下次 GC | ThreadLocal、WeakHashMap |
| 虚引用 | PhantomReference | 随时可能 | 跟踪对象回收 |

#### finalize() 方法

- 对象第一次被标记为可回收时，判断是否需要执行 finalize()
- 如果对象未覆盖 finalize() 或已被调用过，则不执行
- 如果需要执行，放入 F-Queue 队列，由低优先级线程执行
- 对象可以在 finalize() 中拯救自己（重新建立引用链）

```java
public class FinalizeEscape {
    private static FinalizeEscape SAVE_ME;
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        SAVE_ME = this;  // 拯救自己
    }
}
```

### 垃圾回收算法

#### 标记-清除（Mark-Sweep）

```
优点：简单
缺点：1. 效率不高（标记和清除都遍历）
      2. 内存碎片（大量不连续空间）
```

#### 复制算法（Copying）

```
优点：无碎片、分配高效（指针碰撞）
缺点：空间利用率低（需要预留一半）
适用：新生代（存活率低）

优化：Eden:S0:S1 = 8:1:1，只浪费 10%
```

#### 标记-整理（Mark-Compact）

```
优点：无碎片、空间利用率高
缺点：移动对象成本高
适用：老年代（存活率高）
```

#### 分代收集

| 区域 | 特点 | 收集算法 |
|------|------|----------|
| 新生代 | 对象存活率低（90%+ 是垃圾） | 复制算法 |
| 老年代 | 对象存活率高 | 标记-清除 / 标记-整理 |

## 垃圾收集器

### 收集器对比

| 收集器 | 类型 | 算法 | 适用场景 |
|--------|------|------|----------|
| Serial | 新生代 | 复制 | 单核、客户端 |
| Serial Old | 老年代 | 标记-整理 | 单核、客户端 |
| ParNew | 新生代 | 复制 | 多核、配合 CMS |
| Parallel Scavenge | 新生代 | 复制 | 吞吐量优先 |
| Parallel Old | 老年代 | 标记-整理 | 吞吐量优先 |
| CMS | 老年代 | 标记-清除 | 响应时间优先 |
| G1 | 全堆 | 复制 + 标记-整理 | 响应时间可控 |
| ZGC | 全堆 | 并发整理 | 超低延迟 |

### CMS 收集器

**四个阶段：**

```
1. 初始标记 (STW)     ─── 标记 GC Roots 直接关联的对象
2. 并发标记           ─── 从 GC Roots 遍历整个堆（用户线程并发）
3. 重新标记 (STW)     ─── 修正并发标记期间的变动（增量更新）
4. 并发清除           ─── 清除垃圾对象（用户线程并发）
```

**问题：**
1. CPU 敏感：默认启动 (CPU+3)/4 个线程
2. 浮动垃圾：并发清理时新产生的垃圾
3. 内存碎片：标记-清除算法导致
4. Concurrent Mode Failure：老年代空间不足触发 Full GC

**参数：**
```bash
-XX:+UseConcMarkSweepGC          # 启用 CMS
-XX:CMSInitiatingOccupancyFraction=75  # 老年代占用 75% 时触发
-XX:+UseCMSCompactAtFullCollection     # Full GC 时压缩
-XX:CMSFullGCsBeforeCompaction=0       # 每次 Full GC 都压缩
```

### G1 收集器

**Region 分区：**
```
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ E  │ S  │    │ O  │ O  │ O  │ H  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ O  │    │ E  │ E  │ S  │ O  │ H  │
└────┴────┴────┴────┴────┴────┴────┴────┘

E = Eden    S = Survivor    O = Old    H = Humongous（大对象）
```

**四个阶段：**
```
1. 初始标记 (STW)     ─── 标记 GC Roots 直接关联的对象
2. 并发标记           ─── 从 GC Roots 遍历整个堆（SATB）
3. 最终标记 (STW)     ─── 处理 SATB 队列中的变更
4. 筛选回收 (STW)     ─── 复制存活对象到新 Region
```

**优点：**
- 可预测停顿时间：`-XX:MaxGCPauseMillis=200`
- 无内存碎片：复制算法
- 大对象优化：Humongous Region

**参数：**
```bash
-XX:+UseG1GC                    # 启用 G1
-XX:MaxGCPauseMillis=200        # 最大停顿时间目标
-XX:G1HeapRegionSize=4m         # Region 大小（1-32MB，2 的幂）
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用 45% 触发并发标记
```

### ZGC 收集器

**特点：**
- 停顿时间 < 10ms（JDK17+ < 1ms）
- 支持堆大小 8MB ~ 16TB
- JDK15 正式可用，JDK21 分代 ZGC

**核心技术：**
1. **着色指针**：在指针中存储标记信息
2. **读屏障**：在引用读取时修正指针
3. **多重映射**：同一物理内存映射到多个虚拟地址

**参数：**
```bash
-XX:+UseZGC                     # 启用 ZGC
-XX:+ZGenerational              # 启用分代 ZGC（JDK21+）
-Xmx16g                         # 最大堆大小
```

## 面试高频问题

### 1. Minor GC / Major GC / Full GC 区别？

| GC 类型 | 作用区域 | 触发条件 |
|---------|---------|---------|
| Minor GC / Young GC | 新生代 | Eden 区满 |
| Major GC | 老年代 | 老年代空间不足 |
| Full GC | 整个堆 | System.gc()、老年代满、方法区满 |

### 2. 对象如何进入老年代？

**方式一：年龄达到阈值**
- 默认 15 岁（`-XX:MaxTenuringThreshold=15`）
- 每次 Minor GC 存活年龄 +1

**方式二：大对象直接进入**
- 超过 `-XX:PretenureSizeThreshold` 的对象

**方式三：动态年龄判定**
- Survivor 区相同年龄对象总和 > Survivor 区一半
- 年龄 >= 该年龄的对象进入老年代

**方式四：空间担保失败**
- Minor GC 前，老年代最大可用连续空间 < 新生代所有对象总大小
- 且不允许担保失败，则 Full GC

### 3. CMS 和 G1 的区别？

| 对比项 | CMS | G1 |
|--------|-----|-----|
| 内存布局 | 传统分代 | Region 分区 |
| 算法 | 标记-清除 | 复制 |
| 碎片 | 有 | 无 |
| 停顿时间 | 不可控 | 可预测 |
| 适用堆 | < 6GB | > 6GB |
| 并发标记 | 增量更新 | SATB |

### 4. 如何排查 OOM？

**步骤：**
```bash
# 1. 查看堆内存使用
jmap -heap <pid>

# 2. 导出堆转储
jmap -dump:format=b,file=heap.hprof <pid>

# 3. 分析堆转储
# 使用 MAT、JProfiler、VisualVM 等工具

# 4. 查看线程栈
jstack <pid> > thread.txt

# 5. 查看 GC 日志
# 启动参数添加：-Xlog:gc*:file=gc.log
```

**常见 OOM 类型：**
| 异常 | 原因 | 排查 |
|------|------|------|
| Java heap space | 堆内存不足 | 检查是否有内存泄漏 |
| Metaspace | 类加载过多 | 检查动态代理、反射 |
| GC overhead limit exceeded | GC 时间过长 | 检查对象创建频率 |
| Direct buffer memory | 直接内存不足 | 检查 NIO 使用 |

## 性能调优

### 常用参数

```bash
# 堆大小
-Xms2g                          # 初始堆大小
-Xmx2g                          # 最大堆大小（与 Xms 相同避免动态扩容）

# 新生代
-Xmn1g                          # 新生代大小
-XX:NewRatio=2                  # 新生代:老年代 = 1:2
-XX:SurvivorRatio=8             # Eden:S0:S1 = 8:1:1

# 元空间
-XX:MetaspaceSize=256m          # 初始元空间大小
-XX:MaxMetaspaceSize=512m       # 最大元空间大小

# GC 日志（JDK9+）
-Xlog:gc*:file=gc.log:time,tags:filecount=5,filesize=10m

# 堆转储（OOM 时自动导出）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heap.hprof
```

### 调优思路

**1. 确定目标**
- 吞吐量优先：Parallel Scavenge + Parallel Old
- 响应时间优先：G1 / ZGC

**2. 分析现状**
```bash
# 查看 GC 统计
jstat -gcutil <pid> 1000

# 查看 GC 原因
jstat -gccause <pid> 1000
```

**3. 调整参数**
- 频繁 Minor GC：增大新生代
- 频繁 Full GC：增大老年代、检查内存泄漏
- GC 时间过长：减少堆大小、换低延迟收集器
