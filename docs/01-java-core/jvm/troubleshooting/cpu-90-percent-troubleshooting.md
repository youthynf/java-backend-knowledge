# 线上 Java 服务 CPU 飙到 90% 以上，第一步操作是什么？

> 凌晨两点，报警群炸了。"XX 服务 CPU 92%，请求超时！"
> 新手第一反应是 `tail -f error.log`，翻三分钟啥也没找到。
> 老手已经敲完 `top -c`，锁定了 PID，甚至已经知道是哪行代码在搞事情。
>
> **看日志没错，但日志是"结果"，CPU 是"现场"。**
> 错误日志可能还没刷盘，或者压根不是抛异常——纯计算也能把 CPU 吃满。
> 所以排查 CPU 的第一原则永远是：**先定位"谁"在跑，再定位"哪"在跑**。

---

## 一、先别急着翻日志，top 才是第一现场

**面试官**：线上 Java 服务 CPU 飙到 90% 以上，你的第一步操作是什么？为什么不是直接看日志？

**候选人**：第一步一定是 `top -c`，或者 `htop`。先确认占用 CPU 的进程到底是谁。如果是 Java 进程，记下 PID，再往下钻。

**面试官**：为什么不是先看日志？

**候选人**：因为 CPU 高不一定是抛异常导致的。比如**死循环、正则回溯、频繁 Full GC**，这些场景业务日志可能完全正常，甚至没有任何 ERROR。
日志是"事后诸葛亮"，而 top 看到的是"正在发生的犯罪现场"。先看日志，很可能浪费黄金三分钟。

**面试官**：如果一台机器上有多个 Java 进程，怎么快速确定是哪个？

**候选人**：`top -c` 会显示完整启动命令，一眼就能看出哪个是业务进程。或者 `ps -ef | grep java` 配合 CPU 占用率对比，也能快速筛选。

---

## 二、从进程到线程：精准定位"凶手"

**面试官**：已经定位到 Java 进程 PID，下一步怎么找到具体是哪个线程在消耗 CPU？

**候选人**：`top -Hp <pid>`。
- `-H` 显示线程视图
- `-p` 指定进程

这时候能看到该进程下所有线程的 CPU 占用，找到 `%CPU` 最高的那个，记下 TID。

**面试官**：拿到 TID 之后呢？

**候选人**：jstack 导出的线程 ID 是十六进制，而 top 里看到的是十进制，需要转换：

```bash
printf "%x\n" <tid>
# 得到十六进制值，比如 0x3a2f
```

**面试官**：然后怎么定位代码？

**候选人**：

```bash
# 方式一：直接 grep
jstack <pid> | grep -A 30 <十六进制tid>

# 方式二：导文件再搜
jstack <pid> > stack.log
# 在文件里搜 nid=0x<十六进制tid>
```

`nid` 就是 **Native Thread ID**，对上了就能看到完整的调用栈，直接定位到类名、方法名、代码行。

---

## 三、jstack 之后，怎么看栈？

**面试官**：jstack 之后，线程状态是 `RUNNABLE`，栈顶卡在业务代码的一个 `for` 循环里，说明什么？

**候选人**：大概率是**死循环**，或者批量处理的数据量远超预期。
`RUNNABLE` 表示线程正在运行，如果长时间卡在一个业务方法的循环体内，基本就是代码逻辑问题。

**面试官**：如果栈顶卡在 `java.util.regex.Pattern` 呢？

**候选人**：这是典型的**正则回溯灾难**。比如用了 `.*` 贪婪匹配，输入字符串稍微复杂一点，CPU 就直接爆炸。
解决方式：
- 优化正则，改成**独占模式** `++` 或 `*+`
- 预校验字符串长度

**面试官**：那如果线程状态是 `BLOCKED` 呢？

**候选人**：说明线程在抢锁。如果大量线程都 `BLOCKED` 在同一把 synchronized 锁上，就要考虑**锁粒度**是不是太粗了。比如：
- 把 `HashMap` 换成 `ConcurrentHashMap`
- 把同步块拆小
- 甚至改成无锁设计（CAS / 原子类）

**面试官**：怎么判断 CPU 高是 GC 引起的？

**候选人**：两个办法：

1. **jstack 里看线程名**：如果出现 `VM Thread`、`GC task thread#`，说明 GC 线程在疯狂工作。
2. **jstat 持续观察**：
   ```bash
   jstat -gcutil <pid> 1000
   ```
   如果 YGC/FGC 次数每秒都在涨，且 CPU 曲线和 GC 时间吻合，那就是 GC 问题。
   接下来就要转去排查**内存泄漏**或者**大对象**了。

---

## 四、生产环境的"核武器"：Arthas

**面试官**：生产环境很多时候不让随便 jstack，或者 jstack 会触发 STW，你有什么更优雅的方案？

**候选人**：**Arthas**。直接：

```bash
thread -n 5
```

不用手动转十六进制，不用导文件，直接列出 CPU 占用最高的 5 个线程，调用栈一目了然。

**面试官**：Arthas 的 `thread` 和 `profiler` 有什么区别？

**候选人**：
- `thread`：快照，适合快速定位"此刻"谁在跑
- `profiler`：采样，生成火焰图，适合看一段时间内的方法级热点

如果 CPU 高是**持续的、方法调用复杂**的场景，火焰图更直观，一眼就能看出是哪个方法的哪个子调用在烧 CPU。

---

## 五、灵魂追问：如果 8 个核全被打满呢？

**面试官**：最后一题。8 核机器，CPU 飙到 800%，`top -Hp` 发现有 8 个线程各占 100%，jstack 显示它们都在执行同一个业务方法，但代码 review 并没有死循环。给出三种可能的根因。

**候选人**：

1. **线程池配置失控**
   比如某个接口把线程池核心线程数设成了 100，同时进来 8 个请求，每个请求内部又起了 N 个线程做并行计算，直接把 CPU 压满。

2. **数据量突增**
   比如平时处理 1000 条数据，今天上游灌了 100 万条，同一个方法没有分页，单线程处理超时，线程堆积，8 个核全在跑同一个逻辑。

3. **外部依赖超时引发重试风暴**
   比如调用下游接口超时，框架或业务代码写了重试机制，8 个线程各自疯狂重试，CPU 全耗在序列化和连接建立上。

**面试官**：如果排查到最后，发现是业务代码写了一个 `List.contains()` 做去重，数据量大了变成 O(n²)，这算 Bug 还是性能优化？

**候选人**：线上故障复盘时，**这算 Bug**。因为代码在数据量小的时候能跑，不代表逻辑正确。`contains` 去重是**隐性陷阱**，属于代码逻辑缺陷，而不是简单的"可以优化"。

---

## 六、排查口诀（建议背下来）

> **top 找进程 → top -Hp 找线程 → printf 转十六进制 → jstack 看栈 → 结合代码断案**

如果是**线上救急**，记住 Arthas 最小操作集：

```bash
thread -n 5
profiler start --event cpu
profiler stop --file /tmp/cpu.html
```

三行命令，五分钟出火焰图，足够应付 90% 的 CPU 飙高场景。

---

## 延伸思考

- **CPU 高 vs 内存高**：排查思路完全不同，内存高优先看 `jmap -heap` 和 dump 分析
- **容器环境**：在 Docker / K8s 里，`top` 看到的 CPU 利用率可能不准，建议用 `pidstat -p <pid> 1` 或容器内的 cgroup 指标
- **JFR (Java Flight Recorder)**：JDK 11+ 内置，比 Arthas 更轻量的生产级 profiler
- **async-profiler**：Arthas profiler 的底层实现，可以独立使用，生成更精细的火焰图

# 面试复习补充

## 核心概念补充

这篇文章的主题是 **线上 Java 服务 CPU 飙到 90% 以上，第一步操作是什么？**。复习时应先给出定义，再说明它在 Java 并发或 JVM 体系中的位置，最后结合使用场景、限制条件和常见误区展开。

## 面试官想考什么

- 是否能用自己的话讲清楚概念，而不是只背术语。
- 是否理解底层机制、关键流程以及它和相邻知识点的区别。
- 是否能把知识点落到真实项目：如何使用、如何排查、如何调优、什么时候不该用。
- 是否知道常见坑点，例如线程安全、可见性、阻塞、内存泄漏、GC 停顿或参数误用。

## 标准回答

JVM 排查题要先稳定现场，再定位范围：CPU、内存、线程、GC、IO 或外部依赖。回答时要给出工具链和判断依据，例如 `top`/`jstack` 定位高 CPU 线程，`jmap`/堆转储分析内存，GC 日志判断停顿来源，最后给出修复和复盘措施。

## 深挖追问

- CPU 飙高时如何把 OS 线程 id 对应到 Java 线程栈？
- OOM 后如何区分堆溢出、元空间、直接内存、线程数过多？
- 线上抓取 dump 有什么风险，如何降低对服务的影响？
- 调参前后如何验证收益，避免只凭感觉修改参数？

## 实战场景/代码示例

```bash
jcmd <pid> Thread.print > thread-dump.txt
jcmd <pid> GC.heap_dump /tmp/heap.hprof
jcmd <pid> VM.native_memory summary   # 需提前开启 NMT 才有完整信息
```

## 易错点/总结

- 不要脱离场景背结论：并发和 JVM 问题通常都和负载、线程数、内存大小、JDK 版本有关。
- 面试回答建议采用“定义 → 原理 → 场景 → 风险/排查”的顺序。
- 如果涉及源码或参数，说明核心思路即可；不确定的版本差异要明确限定，不要绝对化。
