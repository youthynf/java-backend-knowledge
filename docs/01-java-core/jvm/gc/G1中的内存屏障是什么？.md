# G1 中的内存屏障是什么

## 核心概念

G1 中的内存屏障是 JVM 在引用更新时插入的代码片段，用于维护 SATB 快照和 RSet 一致性。G1 用两种屏障：写前屏障（Pre-Write Barrier）和写后屏障（Post-Write Barrier）。一句普通的 Java 赋值 `obj.field = other`，JVM 实际会插入两条屏障代码。

屏障的目的不是"内存可见性"（那是 JMM 的内存屏障），而是"GC 一致性"：让 GC 线程能正确捕获引用变化，避免漏标和 RSet 失效。

```text
Java 代码: obj.field = other;

JVM 实际执行：
   1. 写前屏障：处理 SATB，记录旧引用到 SATB 缓冲区
   2. 真正赋值：obj.field = other
   3. 写后屏障：处理 RSet，把 Card 加入 DCQ
```

## 标准回答

G1 用两种内存屏障维护 GC 数据结构：写前屏障处理 SATB（保证并发标记正确性），写后屏障处理 RSet（记录跨 Region 引用变化）。两者都由 JIT 在引用更新处自动插入，对程序员透明。SATB 把旧引用入队供最终标记处理；RSet 通过 DCQ + Refine 线程异步更新。

要点：

1. **写前屏障 Pre-Write Barrier**：处理 SATB，记录引用消失前的旧值。
2. **写后屏障 Post-Write Barrier**：处理 RSet，把脏 Card 加入 DCQ。
3. **目的**：保证并发标记正确性 + RSet 实时性。
4. **JIT 透明插入**：程序员无需关心，JVM 自动处理。
5. **开销**：每次引用更新都要执行屏障，有一定性能开销。

## 两种屏障

### 写前屏障（Pre-Write Barrier）

在引用更新前执行，记录"即将被覆盖的旧引用"。目的是 SATB（Snapshot-At-The-Beginning）：把并发标记开始时的对象图快照保留下来。

```text
def pre_write_barrier(src, field, new_value):
    if 并发标记阶段:
        old_value = src.field
        if old_value != null:
            satb_buffer.add(old_value)   # 旧引用入队
    src.field = new_value                # 真正赋值
```

为什么需要？并发标记中，如果灰色对象 A 引用白色对象 C，引用消失（A.field = null）后 C 应该被漏标。SATB 在消失前记录 C 到缓冲区，最终标记阶段强制标记为存活，避免漏标。

### 写后屏障（Post-Write Barrier）

在引用更新后执行，记录"新建立的引用关系"。目的是 RSet 维护：跨 Region 引用需要被记录到被引用方的 RSet。

```text
def post_write_barrier(src, field, new_value):
    if 跨 Region 引用:
        card = get_card(src, field)
        mark_card_dirty(card)
        dcq.add(card)                    # 脏 Card 入队
```

为什么需要？跨 Region 引用是 RSet 维护的依据。写后屏障把脏 Card 加入 DCQ，Refine 线程消费 DCQ 更新 RSet。

## 一句赋值的三条伪代码

G1 中 `obj.field = other` 实际是：

```text
1. JVM → Insert Pre-Write Barrier
   处理 SATB，记录旧引用到 SATB 缓冲区

2. obj.field = other
   真正的赋值

3. JVM → Insert Post-Write Barrier
   处理 RSet，把脏 Card 加入 DCQ
```

SATB 缓冲区的对象在最终标记阶段被强制标记为存活，避免漏标。DCQ 中的脏 Card 由 Refine 线程消费，更新 RSet。

## 屏障与 SATB / RSet 的关系

| 屏障 | 维护的数据结构 | 目的 |
|------|--------------|------|
| 写前屏障 | SATB 缓冲区 | 并发标记正确性（避免漏标） |
| 写后屏障 | DCQ → RSet | 跨 Region 引用跟踪（避免全堆扫描） |

两者分工明确：写前关注"引用消失"（保 SATB 快照），写后关注"引用建立"（保 RSet 索引）。

## 屏障的开销

每次引用更新都触发两条屏障，开销包括：

- 判断并发标记阶段是否进行（写前屏障）。
- 判断是否跨 Region 引用（写后屏障）。
- 入队操作（SATB 缓冲区、DCQ）。

HotSpot 通过快速过滤优化：

- 大部分引用更新不是跨 Region 的，写后屏障快速跳过。
- SATB 只在并发标记阶段启用，非并发标记阶段写前屏障几乎无开销。

实测 G1 的屏障开销约 5%~10% 的吞吐量损失，换来可预测停顿。

## 代码示例

理解屏障触发：

```java
public class BarrierDemo {
    Object field;

    public void set(Object other) {
        // 实际执行：
        // 1. 写前屏障（SATB）：记录 field 当前值（旧引用）
        // 2. this.field = other
        // 3. 写后屏障（RSet）：标记 this 所在 Card 为脏，加入 DCQ
        this.field = other;
    }
}
```

## 实战场景

| 场景 | 关注点 | 调优 |
|------|--------|------|
| G1 吞吐量低 | 屏障开销 | 评估是否用 Parallel GC |
| Refine 线程 CPU 高 | DCQ 增长过快 | 增大 `-XX:G1ConcRefinementThreads` |
| 并发标记阶段 CPU 高 | SATB 处理 | 评估 `-XX:ConcGCThreads` |

## 深挖追问

### G1 的屏障和 JMM 的内存屏障是一回事吗？

不是。JMM 的内存屏障（LoadLoad、StoreStore 等）是禁止 CPU 重排序的硬件指令，用于保证内存可见性。G1 的屏障是 JIT 在引用更新处插入的代码片段，用于维护 GC 数据结构（SATB、RSet）。两者名字相同但含义不同。

### 为什么 CMS 只用增量更新，G1 用 SATB？

CMS 用增量更新（关注引用增加），把黑色重新标记为灰色，重新扫描。G1 用 SATB（关注引用消失），把旧引用入队最终标记。G1 选 SATB 是因为 RSet 已经记录了跨 Region 引用，最终标记处理 SATB 缓冲区时可以直接通过 RSet 找到引用源，效率高。CMS 没有 RSet，SATB 没有优势。

### 写前屏障为什么只在并发标记阶段执行？

SATB 的目的是保留并发标记开始时的快照。非并发标记阶段没有并发标记线程在跑，引用变化不需要记录。JIT 通过读全局状态判断当前是否在并发标记阶段，避免无谓开销。

### 屏障会对所有引用更新执行吗？

是的，所有引用字段更新都触发屏障。但写后屏障会先判断"是否跨 Region 引用"，同 Region 引用直接跳过 RSet 维护。读屏障（ZGC 才有）G1 没有。

## 易错点

- 把 G1 的屏障和 JMM 的内存屏障混淆，前者是 GC 数据结构维护，后者是内存可见性。
- 以为屏障只在 GC 时执行，实际每次引用更新都执行。
- 以为写前屏障和写后屏障功能相同，前者维护 SATB，后者维护 RSet。
- 以为 SATB 缓冲区的对象会被回收，实际被强制标记为存活（避免漏标）。
- 把 DCQ 和 SATB 缓冲区混淆，DCQ 给 RSet 用，SATB 缓冲区给并发标记用。

## 总结

G1 用两种内存屏障维护 GC 数据结构：写前屏障处理 SATB（避免漏标），写后屏障处理 RSet（跨 Region 引用跟踪）。每次引用更新都触发两条屏障，由 JIT 自动插入。SATB 缓冲区供最终标记处理，DCQ 供 Refine 线程消费更新 RSet。屏障的开销约 5%~10% 吞吐量，是 G1 可预测停顿的代价。理解屏障是理解 G1 并发标记和 RSet 维护的基础。
