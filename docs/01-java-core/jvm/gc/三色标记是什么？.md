# 三色标记是什么

## 核心概念

三色标记（Tri-color Marking）是并发标记阶段的核心算法，用白、灰、黑三种颜色表示对象的标记状态。它解决了"并发标记时用户线程还在修改引用关系"的问题，是 CMS、G1、ZGC、Shenandoah 等现代并发收集器的共同理论基础。

三色标记的核心思想：白色未访问、灰色已访问但子引用未扫完、黑色完全标记。GC 开始时所有对象白色，结束后白色对象就是垃圾。

```text
GC 开始：所有对象白色
GC Roots 入栈：根对象变灰
循环：取出灰色对象 → 扫描子引用 → 子引用变灰 → 当前对象变黑
GC 结束：黑色存活，白色垃圾
```

## 标准回答

三色标记是并发标记算法，用三种颜色表示对象状态。一句话结论：白色未访问，灰色已访问但子引用未扫完，黑色完全标记；GC 结束后白色对象是垃圾。并发标记的最大挑战是"漏标"，通过写屏障 + 增量更新（CMS）或 SATB（G1）解决。

要点：

1. **三色定义**：白（未访问）、灰（已访问子未完）、黑（完全标记）。
2. **算法流程**：根对象入栈变灰 → 取灰色对象扫子引用变灰 → 当前变黑 → 直到无灰色。
3. **不变式**：强三色不变式（黑不直接指向白）、弱三色不变式（白可被灰指向）。
4. **漏标问题**：并发标记中"黑指白 + 灰指白引用消失"导致漏标。
5. **解决方案**：增量更新（CMS）、SATB（G1）、染色指针 + 读屏障（ZGC）。

## 三种颜色

| 颜色 | 含义 |
|------|------|
| 白色 White | 未被访问（垃圾候选） |
| 灰色 Gray | 已被访问，但引用字段未完全扫描 |
| 黑色 Black | 已被访问，且所有引用字段都已扫描 |

GC 开始前所有对象都是白色，结束后黑色对象存活，白色对象回收。

## 标记过程

```text
初始阶段：
GC_ROOTS → 全部入栈变灰

循环（直到无灰色对象）：
  取出一个灰色对象 obj
  for field in obj 的引用字段:
      if field.color == WHITE:
          field.color = GRAY  # 子引用变灰入栈
  obj.color = BLACK          # 当前对象变黑

结束：
  无灰色对象
  黑色对象存活
  白色对象回收
```

伪代码：

```python
def mark(roots):
    stack = []
    for root in roots:
        if root.color == WHITE:
            root.color = GRAY
            stack.append(root)

    while stack:
        obj = stack.pop()
        for field in obj.references:
            if field.color == WHITE:
                field.color = GRAY
                stack.append(field)
        obj.color = BLACK
```

## 并发标记的挑战

标记线程和用户线程并发执行时，引用关系会变化，可能导致漏标。

### 漏标的两个条件

漏标发生当且仅当同时满足：

1. **黑色对象指向了白色对象**（新引用）。
2. **灰色对象指向白色对象的引用消失**（旧引用断开）。

```text
原本：  GreyA → WhiteC  (C 还没被扫)
变化：  GreyA -X-> WhiteC    (A 不再引用 C)
        BlackB → WhiteC     (B 新引用 C)
结果：  B 是黑色不会再扫，A 不再引用 C
        C 被漏标，被错误回收！
```

### 不变式

- **强三色不变式**：黑色对象不能直接引用白色对象。
- **弱三色不变式**：黑色对象可以引用白色对象，但该白色对象必须有灰色对象保护（即存在灰色到白色的路径）。

只要打破漏标的两个条件之一，就能保证正确性。

## 解决方案：写屏障

写屏障（Write Barrier）是在引用更新时插入的代码，记录变化的对象。两种实现：

### 1. 增量更新（Incremental Update）— CMS

关注"引用增加"：黑色对象新引用白色对象时，把黑色对象重新标记为灰色，重新扫描。代价是并发标记结束后的重新标记阶段要重新扫描这些黑色对象。

```text
写屏障伪代码：
def write_barrier(src, field, new_value):
    if src.color == BLACK and new_value.color == WHITE:
        src.color = GRAY  # 黑色重新变灰，重新扫描
    src.field = new_value
```

### 2. SATB（Snapshot-At-The-Beginning）— G1

关注"引用消失"：灰色对象引用白色对象的引用消失前，把旧引用记录到 SATB 缓冲区，最终标记阶段处理。详见 [SATB 算法是什么](SATB算法是什么？.md)。

```text
写屏障伪代码：
def write_barrier(src, field, new_value):
    old_value = src.field
    if old_value.color == WHITE:
        satb_buffer.add(old_value)  # 旧引用入队，最终标记处理
    src.field = new_value
```

### 3. 染色指针 + 读屏障 — ZGC

ZGC 不用写屏障，用染色指针在指针中编码标记信息，读屏障在读取引用时修正。详见 [染色指针算法是怎么样的](染色指针算法是怎么样的？.md) 和 [ZGC 垃圾收集器基本原理](ZGC垃圾收集器基本原理.md)。

## 三色标记在各 GC 中的实现

### CMS

- 初始标记（STW）→ 并发标记 → 重新标记（STW）→ 并发清除。
- 用增量更新处理并发标记中的引用变化。

### G1

- 初始标记（STW）→ 并发标记 → 最终标记（STW）→ 筛选回收（STW）。
- 用 SATB + 写屏障处理并发标记。
- 用 RSet 减少跨 Region 扫描。

### ZGC

- 染色指针把颜色信息编码到指针中。
- 读屏障在读取引用时更新指针。
- 几乎完全并发，STW 极短（< 10ms）。

## 性能优化技术

### 并行标记

多个 GC 线程同时标记，每个线程有本地标记栈减少竞争。

### 增量标记

把标记过程分成多个小步骤，与用户线程交替执行，避免长时间 STW。

### 记忆集（Remembered Set）

记录跨代/跨 Region 引用，减少全堆扫描。G1 中每个 Region 有 RSet。

## 代码示例

用对象图理解漏标：

```java
class A { B b; C c; }   // 假设 A 是灰色（已访问但 b/c 未扫）
class B { C c; }        // 假设 B 是黑色（已访问且 c 已扫）
class C { }             // 白色

// 并发执行：
A a = ...; B b = ...; C c = ...;
a.c = c;       // 黑色 b 不再引用 c（A 之前通过 b 引用 c）
b.c = null;    // 灰色 A 的 c 引用赋给 b → 漏标！
// 此时 c 既不被 A 间接引用，也不被 B 引用，但 B 是黑色不会再扫
// c 应该是存活的（A.c 引用），但被漏标
```

写屏障解决：

```text
SATB：A.b = null 时，旧值 b 入 SATB 队列，最终标记处理
增量更新：B.c = c 时，B 重新变灰，重新扫描 B.c
```

## 深挖追问

### 三色标记和分代收集是什么关系？

三色标记是并发标记阶段的算法，分代收集是堆的划分策略。两者不冲突：分代收集中每个代的标记都可以用三色标记。CMS、G1 在老年代用三色标记，新生代通常用复制算法（不需要标记）。

### 强三色不变式和弱三色不变式有什么区别？

强三色：黑色对象完全不能直接引用白色对象。
弱三色：黑色对象可以引用白色，但白色必须有灰色对象保护（即存在灰色到白色的路径）。

破坏漏标条件的方式：

- 增量更新破坏"黑指白"：黑色变灰重新扫描。
- SATB 破坏"灰指白消失"：旧引用入队最终标记。

### 为什么 ZGC 不用写屏障？

写屏障在引用更新时插入，开销与引用更新频率成正比。ZGC 用染色指针把标记信息编码在指针中，配合读屏障在读取引用时修正，避免了写屏障的全量拦截。读屏障由 JIT 在每次引用读取处插入，但有硬件优化（预测加载），实际开销很低。

### G1 用 SATB 而不用增量更新，为什么？

SATB 配合 RSet 浑然天成：RSet 记录跨 Region 引用，SATB 把旧引用入队后，最终标记阶段处理时可以直接通过 RSet 找到引用源，不需要全堆扫描。增量更新把黑色变灰后还要重新扫描所有字段，效率低。

## 易错点

- 把"白色"理解为"存活"，实际白色是"未访问"，GC 结束后是垃圾。
- 以为三色标记只在 G1 用，CMS、ZGC、Shenandoah 都用。
- 把"写屏障"和"内存屏障"（JMM）混淆，前者是 GC 概念，后者是并发编程概念。
- 以为漏标只会少回收对象，实际是错误回收存活对象，导致程序崩溃。
- 把 SATB 和增量更新混淆，SATB 关注引用消失，增量更新关注引用增加。

## 总结

三色标记是并发标记的核心算法，用白、灰、黑表示对象状态。并发标记的挑战是漏标，通过写屏障 + 增量更新（CMS）或 SATB（G1）或染色指针 + 读屏障（ZGC）解决。理解三色不变式、漏标条件、解决方案，是阅读 CMS、G1、ZGC 源码和调优并发收集器的基础。
