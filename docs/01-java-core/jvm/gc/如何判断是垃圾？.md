# 如何判断是垃圾

## 核心概念

判断对象是否是垃圾是 GC 的第一步。主流方法有两种：引用计数法和可达性分析算法。JVM 主流实现（HotSpot、Zing、GraalVM）都用可达性分析，因为引用计数无法解决循环引用。

可达性分析的核心是 GC Roots：从一组根对象出发，沿引用链遍历，不可达的对象就是垃圾。GC Roots 的选择决定了哪些对象能被回收，是 JVM 内存管理的基石。

```text
GC Roots
   │
   ├──> Object A ──> Object C    (可达)
   │
   ├──> Object B ──> Object D    (可达)
   │
   └──> Object E                 (可达)

         Object F (不可达)       (垃圾)
         Object X ←─→ Object Y   (循环引用但不可达，垃圾)
```

## 标准回答

JVM 用可达性分析判断对象是否是垃圾，从 GC Roots 出发沿引用链遍历，不可达的对象是垃圾。引用计数法因为无法解决循环引用被 JVM 弃用。GC Roots 包括虚拟机栈中的引用、方法区类静态属性、常量、本地方法栈 JNI 引用、JVM 内部引用、synchronized 持有的对象。

要点：

1. **引用计数法**：每个对象维护引用计数，加 1 减 1，为 0 时回收；致命缺陷是无法解决循环引用。
2. **可达性分析**：从 GC Roots 出发，不可达的对象判定为垃圾。
3. **GC Roots**：栈帧局部变量、静态字段、常量、JNI、JVM 内部引用、锁对象。
4. **方法区回收**：废弃常量和无用类，需要满足三个条件。
5. **finalize()**：JDK 9+ 已废弃，推荐 `try-with-resources` 或 `Cleaner`。

## 引用计数法

每个对象维护一个引用计数器，被引用时 +1，引用断开时 -1，为 0 时回收。Python 等语言用此算法，但 JVM 不用。

致命缺陷：无法解决循环引用。

```java
public class ReferenceCountingGC {
    public Object instance = null;
    public static void main(String[] args) {
        ReferenceCountingGC a = new ReferenceCountingGC();
        ReferenceCountingGC b = new ReferenceCountingGC();
        a.instance = b;
        b.instance = a;     // 循环引用
        a = null; b = null; // 引用计数都不为 0，但实际已是垃圾
        System.gc();        // JVM 仍能回收，说明不用引用计数
    }
}
```

## 可达性分析算法

JVM 主流算法。从 GC Roots 出发，沿引用链向下搜索，搜索走过的路径称为引用链。如果一个对象到 GC Roots 没有任何引用链，则不可达，可被回收。

### GC Roots

| 类型 | 示例 |
|------|------|
| 虚拟机栈中引用的对象 | 方法中的局部变量、参数 |
| 方法区中类静态属性引用的对象 | `static` 变量引用的对象 |
| 方法区中常量引用的对象 | 常量池引用 |
| 本地方法栈中 JNI 引用的对象 | native 方法引用 |
| JVM 内部引用 | 基本类型 Class 对象、常驻异常对象、类加载器 |
| 同步锁持有的对象 | `synchronized` 锁对象 |
| JMXBean、JVMTI 等 | JVM 内部管理对象 |

为什么这些是 GC Roots？因为它们是程序"仍然能直接访问"的对象入口，从它们出发能找到的对象肯定还在用。

## 方法区的回收

方法区（JDK 8+ 元空间）的回收率远低于新生代，但 JVM 仍会回收废弃常量和无用类。

### 废弃常量

常量池中的字面量（如字符串）没有任何地方引用时，可被回收。

### 无用类

类被卸载需要同时满足三个条件：

1. 该类所有实例都已被回收（堆中不存在该类及子类的实例）。
2. 加载该类的 ClassLoader 已被回收。
3. 该类对应的 `Class` 对象没有在任何地方被引用（无法通过反射访问）。

满足三个条件后 JVM **允许**卸载，但不一定立即卸载。HotSpot 通过 `-Xnoclassgc` 控制是否对类进行卸载。在大量使用反射、动态代理（CGLIB）、JSP、OSGi 的场景，类卸载非常重要，否则元空间会持续增长。

## finalize()（已废弃）

`finalize()` 类似 C++ 析构函数，用于对象回收前的资源清理。当 GC 判定对象不可达后，如果重写了 `finalize()` 且未被调用过，JVM 把它放入 F-Queue，由 Finalizer 线程执行。`finalize()` 中可重新建立与 GC Roots 的引用链实现"自救"，但只允许一次。

JDK 9+ 将 `finalize()` 标记为 `Deprecated`，原因：

- 执行时机不确定。
- 执行顺序不保证。
- JVM 退出时可能不执行。
- 性能开销大。
- 容易内存泄漏。

替代方案：

```java
// 推荐：try-with-resources
public class Resource implements AutoCloseable {
    @Override public void close() { /* 释放 */ }
}

try (Resource r = new Resource()) {
    // 使用资源
}

// 或 Cleaner（JDK 9+）
public class Resource implements AutoCloseable {
    private static final Cleaner cleaner = Cleaner.create();
    private final Cleaner.Cleanable cleanable;

    public Resource() {
        this.cleanable = cleaner.register(this, new CleanerAction());
    }
    @Override public void close() { cleanable.clean(); }
}
```

## 代码示例

自救演示：

```java
public class FinalizeEscape {
    private static FinalizeEscape SAVE_HOOK = null;

    public void isAlive() { System.out.println("alive"); }

    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        SAVE_HOOK = this; // 自救
    }

    public static void main(String[] args) throws InterruptedException {
        SAVE_HOOK = new FinalizeEscape();
        // 第一次：自救成功
        SAVE_HOOK = null;
        System.gc();
        Thread.sleep(500);
        if (SAVE_HOOK != null) SAVE_HOOK.isAlive();
        else System.out.println("dead");

        // 第二次：自救失败（finalize 只调用一次）
        SAVE_HOOK = null;
        System.gc();
        Thread.sleep(500);
        if (SAVE_HOOK != null) SAVE_HOOK.isAlive();
        else System.out.println("dead");
    }
}
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 内存泄漏排查 | jmap -histo 找实例最多的类 | 检查静态集合、ThreadLocal、监听器 |
| 类卸载 | 大量动态生成类后看元空间 | CGLIB、JSP 重编译可能不卸载类 |
| 资源管理 | try-with-resources | 优于 finalize |
| 监控 GC | jstat -gcutil 看 YGC/FGC | 配合 GC 日志定位 |

## 深挖追问

### 引用计数和可达性分析的区别？

引用计数：每个对象维护计数，加 1 减 1，为 0 即回收。优点是即时回收，缺点是无法解决循环引用、计数更新开销大。

可达性分析：从 GC Roots 出发遍历引用链，不可达即回收。优点是能解决循环引用，缺点是需要 STW 保证一致性。

### GC Roots 为什么不包括堆中对象？

GC Roots 是"程序仍然能直接访问"的对象入口。堆中对象如果不可达，本身就是垃圾；如果可达，必然能从 GC Roots 沿引用链找到。所以从 GC Roots 出发能覆盖所有存活对象，不需要把堆中对象作为 Roots。

### 类卸载为什么这么难？

因为 ClassLoader、类的实例、`Class` 对象、静态字段引用链中只要有一个仍可达，类就不能卸载。Tomcat 重新部署 WebApp 时，如果旧的 ClassLoader 没被回收（典型是 ThreadLocal、线程池、静态日志器持有），就会导致 PermGen / Metaspace 泄漏。

### finalize() 还能用吗？

能用但不推荐。JDK 9+ 标记 Deprecated，JDK 18+ 标记 forRemoval。新代码不要用，老代码改造时优先迁移到 `try-with-resources` 或 `Cleaner`。`System.gc()` 触发 finalize 的时机不确定，依赖它做资源释放很容易泄漏。

## 易错点

- 把"引用计数"当作 JVM 用的算法，实际是 Python 等用，JVM 用可达性分析。
- 以为堆中对象也是 GC Roots，只有栈、静态字段、常量、JNI 等是。
- 以为 `System.gc()` 立即触发 GC，实际只是建议。
- 在 finalize() 中关闭资源，导致延迟或泄漏。
- 以为类一旦不可用就立即卸载，JVM 只"允许"卸载，不一定立即执行。

## 总结

判断垃圾是 GC 的第一步。JVM 用可达性分析算法，从 GC Roots 出发遍历引用链，不可达的对象是垃圾。GC Roots 包括栈帧、静态字段、常量、JNI、JVM 内部引用、锁对象。方法区也会回收废弃常量和满足三条件的无用类。finalize() 已废弃，推荐 try-with-resources 或 Cleaner。理解 GC Roots 是排查内存泄漏的关键：静态集合、ThreadLocal、监听器是最常见的"隐性引用"。
