# 垃圾回收什么时候会触发

## 核心概念

GC 触发时机是 GC 调优的核心问题。不同 GC 类型（Minor、Major、Mixed、Full）有不同的触发条件，理解触发条件才能在 GC 频繁或 Full GC 异常时定位根因。

简单总结：Minor GC 由 Eden 满触发，Full GC 由老年代/元空间不足、System.gc()、空间分配担保失败等触发，Mixed GC 由 G1 在并发标记周期后触发。

```text
Eden 满 ──────────→ Minor GC
老年代满 ─────────→ Full GC
元空间满 ─────────→ Full GC
System.gc() ──────→ Full GC（建议）
分配担保失败 ─────→ Full GC
CMS Concurrent Mode Failure → Full GC
G1 IHOP 阈值 ────→ 并发标记周期 → Mixed GC
```

## 标准回答

GC 触发分两类：Minor GC 由 Eden 区满触发；Full GC 由多种条件触发。Mixed GC 是 G1 特有，由并发标记周期后触发。理解每种触发条件能帮助排查"为什么 GC 这么频繁"或"为什么突然 Full GC"。

要点：

1. **Minor GC**：Eden 区满即触发。
2. **Full GC**：老年代不足、元空间不足、System.gc()、分配担保失败、CMS Concurrent Mode Failure。
3. **Mixed GC**：G1 在并发标记周期完成后，老年代分区加入 CSet 触发。
4. **System.gc()**：建议而非强制，可用 `-XX:+DisableExplicitGC` 禁用。
5. **规避**：避免大对象、调大新生代、降低 IHOP、禁止 System.gc()。

## Minor GC（Young GC）触发

**唯一触发条件**：Eden 区空间不足以分配新对象。

- 新对象优先在 Eden 分配。
- Eden 满 → 触发 Minor GC。
- 存活对象复制到 Survivor，年龄 +1。
- 年龄达到阈值的对象晋升老年代。

注意：Minor GC 不一定回收新生代所有对象，只回收不可达的。如果大部分对象存活（异常情况），Survivor 装不下，触发空间分配担保。

## Full GC 触发条件

Full GC 回收整堆 + 方法区，停顿时间长，应该尽量避免。常见触发条件：

### 1. System.gc()

应用代码显式调用 `System.gc()` 或 `Runtime.getRuntime().gc()`。

- 只是建议，JVM 可以选择忽略。
- 默认触发 Full GC。
- 可用 `-XX:+DisableExplicitGC` 禁用，避免应用代码意外触发。

```java
System.gc(); // 建议触发 Full GC，但 -XX:+DisableExplicitGC 后被忽略
```

### 2. 老年代空间不足

老年代空间不足以容纳新晋升的对象。常见原因：

- 大对象直接进老年代（超过 `-XX:PretenureSizeThreshold`）。
- 长期存活对象晋升老年代。
- 内存泄漏导致老年代持续增长。

应对：

- 增大堆 `-Xmx`。
- 避免大对象。
- 调大 `-XX:MaxTenuringThreshold` 让对象在新生代多存活。

### 3. 元空间不足

JDK 8+ 元空间（本地内存）不足触发 Full GC。

- 大量动态生成类（CGLIB、JSP、Groovy）。
- 类加载器泄漏。

应对：

- 增大 `-XX:MaxMetaspaceSize`。
- 排查动态生成类的源头。
- 检查类加载器是否泄漏。

### 4. 空间分配担保失败

Minor GC 前检查老年代是否容纳新生代所有对象（最坏情况）：

- 不允许冒险 → 直接 Full GC。
- 允许冒险但老年代小于历次晋升平均大小 → Full GC。
- 允许冒险且老年代足够 → Minor GC，但 Survivor 装不下时老年代担保，仍失败则 Full GC。

`-XX:-HandlePromotionFailure` 控制是否允许冒险（JDK 6 后默认允许）。

### 5. CMS Concurrent Mode Failure

CMS 在并发清除阶段，老年代空间不足以容纳新晋升对象。

- 浮动垃圾过多。
- CMS 触发阈值过高（默认 92%）。

应对：

- 降低 `-XX:CMSInitiatingOccupancyFraction=70`。
- 增大老年代。

### 6. JDK 7 及以前永久代不足

JDK 7 及以前，永久代存放类元信息，空间不足触发 Full GC。JDK 8+ 移除永久代，元空间在本地内存，此问题不再常见。

## G1 的 Mixed GC 触发

G1 的 GC 触发更复杂：

1. **Young GC**：Eden 满触发，只回收新生代 Region。
2. **并发标记周期**：老年代占用超过 IHOP（默认 45%）触发，标记存活对象。
3. **Mixed GC**：并发标记周期后，在接下来的几次 Young GC 中加入老年代 Region，进行混合回收。
4. **Full GC**：转移失败（Evacuation Failure）时触发，单线程 Serial Full GC（JDK 10+ 并行），性能差。

G1 参数：

- `-XX:InitiatingHeapOccupancyPercent=45`：触发并发标记的堆占用阈值。
- `-XX:G1MixedGCLiveThresholdPercent=85`：Mixed GC 准入存活率。
- `-XX:G1MixedGCCountTarget=8`：Mixed GC 次数目标。

## ZGC 的触发

ZGC 触发机制：

| 触发方式 | 说明 |
|---------|------|
| Allocation Stall | 堆满阻塞分配，应避免 |
| Allocation Rate | 自适应算法，主流方式 |
| Timer | 固定间隔，应对流量突增 |
| Proactive | ZGC 主动触发 |
| Warmup | 启动预热 |
| System.gc() | 外部触发 |
| Metadata GC Threshold | 元空间不足 |

## 代码示例

观察 GC 触发：

```bash
# JDK 8
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCCause

# JDK 9+
-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=50M
```

GC 日志中的关键字：

```text
GC (Allocation Failure)        # Eden 满，触发 Minor GC
GC (System.gc())               # 显式触发
GC (Metadata GC Threshold)     # 元空间不足
GC (CMS Final Remark)          # CMS 重新标记
Full GC (Ergonomics)           # 自适应触发
Full GC (Allocation Failure)   # 分配失败
```

## 实战场景

| 场景 | 原因 | 应对 |
|------|------|------|
| Minor GC 频繁 | Eden 太小 | 增大 `-Xmn` 或 `-XX:NewRatio` |
| Full GC 频繁（老年代） | 内存泄漏/堆太小 | dump 分析，增大堆 |
| Full GC 频繁（元空间） | 动态生成类多 | 增大 `-XX:MaxMetaspaceSize` |
| Full GC 频繁（System.gc） | 代码显式调用 | `-XX:+DisableExplicitGC` |
| CMS Concurrent Mode Failure | 浮动垃圾多 | 降低 CMSInitiatingOccupancyFraction |

## 深挖追问

### System.gc() 一定会触发 Full GC 吗？

不一定。`System.gc()` 只是建议，JVM 可以选择忽略。默认行为是触发 Full GC。可以用 `-XX:+DisableExplicitGC` 完全禁用，或用 `-XX:+ExplicitGCInvokesConcurrent` 改为并发 GC（CMS/G1 模式下）。

### 为什么不建议调大 MaxTenuringThreshold？

调大 `MaxTenuringThreshold` 让对象在新生代多存活，理论上减少老年代压力。但 Survivor 空间有限，对象在 Survivor 间反复复制有开销。如果 Survivor 装不下，触发空间分配担保，反而进老年代。一般保持默认 15 即可。

### G1 的 Full GC 是单线程吗？

JDK 10 之前是单线程 Serial Full GC，性能极差。JDK 10（[JEP 307](https://openjdk.org/jeps/307)）改为并行。但 Full GC 仍是要避免的，因为它表示 G1 无法在并发模式下跟上分配速度。

### 为什么调 -Xms 和 -Xmx 一样能减少 Full GC？

堆动态扩缩容会触发 Full GC：扩容时如果申请内存失败，触发 Full GC 整理；缩容后再扩容又要重复。固定堆大小避免这些开销，配合 `-XX:+AlwaysPreTouch` 启动时预分配物理内存，减少运行时抖动。

## 易错点

- 把 Minor GC 当作"小 GC"、Full GC 当作"大 GC"，实际是回收区域不同。
- 以为 System.gc() 立即触发，只是建议，且可禁用。
- 把 Major GC 和 Full GC 等同，Major GC 一般只回收老年代，Full GC 回收整堆。
- 以为 G1 不会 Full GC，转移失败仍会触发。
- 忽略空间分配担保，导致 Survivor 装不下时频繁 Full GC。

## 总结

GC 触发时机是 GC 调优的核心。Minor GC 由 Eden 满触发，Full GC 由老年代/元空间不足、System.gc()、分配担保失败、CMS Concurrent Mode Failure 等触发，G1 的 Mixed GC 由并发标记周期后触发。理解每种触发条件能帮助排查"为什么 GC 频繁"或"为什么突然 Full GC"。生产环境建议 `-XX:+DisableExplicitGC`、`-Xms=-Xmx`、合理 IHOP，避免意外 Full GC。
