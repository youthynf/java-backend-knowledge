# 内存泄漏和内存溢出怎么区分

## 核心概念

内存泄漏（Memory Leak）和内存溢出（OutOfMemoryError，OOM）是两个相关但不同的概念：

- **内存泄漏**：对象不再使用但仍被引用，无法被 GC 回收，可用内存逐渐减少。
- **内存溢出**：JVM 申请内存时找不到足够空间，抛 `OutOfMemoryError`。

内存泄漏是"原因"，内存溢出是"结果"。长期内存泄漏最终会导致内存溢出。但内存溢出不一定由泄漏引起，也可能是堆确实不够大。

```text
内存泄漏（隐性）           内存溢出（显性）
─────────────────          ─────────────────
对象不被回收               JVM 抛 OOM
堆占用持续增长             程序崩溃
可用内存逐渐减少           堆/元空间/直接内存等不足
原因：隐性引用             原因：泄漏或堆太小
排查：dump + MAT           排查：先看 OOM 类型
```

## 标准回答

内存泄漏是"对象不再使用但仍被引用无法回收"，内存溢出是"JVM 申请内存失败抛 OOM"。泄漏是原因，溢出是结果。判断方法：多次 jmap -histo 对比，某类对象持续增长且不被回收，是泄漏；堆确实不够大，是溢出。常见泄漏：静态集合、ThreadLocal、监听器、未关闭资源。常见溢出：堆 OOM、元空间 OOM、直接内存 OOM、GC overhead。

要点：

1. **内存泄漏**：对象不可达但被隐性引用阻止回收。
2. **内存溢出**：JVM 申请内存失败。
3. **关系**：泄漏长期积累导致溢出，但溢出不一定因泄漏。
4. **判断方法**：多次 dump 对比对象数量趋势。
5. **常见泄漏源**：静态集合、ThreadLocal、监听器、未关闭资源。
6. **常见 OOM 类型**：Java heap space、Metaspace、Direct buffer memory、GC overhead。

## 内存泄漏

对象不再使用但仍被引用，GC 无法回收。常见原因：

### 1. 静态集合泄漏

静态集合持有对象引用，永远不移除，持续增长。

```java
// 错误：静态 Map 只增不减
private static Map<String, byte[]> cache = new HashMap<>();
public void put(String key, byte[] data) {
    cache.put(key, data); // 永远不移除
}

// 正确：用 Caffeine 等带容量上限的缓存
private static Cache<String, byte[]> cache = Caffeine.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(30, TimeUnit.MINUTES)
    .build();
```

### 2. ThreadLocal 泄漏

线程池中 ThreadLocal 用完不 `remove()`，线程被复用，ThreadLocal 中的对象不会被 GC。

```java
// 错误
private static ThreadLocal<User> userTl = new ThreadLocal<>();
public void handle(Request req) {
    userTl.set(req.getUser());
    // 忘记 remove()
}

// 正确
public void handle(Request req) {
    try {
        userTl.set(req.getUser());
        // 业务处理
    } finally {
        userTl.remove();
    }
}
```

### 3. 监听器泄漏

注册监听器但未取消，被监听对象一直引用监听器，监听器引用的资源无法回收。

```java
// 错误：注册后不取消
button.addActionListener(listener);

// 正确
button.addActionListener(listener);
// 不用时
button.removeActionListener(listener);
```

### 4. 资源未关闭

文件、数据库连接、网络流等未关闭，底层资源泄漏。

```java
// 错误
public void read(String file) {
    InputStream is = new FileInputStream(file);
    // 异常时不关闭
}

// 正确：try-with-resources
public void read(String file) {
    try (InputStream is = new FileInputStream(file)) {
        // 使用
    }
}
```

### 5. 内部类持有外部类

非静态内部类隐式持有外部类引用，外部类无法回收。

```java
// 错误：非静态内部类
class Outer {
    class Inner { } // Inner 持有 Outer.this
}

// 正确：静态内部类
class Outer {
    static class Inner { } // 不持有 Outer.this
}
```

## 内存溢出

JVM 申请内存时找不到足够空间，抛 `OutOfMemoryError`。常见类型：

| OOM 类型 | 原因 | 排查方向 |
|---------|------|---------|
| `Java heap space` | 堆内存不足 | 内存泄漏 / 对象过多 |
| `Metaspace` | 元空间不足 | 大量动态生成类 |
| `GC overhead limit exceeded` | GC 回收比例过低 | 堆太小 / 内存泄漏 |
| `Direct buffer memory` | 直接内存不足 | NIO 堆外内存泄漏 |
| `Unable to create new native thread` | 无法创建新线程 | 线程数过多 / OS 限制 |
| `Out of swap space` | 交换空间不足 | 物理内存不足 |

### 堆 OOM

```java
List<byte[]> list = new ArrayList<>();
while (true) list.add(new byte[1024 * 1024]); // 不断分配 1MB
// OutOfMemoryError: Java heap space
```

### 元空间 OOM

大量动态生成类（CGLIB、JSP、Groovy）：

```java
while (true) {
    Enhancer enhancer = new Enhancer();
    enhancer.setSuperclass(MyClass.class);
    enhancer.setCallback((MethodInterceptor) (o, m, a, p) -> p.invokeSuper(o, a));
    enhancer.create();
}
// OutOfMemoryError: Metaspace
```

### 直接内存 OOM

NIO DirectByteBuffer 泄漏：

```java
List<ByteBuffer> buffers = new ArrayList<>();
while (true) buffers.add(ByteBuffer.allocateDirect(1024 * 1024));
// OutOfMemoryError: Direct buffer memory
```

### GC overhead limit exceeded

GC 占用 98% 以上时间但回收不到 2% 堆，持续 5 次：

```text
堆满 → Full GC → 回收少 → 堆又满 → Full GC → ...
```

通常是内存泄漏导致 GC 无法释放足够空间。

## 区分泄漏和溢出

| 维度 | 内存泄漏 | 内存溢出 |
|------|---------|---------|
| 表现 | 堆占用持续增长 | 程序崩溃，OOM |
| 可观测 | dump 分析找到"保留链" | 日志有 OOM 类型 |
| 修复 | 找到泄漏源，断开引用 | 增大堆 / 减少对象 |
| 长期 | 不修复最终导致溢出 | 修复后立即恢复 |

### 判断方法

多次 jmap -histo 对比，看对象数量趋势：

```bash
# 间隔 1 分钟执行 3 次
jmap -histo <pid> | head -20 > histo1.txt
sleep 60
jmap -histo <pid> | head -20 > histo2.txt
sleep 60
jmap -histo <pid> | head -20 > histo3.txt

# 对比三次输出
# 如果某类对象持续增长 → 内存泄漏
# 如果对象数量稳定但堆仍满 → 堆太小，是溢出
```

## 排查流程

```text
1. 看日志确认 OOM 类型
   ├─ Java heap space → 堆问题
   ├─ Metaspace → 类元数据问题
   ├─ Direct buffer memory → 直接内存问题
   └─ GC overhead → 持续 Full GC

2. dump 堆（启动时配置自动 dump）
   -XX:+HeapDumpOnOutOfMemoryError
   -XX:HeapDumpPath=/var/log/heap.hprof

3. 用 MAT 分析
   - Leak Suspects Report
   - Dominator Tree
   - Path to GC Roots
   - Histogram

4. 找到泄漏源
   - 哪个对象保留最多内存
   - GC Roots 引用链
   - 静态集合 / ThreadLocal / 监听器

5. 修复代码
   - 断开引用
   - 添加 remove / close
   - 用带容量上限的缓存
```

## 代码示例

完整排查示例：

```bash
# 1. 启动时配置自动 dump
java -Xms1g -Xmx1g \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/heap.hprof \
  -Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags \
  -jar app.jar

# 2. 实时监控
jstat -gcutil <pid> 1000   # 每秒 GC 统计
jmap -histo <pid> | head -20  # 对象统计

# 3. 手动 dump
jcmd <pid> GC.heap_dump /tmp/heap.hprof
# 或
jmap -dump:format=b,file=/tmp/heap.hprof <pid>

# 4. 用 MAT 分析
# 打开 heap.hprof → Leak Suspects Report
```

## 实战场景

| 场景 | 类型 | 排查 |
|------|------|------|
| 堆持续增长，Full GC 后不降 | 泄漏 | dump + MAT |
| 突然 OOM，业务量突增 | 溢出 | 增大堆，限流 |
| 元空间持续增长 | 泄漏 | 检查 CGLIB / JSP |
| 直接内存增长 | 泄漏 | 检查 NIO / Netty |
| 线程数过多 | 溢出 | 调整线程池 |

## 深挖追问

### 内存泄漏一定会导致 OOM 吗？

不一定。如果堆足够大、泄漏速率低，可能很久才 OOM。但泄漏是"持续增长"的，最终必然导致 OOM。生产环境监控要关注堆占用趋势，发现持续增长及时处理。

### 为什么 ThreadLocal 用完要 remove？

ThreadLocal 的 `ThreadLocalMap` 中 Entry 继承 WeakReference，key 是弱引用。ThreadLocal 对象本身被回收后，key 变 null。但 value 仍是强引用，存在 `ThreadLocalMap.Entry` 中。线程池中线程被复用，ThreadLocalMap 也复用，value 永远不被回收，导致泄漏。`remove()` 显式清除 entry，避免泄漏。

### GC overhead limit exceeded 和 Java heap space 有什么区别？

`Java heap space`：堆确实满了，无法分配新对象。
`GC overhead limit exceeded`：GC 占用 98% 以上时间但回收不到 2%，持续 5 次。是"GC 太频繁但效果差"的告警，通常预示即将 OOM。

### 怎么判断是泄漏还是堆太小？

多次 dump 对比：

- 对象数量持续增长且不被回收 → 泄漏。
- 对象数量稳定但堆仍满 → 堆太小（业务量超过堆容量）。
- Full GC 后老年代使用率仍高 → 泄漏概率大。

## 易错点

- 把"内存泄漏"和"内存溢出"当作同义词，前者是原因，后者是结果。
- 以为 OOM 一定是泄漏，也可能是堆太小或业务量突增。
- 只看对象数量不看 Retained Size，可能错过泄漏。
- 用 `System.gc()` 试图解决泄漏，治标不治本。
- 排查时直接重启丢失现场，应该先 dump。

## 总结

内存泄漏是"对象不被回收"，内存溢出是"JVM 申请内存失败"。泄漏是原因，溢出是结果，长期泄漏导致溢出。常见泄漏源：静态集合、ThreadLocal、监听器、未关闭资源、内部类持有外部类。常见 OOM 类型：Java heap space、Metaspace、Direct buffer memory、GC overhead。排查流程：看 OOM 类型 → dump → MAT 分析 → 找泄漏源 → 修复代码。生产环境一定要配置 `-XX:+HeapDumpOnOutOfMemoryError` 自动 dump。
