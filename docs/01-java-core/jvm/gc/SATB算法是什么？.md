# SATB 算法是什么

## 核心概念

SATB（Snapshot-At-The-Beginning，起始快照）是 G1 垃圾收集器用于并发标记阶段的核心算法。它通过保留"并发标记开始时的对象图快照"，确保并发标记过程中即使引用关系变化，也不会漏标存活对象。

SATB 解决的问题是并发标记的漏标：标记线程和用户线程并发执行时，如果灰色对象引用白色对象的引用消失，且黑色对象新引用该白色对象，会导致白色对象被错误回收。SATB 通过"在引用消失前记录旧值"避免漏标。

```text
并发标记开始时建立快照（所有可达对象视为存活）
   ↓
并发标记中：
  - 引用消失前，写前屏障把旧引用加入 SATB 缓冲区
  - 缓冲区满 → 提交到全局队列
   ↓
最终标记阶段（STW）：
  - 处理 SATB 缓冲区残留
  - 强制标记为存活
   ↓
副作用：浮动垃圾（快照存活但实际已死的对象本次不回收）
```

## 标准回答

SATB 是 G1 并发标记的核心算法，在标记开始时建立对象图快照，确保快照中存活的对象不会被漏标。它通过写前屏障捕获引用消失前的旧值，加入 SATB 缓冲区，最终标记阶段强制标记为存活。代价是产生浮动垃圾：快照中存活但实际已死的对象本次不回收，要等下次 GC。

要点：

1. **核心思想**：保留标记开始时的快照，确保快照中存活对象不被误回收。
2. **写前屏障**：引用消失前记录旧值到 SATB 缓冲区。
3. **缓冲区机制**：每线程一个 256 条目的缓冲区，满后提交全局队列。
4. **最终标记处理**：STW 阶段处理 SATB 缓冲区，强制标记存活。
5. **代价**：浮动垃圾——快照中存活但实际已死的对象本次不回收。

## 核心问题：并发标记漏标

并发标记中，标记线程和用户线程并发执行，引用关系会变化。漏标发生的两个条件：

1. **黑色对象指向白色对象**（新引用）。
2. **灰色对象指向白色对象的引用消失**（旧引用断开）。

如果两个条件同时满足，白色对象会漏标，被错误回收。

## 解决思路

### 增量更新（CMS 用）

关注"引用增加"：黑色对象新引用白色对象时，把黑色重新标记为灰色，重新扫描。

### SATB（G1 用）

关注"引用消失"：灰色对象引用白色对象的引用消失前，把旧引用记录到 SATB 缓冲区。最终标记阶段处理缓冲区，强制标记为存活。

G1 选 SATB 的原因：RSet 已经记录了跨 Region 引用，处理 SATB 缓冲区时可以直接通过 RSet 找到引用源，不需要全堆扫描，效率高。CMS 没有 RSet，SATB 没有优势。

## 实现机制

### 三色标记基础

SATB 基于三色标记：

- 白色 White：未标记。
- 灰色 Gray：已标记，引用未扫完。
- 黑色 Black：已标记，引用全扫完。

SATB 保证"标记开始时存活的对象"最终都被标记为黑色或灰色，不被错误回收。

### 写前屏障

引用更新前插入，记录旧值：

```text
def pre_write_barrier(src, field, new_value):
    if 并发标记阶段:
        old_value = src.field
        if old_value != null:
            satb_buffer.add(old_value)   # 旧引用入队
    src.field = new_value
```

为什么记录旧值？如果旧引用消失后，黑色对象又新引用该白色对象，标记线程可能漏标。把旧值入队后，最终标记阶段强制标记为存活，避免漏标。

### SATB 缓冲区

- 每线程一个固定大小的缓冲区（默认 256 条目）。
- 缓冲区满 → 提交到全局队列，分配新缓冲区。
- 并发标记线程定期检查全局队列，处理缓冲区内容。

### 工作流程

```text
1. 初始标记（STW）：
   - 标记 GC Roots 直接引用的对象（变灰）
   - 此时建立快照

2. 并发标记：
   - 与用户线程并发
   - 从灰色对象遍历引用链
   - 写前屏障捕获引用消失，旧值入 SATB 缓冲区
   - 新引用通过常规标记流程处理

3. 最终标记（STW）：
   - 处理 SATB 缓冲区剩余对象
   - 把缓冲区对象标记为灰色
   - 扫描其引用字段
   - 更新为黑色

4. 清理：
   - 回收未标记（白色）的对象
   - 释放完全空闲的 Region
```

## SATB 优缺点

### 优点

- 避免漏标：快照中存活对象一定被标记为存活。
- 减少重新标记开销：相比增量更新，SATB 不需要重新扫描黑色对象。
- 支持并发标记实现，减少 STW 停顿。
- 与 RSet 配合天然：处理 SATB 缓冲区时通过 RSet 找引用源。

### 缺点

- **浮动垃圾**：快照中存活但实际已死的对象本次不回收，等下次 GC。
- **写屏障开销**：每次引用更新都触发写前屏障，影响吞吐量。
- **缓冲区内存**：SATB 缓冲区占用额外内存。

## 浮动垃圾

SATB 的核心代价是浮动垃圾。引用消失前的对象被记入 SATB 缓冲区，最终标记阶段强制标记为存活。即使该对象实际已经死亡，本次 GC 也不回收，要等下次 GC。

```text
并发标记开始：obj 还存活
并发标记中：obj.field = null（引用断开）
            但 obj 之前引用的对象已入 SATB 缓冲区
最终标记：缓冲区对象强制标记为存活
本次 GC：缓冲区对象不被回收（即使实际已死）
下次 GC：重新标记，确认是垃圾，回收
```

浮动垃圾不会导致内存泄漏，只是延迟回收。但如果业务对象分配速率高，浮动垃圾可能积累导致堆占用过高，需要 G1 提前触发下一次 GC。

## 代码示例

理解 SATB 行为：

```java
public class SatbDemo {
    Object field1;  // 引用 obj1
    Object field2;  // 引用 obj2

    public void test() {
        // 并发标记阶段：
        // 写前屏障触发，记录 obj1（旧值），加入 SATB 缓冲区
        field1 = field2;     // field1 不再引用 obj1

        // 即使 obj1 实际已不可达，本次 GC 也会被强制标记为存活
        // 下次 GC 才会被回收（浮动垃圾）
    }
}
```

## 实战场景

| 场景 | 关注点 | 调优 |
|------|--------|------|
| G1 浮动垃圾多 | 堆占用增长快 | 适当降低 IHOP，提前触发并发标记 |
| SATB 缓冲区处理慢 | 最终标记耗时长 | 增大 `-XX:ConcGCThreads` |
| 写屏障开销大 | 吞吐量低 | 评估是否用 Parallel GC |

## 深挖追问

### SATB 为什么会产生浮动垃圾？

SATB 在引用消失前把旧值入队，最终标记阶段强制标记为存活。即使该对象实际已经死亡（被覆盖前的引用已经断开），本次 GC 也不回收。要等下次 GC 重新从 GC Roots 出发可达性分析，确认是垃圾后才能回收。

### SATB 和增量更新的区别？

| 维度 | SATB | 增量更新 |
|------|------|---------|
| 关注 | 引用消失 | 引用增加 |
| 处理方式 | 旧值入队，最终标记强制标记 | 黑色重新变灰，重新扫描 |
| 浮动垃圾 | 有 | 无 |
| 适用 | G1（有 RSet） | CMS（无 RSet） |

### SATB 缓冲区满了怎么办？

缓冲区满后，线程把它提交到全局队列，并分配新缓冲区继续工作。并发标记线程定期检查全局队列，处理缓冲区中的引用。如果应用线程产生 SATB 记录的速度超过标记线程处理速度，全局队列会堆积，最终标记阶段需要更长时间处理。

### G1 为什么用 SATB 不用增量更新？

SATB 配合 RSet 浑然天成：处理 SATB 缓冲区时，可以通过 RSet 直接找到引用源，不需要全堆扫描。增量更新把黑色重新变灰后，重新扫描时需要遍历该对象的所有引用字段，效率低。G1 已有 RSet 基础设施，SATB 自然成为选择。

## 易错点

- 把 SATB 当作"完整快照算法"，实际只记录"引用消失"的对象。
- 以为 SATB 缓冲区的对象会被立即回收，实际被强制标记为存活（避免漏标）。
- 把 SATB 和 CMS 的增量更新混淆，SATB 关注引用消失，增量更新关注引用增加。
- 以为浮动垃圾会导致内存泄漏，实际只是延迟回收。
- 以为 SATB 没有 STW，初始标记和最终标记都 STW，只是并发标记不 STW。

## 总结

SATB 是 G1 并发标记的核心算法，通过"引用消失前记录旧值"避免漏标。代价是浮动垃圾——快照中存活但实际已死的对象本次不回收。SATB 配合 RSet 实现高效处理，是 G1 区别于 CMS 的关键设计。理解 SATB 是理解 G1 并发标记、浮动垃圾、写前屏障的基础。
