# G1 垃圾收集器基本原理

## 核心概念

G1（Garbage First）是 JDK 7 引入、JDK 9 起作为默认的垃圾收集器。它通过把堆划分为多个 Region，按回收价值（垃圾占比）排序，优先回收垃圾最多的 Region，实现"可预测的停顿时间"。G1 解决了 CMS 的内存碎片、停顿不可控、不适合大堆等问题，是当前主流服务端应用的默认选择。

G1 的核心创新：分区（Region）+ 可预测停顿（MaxGCPauseMillis）+ 混合回收（Mixed GC）+ SATB 并发标记。它仍保留分代概念，但物理上不再有连续的新生代和老年代边界，每个 Region 可动态切换为 Eden、Survivor、Old、Humongous。

```text
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ H  │ E  │ O  │ E  │ O  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ E  │ E  │ O  │ S  │ E  │ O  │ E  │
└────┴────┴────┴────┴────┴────┴────┴────┘
E=Eden  S=Survivor  O=Old  H=Humongous
Region 大小 1~32MB，默认约 2048 个
```

## 标准回答

G1 是面向服务端的分区化、可预测停顿的垃圾收集器。一句话结论：把堆划分为多个 Region，每个 Region 可动态切换为 Eden/Survivor/Old/Humongous，按回收价值排序优先回收垃圾最多的（Garbage First），通过 MaxGCPauseMillis 控制停顿目标，通过 SATB 实现并发标记。

要点：

1. **分区 Region**：1~32MB，默认 2048 个，每个可动态切换代。
2. **Humongous**：超过 Region 50% 的对象直接进 Humongous Region。
3. **RSet**：记录跨 Region 引用，避免全堆扫描。
4. **Card Table**：Region 内细分 Card（512B），用 Card 标记脏。
5. **SATB**：并发标记算法，处理漏标。
6. **CSet**：本次回收的 Region 集合。
7. **四个阶段**：初始标记（STW）→ 并发标记 → 最终标记（STW）→ 筛选回收（STW）。

## G1 内存模型

### Region 分区

G1 把堆划分为多个大小相等的 Region，每个 Region 在不同时期可以是 Eden、Survivor、Old 或 Humongous。`-XX:G1HeapRegionSize` 指定大小（1~32MB，2 的幂），默认按堆大小自动计算，目标 2048 个。

```bash
-XX:G1HeapRegionSize=4m   # 显式指定
```

### Card 卡片

每个 Region 逻辑上划分为多个 Card（默认 512 字节）。Card Table 是字节数组，每个字节标记对应 Card 是否被修改过（脏）。当对象引用变化时，写屏障把对应 Card 标记为脏，Refine 线程消费脏 Card 更新 RSet。

### Humongous Region

大对象（超过 Region 大小的 50%）直接在 Humongous Region 分配，可能跨多个连续 Region：

- `StartsHumongous`：第一个 Region。
- `ContinuesHumongous`：后续连续 Region。

大对象回收成本高，G1 在并发标记阶段后可单独回收无引用的 Humongous Region。

## 已记忆集合 RSet

RSet（Remembered Set）记录"哪些其他 Region 引用了当前 Region 的对象"，是 Point-In 方向的反向索引。回收某 Region 时，通过 RSet 快速找到引用方，避免全堆扫描。

### 三种粒度

RSet 在 PRT（Per Region Table）中按引用密度分三种模式：

| 模式 | 适用场景 | 记录方式 |
|------|---------|---------|
| 稀少 Sparse | 极少其他 Region 引用 | 哈希表记录 Card 索引 |
| 细粒度 Fine | 几十到几百个 Card 引用 | 位图，每个 bit 对应一个 Card |
| 粗粒度 Coarse | 大量引用 | 位图，每个 bit 对应一个 Region |

### 不需要记录的引用

- 同 Region 内引用（扫描时自然覆盖）。
- 新生代 → 新生代（YGC 全量扫描新生代）。
- 新生代 → 老年代（YGC 不扫描老年代，Mixed GC 用新生代作根）。

### 需要记录的引用

- 老年代 → 老年代（Mixed GC 部分回收老年代）。
- 老年代 → 新生代（YGC 时需要找老年代对新生代的引用）。

### RSet 维护

- 写后屏障（Post-Write Barrier）捕获引用变化，把对应 Card 加入 Dirty Card Queue（DCQ）。
- Refine 线程消费 DCQ，更新 RSet。
- 默认 Refine 线程数等于 `-XX:ParallelGCThreads`。

## 收集集合 CSet

CSet（Collection Set）是本次 GC 暂停要回收的 Region 集合。所有 CSet 内 Region 的存活对象都会被转移到其他 Region，原 Region 释放。

- **YGC**：CSet 只含新生代 Region。
- **Mixed GC**：CSet 含所有新生代 + 部分老年代（按回收价值排序）。
- **Full GC**：单线程 Serial Full GC，转移失败时触发，要避免。

### Mixed GC 准入条件

- 老年代分区中存活对象占比 < `-XX:G1MixedGCLiveThresholdPercent`（默认 85%）。
- 每次最多回收老年代分区数 ≤ 老年代总分区数 × `-XX:G1OldCSetRegionThresholdPercent`（默认 10%）。

## G1 的活动周期

### 1. 年轻代收集（Young GC）

Eden 满时触发，STW。

- 扫描 GC Roots 直接引用的新生代对象。
- 通过 RSet 扫描老年代引用新生代的对象。
- 复制 Eden + Survivor 存活对象到新 Survivor 或老年代。
- 清空原 Eden + Survivor Region，归还空闲列表。

### 2. 并发标记周期（Concurrent Marking Cycle）

老年代占用超过 IHOP（`-XX:InitiatingHeapOccupancyPercent`，默认 45%）触发。包含五个阶段：

| 阶段 | STW | 说明 |
|------|-----|------|
| 初始标记 Initial Mark | 是 | 标记 GC Roots 直接关联对象，借道 YGC |
| 根分区扫描 Root Region Scan | 否 | 扫描 Survivor 分区作为根 |
| 并发标记 Concurrent Mark | 否 | 多线程扫描分区，标记存活对象，处理 SATB 缓冲区 |
| 重新标记 Remark | 是 | 处理剩余 SATB 缓冲区，完成引用处理 |
| 清理 Cleanup | 部分 STW | 统计存活数据，识别空闲 Region，选 Mixed GC 候选 |

### 3. 混合收集周期（Mixed Collection Cycle）

并发标记周期后，G1 在接下来的几次 YGC 中加入老年代 Region，进行混合回收。

- 单次混合回收与 YGC 过程类似，只是 CSet 包含部分老年代 Region。
- 可能连续多次执行，直到回收收益低于阈值或达到 `-XX:G1MixedGCCountTarget`（默认 8）。

### 4. Full GC（避免）

转移失败（Evacuation Failure）时触发，单线程 Serial Full GC，性能极差。

触发条件：

- 年轻代转移时找不到空闲 Region。
- 老年代转移时找不到空闲 Region。
- Humongous 对象分配时找不到连续 Region。

避免方式：

- `-XX:G1ReservePercent=10` 保留空间（默认 10%，最大 50%）。
- 调大堆或调小 IHOP。

## SATB 与并发标记

G1 用 SATB（Snapshot-At-The-Beginning）算法解决并发标记的漏标问题。

- 标记开始时建立对象图快照。
- 写前屏障把被删除的旧引用加入 SATB 缓冲区。
- 最终标记阶段处理 SATB 缓冲区，强制标记为存活。
- 详见 [SATB 算法是什么](SATB算法是什么？.md)。

副作用：浮动垃圾。被 SATB 强制存活的"快照中存活但实际已死"的对象，本次 GC 不回收，下次再回收。

## G1 关键参数

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `-XX:+UseG1GC` | 启用 G1 | JDK 9+ 默认 |
| `-XX:MaxGCPauseMillis` | 目标最大停顿 | 200ms |
| `-XX:G1HeapRegionSize` | Region 大小 | 自动（1~32M） |
| `-XX:InitiatingHeapOccupancyPercent` | 触发并发标记的堆占用 | 45 |
| `-XX:G1NewSizePercent` | 新生代最小比例 | 5 |
| `-XX:G1MaxNewSizePercent` | 新生代最大比例 | 60 |
| `-XX:G1MixedGCLiveThresholdPercent` | Mixed GC 准入存活率 | 85 |
| `-XX:G1MixedGCCountTarget` | Mixed GC 次数目标 | 8 |
| `-XX:G1ReservePercent` | 保留空间防晋升失败 | 10 |
| `-XX:ParallelGCThreads` | STW GC 线程数 | CPU 核数 60% |
| `-XX:ConcGCThreads` | 并发 GC 线程数 | ParallelGCThreads 的 1/4 |

## 推荐配置

```bash
# Spring Boot 微服务（JDK 11+）
-Xms4g -Xmx4g \
-XX:+UseG1GC \
-XX:MaxGCPauseMillis=100 \
-XX:InitiatingHeapOccupancyPercent=35 \
-XX:G1ReservePercent=15 \
-XX:+HeapDumpOnOutOfMemoryError \
-XX:HeapDumpPath=/var/log/heap.hprof \
-XX:+DisableExplicitGC \
-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags:filecount=10,filesize=50M
```

## 实战场景

| 场景 | 调优方向 | 参数 |
|------|---------|------|
| 偶发 500ms+ 停顿 | 调小 MaxGCPauseMillis | `-XX:MaxGCPauseMillis=100` |
| Full GC 频繁 | 增大 G1ReservePercent、降低 IHOP | `-XX:G1ReservePercent=15 -XX:InitiatingHeapOccupancyPercent=35` |
| Humongous 对象多 | 增大 Region 或避免大对象 | `-XX:G1HeapRegionSize=8m` |
| 大堆（16GB+） | 考虑 ZGC | `-XX:+UseZGC` |

## 深挖追问

### G1 的 "Garbage First" 是什么意思？

筛选回收阶段，G1 按每个 Region 的垃圾占比（回收价值）排序，在停顿时间允许的范围内优先回收垃圾最多的 Region，即 Garbage First。这种策略让 G1 在有限停顿时间内回收最多内存。

### G1 为什么用 SATB 不用增量更新？

SATB 配合 RSet 浑然天成：RSet 记录跨 Region 引用，SATB 把旧引用入队后，最终标记阶段处理时可以直接通过 RSet 找到引用源，不需要全堆扫描。增量更新把黑色变灰后还要重新扫描所有字段，效率低。

### G1 的 Full GC 为什么是单线程？

JDK 10 之前 G1 的 Full GC 退化为单线程 Serial Full GC，性能极差。JDK 10（[JEP 307](https://openjdk.org/jeps/307)）将其改为并行，性能大幅提升。但 Full GC 仍是要极力避免的，因为它表示 G1 无法在并发模式下跟上分配速度。

### G1 适合多大的堆？

官方建议 ≥ 6GB。堆太小（< 2GB）时 Region 数量少，G1 优势不明显，Parallel Scavenge 可能更好。堆超大（> 64GB）时 G1 的 RSet 维护开销变大，ZGC 更合适。

## 易错点

- 把 G1 当作"不分代"收集器，G1 仍逻辑分代，只是物理上 Region 化。
- 以为 MaxGCPauseMillis 越小越好，设太小会导致 GC 频繁，反而降低吞吐。
- 把 IHOP 设过高，导致并发标记启动太晚，引发 Full GC。
- 忽略 Humongous 对象，大对象直接进 Humongous Region 可能引发 Full GC。
- 以为 G1 没有 Full GC，转移失败仍会触发，且 JDK 10 前是单线程。

## 总结

G1 是 JDK 9+ 默认垃圾收集器，通过 Region 分区、SATB 并发标记、可预测停顿实现低延迟大堆回收。核心数据结构是 RSet（跨 Region 引用）和 Card Table（细粒度脏标记）。回收流程分年轻代收集、并发标记周期、混合收集周期，Full GC 是兜底要避免。理解 G1 的工作原理是 JDK 9+ 服务端调优的基础。
