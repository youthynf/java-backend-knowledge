# GC 只会对堆进行回收吗？

## 核心概念

GC 的主要回收对象是 Java 堆中的对象实例，但并不只关注堆。JVM 还可能对方法区/元空间中的无用类元数据、常量等进行清理。不同 JDK 版本和不同垃圾收集器在实现细节上会有差异。

常见区域：

- **Java 堆**：对象实例，是 GC 最主要的工作区域。
- **方法区/元空间**：类元数据、运行时常量池等，满足条件时也可能卸载类并回收相关元数据。
- **直接内存**：不在 Java 堆内，通常依赖引用对象和 Cleaner 等机制触发释放，排查时也要关注。

## 面试官想考什么

- 是否知道 GC 不等于“只回收堆”。
- 是否能说明类卸载的大致条件。
- 是否能区分堆内存、元空间和直接内存问题。

## 标准回答

GC 最主要回收 Java 堆，因为对象实例大多分配在堆上。除此之外，方法区/元空间也存在回收，例如废弃常量回收、无用类卸载等。类卸载通常需要满足该类的所有实例已不可达、加载该类的类加载器已不可达、该类对应的 `Class` 对象不可达等条件。

所以面试中不能简单回答“GC 只回收堆”。更准确的说法是：堆是 GC 的主要回收区域，方法区/元空间也可能被回收；直接内存不属于 Java 堆，但其释放也可能和 GC 触发有关。

## 深挖追问

### 元空间 OOM 和堆 OOM 有什么区别？

堆 OOM 通常是对象实例过多或泄漏；元空间 OOM 常见于动态生成类过多、类加载器泄漏、热部署不彻底等。

### 类卸载为什么比较难发生？

只要类加载器、类的实例、静态引用链或 `Class` 对象仍可达，类元数据就不能安全卸载。

### 直接内存怎么排查？

关注 `-XX:MaxDirectMemorySize`、NIO/Netty Buffer 使用、Native Memory Tracking、进程 RSS 和堆占用是否不匹配。

## 实战场景/代码示例

```bash
jcmd <pid> GC.heap_info
jcmd <pid> VM.native_memory summary   # 需要提前开启 NMT 才完整
jcmd <pid> GC.class_histogram
```

典型现象：堆看起来不大，但进程 RSS 很高，可能要怀疑直接内存、线程栈、元空间或 native 内存。

## 易错点/总结

- GC 主要回收堆，但不是只回收堆。
- JDK 8 之后永久代被元空间替代，元空间默认使用本地内存。
- 看到 OOM 要先看具体错误类型：`Java heap space`、`Metaspace`、`Direct buffer memory` 的排查方向不同。
