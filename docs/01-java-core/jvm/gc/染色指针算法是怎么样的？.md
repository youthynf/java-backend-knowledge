# 染色指针算法是怎么样的

## 核心概念

染色指针（Colored Pointer）是 ZGC 和 Shenandoah 等现代低延迟垃圾收集器的核心技术。它把 GC 标记信息（颜色、转移状态等）直接编码到 64 位对象指针的高位，无需访问对象头即可知道对象状态。

染色指针解决的问题是：让 GC 在不访问对象、不 STW 的前提下，完成标记、转移、重定位。它把传统的"遍历对象图标记对象"变为"遍历引用图标记引用"，是 ZGC 实现 < 1ms 停顿的关键。

```text
传统 GC：标记信息存对象头 Mark Word
        访问对象 → 读 Mark Word → 知道颜色
        代价：必须访问堆中对象

染色指针：标记信息存指针本身
        访问指针 → 知道颜色
        代价：指针操作（寄存器级，快得多）
```

## 标准回答

染色指针把 GC 标记信息编码到 64 位对象指针的高 4 位，无需访问对象头即可知道对象状态。ZGC 用 Marked0、Marked1、Remapped、Finalizable 四个标志位，分别表示标记周期、重映射状态、finalize 处理。染色指针让 ZGC 实现并发标记和并发转移，STW 时间 < 1ms。配合读屏障实现指针自愈。

要点：

1. **基本原理**：64 位指针高 16 位通常为 0，用高 4 位存 GC 标记。
2. **四个标志位**：Marked0、Marked1、Remapped、Finalizable。
3. **内存寻址**：44 位地址空间，最大支持 16TB 堆。
4. **优势**：无需访问对象头，标记效率高；并发执行不需 STW。
5. **配合读屏障**：实现指针自愈，转移期间无需暂停应用。

## 背景：传统 GC 标记方案

在 ZGC 之前，主流 GC 把标记信息存放在对象头的 Mark Word 中：

```text
对象头 Mark Word（64 位）：
┌───────────────────────────────────────────┐
│ hash | age | biased | lock | reserved    │
└───────────────────────────────────────────┘
```

不同收集器的实现：

| 收集器 | 标记方案 | 代价 |
|--------|---------|------|
| Serial、PS、ParNew、CMS | 对象头 Mark Word | 必须访问堆对象 |
| G1、Shenandoah | BitMap（独立数据结构） | 占堆 1/64，需查 BitMap |
| ZGC | 染色指针（指针本身） | 寄存器操作，最快 |

对象头方案的局限：访问对象才能知道状态，GC 必须访问堆内存，开销大。BitMap 方案需要额外数据结构。染色指针直接用指针存储，最快。

## 染色指针的布局

64 位 JVM 中，对象指针 64 位。ZGC 用高 4 位作为标志位，剩余 44 位寻址（最大 16TB 堆）：

```text
ZGC 64 位指针布局：
┌─┬─┬─┬─┬──────────────────────────────────────┐
│F│R│M│M│        44 位对象地址（16TB）          │
│i│e│1│0│                                       │
│n│m│ │ │                                       │
│a│a│ │ │                                       │
│l│p│ │ │                                       │
└─┴─┴─┴─┴──────────────────────────────────────┘
  │ │ │ │
  │ │ │ └── Marked0：标记周期 1
  │ │ └── Marked1：标记周期 2
  │ └── Remapped：是否已重映射
  └── Finalizable：finalize 处理
```

四个标志位：

- **Marked0 / Marked1**：交替用于标记周期，区分两次 GC。第一次标记用 Marked0，第二次用 Marked1，依次交替。
- **Remapped**：表示对象已转移完成，指针指向新地址。
- **Finalizable**：用于 finalize 处理，标记需要执行 finalize 的对象。

## 三色标记的物理实现

染色指针是三色标记的物理实现：

| 颜色 | 染色指针状态 |
|------|------------|
| 白色 | 高 4 位全 0（未标记） |
| 灰色 | Marked0/1 置 1，对象在标记栈中 |
| 黑色 | Marked0/1 置 1，对象不在标记栈 |

GC 通过指针的标志位就能知道对象状态，无需访问对象头。

## 多重映射

ZGC 通过多重映射让同一物理内存映射到三个虚拟地址空间：

```text
虚拟地址空间：         物理内存：
Marked0  ────────┐
Marked1  ────────┼──→  同一物理内存
Remapped ────────┘
```

操作系统层面，三个虚拟地址指向同一物理内存。ZGC 通过切换"当前有效视图"实现 GC 周期切换：

- 标记阶段：当前视图为 Marked0（或 Marked1）。
- 转移阶段：当前视图为 Remapped。
- 应用线程通过读屏障检测指针视图与当前是否一致，不一致则修正。

## 工作流程

### 标记阶段

1. 初始时视图设为 Remapped。
2. 进入标记阶段，视图切到 Marked0。
3. GC 线程访问对象时，把对象指针视图从 Remapped 切到 Marked0。
4. 应用线程通过读屏障同样切换视图。
5. 标记结束后：Marked0 视图对象是存活的，Remapped 视图对象是垃圾。

### 转移阶段

1. 转移阶段视图切回 Remapped。
2. GC 线程把 Marked0 视图的对象转移到新 Region，指针视图设为 Remapped。
3. 应用线程通过读屏障访问转移中的对象时，自动修正指针（自愈）。
4. 转移完成后，旧地址的对象仍可通过多重映射访问，避免悬挂指针。

## 优势与挑战

### 优势

- **极低停顿**：亚毫秒级，且不随堆增长。
- **内存开销小**：元数据内联在指针中，不需要额外 BitMap。
- **并发执行效率高**：标记、转移都并发，无需 STW 更新引用。
- **大堆友好**：支持 16TB 堆，停顿时间不变。

### 挑战

- **硬件要求**：需要 64 位 JVM + 操作系统支持多重映射。
- **指针操作复杂**：JVM 实现复杂度增加。
- **调试难度**：指针含元数据，调试工具需特殊处理。
- **平台限制**：最初仅 Linux/x64，后扩展（macOS、Windows）。

## 与其他方案的对比

| 方案 | 标记位置 | 访问开销 | 内存开销 | 典型 GC |
|------|---------|---------|---------|---------|
| 对象头 Mark Word | 对象头 | 高（访问堆） | 0 | CMS、Serial |
| BitMap | 独立数据结构 | 中（查 BitMap） | 堆 1/64 | G1、Shenandoah |
| 染色指针 | 指针本身 | 低（寄存器） | 0 | ZGC |

## 代码示例

理解染色指针的视图切换：

```java
public class ColoredPointerDemo {
    Object field;

    public void test() {
        // 假设当前 GC 视图为 Marked0
        // obj 指针的视图为 Remapped（未标记）

        Object obj = this.field;  // 读屏障触发
        // 读屏障检查 obj 的指针视图：
        //   如果 Remapped != 当前视图 Marked0
        //   说明 obj 未标记，需要标记
        //   把 obj 指针视图切到 Marked0
    }
}
```

## 实战场景

| 场景 | 关注点 | 调优 |
|------|--------|------|
| ZGC 调优 | 读屏障开销 | 监控 CPU，必要时增加 ConcGCThreads |
| 大堆低延迟 | 染色指针优势 | 堆 > 64GB 时 ZGC 优势明显 |
| 兼容性 | 平台支持 | macOS / Windows / Linux 都支持 |

## 深挖追问

### 染色指针为什么用 44 位寻址？

64 位指针高 16 位通常为 0（操作系统虚拟地址空间一般 < 48 位）。ZGC 用高 4 位作为标志位，剩余 44 位寻址，最大支持 16TB 堆。这已经远超当前服务器物理内存上限，未来可扩展到 56 位寻址（64TB）。

### 染色指针和指针压缩（UseCompressedOops）什么关系？

两者无关。指针压缩是把 64 位指针压缩为 32 位（堆 < 32GB 时），节省内存。染色指针是在指针的高位存 GC 标记信息。两者可以共存：压缩指针时低位寻址，染色指针时高位存标记。但 ZGC 实际上不使用压缩指针，因为染色指针需要 64 位空间。

### 读屏障如何实现指针自愈？

读屏障是 JIT 在每次从堆读取对象引用时插入的代码。读屏障检查指针视图：

- 视图与当前一致：无操作。
- 视图不一致但对象未转移：标记对象，更新指针视图。
- 视图不一致且对象已转移：把指针更新到新地址（自愈）。

应用线程下次访问该指针时已是修正后的视图，无开销。

### Shenandoah 也用染色指针吗？

Shenandoah 早期用 Brooks 转发指针（对象头额外字段），JDK 13+ 也引入了染色指针。但 Shenandoah 的染色指针布局与 ZGC 不同，标志位含义也不同。

## 易错点

- 把染色指针和指针压缩混淆，两者是不同技术。
- 以为染色指针会缩小可用堆空间，实际只占用指针高位，不影响寻址。
- 以为染色指针需要应用代码改造，完全 JIT 透明。
- 把"多重映射"当作"虚拟内存"，多重映射是同一物理内存映射到多个虚拟地址。
- 以为 ZGC 不需要读屏障，染色指针需要读屏障配合实现自愈。

## 总结

染色指针是 ZGC 的核心技术，把 GC 标记信息编码到 64 位指针的高 4 位，无需访问对象头即可知道对象状态。配合多重映射和读屏障，ZGC 实现并发标记、并发转移、指针自愈，STW 时间 < 1ms 且不随堆增长。染色指针是 ZGC 实现亚毫秒级停顿和大堆支持的关键。
