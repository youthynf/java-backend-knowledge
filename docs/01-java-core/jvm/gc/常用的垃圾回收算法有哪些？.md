# 常用的垃圾回收算法有哪些

## 核心概念

垃圾回收算法是垃圾收集器（GC）的理论基础，规定了"如何标记垃圾、如何回收内存"。三大基础算法：标记-清除、复制、标记-整理。分代收集是把不同算法组合应用到不同代；分区收集（G1）把堆分成 Region 灵活回收。理解算法思想是理解各 GC 收集器设计取舍的前提。

每个算法都在"内存利用率、内存碎片、停顿时间、吞吐量"四个维度做权衡，没有银弹。

```text
┌────────────────────────────────────────────────────────────────┐
│  算法        适用代       碎片   空间利用率   STW   典型GC    │
├────────────────────────────────────────────────────────────────┤
│  标记-清除   老年代       有     高           中    CMS       │
│  复制        新生代       无     低（减半）   短    Serial/PS │
│  标记-整理   老年代       无     高           长    Serial Old│
│  分代收集    全堆         -      高           短    所有      │
│  分区收集    全堆         无     高           可控  G1/ZGC   │
└────────────────────────────────────────────────────────────────┘
```

## 标准回答

三大基础 GC 算法 + 分代 + 分区。一句话结论：标记-清除适合存活多但有碎片；复制适合存活少但浪费空间；标记-整理无碎片但移动开销大；分代把新生代用复制、老年代用标记-整理/清除组合应用；分区（G1）按 Region 灵活回收。

要点：

1. **标记-清除 Mark-Sweep**：标记所有可达对象，统一回收未标记对象；有碎片，效率低。
2. **复制 Copying**：内存分两块，每次只用一块，GC 时复制存活对象到另一块；无碎片但空间减半。
3. **标记-整理 Mark-Compact**：标记存活对象后向一端移动；无碎片但移动开销大。
4. **分代收集 Generational**：新生代用复制，老年代用标记-整理/清除。
5. **分区收集 Region-based**：G1 把堆分 Region，按回收价值排序，优先回收垃圾多的 Region。

## 标记-清除算法（Mark-Sweep）

```text
标记前: [A][B][C][D][E][F][G][H]
标记后: [A][✗B][C][✗D][E][✗F][G][✗H]  (B/D/F/H 不可达)
清除后: [A][  ][C][  ][E][  ][G][  ]
```

过程：

1. **标记阶段**：从 GC Roots 出发遍历，标记所有可达对象。
2. **清除阶段**：遍历堆，回收未标记对象。

优点：实现简单，不需要移动对象。

缺点：

- **内存碎片**：清除后产生大量不连续空间，分配大对象时找不到连续空间，触发 Full GC。
- **效率低**：标记和清除都要遍历全堆。
- **STW**：标记和清除都需要暂停用户线程。

适用：CMS（老年代）。

## 复制算法（Copying）

```text
复制前 (From 区): [A][B][C][D][E]  To 区: [空]
                  ↓ 存活对象复制
复制后 (From 区): [空]              To 区: [A][B][C][D][E]
```

过程：

1. 内存分为两块，每次只用一块。
2. GC 时把存活对象复制到另一块。
3. 清空当前块。

优点：

- 无内存碎片。
- 分配快（指针碰撞法）。

缺点：

- 空间浪费：可用内存减半。
- 存活对象多时复制开销大。

新生代优化：HotSpot 把新生代分为 Eden + 2 个 Survivor（8:1:1），每次只用 Eden + 一个 Survivor，浪费仅 10%。

适用：Serial、ParNew、Parallel Scavenge（新生代）。

## 标记-整理算法（Mark-Compact）

```text
标记前: [A][✗B][C][✗D][E][✗F][G][✗H]
整理后: [A][C][E][G][        空闲        ]
```

过程：

1. 标记阶段：与标记-清除相同。
2. 整理阶段：把存活对象向一端移动，清理边界外内存。

优点：

- 无内存碎片。
- 不浪费空间。

缺点：

- 移动对象需要更新引用，开销大。
- STW 时间较长。

适用：Serial Old、Parallel Old（老年代）。

## 分代收集算法

基于两个假说：

- **弱分代假说**：绝大多数对象朝生夕灭。
- **强分代假说**：熬过多次 GC 的对象越难回收。

把堆分为新生代和老年代，每代用最适合的算法：

```text
┌─────────────────────────────────────────────────────┐
│                     堆内存                          │
├─────────────────────────┬───────────────────────────┤
│       新生代             │         老年代            │
│  (存活率低 ~98%)         │   (存活率高)              │
│  → 复制算法              │   → 标记-整理/标记-清除   │
│  → Minor GC 频繁但快     │   → Major GC 少但慢       │
└─────────────────────────┴───────────────────────────┘
```

跨代引用通过记忆集（Remembered Set）解决，避免全堆扫描。

### 对象晋升老年代的条件

1. **年龄达到阈值**：默认 15，`-XX:MaxTenuringThreshold`。对象头的 Mark Word 中 4 位存年龄，所以最大 15。
2. **大对象直接进老年代**：超过 `-XX:PretenureSizeThreshold` 直接进老年代，避免在 Eden 和 Survivor 间反复复制。
3. **动态年龄判断**：Survivor 区相同年龄对象大小总和超过 Survivor 一半，年龄 ≥ 该年龄的对象直接晋升。
4. **空间分配担保**：Minor GC 后 Survivor 放不下，通过担保机制进入老年代。

### GC 类型

| GC 类型 | 回收区域 | 频率 | 速度 | 触发条件 |
|---------|---------|------|------|---------|
| Minor GC / Young GC | 新生代 | 频繁 | 快 | Eden 区满 |
| Major GC / Old GC | 老年代 | 较少 | 慢 | 老年代空间不足 |
| Mixed GC | 新生代 + 部分老年代 | - | 中 | G1 触发 |
| Full GC | 整堆 + 方法区 | 极少 | 最慢 | 多种条件 |

## 分区收集算法（Region-based）

G1、ZGC、Shenandoah 都基于分区思想：把堆划分为多个大小相等的 Region，每个 Region 可独立回收。无需固定新生代/老年代边界，根据回收价值灵活选择。

```text
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ H  │ E  │ O  │ E  │ O  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ E  │ E  │ O  │ S  │ E  │ O  │ E  │
└────┴────┴────┴────┴────┴────┴────┴────┘
E=Eden  S=Survivor  O=Old  H=Humongous
```

优点：

- 减少大对象回收停顿。
- 可预测停顿时间（MaxGCPauseMillis）。
- 适合大堆。

缺点：

- 实现复杂。
- 元数据开销（RSet 等）。

## 算法对比

| 算法 | 适合场景 | 内存碎片 | 空间利用率 | STW 时间 |
|------|---------|---------|-----------|---------|
| 标记-清除 | 存活对象多 | 有 | 高 | 中等 |
| 复制 | 存活对象少 | 无 | 低（减半） | 短 |
| 标记-整理 | 存活对象多 | 无 | 高 | 长 |
| 分代收集 | 通用 | - | 高 | 优化后最短 |
| 分区收集 | 大堆 | 无 | 高 | 可控 |

## 代码示例

观察分代晋升行为：

```java
// -Xms1g -Xmx1g -Xmn512m -XX:+UseParallelGC -XX:+PrintGCDetails
public class GenerationDemo {
    public static void main(String[] args) {
        // 大对象直接进老年代（默认 PretenureSizeThreshold 对 PS 无效，需 Serial/ParNew）
        byte[] big = new byte[1024 * 1024 * 100]; // 100MB

        // 短期对象在 Eden 分配，Minor GC 后回收
        for (int i = 0; i < 10000; i++) {
            byte[] temp = new byte[1024 * 10]; // 10KB
        }
    }
}
```

## 实战场景

| 场景 | 算法选择 | 参数 |
|------|---------|------|
| 客户端小堆 | Serial（复制 + 标记整理） | `-XX:+UseSerialGC` |
| 后台吞吐 | Parallel Scavenge + Parallel Old | `-XX:+UseParallelGC` |
| 低延迟中等堆 | G1（分区） | `-XX:+UseG1GC -XX:MaxGCPauseMillis=200` |
| 超大堆低延迟 | ZGC | `-XX:+UseZGC` |

## 深挖追问

### 为什么新生代用复制算法，老年代用标记-整理？

新生代对象存活率低（约 98% 死亡），复制少量存活对象效率高，且 Eden+2 Survivor 只浪费 10%。老年代存活率高，复制算法要复制大量对象，效率反而低，因此用标记-整理（无碎片）或标记-清除（CMS，有碎片）。

### 什么叫 stop the world？能避免吗？

STW 指 GC 时暂停所有用户线程。目前所有 GC 都无法完全避免 STW，因为可达性分析必须在一个一致性的快照中进行。但可以通过并发收集缩短 STW：CMS、G1、ZGC 都把大部分工作并发执行，STW 时间从秒级降到毫秒级。

### 对象在什么情况下进入老年代？

1. 年龄达到 MaxTenuringThreshold（默认 15）。
2. 大对象超过 PretenureSizeThreshold 直接进入（仅 Serial/ParNew 生效）。
3. Survivor 区相同年龄对象总和超过 Survivor 一半，动态年龄判断。
4. Minor GC 后 Survivor 放不下，通过担保机制进入。
5. G1 中超过 Region 50% 的对象进入 Humongous Region（视为老年代）。

### 分代收集的两个假说是什么？

弱分代假说：绝大多数对象朝生夕灭。强分代假说：熬过多次 GC 的对象越难回收。基于这两个假说，把堆分新生代和老年代，分别用复制和标记-整理，效率最高。跨代引用通过记忆集（RSet）解决，避免全堆扫描。

## 易错点

- 把"标记-清除"和"标记-整理"混淆，前者有碎片，后者无碎片。
- 以为复制算法在老年代也适用，老年代存活率高，复制开销大。
- 把 Major GC 和 Full GC 等同，Major GC 一般只回收老年代，Full GC 回收整堆。
- 以为对象年龄默认是 16，对象头 Mark Word 中 4 位存年龄，最大 15。
- 以为 PretenureSizeThreshold 对所有 GC 生效，实际只对 Serial 和 ParNew 有效。

## 总结

三大基础 GC 算法各有取舍：标记-清除实现简单但有碎片，复制无碎片但浪费空间，标记-整理无碎片但移动开销大。分代收集根据对象存活特征把不同算法组合应用到不同代，是 HotSpot 的主流策略。分区收集（G1）按 Region 灵活回收，实现可预测停顿。理解算法是理解 CMS、G1、ZGC 设计取舍的基础。
