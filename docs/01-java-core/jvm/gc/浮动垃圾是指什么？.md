# 浮动垃圾是指什么

## 核心概念

浮动垃圾（Floating Garbage）是并发垃圾收集器（CMS、G1、ZGC 等）特有的现象：在并发标记阶段已经成为垃圾的对象，因为标记时还在"存活快照"中，本次 GC 不回收，要等下一次 GC 才能回收。

浮动垃圾不是内存泄漏，只是延迟回收。但如果不控制产生速率，会让堆占用持续增长，提前触发 Full GC。

```text
并发标记开始：对象 A 还存活（被引用）
并发标记中：A 的引用断开，A 成为垃圾
但 A 已在"存活快照"中，被强制标记为存活
本次 GC：A 不被回收（浮动垃圾）
下次 GC：A 重新标记，确认为垃圾，回收
```

## 标准回答

浮动垃圾是并发收集器在并发标记阶段产生的"标记后才死亡的垃圾"，本次 GC 不回收，下次 GC 才回收。CMS 在并发清除阶段产生的垃圾、G1 通过 SATB 强制存活的"快照中存活但实际已死"对象，都是浮动垃圾。浮动垃圾不是泄漏，但需要预留堆空间容纳。

要点：

1. **产生原因**：并发标记/清除时用户线程还在产生新垃圾。
2. **CMS 浮动垃圾**：并发清除阶段产生的新垃圾。
3. **G1 浮动垃圾**：SATB 强制存活的"快照存活但实际已死"对象。
4. **不是泄漏**：下次 GC 会回收。
5. **需要预留空间**：浮动垃圾占用堆，需要降低 GC 触发阈值。

## CMS 的浮动垃圾

CMS 在并发清除阶段（Concurrent Sweep），用户线程还在运行，会产生新垃圾。这些新垃圾本次 CMS 不回收，因为标记阶段已经结束，要等下一次 CMS。

```text
CMS 四阶段：
1. 初始标记（STW）：标记 GC Roots 直接关联
2. 并发标记：遍历引用链
3. 重新标记（STW）：修正并发标记变化
4. 并发清除：清除未标记对象 ← 用户线程还在产生垃圾，这些是浮动垃圾
```

CMS 需要预留空间容纳浮动垃圾：

- `-XX:CMSInitiatingOccupancyFraction=75`：老年代使用 75% 时触发 CMS。
- 预留不足 → Concurrent Mode Failure → 退用 Serial Old 全量收集（很慢）。

## G1 的浮动垃圾

G1 用 SATB 算法，并发标记开始时建立对象图快照。引用消失前，旧值通过写前屏障加入 SATB 缓冲区，最终标记阶段强制标记为存活。即使该对象实际已经死亡，本次 GC 也不回收。

```text
并发标记开始：对象 B 还存活
并发标记中：B 引用断开（field = null）
            旧值 B 入 SATB 缓冲区
最终标记：B 被强制标记为存活
本次 GC：B 不回收（浮动垃圾）
下次 GC：B 重新判定为垃圾，回收
```

G1 与 CMS 浮动垃圾的区别：

- CMS 浮动垃圾是"并发清除阶段新产生的"。
- G1 浮动垃圾是"并发标记阶段引用消失但被 SATB 强制存活的"。

## ZGC 的浮动垃圾

ZGC 也用并发标记，同样会产生浮动垃圾。ZGC 通过染色指针的多个视图（Marked0、Marked1、Remapped）区分两次标记周期，下次 GC 时上次未回收的对象会被重新评估。

## 浮动垃圾的影响

### 堆占用增长

浮动垃圾占用堆空间，但本次不回收。如果业务对象分配速率高，浮动垃圾可能让堆占用持续增长。

### 触发 GC 提前

为容纳浮动垃圾，需要降低 GC 触发阈值：

- CMS：`-XX:CMSInitiatingOccupancyFraction=70`（默认 92%，建议 70~80）。
- G1：`-XX:InitiatingHeapOccupancyPercent=35`（默认 45，浮动垃圾多时可降低）。

### 不算内存泄漏

浮动垃圾下次 GC 一定会回收，不是内存泄漏。但需要监控堆占用，避免因浮动垃圾积累导致 Full GC。

## 代码示例

理解浮动垃圾的产生：

```java
public class FloatingGarbageDemo {
    static Object cache;

    public void test() {
        // 并发标记阶段：
        Object obj = new Object();   // obj 在快照中存活
        cache = obj;

        // 并发标记中：
        cache = null;                // obj 实际已不可达
        // 但 obj 已在快照中，被强制标记为存活
        // 本次 GC 不回收（浮动垃圾）
        // 下次 GC 才回收
    }
}
```

## 实战场景

| 场景 | 关注点 | 调优 |
|------|--------|------|
| CMS 频繁 Concurrent Mode Failure | 浮动垃圾过多 | 降低 `-XX:CMSInitiatingOccupancyFraction` |
| G1 堆占用持续增长 | 浮动垃圾未及时回收 | 降低 `-XX:InitiatingHeapOccupancyPercent` |
| ZGC 堆压力大 | 浮动垃圾 + 大堆 | 调整 `-XX:ZAllocationSpikeTolerance` |

## 深挖追问

### 浮动垃圾和内存泄漏的区别？

浮动垃圾：本次 GC 不回收，下次 GC 一定回收。是并发收集器的固有现象。
内存泄漏：对象永远不会被回收，因为存在隐性引用（静态集合、ThreadLocal、监听器等）。需要修复代码。

### CMS 和 G1 的浮动垃圾有什么不同？

CMS 浮动垃圾是"并发清除阶段新产生的"，标记阶段已经结束，新垃圾本次不回收。
G1 浮动垃圾是"并发标记阶段引用消失但被 SATB 强制存活的"，因为 SATB 保留快照，已断开的引用对象仍被标记为存活。

### 浮动垃圾会无限增长吗？

不会。浮动垃圾下次 GC 一定回收。但如果业务分配速率持续高于 GC 速率，浮动垃圾 + 新对象会让堆满，触发 Full GC（CMS 的 Concurrent Mode Failure、G1 的 Evacuation Failure）。

### 如何减少浮动垃圾？

- 缩短 GC 周期：降低触发阈值，让 GC 更频繁。
- 减少对象分配速率：优化代码，避免无谓对象创建。
- 增大堆：给浮动垃圾更多空间，但治标不治本。
- 换 GC：ZGC 的浮动垃圾相对较少（染色指针 + 多视图）。

## 易错点

- 把浮动垃圾当作内存泄漏，实际下次 GC 会回收。
- 以为浮动垃圾只在 CMS 有，G1、ZGC 都有。
- 以为降低 GC 触发阈值能消除浮动垃圾，只是减少其累积。
- 把"并发标记中的新对象"当作浮动垃圾，新对象在 G1 中默认标记为存活（黑色），属于正常处理。
- 以为浮动垃圾可以避免，并发收集器必然有，只是多少的区别。

## 总结

浮动垃圾是并发收集器的固有现象：标记后才死亡的垃圾本次不回收，下次 GC 才回收。CMS 在并发清除阶段产生，G1 通过 SATB 强制存活产生。浮动垃圾不是泄漏，但需要预留堆空间容纳，通过降低 GC 触发阈值应对。理解浮动垃圾是理解 CMS Concurrent Mode Failure、G1 SATB、并发收集器堆占用增长的基础。
