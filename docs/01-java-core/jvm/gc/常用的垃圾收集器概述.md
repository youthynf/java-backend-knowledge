# 常用的垃圾收集器概述

## 核心概念

垃圾收集器（GC Collector）是 GC 算法的具体实现，决定了"什么时候回收、回收哪些区域、停顿多长、并发还是并行"。HotSpot 提供了 7 款主流收集器：Serial、ParNew、Parallel Scavenge、Serial Old、Parallel Old、CMS、G1，加上 JDK 11+ 的 ZGC 和 JDK 12+ 的 Shenandoah。

选择收集器的核心是平衡三个目标：停顿时间（延迟）、吞吐量、内存占用。没有银弹，看场景选。

```text
      新生代收集器                    老年代收集器
  ┌──────────────┐              ┌───────────────────┐
  │   Serial     │────搭配──────│  Serial Old (MSC) │
  └──────────────┘              └───────────────────┘
  ┌──────────────┐              ┌───────────────────┐
  │  ParNew      │────搭配──────│      CMS          │
  └──────────────┘              └───────────────────┘
  ┌──────────────┐              ┌───────────────────┐
  │  Parallel    │────搭配──────│   Parallel Old    │
  │  Scavenge    │              └───────────────────┘
  └──────────────┘

  ┌──────────────────────────────────────────────────────┐
  │           G1（整堆分区收集，JDK 9+ 默认）            │
  └──────────────────────────────────────────────────────┘
  ┌──────────────────────────────────────────────────────┐
  │        ZGC / Shenandoah（超低延迟，TB 级堆）         │
  └──────────────────────────────────────────────────────┘
```

## 标准回答

7 款经典收集器 + 2 款现代收集器。一句话结论：吞吐量优先选 Parallel Scavenge + Parallel Old，延迟敏感选 CMS（JDK 8）或 G1（JDK 9+），超大堆低延迟选 ZGC/Shenandoah。JDK 8 默认 Parallel Scavenge，JDK 9+ 默认 G1，JDK 15+ ZGC 生产可用。

要点：

1. **Serial**：单线程，客户端模式或小堆。
2. **ParNew**：Serial 的多线程版本，CMS 的搭档。
3. **Parallel Scavenge**：吞吐量优先，JDK 8 默认。
4. **Serial Old / Parallel Old**：对应老年代。
5. **CMS**：低停顿，标记-清除，JDK 9 废弃，JDK 14 移除。
6. **G1**：分区 + 可预测停顿，JDK 9+ 默认。
7. **ZGC**：染色指针 + 读屏障，TB 级堆，亚毫秒停顿。
8. **Shenandoah**：RedHat 出品，Brooks 转发指针。

## Serial 收集器

- 单线程收集，STW 时只用一个 GC 线程。
- 新生代用复制算法，老年代用标记-整理。
- 简单高效，无多线程交互开销。

```bash
-XX:+UseSerialGC
```

适用：客户端模式、微服务小堆（几十 MB）、嵌入式。

## ParNew 收集器

- Serial 的多线程版本，新生代并行收集。
- CMS 的默认新生代搭档。
- JDK 9 起标记 Deprecated，JDK 14 移除（跟随 CMS）。

```bash
-XX:+UseParNewGC              # ParNew + Serial Old（JDK 9 起废弃）
-XX:+UseConcMarkSweepGC       # CMS + ParNew
-XX:ParallelGCThreads=4       # GC 线程数
```

适用：配合 CMS 使用，中等堆大小服务端。

## Parallel Scavenge 收集器

- **吞吐量优先**，新生代并行复制算法。
- JDK 8 默认收集器。
- 自适应调节（`-XX:+UseAdaptiveSizePolicy`）。

```bash
-XX:+UseParallelGC            # Parallel Scavenge + Serial Old（JDK 8 默认新生代）
-XX:+UseParallelOldGC         # Parallel Scavenge + Parallel Old
-XX:MaxGCPauseMillis=200      # 最大 GC 停顿目标（尽力而为）
-XX:GCTimeRatio=99            # 吞吐量大小（GC 时间 = 1/(1+99) = 1%）
-XX:+UseAdaptiveSizePolicy    # 自适应调节
```

与 ParNew 的区别：

- ParNew 配合 CMS，关注延迟。
- Parallel Scavenge 关注吞吐量，自适应调节。

适用：后台计算型任务、批处理、离线分析。

## Serial Old 收集器

- Serial 的老年代版本，单线程标记-整理。
- 客户端模式或 CMS 后备方案（Concurrent Mode Failure 时）。

## Parallel Old 收集器

- Parallel Scavenge 的老年代版本，多线程标记-整理。
- 与 Parallel Scavenge 配合，吞吐量优先。

## CMS 收集器（Concurrent Mark Sweep）

目标：最短回收停顿时间。第一款真正意义上的并发收集器。基于标记-清除算法。

### 四个阶段

```text
──1.初始标记──2.并发标记──3.重新标记──4.并发清除──
   STW 短       并发        STW 稍长    并发
```

| 阶段 | STW | 说明 |
|------|-----|------|
| 初始标记 | 是 | 标记 GC Roots 直接关联对象，速度快 |
| 并发标记 | 否 | 从 GC Roots 遍历引用链，与用户线程并发 |
| 重新标记 | 是 | 修正并发标记期间变动（增量更新），稍长 |
| 并发清除 | 否 | 清除未标记对象，与用户线程并发 |

### CMS 的三个致命问题

1. **CPU 敏感**：默认占用 (CPU 核数+3)/4 个 GC 线程，CPU 少时影响吞吐。
2. **浮动垃圾**：并发清理阶段产生的新垃圾，只能下次 GC 回收。预留空间不足 → Concurrent Mode Failure → 退用 Serial Old 全量收集（很慢）。
3. **内存碎片**：标记-清除产生碎片，可能空间足够但找不到连续空间分配大对象，触发 Full GC。

```bash
-XX:+UseConcMarkSweepGC
-XX:CMSInitiatingOccupancyFraction=75   # 老年代使用 75% 触发
-XX:+UseCMSCompactAtFullCollection       # Full GC 后压缩
-XX:CMSFullGCsBeforeCompaction=2         # 2 次 Full GC 后压缩
-XX:+CMSParallelRemarkEnabled            # 并行重新标记
```

**JDK 9 起 CMS 标记 Deprecated，JDK 14 正式移除。**

## G1 收集器（Garbage First）

JDK 9 起默认收集器。面向服务端，追求停顿时间可控。详见 [G1 垃圾收集器基本原理](G1垃圾收集器基本原理.md)。

### 核心设计

- 把堆划分为多个 Region（1~32MB，默认约 2048 个）。
- 每个 Region 可动态切换为 Eden / Survivor / Old / Humongous。
- 大对象（超过 Region 50%）直接分配在 Humongous Region。

### 四个阶段

| 阶段 | STW | 说明 |
|------|-----|------|
| 初始标记 | 是 | 标记 GC Roots 直接关联对象 |
| 并发标记 | 否 | 从 Roots 遍历引用链，使用 SATB 快照 |
| 最终标记 | 是 | 处理 SATB 缓冲区残留引用 |
| 筛选回收 | 是 | 按回收价值排序 Region，优先回收收益高的 |

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200                  # 目标停顿时间
-XX:G1HeapRegionSize=4m                   # Region 大小
-XX:InitiatingHeapOccupancyPercent=45     # 触发并发标记的堆占用阈值
-XX:G1NewSizePercent=5                    # 新生代最小比例
-XX:G1MaxNewSizePercent=60                # 新生代最大比例
```

### G1 vs CMS

| 对比项 | CMS | G1 |
|--------|-----|-----|
| 算法 | 标记-清除 | 标记-整理/复制 |
| 碎片 | 有 | 无 |
| 停顿预测 | 不可控 | 可控（MaxGCPauseMillis） |
| 堆大小 | 建议 < 6GB | 建议 > 6GB |
| 内存布局 | 分代连续 | Region 化 |
| JDK 版本 | ≤ 8 | ≥ 9（默认） |

## ZGC 收集器（JDK 11+）

超低延迟收集器，停顿时间 < 10ms（JDK 16+ < 1ms），且不随堆大小增长。详见 [ZGC 垃圾收集器基本原理](ZGC垃圾收集器基本原理.md)。

### 核心技术

| 技术 | 作用 |
|------|------|
| 染色指针 | 在指针中存储 GC 标记信息 |
| 读屏障 | 读取对象引用时执行，修正指针（自愈） |
| 多重映射 | 同一物理内存映射到不同虚拟地址 |

```bash
-XX:+UnlockExperimentalVMOptions -XX:+UseZGC  # JDK 11-14
-XX:+UseZGC                                     # JDK 15+
-XX:ZCollectionInterval=0                       # GC 间隔
-XX:ZAllocationSpikeTolerance=2                 # 分配尖峰容忍度
-XX:SoftMaxHeapSize=4g                          # 软最大堆
```

适用：超大堆（TB 级）、对延迟极其敏感的在线服务。JDK 21 引入分代 ZGC，进一步优化。

## Shenandoah 收集器（JDK 12+）

RedHat 开发的低延迟收集器，目标停顿 < 10ms。

| 对比项 | Shenandoah | ZGC |
|--------|-----------|-----|
| 开发者 | RedHat | Oracle |
| 指针技术 | Brooks 转发指针（对象头额外字段） | 染色指针（指针本身编码） |
| 读屏障 | 有（修正转发指针） | 有（自愈染色指针） |
| 并发整理 | 对象移动与用户线程并发 | 对象移动与用户线程并发 |

```bash
-XX:+UseShenandoahGC
```

## 收集器选择指南

```text
你的需求是什么？
│
├── 吞吐量优先（批处理/离线分析）
│   └── Parallel Scavenge + Parallel Old
│
├── 延迟敏感（在线服务）
│   ├── JDK 8
│   │   └── CMS（+ ParNew）
│   ├── JDK 9-14
│   │   └── G1
│   └── JDK 15+
│       ├── G1（通用）
│       ├── ZGC（超低延迟，大堆）
│       └── Shenandoah（超低延迟，RedHat）
│
├── 小堆（< 512MB）
│   └── Serial / G1
│
└── 默认选择
    ├── JDK 8  → Parallel Scavenge + Parallel Old
    └── JDK 9+ → G1
```

## 实战场景

| 场景 | GC 选择 | 参数 |
|------|---------|------|
| Spring Boot 微服务 | G1 | `-Xms2g -Xmx2g -XX:+UseG1GC -XX:MaxGCPauseMillis=100` |
| 后台批处理 | Parallel | `-Xms4g -Xmx4g -XX:+UseParallelGC -XX:GCTimeRatio=99` |
| 大堆低延迟 | ZGC | `-Xms16g -Xmx16g -XX:+UseZGC -XX:SoftMaxHeapSize=12g` |
| 容器小堆 | Serial/G1 | `-Xmx512m -XX:+UseSerialGC` 或 G1 |

## 深挖追问

### CMS 和 G1 的区别？

CMS 用标记-清除算法，有内存碎片问题；G1 用 Region 分区 + 复制算法，无碎片。CMS 停顿时间不可控，G1 可通过 MaxGCPauseMillis 设定目标。CMS 需要连续的堆空间，G1 的 Region 化更灵活。JDK 9 起 G1 成为默认，CMS 在 JDK 14 被移除。

### G1 的 "Garbage First" 是什么意思？

G1 在筛选回收阶段会根据每个 Region 的垃圾占比（回收价值）排序，优先回收垃圾最多的 Region，这就是 "Garbage First"。这种策略让 G1 在有限停顿时间内回收最多内存。

### ZGC 为什么能做到亚毫秒级停顿？

ZGC 将几乎所有 GC 工作并发执行（标记、转移），只在初始标记和初始转移时短暂 STW。核心是染色指针 + 读屏障技术，对象移动时用户线程通过读屏障自动修正指针（自愈），不需要 STW 更新引用。STW 时间只依赖 GC Roots 数量，与堆大小无关。

### 如何选择 GC 收集器？

看场景：

- 吞吐优先用 Parallel，延迟敏感用 G1/ZGC。
- JDK 8 默认 Parallel，JDK 9+ 默认 G1。
- 堆大于 6GB 建议用 G1。
- 对延迟要求极致（< 10ms）且堆较大（> 16GB），用 ZGC。
- 容器小堆（< 512MB）用 Serial 也行。

## 易错点

- 把"并发"和"并行"混淆：并行（Parallel）是多个 GC 线程同时工作但 STW；并发（Concurrent）是 GC 线程与用户线程同时执行。
- 以为 CMS 是 JDK 9+ 默认，实际 JDK 9+ 默认是 G1，CMS 已废弃。
- 以为 ZGC 不需要 STW，所有 GC 都需要 STW，ZGC 只是 STW 极短。
- 把 `-XX:+UseParallelGC` 和 `-XX:+UseParallelOldGC` 当作独立选择，实际 ParallelGC 控制新生代，ParallelOldGC 控制老年代。
- 以为 G1 不需要分代，G1 逻辑上仍分代，只是物理上 Region 化。

## 总结

7 款经典 + 2 款现代收集器各有取舍。Serial/ParNew/Parallel 是新生代，Serial Old/Parallel Old/CMS 是老年代，G1/ZGC/Shenandoah 是整堆。JDK 8 默认 Parallel，JDK 9+ 默认 G1，JDK 15+ ZGC 生产可用。选择看场景：吞吐优先 Parallel，延迟敏感 G1/ZGC，超大堆 ZGC。理解各收集器的算法、阶段、参数是 GC 调优的基础。
