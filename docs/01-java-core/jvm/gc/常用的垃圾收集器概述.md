# 常用的垃圾收集器概述

常用的垃圾收集器概述
Serial收集器（复制算法）
串行收集器，最早诞生的垃圾回收器，以单线程的方式进行垃圾收集，进行垃圾收集时，必须暂停其他所有的工作线程，直到它收集结束（STW）。优点就是简单高效，对于单个CPU环境来说，由于没有线程交互的开销，因此拥有最高的单线程收集效率。
•  特点：单线程、简单高效（与其他收集器的单线程相比），对于限定单个CPU的环境来说，由于没有线程交互开销，效率较高。
•  适用场景：对于内存较小的场景，并且对暂停时间要求不是特别高的应用，如一些简单命令行工具或Client模式下的应用等；
•  工作原理：采用复制算法，将新生代分为一个Eden区和两个Survivor区（一般比例是8:1:1）。新建的对象首先被分配到Eden区，当Eden区满时，触发一次MinorGC，将存活的对象复制到一个Survivor区，然后清空Eden区。当Survivor区满时，存活的对象会被复制到另一个Survivor区或者老年代。垃圾收集只有一个阶段，就是暂停应用线程，单线程执行垃圾回收。

ParNew收集器（复制算法）
并行收集器，Serial收集器的多线程版本，除了使用多线程进行回收垃圾外，它和Serial收集器的行为基本类似，垃圾收集过程中需要STW，在多核CPU环境下有着比Serial收集器更好的表现。它是Server模式下的虚拟机首选新生代收集器，除了性能原因外，主要是因为除了Serial收集器，只有它能与CMS收集器配合工作。
•  适用场景：主要用于新生代的垃圾收集器，并且在Server模式下与CMS收集器配合使用。
•  工作原理：采用复制算法，将新生代分为Eden区和Survivor区，在垃圾收集时，在多CPU环境下，多个线程可以同时进行复制操作，从而提高垃圾收集的效率，减少垃圾收集时间。垃圾收集只有一个阶段，就是暂停应用线程，多线程并行执行垃圾回收。

Parallel Scavenge收集器 (复制算法)
并行收集器，与ParNew一样是多线程收集器，垃圾收集过程中需要STW，但支持GC自适应调节策略，可设置-XX:+UseAdptiveSizePolicy参数。它的目标是达到一个可控制的吞吐量，而不是关注用户线程的停顿时间，它被称为吞吐量优先收集器。当开关打开时不需要手动指定新生代的大小（-Xmn）、Eden与Survivor区的比例（-XX:SurvivorRation）、晋升老年代的对象年龄（-XX:PretenureSizeThreshold）等，虚拟机会根据系统的运行状况收集性能监控信息，动态设置这些参数以提供最优的停顿时间和最高的吞吐量。与Parallel Old收集器配合使用。
•  适用场景：适用于吞吐量要求较高的场景，比如后台任务处理、科学计算等应用程序。这些应用更加关注一段时间内能够处理多少任务，而不是每次垃圾收集的暂停时间；
•  工作原理：采用复制算法，新生代垃圾收集。可以通过参数控制吞吐量，如-XX:MaxGCPauseMillis设置最大垃圾收集停顿时间，根据应用需求来优化垃圾收集的性能；可以通过一个开关参数打开 GC 自适应的调节策略(GC Ergonomics)，就不需要手动指定新生代的大小(-Xmn)、Eden 和 Survivor 区的比例、晋升老年代对象年龄等细节参数了。虚拟机会根据当前系统的运行情况收集性能监控信息，动态调整这些参数以提供最合适的停顿时间或者最大的吞吐量。

Serial Old收集器（标记-整理算法）
串行收集器，Serial收集器的老年代版本，同样是单线程收集器。它也是给Client模式下的虚拟机使用，如果用在 Server 模式下，在 JDK 1.5 以及之前版本(Parallel Old 诞生以前)中与Parallel Scavenge收集器搭配使用，以及作为 CMS 收集器的后备预案，在并发收集发生Concurrent Mode Failure 时使用。
•  适用场景：主要用于Client模式下的老年代收集，或者在一些特定场景下作为CMS收集器的后备预案。
•  工作原理：采用标记-整理算法，首先标记出所有需要回收的对象，然后将存活的对象向一端移动，最后直接清理掉端边界以外的内存空间，这种算法可以避免内存碎片问题，适合老年代内存空间的回收。

Parllel Old收集器（标记-整理算法）
它是Parallel Scavenge收集器的老年代版本，老年代并行收集器，追求高吞吐量，高效利用 CPU。
•  适用场景：对吞吐量要求较高，并且老年代垃圾收集也需要高效处理的应用场景。与Parallel Scavenge收集器配合使用，能够在整体上提高应用程序的吞吐量，适用于一些大数据处理、批量处理等应用。
•  工作原理：采用标记 - 整理算法，先标记出需要回收的对象，然后将存活的对象向一端移动，清理掉边界以外的内存空间。这样可以在保证较高吞吐量的同时，避免老年代出现大量的内存碎片。

CMS（Concurrent Mark Sweep）收集器（标记-清除算法）
CMS(Concurrent Mark Sweep)，Mark Sweep 指的是标记 - 清除算法，整个回收期间分为四个阶段：初始标记、并发标记、重新标记、并发清除等，其中初始标记和重新标记阶段需要STW暂停应用线程，在并发标记和并发清除阶段可以与应用线程并发运行。这个算法通常适用于老年代，年轻代可以使用并行复制回收或串行复制回收。同样，在老生代回收时，因为并发执行，如果在分配内存时内存不足，则需要进行FGC，也需要STW并对整个内存进行穿行回收。
•  适用场景：适用于那些对响应时间要求较高，不能长时间停顿的应用程序，比如互联网的B/S加购的服务器、电子商务网站。这些应用需要快速响应用户请求，不能因垃圾收集而长时间阻塞用户。但是以牺牲吞吐量为代价，导致CPU利用率不高，无法处理浮动垃圾，同时使用标记-清除算法会导致空间碎片，往往出现老年代空间剩余但无法找到足够大连续空间来分配而触发一次FullGC，需要对整个内存进行STW进行串行回收。
•  工作原理：整个垃圾收集过程分为4个主要阶段：初始标记（短暂STW，标记处GC Roots能够直接关联的老年代对象，这个阶段速度很快）、并发标记（垃圾收集线程和用户线程同时运行，基于初始标记的结果，进一步标记出所有可达的老年代对象，这个过程可能会比较长，但是由于并发，所以不会导致长时间停顿）、重新标记（短暂STW，因为在并发标记阶段，用户可能会改变对象的引用关系，所以需要重新标记，以确保标记的准确性）、并发清除（垃圾收集线程与用户线程并发执行，主要是清除那些没有被标记的老年代对象）

G1（Garbage First）收集器（标记-整理算法）
G1是JDK7 update4引入的一款并发垃圾收集器，极大简化了CMS复杂性（上百个参数），G1致力于在多CPU和大内存服务器上对垃圾回收提供软实时目标和高吞吐量。
•  适用场景：适用于大内存，多处理器的机器，并且对停顿时间有一定要求的应用，比如大型企业级应用、云计算环境中的应用，能不牺牲太多吞吐量的情况下，有效控制垃圾收集的停顿时间；
•  工作原理：
G1 把堆划分成多个大小相等的独立区域(Region)，新生代和老年代不再物理隔离。在垃圾收集时，它根据各个区域的垃圾堆积程度，优先手机垃圾最多的区域，从而提高垃圾收集的效率。这种划分方法带来了很大的灵活性，使得可预测的停顿时间模型成为可能。通过记录每个 Region 垃圾回收时间以及回收所获得的空间(这两个值是通过过去回收的经验获得)，并维护一个优先列表，每次根据允许的收集时间，优先回收价值最大的 Region。每个 Region 都有一个 Remembered Set，用来记录该 Region 对象的引用对象所在的 Region。通过使用 Remembered Set，在做可达性分析的时候就可以避免全堆扫描。
G1的年轻代回收方式是并行回收，单阶段完成，采用复制算法，和其他垃圾收集器一样，一旦发生一次年轻代的回收，整个年轻代都会被回收，也就是我们说的YGC。但是与其他垃圾收集器不同的是，G1会根据预测时间动态调整年轻代的大小，同时对于老年代的回收不会为了释放老年代的分区而对整个老年代进行回收，任意时刻只会对部分老年代分区进行回收，选择垃圾多的老年代分区，并且这部分的老年代分区回收会在下一次增量回收中与年轻代一起被回收，这就是所谓的混合回收（Mixed GC）。对于老年代的回收涉及G1的并发标记算法，这个过程称为并发标记阶段，主要分为4个子阶段：初始标记子阶段（和CMS类似，短暂STW，标记出GC Roots能直接关联到的对象，该阶段和年轻代的的回收阶段几乎一致，实际上MixedGC借用了年轻代的收集结果，无需单独一次初始标记，这就是所谓的“借道”）、并发标记子阶段（YGC之后，内存消耗达到阈值后启动，使用多个标记线程与用户线程并发执行，每个线程每次只扫描一个分区，标记出存活的对象。标记时会记录存活对象的数量、内存占用大小并记入分区空间。使用STAB算法进行并发标记）、再标记子阶段（短暂STW，并行执行，主要也是处理并发标记阶段中引用关系的变化，确保标记的准确性）、（筛选回收）清理子阶段（需要STW，并行执行，根据各个区域的垃圾堆积程度进行排序，将空闲分区放到空闲分区列表中，该阶段不会对垃圾对象进行清理，也不会执行存活对象的复制，真正的垃圾回收是在混合回收中）。在并发标记阶段完成之后，在下一次的垃圾回收的时候就会回收垃圾比较多的老年代分区，这时候的回收就是混合回收，与YGC最大的不同就是同时进行年轻代分区和部分老年代分区的回收。
由于G1对停顿时间有要求，在并发标记阶段可能存在多次YGC；混合回收有可能执行多次，需要根据预测时间来决定一次回收的老年代分区数量。
JDK10在内存不足时将串行回收优化成并行回收提升了回收效率。

ZGC（Z Garbage Collector）
ZGC是一种可伸缩的低延迟垃圾收集器，目标是在处理非常大的堆内存（例如数 TB 的堆）时，也能将暂停时间控制在极低的水平，通常在 10 毫秒以内，而且它不会随着堆的增大而显著增加暂停时间。ZGC的引入是为了解决G1收集器的不足的，主要包括：停顿时间过长、内存利用率不高，通常引用关系的处理需要额外消耗内存（占整个内存的1%-20%）、不适合超大内存系统（100GB以上停顿明显增加）。
•  适用场景：对响应时间要求极高的应用，如电商网站的实时交易处理、云计算平台的实时资源分配等；处理大规模数据的应用，尤其是堆内存占用较大的情况，像大数据分析系统、分布式存储系统等；
•  工作原理：主要有4个阶段：并发标记（与用户线程并发，它从根集合（如线程栈、静态变量等）开始，通过对象引用关系，标记所有存活的对象）、并发预备重分配（主要是确定哪些区域的对象需要被重分配，并将这些对象所在的页面标记为“待重分配”状态）、并发重分配（关键阶段，与用户线程并发执行，应用线程在运行过程中，当访问到一个位于 “待重分配” 页面中的对象引用时，会通过写屏障（Write Barrier）将引用更新到新的位置）、并发重映射（与用户线程并发执行，对象重分配完成后，需要对引用进行重映射，清理那些指向旧位置的引用，使所有引用都指向对象的新位置）；

注：
CMS和G1的STW标记阶段都是使用多线程，而非单线程，从而缩短STW时间；
G1筛选回收阶段，新区域是指在垃圾回收过程中，G1垃圾收集器为存活对象分配的新内存区域。存活对象会从旧区域复制到这些新区域中。
