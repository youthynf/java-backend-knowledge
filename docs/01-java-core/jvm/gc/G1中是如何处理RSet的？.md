# G1中是如何处理RSet的？

G1中是如何处理RSet的？
概述
记忆集（Remembered Set，RSet） 是G1垃圾收集器的关键数据结构，用于记录跨Region（跨代）的引用关系，避免全堆扫描，提升GC效率。
背景
JVM使用的是根对象引用的收集算法，即从根集合出发，标记所有存活的对象，然后遍历对象的每一个成员变量并继续标记，直到所有对象标记完毕。在分代垃圾回收算法中，年轻代和老年代处于不同的回收阶段，如果还是采用这种标记算法，在进行年轻代回收时，我们需要同时把老年代的活跃对象全部标记，但回收时并没有回收老年代，则浪费了时间。只有我们进行FGC时，才真正需要对内存做全部标记。因此，算法设计者设计了使用RSet记录从非收集部分指向收集部分的指针集合，而这个集合称为对象的引用关系。
记录方法
RSet引用关系记录方法可以分为两种，一种是Point Out，第二种是Point In，前者记录的是当前分区对象对其他分区对象的引用，而后者记录的是其他分区对象对当前分区对象的引用。
•  Point Out：维护简单，只需要维护当前分区是否继续引用外部分区即可，但是标记时需要对RSet做全部扫描，检查引用的对象是否存活；
•  Point In：维护复杂，需要维护引用当前的所有分群的引用关系是否正常，但是标记时高效，可以判断当前对象是否活跃，无需进行额外扫描，因为RSet里面的对象可以看作跟对象。G1使用的是Point In的方式。

数据结构
为了提高RSet的存储效率，使用了3种数据结构：
•  稀疏哈希表：当前Region被极少数其他Region中的对象引用时使用，利用哈希表存储引用源所在的Card索引（卡表条目），key是引用源RegionID+Card索引，value无（仅需记录存在引用），通过检查某Card是否存在集合中来判断是否存在引用。可以精确到对象级别，但是开销较高，每个引用占据一个条目，适合引用极度稀疏的场景。
•  细粒度位图：当前Region被几十到几百Card引用时使用，通过位图方式，每个bit代表一个Card（512Byte），若该Card有引用则置1。空间效率较高，位操作快速查询，但无法精确定位对象，需要扫描整个Card；
•  粗粒度位图：当前Region被大量引用，如某个Region存在超过50%的Card引用当前Region对象，则使用位图方式，每个bit标记对应的外部Region是否引用当前Region，空间占用最小，1个bit对应一个Region，但是无法定位引用位置，只能全Region扫描。

G1中RSet维护
G1新引入了Refine线程，它实际上是一个线程池，主要用于处理新生代分区的抽样，并且在满足相应时间的这个指标的情况下，更新新生代分区的数目，通常由一个单独的线程来处理；此外还负责更新RSet，对于RSet的更新并不是同步完成的，G1会把所有引用关系放入一个队列中，成为Dirty Card Queue（DCQ），然后使用Refine线程来消费这个队列，完成引用关系的记录。
RSet保存的引用关系分析
不需要记录的引用关系：
•  分区内部：扫描是按分区进行的，因此同分区的所有引用关系在扫描时能全部覆盖，无需额外记录；
•  新生代分区→新生代分区：年轻代分区在YGC回收时会全量扫描，因此年轻代分区间的引用关系会被全部覆盖到，因此也无需额外记录；
•  新生代分区→老年代分区：YGC时无需扫描老年代分区，因此不需要知道这个引用关系，而混合回收时，G1会使用新生代的分区作为根，自然能扫描出新生代到老年代分区的引用，因此也无需记录；FGC所有分区都会处理，更加不用记录；

需要记录的引用关系：
•  老年代分区→老年代分区：在混合回收的时候，可能只有部分分区被回收，因此需要通过记录这个引用关系，快速找到存活对象；
•  老年代分区→年轻代分区：在YGC时有两种根，一个是栈空间/全局空间变量的引用，另一个是老年代分区到新生代分区的引用，因此需要记录；

## 面试总结

围绕「G1中是如何处理RSet的？」，面试官通常不只考概念定义，更关注你能否把机制、使用场景和线上问题串起来。

### 核心回答

1. GC 的核心目标是在停顿时间、吞吐量和内存占用之间做权衡。
2. 判断对象是否可回收要理解可达性分析、引用类型、分代假设和具体收集器机制。
3. G1/ZGC 等现代收集器更强调可预测停顿，但仍需要根据对象分配速率和堆布局调优。

### 实战落地

- **排查类问题**：先收集监控、日志和 JVM 现场信息，再用工具验证假设，避免凭经验改参数。
- **调优类问题**：先明确目标是降低停顿、提升吞吐还是减少内存，再选择收集器、堆大小和业务代码优化。
- **面试表达**：用“现象 → 原理 → 工具验证 → 解决方案 → 风险边界”的链路回答。

### 易错点

- 不要只看平均停顿，要看 P99 和业务尖峰时段。
- 调优前先定位问题，别上来就改 Xmx、G1 参数。
