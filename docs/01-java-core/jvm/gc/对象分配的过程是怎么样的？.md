# 对象分配的过程是怎么样的

## 核心概念

对象分配过程指 JVM 决定"对象分配在堆的哪个位置"的策略。它不是单点动作，而是一组规则：优先 TLAB → Eden → 大对象进老年代 → 长期存活晋升老年代 → 动态年龄判断 → 空间分配担保。理解这套规则能帮助排查"为什么对象进老年代了"、"为什么 Full GC 频繁"。

对象分配的核心目标是：让短期对象在新生代回收，避免进老年代增加 Full GC 压力；让长期对象进老年代，避免在新生代反复复制。

```text
new 一个对象 →
   1. 优先 TLAB（Eden 内线程私有缓冲）
   2. TLAB 失败 → Eden 加锁分配
   3. Eden 满 → 触发 Minor GC
   4. 大对象 → 直接进老年代
   5. Survivor 装不下 → 空间分配担保进老年代
   6. 年龄达阈值 → 晋升老年代
```

## 标准回答

对象分配按"优先 TLAB → Eden → 老年代"的层次进行。一句话结论：TLAB 是 Eden 内的线程私有缓冲，分配无锁；Eden 满触发 Minor GC；大对象直接进老年代；长期存活对象按年龄晋升；Survivor 装不下通过空间分配担保进老年代；动态年龄判断让相同年龄对象总和超 Survivor 一半时整体晋升。

要点：

1. **TLAB**：Eden 内线程私有缓冲，分配无锁。
2. **Eden 分配**：TLAB 失败后走 Eden 加锁分配。
3. **大对象**：超过 `-XX:PretenureSizeThreshold` 直接进老年代。
4. **年龄晋升**：默认 15，`-XX:MaxTenuringThreshold`。
5. **动态年龄判断**：Survivor 中相同年龄对象总和超一半，整体晋升。
6. **空间分配担保**：Survivor 装不下时进老年代。

## 对象优先在 Eden 分配

大多数情况下，对象在新生代 Eden 区分配。当 Eden 区空间不够时，触发 Minor GC。

### TLAB（Thread Local Allocation Buffer）

JVM 为每个线程在 Eden 区分配一小块私有内存（TLAB），对象优先在 TLAB 中分配，避免多线程竞争全局锁。

- 默认开启：`-XX:+UseTLAB`。
- 默认占 Eden 的 1%：`-XX:TLABWasteTargetPercent=1`。
- TLAB 失败 → 申请新 TLAB。
- 新 TLAB 失败 → Eden 加锁分配。
- Eden 不足 → 触发 Minor GC。

```text
分配流程：
1. 当前线程 TLAB 是否足够？
   ├─ 是 → 在 TLAB 中分配（无锁，最快）
   └─ 否 → 申请新 TLAB
2. 新 TLAB 是否成功？
   ├─ 是 → 在新 TLAB 中分配
   └─ 否 → Eden 加锁分配
3. Eden 是否足够？
   ├─ 是 → 加锁分配（CAS）
   └─ 否 → 触发 Minor GC
4. Minor GC 后 Eden 是否足够？
   ├─ 是 → 分配
   └─ 否 → 大对象直接进老年代 / 空间分配担保
```

## 大对象直接进入老年代

大对象（如长数组、超大字符串）需要连续内存空间，在新生代间反复复制代价高。`-XX:PretenureSizeThreshold` 设置阈值，超过直接进老年代。

```bash
-XX:PretenureSizeThreshold=1048576   # 1MB，超过直接进老年代
```

注意：

- 该参数只对 Serial 和 ParNew 生效，Parallel Scavenge 不支持。
- G1 用 Humongous Region 处理大对象（超过 Region 50%）。
- 频繁创建大对象会快速填满老年代，触发 Full GC。

## 长期存活的对象进入老年代

JVM 为对象定义年龄计数器（4 位，存对象头 Mark Word）。

- 对象在 Eden 出生。
- 经历 Minor GC 仍存活，复制到 Survivor，年龄 +1。
- 年龄达到 `-XX:MaxTenuringThreshold`（默认 15），晋升老年代。

```bash
-XX:MaxTenuringThreshold=15   # 默认 15，最大也是 15（4 位限制）
```

为什么最大是 15？因为对象头 Mark Word 中分代年龄字段是 4 位，4 位无符号最大 15。

## 动态对象年龄判定

JVM 不一定要求年龄达到 MaxTenuringThreshold 才晋升。如果在 Survivor 中相同年龄所有对象大小的总和大于 Survivor 空间的一半，年龄大于或等于该年龄的对象直接进入老年代。

```text
Survivor 容量 100MB
年龄 2 的对象总大小 = 60MB > 50MB（一半）
→ 年龄 ≥ 2 的对象全部晋升老年代
```

这个机制让"批量晋升"成为可能，避免 Survivor 反复复制。

## 空间分配担保

Minor GC 前检查老年代是否容纳新生代所有对象（最坏情况）：

```text
Minor GC 前：
1. 检查老年代最大可用连续空间是否 > 新生代所有对象总空间
   ├─ 是 → Minor GC 安全
   └─ 否 → 检查 HandlePromotionFailure 是否允许冒险
              ├─ 不允许 → Full GC
              └─ 允许 → 检查老年代是否 > 历次晋升平均大小
                          ├─ 是 → 尝试 Minor GC（冒险）
                          │       ├─ 成功 → 完成
                          │       └─ 失败（Survivor 装不下）→ 老年代担保
                          │               ├─ 老年代够 → 晋升老年代
                          │               └─ 老年代不够 → Full GC
                          └─ 否 → Full GC
```

`-XX:HandlePromotionFailure` 控制是否允许冒险（JDK 6 后默认允许）。

## G1 的对象分配

G1 的对象分配略有不同：

- 优先 TLAB（Eden Region 内）。
- 大对象（超过 Region 50%）直接进 Humongous Region。
- 不存在"新生代连续空间"概念，每个 Region 独立。
- IHOP 阈值（默认 45%）触发并发标记，进入 Mixed GC 周期。

## 代码示例

观察对象分配行为：

```java
// -Xms1g -Xmx1g -Xmn512m -XX:SurvivorRatio=8 -XX:+PrintGCDetails
public class AllocationDemo {
    public static void main(String[] args) {
        // 1. 小对象在 Eden 分配（通过 TLAB）
        byte[] small = new byte[1024];      // 1KB

        // 2. 中等对象也在 Eden 分配
        byte[] medium = new byte[1024 * 100]; // 100KB

        // 3. 大对象直接进老年代（需 Serial/ParNew）
        // -XX:PretenureSizeThreshold=1048576
        byte[] big = new byte[1024 * 1024 * 2]; // 2MB

        // 4. 长期存活对象晋升
        List<byte[]> list = new ArrayList<>();
        for (int i = 0; i < 100; i++) {
            list.add(new byte[1024 * 100]); // 持续持有，年龄增长
        }
    }
}
```

## 实战场景

| 场景 | 关注点 | 优化 |
|------|--------|------|
| Minor GC 频繁 | Eden 太小 | 增大 `-Xmn` 或 `-XX:NewRatio` |
| 老年代增长快 | 大对象 / 长期对象 | 控制对象大小，排查缓存 |
| Full GC 频繁 | 空间分配担保失败 | 增大堆，调整 SurvivorRatio |
| G1 Humongous 多 | 大对象多 | 增大 `-XX:G1HeapRegionSize` |

## 深挖追问

### 为什么大对象直接进老年代？

大对象在新生代间复制代价高（每次 Minor GC 都要复制）。直接进老年代避免复制开销。但代价是老年代空间被快速占用，可能触发 Full GC。所以应该控制大对象的创建频率。

### MaxTenuringThreshold 为什么最大是 15？

对象头 Mark Word 中分代年龄字段是 4 位，4 位无符号最大 15。如果要支持更大年龄，需要扩大字段，影响对象头布局。15 对绝大多数场景已经够用。

### 动态年龄判断是怎么计算的？

Minor GC 后，JVM 计算 Survivor 中每个年龄的对象总大小。从年龄 1 开始累加，当累加和超过 Survivor 一半时，年龄 ≥ 当前年龄的对象全部晋升老年代。这种"批量晋升"避免 Survivor 反复复制。

### 空间分配担保什么时候触发 Full GC？

- 老年代最大可用连续空间 < 新生代所有对象总空间，且不允许冒险 → Full GC。
- 允许冒险但老年代 < 历次晋升平均大小 → Full GC。
- Minor GC 时 Survivor 装不下，老年代也装不下 → Full GC。

## 易错点

- 把 TLAB 当作"栈上分配"，TLAB 在堆的 Eden 区。
- 以为 PretenureSizeThreshold 对所有 GC 生效，只对 Serial/ParNew 有效。
- 以为 MaxTenuringThreshold 可以设很大，最大 15（4 位限制）。
- 把"动态年龄判断"和"年龄达阈值"混淆，前者是批量晋升。
- 忽略空间分配担保，导致 Survivor 装不下时频繁 Full GC。

## 总结

对象分配是分层次的策略：优先 TLAB（无锁），失败走 Eden 加锁；Eden 满触发 Minor GC；大对象直接进老年代；长期存活对象按年龄晋升；Survivor 装不下通过空间分配担保进老年代；动态年龄判断让相同年龄对象总和超 Survivor 一半时整体晋升。理解这套规则能帮助排查 Full GC 频繁、老年代增长过快等问题。
