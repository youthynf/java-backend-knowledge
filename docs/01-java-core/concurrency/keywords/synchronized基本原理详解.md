# synchronized基本原理详解

synchronized基本原理详解
概述
synchronized 是 Java 中用于实现线程同步的关键字，主要作用是在多线程环境下确保同一时刻只有一个线程可以访问被修饰的代码块或方法，保证：
原子性：对共享变量的操作不可分割；
可见性：一个线程对共享变量的修改对其他线程立即可见；
有序性：防止指令重排序；

使用方式
修饰代码块：需指定锁对象；
修饰实例方法：锁定当前对象实例 (this)；
修饰静态方法：锁定类对象 (Class 对象)；

实现原理
基于 JVM 的 Monitor（监视器）机制，通过字节码指令实现：
•  monitorenter：获取锁（计数器加1）
•  monitorexit：释放锁（计数器减1）
每个对象关联一个 monitor 锁，具有以下特性：
•  同一时刻只能被一个线程获取；
•  支持可重入（计数器机制）；
•  获取失败会进入阻塞状态，加入同步队列；

优化机制
锁消除（Lock Elimination）
JIT 编译器通过逃逸分析，消除不可能存在共享资源竞争的锁。
锁粗化（Lock Coarsening）
将多个连续的锁操作合并为一个更大范围的锁，减少不必要的加锁/解锁开销。
轻量级锁（Lightweight Lock）
适用场景：无实际竞争或只有短暂竞争；
实现：通过 CAS 操作尝试获取锁；
失败处理：CAS 失败后直接膨胀为重量级锁（不重试）。

偏向锁（Biased Locking）
适用场景：长期只有一个线程访问同步块；
特点：首次通过 CAS 设置线程ID，后续无需同步操作；
现状：JDK 15 已移除；

自适应自旋锁（Adaptive Spinning）
自旋次数动态调整（基于历史成功率），减少线程阻塞/唤醒的开销
锁升级过程（膨胀过程）
无锁状态
•  新创建对象初始状态，Mark Word 存储对象哈希码和分代年龄等信息。

偏向锁
•  通过 CAS 将当前申请锁的线程ID写入Mark Word，偏向状态置1，锁定标志保持为01（偏向锁和无锁共享相同的锁标志）；
•  适合单线程重复访问场景；
•  发生竞争或重偏向次数超过阈值（默认20次）时撤销偏向锁；

轻量级锁
•  线程栈中创建 Lock Record，用于存储锁对象目前的Mark Word备份，官方称为Displace Mark Word；
•  通过 CAS 尝试获取锁，将当前锁对象的Mark Word拷贝到Lock Record中，并且将Mark Word更新为指向Lock Record的指针，如果更新成功则拥有该对象的锁，并且对象的Mark Word的锁标志位更新为00；
•  竞争失败时直接膨胀（不自旋重试），没有获得锁的线程会被阻塞。此时锁标志位为10，Mark Word中存储的是指向重量级锁的指针；

重量级锁
•  关联操作系统互斥量（mutex）；
•  竞争失败线程进入阻塞状态；
•  涉及用户态/内核态切换，开销最大；
•  发生轻量级锁竞争/长时间持有轻量级锁/调用了wait()方法都会导致升级重量级锁；

特别说明：轻量级锁不存在自旋，只有重量级锁加锁失败才会自旋。重量级锁加锁失败，会多次尝试cas和自适应自旋，如果一直加锁失败，就会阻塞当前线程等待唤醒
轻量级锁竞争没有自旋的原因其实是其设计并不是用于处理过于激烈的竞争场景，而是为了应对线程之间交替获取锁的场景。

优缺点分析
优点
•  语法简单，JVM 自动管理锁
•  自动处理加锁/解锁过程
•  默认非公平锁，吞吐量较高

缺点
•  灵活性不足：
•  不支持中断；
•  不支持超时获取锁；
•  无法查询锁状态；
•  功能限制，单一条件变量（对比 ReentrantLock 的多个条件），无法实现公平锁（除非通过 ReentrantLock）。
•  性能问题，重量级锁状态下性能较差，不区分读/写场景。

## 面试总结

围绕「synchronized基本原理详解」，面试官通常不只考概念定义，更关注你能否把机制、使用场景和线上问题串起来。

### 核心回答

1. synchronized 基于对象监视器实现互斥和可见性，进入/退出临界区天然建立 happens-before。
2. 锁对象不同，保护的共享资源范围就不同；锁升级会经历偏向、轻量级、重量级等阶段。
3. 和 ReentrantLock 相比，synchronized 语法简单自动释放，ReentrantLock 提供可中断、超时、公平、多个条件队列。

### 高频追问

- 锁升级为什么不能随意降级？
- wait/notify 为什么必须在同步块内？
- 如何选择 synchronized 和 ReentrantLock？

### 实战落地

- **选型前**：先判断是互斥访问、线程协作、任务编排，还是限流隔离。
- **编码时**：控制共享变量范围，明确锁对象、超时策略、异常处理和资源释放。
- **上线后**：观察线程数、队列长度、阻塞时间、拒绝次数和 RT 抖动，必要时用线程 Dump 验证。

### 易错点

- 不要锁可变对象或字符串常量。
- 减少锁粒度但不能破坏共享状态的一致性。
