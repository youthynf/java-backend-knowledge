# 线程状态如何转换？

## 核心概念

Java 线程状态定义在 `Thread.State` 中，主要包括：

- `NEW`：线程对象已创建，还未调用 `start()`。
- `RUNNABLE`：可运行状态，可能正在运行，也可能等待 CPU 时间片。
- `BLOCKED`：等待进入 `synchronized` 保护的临界区。
- `WAITING`：无限期等待其他线程唤醒。
- `TIMED_WAITING`：带超时时间的等待。
- `TERMINATED`：线程执行结束。

## 面试官想考什么

- 是否能区分 Java 线程状态和操作系统线程状态。
- 是否能区分 `BLOCKED`、`WAITING`、`TIMED_WAITING`。
- 是否知道不同 API 会让线程进入什么状态，以及是否释放锁。

## 标准回答

线程创建后处于 `NEW`，调用 `start()` 后进入 `RUNNABLE`。如果竞争 `synchronized` 锁失败，会进入 `BLOCKED`。如果调用 `Object.wait()`、`Thread.join()`、`LockSupport.park()` 等等待唤醒的方法，会进入 `WAITING`；如果调用带超时版本、`Thread.sleep()`，会进入 `TIMED_WAITING`。任务执行结束或异常退出后进入 `TERMINATED`。

要注意，Java 的 `RUNNABLE` 同时覆盖了操作系统层面的 ready 和 running；看到 `RUNNABLE` 不代表一定正在占用 CPU。

## 深挖追问

### `BLOCKED` 和 `WAITING` 有什么区别？

- `BLOCKED`：被动等待进入 monitor 锁，常见于竞争 `synchronized`。
- `WAITING`：线程主动等待某个条件或信号，例如 `wait()`、`join()`、`park()`。

### `sleep()` 会释放锁吗？`wait()` 呢？

`sleep()` 不释放已持有的 monitor 锁；`wait()` 必须在持有对象 monitor 时调用，调用后会释放该 monitor，等待被 `notify/notifyAll` 唤醒或中断。

### 中断会让线程进入什么状态？

中断本身只是设置中断标记。如果线程正在 `sleep/wait/join` 等可中断阻塞中，会抛出 `InterruptedException` 并清除中断标记。

## 实战场景/代码示例

```java
synchronized (lock) {
    while (!condition) {
        lock.wait(); // WAITING，并释放 lock 的 monitor
    }
}
```

排查线程状态：

```bash
jcmd <pid> Thread.print
# 或 jstack <pid>
```

## 易错点/总结

- `RUNNABLE` 不等于一定在运行。
- `BLOCKED` 主要对应进入 `synchronized` 失败，不是所有阻塞都叫 `BLOCKED`。
- `sleep()` 不释放锁，`wait()` 会释放调用对象的 monitor。
- 面试回答要把“调用什么 API → 进入什么状态 → 如何恢复”讲清楚。
