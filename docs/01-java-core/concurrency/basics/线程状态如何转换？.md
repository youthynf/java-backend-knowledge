# 线程状态如何转换

## 核心概念

Java 线程状态由 `java.lang.Thread.State` 枚举定义，共 6 种：`NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING`、`TERMINATED`。这是 JVM 层面的抽象，和操作系统层面的线程状态（如 Linux 的 ready/running/blocked）不是一回事——Java 把 OS 的 ready 和 running 合并成 `RUNNABLE`。

理解状态转换的关键是"调用什么 API → 进入什么状态 → 如何恢复"。同一个"线程看起来停住了"，可能是因为等锁（`BLOCKED`）、等通知（`WAITING`）、等超时（`TIMED_WAITING`），三者的触发条件和恢复路径完全不同，排查问题时不能混为一谈。

## 标准回答

Java 线程 6 种状态的转换路径：

1. **NEW → RUNNABLE**：`thread.start()`。
2. **RUNNABLE → BLOCKED**：竞争 `synchronized` 锁失败，进入 monitor 的 entry list。
3. **RUNNABLE → WAITING**：调用 `Object.wait()` / `Thread.join()` / `LockSupport.park()` / `Condition.await()`。
4. **RUNNABLE → TIMED_WAITING**：调用带超时的 `wait(timeout)` / `join(timeout)` / `sleep(ms)` / `parkNanos` / `await(timeout)`。
5. **WAITING / TIMED_WAITING → RUNNABLE**：被 `notify/notifyAll`、`unpark`、目标线程结束、中断、超时触发。
6. **BLOCKED → RUNNABLE**：拿到 monitor 锁。
7. **RUNNABLE → TERMINATED**：`run()` 正常返回或抛出未捕获异常。

注意 Java 的 `RUNNABLE` 包含了 OS 的 ready 和 running，看到 `RUNNABLE` 不代表线程正在占用 CPU——它可能正在等 IO、等 CPU 时间片。`BLOCKED` 只对应 `synchronized` 等锁，`ReentrantLock.lock()` 等待时是 `WAITING`，不是 `BLOCKED`。

## 实现原理

### 六种状态的定义

| 状态 | 含义 | 触发条件 |
|------|------|----------|
| NEW | 已创建未启动 | `new Thread()` 后未调 `start()` |
| RUNNABLE | 可运行（含就绪和运行） | `start()` 后；被唤醒拿到锁后 |
| BLOCKED | 等 synchronized monitor | 竞争内置锁失败 |
| WAITING | 无限期等待 | `wait()` / `join()` / `park()` / `await()` 无超时 |
| TIMED_WAITING | 限时等待 | `sleep(ms)` / `wait(ms)` / `join(ms)` / `parkNanos` / `await(ms)` |
| TERMINATED | 终止 | `run()` 正常结束或异常退出 |

### 为什么 BLOCKED 只对应 synchronized？

`BLOCKED` 在 JVM 规范里定义为"等待 monitor 锁"。`synchronized` 用 monitor 实现，竞争失败进 `BLOCKED`。而 `ReentrantLock` 基于 AQS，等待时调用 `LockSupport.park()`，状态是 `WAITING`，不会进 `BLOCKED`。所以线程 dump 里 `BLOCKED` 多，通常是 synchronized 用得多；不是"所有阻塞都叫 BLOCKED"。

### 中断如何影响状态？

中断本身不改变线程状态，只设置 `interruptFlag`。如果线程正处在可中断阻塞（`WAITING` / `TIMED_WAITING` 中由 `sleep/wait/join/park` 触发的）中，JVM 会把它唤醒，抛 `InterruptedException`，并清除中断标志。状态会回到 `RUNNABLE`，然后异常向上传播，最终如果没被 catch，线程进入 `TERMINATED`。

### 状态转换图

```
        new Thread()             start()
NEW ─────────────────► [对象已创建] ──────► RUNNABLE
                                                  │
                                                  ├── synchronized 竞争失败 ──► BLOCKED
                                                  │                              │
                                                  │                              │ 拿到锁
                                                  │                              ▼
                                                  │◄─────────────────────────────┘
                                                  │
                                                  ├── wait()/join()/park() ──► WAITING
                                                  │   notify/notifyAll/unpark    │
                                                  │◄─────────────────────────────┘
                                                  │
                                                  ├── sleep(ms)/wait(ms)/join(ms) ──► TIMED_WAITING
                                                  │   超时/notify/unpark              │
                                                  │◄─────────────────────────────────┘
                                                  │
                                                  └── run() 结束 / 未捕获异常 ──► TERMINATED
```

## 代码示例

### 观察六种状态

```java
public class ThreadStateDemo {

    public static void main(String[] args) throws InterruptedException {
        // NEW
        Thread t = new Thread(() -> {
            try { Thread.sleep(100); } catch (InterruptedException ignored) {}
        });
        System.out.println("state-1: " + t.getState()); // NEW

        // RUNNABLE
        t.start();
        System.out.println("state-2: " + t.getState()); // RUNNABLE

        // TIMED_WAITING (sleep)
        Thread.sleep(50);
        System.out.println("state-3: " + t.getState()); // TIMED_WAITING

        // TERMINATED
        t.join();
        System.out.println("state-4: " + t.getState()); // TERMINATED
    }
}
```

### BLOCKED vs WAITING

```java
public class BlockedVsWaiting {
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        Thread holder = new Thread(() -> {
            synchronized (lock) {
                try { Thread.sleep(60_000); } catch (InterruptedException ignored) {}
            }
        });
        holder.start();
        Thread.sleep(100);

        Thread syncWaiter = new Thread(() -> {
            synchronized (lock) { System.out.println("got lock"); }
        });
        syncWaiter.start();
        Thread.sleep(100);
        System.out.println("syncWaiter: " + syncWaiter.getState()); // BLOCKED

        Thread lockWaiter = new Thread(() -> {
            ReentrantLock rl = new ReentrantLock();
            rl.lock();
            try { rl.lockInterruptibly(); } catch (InterruptedException ignored) {}
            // 上面是 demo 用法，实际场景见下文
        });
        // 实际观察 ReentrantLock 等待时的状态：
        ReentrantLock rl = new ReentrantLock();
        rl.lock();
        Thread rlWaiter = new Thread(() -> {
            rl.lock(); // 等待中
        });
        rlWaiter.start();
        Thread.sleep(100);
        System.out.println("rlWaiter: " + rlWaiter.getState()); // WAITING
    }
}
```

### 用 jstack 排查状态

```bash
# 找到 Java 进程 PID
jps -l

# 打印线程栈
jstack <pid> > thread.dump

# 或用 jcmd
jcmd <pid> Thread.print > thread.dump
```

线程 dump 中会看到 `java.lang.Thread.State: BLOCKED (on object monitor)` / `WAITING (on object monitor)` / `WAITING (parking)` 等明细。

## 实战场景

| 场景 | 表现 | 排查方法 |
|------|------|----------|
| 接口卡死 | 大量线程 `BLOCKED` 在同一把 synchronized | jstack 找持锁线程，看它在干什么 |
| 线程池任务积压 | worker 全部 `WAITING (parking)` 在队列 take | 检查任务执行时间、是否有死循环 |
| 死锁 | 两个线程互相 `BLOCKED` | jstack 自动检测 `Found Java-level deadlock` |
| CPU 飙高 | 多个线程 `RUNNABLE` 且占用 CPU | `top -Hp <pid>` 找高 CPU 线程，jstack 对应 nid |
| 中断未响应 | `WAITING` 线程调 `interrupt()` 仍不退出 | 检查任务是否吞掉 `InterruptedException` |

## 深挖追问

### sleep()、wait()、park() 各让线程进入什么状态？

- `Thread.sleep(ms)` → `TIMED_WAITING`
- `Object.wait()` → `WAITING`；`Object.wait(ms)` → `TIMED_WAITING`
- `Thread.join()` → `WAITING`；`join(ms)` → `TIMED_WAITING`
- `LockSupport.park()` → `WAITING`；`parkNanos` / `parkUntil` → `TIMED_WAITING`

### wait() 为什么必须放在 while 循环里？

防止"虚假唤醒"（spurious wakeup）。JVM 规范允许 `wait()` 在没有 `notify` 的情况下自己返回，因此醒来后必须再次检查条件：

```java
synchronized (lock) {
    while (!condition) {  // 不是 if
        lock.wait();
    }
    // 条件满足，执行业务
}
```

### 一个线程能 start 多次吗？

不能。Thread 内部 `threadStatus` 字段记录状态，`start()` 只允许在 `0`（NEW）状态调用。再次 `start()` 抛 `IllegalThreadStateException`。

### wait() 释放哪个锁？

`obj.wait()` 释放 `obj` 这个对象的 monitor，**不释放**当前线程持有的其他对象锁。所以如果一个线程持有 A、B 两把锁，调 `A.wait()` 只释放 A，B 仍持有。

### Interrupt 后线程会立即终止吗？

不会。`interrupt()` 只设置标志位。如果线程在 `sleep/wait/join/park` 中，会被唤醒并抛 `InterruptedException`，但状态会先回到 `RUNNABLE`，再抛异常。业务不 catch 也不重新抛，线程会继续跑。所以中断是"协作式"的。

## 易错点

- 把 `RUNNABLE` 等同于"正在执行"——它包含 ready，可能正在等 CPU 或 IO。
- 把所有阻塞都叫 `BLOCKED`——只有 synchronized 等锁才是 `BLOCKED`，`ReentrantLock` 等待是 `WAITING`。
- `wait()` 用 `if` 而不是 `while`——遇到虚假唤醒或 `notifyAll` 后条件不满足，会误执行。
- 误以为 `interrupt()` 会停线程——只是设标志，业务必须主动响应。
- `sleep()` 不释放锁却拿来"等条件"——其他线程拿不到锁推不动条件，可能死锁。

## 总结

Java 线程 6 种状态的核心是"调用什么 API 进入什么状态、怎么恢复"。`RUNNABLE` 包含 OS 的 ready 和 running，`BLOCKED` 只对应 synchronized，`ReentrantLock` 等待是 `WAITING`。`sleep` 不释放锁、`wait` 释放 monitor、`park` 不需要锁。中断是协作式而非强制式。排查生产问题用 `jstack` / `jcmd Thread.print` 看线程状态明细，是定位死锁、CPU 飙高、任务积压的第一步。
