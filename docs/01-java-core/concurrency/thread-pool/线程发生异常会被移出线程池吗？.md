# 线程发生异常会被移出线程池吗

## 核心概念

线程池中的工作线程执行任务时如果抛出未捕获的运行时异常或错误，**当前工作线程会异常终止**，从线程池中移出。但 `ThreadPoolExecutor` 会自动创建新的工作线程补位，让线程池继续可用。所以"线程被移出"和"线程池缩水"是两件事，需要分开看。

异常处理还和提交方式有关：`execute()` 抛出的异常会冒到 `UncaughtExceptionHandler`；`submit()` 抛出的异常被封装进 `Future`，必须调 `future.get()` 才能感知。

## 标准回答

工作线程执行任务时抛出未捕获异常的流程：

1. 任务 `run()` 抛出 `RuntimeException`（或 `Error`）。
2. `runWorker` 的 `try/catch` 捕获到 `Throwable`，调用 `afterExecute(task, ex)`。
3. `finally` 块执行完毕，进入 `processWorkerExit`，当前 Worker 被从 `workers` 集合移除。
4. `processWorkerExit` 内部调用 `addWorker(null, false)` 尝试补一个新的工作线程。
5. 异常根据提交方式分流：
   - `execute()` 提交：异常经 `Thread.dispatchUncaughtException` 走 `UncaughtExceptionHandler`，默认打印到 `System.err`。
   - `submit()` 提交：任务被包装成 `FutureTask`，异常被 `setException` 存进 `Future`，`future.get()` 时抛 `ExecutionException`。

所以：**线程会被移出，但线程池不会缩水**（除非线程池已关闭或达到拒绝补位的条件）。

## 实现原理

### runWorker 的异常处理

```java
final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    w.firstTask = null;
    w.unlock();
    boolean completedAbruptly = true;
    try {
        while (task != null || (task = getTask()) != null) {
            w.lock();
            // ... 状态检查 ...
            try {
                beforeExecute(wt, task);
                task.run();              // 任务在这里抛异常
                afterExecute(task, null);
            } catch (Throwable ex) {
                afterExecute(task, ex);   // 异常被 catch，传给 afterExecute
                throw ex;                 // 重新抛出，触发线程退出
            } finally {
                task = null;
                w.completedTasks++;
                w.unlock();
            }
        }
        completedAbruptly = false;
    } finally {
        processWorkerExit(w, completedAbruptly);   // true = 异常退出
    }
}
```

注意 `throw ex` 把异常重新抛出，工作线程的 `run()` 方法因异常退出，`Thread` 终止。`completedAbruptly = true` 标记是"非正常退出"。

### processWorkerExit 的补位逻辑

```java
private void processWorkerExit(Worker w, boolean completedAbruptly) {
    if (completedAbruptly)
        decrementWorkerCount();          // 异常退出需要手动减计数
    // ... 正常退出时 getTask 已减计数 ...

    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        completedTaskCount += w.completedTasks;
        workers.remove(w);               // 从 workers 移除
    } finally {
        mainLock.unlock();
    }

    tryTerminate();                       // 尝试终止线程池（如果该终止了）

    int c = ctl.get();
    if (runStateLessThan(c, STOP)) {      // 还在 RUNNING 或 SHUTDOWN
        if (!completedAbruptly) {
            int min = allowCoreThreadTimeOut ? 0 : corePoolSize;
            if (min == 0 && !workQueue.isEmpty())
                min = 1;
            if (workerCountOf(c) >= min)
                return;                   // 数量够，不补
        }
        addWorker(null, false);           // 补一个非核心线程
    }
}
```

关键点：异常退出时（`completedAbruptly = true`）跳过 "数量够不够" 的判断，直接调 `addWorker` 补一个。这是线程池的容错机制——异常退出不应该让线程池缩水。

### execute 与 submit 的异常路径差异

`submit` 把任务包装成 `FutureTask`：

```java
public Future<?> submit(Runnable task) {
    RunnableFuture<Void> ftask = newTaskFor(task, null);
    execute(ftask);
    return ftask;
}
```

`FutureTask.run()` 内部 try/catch 所有异常：

```java
public void run() {
    try {
        Callable<V> c = callable;
        V result = c.call();
        set(result);
    } catch (Throwable ex) {
        setException(ex);   // 异常存进 Future，不抛出
    }
}
```

所以 `submit` 提交的任务即使抛异常，`runWorker` 也看不到——异常被 `FutureTask` 自己吞了，存在 `outcome` 字段里，`get()` 时包装成 `ExecutionException` 抛出。

`execute` 提交普通 `Runnable`，异常直接冒到 `runWorker` 的 catch，重新抛出，触发线程的 `UncaughtExceptionHandler`。

## 代码示例

### execute 异常路径

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        1, 1, 0L, TimeUnit.SECONDS, new LinkedBlockingQueue<>(),
        r -> {
            Thread t = new Thread(r);
            t.setUncaughtExceptionHandler((th, ex) ->
                log.error("线程 {} 未捕获异常", th.getName(), ex));
            return t;
        });

executor.execute(() -> { throw new RuntimeException("bad task"); });
// UncaughtExceptionHandler 被触发，日志记录异常
// 线程池自动补位，可以继续提交任务
```

### submit 异常路径

```java
Future<?> future = executor.submit(() -> {
    throw new IllegalStateException("bad task");
});

try {
    future.get();
} catch (ExecutionException e) {
    // e.getCause() 才是真正的异常
    log.error("任务执行失败", e.getCause());
}
// 如果不调 get()，异常被静默保存在 Future 中，无人感知
```

### 推荐做法：任务内部兜底

```java
executor.execute(() -> {
    try {
        doBiz();
    } catch (Exception e) {
        log.error("异步任务执行失败", e);
        // 视情况发送告警、补偿、回滚
    }
});
```

### 重写 afterExecute 统一处理

```java
class LoggingExecutor extends ThreadPoolExecutor {
    public LoggingExecutor(int core, int max, BlockingQueue<Runnable> queue) {
        super(core, max, 0L, TimeUnit.SECONDS, queue);
    }

    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        // submit 提交的 FutureTask 异常不会进 t
        if (t == null && r instanceof Future<?> && ((Future<?>) r).isDone()) {
            try { ((Future<?>) r).get(); }
            catch (CancellationException ce) { /* 被取消 */ }
            catch (ExecutionException ee) { t = ee.getCause(); }
            catch (InterruptedException ie) { Thread.currentThread().interrupt(); }
        }
        if (t != null) {
            log.error("任务执行异常", t);
        }
    }
}
```

## 实战场景

| 场景 | 处理方式 | 注意点 |
|------|----------|--------|
| 异步日志/埋点 | 任务内部 try-catch | 失败不影响主链路 |
| 关键业务异步处理 | submit + get + 补偿 | 不调 get 异常会丢 |
| 监控任务异常率 | 自定义 ThreadFactory + UncaughtExceptionHandler | 配合告警系统 |
| 统一异常收集 | 重写 afterExecute | 注意 submit 的 FutureTask 需特殊处理 |

## 深挖追问

### execute 和 submit 的异常有什么区别？

- `execute`：异常直接冒到工作线程的 `run()`，触发 `UncaughtExceptionHandler`，控制台会看到栈。
- `submit`：任务被 `FutureTask` 包装，异常被 `setException` 存进 `Future`，工作线程不感知异常，必须 `future.get()` 才能拿到。

两者都会让当前任务的工作线程退出（execute 路径）或正常复用（submit 路径），但线程池都会补位。

### 线程异常退出会让线程池线程变少吗？

不会。`processWorkerExit` 在异常退出时跳过数量检查直接 `addWorker`，补一个新线程。只有线程池已关闭（STOP 以上）或 `addWorker` 失败（如达到 max）才不补。

### submit 不调 get 会发生什么？

异常被静默保存在 `Future` 的 `outcome` 字段。`Future` 对象如果一直被引用，异常信息可以事后通过 `get()` 取出；如果 `Future` 被丢弃，异常就丢失了。生产中常见"任务莫名失败但没日志"的根因。

### 如何让所有任务异常都被记录？

三种方式：

1. 任务内部统一 try-catch（最简单，最推荐）。
2. 自定义 `ThreadFactory` 设置 `UncaughtExceptionHandler`（只对 `execute` 路径有效）。
3. 重写 `afterExecute`（对 `execute` 和 `submit` 都有效，但 `submit` 要额外处理 `FutureTask`）。

### Error 也会被处理吗？

会。`runWorker` 的 catch 是 `Throwable`，能捕获 `Error`（如 `OutOfMemoryError`、`StackOverflowError`）。但 `Error` 通常意味着 JVM 严重问题，仅靠线程池补位解决不了根因。

## 易错点

- 用 `submit` 提交任务但不调 `get()`，异常被静默吞掉，线上没日志没告警。
- 以为 `UncaughtExceptionHandler` 能捕获 `submit` 任务抛出的异常，实际只对 `execute` 有效。
- `afterExecute` 直接信任第二个参数 `Throwable t`，对 `submit` 提交的 `FutureTask` 永远是 null。
- 异常吞掉后不发送告警，等问题暴露已经晚了。

## 总结

工作线程抛未捕获异常会退出，但线程池会自动补位，所以线程池"看起来"不会缩水。异常处理路径因提交方式而异：`execute` 走 `UncaughtExceptionHandler`，`submit` 把异常藏进 `Future`。生产中推荐任务内部 try-catch 兜底，或重写 `afterExecute` 做统一收集，避免异常被静默丢失。
