# 如何撤回提交给线程池中的任务

## 核心概念

提交到线程池的任务能否撤回，取决于它处于哪个阶段：

1. **还在队列里**：可以从队列移除，或通过 `Future.cancel(false)` 取消。
2. **已被线程取走在执行**：只能通过中断协作式取消，调用 `Future.cancel(true)`，能否真停下取决于任务是否响应中断。
3. **已执行完成**：无法撤回，只能业务补偿。

Java 没有安全的"强制杀死线程"机制，`Thread.stop()` 已被废弃。撤回本质是**协作式取消**——发一个信号，由任务自己决定何时退出。

## 标准回答

撤回任务有两条路径：

- **通过 `Future.cancel(mayInterruptIfRunning)`**：适用于 `submit()` 提交的任务。`cancel(false)` 只取消未启动的任务（已启动的不动）；`cancel(true)` 在任务已启动时给执行线程发中断信号。
- **通过 `ThreadPoolExecutor.getQueue().remove(task)`**：适用于 `execute()` 提交、还在队列中的任务。要求能拿到同一个 `Runnable` 对象引用。

运行中的任务无法被强杀，只能靠任务代码响应中断（检查 `isInterrupted()` 或处理 `InterruptedException`）主动退出。

## 实现原理

### Future.cancel 的源码语义

`FutureTask.cancel` 的核心逻辑：

```java
public boolean cancel(boolean mayInterruptIfRunning) {
    if (!(state == NEW &&
          UNSAFE.compareAndSwapInt(this, stateOffset, NEW,
              mayInterruptIfRunning ? INTERRUPTING : CANCELLED)))
        return false;   // 任务已完成或已取消，取消失败
    try {
        if (mayInterruptIfRunning) {
            try {
                Thread t = runner;
                if (t != null)
                    t.interrupt();    // 给执行线程发中断
            } finally {
                UNSAFE.putOrderedInt(this, stateOffset, INTERRUPTED);
            }
        }
    } finally {
        finishCompletion();
    }
    return true;
}
```

`cancel(false)`：状态从 NEW → CANCELLED，任务还没启动的话不会被调度执行；已启动的继续跑完。
`cancel(true)`：状态从 NEW → INTERRUPTING → INTERRUPTED，并给 runner 线程发 `interrupt()`。任务能不能停下取决于是否响应中断。

### ThreadPoolExecutor.remove

```java
public boolean remove(Runnable task) {
    boolean removed = workQueue.remove(task);
    tryTerminate();   // 如果 SHUTDOWN 且队列空，触发终止
    return removed;
}
```

`workQueue.remove(task)` 用 `BlockingQueue.remove(Object)`，依赖 `equals` 比较。如果是 `execute(runnable)` 提交的，队列里存的就是原始 `Runnable`，能匹配；如果是 `submit(callable)` 提交的，队列里是 `FutureTask`，需要传入同一个 `Future` 引用。

### 任务状态对 cancel 的影响

| 任务状态 | cancel(false) | cancel(true) |
|----------|---------------|--------------|
| 还在队列，未启动 | 取消，不再执行 | 取消，不再执行 |
| 正在执行 | 无效，任务继续 | 发中断信号，看任务是否响应 |
| 已完成 | 返回 false | 返回 false |
| 已取消 | 返回 false | 返回 false |

## 代码示例

### cancel(true) 撤回运行中任务

```java
Future<?> future = executor.submit(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        doOneBatch();   // 假设每个 batch 耗时较短
    }
    cleanup();
});

// 等一会儿再取消
TimeUnit.SECONDS.sleep(5);
boolean cancelled = future.cancel(true);   // 发中断
System.out.println("取消结果: " + cancelled);
```

任务内部必须响应中断，否则 `cancel(true)` 只是发了信号，任务继续跑：

```java
executor.submit(() -> {
    try {
        while (true) {
            String item = queue.poll(1, TimeUnit.SECONDS);  // 可被中断唤醒
            if (item != null) process(item);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();   // 重新设置标志
        cleanup();
    }
});
```

### remove 撤回排队中任务

```java
Runnable task = () -> {
    try { TimeUnit.SECONDS.sleep(3); } catch (InterruptedException ignored) {}
    System.out.println("任务执行");
};

ThreadPoolExecutor executor = new ThreadPoolExecutor(
        1, 1, 0L, TimeUnit.SECONDS, new LinkedBlockingQueue<>());

// 先占住核心线程
executor.execute(() -> { try { TimeUnit.SECONDS.sleep(2); } catch (InterruptedException ignored) {} });

// 再提交一个会进队列的任务
executor.execute(task);
TimeUnit.MILLISECONDS.sleep(100);

// 任务还在队列里，可以移除
boolean removed = executor.remove(task);
System.out.println("移除结果: " + removed);
```

### 业务级取消：状态标记

对于无法响应中断的任务（如调用第三方接口），用业务标记：

```java
class CancellableJob {
    private final AtomicBoolean cancelled = new AtomicBoolean(false);

    public void run() {
        for (Item item : items) {
            if (cancelled.get()) return;
            callExternal(item);
        }
    }

    public void cancel() {
        cancelled.set(true);
    }
}
```

## 实战场景

| 场景 | 撤回方式 | 注意点 |
|------|----------|--------|
| 长耗时计算任务超时 | submit + Future.cancel(true) | 任务必须响应中断 |
| 用户取消订单触发任务回滚 | 业务标记 + remove | remove 只对队列中任务有效 |
| 限流场景丢掉积压任务 | drainQueue + 重新入队 | 需谨慎，可能丢数据 |
| 防止重复提交 | ConcurrentHashMap 存 Future + cancel | 配合幂等设计 |

## 深挖追问

### cancel(true) 会立刻停止线程吗？

不会。它只是调用 `Thread.interrupt()`，设置中断标志。任务如果在 `sleep/wait/take` 等可中断方法上，会抛 `InterruptedException`；如果在 CPU 循环里且不检查 `isInterrupted()`，会继续跑完。

### 为什么不推荐 Thread.stop？

`Thread.stop()` 已被废弃。它会在任意位置抛 `ThreadDeath`，可能破坏对象状态、释放被持有的锁、导致不变量失效。中断机制是协作式的，由任务自己决定何时退出，更安全。

### remove 能撤回 submit 提交的任务吗？

可以，但要传入 `Future` 引用而非原始 `Runnable`。因为 `submit` 把 `Runnable/Callable` 包装成 `FutureTask` 存进队列。`future.cancel(false)` 比 `remove` 更直接。

### Future.cancel 后还能再取消吗？

不能。`cancel` 返回 false 表示任务已完成或已取消。`Future` 一旦进入 CANCELLED 状态不可恢复。

### 任务被取消后还会执行吗？

`cancel(false)` 后任务如果还没启动，永远不会启动；`cancel(true)` 后任务如果已启动，会继续执行直到自然结束或响应中断退出。`FutureTask.run()` 开头会检查状态，CANCELLED 状态直接返回。

## 易错点

- `cancel(true)` 后以为任务一定停了，结果任务不响应中断继续跑。
- 任务 catch 了 `InterruptedException` 后吞掉不重设中断标志，导致上层无法感知。
- 用 `Thread.stop` 强杀，引发数据不一致。
- 长耗时任务不做超时和取消设计，线上无法止血。

## 总结

撤回线程池任务的本质是协作式取消：队列中的任务通过 `remove` 或 `cancel(false)` 移除，运行中的任务通过 `cancel(true)` 发中断信号、由任务自己响应。Java 没有"强制杀死线程"的安全方式，长耗时任务必须在设计阶段就考虑超时、中断检查和业务补偿标记，否则线上故障时无法止血。
