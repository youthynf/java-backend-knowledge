# 如何撤回提交给线程池中的任务？

## 核心概念

提交给线程池的任务能否撤回，取决于任务所处阶段：

1. 还在队列中：可以尝试从队列移除或取消 `Future`。
2. 已经被线程取走但尚未执行完：只能通过中断协作式取消。
3. 已经执行完成：无法撤回，只能补偿。

Java 线程没有安全的“强制杀死任务”机制，取消本质上是协作式的。

## 面试官想考什么

- 是否知道 `Future.cancel(boolean mayInterruptIfRunning)` 的语义。
- 是否理解中断不是强制停止，而是给线程一个取消信号。
- 是否能区分排队任务、运行中任务和已完成任务。

## 标准回答

如果任务通过 `submit()` 提交，可以拿到 `Future`，调用 `cancel(false)` 尝试取消未运行的任务，调用 `cancel(true)` 会在任务已运行时尝试中断执行线程。是否能真正停止，取决于任务代码是否响应中断。

如果通过 `execute()` 提交且任务仍在阻塞队列中，可以使用 `ThreadPoolExecutor#getQueue().remove(task)` 移除，但这要求还能拿到同一个任务对象。运行中的任务不能安全强杀，只能设计超时、中断检查或业务取消标记。

## 深挖追问

### `cancel(true)` 会立刻停止线程吗？

不会。它通常只是调用线程中断。任务如果在可中断阻塞方法中，可能抛出 `InterruptedException`；如果在 CPU 循环中，需要主动检查 `Thread.currentThread().isInterrupted()`。

### 为什么不推荐 `Thread.stop()`？

`Thread.stop()` 会在任意位置强制终止线程，可能破坏对象状态和锁保护的不变量，已经不推荐使用。

## 实战场景/代码示例

```java
Future<?> future = executor.submit(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        doOneBatch();
    }
});

boolean cancelled = future.cancel(true);
```

任务内部要正确处理中断：

```java
try {
    queue.take();
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    return;
}
```

## 易错点/总结

- 取消不是强制删除运行中的代码。
- 长耗时任务应主动支持超时、中断和幂等补偿。
- 如果业务需要撤销结果，要设计状态机或补偿逻辑，而不是只依赖线程取消。
