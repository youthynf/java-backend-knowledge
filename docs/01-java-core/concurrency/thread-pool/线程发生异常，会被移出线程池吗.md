# 线程发生异常，会被移出线程池吗？

## 核心概念

线程池中的工作线程执行任务时，如果任务抛出未捕获的运行时异常或错误，当前工作线程会异常结束。`ThreadPoolExecutor` 会根据当前线程池状态和任务队列情况补充新的工作线程，以维持线程池可用性。

需要区分两件事：

- **线程是否还活着**：抛出未捕获异常的工作线程通常会终止。
- **线程池是否还能继续工作**：线程池一般会创建新的 worker 继续处理后续任务。

## 面试官想考什么

- 是否理解线程池复用的是工作线程，但线程不是永远不死。
- 是否知道 `execute()` 和 `submit()` 对异常的表现不同。
- 是否知道生产中要捕获任务异常，避免问题被吞掉或日志缺失。

## 标准回答

如果通过 `execute()` 提交任务，任务抛出未捕获异常时，异常会交给线程的 `UncaughtExceptionHandler`，当前工作线程会退出，线程池可能补充新线程继续运行。

如果通过 `submit()` 提交任务，异常会被封装到 `Future` 中，调用 `future.get()` 时才会抛出 `ExecutionException`；如果不调用 `get()`，异常容易被忽略。

因此生产中建议在任务内部统一捕获异常并记录日志，或者扩展 `afterExecute()` 做统一处理。

## 深挖追问

### `execute()` 和 `submit()` 的异常有什么区别？

- `execute()`：异常直接暴露在线程执行过程中，通常能在日志中看到。
- `submit()`：异常被保存到 `Future`，需要调用 `get()` 才能感知。

### 线程异常退出会不会导致线程池线程越来越少？

正常情况下不会。`ThreadPoolExecutor` 的 worker 退出流程会尝试根据线程池状态、队列和核心线程数补充 worker。但如果线程池已经关闭或拒绝创建线程，就不会补充。

## 实战场景/代码示例

```java
Future<?> future = executor.submit(() -> {
    throw new IllegalStateException("bad task");
});

try {
    future.get();
} catch (ExecutionException e) {
    log.error("task failed", e.getCause());
}
```

更推荐在任务内部兜底：

```java
executor.execute(() -> {
    try {
        doBiz();
    } catch (Exception e) {
        log.error("async task failed", e);
    }
});
```

## 易错点/总结

- 不要以为线程池会自动记录所有任务异常。
- `submit()` 不取 `Future#get()` 时，异常可能被“静默保存”。
- 异步任务必须有日志、监控和告警，否则线上故障很难定位。
