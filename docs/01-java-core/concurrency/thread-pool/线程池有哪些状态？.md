# 线程池有哪些状态

## 核心概念

`ThreadPoolExecutor` 用一个 `volatile int ctl` 字段同时保存"线程池状态"和"工作线程数"。高 3 位表示状态，低 29 位表示线程数。线程池状态共有 5 种，按流转顺序依次为：`RUNNING` → `SHUTDOWN` → `STOP` → `TIDYING` → `TERMINATED`。

理解状态机的关键是抓住两条主线：**是否接收新任务**、**是否处理队列中的剩余任务**。`shutdown()` 触发温和关闭（处理完排队任务再退），`shutdownNow()` 触发强制关闭（不再处理排队任务，并中断工作线程）。

## 标准回答

线程池有 5 种状态，定义在 `ThreadPoolExecutor` 的静态常量中：

1. `RUNNING`：接收新任务，处理队列任务。线程池被创建后即处于此状态。
2. `SHUTDOWN`：不接收新任务，但继续处理队列中的任务。`shutdown()` 触发。
3. `STOP`：不接收新任务，不处理队列任务，中断正在执行的任务。`shutdownNow()` 触发。
4. `TIDYING`：所有任务已终止，工作线程数为 0，即将调用 `terminated()` 钩子。
5. `TERMINATED`：`terminated()` 执行完毕，线程池彻底死亡。

状态迁移不可逆，只能从左向右流转。

## 实现原理

### 状态值定义

`ctl` 高 3 位编码状态，源码常量如下：

```java
private static final int COUNT_BITS = Integer.SIZE - 3;     // 29
private static final int CAPACITY   = (1 << COUNT_BITS) - 1; // 线程数上限约 5 亿

// runState 存储在 ctl 高 3 位
private static final int RUNNING    = -1 << COUNT_BITS; // 111
private static final int SHUTDOWN   =  0 << COUNT_BITS; // 000
private static final int STOP       =  1 << COUNT_BITS; // 001
private static final int TIDYING    =  2 << COUNT_BITS; // 010
private static final int TERMINATED =  3 << COUNT_BITS; // 011
```

数值大小关系：`RUNNING < SHUTDOWN < STOP < TIDYING < TERMINATED`。判断状态时常用 `runStateAtLeast(ctl, SHUTDOWN)` 这类比较。

### 状态流转图

```
        shutdown()
  RUNNING ───────────► SHUTDOWN
     │                    │
     │ shutdownNow()      │ 队列空 且 工作线程=0
     │                    ▼
     ▼                  TIDYING
   STOP ─────────────────►
     │  工作线程=0
     ▼
   TIDYING ─────────────► TERMINATED
            terminated()
```

补充细节：

- `RUNNING → SHUTDOWN`：调用 `shutdown()`，或线程池对象被 GC 时 `finalize()` 触发的 `shutdown()`。
- `RUNNING/SHUTDOWN → STOP`：调用 `shutdownNow()`。
- `SHUTDOWN → TIDYING`：当队列为空且工作线程数为 0 时自动切换。
- `STOP → TIDYING`：当工作线程数为 0 时自动切换（此时队列可能还有任务，但会被丢弃）。
- `TIDYING → TERMINATED`：`terminated()` 钩子方法执行完毕后切换。`awaitTermination()` 一直在等待这个状态。

### 状态转换的触发点

| 调用 | 状态变化 | 行为 |
|------|----------|------|
| `shutdown()` | → SHUTDOWN | 拒绝新任务，继续处理队列 |
| `shutdownNow()` | → STOP | 拒绝新任务，丢弃队列，中断工作线程 |
| `awaitTermination()` | 不变化 | 阻塞等待直到 TERMINATED |

## 代码示例

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        2, 4, 30L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(10));

executor.submit(() -> System.out.println("task running"));

executor.shutdown();
if (!executor.awaitTermination(5, TimeUnit.SECONDS)) {
    // 5 秒还没结束，强制停止
    List<Runnable> remaining = executor.shutdownNow();
    System.out.println("未执行任务数: " + remaining.size());
}

System.out.println("是否已终止: " + executor.isTerminated());
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 应用优雅停机 | 注册 ShutdownHook，先 `shutdown()` 再 `awaitTermination()`，超时再 `shutdownNow()` | 必须设超时，否则进程无法退出 |
| 拒绝服务排查 | `isShutdown()` 判断是否进入停止流程，避免在停止后还提交任务 | `shutdown()` 后 `execute()` 会触发拒绝策略 |
| 等待所有任务完成 | `awaitTermination(timeout)` 阻塞当前线程 | 不传超时会永久阻塞，慎用 |

## 深挖追问

### 线程池状态和线程状态是一回事吗？

不是。线程池状态描述的是线程池这个管理对象的生命周期；线程状态（`Thread.State`：NEW/RUNNABLE/BLOCKED/WAITING/TIMED_WAITING/TERMINATED）描述的是单个线程的执行状态。线程池处于 RUNNING 时，里面的工作线程可能正处于 BLOCKED。

### 为什么用 1 个 int 同时存状态和线程数？

为了在一次 CAS 中同时更新两者，避免状态和线程数不一致带来的并发问题。`ctl` 是 `volatile int`，通过 `Unsafe.compareAndSwapInt` 原子更新。

### TERMINATED 状态后还能复用线程池吗？

不能。状态不可逆，必须 `new` 一个新的 `ThreadPoolExecutor`。

## 易错点

- 把 `shutdown()` 当成"立即停止"，实际它只是"关门谢客"，正在执行和队列中的任务还会继续。
- 以为 `shutdownNow()` 一定能中断所有线程。如果任务不响应中断（没有可中断的阻塞、不检查中断标志），它仍会跑完。
- 混淆 `isShutdown()` 和 `isTerminated()`：前者进入 SHUTDOWN/STOP 即为 true，后者只有走到 TERMINATED 才为 true。

## 总结

线程池状态机围绕"是否接新任务、是否处理队列任务"两条线设计，5 个状态对应 5 个生命周期阶段。掌握 `ctl` 的高 3 位编码方式、`shutdown()` 与 `shutdownNow()` 的触发位置、`TIDYING` 到 `TERMINATED` 的钩子调用，是看懂线程池源码和写出正确优雅停机代码的前提。
