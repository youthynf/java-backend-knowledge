# 线程池有哪些状态？

线程池有哪些状态？
线程池的五种状态：
RUNNING：线程池正常运行状态，可以接受任务和处理队列中的任务；
SHUTDOWN：不接受新任务，并且会处理队列中的任务；
STOP：不接受新任务，并且不处理队列中的任务，而且支持任务中断，前提任务支持；
TIDYING：所有任务终止了，线程池中也没有线程了。到达该状态后会调用线程池的terminated()方法；
TERMINATED：terminated()方法执行完成后会转变为TERMINATED状态。

状态的流转：
RUNNING → SHUTDOWN：执行shutdown()方法，或者线程池对象GC是会调用finalize()方法从而调用shutdown()方法；
RUNNING / SHUTDOWN → STOP：执行shutdownnow()方法，或者先执行shutdown()后接着执行shutdownnow()方法；
SHUTDOWN → TIDYING：SHUTDOWN状态下，队列中任务会继续消费，当队列为空且线程池中没有线程时自动切换；
STOP → TIDYING：STOP状态下，队列任务可能不为空，只要线程池中没有线程就自动切换；
TIDYING → TERMINATED：执行terminated()方法后自动切换。

特别说明，这里的线程池状态和线程状态不是一个东西。

## 面试总结

围绕「线程池有哪些状态？」，面试官通常不只考概念定义，更关注你能否把机制、使用场景和线上问题串起来。

### 核心回答

1. 线程池通过复用线程降低创建销毁成本，并用队列、拒绝策略和参数控制并发压力。
2. 核心参数要结合任务类型、RT、吞吐、下游容量和机器资源一起评估。
3. 线上重点关注活跃线程数、队列积压、拒绝次数、任务耗时和异常吞噬。

### 高频追问

- 为什么不建议直接使用 Executors 默认工厂？
- CPU 密集型和 IO 密集型线程数如何估算？
- 队列满、线程满、下游慢时如何降级和止血？

### 实战落地

- **选型前**：先判断是互斥访问、线程协作、任务编排，还是限流隔离。
- **编码时**：控制共享变量范围，明确锁对象、超时策略、异常处理和资源释放。
- **上线后**：观察线程数、队列长度、阻塞时间、拒绝次数和 RT 抖动，必要时用线程 Dump 验证。

### 易错点

- 不要使用无界队列掩盖流量问题。
- 异步任务要显式处理异常、超时和上下文传递。
