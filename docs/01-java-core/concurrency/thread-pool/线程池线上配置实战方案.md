# 线程池线上配置实战方案

## 核心原则

> 线程池配置没有银弹，核心是：**明确任务类型 → 估算吞吐需求 → 压测验证 → 动态调优**

---

## 一、任务类型分析

在配置线程池前，必须先搞清楚你的任务属于哪一类：

| 类型 | 特征 | 典型场景 | 线程数公式 |
|------|------|----------|-----------|
| CPU 密集型 | 计算为主，几乎不等待 | 加密解密、排序、压缩 | **N + 1**（N = CPU 核数） |
| IO 密集型 | 大量等待外部响应 | HTTP 调用、DB 查询、文件读写 | **N × 2** 或 **N / (1 - 阻塞系数)** |
| 混合型 | 既有计算又有 IO | 业务处理（查DB + 计算 + 写DB） | 按瓶颈端配置，或拆分为两个池 |

### 阻塞系数估算

```
阻塞系数 = 等待时间 / (等待时间 + 计算时间)
```

- 纯计算任务：阻塞系数 ≈ 0，线程数 ≈ N + 1
- 纯 IO 任务：阻塞系数 ≈ 0.9，线程数 ≈ N / (1 - 0.9) = 10N
- 一般业务：阻塞系数 ≈ 0.5~0.8，线程数 ≈ 2N ~ 5N

## 二、线上配置方案（按场景）

### 方案 1：Web 服务 - 异步调用远程接口

```java
/**
 * 场景：订单服务调用库存、支付、物流等远程接口
 * 特征：IO 密集型，RT 50~200ms，QPS 峰值约 500
 * 服务器：8 核
 */
@Bean
public ThreadPoolExecutor remoteCallExecutor() {
    int cpuCores = Runtime.getRuntime().availableProcessors(); // 8
    
    return new ThreadPoolExecutor(
        cpuCores * 2,                              // corePoolSize = 16
        cpuCores * 4,                              // maximumPoolSize = 32
        60L, TimeUnit.SECONDS,                     // 非核心线程 60s 回收
        new LinkedBlockingQueue<>(500),             // 有界队列 500
        new CustomNamedThreadFactory("remote-call"), // 自定义线程名
        new ThreadPoolExecutor.CallerRunsPolicy()   // 降级：调用者线程执行
    );
}
```

**配置理由**：
- **core = 2N = 16**：IO 等待期间 CPU 空闲，多线程充分利用 CPU
- **max = 4N = 32**：突发流量缓冲，超过核心的线程在空闲后自动回收
- **队列 500**：按峰值 QPS × 平均 RT 估算，500 × 0.1s = 50 在途，500 容量留足余量
- **CallerRunsPolicy**：远程调用不能丢，让调用者自己执行可以自然限流

### 方案 2：数据处理 - 批量计算任务

```java
/**
 * 场景：报表生成、数据清洗、批量计算
 * 特征：CPU 密集型，单任务耗时 1~10s
 * 服务器：8 核
 */
@Bean
public ThreadPoolExecutor computeExecutor() {
    int cpuCores = Runtime.getRuntime().availableProcessors(); // 8
    
    return new ThreadPoolExecutor(
        cpuCores + 1,                              // corePoolSize = 9
        cpuCores + 1,                              // maximumPoolSize = 9（固定）
        0L, TimeUnit.SECONDS,                      // 不需要回收
        new LinkedBlockingQueue<>(1000),            // 大队列缓冲
        new CustomNamedThreadFactory("compute"),
        new ThreadPoolExecutor.AbortPolicy()        // 队列满了直接拒绝，避免雪崩
    );
}
```

**配置理由**：
- **core = max = N + 1 = 9**：CPU 密集型多线程无益，反而增加上下文切换
- **+1 的原因**：偶发缺页中断时，多一个线程保证 CPU 不空转
- **大队列 1000**：计算任务耗时长，需要足够的队列容量缓冲
- **AbortPolicy**：计算任务堆积说明系统过载，快速失败比慢慢拖死好

### 方案 3：日志/消息消费 - 高吞吐写入

```java
/**
 * 场景：Kafka 消费者处理消息、异步日志写入
 * 特征：IO 密集型（写DB/ES），要求不丢消息
 * 服务器：8 核
 */
@Bean
public ThreadPoolExecutor messageConsumeExecutor() {
    int cpuCores = Runtime.getRuntime().availableProcessors(); // 8
    
    ThreadPoolExecutor executor = new ThreadPoolExecutor(
        cpuCores * 3,                              // corePoolSize = 24
        cpuCores * 6,                              // maximumPoolSize = 48
        120L, TimeUnit.SECONDS,                    // 较长回收时间
        new LinkedBlockingQueue<>(2000),            // 大容量队列
        new CustomNamedThreadFactory("msg-consume"),
        new RetryRejectedExecutionHandler()         // 自定义：持久化后重试
    );
    
    // 允许核心线程超时回收（低峰期释放资源）
    executor.allowCoreThreadTimeOut(true);
    return executor;
}
```

**配置理由**：
- **core = 3N = 24**：消息消费是典型 IO 密集，需要高并发写入
- **max = 6N = 48**：消费堆积时快速扩容
- **allowCoreThreadTimeOut**：低峰期自动缩容，节省资源
- **自定义拒绝策略**：消息不能丢，持久化到 DB 后由定时任务重试

### 方案 4：定时任务调度

```java
/**
 * 场景：ScheduledExecutorService 替代 Timer
 * 特征：任务量少但要求准时触发
 */
@Bean
public ScheduledExecutorService scheduledExecutor() {
    return new ScheduledThreadPoolExecutor(
        4,                                          // 4 个核心线程足够
        new CustomNamedThreadFactory("scheduled")
    );
}
```

**配置理由**：
- 定时任务数量有限，4 核心线程覆盖大部分场景
- 使用 ScheduledThreadPoolExecutor 而非 Timer（Timer 不处理异常会终止所有任务）

## 三、自定义拒绝策略（实战版）

### 3.1 持久化重试策略

```java
/**
 * 被拒绝的任务持久化到数据库，由定时任务捞取重试
 */
public class RetryRejectedExecutionHandler implements RejectedExecutionHandler {
    
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        if (r instanceof RetryableTask) {
            RetryableTask task = (RetryableTask) r;
            try {
                // 持久化到数据库
                taskRepository.save(TaskPO.builder()
                    .taskId(task.getTaskId())
                    .payload(task.serialize())
                    .retryCount(0)
                    .status(TaskStatus.PENDING)
                    .createTime(new Date())
                    .build());
                log.warn("任务被拒绝，已持久化等待重试: taskId={}", task.getTaskId());
            } catch (Exception e) {
                log.error("持久化失败，任务丢失: taskId={}", task.getTaskId(), e);
                // 降级：发送告警
                alertService.sendAlert("线程池任务持久化失败: " + task.getTaskId());
            }
        } else {
            log.error("非 RetryableTask 被拒绝，无法持久化: {}", r.getClass().getName());
        }
    }
}
```

### 3.2 带降级的拒绝策略

```java
/**
 * 优先让调用者执行，如果调用者线程也满则降级
 */
public class FallbackRejectedExecutionHandler implements RejectedExecutionHandler {
    
    private final RejectedExecutionHandler fallback;
    
    public FallbackRejectedExecutionHandler() {
        this.fallback = new ThreadPoolExecutor.AbortPolicy();
    }
    
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        if (!executor.isShutdown()) {
            try {
                // 尝试调用者线程执行
                if (Thread.currentThread().getName().startsWith("remote-call")) {
                    // 如果调用者本身就是线程池线程，避免递归
                    throw new RejectedExecutionException("调用者线程也在线程池中，避免递归");
                }
                r.run();
            } catch (Exception e) {
                // 降级：记录日志 + 告警
                log.error("任务执行降级失败", e);
                alertService.sendAlert("线程池任务降级失败");
            }
        }
    }
}
```

## 四、动态调参方案

### 4.1 基于配置中心的动态调整

```java
/**
 * 接入 Nacos/Apollo 配置中心，实现线程池参数动态调整
 */
@Component
@Slf4j
public class DynamicThreadPoolManager {
    
    private final Map<String, ThreadPoolExecutor> executorMap = new ConcurrentHashMap<>();
    
    /**
     * 注册线程池到管理器
     */
    public void register(String name, ThreadPoolExecutor executor) {
        executorMap.put(name, executor);
    }
    
    /**
     * 动态调整参数（由配置变更回调触发）
     */
    @NacosConfigListener(dataId = "thread-pool-config")
    public void onConfigChange(String config) {
        JSONObject json = JSON.parseObject(config);
        executorMap.forEach((name, executor) -> {
            if (json.containsKey(name)) {
                JSONObject poolConfig = json.getJSONObject(name);
                int newCore = poolConfig.getIntValue("corePoolSize");
                int newMax = poolConfig.getIntValue("maximumPoolSize");
                
                // ⚠️ 注意：setMaximumPoolSize 必须在 setCorePoolSize 之前调用
                // 因为 setCorePoolSize 会校验 core <= max
                if (newMax < executor.getCorePoolSize()) {
                    executor.setMaximumPoolSize(newMax);
                    executor.setCorePoolSize(newCore);
                } else {
                    executor.setCorePoolSize(newCore);
                    executor.setMaximumPoolSize(newMax);
                }
                
                log.info("线程池[{}]参数已更新: core={}, max={}", 
                    name, executor.getCorePoolSize(), executor.getMaximumPoolSize());
            }
        });
    }
}
```

### 4.2 Nacos 配置示例

```json
{
  "remote-call": {
    "corePoolSize": 16,
    "maximumPoolSize": 32
  },
  "compute": {
    "corePoolSize": 9,
    "maximumPoolSize": 9
  },
  "msg-consume": {
    "corePoolSize": 24,
    "maximumPoolSize": 48
  }
}
```

### 4.3 动态调参注意事项

```java
// ⚠️ 关键注意点：

// 1. setCorePoolSize 时，如果新值 < 当前核心线程数，多余线程会立即被回收
executor.setCorePoolSize(8); // 如果当前有16个核心线程，8个会被立即回收

// 2. setMaximumPoolSize 时，如果新值 < 当前最大线程数，多余线程会在空闲后回收
executor.setMaximumPoolSize(20);

// 3. 调参顺序很重要！
// 错误：先设 core=20 再设 max=10 → core > max 报错
// 正确：先设 max=10 再设 core=20 → 不行，core 不能 > max
// 正确做法：先设 max=20 再设 core=20

// 4. 队列容量无法动态调整（ArrayBlockingQueue 容量 final）
// 如需调整队列容量，需要重建线程池
```

## 五、线程池监控告警

### 5.1 监控指标采集

```java
@Component
@Slf4j
public class ThreadPoolMonitor {
    
    @Scheduled(fixedRate = 5000) // 每 5 秒采集一次
    public void collectMetrics() {
        DynamicThreadPoolManager.getExecutors().forEach((name, executor) -> {
            // 上报 Micrometer 指标
            Metrics.gauge("threadpool.active", Tags.of("name", name), executor.getActiveCount());
            Metrics.gauge("threadpool.pool.size", Tags.of("name", name), executor.getPoolSize());
            Metrics.gauge("threadpool.queue.size", Tags.of("name", name), executor.getQueue().size());
            
            // 告警判断
            double queueUsage = (double) executor.getQueue().size() / getMaxQueueSize(executor);
            double poolUsage = (double) executor.getPoolSize() / executor.getMaximumPoolSize();
            
            if (queueUsage > 0.8 || poolUsage > 0.9) {
                alertService.sendAlert(String.format(
                    "线程池[%s]负载过高: 队列使用率=%.1f%%, 线程使用率=%.1f%%, 活跃=%d, 队列=%d",
                    name, queueUsage * 100, poolUsage * 100,
                    executor.getActiveCount(), executor.getQueue().size()
                ));
            }
        });
    }
}
```

### 5.2 关键告警规则

| 指标 | 告警阈值 | 含义 |
|------|---------|------|
| 队列使用率 | > 80% | 任务堆积，即将触发拒绝策略 |
| 线程使用率 | > 90% | 线程资源耗尽 |
| 拒绝次数 | > 0 | 已有任务被拒绝，需扩容 |
| 活跃线程 / 核心线程 | 持续 = 100% | 核心线程不够用 |

## 六、常见线上事故与防范

### 6.1 OOM：无界队列 + FixedThreadPool

```java
// ❌ 错误：无界队列
ExecutorService pool = Executors.newFixedThreadPool(10);
// 如果任务提交速度 > 处理速度，队列无限增长 → OOM

// ✅ 正确：有界队列
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    10, 10, 0L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(500),  // 有界！
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### 6.2 线程泄漏：未关闭线程池

```java
// ❌ 错误：Spring Bean 销毁时未关闭线程池
@Bean
public ThreadPoolExecutor executor() { ... }
// 容器关闭时线程池中的线程仍在运行

// ✅ 正确：实现 DisposableBean
@Bean
public ThreadPoolExecutor executor() { ... }

@Override
public void destroy() {
    executor.shutdown();
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        executor.shutdownNow();
    }
}
```

### 6.3 任务丢失：异常吞没

```java
// ❌ 错误：execute 提交的任务异常被吞
executor.execute(() -> {
    throw new RuntimeException("业务异常"); // 不会打印任何日志！
});

// ✅ 正确方案 1：submit + Future.get
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("业务异常");
});
try {
    future.get(); // 会抛出 ExecutionException
} catch (ExecutionException e) {
    log.error("任务执行失败", e.getCause());
}

// ✅ 正确方案 2：自定义 ThreadFactory 设置 UncaughtExceptionHandler
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setUncaughtExceptionHandler((thread, ex) -> log.error("线程[{}]异常", thread.getName(), ex));
    return t;
};
```

### 6.4 死锁：CallerRunsPolicy + 线程池间依赖

```java
// ❌ 错误场景：
// 线程池A 的任务中调用了线程池B 的submit，且B也满了
// A 的线程被 B 的 CallerRunsPolicy 阻塞 → A 也满了 → 互相等待

// ✅ 正确：避免线程池间直接依赖，或使用独立线程提交
```

## 七、配置速查表

| 场景 | core | max | 队列 | 拒绝策略 | 关键点 |
|------|------|-----|------|---------|--------|
| 远程调用 | 2N | 4N | 500 | CallerRunsPolicy | 不能丢，自然限流 |
| CPU 计算 | N+1 | N+1 | 1000 | AbortPolicy | 固定线程，快速失败 |
| 消息消费 | 3N | 6N | 2000 | 持久化重试 | 不丢消息，允许核心超时 |
| 定时任务 | 4 | 4 | - | AbortPolicy | 用 ScheduledExecutor |
| 日志写入 | N | 2N | 5000 | DiscardPolicy | 可接受少量丢失 |
| 文件处理 | 2N | 4N | 1000 | CallerRunsPolicy | IO 密集，限流保护 |

> **N = CPU 核心数**

## 八、压测验证清单

配置完成后，必须通过压测验证：

- [ ] **正常负载**：线程池利用率 50%~70%，队列使用率 < 30%
- [ ] **2倍流量**：队列开始积压，max 线程逐步扩出
- [ ] **5倍流量**：触发拒绝策略，确认降级逻辑正常
- [ ] **持续高压 30min**：无 OOM，无线程泄漏，内存稳定
- [ ] **流量恢复后**：非核心线程正常回收，资源释放

# 面试复习补充

## 核心概念补充

这篇文章的主题是 **线程池线上配置实战方案**。复习时应先给出定义，再说明它在 Java 并发或 JVM 体系中的位置，最后结合使用场景、限制条件和常见误区展开。

## 面试官想考什么

- 是否能用自己的话讲清楚概念，而不是只背术语。
- 是否理解底层机制、关键流程以及它和相邻知识点的区别。
- 是否能把知识点落到真实项目：如何使用、如何排查、如何调优、什么时候不该用。
- 是否知道常见坑点，例如线程安全、可见性、阻塞、内存泄漏、GC 停顿或参数误用。

## 标准回答

线程池相关问题要围绕 `ThreadPoolExecutor` 的核心参数、任务提交流程和生命周期来回答：先看核心线程数、最大线程数、阻塞队列、拒绝策略，再看任务执行过程中线程如何复用、如何退出、异常如何处理。生产中不建议无脑使用 `Executors` 的快捷方法，应根据业务类型配置有界队列、命名线程工厂、拒绝策略和监控指标。

## 深挖追问

- `execute()` 和 `submit()` 在异常表现、返回值上的区别是什么？
- 核心线程、非核心线程分别什么时候创建和回收？
- 阻塞队列选择无界、有界、同步移交队列时，对吞吐和稳定性有什么影响？
- 拒绝策略应该如何和降级、限流、告警结合？

## 易错点/总结

- 不要脱离场景背结论：并发和 JVM 问题通常都和负载、线程数、内存大小、JDK 版本有关。
- 面试回答建议采用“定义 → 原理 → 场景 → 风险/排查”的顺序。
- 如果涉及源码或参数，说明核心思路即可；不确定的版本差异要明确限定，不要绝对化。
