# 线程池中的线程是如何关闭的

## 核心概念

线程池中的工作线程关闭依赖 Java 的**中断机制**——`Thread.interrupt()` 设置中断标志，工作线程在合适的时机检查标志并退出。这是一种"协作式"停止，而不是 `Thread.stop()` 那种"强制终止"。

线程池关闭线程的入口有两类：

- **`shutdown()`**：只中断"空闲"工作线程（在 `workQueue.take()` 上阻塞的）。正在执行任务的线程不打扰。
- **`shutdownNow()`**：中断所有工作线程，包括正在执行任务的。

无论哪种方式，线程真正停下来要靠任务代码响应中断。

## 标准回答

线程池关闭工作线程的机制：

1. `shutdown()` / `shutdownNow()` 调用 `interruptIdleWorkers()` / `interruptWorkers()`。
2. 内部遍历所有 `Worker`，调用 `Worker.interruptIfStarted()` 给工作线程发 `Thread.interrupt()`。
3. 工作线程如果在 `take/poll/sleep` 等可中断阻塞方法上，会抛 `InterruptedException` 退出阻塞。
4. `getTask()` 检测到线程池状态变为 SHUTDOWN 且队列空、或 STOP，返回 null。
5. `runWorker` 循环退出，工作线程结束。

核心一句话：**线程池通过 `Thread.interrupt()` 发信号，工作线程在 `getTask` 循环中根据状态和中断标志退出**。

## 实现原理

### interruptIfStarted 源码

`Worker` 是 `ThreadPoolExecutor` 的内部类，继承 AQS 实现独占锁，记录工作线程引用：

```java
private final class Worker extends AbstractQueuedSynchronizer implements Runnable {
    final Thread thread;
    Runnable firstTask;

    Worker(Runnable firstTask) {
        setState(-1); // inhibit interrupts until runWorker
        this.firstTask = firstTask;
        this.thread = getThreadFactory().newThread(this);
    }

    public void run() {
        runWorker(this);
    }

    void interruptIfStarted() {
        Thread t;
        if (getState() >= 0 && (t = thread) != null && !t.isInterrupted()) {
            try {
                t.interrupt();
            } catch (SecurityException ignore) {
            }
        }
    }
}
```

`getState() >= 0` 这个判断很关键：Worker 构造时 `setState(-1)`，runWorker 启动后才 `unlock`（也就是 `setState(0)`）。这避免了在线程还没启动时就发中断的边界情况。

### interruptIdleWorkers

```java
private void interruptIdleWorkers(boolean onlyOne) {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        for (Worker w : workers) {
            Thread t = w.thread;
            if (!t.isInterrupted() && w.tryLock()) {   // tryLock 成功说明线程空闲
                try {
                    t.interrupt();
                } catch (SecurityException ignore) {
                } finally {
                    w.unlock();
                }
            }
            if (onlyOne) break;
        }
    } finally {
        mainLock.unlock();
    }
}
```

关键在 `w.tryLock()`：`Worker` 在执行任务前会 `lock()`，执行完 `unlock()`。`tryLock` 成功意味着当前没在执行任务（在 `take()` 阻塞），可以安全中断；失败说明正在执行任务，`shutdown` 不打扰它。

`shutdownNow` 调用的 `interruptWorkers` 不做 `tryLock` 判断，直接对所有 worker 中断：

```java
private void interruptWorkers() {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        for (Worker w : workers)
            w.interruptIfStarted();   // 不区分是否在执行
    } finally {
        mainLock.unlock();
    }
}
```

### getTask 对中断的响应

```java
private Runnable getTask() {
    boolean timedOut = false;
    for (;;) {
        try {
            Runnable r = timed ?
                workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) :
                workQueue.take();          // 在这里被 interrupt 唤醒
            if (r != null) return r;
            timedOut = true;
        } catch (InterruptedException retry) {
            timedOut = false;
            // 不直接 return，进入下一轮 for 重新检查状态
        }
    }
}
```

`take()` 在被中断时抛 `InterruptedException`，进入 catch，重新循环。下一轮会检查 `runStateAtLeast(c, STOP)` 或 `SHUTDOWN && queue.isEmpty()`，满足条件返回 null，工作线程退出。

### runWorker 的退出

```java
final void runWorker(Worker w) {
    try {
        while (task != null || (task = getTask()) != null) {
            // 执行任务
        }
    } finally {
        processWorkerExit(w, false);   // 清理 worker、补充新 worker（必要时）
    }
}
```

`getTask` 返回 null 时退出循环，进入 `processWorkerExit` 完成清理。

## 代码示例

### 任务代码响应中断

```java
executor.submit(() -> {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            String line = queue.poll(100, TimeUnit.MILLISECONDS);  // 可被中断
            if (line != null) process(line);
        }
    } catch (InterruptedException e) {
        // 被中断，退出循环
        Thread.currentThread().interrupt();   // 重新设置中断标志，便于上层感知
    } finally {
        cleanup();
    }
});
```

### 不可中断的 CPU 密集任务

```java
executor.submit(() -> {
    long total = 0;
    for (long i = 0; i < Long.MAX_VALUE; i++) {
        // 不检查中断，shutdownNow 也无法停止
        total += i;
    }
});
```

这种任务即使 `shutdownNow` 也只能等它跑完。正确写法：

```java
executor.submit(() -> {
    long total = 0;
    for (long i = 0; i < Long.MAX_VALUE; i++) {
        if (Thread.currentThread().isInterrupted()) {
            return;   // 主动检查
        }
        total += i;
    }
});
```

## 实战场景

| 场景 | 中断方式 | 注意点 |
|------|----------|--------|
| 优雅停机 | shutdown + awaitTermination | 任务要响应中断 |
| 长耗时批处理 | 周期性检查 isInterrupted | 配合 shutdownNow 才能真停 |
| 阻塞在 IO 上 | interrupt 抛 ClosedByInterruptException | NIO Channel 会自动关闭 |
| 阻塞在锁上 | interrupt 不响应 | ReentrantLock.lock 不响应中断，需用 lockInterruptibly |

## 深挖追问

### 为什么不能用 Thread.stop？

`Thread.stop()` 已被废弃。它会在任意位置抛 `ThreadDeath` 错误终止线程，可能破坏对象状态（如正在写一个 Map 时被强停，状态不一致）、释放被持有的锁导致不变量失效。中断机制是协作式的，任务自己决定何时退出，更安全。

### 中断一个正在执行的线程会立刻停吗？

不会。`interrupt()` 只设置中断标志。如果线程在 `sleep/wait/join` 或 `BlockingQueue.take/poll` 等可中断方法上，会抛 `InterruptedException` 退出阻塞；否则只是设置标志，需要任务代码主动检查 `isInterrupted()` 才能停下。

### Worker 为什么要继承 AQS？

用 AQS 实现一把"独占锁"来表示工作线程是否在执行任务。`tryLock` 成功表示空闲，`shutdown` 据此判断哪些 worker 可以安全中断。这比单独维护一个 boolean 标志更高效且并发安全。

### 工作线程异常退出会被补充吗？

会。`processWorkerExit` 中会调用 `addWorker(null, false)` 尝试补充一个非核心线程（前提是线程池还在运行、且未达到核心数下限）。所以单个任务抛未捕获异常不会让线程池"缩水"。详见 [线程发生异常会被移出线程池吗？](线程发生异常会被移出线程池吗？.md)。

### shutdownNow 后线程一定退出吗？

不一定。如果任务不响应中断（既没有可中断的阻塞，也不检查中断标志），线程会一直跑完任务才退出。`shutdownNow` 只是发信号，不是强制停止。

## 易错点

- 任务里 catch 了 `InterruptedException` 后吞掉不处理，导致中断信号丢失，任务停不下来。
- 长循环里不检查 `isInterrupted()`，`shutdownNow` 后任务依然跑几分钟。
- 误以为 `Thread.interrupt()` 能像 `Thread.stop()` 一样立刻停线程。
- `shutdown` 后期望"立刻停"，结果队列任务还在执行，造成困惑。

## 总结

线程池关闭工作线程依赖 Java 中断机制：`shutdown` 只中断空闲线程，`shutdownNow` 中断所有线程，但都只是发信号。工作线程在 `getTask` 循环中根据线程池状态和中断标志退出。任务代码必须响应中断——要么在可中断阻塞方法上让异常抛出，要么主动检查 `isInterrupted()`——否则强制关闭也停不下来。
