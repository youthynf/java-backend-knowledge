# 为什么核心线程池满了之后是先加入阻塞队列而不是直接创建线程？

## 核心概念

`ThreadPoolExecutor` 的默认提交流程是：

1. 当前工作线程数小于 `corePoolSize`：创建核心线程执行任务。
2. 核心线程已满：尝试把任务放入阻塞队列。
3. 队列也满：再尝试创建非核心线程，直到 `maximumPoolSize`。
4. 线程数达到最大且队列满：执行拒绝策略。

这体现的是一种“优先复用核心线程、用队列削峰”的设计。

## 面试官想考什么

- 是否熟悉线程池任务提交流程。
- 是否理解队列在削峰、限流、稳定性上的作用。
- 是否知道不同队列会改变线程池扩容行为。

## 标准回答

核心线程满后先入队，是为了复用已有线程并限制线程数量，避免瞬时流量导致大量创建线程，引发上下文切换和资源竞争。阻塞队列可以缓冲短时间高峰，让核心线程慢慢消费任务。只有当队列也放不下时，才说明当前处理能力明显不足，线程池才会创建非核心线程应急扩容。

但这不是绝对最优策略。如果使用无界队列，队列几乎不会满，`maximumPoolSize` 基本失效；如果使用 `SynchronousQueue`，任务无法排队，会更倾向于直接创建线程。

## 深挖追问

### 为什么不先创建到最大线程数？

因为线程不是免费资源。大量线程会增加内存占用、调度开销和锁竞争，甚至压垮下游系统。

### 队列太大会怎样？

队列太大可能掩盖处理能力不足，导致请求长时间排队，超时后仍继续执行，产生无效工作和内存压力。

### 队列太小会怎样？

队列太小会更快触发扩容和拒绝，适合低延迟、快速失败场景，但需要配合降级和限流。

## 实战场景/代码示例

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        8,
        32,
        60, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(500),
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

如果要更积极扩容，可以选择较小队列；如果要更强削峰，可以选择适中的有界队列。

## 易错点/总结

- 任务提交顺序是“核心线程 → 队列 → 非核心线程 → 拒绝”。
- 无界队列会让最大线程数难以发挥作用。
- 队列容量是稳定性参数，不是越大越好。
