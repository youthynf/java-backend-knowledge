# 为什么核心线程池满了之后是先加入阻塞队列而不是直接创建线程

## 核心概念

`ThreadPoolExecutor` 提交任务的顺序是：**核心线程 → 阻塞队列 → 非核心线程 → 拒绝策略**。核心线程数达到 `corePoolSize` 后，新任务先尝试入队，而不是立即扩容到 `maximumPoolSize`。这个顺序是 JDK 线程池设计上最有争议、也是面试最常被追问的点。

设计动机可以一句话概括：**线程是昂贵资源，能用队列削峰就不要轻易扩容**。新建线程要分配栈空间（默认 1MB）、注册到 OS、增加调度开销，而队列只是堆内一个对象引用。

## 标准回答

核心线程满后先入队，主要有三个原因：

1. **复用已有线程**：核心线程处理完手头任务后会立刻从队列取下一个任务，队列中的任务不需要等新线程创建就能被执行。
2. **避免线程数过早膨胀**：突发流量下若直接扩到 `maximumPoolSize`，会瞬间创建大量线程，带来内存压力、上下文切换和锁竞争。
3. **保护下游**：线程数膨胀意味着下游并发请求量也膨胀，可能直接压垮数据库、第三方接口。队列削峰等于给下游一个缓冲垫。

只有当队列也放不下时，才说明缓冲已饱和、需要应急扩容；如果连 max 都打满了，说明真的处理不过来，必须走拒绝策略。

## 实现原理

源码中 `execute` 的关键判断顺序：

```java
public void execute(Runnable command) {
    int c = ctl.get();
    if (workerCountOf(c) < corePoolSize) {              // 步骤 1：少核心，直接建
        if (addWorker(command, true)) return;
        c = ctl.get();
    }
    if (isRunning(c) && workQueue.offer(command)) {     // 步骤 2：入队
        // ...
    }
    else if (!addWorker(command, false)) {              // 步骤 3：队列满，建非核心
        reject(command);                                // 步骤 4：拒绝
    }
}
```

注意 `workQueue.offer(command)` 是非阻塞入队——满了立即返回 false，进入步骤 3。这与 `put`（阻塞）和 `add`（抛异常）不同。

### 不同队列对扩容行为的影响

| 队列类型 | 行为 | 对 max 的影响 |
|----------|------|---------------|
| `LinkedBlockingQueue`（无界） | 永远不返回 false | `maximumPoolSize` 完全失效 |
| `LinkedBlockingQueue(cap)`（有界） | 满了返回 false | 正常触发扩容 |
| `ArrayBlockingQueue(cap)`（有界） | 满了返回 false | 正常触发扩容 |
| `SynchronousQueue` | 没有容量，必须有消费者在等才能入队 | 几乎每次都直接扩容 |
| `PriorityBlockingQueue`（无界） | 永远不返回 false | `maximumPoolSize` 完全失效 |

`Executors.newFixedThreadPool` 用无界 `LinkedBlockingQueue`，所以 max 永远不会触发；`newCachedThreadPool` 用 `SynchronousQueue`，所以核心线程数 = 0，几乎每来一个任务都新建线程。

## 代码示例

```java
// 用小队列让扩容行为可观察
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        2, 4, 30L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(2),     // 队列 2
        new ThreadPoolExecutor.CallerRunsPolicy());

// 提交 6 个耗时任务，观察线程数变化
for (int i = 1; i <= 6; i++) {
    final int idx = i;
    executor.execute(() -> {
        System.out.println("任务 " + idx + " 在 " + Thread.currentThread().getName());
        try { TimeUnit.SECONDS.sleep(2); } catch (InterruptedException ignored) {}
    });
}
// 任务 1、2：核心线程
// 任务 3、4：入队
// 任务 5、6：扩容到 max=4
```

## 实战场景

| 场景 | 队列选择 | 行为 |
|------|----------|------|
| 削峰填谷 | 中等有界队列（如 500） | 短期高峰进队列，慢慢消费 |
| 快速失败 | 小队列（如 10） | 队列迅速满，扩容或拒绝 |
| 实时响应 | `SynchronousQueue` | 不缓存，直接扩容到 max |
| 串行处理 | 1 核 1 max + 大队列 | 顺序消费 |

## 深挖追问

### 为什么不先扩到 max 再入队？

线程不是免费的：每个线程默认占 1MB 栈，几百个线程就是几百 MB；OS 线程调度成本随数量上升而上升；线程越多锁竞争越激烈。队列削峰能用极小代价吸收短期流量，是更经济的选择。

### 队列设太大有什么风险？

无界队列会掩盖处理能力不足的问题——任务长时间排队，调用方早超时返回，但任务还在跑，浪费资源并占内存，最终可能 OOM。生产环境**必须用有界队列**。

### 用 SynchronousQueue 是不是就不入队了？

`SynchronousQueue` 没有容量，`offer` 只有在有空闲线程正在 `take` 时才成功。`CachedThreadPool` 用它意味着"有空闲线程就转交，否则新建"。从效果上看几乎不走"队列缓冲"那一步。

### Tomcat 线程池为什么不一样？

Tomcat 自定义的 `org.apache.tomcat.util.threads.ThreadPoolExecutor` 重写了 `execute` 流程：核心满后直接尝试扩容到 max，再入队。原因是 Tomcat 处理 HTTP 请求时希望尽快响应、减少排队延迟。这恰好说明 JDK 默认顺序不是"绝对正确"，而是"通用场景下的折中"。

## 易错点

- 误以为"先扩到 max 再入队"，导致面试时把流程画反。
- 用无界队列还想靠 `maximumPoolSize` 兜底，实际永远不会触发。
- 把队列大小当越多越好，结果高峰期任务排队超时、内存暴涨。
- 用 `LinkedBlockingQueue()` 无参构造（默认 `Integer.MAX_VALUE`）当有界队列用。

## 总结

"先入队再扩容"是 JDK 线程池在通用场景下做的经济性取舍——优先复用线程、用队列吸收短期波动、把扩容作为最后手段。理解这一点不仅是为了答面试，更是为了在选队列和设参数时知道每个选择背后意味着什么。
