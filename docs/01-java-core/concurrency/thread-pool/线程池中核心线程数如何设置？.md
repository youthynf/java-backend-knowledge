# 线程池中核心线程数如何设置

## 核心概念

`corePoolSize` 决定线程池在常态负载下保留多少工作线程。它不是一个公式能直接算出来的值，而是由**任务类型、机器 CPU 资源、下游容量、队列长度、响应时间目标**共同决定的工程参数。

业界流传的经验公式只是起点：

- CPU 密集型：`N` 或 `N + 1`（N = CPU 核数）
- IO 密集型：`2N` 或 `N / (1 - 阻塞系数)`
- 混合型：拆分多个线程池，按瓶颈端单独配置

最终值必须通过压测修正，配合监控指标持续调整。

## 标准回答

核心线程数的设置遵循三步：

1. **按任务类型估初值**：CPU 密集用 `N+1`，IO 密集用 `2N` 或基于阻塞系数的公式。
2. **结合下游容量修正**：线程数放大并发请求量，不能超过下游（DB、Redis、第三方接口）的承载能力。
3. **压测验证 + 监控调整**：观察吞吐、P99、CPU 利用率、队列积压，迭代到合理区间。

公式背后的本质：线程数应该让 CPU 在"计算时间 + 等待时间"内尽量不空转，又不至于过多导致上下文切换损耗。Brian Goetz 在《Java 并发编程实战》给出的公式是：

```
线程数 = CPU 核数 × (1 + 等待时间 / 计算时间)
        = CPU 核数 / (1 - 阻塞系数)
```

其中阻塞系数 = 等待时间 / (等待时间 + 计算时间)，纯计算 ≈ 0，纯 IO ≈ 0.9。

## 实现原理

### 为什么 CPU 密集型是 N+1

CPU 密集型任务几乎不阻塞，N 个线程就够把 N 个核打满。多出的线程会带来上下文切换，反而降低吞吐。`+1` 是为了应对偶发的缺页中断、GC 暂停等让某个线程短暂停顿的情况，多一个线程保证 CPU 不空转。

### 为什么 IO 密集型是 2N 或更多

IO 任务大部分时间在等待网络或磁盘响应，CPU 空闲。此时让更多线程进来，当一个线程在等 IO 时，CPU 可以执行另一个线程的计算部分。`2N` 是经验值，更精确的估算需要知道任务的"等待/计算"比例。

### 阻塞系数的实测

```
阻塞系数 = 等待时间 / (等待时间 + 计算时间)
```

- 纯计算任务：阻塞系数 ≈ 0，线程数 ≈ N + 1
- 纯 IO 任务（如远程调用 RT 200ms，CPU 处理 1ms）：阻塞系数 ≈ 0.995，理论线程数 ≈ 200N（实际不会设这么大，下游扛不住）
- 一般业务：阻塞系数 0.5~0.8，线程数 2N ~ 5N

公式算出来很大时，要回到下游容量做上限约束。

### 与队列、max 的配合

核心数不能孤立设计。如果队列很大，核心数小一点也能削峰；如果队列小，核心数要够大才能及时处理。常见组合：

- 快速响应：core 中等、max 大、队列小
- 削峰填谷：core 中等、max 不大、队列大
- 突发流量：core 小、max 大、队列小（让 max 尽快扩出）

## 代码示例

```java
int cpu = Runtime.getRuntime().availableProcessors();

// IO 密集型：远程调用线程池
ThreadPoolExecutor ioExecutor = new ThreadPoolExecutor(
        cpu * 2,                            // core
        cpu * 4,                            // max
        60L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(500),
        new ThreadFactory() {
            private final AtomicInteger seq = new AtomicInteger();
            public Thread newThread(Runnable r) {
                Thread t = new Thread(r, "io-worker-" + seq.incrementAndGet());
                t.setUncaughtExceptionHandler((th, ex) -> log.error("io 异常", ex));
                return t;
            }
        },
        new ThreadPoolExecutor.CallerRunsPolicy());

// CPU 密集型：报表计算线程池
ThreadPoolExecutor cpuExecutor = new ThreadPoolExecutor(
        cpu + 1, cpu + 1,
        0L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(1000),
        new ThreadPoolExecutor.AbortPolicy());
```

## 实战场景

| 场景 | 推荐配置 | 关键指标 |
|------|----------|----------|
| HTTP 远程调用 | core=2N, max=4N, 队列 500 | P99、下游 RT、连接池利用率 |
| DB 查询 | core 不超过连接池大小 | DB 连接池等待时间 |
| CPU 计算（如加密/压缩） | core=N+1, max=N+1 | CPU 利用率 70~80% |
| 消息消费 | core=2~3N, max=4~6N | 消费堆积、消费速率 |
| 日志异步写入 | core=N, 队列大 | 队列积压、磁盘 IO |

## 深挖追问

### 线程数是不是越多越好？

不是。线程过多会带来：

1. **上下文切换开销**：OS 调度器频繁切换，CPU 时间浪费在寄存器/缓存恢复。
2. **内存占用**：每线程默认 1MB 栈，1000 线程就是 1GB。
3. **锁竞争加剧**：共享资源被更多线程争用。
4. **下游过载**：并发请求放大，DB 连接池、第三方接口被打爆。

### 2N 一定适合 IO 密集吗？

不一定。`2N` 是粗略经验，更准确的做法是用阻塞系数公式或直接压测。如果 IO 等待时间特别长（如 RT 几秒），2N 远远不够；如果 IO 极快（如本地 SSD），2N 都偏多。

### 为什么不能多个业务共用一个大线程池？

慢任务会占据线程，把快任务饿死。例如发短信和订单查询共用一个池，短信接口抖动慢了，所有订单查询线程都被占住，订单接口跟着挂。建议按业务隔离，关键链路单独配置。

### corePoolSize 设 0 行不行？

可以。`CachedThreadPool` 就是 core=0 的例子。core=0 时所有线程都是非核心，空闲超过 `keepAliveTime` 都会被回收，低峰期 0 线程。代价是高峰期第一批任务要等线程创建。如果希望低峰也能立即响应，core 至少 1。

### allowCoreThreadTimeOut 是干什么的？

允许核心线程在空闲超过 `keepAliveTime` 后也被回收，低峰期释放资源。代价是下一次任务到来需要重新创建线程，有微小延迟。适合流量波动大、低峰期长的场景。

## 易错点

- 把公式当真理，不做压测就上线，结果线程数完全不对。
- IO 线程数设得很大，但 DB 连接池只有 10，多余线程全在等连接。
- 多个业务共用一个池，慢任务拖死快任务。
- core 设 0 又想低延迟响应，矛盾需求没考虑清楚。
- 改了 corePoolSize 不重启压测，结果高峰期才发现配置不对。

## 总结

核心线程数的设置本质是"让 CPU 不空转、不让线程过多、不让下游过载"三者的平衡。经验公式（CPU 密集 N+1、IO 密集 2N）只是起点，真正的值要靠压测和监控持续调整。生产环境还要考虑业务隔离、队列配合、拒绝策略，构成一个完整的线程池设计。
