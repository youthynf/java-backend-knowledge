# 线程池中核心线程数如何设置？

## 核心概念

核心线程数 `corePoolSize` 决定线程池在常态负载下保留多少工作线程。它不是一个固定公式能直接算出来的值，而是由 **任务类型、机器资源、下游容量、队列长度、响应时间目标** 共同决定的。

常见经验：

- CPU 密集型：接近 CPU 核数，常见经验是 `N` 或 `N + 1`，避免过多线程导致上下文切换。
- I/O 密集型：线程数可以大于 CPU 核数，因为大量时间消耗在网络、数据库、磁盘等待上。
- 混合型任务：建议拆分线程池，例如计算线程池、DB 访问线程池、远程调用线程池分别配置。

## 面试官想考什么

- 是否只会背 `CPU 密集 N+1、IO 密集 2N+1`，还是理解背后的等待时间和计算时间。
- 是否知道线程池参数要和队列、最大线程数、拒绝策略一起设计。
- 是否能结合业务指标压测和监控，而不是拍脑袋配置。

## 标准回答

核心线程数应先按任务类型估算，再通过压测修正。CPU 密集型任务线程数不宜远大于 CPU 核数；I/O 密集型任务可以适当增加线程数，提高等待期间的 CPU 利用率。但最终要看吞吐、响应时间、错误率、CPU 利用率、队列积压和下游承载能力。

如果任务中既有计算又有 I/O，最好拆分为不同线程池，避免慢 I/O 把计算任务拖垮。生产环境还要配合有界队列、合理的最大线程数、线程命名、监控和拒绝策略。

## 深挖追问

### 线程数是不是越多越好？

不是。线程过多会带来上下文切换、内存占用、锁竞争、下游连接池耗尽等问题，吞吐可能下降，延迟可能升高。

### `2N+1` 一定适合 I/O 密集吗？

不一定。它只是粗略经验。更准确的思路是：线程数 ≈ CPU 核数 ×（1 + 等待时间 / 计算时间）。但等待时间通常需要压测或线上观测获得。

### 为什么要关注下游容量？

线程池扩大只会增加并发请求量，如果数据库连接池、Redis、第三方接口或磁盘已经到瓶颈，继续加线程只会放大超时和重试风暴。

## 实战场景/代码示例

```java
int cpu = Runtime.getRuntime().availableProcessors();

ThreadPoolExecutor ioExecutor = new ThreadPoolExecutor(
        cpu * 2,
        cpu * 4,
        60, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(1000),
        r -> new Thread(r, "order-io-worker-" + r.hashCode()),
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

上线后重点观察：

- `activeCount`：活跃线程是否长期接近上限。
- `queue.size()`：队列是否持续增长。
- 任务耗时分布：P95/P99 是否恶化。
- 拒绝次数和超时次数。
- 下游连接池和接口错误率。

## 易错点/总结

- 经验公式只能作为起点，最终以压测和监控为准。
- 不要多个业务共用一个大线程池，容易互相拖垮。
- 队列不能无界，否则高峰期可能把内存撑爆。
- 核心线程数、最大线程数、队列容量和拒绝策略必须一起设计。
