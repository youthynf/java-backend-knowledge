# 线程池为什么一定使用阻塞队列

## 核心概念

`ThreadPoolExecutor` 的构造方法第 5 个参数是 `BlockingQueue<Runnable> workQueue`，必须是阻塞队列。原因不在 API 限制（构造方法签名要求 `BlockingQueue` 接口），而在于线程池的工作模型需要一个**能阻塞等待、能线程安全转移任务**的数据结构。

工作线程的本质行为是"循环从队列取任务，取不到就等"。`BlockingQueue` 的 `take()` 在队列空时会挂起当前线程，`offer()` 满了立即返回 false；这种"空阻塞、满拒绝"的语义恰好匹配线程池的需要。

## 标准回答

线程池必须用阻塞队列，原因有三：

1. **生产消费解耦**：提交任务（生产者）和执行任务（消费者）通过队列解耦，避免直接耦合带来的同步开销。
2. **线程复用靠阻塞等待**：核心线程取不到任务时调用 `take()` 阻塞，不消耗 CPU；队列来任务后被唤醒继续工作。如果用普通队列，线程只能 busy-wait 轮询，CPU 飙满。
3. **线程安全**：`BlockingQueue` 内部用锁或 CAS 保证并发入队出队安全，不需要线程池再做同步。

## 实现原理

### Worker 取任务的源码

`ThreadPoolExecutor.getTask()` 是工作线程从队列取任务的入口：

```java
private Runnable getTask() {
    boolean timedOut = false;
    for (;;) {
        // ... 状态判断 ...
        boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;
        try {
            Runnable r = timed ?
                workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) :
                workQueue.take();   // 核心线程阻塞等待
            if (r != null) return r;
            timedOut = true;
        } catch (InterruptedException retry) {
            timedOut = false;
        }
    }
}
```

- 核心线程：`workQueue.take()`，队列空时无限期阻塞，直到有任务进来或被中断。
- 非核心线程：`workQueue.poll(keepAliveTime)`，最多阻塞 `keepAliveTime`，超时返回 null，外层逻辑会让该线程退出。

### 入队与扩容的配合

```java
// execute 中
if (isRunning(c) && workQueue.offer(command)) {  // 非阻塞入队
    // 入队成功
}
else if (!addWorker(command, false)) {  // 队列满，扩容
    reject(command);
}
```

`offer` 是非阻塞的：满了立即返回 false，触发后续的扩容和拒绝逻辑。如果用 `put`（阻塞）会让提交线程被阻塞，这违反了线程池"快速失败 + 拒绝策略"的设计意图。

### 常用阻塞队列对比

| 队列 | 数据结构 | 容量 | 适用场景 |
|------|----------|------|----------|
| `ArrayBlockingQueue` | 数组 | 有界 | 通用，削峰填谷 |
| `LinkedBlockingQueue` | 链表 | 默认无界（`Integer.MAX_VALUE`） | 一般场景，但需显式设容量 |
| `SynchronousQueue` | 不存元素 | 0 | 直接移交，CachedThreadPool 用 |
| `PriorityBlockingQueue` | 堆 | 无界 | 任务有优先级 |
| `DelayedWorkQueue` | 堆 | 无界 | 定时任务，按延迟时间排序 |

### 阻塞实现的底层

`ArrayBlockingQueue` 用 `ReentrantLock` + 两个 `Condition`（`notEmpty`、`notFull`）实现：

```java
public E take() throws InterruptedException {
    final ReentrantLock lock = this.lock;
    lock.lockInterruptibly();
    try {
        while (count == 0)
            notEmpty.await();   // 队列空，挂起
        return dequeue();
    } finally {
        lock.unlock();
    }
}

public boolean offer(E e) {
    lock.lock();
    try {
        if (count == items.length) return false;  // 满了直接返回
        enqueue(e);
        return true;
    } finally {
        lock.unlock();
    }
}
```

`LinkedBlockingQueue` 用两把分离的 `ReentrantLock`（put 和 take 各一把），并发度更高。

## 代码示例

```java
// 标准用法：有界队列
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        4, 8, 60L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(100),    // 有界阻塞队列
        new ThreadPoolExecutor.CallerRunsPolicy());

// 优先级队列示例：高优先级任务先执行
ThreadPoolExecutor priorityExecutor = new ThreadPoolExecutor(
        2, 4, 30L, TimeUnit.SECONDS,
        new PriorityBlockingQueue<>(),     // 无界，按 Comparable 排序
        new ThreadPoolExecutor.AbortPolicy());
```

任务实现 `Comparable` 才能用优先级队列：

```java
class PriorityTask implements Runnable, Comparable<PriorityTask> {
    private final int priority;
    private final Runnable work;

    public PriorityTask(int priority, Runnable work) {
        this.priority = priority;
        this.work = work;
    }

    @Override
    public int compareTo(PriorityTask o) {
        return Integer.compare(o.priority, this.priority); // 高优先级排前
    }

    @Override
    public void run() { work.run(); }
}
```

## 实战场景

| 场景 | 队列选择 | 注意点 |
|------|----------|--------|
| 通用业务线程池 | `ArrayBlockingQueue`（有界） | 容量按峰值估算 |
| 异步日志/事件 | `LinkedBlockingQueue`（显式容量） | 避免无界默认 |
| 任务直转，无缓冲 | `SynchronousQueue` | 配合较大 max |
| 延时/定时任务 | `DelayedWorkQueue` | 用 `ScheduledThreadPoolExecutor` |
| 重要任务插队 | `PriorityBlockingQueue` | 注意无界风险 |

## 深挖追问

### 能不能用 ConcurrentLinkedQueue？

不能直接用。`ConcurrentLinkedQueue` 是非阻塞队列，`poll` 在空时返回 null 而不是阻塞等待。线程池若用它，工作线程必须 busy-wait 轮询，CPU 飙满；或者自己写 `wait/notify` 包装，等于重新实现 `BlockingQueue`。

### 为什么不用普通 List + synchronized？

性能差且容易写错。`BlockingQueue` 是经过充分测试的高并发数据结构，`ArrayBlockingQueue` 一把锁、`LinkedBlockingQueue` 两把锁分离读写、`SynchronousQueue` 直接 transfer，各有优化。手写同步 List 既不高效也容易出 bug。

### LinkedBlockingQueue 默认容量是 Integer.MAX_VALUE，为什么这么设计？

为了"通用性"——让不知道设多大容量的用户也能用。但这恰好是阿里规范禁用 `Executors` 的原因：默认值掩盖了真实风险。生产环境必须显式指定容量。

### 队列大小怎么估算？

经验公式：`队列容量 ≈ 峰值 QPS × 平均 RT × 安全系数`。例如峰值 500 QPS、平均 RT 0.1s、安全系数 2，队列 ≈ 100。但更可靠的是压测观察队列积压曲线。

## 易错点

- 用 `LinkedBlockingQueue` 无参构造，实际无界，掩盖容量问题。
- 把 `PriorityBlockingQueue` 当有界队列用，任务无限堆积。
- 自定义 `BlockingQueue` 实现漏掉中断响应，导致 `shutdownNow` 无效。
- 误以为 `SynchronousQueue` 是"容量 1 的队列"，实际容量 0，必须有消费者在等。

## 总结

线程池用阻塞队列不是 API 强制那么简单，而是工作模型本身的必然要求：核心线程要阻塞等待任务、入队要快速失败触发扩容、并发入队出队要线程安全。`BlockingQueue` 的 `take/offer/poll/put` 四种语义完美匹配这些需求。理解队列选型背后的语义差异，是写好线程池配置的基础。
