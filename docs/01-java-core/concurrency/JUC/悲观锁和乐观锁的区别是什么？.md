# 悲观锁和乐观锁的区别是什么？

悲观锁和乐观锁的区别是什么？
基本概念
•  乐观锁：
乐观锁认为竞争不总会发生，因此它不需要持有锁，将比较和替换两个动作作为一个原子操作去尝试修改内存中的变量，如果失败则表示存在竞争，那么就应该有相应的重试机制；
•  悲观锁：
悲观锁认为竞争总会发生，因此每次对资源进行操作是，都会持有一个独占的锁，就像synchronized，直接上锁操作资源。

Java中实现乐观锁的方式
CAS（Compare and Swap）
CAS是乐观锁的基础，Java提供了java.util.concurrent.atomic包，包含了各种原子变量类，这些类使用了CAS操作实现了线程安全的原子操作，可以用来实现乐观锁；
版本号控制
增加一个版本号来记录数据修改的版本，每次更新时递增版本号。在更新数据的时候同时比较版本号，只有版本号一致才更新成功，否则更新失败；
时间戳
使用时间戳记录数据的更新时间，在更新数据的时候，比较时间戳是否一致，一致则更新成功，否则更新失败；

CAS有什么缺点
可能产生ABA问题
并发环境下，一个变量在两次检查之间可能被其他线程修改并恢复为原值，导致错误的判断，Java通过AtomicStampedReference通过维护一个引用值和版本戳来实现，每次更新引用同时更新版本戳，从而是的引用的变化是可检的；
循环时间长开销大
自旋CAS方式，如果长时间不成功，则会给CPU带来很大的开销；
只能保证一个共享变量的原子操作
多个可以通过AtomicStampedReference来处理或者使用synchronized实现；

# 面试复习补充

## 核心概念补充

这篇文章的主题是 **悲观锁和乐观锁的区别是什么？**。复习时应先给出定义，再说明它在 Java 并发或 JVM 体系中的位置，最后结合使用场景、限制条件和常见误区展开。

- 是否能用自己的话讲清楚概念，而不是只背术语。

- 是否能把知识点落到真实项目：如何使用、如何排查、如何调优、什么时候不该用。

## 标准回答

JUC 相关问题要说明它解决的并发痛点：减少手写同步代码、提供高性能线程安全容器、原子类、锁、同步器和任务编排工具。回答时需要把 API 用法和底层机制联系起来，例如 CAS、AQS、队列、阻塞/唤醒、弱一致性迭代等。
