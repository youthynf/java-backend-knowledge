# 悲观锁和乐观锁的区别是什么

## 核心概念

悲观锁和乐观锁不是某一种具体的锁实现，而是两种并发控制思想：

- **悲观锁**：假设冲突总会发生，每次访问共享资源前都先加锁，独占资源后再操作。典型代表 `synchronized`、`ReentrantLock`、数据库的 `SELECT ... FOR UPDATE`。
- **乐观锁**：假设冲突通常不发生，不加锁，先读出来做事，提交时再通过版本号或 CAS 校验冲突，校验失败就重试或放弃。典型代表 CAS、`AtomicInteger`、数据库的版本号机制。

一句话：**悲观锁"先锁后用"，乐观锁"先做后验"。悲观锁适合写多读少、竞争激烈的场景；乐观锁适合读多写少、冲突较少的场景。**

## 标准回答

悲观锁和乐观锁的核心区别在于对冲突的假设和处理时机：

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 冲突假设 | 总会发生 | 一般不发生 |
| 加锁时机 | 操作前加锁 | 操作时不加锁，提交时校验 |
| 冲突处理 | 等待锁释放 | 重试或放弃 |
| 并发性 | 低（互斥） | 高（无锁） |
| 适用场景 | 写多读少、竞争激烈 | 读多写少、冲突少 |
| 典型实现 | `synchronized`、`ReentrantLock`、行锁 | CAS、`Atomic*`、版本号 |

Java 中乐观锁的实现主要依赖 CAS。`java.util.concurrent.atomic` 包下的所有原子类都是 CAS 乐观锁的典型应用。

## 实现原理

### Java 中的乐观锁：CAS

CAS（Compare-And-Swap）的思想是：比较当前内存值是否等于预期旧值，相等则更新为新值，否则什么都不做。CAS 是一条 CPU 原子指令（x86 的 `cmpxchg`），Java 通过 `Unsafe.compareAndSwapInt/Long/Object` 调用。

```java
public final int incrementAndGet() {
    int oldValue;
    do {
        oldValue = value;              // 读取当前值
    } while (!compareAndSet(oldValue, oldValue + 1)); // CAS 重试
    return oldValue + 1;
}
```

ABA 问题是 CAS 的经典缺陷：线程 1 读到 A，线程 2 把 A 改成 B 又改回 A，线程 1 的 CAS 仍然成功，但中间状态丢失了。解决方案：

- `AtomicStampedReference`：加版本号，每次更新版本号递增。
- `AtomicMarkableReference`：加布尔标记。

### 数据库中的乐观锁

数据库通常用版本号或时间戳实现乐观锁：

```sql
UPDATE goods SET stock = stock - 1, version = version + 1
WHERE id = ? AND version = ?;
```

如果 `version` 不匹配，更新影响行数为 0，业务层重试或返回失败。

### 数据库中的悲观锁

```sql
SELECT * FROM goods WHERE id = 1 FOR UPDATE;
```

行级排他锁持有到事务结束，其他事务必须等待。

## 代码示例

### 乐观锁：AtomicInteger 计数器

```java
import java.util.concurrent.atomic.AtomicInteger;

public class OptimisticCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // CAS 自旋
    }

    public int get() {
        return count.get();
    }
}
```

### 悲观锁：ReentrantLock 计数器

```java
import java.util.concurrent.locks.ReentrantLock;

public class PessimisticCounter {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }

    public int get() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }
}
```

### 乐观锁解决 ABA：AtomicStampedReference

```java
import java.util.concurrent.atomic.AtomicStampedReference;

public class AbaDemo {
    public static void main(String[] args) {
        AtomicStampedReference<Integer> ref =
            new AtomicStampedReference<>(100, 1);

        new Thread(() -> {
            int stamp = ref.getStamp();
            Integer value = ref.getReference();
            ref.compareAndSet(value, 200, stamp, stamp + 1);
            ref.compareAndSet(200, 100, stamp + 1, stamp + 2); // 改回 100
        }).start();

        new Thread(() -> {
            int stamp = ref.getStamp();
            try { Thread.sleep(100); } catch (InterruptedException ignored) {}
            // 即使值改回 100，stamp 已经变了，CAS 失败
            boolean ok = ref.compareAndSet(100, 300, stamp, stamp + 1);
            System.out.println("CAS result: " + ok);
        }).start();
    }
}
```

## 实战场景

| 场景 | 选型 | 用法 | 注意点 |
|------|------|------|--------|
| 高并发计数 | 乐观锁 | `AtomicLong` 或 `LongAdder` | `LongAdder` 在热点计数场景更优 |
| 库存扣减 | 悲观锁 | `SELECT FOR UPDATE` 行锁 | 注意死锁，按固定顺序加锁 |
| 库存扣减（弱冲突） | 乐观锁 | 版本号 CAS | 失败要重试，重试次数限制 |
| 状态机更新 | 乐观锁 | `UPDATE ... WHERE status=?` | 利用业务字段当版本 |
| 银行转账 | 悲观锁 | 事务 + 行锁 | 严格一致性，避免脏读 |
| 缓存更新 | 乐观锁 | CAS 比较旧值 | 写多时自旋开销大 |

## 深挖追问

### 1. CAS 有什么缺点？

- **ABA 问题**：值从 A 改 B 再改回 A，CAS 检测不到中间变化。`AtomicStampedReference` 加版本号解决。
- **自旋开销大**：长时间 CAS 失败会占用 CPU。高并发场景 `LongAdder` 通过分散热点缓解。
- **只能保证单个变量原子**：多个变量需要 `AtomicReference` 包装成对象，或用 `synchronized`。

### 2. 悲观锁和乐观锁能混用吗？

可以。典型场景是数据库层用乐观锁（版本号），应用层用悲观锁（`synchronized` 或 `ReentrantLock`）保护内存状态。Redis 分布式锁可视作分布式悲观锁。

### 3. CAS 和 synchronized 哪个性能更好？

- 无竞争或低竞争：CAS 更快，无锁，无上下文切换。
- 中等竞争：CAS 自旋开始有开销，`synchronized` 经过偏向锁、轻量级锁升级，开销也可控。
- 高竞争：CAS 自旋浪费 CPU，`synchronized` 阻塞唤醒更合适。

JVM 的 `synchronized` 经过锁升级优化（偏向锁 -> 轻量级锁 -> 重量级锁），实际性能并不差。简单临界区优先 `synchronized`。

### 4. LongAdder 为什么比 AtomicLong 更适合高并发？

`AtomicLong` 所有线程竞争同一个 `value` 字段，高并发时 CAS 失败率高。`LongAdder` 把计数分散到 `Cell[]` 数组，每个线程哈希到不同槽位独立累加，`sum()` 时再汇总。竞争越激烈，`LongAdder` 优势越明显。

### 5. 乐观锁一定无锁吗？

不一定。乐观锁是一种"先做后验"的思想，实现可以无锁（CAS）也可以带锁（数据库版本号其实依赖事务和行锁，但业务层逻辑是乐观的）。乐观的本质是不阻塞读，提交时校验。

## 易错点

- 把"乐观锁"等同于"CAS"：CAS 只是乐观锁的一种实现，版本号、时间戳也是乐观锁。
- 高竞争场景硬用 CAS：自旋开销可能比 `synchronized` 还大。
- 忽略 ABA 问题：单纯 CAS 不感知中间变化，关键场景加版本号。
- 乐观锁不重试就放弃：版本号失败要重试，但必须有重试上限，否则活锁。
- 悲观锁持有时间过长：临界区里包含 IO、RPC 会导致吞吐崩塌。

## 总结

悲观锁假设冲突必然发生，先加锁再操作；乐观锁假设冲突很少发生，先操作提交时校验。Java 乐观锁主要靠 CAS 实现（`Atomic*`），悲观锁主要靠 `synchronized` 和 `ReentrantLock`。选型看竞争强度和读写比例：低竞争读多用乐观，高竞争写多用悲观。CAS 的三大缺点是 ABA、自旋开销、单变量原子；`AtomicStampedReference`、`LongAdder`、版本号机制分别应对。
