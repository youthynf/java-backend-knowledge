# 公平锁与非公平锁实现原理与区别是什么？

公平锁与非公平锁实现原理与区别是什么？
在 Java 中，公平锁与非公平锁主要通过 ReentrantLock 类来体现，以下从多个方面对它们进行详细比较：

锁获取顺序
•  公平锁：
遵循先来先得的原则，按照线程请求锁的先后顺序来分配锁。例如，有线程 A、B、C 依次请求锁，在锁可用时，一定会先把锁分配给线程 A，只有当线程 A 释放锁后，才会考虑线程 B，依此类推。它通过内部的队列机制（借助 AbstractQueuedSynchronizer，即 AQS 的相关逻辑）严格维护等待线程的顺序，确保每个线程都能按照请求的先后顺序获取到锁，不会出现后请求的线程插队获取锁的情况。

•  非公平锁：
线程获取锁的顺序具有随机性，不保证与请求锁的顺序一致。多个线程竞争锁时，每个线程获取锁的机会是均等的，即使线程 A 先请求锁，但是线程 B 在锁恰好可用时，有可能会直接抢到锁并占有，并不一定遵循先来后到的顺序。新到来的线程和已经等待的线程在锁释放时会同时参与竞争，谁先抢到算谁的。默认情况下，ReentrantLock 构造出来的就是非公平锁。

实现机制
•  公平锁：
在 ReentrantLock 内部，公平锁的实现依赖于 AQS 的同步队列（CLH 队列的一种变体）。当线程请求锁时，如果锁已经被占用，该线程会被添加到同步队列的尾部，按照入队顺序排队等待。当锁被释放时，锁会从队列头部唤醒下一个等待线程，将锁分配给它，然后该线程出队获取锁并执行相应代码，通过这种严格的队列操作来保证公平性。

•  非公平锁：
同样基于 AQS 机制，但在锁释放时，不会只从队列头部唤醒等待线程来获取锁。而是当锁被释放的瞬间，新到来请求锁的线程和同步队列中等待的线程会一同竞争锁，利用底层的 CAS （比较与交换）操作等机制来尝试获取锁，谁先通过 CAS 操作成功修改锁的状态（比如将锁的拥有者从当前线程置为自己），谁就能获取到锁，所以其实现过程相对公平锁来说少了严格按顺序唤醒队列中线程的操作，更具随机性。

性能表现
公平锁：
•  优点：能确保每个线程都有公平获取锁的机会，避免了线程长时间等待却始终无法获取锁的 “饥饿” 现象，在对资源访问顺序要求严格、需要保证公平性的场景下非常适用，例如一些资源分配系统，要求按照客户端申请资源的先后顺序来分配资源，使用公平锁可以很好地满足这种需求，使系统的并发行为更加有序、可预测。
•  缺点：由于要严格维护等待线程的队列顺序，每次释放锁都需要从队列头部唤醒下一个线程，涉及更多的队列操作以及线程上下文切换等开销，在高并发场景下，获取锁的效率相对较低，整体性能可能不如非公平锁。尤其是在锁竞争激烈的情况下，频繁的队列操作会成为性能瓶颈。

非公平锁：
•  优点：在高并发场景下，因为不需要严格按照顺序排队等待锁，多个线程可以直接竞争，有机会让新到来的线程更快地抢到锁（如果正好锁释放时它能抢到），减少了线程等待的时间，整体获取锁的效率往往更高，能提升系统的并发性能，更适合对性能要求较高、对获取锁顺序没有严格要求的场景，比如常见的线程池执行任务场景，多个任务由不同线程执行，使用非公平锁获取锁去访问共享资源（如数据库连接池中的连接等）可以加快任务执行速度。
•  缺点：由于是随机抢占锁，可能会导致部分线程长时间得不到锁，出现 “饥饿” 现象，特别是在竞争比较激烈的情况下，一些运气不好的线程可能会一直处于等待状态，无法及时获取到锁进行资源的访问，不符合对公平性有严格要求的应用场景。

适用场景
•  公平锁：
适用于那些对公平性要求极高，希望各个线程按照请求顺序依次获取锁、访问共享资源的场景。例如，在一个多用户排队购票系统中，每个用户相当于一个线程，为了保证购票顺序的公平公正，就需要使用公平锁来控制对票务资源的访问，按照用户发起购票请求的先后顺序来分配锁，让用户依次进行购票操作。

•  非公平锁：
应用场景更为广泛，特别是在并发程度较高、对获取锁顺序没有严格要求的情况下，优先考虑使用非公平锁来提升系统性能。比如，在一个电商系统的后台，多个线程负责处理不同订单的相关操作（如库存更新、物流信息更新等），这些操作对顺序并没有严格要求，使用非公平锁可以让线程更快速地获取锁执行任务，提高系统整体的处理效率。

总的来说，Java 中的公平锁和非公平锁各有特点，在实际应用中需要根据具体的业务场景、对公平性的要求以及性能考量等因素综合选择合适的锁机制。
