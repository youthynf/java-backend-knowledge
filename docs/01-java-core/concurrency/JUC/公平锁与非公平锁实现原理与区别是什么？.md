# 公平锁与非公平锁实现原理与区别是什么

## 核心概念

公平锁和非公平锁是按"线程获取锁的顺序"划分的两种锁策略：

- **公平锁**：线程按请求锁的先后顺序排队获取锁，先到先得。AQS 队列中排在最前面的线程一定先获得锁。
- **非公平锁**：新到来的线程可以"插队"尝试 CAS 抢锁，抢不到再排队。即使队列里有等待线程，新线程也可能插队成功。

`ReentrantLock` 默认是非公平锁：

```java
public ReentrantLock() { sync = new NonfairSync(); }
public ReentrantLock(boolean fair) { sync = fair ? new FairSync() : new NonfairSync(); }
```

`Semaphore`、`ReentrantReadWriteLock` 同样支持公平/非公平两种模式。

一句话：**公平锁保证先到先得但吞吐量低；非公平锁允许插队，吞吐量更高但可能让队列线程"饥饿"。Java 默认非公平，因为多数业务对顺序无强需求。**

## 标准回答

公平锁和非公平锁的核心差异是：锁被释放的瞬间，新来的线程是否可以和队列头部的等待线程一起竞争。

| 维度 | 公平锁 | 非公平锁 |
|------|--------|----------|
| 获取顺序 | 严格 FIFO，先到先得 | 新线程可插队 |
| 队列检查 | `tryAcquire` 前调用 `hasQueuedPredecessors()` | 不检查 |
| 吞吐量 | 低 | 高 |
| 饥饿风险 | 几乎无 | 可能存在 |
| 上下文切换 | 多 | 少 |
| 典型实现 | `new ReentrantLock(true)` | `new ReentrantLock()` / `new ReentrantLock(false)` |

非公平锁吞吐更高的原因：当锁释放的瞬间，新线程恰好在线程调度上"热度更高"，可以直接 CAS 拿到锁，避免了从队列里 unpark 一个老线程的上下文切换开销。

## 实现原理

### 非公平锁 NonfairSync.tryAcquire

```java
// ReentrantLock.NonfairSync
final void lock() {
    if (compareAndSetState(0, 1))              // 上来直接 CAS 抢
        setExclusiveOwnerThread(Thread.currentThread());
    else
        acquire(1);
}

protected final boolean tryAcquire(int acquires) {
    return nonfairTryAcquire(acquires);
}

final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        if (compareAndSetState(0, acquires)) { // 不检查队列，直接 CAS
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        int nextc = c + acquires;              // 重入
        setState(nextc);
        return true;
    }
    return false;
}
```

关键点：`c == 0` 时直接 CAS 抢锁，**不检查队列中是否有前驱节点**。

### 公平锁 FairSync.tryAcquire

```java
// ReentrantLock.FairSync
final void lock() { acquire(1); }

protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        if (hasQueuedPredecessors() == false && // 关键差异：检查队列
            compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        int nextc = c + acquires;
        setState(nextc);
        return true;
    }
    return false;
}
```

`hasQueuedPredecessors()` 检查 AQS 队列中是否有排在自己前面的线程：

```java
public final boolean hasQueuedPredecessors() {
    Node h, s;
    return (h = head) != null &&
           ((s = h.next) == null || s.thread != Thread.currentThread());
}
```

如果队列里有前驱线程，公平锁会让出 CPU 进入队列排队，不会插队。

### 性能差异的根源

锁释放时的线程调度路径：

- 非公平：持锁线程 `unlock()` -> CAS 把 `state` 改 0 -> 唤醒后继 -> 后继被 OS 调度 -> 后继真正运行。这一过程可能耗时数百微秒。期间新来的线程刚好在 CPU 上，可以直接 CAS 拿到锁，立刻进入临界区，省去了唤醒老线程的开销。
- 公平：持锁线程 `unlock()` -> 唤醒后继 -> 后继被 OS 调度 -> 后继 `tryAcquire` 通过 -> 进入临界区。即使新线程想抢，`hasQueuedPredecessors()` 返回 true，必须排队。

非公平的代价是队列中的线程可能被反复插队，长时间拿不到锁（饥饿）。但实际 JVM 调度有平衡机制，饥饿是罕见情况。

## 代码示例

### 非公平锁（默认）

```java
import java.util.concurrent.locks.ReentrantLock;

ReentrantLock lock = new ReentrantLock(); // 默认非公平
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();
}
```

### 公平锁

```java
ReentrantLock fairLock = new ReentrantLock(true);
```

### 验证公平性

```java
import java.util.concurrent.locks.ReentrantLock;

public class FairnessDemo {
    private static final ReentrantLock lock = new ReentrantLock(true); // 改成 false 试试

    public static void main(String[] args) {
        for (int i = 0; i < 5; i++) {
            new Thread(() -> {
                for (int j = 0; j < 3; j++) {
                    lock.lock();
                    try {
                        System.out.println(Thread.currentThread().getName() + " 持有锁");
                    } finally {
                        lock.unlock();
                    }
                }
            }, "T" + i).start();
        }
    }
}
```

公平模式下，输出顺序更接近 `T0 T1 T2 T3 T4 T0 T1 ...`；非公平模式下，可能看到 `T0 T0 T0 T1 T1 T1 ...` 这样的连续获取。

## 实战场景

| 场景 | 选型 | 注意点 |
|------|------|--------|
| 通用业务临界区 | 非公平（默认） | 吞吐优先 |
| 订单库存扣减 | 非公平 | 顺序无强需求 |
| 资源分配系统 | 公平 | 按申请顺序分配，避免饥饿 |
| 多用户排队购票 | 公平 | 业务要求先到先得 |
| 高频 RPC 入口保护 | 非公平 | 减少上下文切换 |
| 限流场景 | 视业务而定 | `Semaphore` 默认非公平，公平模式可能让长连接饥饿 |

## 深挖追问

### 1. 公平锁一定不会饥饿吗？

理论上几乎不会，因为严格 FIFO 保证每个线程都能轮到。但极端情况下，如果队列很长且新任务源源不断，排队时间仍然可能很长。公平锁解决的是"插队导致的饥饿"，不是"排队等待时间过长"。

### 2. 非公平锁的"插队"具体是怎么发生的？

发生在两个时机：

- **`lock()` 入口**：`NonfairSync.lock()` 第一步直接 CAS 抢锁，连队列都不看。
- **`tryAcquire` 时**：`c == 0` 时直接 CAS，不调用 `hasQueuedPredecessors()`。

如果抢不到才走 `acquire` -> `addWaiter` 入队 -> `acquireQueued` 排队。

### 3. 为什么 AQS 默认是非公平？

Doug Lea 在 AQS 注释里明确说明：非公平锁在吞吐量上有显著优势。原因有二：

- 唤醒一个被 `park` 的线程有 OS 调度延迟，期间锁是空闲的，新线程直接拿锁更高效。
- 公平锁严格排队导致上下文切换更频繁。

只有业务对顺序敏感或确认出现饥饿时，才用公平锁。

### 4. `tryLock()` 是公平还是非公平？

`tryLock()`（无参）是**非公平**的——它直接调用 `nonfairTryAcquire`，即使队列里有等待线程也会插队。这是 AQS 设计上的有意选择，避免"试探性获取"也走完整排队流程。

`tryLock(timeout, unit)` 走的是公平或非公平（取决于锁类型），但同样是先尝试 CAS。

### 5. `ReentrantReadWriteLock` 的公平和非公平有什么特殊处理？

公平模式下，如果队列中有等待写锁的线程，新的读锁请求也会排队（避免写锁饥饿）。

非公平模式下，读锁可以"插队"——只要当前是读锁持有，新的读锁请求可以立即获取，但写锁仍然要等所有读锁释放。这导致写锁容易饥饿。

## 易错点

- 误以为 `ReentrantLock` 默认是公平锁：默认是非公平。
- 误以为公平锁性能更好：实际吞吐通常更低。
- 误以为非公平锁一定饥饿：JVM 调度有平衡，饥饿是罕见情况。
- 误以为 `tryLock()` 是公平的：无参 `tryLock()` 是非公平的。
- 公平场景下滥用公平锁：严格 FIFO 增加上下文切换，可能拖慢整体吞吐。

## 总结

公平锁按 FIFO 严格排队，先到先得但吞吐低；非公平锁允许新线程插队 CAS 抢锁，吞吐高但可能饥饿。两者在 AQS 中的核心差异是 `tryAcquire` 前是否调用 `hasQueuedPredecessors()` 检查队列。`ReentrantLock`、`Semaphore`、`ReentrantReadWriteLock` 默认都是非公平，因为唤醒被 `park` 线程有调度开销，新线程直接抢锁能省去这部分。只有业务要求严格先来后到或确认饥饿时才用公平锁。
