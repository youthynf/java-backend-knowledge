# 行为型模式-模板方法模式

行为型模式-模板方法模式
定义一个操作中的算法骨架，将一些步骤延迟到子类中，使子类可以不改变算法结构而重定义特定步骤。关键要点：抽象类定义模板方法（固定流程）+ 抽象方法（子类实现可变步骤），模板方法用final修饰防止篡改流程。

// 抽象类（定义算法骨架）
public abstract class CaffeineBeverage {
    // 模板方法：固定流程，声明为final防止子类修改
    public final void prepareRecipe() {
        boilWater();       // 步骤1：煮水（固定）
        brew();            // 步骤2：冲泡（子类实现）
        pourInCup();       // 步骤3：倒入杯子（固定）
        addCondiments();   // 步骤4：加调料（子类实现）
    }

    // 抽象方法：由子类实现
    protected abstract void brew();
    protected abstract void addCondiments();

    // 具体方法：固定实现
    private void boilWater() {
        System.out.println("煮水");
    }

    private void pourInCup() {
        System.out.println("倒入杯子");
    }
}

// 具体子类：咖啡
public class Coffee extends CaffeineBeverage {
    @Override
    protected void brew() {
        System.out.println("用沸水冲泡咖啡");
    }

    @Override
    protected void addCondiments() {
        System.out.println("加糖和牛奶");
    }
}

// 具体子类：茶
public class Tea extends CaffeineBeverage {
    @Override
    protected void brew() {
        System.out.println("用沸水浸泡茶叶");
    }

    @Override
    protected void addCondiments() {
        System.out.println("加柠檬");
    }
}

// 使用示例
public class Client {
    public static void main(String[] args) {
        CaffeineBeverage coffee = new Coffee();
        System.out.println("准备咖啡：");
        coffee.prepareRecipe();
        // 输出：
        // 准备咖啡：
        // 煮水
        // 用沸水冲泡咖啡
        // 倒入杯子
        // 加糖和牛奶

        CaffeineBeverage tea = new Tea();
        System.out.println("\n准备茶：");
        tea.prepareRecipe();
        // 输出：
        // 准备茶：
        // 煮水
        // 用沸水浸泡茶叶
        // 倒入杯子
        // 加柠檬
    }
}

## 核心概念
行为型模式-模板方法模式 可以放在“工程实践能力”这条主线里理解。复习时不要只背结论，要先说明它解决的核心问题，再解释关键机制、适用边界和代价。围绕这个知识点，重点关注：定义、原理、边界、取舍、常见问题、排查方法和落地成本。如果面试官继续追问，通常会从“为什么这样设计、在什么场景会失效、线上如何排查”三个方向展开。

## 面试回答与追问
- **标准回答**：先给出 行为型模式-模板方法模式 的定位，再说明它依赖的核心原理，最后结合业务场景说明如何使用。回答时要把“能解决什么问题”和“会带来什么成本”一起讲清楚。
- **常见追问**：如果数据量、并发量或调用链路继续放大，行为型模式-模板方法模式 的瓶颈会出现在哪里？如何观测、如何优化、如何回滚？
- **易错点**：不要把概念和具体实现混在一起，也不要只说 API 名称。面试中更重要的是说清楚边界条件、失败场景和取舍依据。

## 实战场景与排查
典型落地场景包括：真实业务开发、线上问题治理、性能优化、协作交付和面试复盘。实际处理线上问题时，可以按“现象确认 → 指标采集 → 假设验证 → 小步修复 → 复盘沉淀”的路径推进。先看日志、监控、链路追踪和核心指标，再判断是容量问题、配置问题、代码路径问题，还是外部依赖抖动。

## 总结
复习 行为型模式-模板方法模式 时，建议把它和相邻知识点放在一起比较：相同点是什么、区别在哪里、为什么当前场景选择它而不是替代方案。能讲清楚这些内容，才算真正掌握。
