# 抽象类与接口区别是什么

## 核心概念

抽象类（Abstract Class）和接口（Interface）都是 Java 中实现"抽象"的机制，但设计意图不同：

- **抽象类**：对一类事物的**模板抽象**，表达 "is-a" 关系，可以包含状态（字段）和具体方法实现。子类继承抽象类复用代码。
- **接口**：对一组行为的**契约抽象**，表达 "can-do" 能力，只定义方法签名（Java 8+ 可有 default/static 方法）。类实现接口获得能力。

简单说：抽象类定义"是什么"，接口定义"能做什么"。

```java
// 抽象类：模板
abstract class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }  // 抽象类有构造器
    public abstract void speak();                       // 抽象方法
    public void eat() { System.out.println("eating"); } // 具体方法
}

// 接口：契约
interface Flyable {
    void fly();
    default void land() { System.out.println("landing"); }  // Java 8+ default
}
```

## 标准回答

抽象类和接口都用于抽象，但本质不同。要点：

1. **设计意图**：抽象类是"is-a"模板（有状态、有具体实现），接口是"can-do"契约（无状态、行为规范）。
2. **继承关系**：类只能单继承抽象类，可以多实现接口。
3. **字段**：抽象类可有任意成员变量；接口只能有 `public static final` 常量。
4. **构造器**：抽象类有构造器（供子类调用）；接口没有。
5. **方法实现**：抽象类可有具体方法；Java 8+ 接口可有 `default`/`static` 方法，Java 9+ 可有 `private` 方法。
6. **优先用接口**：除非需要共享状态或模板代码，否则优先接口。

## 实现原理

### 1. 关键差异对比

| 特性 | 抽象类 | 接口 |
|------|--------|------|
| 实例化 | 不能 | 不能 |
| 构造方法 | 可以有 | 不能有 |
| 成员变量 | 任意类型 | 隐式 `public static final` |
| 方法实现 | 可有抽象和具体方法 | Java 8+ 可有 `default`/`static`，Java 9+ 可有 `private` |
| 访问修饰符 | 任意 | 方法隐式 `public`，Java 9+ 支持 `private` |
| 多继承 | 单继承 | 多实现 |
| 静态初始化块 | 可以有 | 不能有 |
| 设计语义 | "is-a" 模板 | "can-do" 契约 |

### 2. 抽象类的本质

抽象类用 `abstract` 修饰，可以包含抽象方法（无实现）和具体方法（有实现）。抽象类不能实例化，但可以有构造器供子类调用。

```java
public abstract class AbstractRepository<T> {
    protected final Class<T> entityClass;

    protected AbstractRepository(Class<T> entityClass) {
        this.entityClass = entityClass;   // 构造器，子类 super 时调用
    }

    public abstract T findById(long id);  // 抽象方法

    public void save(T entity) {          // 具体方法，子类直接复用
        System.out.println("saving " + entity);
    }
}

public class UserRepository extends AbstractRepository<User> {
    public UserRepository() {
        super(User.class);
    }

    @Override
    public User findById(long id) {
        // 子类实现具体逻辑
        return new User(id);
    }
}
```

抽象类常用于**模板方法模式**：定义算法骨架，子类填充关键步骤。

### 3. 接口的本质

接口是纯契约的抽象，定义方法签名但不实现（Java 8 前）。接口的所有字段隐式 `public static final`，所有方法隐式 `public abstract`（Java 8 前）。

```java
public interface Comparable<T> {
    int compareTo(T other);
}

public interface Serializable {
    // 标记接口，无方法
}

public class Student implements Comparable<Student>, Serializable {
    @Override
    public int compareTo(Student o) { /* ... */ return 0; }
}
```

一个类可以实现多个接口，相当于获得多种"能力"。

### 4. Java 8 对接口的增强

Java 8 引入 `default` 方法和 `static` 方法，让接口可以提供默认实现：

```java
public interface Flyable {
    void fly();

    default void land() {
        System.out.println("default landing");  // default 方法，实现类可直接用或重写
    }

    static Flyable noop() {
        return () -> System.out.println("noop fly");  // 静态工厂方法
    }
}
```

`default` 方法的引入让接口可以向后演进：给已有接口添加新方法时，提供 default 实现就不会破坏已有实现类。Java 8 在 `Collection` 接口加 `stream()`、`forEach()` 等 default 方法就是典型。

### 5. Java 9 对接口的增强

Java 9 允许接口有 `private` 方法，用于 default 方法之间的代码复用：

```java
public interface Loggable {
    default void logInfo(String msg) { log("INFO", msg); }
    default void logError(String msg) { log("ERROR", msg); }

    private void log(String level, String msg) {  // Java 9+ private 方法
        System.out.println("[" + level + "] " + msg);
    }
}
```

### 6. default 方法冲突

一个类实现多个接口，如果接口之间有同名 default 方法，编译器要求重写：

```java
interface A {
    default void hello() { System.out.println("A"); }
}
interface B {
    default void hello() { System.out.println("B"); }
}

class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();  // 显式选择 A 的实现
    }
}
```

如果父类和接口都有同名方法，**类优先**：父类的方法胜出，接口的 default 被忽略。

### 7. 何时用抽象类，何时用接口

**用抽象类**：

- 多个子类有共同状态（字段）和共享逻辑。
- 需要模板方法模式：父类定义骨架，子类填充步骤。
- 需要 `protected` 访问控制。
- 需要构造器初始化字段。

**用接口**：

- 定义能力契约，不关心实现细节。
- 类型需要多种能力（多实现）。
- 想为类型定义"行为规范"。
- Java 8+ 后，default 方法让接口也能提供实现，进一步倾向接口。

实际开发中**优先用接口**，需要共享状态或模板代码时再引入抽象类。Spring 全家桶大量用接口（`BeanPostProcessor`、`FactoryBean`、`InitializingBean`），抽象类通常是接口的"适配器"（如 `AbstractBeanFactory` 实现 `BeanFactory`，提供默认实现）。

## 代码示例

### 模板方法模式（抽象类）

```java
public abstract class DataPipeline {
    public final void execute() {          // 模板方法，骨架不可变
        String raw = extract();
        String data = transform(raw);
        load(data);
    }

    protected abstract String extract();
    protected abstract String transform(String raw);
    protected abstract void load(String data);
}

public class CsvToDbPipeline extends DataPipeline {
    @Override
    protected String extract() { return "raw csv"; }
    @Override
    protected String transform(String raw) { return "transformed: " + raw; }
    @Override
    protected void load(String data) { System.out.println("save: " + data); }
}
```

### 多能力组合（接口）

```java
public interface Serializable { byte[] serialize(); }
public interface Cloneable { Object clone(); }
public interface Comparable<T> { int compareTo(T o); }

public class Product implements Serializable, Comparable<Product> {
    private final long id;
    private final String name;

    public Product(long id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    public byte[] serialize() {
        return (id + ":" + name).getBytes();
    }

    @Override
    public int compareTo(Product o) {
        return Long.compare(this.id, o.id);
    }
}
```

### 接口 + 抽象类配合

```java
public interface Repository<T> {
    T findById(long id);
    void save(T entity);
}

public abstract class AbstractRepository<T> implements Repository<T> {
    protected final Map<Long, T> store = new HashMap<>();

    @Override
    public T findById(long id) {
        return store.get(id);   // 通用实现
    }
    // save 留给子类
}

public class UserRepository extends AbstractRepository<User> {
    @Override
    public void save(User entity) {
        store.put(entity.getId(), entity);
    }
}
```

接口定义契约，抽象类提供默认实现，具体类完成剩余逻辑。这是 JDK 集合框架（`List` + `AbstractList` + `ArrayList`）和 Spring 框架的常见模式。

## 实战场景

| 场景 | 选择 | 原因 |
|------|------|------|
| Spring `AbstractApplicationContext` | 抽象类 | 模板方法定义生命周期，子类只实现 `refreshBeanFactory()` |
| `List`、`Map` 等集合接口 | 接口 | 定义行为规范，多种实现（`ArrayList`、`LinkedList`） |
| `BaseController` 公共逻辑 | 抽象类 | 统一处理参数校验、异常捕获、日志 |
| Spring Data `Repository` | 接口 | 只定义方法签名，运行时动态代理生成实现 |
| 模板方法模式 | 抽象类 | 父类定骨架，子类填步骤 |
| 策略模式 | 接口 | 多种策略实现同一契约 |
| 标记接口（`Serializable`、`Cloneable`） | 接口 | 仅标记语义，无方法 |

## 深挖追问

### 1. 抽象类和接口的根本区别

抽象类是"类的抽象"——它本身是类，有状态、有构造器、有具体方法，子类继承复用。接口是"行为的抽象"——它是契约，无状态、无构造器，类通过实现获得能力。

抽象类回答"是什么"，接口回答"能做什么"。

### 2. Java 8 之后，抽象类还有存在意义吗

有。default 方法让接口能提供实现，但接口仍然**不能有状态**（字段必须是常量）、**没有构造器**、**不能 `protected` 访问**。需要这些特性时仍要抽象类。模板方法模式依赖父类定骨架、子类填步骤，这种"代码复用"场景抽象类更合适。

### 3. 接口能 `new` 吗

不能直接 `new`，但可以用匿名内部类实现：

```java
Runnable r = new Runnable() {
    @Override
    public void run() { System.out.println("running"); }
};
```

或者 Java 8+ 用 Lambda：

```java
Runnable r = () -> System.out.println("running");
```

这本质是创建实现接口的实例，不是"new 接口"。

### 4. 一个类可以继承多个接口，能继承多个抽象类吗

接口可以多实现，抽象类只能单继承。Java 通过接口避免了 C++ 多继承的"钻石问题"（同一个方法在不同父类有不同实现，调用时歧义）。

接口的 default 方法引入了新的冲突场景，但 Java 规定同名 default 方法必须由实现类重写解决。

### 5. 抽象类能不能有 `main` 方法

可以。`main` 是静态方法，抽象类可以有静态方法。可以直接运行抽象类的 `main` 方法，不需要子类。

### 6. 标记接口有什么用

标记接口（如 `Serializable`、`Cloneable`）没有方法，仅用于"标记"类具备某种属性。`ObjectOutputStream` 通过 `instanceof Serializable` 判断能否序列化。Java 5+ 引入注解后，标记接口的很多场景被注解替代（如 `@FunctionalInterface`），但 `Serializable` 等仍保留接口形式。

## 易错点

- 在接口里写 `private` 字段或非 `final` 字段，编译错误（接口字段隐式 `public static final`）。
- 期望接口有构造器，实际没有。
- 子类继承抽象类但没实现所有抽象方法，必须自己也声明 `abstract`。
- 多个接口有同名 default 方法且未重写，编译错误。
- 父类和接口有同名方法，以为接口 default 方法会"加成"，实际被父类方法覆盖。
- 误以为接口可以继承抽象类，实际接口只能继承接口。
- 抽象类用 `final` 修饰，编译错误（`final` 类不能被继承，但抽象类必须被继承才有意义）。
- 把 `default` 方法当 `abstract` 用，期望实现类必须重写，实际可选。

## 总结

抽象类和接口是 Java 实现抽象的两种机制。抽象类是"is-a"模板，有状态和具体实现，单继承；接口是"can-do"契约，无状态，多实现。Java 8 引入 default 方法后，接口也能提供实现，但仍不能有状态、构造器、`protected` 访问，需要这些特性时仍用抽象类。生产实践中优先用接口定义能力契约，需要模板方法模式或共享状态时引入抽象类。两者常配合使用：接口定义契约，抽象类提供骨架实现，具体类完成剩余逻辑——这是 JDK 集合和 Spring 框架的常见模式。
