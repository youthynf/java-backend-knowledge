# 抽象类与接口区别是什么？

## 核心概念

抽象类（Abstract Class）和接口（Interface）是 Java 中实现抽象的两种机制，它们的设计目的和使用方式有本质区别。

| 特性 | 抽象类 | 接口 |
|------|--------|------|
| **实例化** | 不能实例化 | 不能实例化 |
| **构造方法** | ✅ 可以有 | ❌ 不能有 |
| **成员变量** | 可以有任意成员变量 | 只能有 `public static final` 常量 |
| **方法实现** | 可以有抽象方法和具体方法 | Java 8+ 可有 default/static 方法 |
| **多继承** | ❌ 只能单继承 | ✅ 可以多实现 |
| **访问修饰符** | 方法可以是任意访问权限 | 方法默认 `public` |
| **设计理念** | "is-a" 关系，模板设计 | "can-do" 能力，行为契约 |

## 面试高频问题

### 1. 抽象类和接口的根本区别是什么？

- **抽象类**是对类的抽象，是一种**模板设计**——子类继承父类的特征和行为
- **接口**是对行为的抽象，是一种**规范约束**——实现类必须具备某种能力
- 抽象类定义"是什么"，接口定义"能做什么"

### 2. 什么时候用抽象类，什么时候用接口？

- **用抽象类**：多个子类有共同特征和行为，需要共享代码（如模板方法模式）
- **用接口**：定义能力契约，不关心实现细节（如 `Comparable`、`Runnable`）
- **实际开发**：优先使用接口，需要共享代码时引入抽象类

### 3. Java 8/9 对接口的增强有什么影响？

```java
// Java 8: default 方法
public interface Flyable {
    void fly();
    
    default void land() {
        System.out.println("准备降落"); // 提供默认实现
    }
}

// Java 9: private 方法
public interface Loggable {
    default void logInfo(String msg) { log("INFO", msg); }
    default void logError(String msg) { log("ERROR", msg); }
    
    private void log(String level, String msg) {  // 接口内部复用
        System.out.println("[" + level + "] " + msg);
    }
}
```

> ⚠️ 这使得接口和抽象类的界限变得模糊，但核心区别不变：抽象类仍是单继承，接口仍可多实现。

## 代码示例

### 抽象类：模板方法模式

```java
public abstract class DataParser {
    // 模板方法：定义算法骨架
    public final void parse(String input) {
        String data = readFile(input);
        String result = processData(data);  // 子类实现
        writeOutput(result);
    }
    
    protected abstract String processData(String data);
    
    private String readFile(String input) {
        return "raw: " + input;  // 公共逻辑
    }
    
    private void writeOutput(String result) {
        System.out.println(result);
    }
}

public class JsonParser extends DataParser {
    @Override
    protected String processData(String data) {
        return "JSON parsed → " + data;
    }
}
```

### 接口：多实现组合能力

```java
public interface Serializable {
    byte[] toBytes();
}

public interface Comparable<T> {
    int compareTo(T other);
}

// 一个类可以同时具备多种能力
public class Student implements Serializable, Comparable<Student> {
    private String name;
    private int score;
    
    @Override
    public byte[] toBytes() {
        return name.getBytes();
    }
    
    @Override
    public int compareTo(Student other) {
        return Integer.compare(this.score, other.score);
    }
}
```

## 实战场景

| 场景 | 选择 | 原因 |
|------|------|------|
| Spring 框架的 `AbstractApplicationContext` | 抽象类 | 模板方法定义生命周期，子类只需实现 `refreshBeanFactory()` |
| `List` 接口 | 接口 | 定义集合行为规范，`ArrayList` 和 `LinkedList` 各自实现 |
| `BaseController` 公共逻辑 | 抽象类 | 统一处理参数校验、异常捕获、日志记录 |
| `Repository` 接口 | 接口 | Spring Data JPA 只定义方法签名，运行时动态代理 |

## 延伸思考

- **接口的 default 方法冲突**：一个类实现两个接口，两个接口有同名 default 方法，必须显式重写
- **抽象类 vs 接口 + default**：当接口有了 default 方法，抽象类还有存在意义吗？（答案是：有——共享状态和构造逻辑）
- **钻石继承问题**：Java 通过接口避免了 C++ 的多继承菱形问题，但 default 方法引入了新的冲突场景

## 面试复习版

### 核心概念
- 抽象类表达 is-a 和部分实现复用；接口表达能力契约。
- Java 8 后接口可有 default/static 方法，但状态和构造能力仍不同。

### 面试官想考什么
- 接口和抽象类如何选型。
- 默认方法、多继承冲突、模板复用。

### 标准回答
有共同状态和骨架流程时用抽象类；只定义能力规范、需要多实现组合时用接口。回答可从继承关系、成员能力、构造器、扩展性四方面展开。

### 深挖追问
- 接口可以有成员变量吗？
- 默认方法冲突怎么解决？
- 为什么类只能单继承？

### 实战场景/代码示例
```java
interface Payable { void pay(); }
abstract class AbstractOrder { protected abstract void validate(); }
```

### 易错点/总结
- 不要为少量代码复用滥用继承。
- default 方法不是抽象类万能替代。
