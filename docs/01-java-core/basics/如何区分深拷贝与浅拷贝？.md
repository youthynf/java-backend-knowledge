# 如何区分深拷贝与浅拷贝

## 核心概念

深浅拷贝的区别在于：**复制对象时，是否连同引用类型字段指向的子对象一起复制**。

- **浅拷贝**：复制当前对象本身，引用类型字段仍指向原来的子对象。原对象和副本共享子对象。
- **深拷贝**：递归复制当前对象及其所有引用子对象，形成完全独立的对象图。

判断标准一句话：**修改副本里的引用子对象，是否会影响原对象。** 影响，就是浅拷贝；不影响，就是深拷贝。

注意区分"赋值"和"拷贝"：

```java
Person p1 = new Person("张三", new Address("北京"));
Person p2 = p1; // 这不是拷贝，只是引用赋值，p1 和 p2 指向同一对象
p2.setName("李四");
System.out.println(p1.getName()); // 李四
```

赋值只是把引用复制了一份，两个变量指向同一个对象，谈不上拷贝。

## 标准回答

深浅拷贝是对象复制时的两种策略。浅拷贝只复制对象"外壳"，引用字段仍指向原子对象；深拷贝递归复制整个对象图，副本与原对象完全独立。判断方法看引用字段是否共享。面试要点：

1. **浅拷贝实现**：实现 `Cloneable` 接口，重写 `clone()` 调用 `super.clone()`。
2. **深拷贝实现**：手动递归克隆、序列化/反序列化、复制构造器、第三方工具（Jackson、Kryo）。
3. **`Cloneable` 是标记接口**，没有方法，只是告诉 `Object.clone()` 可以走字段复制路径。
4. **String 是不可变对象**，浅拷贝共享引用也安全，效果等同深拷贝。
5. **生产推荐用复制构造器或序列化**，`clone()` 设计上被认为是失败的特性（Joshua Bloch 的观点）。

## 实现原理

### 1. `Object.clone()` 的语义

`Object.clone()` 是 `native` 方法，做的是"浅拷贝"：为新对象逐字段复制原对象的值。基本类型字段直接复制值，引用类型字段复制引用地址。

```java
protected native Object clone() throws CloneNotSupportedException;
```

如果不实现 `Cloneable` 接口就调用 `clone()`，会抛 `CloneNotSupportedException`。`Cloneable` 本身没有方法，只是一个标记接口，告诉 `Object.clone()` "我可以被克隆"。

### 2. 浅拷贝的行为

```java
class Address {
    String city;
    Address(String city) { this.city = city; }
}

class Person implements Cloneable {
    String name;
    Address address;

    Person(String name, Address address) {
        this.name = name;
        this.address = address;
    }

    @Override
    protected Person clone() throws CloneNotSupportedException {
        return (Person) super.clone(); // 浅拷贝
    }
}

Person p1 = new Person("张三", new Address("北京"));
Person p2 = p1.clone();

p2.setName("李四");                  // 不影响 p1.name（String 不可变 + 浅拷贝创建新引用）
p2.address.setCity("上海");          // 影响 p1.address.city！两者共享 Address
System.out.println(p1.address.getCity()); // 上海
```

浅拷贝后，`p1` 和 `p2` 是两个不同的 `Person` 对象，但它们的 `address` 字段指向同一个 `Address` 对象。修改任意一方的 `address`，另一方都看得见。

### 3. 深拷贝的实现方式

#### 方式 A：重写 `clone()` 递归克隆

```java
class Address implements Cloneable {
    String city;

    @Override
    protected Address clone() throws CloneNotSupportedException {
        return (Address) super.clone();
    }
}

class Person implements Cloneable {
    String name;
    Address address;

    @Override
    protected Person clone() throws CloneNotSupportedException {
        Person cloned = (Person) super.clone();
        cloned.address = this.address.clone(); // 递归克隆引用字段
        return cloned;
    }
}
```

优点：性能好，无序列化开销。缺点：每层都要实现 `Cloneable` 并正确重写 `clone()`，维护成本高，容易漏字段。

#### 方式 B：序列化 + 反序列化

```java
import java.io.*;

class Person implements Serializable {
    String name;
    Address address;

    @SuppressWarnings("unchecked")
    public Person deepCopy() {
        try {
            ByteArrayOutputStream bos = new ByteArrayOutputStream();
            ObjectOutputStream oos = new ObjectOutputStream(bos);
            oos.writeObject(this);

            ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
            ObjectInputStream ois = new ObjectInputStream(bis);
            return (Person) ois.readObject();
        } catch (IOException | ClassNotFoundException e) {
            throw new RuntimeException("深拷贝失败", e);
        }
    }
}
```

优点：自动处理整个对象图，不需要逐字段递归。缺点：所有相关类必须实现 `Serializable`，性能比手动克隆差，且 Java 原生序列化本身有安全和性能问题。

#### 方式 C：复制构造器（推荐）

```java
class Address {
    String city;

    public Address(Address other) {
        this.city = other.city;
    }
}

class Person {
    String name;
    Address address;

    public Person(Person other) {
        this.name = other.name;                       // String 不可变，直接复用
        this.address = new Address(other.address);    // 引用字段新建
    }
}
```

优点：API 清晰、类型安全、不依赖 `Cloneable` 和序列化机制，Joshua Bloch 在 Effective Java 中明确推荐这种写法。缺点：对深层对象图仍要逐层写复制构造器。

#### 方式 D：第三方工具

```java
// Jackson
ObjectMapper mapper = new ObjectMapper();
Person copy = mapper.readValue(mapper.writeValueAsString(original), Person.class);

// Spring BeanUtils（注意：是浅拷贝！）
BeanUtils.copyProperties(source, target);

// Kryo（高性能序列化框架，支持深拷贝）
Kryo kryo = new Kryo();
Person copy = kryo.copy(original);
```

注意 `Spring BeanUtils.copyProperties` 和 `Apache BeanUtils.copyProperties` 都是**浅拷贝**，引用字段不会被复制，业务里要小心。

## 代码示例

### 浅拷贝完整示例

```java
class Address {
    String city;
    Address(String city) { this.city = city; }
    String getCity() { return city; }
    void setCity(String city) { this.city = city; }
}

class Person implements Cloneable {
    String name;
    Address address;

    Person(String name, Address address) {
        this.name = name;
        this.address = address;
    }

    @Override
    protected Person clone() throws CloneNotSupportedException {
        return (Person) super.clone();
    }

    public static void main(String[] args) throws Exception {
        Person p1 = new Person("张三", new Address("北京"));
        Person p2 = p1.clone();

        System.out.println(p1 == p2);                 // false
        System.out.println(p1.address == p2.address); // true，共享 Address
        p2.address.setCity("上海");
        System.out.println(p1.address.getCity());     // 上海，被影响
    }
}
```

### 序列化实现深拷贝（带泛型工具）

```java
import java.io.*;

public final class DeepCopyUtils {
    private DeepCopyUtils() {}

    @SuppressWarnings("unchecked")
    public static <T extends Serializable> T deepCopy(T obj) {
        try {
            ByteArrayOutputStream bos = new ByteArrayOutputStream();
            try (ObjectOutputStream oos = new ObjectOutputStream(bos)) {
                oos.writeObject(obj);
            }
            ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
            try (ObjectInputStream ois = new ObjectInputStream(bis)) {
                return (T) ois.readObject();
            }
        } catch (IOException | ClassNotFoundException e) {
            throw new RuntimeException("深拷贝失败", e);
        }
    }
}
```

### 防御性拷贝

返回内部可变字段时，应返回副本，避免外部修改破坏内部状态。

```java
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class Order {
    private final List<String> items;

    public Order(List<String> items) {
        // 构造时复制一份，避免外部修改影响内部
        this.items = new ArrayList<>(items);
    }

    public List<String> getItems() {
        // 返回不可修改视图，外部调用 add 会抛 UnsupportedOperationException
        return Collections.unmodifiableList(items);
    }
}
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 历史版本快照 | 保存对象到历史列表前深拷贝 | 否则后续修改会污染历史 |
| 多线程共享配置 | 每个线程持有配置的深拷贝副本 | 避免并发修改，但开销大；优先考虑不可变对象 |
| 防御性拷贝 | getter 返回集合副本或不可变视图 | 浅拷贝通常够用，关键是阻止外部修改集合结构 |
| 缓存回源 | 缓存返回对象前深拷贝 | 避免调用方修改污染缓存 |
| RPC 请求/响应 | 复制请求做重试 | 用序列化框架（Hessian、Kryo）实现深拷贝 |

## 深挖追问

### 1. `Cloneable` 接口为什么是空的

它是标记接口，没有方法。它唯一的作用是告诉 `Object.clone()` "这个类允许被克隆"，否则抛 `CloneNotSupportedException`。这种把接口当开关的设计被普遍认为是 Java 的设计缺陷——它没有把 `clone()` 方法放进接口，导致调用者只能依赖 `Object.clone()` 的 protected 行为。Effective Java 建议：除非是 final 类且性能关键，否则不要用 `clone()`，改用复制构造器或静态工厂。

### 2. 浅拷贝会影响 `String` 字段吗

不会。`String` 是不可变对象，浅拷贝复制的是引用，但引用指向的字符串对象无法被修改。所以浅拷贝共享 `String` 引用是安全的，效果上等同深拷贝。

### 3. `BeanUtils.copyProperties` 是深拷贝还是浅拷贝

浅拷贝。Spring 和 Apache 的 `BeanUtils.copyProperties` 都是按属性名复制值，引用类型字段直接复制引用，不会递归克隆子对象。如果业务需要深拷贝，要用 Jackson、Kryo 或手写复制构造器。

### 4. 为什么 Java 不默认提供深拷贝

深拷贝需要类设计者明确知道哪些字段要复制、哪些不需要（如 `Connection`、`Thread`）。JVM 无法自动判断，所以把这个责任交给开发者。`Object.clone()` 只提供浅拷贝，深拷贝由开发者自行实现。

### 5. 不可变对象需要深拷贝吗

不需要。不可变对象的所有字段在构造后就不会改变，引用共享完全安全。这也是为什么 `String`、`Integer`、`LocalDate` 等不可变类不需要专门写拷贝逻辑。设计不可变类是避免深拷贝开销的最佳方式。

## 易错点

- 把"引用赋值"当成"拷贝"，`Person p2 = p1` 不是拷贝。
- 用 `BeanUtils.copyProperties` 期望深拷贝，实际只是浅拷贝。
- 重写 `clone()` 时忘记 `implements Cloneable`，运行时抛 `CloneNotSupportedException`。
- 递归克隆漏掉某个引用字段，导致深拷贝变"半深拷贝"，问题难复现。
- 序列化深拷贝时漏实现 `Serializable`，抛 `NotSerializableException`。
- 在 `clone()` 里调用了可被重写的方法（构造器/clone 中的"构造器不应调用可重写方法"原则同样适用）。
- 以为 `ArrayList` 的 `clone()` 是深拷贝，实际只是浅拷贝，元素仍共享。

## 总结

深浅拷贝的核心区别是引用字段是否共享。浅拷贝简单快速，适合不可变字段或单层对象；深拷贝安全独立，适合复杂可变对象图。生产实践中优先用复制构造器（清晰、类型安全），其次用序列化（自动处理对象图），避免 `Object.clone()`（设计缺陷多）。最佳策略是把对象设计成不可变，从根上避免拷贝问题。
