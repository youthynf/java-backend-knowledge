# 基本数据类型的缓存池是什么

## 核心概念

Java 的部分包装类型内部实现了**缓存池**，会预先创建一段常用取值范围内的对象。通过自动装箱（`Integer.valueOf(int)` 等）创建包装对象时，如果值在缓存范围内，会直接复用缓存对象，不再 `new` 新对象。缓存池的目的是减少频繁创建小整数等常用对象带来的内存和性能开销。

```java
Integer a = 127;   // 等价于 Integer.valueOf(127)，命中缓存
Integer b = 127;   // 命中同一个缓存对象
System.out.println(a == b);   // true，同一个引用

Integer c = 128;   // 超出缓存范围，新建对象
Integer d = 128;   // 又新建一个对象
System.out.println(c == d);   // false，不同引用
```

这是面试的经典题，也是生产中"`Integer == Integer` 有时对有时错"的根源。

## 标准回答

基本数据类型的缓存池是包装类内部预创建的对象池，自动装箱时命中范围复用对象。要点：

1. **缓存范围**：`Byte`/`Short`/`Integer`/`Long` 缓存 `-128~127`；`Character` 缓存 `0~127`；`Boolean` 缓存 `TRUE`/`FALSE`；`Float`/`Double` 不缓存。
2. **触发时机**：仅 `valueOf()` 触发，`new Integer(...)` 永远新建对象。
3. **`Integer` 上限可配置**：通过 `-XX:AutoBoxCacheMax=<size>` 调整。
4. **缓存导致 `==` 不一致**：范围内 `true`、范围外 `false`，比较包装类型必须用 `equals()`。
5. **缓存实现**：静态数组 + 静态代码块，类加载时初始化。

## 实现原理

### 1. 各包装类的缓存范围

| 包装类 | 缓存范围 | 可配置 | 实现位置 |
|--------|---------|--------|---------|
| `Byte` | -128 ~ 127（全部） | 否 | `Byte.ByteCache` |
| `Short` | -128 ~ 127 | 否 | `Short.ShortCache` |
| `Integer` | -128 ~ 127 | 是 | `Integer.IntegerCache` |
| `Long` | -128 ~ 127 | 否 | `Long.LongCache` |
| `Character` | 0 ~ 127 | 否 | `Character.CharacterCache` |
| `Boolean` | TRUE / FALSE | 否 | `Boolean.TRUE` / `Boolean.FALSE` |
| `Float` | 不缓存 | - | - |
| `Double` | 不缓存 | - | - |

### 2. `Integer` 缓存的源码

```java
public final class Integer extends Number implements Comparable<Integer> {
    public static Integer valueOf(int i) {
        if (i >= IntegerCache.low && i <= IntegerCache.high)
            return IntegerCache.cache[i + (-IntegerCache.low)];
        return new Integer(i);
    }

    private static class IntegerCache {
        static final int low = -128;
        static int high;          // 上限，默认 127，可配置
        static final Integer[] cache;

        static {
            // high 默认 127，可通过 JVM 参数 -XX:AutoBoxCacheMax 调整
            int h = 127;
            String prop = System.getProperty("java.lang.Integer.IntegerCache.high");
            if (prop != null) {
                try {
                    h = Math.max(parseInt(prop), 127);
                } catch (NumberFormatException e) { /* 用默认 */ }
            }
            high = h;

            cache = new Integer[(high - low) + 1];
            int j = low;
            for (int k = 0; k < cache.length; k++)
                cache[k] = new Integer(j++);
        }
    }
}
```

`IntegerCache` 是 `Integer` 的私有静态内部类，类加载时通过静态代码块预创建缓存数组。`valueOf` 检查值是否在范围内，命中则直接返回缓存对象。

### 3. 为什么 `Byte`/`Short`/`Long` 不支持配置

`Byte` 范围就是 `-128~127`，全缓存。`Short` 和 `Long` 选择 `-128~127` 作为常用范围，没提供 JVM 参数。可能是 JVM 设计者认为：超出 `Byte` 范围的整数使用频率较低，缓存收益不明显，没必要配置。

### 4. `Boolean` 的"缓存"

```java
public final class Boolean {
    public static final Boolean TRUE = new Boolean(true);
    public static final Boolean FALSE = new Boolean(false);

    public static Boolean valueOf(boolean b) {
        return b ? TRUE : FALSE;
    }
}
```

`Boolean` 只有两个值，直接用静态常量 `TRUE`/`FALSE`。"缓存"其实就是两个静态字段。

### 5. `Float`/`Double` 为什么不缓存

浮点数有无限多个可能值（理论上 `2^32` 个不同的 `float`），不可能预创建缓存。即便缓存小范围浮点数，命中概率也极低（业务场景不会反复用 `0.1f`），收益不抵开销。

### 6. `valueOf` vs `new`

```java
Integer a = Integer.valueOf(127);    // 命中缓存，复用对象
Integer b = Integer.valueOf(127);    // 复用同一对象
System.out.println(a == b);          // true

Integer c = new Integer(127);        // 永远新建对象
Integer d = new Integer(127);        // 又新建对象
System.out.println(c == d);          // false
```

只有 `valueOf()` 走缓存，`new` 永远新建。`new Integer(...)` 已被 Java 9 标记 `@Deprecated`，应该用 `valueOf`。自动装箱 `Integer a = 127` 用的就是 `valueOf`，所以会命中缓存。

### 7. 缓存命中场景

```java
// 自动装箱：编译器插入 valueOf，命中缓存
Integer a = 127;

// 显式 valueOf：命中缓存
Integer b = Integer.valueOf(127);

// 集合 add：自动装箱，命中缓存
List<Integer> list = new ArrayList<>();
list.add(127);

// 三元表达式：自动装箱
Integer c = condition ? 127 : 200;

// 字面量赋值
Integer d = 100;        // 命中缓存
```

不命中场景：

```java
Integer a = new Integer(127);     // 永远不命中
Integer b = Integer.getInteger("key"); // 从系统属性读取，不走缓存
```

### 8. 包装类型 `==` 的真假分布

```java
Integer a = 127, b = 127;
System.out.println(a == b);   // true，缓存命中

Integer c = 128, d = 128;
System.out.println(c == d);   // false，超出缓存范围

Integer e = new Integer(127);
Integer f = new Integer(127);
System.out.println(e == f);   // false，new 永远新建

Integer g = 127;
Integer h = new Integer(127);
System.out.println(g == h);   // false，缓存对象 ≠ new 对象

Integer i = 127;
int j = 127;
System.out.println(i == j);   // true，Integer 自动拆箱为 int 比较值
```

最后一种情况要特别注意：`Integer == int` 会自动拆箱，比较的是值。但如果 `Integer` 为 `null`，拆箱抛 `NullPointerException`。

## 代码示例

### 缓存验证

```java
public class CacheDemo {
    public static void main(String[] args) {
        // Integer 缓存
        System.out.println(Integer.valueOf(127) == Integer.valueOf(127));   // true
        System.out.println(Integer.valueOf(128) == Integer.valueOf(128));   // false

        // Long 缓存
        System.out.println(Long.valueOf(127L) == Long.valueOf(127L));       // true
        System.out.println(Long.valueOf(128L) == Long.valueOf(128L));       // false

        // Character 缓存
        System.out.println(Character.valueOf((char) 127) == Character.valueOf((char) 127)); // true
        System.out.println(Character.valueOf((char) 128) == Character.valueOf((char) 128)); // false

        // Boolean 缓存
        System.out.println(Boolean.valueOf(true) == Boolean.valueOf(true));  // true

        // Byte/Short 缓存（范围 -128~127，全缓存或近似）
        System.out.println(Byte.valueOf((byte) 100) == Byte.valueOf((byte) 100));  // true
        System.out.println(Short.valueOf((short) 100) == Short.valueOf((short) 100)); // true

        // Float/Double 不缓存
        System.out.println(Float.valueOf(1.0f) == Float.valueOf(1.0f));   // false
        System.out.println(Double.valueOf(1.0) == Double.valueOf(1.0));   // false
    }
}
```

### 调整 `Integer` 缓存上限

```bash
# 启动时指定 JVM 参数
java -XX:AutoBoxCacheMax=1000 -cp . CacheDemo
```

调整后 `Integer.valueOf(500) == Integer.valueOf(500)` 会返回 `true`（在新的缓存范围内）。但生产中很少调整，默认值够用。

### 业务代码中的缓存陷阱

```java
public class CacheTrap {
    public static void main(String[] args) {
        // 错误：用 == 比较库存数量
        Map<String, Integer> stock = new HashMap<>();
        stock.put("A", 100);
        stock.put("B", 200);

        Integer a = stock.get("A");
        Integer b = stock.get("A");
        System.out.println(a == b);   // 可能为 true（命中缓存），但行为依赖缓存范围

        // 正确：用 equals 比较
        System.out.println(a.equals(b));   // 永远 true

        // 错误：循环计数用 Integer
        Integer count = 0;
        for (int i = 0; i < 200; i++) {
            count++;   // 自动拆箱+加法+自动装箱，超出 127 后每次新建对象
        }

        // 正确：用 int
        int count2 = 0;
        for (int i = 0; i < 200; i++) {
            count2++;
        }
    }
}
```

## 实战场景

| 场景 | 行为 | 注意点 |
|------|------|--------|
| 缓存命中判断 | `Integer.valueOf(127)` 复用 | 范围外不命中 |
| 包装类型比较 | 缓存范围内 `==` true | 比较一律用 `equals` |
| 集合存储基本类型 | 自动装箱，可能命中缓存 | 性能敏感考虑 `int[]` |
| 循环计数 | 用 `Integer` 反复拆装 | 改用 `int` |
| 实体 ID 字段 | `Long` 包装类型 | 比较用 `equals`，不要 `==` |
| 状态码字段 | `Integer` 表达状态 | 缓存范围内 `==` 可能"碰巧"工作，迁移后失效 |

## 深挖追问

### 1. 为什么默认缓存 `-128~127`

`Byte` 范围正好是 `-128~127`，所以 `Byte` 全部缓存。`Integer`/`Long`/`Short` 选择同样范围作为"常用值"覆盖，是经验值——大部分小整数运算都在这个范围内（数组下标、循环计数、状态码）。`Character` 是 `0~127`，覆盖 ASCII 字符。

### 2. 为什么 `Integer` 缓存上限可配置

某些应用场景（如大量小整数运算）扩大缓存能减少对象创建。`-XX:AutoBoxCacheMax=<size>` 可以调整上限（下限固定 -128）。但缓存大小直接影响内存占用，需要权衡。生产中很少调整。

### 3. `Integer a = 127` 和 `Integer b = 127` 为什么 `a == b`

自动装箱 `Integer a = 127` 编译为 `Integer.valueOf(127)`，`127` 在缓存范围内，返回缓存数组中的同一对象。`b` 同理。所以 `a` 和 `b` 指向同一个缓存对象，`==` 返回 `true`。

### 4. 缓存对象不会被 GC 回收吗

不会。缓存对象存储在静态字段中（如 `IntegerCache.cache`），静态字段引用的对象会一直存在到 ClassLoader 卸载。在主应用中，bootstrap ClassLoader 永远不卸载，缓存对象等于 JVM 生命周期。

### 5. 自动拆箱会不会走缓存

不会。拆箱是 `Integer.intValue()` 返回 `int`，没有对象创建过程。装箱才涉及缓存。

### 6. 缓存是线程安全的吗

是。缓存数组在类加载时通过静态代码块初始化，初始化完成后只读不写，所有线程共享同一份缓存。`valueOf` 只是查数组返回引用，无并发问题。

### 7. 为什么 `Float`/`Double` 不缓存

浮点数取值空间太大（`2^32` 个不同的 `float`），且业务中重复使用同一浮点值的概率极低，缓存收益不抵开销。`Boolean` 只有两个值，全缓存。

## 易错点

- 用 `==` 比较包装类型，缓存范围内通过、范围外失败。
- 用 `new Integer(127)` 创建对象，永远不命中缓存。
- 在循环里用 `Integer` 累加，超出 127 后每次新建对象。
- 把缓存行为当业务逻辑依赖，迁移环境后行为变化。
- 集合中存 `Integer`，遍历比较时误用 `==`。
- 误以为 `Float`/`Double` 有缓存，实际没有。
- 在多线程下用包装类型做计数器，没有同步导致计数错误（包装类型不可变，重新赋值不是原子的）。
- 实体字段是 `Long`，主键比较大时 `==` 返回 false。

## 总结

基本数据类型的缓存池是包装类内部的预创建对象池，目的是减少常用值对象的重复创建。`Byte`/`Short`/`Integer`/`Long` 缓存 `-128~127`，`Character` 缓存 `0~127`，`Boolean` 全缓存，`Float`/`Double` 不缓存。`Integer` 缓存上限可通过 `-XX:AutoBoxCacheMax` 调整。生产中要记住铁律：包装类型比较永远用 `equals()`，不要依赖 `==`——缓存范围内的"碰巧 true"是最隐蔽的 bug 来源。循环计数用 `int`，避免装箱开销。理解缓存机制能解释大量"包装类型 `==` 行为异常"的疑惑。
