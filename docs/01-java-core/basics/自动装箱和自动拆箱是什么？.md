# 自动装箱和自动拆箱是什么

## 核心概念

自动装箱（Autoboxing）和自动拆箱（Unboxing）是 Java 5 引入的语法糖，用于在基本类型和对应包装类型之间自动转换，让开发者无需手写 `Integer.valueOf(1)` 或 `intValue()`。

- **自动装箱**：基本类型 → 包装类对象。编译器插入 `valueOf()` 调用。
- **自动拆箱**：包装类对象 → 基本类型。编译器插入 `xxxValue()` 调用。

```java
Integer a = 10;     // 编译为 Integer a = Integer.valueOf(10);
int b = a;          // 编译为 int b = a.intValue();
```

这层语法糖让代码简洁，但底层并不是"零成本"——它隐藏了对象创建和方法调用，性能问题和 NPE 风险都源于此。

## 标准回答

自动装箱/拆箱是 Java 5 的语法糖，编译期把基本类型与包装类型的转换改写为 `valueOf()` 和 `xxxValue()` 方法调用。要点：

1. **自动装箱**：`Integer a = 10` 编译为 `Integer.valueOf(10)`，命中缓存时复用对象。
2. **自动拆箱**：`int b = a` 编译为 `a.intValue()`，`a` 为 `null` 时抛 `NullPointerException`。
3. **包装类型缓存**：`Integer`/`Long`/`Short`/`Byte` 缓存 `-128~127`，`Character` 缓存 `0~127`，`Boolean` 缓存 `TRUE`/`FALSE`。
4. **比较风险**：包装类型 `==` 比较的是引用，缓存范围外会返回 `false`，必须用 `equals()`。
5. **性能陷阱**：循环里大量装箱拆箱会创建大量临时对象，影响性能。

## 实现原理

### 1. 编译期改写

```java
// 源码
Integer a = 100;
int b = a;

// 编译后等价于
Integer a = Integer.valueOf(100);
int b = a.intValue();
```

各包装类的装箱/拆箱方法：

| 包装类 | 装箱方法 | 拆箱方法 |
|--------|---------|---------|
| `Integer` | `Integer.valueOf(int)` | `intValue()` |
| `Long` | `Long.valueOf(long)` | `longValue()` |
| `Double` | `Double.valueOf(double)` | `doubleValue()` |
| `Boolean` | `Boolean.valueOf(boolean)` | `booleanValue()` |
| `Character` | `Character.valueOf(char)` | `charValue()` |

### 2. 包装类型的缓存机制

`Integer.valueOf` 不是每次都 `new`，而是先查缓存：

```java
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

`IntegerCache` 默认缓存 `-128 ~ 127`，上限可通过 JVM 参数 `-XX:AutoBoxCacheMax=<size>` 调整。

各包装类的缓存范围：

| 包装类 | 缓存范围 | 可配置 |
|--------|---------|--------|
| `Byte` | -128 ~ 127（全部） | 否 |
| `Short` | -128 ~ 127 | 否 |
| `Integer` | -128 ~ 127 | 是（`-XX:AutoBoxCacheMax`） |
| `Long` | -128 ~ 127 | 否 |
| `Character` | 0 ~ 127 | 否 |
| `Boolean` | TRUE / FALSE | 否 |
| `Float` / `Double` | 不缓存 | - |

### 3. 自动装箱的场景

#### 赋值转换

```java
Integer a = 1;          // 自动装箱
int b = a;              // 自动拆箱
Double d = 3.14;        // 自动装箱
```

#### 运算场景

```java
Integer a = 1;
Integer b = 2;
int c = a + b;          // a、b 先自动拆箱为 int，相加后再赋值
```

`a + b` 实际是 `a.intValue() + b.intValue()`。

#### 集合场景

Java 泛型不支持基本类型，集合存基本类型必须先装箱：

```java
List<Integer> list = new ArrayList<>();
list.add(1);            // 自动装箱：1 -> Integer.valueOf(1)
int value = list.get(0); // 自动拆箱：Integer.intValue()
```

集合场景是装箱拆箱最频繁的地方，性能敏感场景要警惕。

#### 方法参数

```java
void process(Integer x) { ... }
process(1);             // 自动装箱
```

### 4. 自动拆箱的 NPE 陷阱

包装类型可以是 `null`，自动拆箱 `null` 会抛 `NullPointerException`：

```java
Integer a = null;
int b = a;              // NullPointerException

// 比较运算也涉及拆箱
Integer x = null;
if (x == 0) { }         // NullPointerException，x 自动拆箱时 NPE

// 三元运算符
Integer a = null;
Integer b = 1;
int c = (a == null) ? 0 : a;  // OK，因为 a == null 为 true，不会执行 a
int d = (a != null) ? a : 0;  // NPE，a != null 为 false，但编译器对三元表达式做自动拆箱
```

实际项目中最常见的 NPE 来源：从 `Map` 取包装类型字段时未判空就赋给基本类型。

```java
Map<String, Integer> counts = ...;
int count = counts.get("key");  // 如果 key 不存在，返回 null，自动拆箱抛 NPE
```

正确写法：

```java
int count = counts.getOrDefault("key", 0);
```

### 5. 性能陷阱

装箱会创建对象，频繁装箱会带来 GC 压力：

```java
Long sum = 0L;
for (int i = 0; i < Integer.MAX_VALUE; i++) {
    sum += i;  // 每次都创建 Long 对象（i 装箱为 Long，然后 sum + Long 创建新 Long）
}
```

`sum` 是 `Long`（不是 `long`），`sum += i` 编译为 `sum = Long.valueOf(sum.longValue() + i)`，每次循环都创建新 `Long` 对象。`Integer.MAX_VALUE` 次循环约创建 21 亿个临时对象，性能远低于直接用 `long`。

修正：

```java
long sum = 0L;
for (int i = 0; i < Integer.MAX_VALUE; i++) {
    sum += i;
}
```

Effective Java Item 61："基本类型优于包装类型"。能用基本类型就用基本类型，包装类型只在需要对象语义（泛型集合、null 表达"未设置"、反射）时用。

### 6. `==` 比较的陷阱

```java
Integer a = 127;
Integer b = 127;
System.out.println(a == b);   // true，缓存范围内同一对象

Integer c = 128;
Integer d = 128;
System.out.println(c == d);   // false，超出缓存范围是不同对象
System.out.println(c.equals(d)); // true，比较内容
```

`Integer` 缓存导致 `==` 在 `-128~127` 范围内"碰巧"返回 `true`，超出范围返回 `false`。这种"有时对有时错"的 bug 非常隐蔽。

正确做法：包装类型比较永远用 `equals()`。

### 7. `new Integer()` 与 `Integer.valueOf()` 的区别

```java
Integer a = new Integer(127);  // 总是新建对象
Integer b = new Integer(127);
System.out.println(a == b);    // false

Integer c = Integer.valueOf(127);
Integer d = Integer.valueOf(127);
System.out.println(c == d);    // true，缓存复用
```

`new Integer()` 已被 Java 9 标记 `@Deprecated`，应该用 `Integer.valueOf()`。自动装箱用的就是 `valueOf()`。

## 代码示例

### 装箱拆箱的字节码

```java
public class BoxingDemo {
    public static void main(String[] args) {
        Integer a = 10;
        int b = a;
        Integer c = a + b;
    }
}
```

`javap -c BoxingDemo` 查看字节码：

```text
0: bipush        10
2: invokestatic  #2  // Method Integer.valueOf:(I)Integer;
5: astore_1
6: aload_1
7: invokevirtual #3  // Method Integer.intValue:()I
10: istore_2
11: aload_1
12: invokevirtual #3
15: iload_2
16: iadd
17: invokestatic  #2  // Method Integer.valueOf
20: astore_3
```

可以清楚看到 `Integer.valueOf` 和 `Integer.intValue` 的调用。

### 缓存陷阱

```java
public class CacheTrap {
    public static void main(String[] args) {
        Integer a = 127, b = 127;
        System.out.println(a == b);   // true

        Integer c = 128, d = 128;
        System.out.println(c == d);   // false

        Integer e = Integer.valueOf(127);
        Integer f = Integer.valueOf(127);
        System.out.println(e == f);   // true，valueOf 走缓存

        Integer g = new Integer(127);
        Integer h = new Integer(127);
        System.out.println(g == h);   // false，new 总是新建
    }
}
```

### NPE 示例

```java
public class NpeTrap {
    public static void main(String[] args) {
        Map<String, Integer> counts = new HashMap<>();
        counts.put("a", 1);

        // 错误：key "b" 不存在，返回 null，自动拆箱 NPE
        // int count = counts.get("b");

        // 正确：使用 getOrDefault
        int count = counts.getOrDefault("b", 0);

        // 正确：显式判空
        Integer value = counts.get("b");
        if (value != null) {
            int v = value;
        }
    }
}
```

### 性能对比

```java
public class PerformanceTrap {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();

        // 错误：Long 装箱，每次创建对象
        Long sum1 = 0L;
        for (int i = 0; i < 100_000_000; i++) {
            sum1 += i;
        }
        System.out.println("Long: " + (System.currentTimeMillis() - start) + "ms");

        // 正确：用 long 基本类型
        start = System.currentTimeMillis();
        long sum2 = 0L;
        for (int i = 0; i < 100_000_000; i++) {
            sum2 += i;
        }
        System.out.println("long: " + (System.currentTimeMillis() - start) + "ms");
    }
}
```

`Long` 版本通常比 `long` 版本慢 5~10 倍，差距随循环次数放大。

## 实战场景

| 场景 | 行为 | 注意点 |
|------|------|--------|
| 泛型集合 | `List<Integer>` 存 int 会装箱 | 性能敏感场景考虑 `int[]` 或 `TIntArrayList` |
| `Map.get()` 返回包装类型 | 直接赋给基本类型会拆箱 NPE | 用 `getOrDefault` 或显式判空 |
| 数据库字段 | 实体字段用包装类型表达 null | "未设置"和"0"语义不同 |
| 三元运算符 | 自动拆箱可能 NPE | 类型推断可能拆箱，谨慎用 |
| `==` 比较 | 缓存范围内"碰巧"true | 包装类型一律用 `equals()` |
| 反序列化 | JSON 字段是 Integer，可能为 null | 反序列化为基本类型字段可能 NPE |
| 缓存命中 | `valueOf` 命中缓存省内存 | 不要 `new Integer`，已废弃 |

## 深挖追问

### 1. `Integer` 缓存为什么默认是 `-128~127`

`Byte` 范围正好是 `-128~127`，所以 `Byte` 全部缓存。`Integer` 选择同样范围作为"常用值"覆盖，是经验值——大部分小整数计算都在这个范围内。`Character` 是 `0~127`，覆盖 ASCII 字符。`Boolean` 只有两个值，全缓存。

### 2. 为什么 `Integer` 缓存上限可配置

JVM 参数 `-XX:AutoBoxCacheMax=<size>` 可以调整上限（下限固定 -128）。某些应用场景（如大量小整数运算）扩大缓存能减少对象创建。但缓存大小直接影响内存占用，需要权衡。

### 3. `Integer a = 128; Integer b = 128;` 为什么 `a != b`

`128` 超出默认缓存范围 `[-128, 127]`，`Integer.valueOf(128)` 每次都 `new` 一个新对象。`==` 比较引用地址，两个不同对象自然不等。改用 `equals()` 比较内容。

### 4. 自动装箱在条件表达式中如何工作

```java
Integer a = null;
if (a == 0) { }   // 等价于 a.intValue() == 0，a 是 null 时 NPE
```

`a == 0` 编译时，`a` 是 `Integer`、`0` 是 `int`，编译器把 `a` 自动拆箱为 `int` 再比较。`a` 为 `null` 时拆箱抛 NPE。

### 5. `Integer` 和 `int` 比较

```java
Integer a = 100;
int b = 100;
System.out.println(a == b);  // true
```

`a == b` 时，`a` 自动拆箱为 `int`，比较的是两个 `int` 值。**如果 `a` 是 `null` 会 NPE**。所以"`Integer` 和 `int` 用 `==` 安全"是误区，仍需判空。

### 6. 装箱拆箱是线程安全的吗

单个包装对象是不可变的，线程安全。但**装箱过程不是原子的**：`Integer a = i` 涉及"读 `i` + 调 `valueOf` + 赋值给 `a`"，多线程下 `i` 可能被其他线程修改。`Integer` 对象本身不可变，引用赋值是原子的（基本类型引用），所以最终 `a` 是某个有效值，不会损坏。

### 7. 为什么 Java 不直接把基本类型当对象

历史和性能原因。Java 1.0 没有泛型，所有对象都有对象头开销（HotSpot 64 位约 12 字节）。`int` 是 4 字节直接存值，`Integer` 是 16 字节对象。如果一切皆对象，性能会大幅下降。Java 5 用装箱拆箱做"妥协"——保留基本类型性能，需要对象语义时自动转换。Valhalla 项目（值类型）未来会进一步模糊界限。

## 易错点

- 包装类型用 `==` 比较，缓存范围内通过、范围外失败。
- `Map.get()` 返回 null 时直接赋给基本类型，NPE。
- 循环里用包装类型累加，每次都创建对象，性能差。
- `Integer a = null; if (a == 0)` 这种条件判断，`a` 为 null 时 NPE。
- 误以为 `Integer a = new Integer(127); Integer b = new Integer(127); a == b` 为 true，实际 false。
- 在三元表达式 `condition ? wrapper1 : wrapper2` 中，一边是 `Integer`、一边是 `int`，整型提升导致拆箱 NPE。
- 在多线程下用包装类型做计数器，没有同步导致计数错误（包装类型不可变，重新赋值不是原子操作）。
- 实体字段用基本类型，反序列化 null 字段时 NPE（应改用包装类型）。

## 总结

自动装箱/拆箱是 Java 5 的语法糖，编译期改写为 `valueOf()` 和 `xxxValue()` 调用，让基本类型与包装类型互转更简洁。它的核心成本是对象创建和方法调用，频繁装箱会产生 GC 压力。生产中要警惕三个陷阱：缓存导致的 `==` 不一致（用 `equals()`）、自动拆箱的 NPE（判空或用 `getOrDefault`）、循环中装箱的性能损耗（用基本类型）。原则：能用基本类型就用基本类型，包装类型只在需要对象语义（集合、null 表达"未设置"、反射）时使用。
