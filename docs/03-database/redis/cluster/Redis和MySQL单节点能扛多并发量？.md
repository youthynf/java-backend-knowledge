# Redis 和 MySQL 单节点能扛多并发量？

## 核心概念

单节点能扛多少并发没有固定答案，取决于硬件配置、数据模型、命令复杂度、SQL 复杂度、连接数、网络、持久化策略、读写比例和响应时间目标。面试中不应只报一个数字，而要说明评估方法。

经验上，Redis 单节点处理简单 `GET/SET` 可达到数万到十几万 QPS；MySQL 单节点简单主键查询可能达到数千到数万 QPS，复杂 Join、事务写入、磁盘 IO 会显著降低吞吐。最终必须以压测和线上监控为准。

## 面试官想考什么

- 是否知道容量评估不能脱离场景；
- 是否理解 Redis 和 MySQL 瓶颈不同；
- 是否会用 QPS、RT、CPU、IO、连接数、慢查询评估；
- 是否能提出扩容和优化方案。

## 标准回答

> Redis 和 MySQL 单节点并发能力取决于业务场景。Redis 简单命令主要受 CPU、网络和大 Key 影响，单节点常见可支撑数万到十几万 QPS；MySQL 受 SQL、索引、事务、锁、磁盘 IO 影响更大，简单主键读可能很高，复杂查询或写事务会下降很多。容量评估应通过压测得到 P95/P99、错误率和资源水位，再决定缓存、读写分离、分库分表或集群扩容。

## 深挖追问

### 为什么 Redis QPS 通常高于 MySQL？

Redis 数据在内存中，命令模型简单，单线程避免大量锁竞争；MySQL 要处理 SQL 解析优化、事务、锁、日志刷盘、磁盘页读写和复杂执行计划，因此成本更高。

### 只看 QPS 够吗？

不够。还要看响应时间、尾延迟、错误率、CPU、内存、磁盘 IO、网络带宽、连接数、主从延迟和业务峰值持续时间。高 QPS 但 P99 很差，对用户仍然不可接受。

## 实战场景 / 压测示例

Redis 可用 `redis-benchmark` 做基础压测：

```bash
redis-benchmark -h 127.0.0.1 -p 6379 -t get,set -n 100000 -c 200 -d 100
```

MySQL 压测要使用接近真实业务的 SQL、数据量和索引，例如 sysbench 或自研压测脚本，并观察慢查询、锁等待和 buffer pool 命中率。

## 易错点 / 总结

- 不要脱离机器配置和业务 SQL 报固定并发数；
- Redis 大 Key、热 Key、慢命令会显著拉低吞吐；
- MySQL 写入能力受事务、索引数量和刷盘策略影响；
- 连接数不等于并发能力，过多连接会增加调度开销；
- 容量结论必须来自压测和监控。

## 面试版详细讲解

### 核心概念

这道题属于 **Redis 集群** 的高频考点，核心要抓住：主从复制、哨兵、Cluster、槽、故障转移、一致性。Redis 主从和集群提升读能力、容量和可用性，但默认异步复制不能保证强一致，故障窗口可能丢少量数据。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：复制延迟、故障切换、脑裂、槽迁移和容量规划；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

Redis 主从和集群提升读能力、容量和可用性，但默认异步复制不能保证强一致，故障窗口可能丢少量数据。

答题时建议用“三段式”：

1. 先给结论，明确适用前提；
2. 再解释底层机制或执行过程；
3. 最后补充业务取舍、风险点和排查手段。

### 深挖追问

- 这个结论在高并发或大数据量下是否仍然成立？
- 它依赖哪些版本、配置、索引/编码或业务一致性要求？
- 线上异常时应该看哪些命令、日志、指标或执行计划？

### 示例 / 实战场景

核心缓存可接受最终一致；强一致金融状态不要只依赖 Redis。Cluster 扩容要关注热槽、大 key 迁移和客户端重定向。

```bash
# 先小范围验证命令复杂度和返回量，避免线上直接扫大 key
redis-cli --scan --pattern 'biz:*' | head
redis-cli --bigkeys
```

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
