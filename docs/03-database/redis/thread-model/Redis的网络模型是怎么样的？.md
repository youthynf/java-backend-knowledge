# Redis的网络模型是怎么样的？

## 核心概念

Redis的网络模型是怎么样的？
在Redis 6.0版本之前，Redis采用的是「单Reactor单线程」模式。这种模式下，所有的网络I/O操作（如接受连接、读取和写入数据）以及命令的执行都是在一个单线程中完成的。Redis通过I/O多路复用技术（如epoll）来同时处理多个客户端连接，从而实现了高效的事件驱动模型。

单Reactor单线程模式的特点
高效性：由于所有操作都在一个线程中进行，避免了多线程之间的上下文切换和同步锁的开销。
简单性：实现相对简单，逻辑清晰，易于维护。
性能瓶颈：在高并发场景下，单线程可能成为性能瓶颈，因为所有操作都必须等待当前操作完成。

Redis 6.0版本的变化
从Redis 6.0版本开始，Redis引入了多线程模型来处理网络I/O操作，以解决网络I/O的性能瓶颈。尽管如此，命令的执行仍然由【主线程】负责。这种改进使得Redis能够更好地利用多核CPU资源，提高网络I/O的处理能力。

## 面试官想考什么

- Redis 单线程主要指命令执行主线程，不是所有模块都单线程。
- IO 多路复用、事件循环、内存操作为什么快。
- 慢命令、大 Key、阻塞操作如何影响整体延迟。

## 标准回答

Redis 快主要来自内存操作、高效数据结构、单线程命令执行避免锁竞争，以及基于 IO 多路复用的事件循环。新版本可用多线程处理部分网络 IO，但慢命令、大 Key、阻塞操作仍会拖慢主线程。

## 深挖追问

1. 单线程为什么还快？内存操作、IO 多路复用、少锁竞争。
2. 什么会阻塞 Redis？慢命令、大 Key、持久化压力、Lua 死循环。
3. 多线程后还要注意什么？命令执行主路径仍怕阻塞。

## 实战场景 / SQL 示例

```text
SLOWLOG GET 10
LATENCY DOCTOR
-- 找出慢命令、大 Key 和延迟尖刺。
```

## 易错点 / 总结

- 不要把“单线程”理解为 Redis 没有后台线程。
- 避免 KEYS、全量 LRANGE/SMEMBERS 等阻塞命令。
- 慢日志和延迟监控要常态化。
