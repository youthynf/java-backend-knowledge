# Redis是单线程吗？

Redis是单线程吗？
Redis的单线程理解
Redis单线程是指【接受客户端请求→解析请求→进行数据读写等操作→发送数据给客户端】这个过程是由一个线程（主线程）来完成的，这也是我们常说Redis是单线程的原因。

Redis的多线程操作
但是Redis程序并不是单线程的，Redis在启动的时候，是会启动后台线程（BIO）的：
Redis在2.6版本，会启动2个后台线程，分别处理关闭文件、AOF刷盘这两个任务；
Redis在4.0版本之后，新增了一个新的后台线程，用来异步释放Redis内存，也就是lazyfree线程。例如执行unlink key/flushdb async/flushall async等命令，会把这些删除操作交给后台线程执行，不导致主线程卡顿。
Redis在6.0版本之后，为了提高网络I/O的并行度，对于网络I/O采用多线程来处理，但是对于命令的执行，Redis仍然使用单线程来处理，所以不要误解Redis有多线程同时执行命令。默认情况下I/O多线程只针对发送响应数据，并不会已多线程方式处理读请求，但支持配置开启io-threads-do-reads配置项设置yes。
之所以 Redis 为「关闭文件、AOF 刷盘、释放内存」这些任务创建单独的线程来处理，是因为这些任务的操作都是很耗时的，如果把这些任务都放在主线程来处理，那么 Redis 主线程就很容易发生阻塞，这样就无法处理后续的请求了。后台线程相当于一个消费者，生产者把耗时任务丢到任务队列中，消费者（BIO）不停轮询这个队列，拿出任务就去执行对应的方法即可。

因此，Redis6.0版本之后，Redis在启动时默认除了主线程外，额外创建6个线程：
Redis-server：Redis的主线程，主要负责执行命令；
bio_close_file、bio_aof_fsync、bio_lazy_free：三个后台线程，分别处理关闭文件任务、AOF刷盘任务、释放内存任务；
io_thd_1、io_thd_2、io_thd_3：三个I/O线程，io-threads默认是4，所以会启动3（4-1）个I/O多线程，用来分担Redis网络I/O的压力。
后台线程相当与一个消费者，生产者把耗时的任务丢到对应的任务队列里面，消费者BIO不断轮询这个队列，拿出任务就去执行对应的方法即可。

Redis中的多进程操作
在 RDB 持久化时，Redis 会 fork 一个【子进程】来执行将内存数据保存为 RDB 文件的操作，这样主进程可以继续处理客户端的请求，而不会阻塞；
在 AOF 重写时，也会 fork 一个【子进程】来进行 AOF 文件的重写操作，主进程继续处理客户端请求，利用了操作系统的写时复制（Copy-On-Write）技术，避免了大量的数据复制，降低了性能开销。
为什么持久化使用进程而不是子线程？
利用写时复制原理达到内存高效一致，fork()子进程时，操作系统复制完全一致的内存地址副本，父子进程只有进行修改时才会发生内存的复制，有效实现内存一致，而如果使用子线程方式，虽然内存直接共享，但是同时存在线程安全问题，多线程操作需要通过同步加锁方式，影响性能；
进程隔离性原因，子进程拥有独立的内存空间和资源，如果子进程在持久化过程中崩溃，不会影响到主进程，但是如果使用子线程，由于共享地址空间，很可能导致整个Redis服务器进程崩溃，造成服务中断；
如果使用子线程，持久化过程中，可能与主线程发生CPU争用问题。

总结
核心操作是单线程：Redis 的核心操作，包括数据结构操作和命令处理，主要是在单线程中完成，以保证操作的原子性和性能。
多进程用于持久化：在持久化操作（RDB 和 AOF 重写）时，会使用多进程来避免阻塞主进程，利用 fork 子进程和写时复制技术来完成数据的持久化操作。
多线程辅助 I/O（Redis 6.0 及以后）：从 Redis 6.0 开始，使用多线程来辅助处理网络 I/O 操作，提高网络 I/O 的并发处理能力，但核心的数据操作依然是单线程。
