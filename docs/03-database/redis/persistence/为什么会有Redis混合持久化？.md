# 为什么会有Redis混合持久化？

为什么会有Redis混合持久化？
RDB 优点是数据恢复速度快，但是快照的频率不好把握。频率太低，丢失的数据就会比较多，频率太高，就会影响性能。AOF 优点是丢失数据少，但是数据恢复不快。为了集成了两者的优点， Redis 4.0 提出了混合使用 AOF 日志和内存快照，也叫混合持久化，既保证了 Redis 重启速度，又降低数据丢失风险。

混合持久化原理
混合持久化文件的生成：
混合持久化结合了 RDB 和 AOF 的优点，在 AOF 重写（AOF rewrite）时，将重写这一刻之前的内存快照以 RDB 的格式写入 AOF 文件的开头，这个过程是通过fork一个子进程来实现的，同时主进程会继续处理客户端的请求。当AOF重写正在进行时，主进程会将新的写操作存储在一个AOF重写缓冲区中。当子进程完成AOF重写后，主进程将AOF重写缓冲区中的AOF指令追加到新的AOF文件末尾，这样生成的 AOF 文件既包含了一个 RDB 格式的全量数据快照，又包含了后续的增量数据的 AOF 日志。
数据恢复：
在进行数据恢复时，Redis 首先会识别 AOF 文件是否为混合持久化文件。如果是，先加载 RDB 部分的全量数据快照，然后再执行 AOF 部分的增量日志，这样既保证了数据的完整性，又利用了 RDB 部分的快速恢复优势，同时减少了 AOF 日志的大小，提高了恢复速度。

开启方式
在 Redis 的配置文件中，可以通过设置 aof-use-rdb-preamble yes 来开启混合持久化，同时需要确保aof持久化开关配置开启 appendonly yes。而 RDB 持久化是默认启用的。
