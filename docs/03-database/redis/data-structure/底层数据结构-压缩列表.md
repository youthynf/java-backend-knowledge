# 底层数据结构-压缩列表

## 核心概念

底层数据结构-压缩列表
什么是压缩列表？
压缩列表最大的特点，就是它被设计成一种内存紧凑型的数据结构，占用一块连续的内存空间，不仅可以利用CPU缓存，而且会针对不同长度的数据，进行相应编码，有效节省内存开销（encoding编码）。但是压缩列表也有缺陷，不能保存过多的元素否则查询效率低，新增或者修改某个元素时，压缩列表的占用空间可能需要重新分配并涉及原数据的拷贝过程，甚至可能引发连锁更新的问题。因此，Redis对象（List、Hash、Zset对象）包含的元素较少或者元素不大的情况下才会使用压缩列表作为底层数据结构。

压缩列表结构设计
压缩列表是Redis为了节约内存而开发的，它是由连续的内存块组成的顺序性数据结构，有点类似于数组。压缩列表在表头有3个字段：
zlbytes：记录整个压缩列表占用的内存字节数，4字节；
zltail：记录压缩列表尾部节点距离起始地址有多少个字节，也就是列表的尾的偏移量，4字节；
zllen：记录压缩列表包含的节点数量，2字节（当数量超过2^16-1时，需要遍历才能知道确切数量）；
zlend：标记压缩列表的结束点，固定值0xFF（十进制255）；
因此，定位第一个元素和最后一个元素可以通过表头三个字段直接定位（zlbytes、zltail、zllen固定的字节长度就是第一个元素的偏移量），时间复杂度O(1)，而查询其他元素时，只能逐个查找，此时时间复杂度O(n)，因此压缩列表不适合保存过多元素。

压缩列表节点包含3部分内容：
prevlen：记录了前一个节点的长度，目的是为了实现从后往前遍历，长度为1字节/5字节；
encoding：记录了当前节点实际数据的类型和长度，类型主要有整型和字符串，长度为1字节/2字节/5字节；
data：记录了当前节点的实际数据，类型和长度都由encoding决定；

压缩列表的每个节点的prevlen属性都记录了前一个节点的长度，而且prevlen属性的空间大小跟前一个节点的长度值有关：
如果前一个节点长度小于254字节，那么prevlen属性需要用1字节空间保存这个长度值；
如果前一个节点长度大于等于254字节，那么prevlen属性需要使用5字节空间来保存这个长度；

压缩列表的每个节点的encoding属性的空间大小跟数据是字符串还是整数，以及字符串的长度有关：
如果当前节点数据是整数，则encoding会使用1字节空间进行编码，也就是encoding长度为1字节；
如果当前节点数据是字符串，根据字符串的长度大小，encoding会使用1字节/2字节/5字节空间进行编码，encoding编码前两个bit表示数据类型，后续的其他bit标识字符串数据的实际长度，即data长度。

连锁更新问题
压缩列表新增某个元素或者修改某个元素时，如果空间不够，压缩列表占用的内存空间就需要重新分配并将原来元素拷贝到新分配空间中。而当新插入或修改的元素较大时，可能会导致后续元素的prevlen占用空间发生变化，从而引起连锁更新问题，导致每个元素的空间都需要重新分配，造成压缩列表性能下降。

示例：如果一个压缩列表中有多个连续的、长度在250~253之间的节点，一旦表头插入或者表头节点修改长度大于等于254时，后续每个节点的prevlen长度由1字节变成5字节，这种在特殊情况下产生连续多次的空间扩展的操作就叫做连锁更新。

压缩列表缺陷总结：
虽然压缩列表紧凑型的内存布局能节省内存开销，但是如果保存的元素数量增加了，或是元素变大了，会导致内存重新分配，最糟糕的时会有连锁更新问题。空间扩展操作也就是重新分配内存，一旦连锁更新发生，就会导致压缩列表占用的空间要多次重新分配，影响访问性能。因此，压缩列表只会用于保存节点数量不多的场景。

Redis针对压缩列表在设计上的不足，在后来的版本，引入quicklist（Redis3.2引入）和listpack（Redis5.0引入），尽可能保持压缩列表节省内存的优势，同时解决压缩列表连锁更新的问题。

## 面试官想考什么

- Redis 类型的使用场景、底层编码和时间复杂度。
- 不同结构的内存占用、适用边界和反模式。
- key 设计、TTL、容量控制是否合理。

## 标准回答

Redis 数据结构题要同时回答使用场景、底层编码和复杂度。选择 String、Hash、List、Set、ZSet、Bitmap、HyperLogLog、GEO、Stream 等结构时，要考虑访问模式、内存占用、key 规模和命令复杂度。

## 深挖追问

1. 为什么有紧凑编码？节省内存并提升局部性。
2. ZSet 为什么适合排行榜？按 score 排序并支持范围查询。
3. Hash 适合存对象吗？适合小对象字段更新，但大 Hash 也可能成为大 Key。

## 实战场景 / SQL 示例

```text
ZADD rank:game 1001 user:1
ZREVRANGE rank:game 0 9 WITHSCORES
HSET user:1 name alice level 5
```

## 易错点 / 总结

- 不要忽略命令复杂度和集合规模。
- 大 Key、热 Key 往往比平均 QPS 更危险。
- 选择结构前先明确读写模式和过期策略。
