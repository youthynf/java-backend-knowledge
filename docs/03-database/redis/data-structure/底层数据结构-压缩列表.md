# 底层数据结构-压缩列表

## 核心概念

压缩列表 ziplist 是 Redis 早期用于小 Hash、小 ZSet、List 等对象的紧凑编码。它把多个元素存放在一块连续内存中，每个 entry 记录前一个节点长度、当前节点编码和内容，因此不需要额外指针，内存利用率高。

需要修正的是：Redis 7.0 以后 ziplist 已废弃，逐步由 listpack 替代。学习 ziplist 主要是理解 Redis 小对象紧凑编码思想，以及为什么后来要改进它。

## 面试官想考什么

- 是否知道 ziplist 是连续内存的紧凑结构。
- 是否能解释节省内存的原因和访问/更新成本。
- 是否知道连锁更新问题。
- 是否知道新版本用 listpack 替代 ziplist。

## 标准回答

ziplist 适合少量小元素，因为它没有链表指针，连续内存也有较好的缓存局部性。但连续内存意味着插入、删除可能触发内存移动；更关键的是，entry 中保存前一个节点长度，如果某个节点长度变化导致编码字节数变化，后续节点的 prevlen 也可能连锁变化，这就是连锁更新。

所以面试中不要只说 ziplist 省内存，还要补充它的缺点和版本演进：Redis 7 后更推荐说 listpack。

## 深挖追问

1. **什么是连锁更新？** 当前 entry 变长后，后一个 entry 记录 prevlen 的空间不够，也要变长，可能层层传递。
2. **ziplist 查询复杂度如何？** 按位置遍历通常 O(N)，不适合大规模随机访问。
3. **为什么小对象仍用紧凑结构？** 小数据下 O(N) 成本可接受，节省内存更重要。
4. **ziplist 和 listpack 区别？** listpack 调整节点编码，降低连锁更新风险，是 ziplist 的替代方案。

## 实战场景 / 代码示例

```bash
# 旧版本小 Hash 可能使用 ziplist，新版本多为 listpack
HSET user:mini name alice age 20
OBJECT ENCODING user:mini
```

如果面试官追问版本，建议回答：“旧版本是 ziplist，Redis 7 已被 listpack 替代，核心思想都是小对象紧凑存储。”

## 易错点 / 总结

- 不要把 ziplist 当成链表，它是连续内存结构。
- 不要忽略连锁更新，这是 ziplist 被替代的重要原因。
- Redis 版本不同，Hash/ZSet 小对象编码名称可能不同。
- ziplist 适合小而少，不适合大对象。
- 总结：ziplist 的关键词是**连续内存、省指针、连锁更新、被 listpack 替代**。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：类型语义、底层编码、复杂度、场景选择和 big key 风险；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

Redis 会根据数据规模选择紧凑或高性能编码。回答时按类型语义、底层结构、复杂度、应用场景、风险治理展开。

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
