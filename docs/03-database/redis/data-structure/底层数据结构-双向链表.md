# 底层数据结构-双向链表

## 核心概念

双向链表是 Redis 早期 List 的底层结构之一，也用于一些内部场景。它由多个节点组成，每个节点保存前驱指针、后继指针和值指针，因此从头尾插入删除很快，时间复杂度 O(1)。但每个节点都有额外指针，内存开销较大，缓存局部性也不如连续内存。

现代 Redis List 的主要底层已经是 quicklist，而不是纯双向链表。学习双向链表主要是理解 List 两端操作为什么高效，以及 quicklist 为什么要在链表基础上做折中。

## 面试官想考什么

- 是否知道双向链表支持 O(1) 头尾插入删除。
- 是否能说出它的内存开销和缓存局部性问题。
- 是否知道 Redis 3.2 后 List 使用 quicklist。
- 是否能区分链表、ziplist/listpack、quicklist 的适用场景。

## 标准回答

双向链表适合频繁从两端插入和删除，节点之间通过 prev/next 指针连接，移动头尾指针即可完成操作。但它的问题也明显：每个元素都要额外保存两个指针，内存不紧凑；遍历时节点分散在内存中，CPU 缓存命中率低。因此 Redis 后来用 quicklist 替代纯链表：每个 quicklist 节点内部存多个元素，减少指针开销。

面试回答时要注意版本：如果问“Redis List 现在底层是什么”，优先回答 quicklist；如果问“链表有什么特点”，再展开双向链表。

## 深挖追问

1. **链表为什么两端操作 O(1)？** 因为有 head/tail 指针，插入删除只改少量指针。
2. **链表按下标查询为什么慢？** 需要从头或尾逐个遍历，复杂度 O(N)。
3. **为什么纯链表不省内存？** 每个节点有 prev、next、value 指针和分配器开销。
4. **quicklist 如何改进？** 把多个元素放进一个 listpack 节点，再用链表串节点。

## 实战场景 / 代码示例

```bash
LPUSH stack:demo a b c
LPOP stack:demo
RPUSH queue:demo a b c
LPOP queue:demo
```

这些命令体现了链表/quicklist 两端操作的优势，但如果执行 `LINDEX` 深位置或 `LRANGE` 超大范围，仍会出现 O(N) 成本。

## 易错点 / 总结

- 不要把当前 Redis List 简化成纯双向链表。
- 双向链表两端快，不代表随机访问快。
- 指针多导致内存开销和缓存不友好。
- 大 List 要控制长度，避免删除、迁移、持久化时产生延迟。
- 总结：双向链表的关键词是**prev/next、头尾 O(1)、随机访问 O(N)、被 quicklist 折中优化**。

## 面试版详细讲解

### 核心概念

这道题属于 **Redis 数据结构** 的高频考点，核心要抓住：redisObject、dict、SDS、listpack、quicklist、intset、skiplist。Redis 会根据数据规模选择紧凑或高性能编码。回答时按类型语义、底层结构、复杂度、应用场景、风险治理展开。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：类型语义、底层编码、复杂度、场景选择和 big key 风险；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

Redis 会根据数据规模选择紧凑或高性能编码。回答时按类型语义、底层结构、复杂度、应用场景、风险治理展开。

答题时建议用“三段式”：

1. 先给结论，明确适用前提；
2. 再解释底层机制或执行过程；
3. 最后补充业务取舍、风险点和排查手段。

### 深挖追问

- 这个结论在高并发或大数据量下是否仍然成立？
- 它依赖哪些版本、配置、索引/编码或业务一致性要求？
- 线上异常时应该看哪些命令、日志、指标或执行计划？

### 示例 / 实战场景

用户资料整体读写用 String，字段更新用 Hash，去重用 Set，排行榜用 Zset，签到用 Bitmap，消息流用 Stream。

```bash
# 先小范围验证命令复杂度和返回量，避免线上直接扫大 key
redis-cli --scan --pattern 'biz:*' | head
redis-cli --bigkeys
```

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
