# 底层数据结构-quicklist

## 核心概念

底层数据结构-quicklist
在Redis3.0之前，List对象的底层数据结构是双向链表或者压缩列表。然后在Redis3.2的时候，List对象的底层改由quicklist数据结构实现。

quicklist实现原理：
quicklist就是双向链表+压缩列表的组合，因为一个quicklist就是一个链表，而链表中的每个元素又是一个压缩列表。通过控制每个链表节点中的压缩列表的大小和元素的个数，来规避连锁更新的问题，因为压缩列表的元素越少或越小，连锁更新带来的影响就越小，从而提供了更好的访问性能。

链表quicklist结构体设计：
head：链表头节点指针；
tail：链表尾节点指针；
count：所有压缩列表中总的元素个数；
len：链表中链表节点的个数；

链表节点quicklistNode结构体设计：
prev：前一个链表节点的指针；
next：后一个链表节点的指针；
zl：指向的压缩列表指针；
sz：压缩列表的字节大小；
count：压缩列表的元素个数；

数据插入过程：
在向quicklist添加一个元素的时候，不会向普通的链表那样，直接新建一个链表节点，而是会检查出插入位置的压缩列表是否能够容纳该元素，如果能容纳，则直接保存到quicklistNode结构里的压缩列表，如果不能容纳，才新建一个新的quicklistNode结构。quicklist会控制quicklistNode结构里的压缩列表大小或者个数（支持参数配置），来规避潜在的连锁更新风险，但并没有完全解决连锁更新问题。当压缩列表的大小超出参数限制，并且插入的位置不是开头和结尾时，涉及到节点分裂的过程，分裂位置为中间位置。

数据查找过程：
Redis会维护每个QuickList节点的元素计数
2. 查找时，首先检查索引是否在第一个节点的范围内
如果是，直接访问该节点的ziplist获取元素
如果不是，累加节点计数并移动到下一个节点
直到找到包含目标索引的节点
特别注意，索引下标元素并不是指 quicklistNode 节点元素，而是所有压缩列表中的元素中对应下标值的元素。查找时利用节点数量这个属性值来进行定位索要查找的元素具体在那个节点下的压缩列表中。

## 面试官想考什么

- Redis 类型的使用场景、底层编码和时间复杂度。
- 不同结构的内存占用、适用边界和反模式。
- key 设计、TTL、容量控制是否合理。

## 标准回答

Redis 数据结构题要同时回答使用场景、底层编码和复杂度。选择 String、Hash、List、Set、ZSet、Bitmap、HyperLogLog、GEO、Stream 等结构时，要考虑访问模式、内存占用、key 规模和命令复杂度。

## 深挖追问

1. 为什么有紧凑编码？节省内存并提升局部性。
2. ZSet 为什么适合排行榜？按 score 排序并支持范围查询。
3. Hash 适合存对象吗？适合小对象字段更新，但大 Hash 也可能成为大 Key。

## 实战场景 / SQL 示例

```text
ZADD rank:game 1001 user:1
ZREVRANGE rank:game 0 9 WITHSCORES
HSET user:1 name alice level 5
```

## 易错点 / 总结

- 不要忽略命令复杂度和集合规模。
- 大 Key、热 Key 往往比平均 QPS 更危险。
- 选择结构前先明确读写模式和过期策略。
