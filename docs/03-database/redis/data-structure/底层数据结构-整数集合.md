# 底层数据结构-整数集合

## 核心概念

整数集合 intset 是 Redis Set 在“小且全为整数”时使用的紧凑编码。它内部用一块连续内存保存有序整数数组，并根据元素大小选择 `int16`、`int32` 或 `int64` 编码。只要新元素超出当前编码范围，就会升级编码；如果元素不是整数，或数量超过 `set-maxintset-entries`（默认 512），Set 会转为 hashtable。

intset 的特点是节省内存、元素有序、支持二分查找；缺点是插入删除可能移动数组元素，适合小集合，不适合大规模频繁修改。

## 面试官想考什么

- 是否知道 Set 小整数集合会用 intset。
- 是否理解编码升级和不可降级。
- 是否能说出 intset 为什么省内存、为什么不适合大集合。
- 是否能联系 Set 的底层编码转换条件。

## 标准回答

当 Set 中元素都是整数且数量较少时，Redis 用 intset 存储。intset 是连续内存数组，按整数大小选择最小可容纳编码，例如全是小整数可用 int16，从而减少内存占用。为了保持有序，查询可以二分；插入时需要找到位置并移动后续元素。

如果插入一个更大的整数，例如从 int16 范围升级到 int32，Redis 会重新分配并转换所有元素；升级后不会因为删除大整数而降级。插入非整数元素时，Set 会转成 hashtable。

## 深挖追问

1. **intset 为什么有序？** 有序数组便于二分查找，也方便判断元素是否存在。
2. **编码升级为什么不可逆？** 降级收益有限且会增加实现复杂度和性能成本，Redis 通常只做单向转换。
3. **intset 插入复杂度是多少？** 查找位置可 O(logN)，但移动元素 O(N)，所以总体可能 O(N)。
4. **什么时候转 hashtable？** 出现非整数元素，或元素数量超过配置阈值。

## 实战场景 / 代码示例

```bash
SADD lucky:ids 1 2 3 100
OBJECT ENCODING lucky:ids
SISMEMBER lucky:ids 3

# 加入字符串后会转为 hashtable
SADD lucky:ids user:5
OBJECT ENCODING lucky:ids
```

小整数黑名单、活动编号集合等可以受益于 intset，但集合持续增长时要关注编码转换和内存变化。

## 易错点 / 总结

- intset 只适用于整数元素，字符串数字和真正整数要看 Redis 解析结果。
- intset 省内存但不是大集合方案，大集合会转 hashtable。
- 编码升级不可逆，删除大元素不会自动降级。
- 连续内存插入删除有移动成本，频繁修改大集合不适合 intset。
- 总结：intset 的关键词是**Set 小整数编码、连续有序数组、编码升级、超过阈值转 hashtable**。

这道题属于 **Redis 数据结构** 的高频考点，核心要抓住：redisObject、dict、SDS、listpack、quicklist、intset、skiplist。Redis 会根据数据规模选择紧凑或高性能编码。回答时按类型语义、底层结构、复杂度、应用场景、风险治理展开。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：类型语义、底层编码、复杂度、场景选择和 big key 风险；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

Redis 会根据数据规模选择紧凑或高性能编码。回答时按类型语义、底层结构、复杂度、应用场景、风险治理展开。

答题时建议用“三段式”：

- 线上异常时应该看哪些命令、日志、指标或执行计划？

# 先小范围验证命令复杂度和返回量，避免线上直接扫大 key

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
