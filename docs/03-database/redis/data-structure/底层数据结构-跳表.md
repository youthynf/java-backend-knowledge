# 底层数据结构-跳表

底层数据结构-跳表
Redis 只有 Zset 对象的底层实现用到了跳表，跳表的优势是能够支持平均 O(logN) 时间复杂度的节点查找。

Zset底层数据结构实现：
Zset 对象使用zset 结构体表示，由跳表 *zsl、哈希表*dict 两个成员变量组成。这样好处是既能进行高效的范围查询，也能进行高效的单点查询。Zset对象在执行数据插入或数据更新的过程中，会依次在跳表和哈希表插入或更新相应的数据，从而保证哈希表和跳表中记录的信息一致。

Zset对象在使用跳表作为数据结构的时候，是使用由哈希表+跳表组成的zset结构体，但是我们讨论的时候，都会说跳表是Zset对象的底层数据结构，而不会提及哈希表，是因为zset中的哈希表只是用于以常数复杂度获取元素权重，大部分的操作都是跳表实现的。

跳表结构实现：
跳表是在链表的基础上改进过来的，实现一种多层的有序链表。跳表的头节点包含指向每个层级的首个跳表节点的指针，然后每个层级的节点都是通过指针链接起来的。

跳表 zskiplist 结构体：
header：指向跳表头节点的指针；
tail：指向跳表尾节点的指针；
length：跳表的长度，即跳表的节点总数量；
level：跳表的最大层数；

跳表节点zskiplistNode结构体：
ele：Zset对象的元素值，sds类型；
score：元素的权重值，double类型；
backward：指向后一个（降序）跳表节点的指针；
level[]：当前节点的level数组，保存每层上的前向（增序）指针和跨度；

跳表的查询过程：优先权重，然后比值
跳表会从头节点的最高层开始，逐一遍历每一层。在遍历某一层的跳表节点时，会用跳表节点的SDS类型的元素值和double类型的元素权重值进行判断：
如果当前节点权重值小于查询的权重值时，跳表就会访问该层的下一个节点，直到找到第一个比查询权重值大或等于的节点；
如果当前节点权重值等于查询的权重值时，并且当前节点的SDS类型数据小于要查询的数据时，跳表就会访问改成的下一个节点；

跳表节点层数设置：
跳表相邻两层的节点数量最理想的比例是2:1，查询时间复杂度可以降到 O(logN)。Redis采用巧妙方法实现，跳表在创建节点的时候，随机生成每个节点的层数，并没有严格维持相邻两层的节点数量比例为2:1的情况。具体做法是，跳表在创建节点时候，会生成范围为[0-1]的一个随机数，如果这个随机数小于0.25，那么层数加1层，然后继续生成下一个随机数，直到随机数大于0.25结束，最终确定该节点层数。相当于每增加一层的概率不超过25%，层数越高，概率越低，层数最高限制是64，在创建跳表「头节点」的时候，就会直接创建 64 层高的头节点。。Redis 7.0 定义为 32，Redis 5.0 定义为 64，Redis 3.0 定义为 32。

为什么用跳表而不用平衡树？
从内存占用上比较，跳表比平衡树更灵活一些。跳表每个节点包含的指针数跟随机参数有关，而平衡树固定是2，按照25%设置，平均每个节点数量1/0.75=1.33个。
在做范围查询的时候，跳表比平衡树操作要简单。平衡树需要中序遍历。
从算法实现难度上比较，跳表比平衡树要简单得多。平衡树需要树调整。
