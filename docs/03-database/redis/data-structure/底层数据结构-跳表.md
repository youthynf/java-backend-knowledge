# 底层数据结构-跳表

## 核心概念

跳表 skiplist 是 Redis ZSet 在大数据量下的核心底层结构之一，通常与 dict 一起使用：dict 负责按 member 快速查 score，skiplist 负责按 score 有序范围查询和排名。跳表由多层有序链表组成，高层链表像索引一样跳过大量节点，使查找、插入、删除平均复杂度达到 O(logN)。

Redis 选择跳表而不是红黑树，一个重要原因是实现简单，范围遍历非常自然；从某个 score 定位到起点后沿底层链表向后扫描即可。

## 面试官想考什么

- 是否知道 ZSet 大对象底层是 skiplist + dict。
- 是否能解释跳表多层索引和 O(logN) 平均复杂度。
- 是否理解范围查询为什么适合跳表。
- 是否能比较跳表与平衡树的取舍。

## 标准回答

跳表是在有序链表上增加多级索引的数据结构。最底层保存所有节点，上层保存部分节点，查找时从最高层开始向右、向下移动，类似二分缩小范围。Redis ZSet 使用跳表保存 score 排序关系，因此可以高效执行 `ZRANGE`、`ZRANGEBYSCORE`、`ZREVRANGE` 等命令。

ZSet 还需要 dict，因为跳表按 score 查范围很方便，但按 member 查 score 不够快；dict 可以 O(1) 找到 member。所以完整回答应说 skiplist + dict 配合。

## 深挖追问

1. **跳表复杂度一定 O(logN) 吗？** 是概率意义上的平均 O(logN)，层数随机生成，极端情况可能退化但概率很低。
2. **为什么适合范围查询？** 定位起点后，沿底层有序链表连续遍历即可返回范围内元素。
3. **同分 score 怎么排？** Redis 使用 member 字典序作为 tie-breaker，保证顺序确定。
4. **为什么不用红黑树？** 跳表实现更简单，范围遍历方便，性能也足够稳定。

## 实战场景 / 代码示例

```bash
ZADD rank:game 100 user:1 90 user:2 100 user:3
ZREVRANGE rank:game 0 9 WITHSCORES
ZRANGEBYSCORE rank:game 90 100 WITHSCORES
ZSCORE rank:game user:1
```

排行榜、延迟队列、按热度分页都依赖 ZSet 的有序范围能力；但要限制集合大小，避免大 ZSet 范围扫描阻塞。

## 易错点 / 总结

- 不要只说 ZSet 底层是跳表，完整答案是 skiplist + dict，小对象可能是 listpack。
- 跳表是有序结构，不是哈希结构。
- `ZRANGE 0 -1` 对大 ZSet 仍然危险，复杂度包含返回元素数 M。
- score 是 double，要注意精度和同分排序。
- 总结：跳表的关键词是**多层索引、平均 O(logN)、范围查询、ZSet、skiplist + dict**。
