# 底层数据结构-哈希表

## 核心概念

哈希表是 Redis 字典 dict 的核心实现，也是 Redis 数据库 key 空间、Hash 类型大对象、Set 类型大对象等结构的基础。Redis dict 使用数组 + 链式哈希解决冲突，并通过渐进式 rehash 扩容/缩容，避免一次性迁移大量元素造成长时间阻塞。

Redis 字典通常包含两个哈希表 `ht[0]` 和 `ht[1]`。正常只用 `ht[0]`；扩容或缩容时分配 `ht[1]`，随后在每次增删查改或定时任务中迁移一部分桶，直到迁移完成。

## 面试官想考什么

- 是否知道 Redis key 空间底层依赖 dict/hashtable。
- 是否能解释哈希冲突和链式解决。
- 是否知道渐进式 rehash 的目的和过程。
- 是否能联系 Hash/Set 编码转换、大 Key、rehash 延迟。

## 标准回答

Redis 哈希表使用数组保存桶，冲突时用链表挂接 dictEntry。为了保证查询性能，元素变多时需要扩容；但 Redis 是单线程事件循环，如果一次性 rehash 全量数据会阻塞请求，所以采用渐进式 rehash：同时保留新旧两个哈希表，每次操作顺带迁移一小部分数据。

查找时如果正在 rehash，需要先查 ht[0]，再查 ht[1]；新增一般写入 ht[1]。迁移完成后释放旧表，新表变成 ht[0]。

## 深挖追问

1. **Redis 如何处理哈希冲突？** 链地址法，同一个桶的多个 entry 用 next 指针连接。
2. **渐进式 rehash 有什么代价？** rehash 期间查询可能查两张表，且每次命令额外做少量迁移工作。
3. **为什么不一次性 rehash？** 数据量大时会长时间阻塞 Redis 主线程。
4. **Hash 类型小对象一定用哈希表吗？** 不一定，小 Hash 通常用 listpack，超过阈值才转 hashtable。

## 实战场景 / 代码示例

```bash
HSET user:1 name alice city shanghai
OBJECT ENCODING user:1

# 当字段数量或值长度超过阈值后，编码可能变为 hashtable
HGET user:1 name
```

排查性能抖动时，如果存在大量大 Hash/大 Set，同时伴随写入扩容，可能需要关注 rehash 和大 Key 影响。

## 易错点 / 总结

- 哈希表平均 O(1)，不是任何情况下都绝对 O(1)，冲突和 rehash 会带来额外成本。
- 渐进式 rehash 降低单次阻塞，但不会让扩容完全无成本。
- 大 Hash/Set 删除、迁移、扩容都可能造成延迟尖刺。
- Redis dict 与 Java HashMap 类似但不是同一实现，回答不要套错红黑树细节。
- 总结：哈希表的关键词是**dict、桶数组、链式冲突、渐进式 rehash、O(1) 平均复杂度**。
