# 底层数据结构-哈希表

## 核心概念

底层数据结构-哈希表
哈希表是一种保存键值对（key-value）的数据结构。Redis的Hash对象类型的底层数据结构实现之一是压缩列表（最新Redis代码已经压缩列表替换成listpack），另一底层数据结构实现就是哈希表。

哈希表优点在于，它能以O(1)时间复杂度快速查询数据。通过将key使用Hash函数的计算，就能定位数据在表中的位置，因为哈希表实际上是数据，所以可以通过索引值快速查到数据。

哈希冲突解决
Redis采用了链式哈希来解决哈希冲突，在不扩容哈希表的前提下，将具有相同哈希值的数据串起来，形成链接，以便这些数据在表中仍然可以被查询到。

哈希表结构设计
哈希表结构 dictht 是一个数组（dictEntry **table），数组中每个元素都是一个指向哈希节点（dictEntry）的指针。

哈希表dictht的结构组成：
table：指向哈希数组的指针，该数组存放着是指向哈希节点（dictEntry）的指针；
size：哈希表的大小；
sizemask：哈希表大小掩码，用于计算索引值；
used：该哈希表已有的节点数量；

哈希节点dictEntry的结构组成：
key：指向键对象的指针；
val：指向值对象的指针，与uint64_t、int64_t、double组成联合体v；
uint64_t：如果值是无符号64位整数时直接存放在该变量中，无需使用val指针指向实际值；
int64_t：如果值是有符号64位整数时直接存放在该变量中，无需使用val指针指向实际值；
double：如果值是浮点数时直接存放在该变量中，无需使用val指针指向实际值；
next：指向下一个哈希表节点，形成单向链表；

重哈希reHash：
Redis使用 dictht 结构体表示哈希表，在实际使用哈希表时，Redis定义了一个 dict 结构体，这个结构体定义了两个哈希表（ht[2]），分别存放两个 dictht 哈希表。正常服务请求截断，插入的数据都会插入哈希表1，此时哈希表2并没有分配空间。

随着数据增加，触发了rehash操作，这个过程分为三步：
给哈希表2分配空间，一般比哈希表1大一倍（两倍的意思）；
将哈希表1的数据迁移到哈希表2中；
迁移完成后，哈希表1的空间会被释放，并把哈希表2设置为哈希表1，然后在哈希表2新创建一个空白的哈希表，为下次rehash做准备。

渐进式rehash：
为了避免rehash在数据迁移过程中，因为拷贝数据耗时，影响Redis性能，所以Redis采用了渐进式rehash，也就是数据迁移工作不是一次性完成，而是分多次迁移。具体步骤如下：
给哈希表2分配空间；
在rehash进行期间，每次哈希表元素进行新增、删除、查找或者更新操作时，Redis除了会执行对应的操作外，还会顺序将哈希表1中索引位置上所有的key-value迁移到哈希表2上；
随着处理客户端发起的哈希表操作请求数量越多，最终在某个时间点会把哈希表1上所有key-value迁移到哈希表2，从而完成rehash操作。

注意点：
在进行渐进式rehash过程中，会有两个哈希表，所以渐进式rehash进行期间，哈希元素的删除、查找、更新操作都会在这两个表进行，如查找时现在哈希表1查找，找不到则从哈希表2查找。而新增则只会在哈希表2上操作，这样保证了哈希表1的数据只会减少，最终称为空表。

rehash触发条件：
触发条件跟负载因子有关系，负载因子=哈希表已保存节点数量/哈希表大小。
当负载因子大于等于1，并且Redis没有在执行bgsave命令或者bgrewiteaof命令，也就是没有执行RDB快照或没有进行AOF重写时，就会进行rehash操作；
当负载因子大于等于5时，此时说明哈希冲突非常严重了，不管有没有在执行RDB快照或AOP重写，都会强制进行rehash操作。

## 面试官想考什么

- Redis 类型的使用场景、底层编码和时间复杂度。
- 不同结构的内存占用、适用边界和反模式。
- key 设计、TTL、容量控制是否合理。

## 标准回答

Redis 数据结构题要同时回答使用场景、底层编码和复杂度。选择 String、Hash、List、Set、ZSet、Bitmap、HyperLogLog、GEO、Stream 等结构时，要考虑访问模式、内存占用、key 规模和命令复杂度。

## 深挖追问

1. 为什么有紧凑编码？节省内存并提升局部性。
2. ZSet 为什么适合排行榜？按 score 排序并支持范围查询。
3. Hash 适合存对象吗？适合小对象字段更新，但大 Hash 也可能成为大 Key。

## 实战场景 / SQL 示例

```text
ZADD rank:game 1001 user:1
ZREVRANGE rank:game 0 9 WITHSCORES
HSET user:1 name alice level 5
```

## 易错点 / 总结

- 不要忽略命令复杂度和集合规模。
- 大 Key、热 Key 往往比平均 QPS 更危险。
- 选择结构前先明确读写模式和过期策略。
