# MVCC实现原理是怎么样的？

MVCC实现原理是怎么样的？
MVCC（多版本并发控制）是一种数据库并发控制方法，通过维护数据的版本链来控制并发事务访问同一个记录，从而提高系统的并发性能。主要依赖「读视图ReadView」和「行记录的隐藏字段」实现。

读视图ReadView
ReadView可以理解为是一个数据快照，对于读提交和可重复读隔离级别的事务来说，它们是通过读视图ReadView来实现的，它们的区别在于创建ReadView的时机不同：
读提交隔离级别：在每个select语句开始前都会重新生成一个ReadView；
可重复读隔离级别：在第一个select语句开始前才会生成一个ReadView；

ReadView有四个重要字段：
creator_trx_id：指创建该ReadView的事务ID；
m_ids：指创建ReadView时，当前数据库中活跃且未提交的事务ID列表；
min_trx_id：指在创建ReadView时，当前数据库中「活跃事务」中事务最小的事务ID，也就是m_ids的最小值；
max_trx_id：指创建ReadView时，当前数据库中应该给下一个事务分配的事务ID值，也就是全局事务中最大的事务ID值+1；

行记录中的隐藏字段
在InnoDB存储引擎中，每一行数据除了包含实际业务数据字段外，还额外附带了一些隐藏字段用于MVCC机制的实现，这些隐藏字段主要有：
trx_id：当一个事务对某条聚簇索引记录进行改动时，就会把该事务的事务 id 记录在 trx_id 隐藏列里；
roll_pointer：每次对某条聚簇索引记录进行改动时，都会把旧版本的记录写入到 undo 日志中，然后这个隐藏列是个指针，指向每一个旧版本记录，于是就可以通过它找到修改前的记录。

如何判断可见性？
一个事务去访问记录的时候，除了自己更新记录总是可见之外，需要结合「读视图ReadView」和「行记录版本链」来确定记录的可见性：
如果记录的trx_id小于ReadView中的min_trx_id值，说明这个行记录版本在创建ReadView之前已经提交的事务生成的，因此可见；
如果记录的trx_id大于等于ReadView中的max_trx_id值，则说明这个版本的记录是创建ReadView之后才启动的事务生成的，所以不可见；
如果记录的trx_id在m_ids列表中，表示生成该版本的记录的事务目前依然活跃未提交，因此不可见；
如果记录的trx_id不在mids列表中，表示生成该版本记录的活跃事务已经被提交，因此可见；
这种通过「版本链」来控制并发事务访问同一个记录时的行为就叫 MVCC（多版本并发控制）。
