# 事务的隔离级别是怎么实现的

## 核心概念

隔离级别规定了并发事务之间“能看到对方的什么”。SQL 标准定义了四个级别：读未提交（Read Uncommitted）、读已提交（Read Committed）、可重复读（Repeatable Read）、串行化（Serializable），严格性递增、并发性递减。

不同隔离级别对应不同的并发异常：脏读、不可重复读、幻读。MySQL InnoDB 默认使用可重复读（RR），并通过 MVCC 和 next-key lock 在很大程度上规避了幻读，这与 SQL 标准里“RR 仍允许幻读”的描述略有出入。

理解隔离级别的关键不是背四个名字，而是搞清楚每个级别“读什么版本的数据、加什么锁、什么时候生成 ReadView”。

## 标准回答

> 四种隔离级别中，读未提交直接读最新数据（可能脏读），串行化对所有读写加锁串行执行，读已提交（RC）和可重复读（RR）通过 ReadView 实现。RC 在每次快照读都生成新的 ReadView，所以同一事务内能看到其他已提交事务的修改（不可重复读）；RR 在事务第一次快照读时生成 ReadView 并复用，所以事务内多次读结果一致。InnoDB 的 RR 在当前读场景下用 next-key lock（记录锁 + 间隙锁）规避幻读。

## 实现原理

### 并发异常与隔离级别的关系

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|----------|------|
| 读未提交 Read Uncommitted | 可能 | 可能 | 可能 |
| 读已提交 Read Committed | 避免 | 可能 | 可能 |
| 可重复读 Repeatable Read | 避免 | 避免 | InnoDB 很大程度避免 |
| 串行化 Serializable | 避免 | 避免 | 避免 |

### 各隔离级别的实现方式

**读未提交（RU）**：直接读取数据页上的最新版本，不加锁、不生成 ReadView。可能读到其他事务未提交的修改，几乎不用。

**读已提交（RC）**：每次快照读都生成新的 ReadView，所以能读到该时刻已提交的最新版本。同一事务内两次读同一行可能不同（不可重复读）。当前读加记录锁，不加间隙锁。

**可重复读（RR）**：事务内第一次快照读生成 ReadView 并复用到事务结束，所以事务内多次读同一行结果一致。当前读加 next-key lock（记录锁 + 间隙锁）防止其他事务在范围内插入。InnoDB 默认级别。

**串行化（Serializable）**：所有读加共享锁、写加排他锁，读写互斥，并发度极低，几乎不用。

### ReadView 的生成时机差异

ReadView 是事务在做快照读时构建的“可见性快照”，包含四个字段：

- `creator_trx_id`：创建该 ReadView 的事务 ID
- `m_ids`：生成 ReadView 时活跃未提交的事务 ID 列表
- `min_trx_id`：`m_ids` 中最小值
- `max_trx_id`：下一个将分配的事务 ID

可见性判断（针对记录的 `trx_id`）：

| 条件 | 可见性 |
|------|--------|
| `trx_id < min_trx_id` | 可见（事务在 ReadView 之前提交） |
| `trx_id >= max_trx_id` | 不可见（事务在 ReadView 之后启动） |
| `min_trx_id <= trx_id < max_trx_id` 且在 `m_ids` 中 | 不可见（活跃未提交） |
| `min_trx_id <= trx_id < max_trx_id` 且不在 `m_ids` 中 | 可见（已提交） |

不可见时，沿 undo log 版本链找上一个版本继续判断，直到可见或链尾。

**RC 与 RR 的唯一区别**就是 ReadView 的生成时机：

- RC：每次 `SELECT` 都新建 ReadView
- RR：事务内第一次 `SELECT` 新建 ReadView，之后复用

### 事务启动时机

```sql
-- 方式一：BEGIN 后第一条 SQL 才真正启动事务（生成 ReadView）
BEGIN;
SELECT ... -- 此刻生成 ReadView

-- 方式二：立即启动事务并生成一致性快照
START TRANSACTION WITH CONSISTENT SNAPSHOT;
```

RR 下用方式二能在第一条 SQL 之前就锁定快照点，常用于备份一致性。

### 设置隔离级别

```sql
-- 全局级别
SET GLOBAL transaction_isolation = 'REPEATABLE-READ';

-- 会话级别
SET SESSION transaction_isolation = 'READ-COMMITTED';

-- 查看当前级别
SELECT @@transaction_isolation;
```

## 实战场景

| 场景 | 推荐级别 | 原因 |
|------|---------|------|
| 报表/统计查询 | RR | 多次查询结果一致，避免数据漂移 |
| 高并发互联网业务 | RC | 锁范围小，间隙锁导致死锁少 |
| mysqldump 一致性备份 | RR + WITH CONSISTENT SNAPSHOT | 快照点固定，备份一致 |
| 金融强一致 | RC + 显式行锁 | 不依赖隔离级别兜底，显式加锁可控 |

## 深挖追问

### 1. 为什么互联网项目常把级别改成 RC？

RR 下当前读会加间隙锁，高并发写入场景下间隙锁容易引发锁冲突和死锁；RC 不加间隙锁，并发更好。互联网业务通常能接受不可重复读，配合乐观锁或显式行锁即可保证一致性。阿里、美团等大厂线上 MySQL 普遍使用 RC。

### 2. RR 真的解决幻读了吗？

快照读通过 MVCC 规避（事务内复用 ReadView）；当前读通过 next-key lock 规避（阻塞范围内插入）。但仍存在例外：事务先快照读、后当前读，或更新了一条原本不可见的记录（更新后 trx_id 变为自己，记录变可见），这两种情况会出现幻读。要彻底避免，可在事务开始时立即执行当前读加锁，或升级到串行化。

### 3. 隔离级别和 MVCC、锁什么关系？

隔离级别是规范，MVCC 和锁是 InnoDB 实现这个规范的两套机制。MVCC 服务于快照读（不加锁），锁服务于当前读和写操作。RR 和 RC 都用 MVCC，区别只在 ReadView 生成时机；串行化退化为纯锁机制。

### 4. ReadView 在 RC 下每次生成，开销大吗？

不大。ReadView 主要是拷贝当前活跃事务 ID 列表，活跃事务少时几乎零成本。真正开销在 undo log 版本链遍历，长事务 + 频繁更新会让链变长。

## 易错点

- 误以为 RR 完全解决幻读：当前读 + 快照读混用仍可能幻读。
- 误以为 `BEGIN` 立即生成 ReadView：RR 下是第一条快照读时才生成。
- 误以为隔离级别越高越好：串行化并发极差，几乎不用。
- 把不可重复读和幻读搞混：前者是同一行内容变了，后者是结果集行数变了。
- 以为 RC 不加锁：当前读仍然加记录锁，只是不加间隙锁。

## 总结

隔离级别本质是“读哪个版本 + 加什么锁”的约定。RC 和 RR 都基于 ReadView + MVCC，差别只在 ReadView 生成时机；RU 直接读最新，Serializable 全程加锁。InnoDB 默认 RR，但互联网项目常切到 RC 以提升并发。下一步可深入 MVCC 版本链和 next-key lock 细节。
