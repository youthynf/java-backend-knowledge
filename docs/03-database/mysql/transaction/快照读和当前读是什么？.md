# 快照读和当前读是什么

## 核心概念

InnoDB 的读操作分为两种：快照读和当前读。两者读取的数据版本不同、加锁行为不同，是理解 MySQL 并发控制的关键。

快照读读取的是历史版本（由 MVCC 决定可见版本），不加锁，读写互不阻塞，是普通 `SELECT` 的默认行为。当前读读取的是最新版本，并加锁（共享锁或排他锁），保证读到的是“此刻”的数据，常用于写操作和加锁读。

区分这两种读的本质是：快照读走 MVCC，当前读走加锁。把两者搞混是幻读、死锁、读到旧数据等问题的常见根源。

## 标准回答

> 普通 `SELECT` 是快照读，走 MVCC，读历史版本不加锁；`SELECT ... FOR UPDATE`、`SELECT ... FOR SHARE`、`INSERT`、`UPDATE`、`DELETE` 是当前读，读最新版本并加锁。RR 隔离级别下快照读复用事务第一次的 ReadView，所以事务内多次读结果一致；当前读加 next-key lock（记录锁 + 间隙锁）防止范围内插入。同一个事务里先快照读后当前读，可能因为当前读看到新插入的行而产生幻读。

## 实现原理

### 快照读

- 读取历史版本，由 MVCC 决定可见版本
- 不加锁，读写不互相阻塞
- RR 隔离级别：事务内第一次快照读生成 ReadView 并复用
- RC 隔离级别：每次快照读都生成新 ReadView
- 触发语句：普通 `SELECT`

### 当前读

- 读取最新版本数据
- 加锁：`FOR UPDATE` 加排他锁，`FOR SHARE` 加共享锁
- RR 下加 next-key lock = 记录锁 + 间隙锁，锁住记录及其前面的间隙
- RC 下只加记录锁，不加间隙锁
- 触发语句：

```sql
SELECT ... FOR UPDATE;
SELECT ... FOR SHARE;
INSERT INTO ...;
UPDATE ... SET ...;
DELETE FROM ...;
```

### 加锁范围示例

假设表 `t` 有 `id`（主键）= 5, 10, 15，隔离级别 RR：

```sql
-- 命中唯一索引等值，退化为记录锁（仅锁 id=10）
SELECT * FROM t WHERE id = 10 FOR UPDATE;

-- 范围查询，加 next-key lock：(10, 15], (15, +∞)
SELECT * FROM t WHERE id > 10 FOR UPDATE;

-- 无索引，全表扫描，每行都加 next-key lock，等价于锁全表
SELECT * FROM t WHERE name = 'x' FOR UPDATE;  -- name 无索引
```

### 快照读与当前读混用的幻读

```sql
-- RR 隔离级别
BEGIN;
SELECT * FROM t WHERE id > 10;          -- 快照读，假设看到 15
-- 其他事务插入 id=12 并提交
SELECT * FROM t WHERE id > 10;          -- 快照读，仍看到 15（ReadView 复用）
UPDATE t SET name='x' WHERE id > 10;    -- 当前读！看到 12 和 15，并加锁
SELECT * FROM t WHERE id > 10;          -- 快照读，现在看到 12（因为 12 的 trx_id 变为自己）
-- 幻读出现
```

## 代码示例

扣减库存的典型场景：先当前读锁定行，再更新，保证扣减基于最新库存。

```sql
START TRANSACTION;

-- 当前读：拿到最新库存并加行锁，防止其他事务并发修改
SELECT stock FROM sku WHERE id = 100 FOR UPDATE;

-- 基于当前读的结果做条件更新
UPDATE sku SET stock = stock - 1 WHERE id = 100 AND stock > 0;

COMMIT;
```

如果用快照读，会读到旧版本，导致超卖：

```sql
START TRANSACTION;
SELECT stock FROM sku WHERE id = 100;  -- 快照读，读到 stock=1，不加锁
-- 另一事务此刻把 stock 减到 0
UPDATE sku SET stock = stock - 1 WHERE id = 100;  -- 当前读 + 写，stock=0，更新 0 行
COMMIT;
```

## 实战场景

| 场景 | 用哪种读 | 原因 |
|------|---------|------|
| 列表查询、详情展示 | 快照读 | 不需要最新，并发好 |
| 扣库存、扣余额 | 当前读 | 必须基于最新数据决策 |
| 唯一性检查后插入 | 当前读 + 唯一索引 | 防止并发插入重复 |
| 备份一致性 | 快照读 | 走 MVCC 不阻塞业务 |
| 范围统计报表 | 快照读 | 不阻塞写入 |

## 深挖追问

### 1. `UPDATE` 语句为什么是当前读？

`UPDATE` 必须基于最新数据修改，否则会覆盖其他事务的更新。执行器先当前读取出最新行加锁，再修改，整个过程中其他事务无法修改该行。

### 2. 快照读一定不加锁吗？

普通快照读不加锁，但 RR 下隔离级别串行化（Serializable）会把所有普通 `SELECT` 隐式转为 `SELECT ... FOR SHARE`，退化为当前读。所以 Serializable 下没有真正的快照读。

### 3. 当前读加锁为什么有时锁全表？

当查询条件无法命中索引时，存储引擎需要扫描聚簇索引所有行，每行都加 next-key lock，效果等价于锁全表。这就是为什么“`UPDATE`/`DELETE` 一定要走索引”是铁律。

### 4. 快照读和当前读在同一事务里看到的数据版本一样吗？

不一定。快照读看 ReadView 决定的历史版本，当前读看最新版本。如果事务开启后其他事务提交了修改，快照读看不到（RR 下），当前读能看到。这就是“先快照读后当前读会幻读”的根源。

## 易错点

- 用快照读做库存扣减：读到旧版本导致超卖。
- `UPDATE`/`DELETE` 不走索引：当前读退化为全表锁。
- 以为快照读永远不加锁：Serializable 下会退化为加锁读。
- 以为 RR 完全无幻读：快照读+当前读混用仍可能幻读。
- 把“读已提交”的“读”和当前读搞混：RC 是隔离级别名，与当前读无关。

## 总结

快照读走 MVCC 读历史版本不加锁，普通 `SELECT` 走这条路径；当前读读最新版本并加锁，所有写操作和加锁读走这条路径。RR 下两者看到的数据版本可能不同，混用会导致幻读。生产中扣减类操作必须用当前读保证基于最新数据，列表查询用快照读提升并发。
