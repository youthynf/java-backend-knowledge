# 滥用事务或一个事务特别多 SQL 会有什么问题？

## 核心概念

事务用于保证一组操作的原子性、一致性、隔离性和持久性，但事务不是越大越安全。一个事务执行时间过长或包含大量 SQL，会持有锁、占用 undo/redo 资源、扩大回滚成本，并影响其他事务并发。

长事务常见表现包括锁等待增多、死锁概率上升、undo log 膨胀、MVCC 旧版本无法及时清理、主从延迟、连接占用时间变长，最终拖慢整个系统。

## 面试官想考什么

- 是否理解事务的收益和成本；
- 是否知道长事务对锁、undo、MVCC、复制延迟的影响；
- 是否能提出拆分、批处理、幂等、补偿等方案；
- 是否能区分数据库事务和分布式业务事务。

## 标准回答

> 滥用事务会降低并发和稳定性。事务越长，锁持有时间越长，其他请求更容易等待或死锁；undo log 不能及时清理，会增加存储和一致性读成本；大量 SQL 回滚代价高；一次提交太大还可能导致主从延迟。实践中应让本地事务尽量短，只包含必须强一致的数据库操作，外部 RPC、耗时计算、文件处理不要放在事务里。

## 深挖追问

### 为什么不要在事务里调用远程接口？

远程接口耗时和成功率不可控，会拉长数据库锁持有时间。如果远程调用超时，数据库事务还可能长时间占用连接和锁。更合理的是先完成本地状态变更，再用消息、任务表或 Saga 补偿驱动外部动作。

### 大批量更新怎么处理？

不要一个事务更新几百万行。可以按主键范围或时间窗口分批，每批限制行数，提交后短暂 sleep，配合幂等标记和进度记录，降低锁冲突和主从延迟。

## 实战场景 / 代码示例

```java
@Transactional
public void createOrder(CreateOrderCmd cmd) {
    // 只放必须强一致的本地数据库操作
    orderMapper.insert(cmd.toOrder());
    stockMapper.decrease(cmd.getSkuId(), cmd.getCount());
    outboxMapper.insert(new OutboxEvent("ORDER_CREATED", cmd.getOrderId()));
}

// 事务外异步发送消息或调用外部系统，失败可重试
```

Outbox 表把“本地事务”和“消息发送”解耦，避免在数据库事务中等待 MQ 或 HTTP。

## 易错点 / 总结

- 事务边界越大，并发能力通常越差；
- 不要把 RPC、文件上传、复杂计算放进事务；
- 长事务会阻碍 undo 清理，影响 MVCC；
- 批处理要分批提交，并设计幂等和断点续跑；
- 事务只能解决本地数据库一致性，跨系统还要补偿机制。
