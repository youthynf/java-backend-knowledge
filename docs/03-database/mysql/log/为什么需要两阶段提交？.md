# 为什么需要两阶段提交

## 核心概念

事务提交时 redo log 和 binlog 都要落盘，但两者是独立的日志，由不同层（InnoDB 和 Server）维护。如果不用协议协调，可能出现一个写成功、另一个没写成功的情况，导致主库恢复数据与 binlog 不一致，进而主从数据不一致。

两阶段提交（2PC）就是解决这个一致性问题的协议。它把事务提交拆成 Prepare 和 Commit 两个阶段，redo log 先 prepare，binlog 写入后再 commit。崩溃恢复时根据 redo log 状态和 binlog 是否完整决定提交或回滚，保证 redo log 与 binlog 始终一致。

没有两阶段提交，主库崩溃后可能出现“主库回滚了事务但从库回放了”或“主库提交了事务但从库没收到”的不一致。

## 标准回答

> 两阶段提交是为了保证 redo log 和 binlog 的一致性。事务提交时 InnoDB 先写 redo log 并标记为 prepare 状态，Server 层再写 binlog，最后把 redo log 改为 commit 状态。崩溃恢复时若 redo log 是 prepare 且 binlog 已完整写入，则提交事务（因为 binlog 已可能传给从库）；若 binlog 不完整则回滚。这样保证主库数据与 binlog 一致，从而保证主从一致。代价是每个事务至少两次刷盘（双 1 配置），通过 binlog 组提交优化。

## 实现原理

### 不用两阶段提交会出什么问题

假设先写 redo log 后写 binlog，中间崩溃：

```
时刻 A：redo log 写完，binlog 还没写 → 崩溃
重启后：redo log 重放，主库事务已提交（name='Li'）
但 binlog 没记录这条更新
→ 从库回放 binlog 时没有这条 → 从库 name 仍是旧值 'Wang'
→ 主从数据不一致
```

假设先写 binlog 后写 redo log，中间崩溃：

```
时刻 B：binlog 写完，redo log 还没写 → 崩溃
重启后：redo log 没记录，事务未生效，主库 name='Wang'
但 binlog 已记录这条更新
→ 从库回放 binlog 时执行了这条 → 从库 name='Li'
→ 主从数据不一致
```

两种顺序都可能不一致，所以需要一个协议保证两者原子性。

### 两阶段提交流程

MySQL 内部 XA 事务，binlog 作为协调者，InnoDB 作为参与者：

```
客户端 COMMIT
   ↓
【Prepare 阶段】
   InnoDB 写 redo log（含 XID），状态置为 PREPARE
   ↓
   redo log fsync 刷盘（innodb_flush_log_at_trx_commit=1）
   ↓
【写 binlog 阶段】
   Server 层把 binlog 写入 binlog 文件
   ↓
   binlog fsync 刷盘（sync_binlog=1）
   ↓
【Commit 阶段】
   InnoDB 把 redo log 状态置为 COMMIT
   ↓
   返回客户端成功
```

注意 Commit 阶段只改 redo log 状态，不需要立即刷盘（write 到 page cache 即可），因为 binlog 已落盘，事务就算成功。

### 两阶段提交时序

```
时间 →

【Prepare】            【binlog write】        【Commit】
   |                       |                      |
   ↓                       ↓                      ↓
redo log: PREPARE  →  binlog 落盘  →  redo log: COMMIT
   |                       |                      |
   ↓                       ↓                      ↓
redo fsync            binlog fsync          redo write (不强制 fsync)
                                                ↓
                                          返回客户端成功

崩溃恢复判断点：
  时刻 A 崩溃（prepare 后、binlog 前）→ binlog 不完整 → 回滚
  时刻 B 崩溃（binlog 后、commit 前）→ binlog 完整 → 提交
```

### 崩溃恢复规则

MySQL 重启后扫描 redo log，对每个处于 PREPARE 状态的事务，拿 XID 去 binlog 查找：

| redo log 状态 | binlog 中有该 XID？ | 恢复动作 |
|--------------|-------------------|---------|
| COMMIT | — | 已提交，跳过 |
| PREPARE | 有（完整） | 提交事务 |
| PREPARE | 无（不完整） | 回滚事务 |

为什么 binlog 完整就提交？因为 binlog 一旦完整写入，就可能已经被从库拉取或用于恢复，主库必须认账，否则主从不一致。

### 双 1 配置的性能代价

`innodb_flush_log_at_trx_commit=1` 和 `sync_binlog=1` 合称“双 1”，是金融级强一致配置。代价是每个事务提交至少两次 fsync：

- redo log fsync（prepare 阶段）
- binlog fsync（写 binlog 阶段）

fsync 是磁盘同步写，高并发下成为性能瓶颈。

### binlog 组提交优化

多个事务并发提交时，把它们的 binlog fsync 合并成一次，大幅减少 I/O。组提交把 Commit 阶段拆成三个子阶段：

```
flush 阶段：多个事务按顺序把 binlog 从 cache write 到文件（不 fsync）
   ↓
sync 阶段：对 binlog 文件做一次 fsync（多个事务合并刷盘）
   ↓
commit 阶段：各事务按顺序做 InnoDB commit
```

每个阶段有队列，第一个进队列的事务是 Leader，负责整队操作，其他是 Follower。锁粒度从“整个提交过程”降到“每个阶段队列”，并发提升。

`binlog_group_commit_sync_delay`（微秒）和 `binlog_group_commit_sync_no_delay_count`（事务数）控制组提交行为：等待 N 微秒或积攒 N 个事务再 fsync，平衡延迟和吞吐。

### redo log 组提交（MySQL 5.7+）

5.6 的组提交只优化 binlog，每个事务 prepare 阶段各自刷 redo log。5.7 把 redo log 刷盘推迟到 flush 阶段一起做，实现 redo log 组提交。这样 binlog 和 redo log 都做了组提交优化。

## 代码示例

查看相关配置：

```sql
-- 双 1 参数
SHOW VARIABLES LIKE 'innodb_flush_log_at_trx_commit';
SHOW VARIABLES LIKE 'sync_binlog';

-- 组提交参数
SHOW VARIABLES LIKE 'binlog_group_commit_sync_delay';
SHOW VARIABLES LIKE 'binlog_group_commit_sync_no_delay_count';
```

设置双 1（金融场景）：

```sql
SET GLOBAL innodb_flush_log_at_trx_commit = 1;
SET GLOBAL sync_binlog = 1;
```

设置组提交优化（高并发场景）：

```sql
-- 等待 1ms 让更多事务一起刷盘
SET GLOBAL binlog_group_commit_sync_delay = 1000;
-- 或积攒 10 个事务再刷盘
SET GLOBAL binlog_group_commit_sync_no_delay_count = 10;
```

## 实战场景

| 场景 | 配置 | 取舍 |
|------|------|------|
| 金融强一致 | 双 1 | 不丢数据，性能损耗 |
| 互联网高并发 | sync_binlog=100~1000 | 性能优先，崩溃丢 N 个事务 |
| 高写入吞吐 | 组提交开启 | 牺牲毫秒级延迟换吞吐 |
| 主从强一致 | 半同步复制 + 双 1 | 主库等从库 binlog 确认 |

## 深挖追问

### 1. 为什么 Commit 阶段不需要 fsync？

Commit 阶段只把 redo log 状态从 PREPARE 改为 COMMIT，这个状态变更即使崩溃丢失也没关系。因为崩溃恢复时只要 binlog 完整，PREPARE 状态的事务也会被提交。所以 Commit 阶段只 write 到 page cache 即可，不需要 fsync。

### 2. 两阶段提交能保证主从完全一致吗？

能保证主库恢复后的数据与 binlog 一致，从库按 binlog 回放就能与主库一致。但“保证一致”是逻辑层面，实际主从延迟、从库回放错误（如 STATEMENT 格式动态函数）仍可能短暂不一致。半同步复制能进一步保证 binlog 至少传到一个从库。

### 3. 组提交为什么能提升性能？

fsync 是昂贵的磁盘同步操作。10 个事务各自 fsync 需要 10 次磁盘 I/O；组提交合并后只 1 次 fsync，I/O 减少 10 倍。代价是 Leader 事务要等待 Follower 加入，增加少量延迟（可配置）。

### 4. 事务没提交时 redo log 会写盘吗？

会。事务执行过程中 redo log 持续写入 redo log buffer，后台线程每秒刷一次。所以崩溃时 redo log 可能已有未提交事务的记录，状态是 PREPARE。崩溃恢复时 binlog 不完整就回滚。

### 5. 为什么用 binlog 作为协调者而不是 redo log？

binlog 是事务是否“对外可见”的标志——一旦 binlog 写入，从库就可能拿到这个事务。所以以 binlog 写入成功作为事务真正提交的标志更合理。redo log 是 InnoDB 内部状态，不能跨引擎传播。

## 易错点

- 以为两阶段提交是为了性能：是为了 redo log 与 binlog 一致性，性能反而是代价。
- 以为 Commit 阶段必须 fsync：Commit 阶段不强制 fsync，靠 binlog 完整性兜底。
- 双 1 配置不懂代价：每事务至少两次 fsync，高并发性能瓶颈。
- 以为组提交是默认最优：开启 `binlog_group_commit_sync_delay` 会增加事务延迟，要权衡。
- 把两阶段提交和分布式事务 XA 搞混：这里是 MySQL 内部 XA，单机事务，不是跨服务分布式事务。

## 总结

两阶段提交保证 redo log 与 binlog 的一致性，从而保证主从数据一致。流程是 redo log prepare → binlog write → redo log commit，崩溃恢复时根据 redo log 状态和 binlog 完整性决定提交或回滚。双 1 配置实现强一致但性能有代价，组提交优化能减少 fsync 次数。理解两阶段提交是理解 MySQL 事务提交和崩溃恢复的关键。
