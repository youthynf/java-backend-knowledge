# 数据库三大范式是什么

## 核心概念

数据库范式（Normal Form，NF）是关系型数据库表设计的规范化规则，目的是消除数据冗余，避免插入、更新、删除异常。三大范式是逐步递进的过程：第一范式要求字段原子化，第二范式要求非主属性完全依赖主键，第三范式要求非主属性不传递依赖主键。

范式解决的核心问题是"冗余导致的不一致"。如果一张表里同一个用户的信息出现多次，更新时漏改一处就会数据不一致。范式通过拆表 + 外键引用让每份数据只存一份，从根上消除冗余。

真实工程中不会教条地追求高范式。读多写少、性能要求高、需要历史快照的场景会主动反范式（冗余字段），但这是有意识的工程取舍，必须配套同步策略和一致性校验。

## 标准回答

> 第一范式（1NF）要求字段不可再拆；第二范式（2NF）在 1NF 基础上要求非主属性完全依赖主键，主要针对联合主键；第三范式（3NF）在 2NF 基础上要求非主属性不传递依赖主键，即非主属性之间不能有依赖关系。范式能减少冗余和更新异常，但读多写少或需要历史快照时会适度反范式。

核心要点：

1. **1NF**：字段原子化，不要把多个值塞一个字段。
2. **2NF**：非主属性完全依赖主键，不能只依赖联合主键的一部分。
3. **3NF**：非主属性直接依赖主键，不能通过其他非主属性间接依赖。
4. **目的**：消除冗余，避免插入/更新/删除异常。
5. **反范式**：读多写少、历史快照场景主动冗余字段。

## 详细机制

### 第一范式（1NF）：字段原子性

每个字段必须是不可再分的最小数据单元。违反 1NF 的典型：把多个手机号、多个标签塞一个字段。

```sql
-- 违反 1NF：一个字段存多个手机号
CREATE TABLE bad_user (
  id BIGINT PRIMARY KEY,
  name VARCHAR(64),
  phones VARCHAR(200)  -- "13800000000,13900000000"
);
```

查询"谁的号是 13900000000"只能用 `LIKE '%13900000000%'`，索引失效、易误匹配。改造：

```sql
-- 满足 1NF：手机号拆到副表
CREATE TABLE `user` (
  id BIGINT PRIMARY KEY,
  name VARCHAR(64) NOT NULL
);

CREATE TABLE `user_phone` (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  phone VARCHAR(20) NOT NULL,
  type TINYINT NOT NULL COMMENT '1家用 2办公 3手机',
  UNIQUE KEY `uk_user_phone` (`user_id`, `phone`),
  KEY `idx_phone` (`phone`)
);
```

### 第二范式（2NF）：完全依赖主键

在 1NF 基础上，所有非主属性必须完全依赖主键，不能只依赖联合主键的一部分。这条规则主要针对联合主键。

```sql
-- 违反 2NF：订单明细表，主键是 (order_id, sku_id)
CREATE TABLE bad_order_item (
  order_id BIGINT,
  sku_id BIGINT,
  sku_name VARCHAR(128),   -- 只依赖 sku_id，不依赖 order_id
  buy_price DECIMAL(10,2),
  quantity INT,
  PRIMARY KEY (order_id, sku_id)
);
```

`sku_name` 只依赖 `sku_id`，会导致：商品改名要改所有订单明细；插入新商品必须先有订单。改造：

```sql
-- 满足 2NF：商品信息单独表
CREATE TABLE `sku` (
  id BIGINT PRIMARY KEY,
  name VARCHAR(128) NOT NULL,
  current_price DECIMAL(10,2) NOT NULL
);

CREATE TABLE `order_item` (
  id BIGINT PRIMARY KEY,
  order_id BIGINT NOT NULL,
  sku_id BIGINT NOT NULL,
  buy_price DECIMAL(10,2) NOT NULL,  -- 历史快照，不算冗余
  quantity INT NOT NULL,
  KEY `idx_order` (`order_id`),
  KEY `idx_sku` (`sku_id`)
);
```

注意 `buy_price` 保留在 `order_item` 不违反 2NF，因为它是订单发生时的快照，依赖整个联合键。

### 第三范式（3NF）：消除传递依赖

在 2NF 基础上，非主属性不能通过其他非主属性间接依赖主键。

```sql
-- 违反 3NF：学生表里有 dept_id 和 dept_name
CREATE TABLE bad_student (
  id BIGINT PRIMARY KEY,
  name VARCHAR(64),
  dept_id BIGINT,
  dept_name VARCHAR(128)  -- 依赖 dept_id，间接依赖 student.id
);
```

`dept_name` 依赖 `dept_id`，`dept_id` 依赖 `id`，所以 `dept_name` 传递依赖主键。改造：

```sql
-- 满足 3NF：部门单独表
CREATE TABLE `department` (
  id BIGINT PRIMARY KEY,
  name VARCHAR(128) NOT NULL
);

CREATE TABLE `student` (
  id BIGINT PRIMARY KEY,
  name VARCHAR(64) NOT NULL,
  dept_id BIGINT NOT NULL,
  KEY `idx_dept` (`dept_id`)
);
```

### 反范式（Denormalization）

读多写少的场景里，过度范式化会导致大量 JOIN，查询性能差。这时主动冗余字段，但必须有同步机制：

- 触发器同步：主表更新时触发器更新冗余字段。
- 应用层双写：业务代码同步写两处。
- 异步同步：监听 binlog，通过 Canal 等工具同步。

订单明细保留 `sku_name`、`buy_price` 是反范式的典型例子，目的是历史快照——商品改名不影响历史订单显示。

## 代码示例

电商订单系统综合示范三大范式：

```sql
-- 1NF：手机号拆到副表
CREATE TABLE `user` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `name` VARCHAR(64) NOT NULL,
  `email` VARCHAR(128) NOT NULL,
  UNIQUE KEY `uk_email` (`email`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE `user_phone` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `user_id` BIGINT UNSIGNED NOT NULL,
  `phone` VARCHAR(20) NOT NULL,
  `type` TINYINT NOT NULL,
  UNIQUE KEY `uk_user_phone` (`user_id`, `phone`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 2NF：商品独立表，订单明细只放订单相关字段 + 历史快照
CREATE TABLE `sku` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `name` VARCHAR(128) NOT NULL,
  `current_price` DECIMAL(10,2) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE `orders` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `order_no` VARCHAR(32) NOT NULL,
  `user_id` BIGINT UNSIGNED NOT NULL,
  `amount` DECIMAL(10,2) NOT NULL,
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_user_time` (`user_id`, `created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE `order_item` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `order_id` BIGINT UNSIGNED NOT NULL,
  `sku_id` BIGINT UNSIGNED NOT NULL,
  `sku_name` VARCHAR(128) NOT NULL COMMENT '历史快照',
  `buy_price` DECIMAL(10,2) NOT NULL COMMENT '历史快照',
  `quantity` INT NOT NULL,
  KEY `idx_order` (`order_id`),
  KEY `idx_sku` (`sku_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 实战场景

| 场景 | 应用 | 说明 |
|------|------|------|
| 用户多手机号 | 1NF | 拆 `user_phone` 副表 |
| 商品规格独立维护 | 2NF | 商品表与订单明细分开 |
| 部门信息复用 | 3NF | 部门单独表，学生/员工只存 `dept_id` |
| 订单保留商品名快照 | 反范式 | 商品改名不影响历史订单 |
| 用户列表带部门名 | 反范式 | 在用户表冗余 `dept_name` + 触发器同步 |
| 朋友圈点赞数 | 反范式 | 单独 `like_count` 字段，避免每次 `COUNT(*)` |

## 深挖追问

### BCNF 比第三范式更严在哪？

BCNF（Boyce-Codd Normal Form）要求每个决定因素都必须是候选键。3NF 允许非主属性被非键属性决定（前提是被决定的属性是主属性的子集），BCNF 不允许。简单说，BCNF 是 3NF 的强化版，处理了主属性之间的依赖。工程上很少刻意追求 BCNF。

### 反范式和缓存有什么区别？

反范式是数据库内的冗余字段，缓存（如 Redis）是数据库外的冗余数据。反范式数据与主数据在同一事务里，一致性较强；缓存需要业务层维护一致性，更易漂移。反范式适合读多写少且字段稳定的场景，缓存适合读多写少且能容忍秒级不一致的场景。

### 反范式怎么保证一致性？

- 同事务双写：业务事务里同时更新主表和冗余字段。
- 触发器：DBA 维护，对应用透明但难调试。
- 异步同步：通过 binlog 监听（Canal、Debezium），最终一致。
- 定时校验：定时任务对账，发现不一致后修复。

### 范式越高越好吗？

不是。高范式减少冗余但增加 JOIN，读性能下降。实际业务表设计通常到 3NF 即可，再往上的 BCNF/4NF/5NF 在 OLTP 几乎用不到。OLAP（分析型）反而常用星型模型/雪花模型，主动反范式以加快聚合。

### 订单明细的 buy_price 算冗余吗？

不算。`buy_price` 是订单发生时的历史快照，依赖 `(order_id, sku_id)` 联合键，不违反任何范式。如果只保留 `sku_id` 然后通过关联 `sku.current_price` 取值，反而违反业务需求——商品调价后历史订单金额会错。

## 易错点

- 把"字段拆细"理解成 1NF 的全部，忽略"不可再分"是相对业务而言。
- 第二范式只看联合主键，单主键表不会违反 2NF。
- 第三范式误认为"不能有任何冗余"，实际上历史快照不算冗余。
- 反范式后不做同步，导致主数据与冗余字段不一致。
- 把范式当成教条，所有表都拆到 3NF，导致复杂 JOIN 拖垮查询。
- 把"部门名"这类常变字段冗余在用户表，每次部门改名要扫全表更新。

## 总结

三大范式是关系型表设计的去冗余规则：1NF 字段原子化、2NF 完全依赖主键、3NF 消除传递依赖。范式能消除插入/更新/删除异常，但读多写少、历史快照场景会主动反范式。反范式必须有同步机制（同事务双写、触发器、binlog 同步）保证一致性。订单明细保留商品名和价格是合法的历史快照，不算违反范式。
