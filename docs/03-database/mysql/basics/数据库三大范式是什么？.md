# 数据库三大范式是什么？

## 核心概念

数据库范式是关系型表设计规范，用来减少数据冗余和插入、删除、更新异常。三大范式通常指：第一范式要求字段原子化；第二范式要求非主属性完全依赖主键，不能只依赖联合主键的一部分；第三范式要求非主属性不能传递依赖主键。

范式强调一致性和可维护性，但真实业务中也会为性能、查询便利、历史快照做适度反范式。

## 面试官想考什么

- 是否能用例子解释三大范式；
- 是否理解范式解决冗余和更新异常；
- 是否知道反范式是有意识的工程取舍；
- 是否能结合订单、用户、商品表设计回答。

## 标准回答

> 第一范式要求字段不可再拆，例如不要把多个手机号放在一个字段；第二范式要求非主属性完全依赖主键，主要针对联合主键；第三范式要求非主属性只依赖主键，不能依赖其他非主属性。遵守范式可以减少冗余和不一致，但高频查询或历史快照场景会适度反范式，例如订单明细保存下单时的商品名和价格。

## 深挖追问

### 什么是传递依赖？

学生表里有 `student_id、dept_id、dept_name`。`student_id` 决定 `dept_id`，`dept_id` 决定 `dept_name`，所以 `dept_name` 对学生是传递依赖。更合理的是学生表保存 `dept_id`，部门表保存 `dept_name`。

### 什么时候可以反范式？

读多写少、跨表 Join 成本高、需要保留历史状态时可以冗余字段，但必须明确同步策略、修复脚本和一致性校验。

## 实战场景 / SQL 示例

```sql
CREATE TABLE user_phone (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  phone VARCHAR(20) NOT NULL,
  type TINYINT NOT NULL,
  UNIQUE KEY uk_user_phone(user_id, phone)
);

CREATE TABLE order_item (
  id BIGINT PRIMARY KEY,
  order_id BIGINT NOT NULL,
  sku_id BIGINT NOT NULL,
  sku_name VARCHAR(128) NOT NULL,
  buy_price DECIMAL(10,2) NOT NULL
);
```

`user_phone` 体现字段原子化；`order_item` 中 `sku_name`、`buy_price` 是为了历史快照而冗余。

## 易错点 / 总结

- 第二范式重点看联合主键；
- 第三范式重点看非主属性之间的依赖；
- 范式不是越高越好，过度范式化会带来复杂 Join；
- 反范式不是随便冗余，要有同步和校验方案；
- 回答时要落到业务表设计，不要只背概念。

## 面试版详细讲解

### 核心概念

这道题属于 **MySQL 基础** 的高频考点，核心要抓住：SQL 语义、数据类型、执行流程、表设计。先保证语义正确，再结合数据量、索引、排序/分组、临时表和网络传输评估成本。类型选择要说明范围、精度、字符集、默认值和扩展性。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：定义、执行顺序、类型边界、索引影响、数据规模变化；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

先保证语义正确，再结合数据量、索引、排序/分组、临时表和网络传输评估成本。类型选择要说明范围、精度、字符集、默认值和扩展性。

答题时建议用“三段式”：

1. 先给结论，明确适用前提；
2. 再解释底层机制或执行过程；
3. 最后补充业务取舍、风险点和排查手段。

### 深挖追问

- 这个结论在高并发或大数据量下是否仍然成立？
- 它依赖哪些版本、配置、索引/编码或业务一致性要求？
- 线上异常时应该看哪些命令、日志、指标或执行计划？

### 示例 / 实战场景

上线前用 EXPLAIN 验证访问类型、扫描行数、key、Extra，重点关注 Using temporary、Using filesort、全表扫描和隐式转换。

```sql
EXPLAIN SELECT * FROM your_table WHERE biz_id = ? ORDER BY created_at DESC LIMIT 20;
```

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
