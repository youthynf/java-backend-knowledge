# 使用联合索引需要注意什么

## 核心概念

联合索引是把多个字段组合成一棵 B+ 树，键是 `(字段1, 字段2, ..., 字段N)` 的组合，按字段顺序字典序排列。它的核心特性是**最左前缀原则**：从最左列开始连续匹配才能用上索引。一旦遇到不连续或范围查询，后续列就用不上索引了。

理解联合索引的关键是"B+ 树按联合键字典序排序"。比如索引 `(a, b, c)`：先按 a 排序，a 相同再按 b 排序，b 相同再按 c 排序。所以 a 是有序的、a 相同时 b 有序、a 和 b 都相同时 c 才有序。如果跳过 a 直接查 b，整个 B+ 树的 b 字段是无序的，索引失效。

## 标准回答

联合索引按字段顺序在 B+ 树中字典序排列，遵循最左前缀原则：从最左列开始连续匹配才能用上索引。`WHERE a=? AND b=?` 可用 `(a,b,c)` 的 a 和 b；`WHERE b=?` 不能用（跳过了 a）。范围查询（`>`、`<`）会停止后续列匹配，但 `>=`、`<=`、`BETWEEN`、`LIKE 'x%'` 不停止。5.6+ 引入索引下推（ICP），在二级索引遍历时直接对索引列过滤，减少回表次数。设计时把区分度高、等值条件的列放左边，范围条件列放右边。

## 最左前缀原则

### 索引 `(a, b, c)` 的可用情况

| 查询条件 | 是否走索引 | 用到的列 |
|----------|-----------|----------|
| `WHERE a=?` | 是 | a |
| `WHERE a=? AND b=?` | 是 | a, b |
| `WHERE a=? AND b=? AND c=?` | 是 | a, b, c |
| `WHERE b=?` | 否 | 无 |
| `WHERE b=? AND c=?` | 否 | 无 |
| `WHERE a=? AND c=?` | 部分 | a（c 用不上，因为跳过 b） |
| `WHERE a=? AND b>? AND c=?` | 部分 | a, b（c 用不上，因为 b 是范围） |

### 为什么会这样

B+ 树按 `(a, b, c)` 字典序排列：

```
(1, 1, 1), (1, 1, 2), (1, 2, 1), (1, 2, 3), (2, 1, 1), (2, 1, 3), (2, 3, 1)...
```

- `a` 全局有序。
- `a` 相同时 `b` 有序，但跨 `a` 值时 `b` 无序。
- `a, b` 都相同时 `c` 有序，否则无序。

所以跳过 `a` 直接查 `b`，索引里 `b` 是无序的，无法用 B+ 树的二分定位。

## 范围查询的细节

联合索引遇到**范围查询会停止后续列匹配**，但范围查询的形式不同，行为也不同：

| 范围形式 | 是否停止后续列匹配 | 说明 |
|----------|-------------------|------|
| `>`, `<` | 停止 | 范围内的后续列无序 |
| `>=`, `<=` | **不停止** | 等值点后续列仍有序 |
| `BETWEEN` | **不停止** | 等价于 `>= AND <=` |
| `LIKE 'x%'` | **不停止** | 前缀匹配，等价于范围 |

### 例子：索引 `(a, b)`

```sql
-- case 1: a > 1 AND b = 2
-- a > 1 范围内，b 是无序的（a=2 时 b 可能是 1 也可能是 3）
-- 只有 a 用到索引，b 用不上
EXPLAIN SELECT * FROM t WHERE a > 1 AND b = 2;

-- case 2: a >= 1 AND b = 2
-- a=1 的等值点 b 有序；a>1 的范围 b 无序
-- a 和 b 都能用上索引（在 a=1 的范围内可以过滤 b）
EXPLAIN SELECT * FROM t WHERE a >= 1 AND b = 2;

-- case 3: a BETWEEN 1 AND 5 AND b = 2
-- 等价于 a>=1 AND a<=5，行为同 case 2
EXPLAIN SELECT * FROM t WHERE a BETWEEN 1 AND 5 AND b = 2;

-- case 4: a LIKE 'j%' AND b = 22
-- 前缀匹配，a 在 [j, k) 范围内
-- a='j' 的等值点 b 有序；其他点 b 无序
-- a 和 b 都能用上索引
EXPLAIN SELECT * FROM t WHERE a LIKE 'j%' AND b = 22;
```

### key_len 验证

通过 `EXPLAIN` 的 `key_len` 字段可以验证联合索引用到了几列：

```sql
-- 索引 (a INT, b INT, c INT)，假设 INT 占 4 字节 + 可空标记 1 字节
EXPLAIN SELECT * FROM t WHERE a = 1;
-- key_len = 5  （只用 a）

EXPLAIN SELECT * FROM t WHERE a = 1 AND b = 2;
-- key_len = 10 （用到 a, b）

EXPLAIN SELECT * FROM t WHERE a > 1 AND b = 2;
-- key_len = 5  （只用 a，b 被范围阻断）
```

## 索引下推（Index Condition Pushdown，ICP）

### 没有 ICP 时（5.6 之前）

```sql
-- 索引 (a, b)
SELECT * FROM t WHERE a > 1 AND b = 2;
```

执行流程：
1. 在二级索引 B+ 树定位到 `a > 1` 的起始位置。
2. 对每条 `a > 1` 的索引记录，**回表**到聚簇索引取整行数据。
3. 在 Server 层用 `b = 2` 过滤回表后的数据。

问题：即使 `b != 2`，也要先回表再过滤，回表次数 = `a > 1` 的记录数。

### 有 ICP 时（5.6+）

执行流程：
1. 在二级索引 B+ 树定位到 `a > 1` 的起始位置。
2. 对每条 `a > 1` 的索引记录，**在二级索引层**直接判断 `b = 2` 是否成立。
3. 只有 `b = 2` 的记录才回表，其他直接跳过。

ICP 把 `b = 2` 的过滤下推到存储引擎层，大幅减少回表次数。`EXPLAIN` 的 `Extra` 字段会显示 `Using index condition`。

## 索引区分度与列顺序

联合索引列顺序的设计原则：

1. **区分度高的放左边**：让最左前缀过滤效果最大化，扫描行数最少。
2. **等值条件放左边，范围条件放右边**：范围条件后续列用不上索引。
3. **高频查询的列放左边**：让更多 SQL 能复用这个联合索引。
4. **ORDER BY 列放末尾**：利用索引有序性避免 filesort。

```sql
-- 区分度计算
SELECT
  COUNT(DISTINCT status) / COUNT(*) AS status_sel,
  COUNT(DISTINCT user_id) / COUNT(*) AS user_id_sel,
  COUNT(DISTINCT created_at) / COUNT(*) AS created_at_sel
FROM orders;
-- 假设 user_id_sel > created_at_sel > status_sel

-- 查询: WHERE user_id = ? AND status = 1 ORDER BY created_at DESC
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
-- user_id 区分度高放左，status 等值放中间，created_at 排序放末尾
```

## 联合索引与排序

利用联合索引的有序性可以避免 filesort：

```sql
-- 索引 (status, created_at)
SELECT * FROM orders WHERE status = 1 ORDER BY created_at DESC;
-- status=1 的记录在索引里已按 created_at 有序
-- Extra: Using where，无 Using filesort

-- 反例：只有 (status) 索引
SELECT * FROM orders WHERE status = 1 ORDER BY created_at DESC;
-- Extra: Using filesort  ← 需要额外排序
```

注意：`ORDER BY` 字段顺序必须与联合索引顺序一致，且方向（ASC/DESC）也要一致（8.0 支持降序索引前可不一致，但有性能损耗）。

## 代码示例

```sql
-- 建联合索引
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);

-- 验证执行计划
EXPLAIN SELECT * FROM orders
WHERE user_id = 1001 AND status = 1
ORDER BY created_at DESC LIMIT 20;
-- type: ref
-- key: idx_user_status_created
-- key_len: 16           ← 用到全部 3 列
-- Extra: Using index condition  ← 使用了 ICP（如果还有其他条件）

-- 强制走某个索引
EXPLAIN SELECT * FROM orders USE INDEX (idx_user_status_created)
WHERE user_id = 1001;

-- 查看索引使用统计（8.0）
SELECT * FROM sys.schema_index_statistics
WHERE table_schema = 'mydb' AND table_name = 'orders';
```

## 实战场景

| 场景 | 索引设计 | 注意点 |
|------|----------|--------|
| 订单列表 `WHERE user_id=? AND status=? ORDER BY created_at` | `(user_id, status, created_at)` | 区分度 user_id > status，排序放末尾 |
| 用户搜索 `WHERE name LIKE '张%' AND age=25` | `(name, age)` | 前缀匹配不阻断 age，ICP 进一步过滤 |
| 报表 `WHERE created_at > ? AND status=? GROUP BY dept_id` | `(status, created_at, dept_id)` | status 等值放左，created_at 范围放中，dept_id 分组放右 |
| 高频点查 `WHERE a=? AND b=? AND c=?` | `(a, b, c)`，区分度高放左 | 三个等值都走索引 |
| 范围 + 排序 `WHERE a=? AND b>? ORDER BY c` | `(a, b, c)` 或 `(a, c, b)` | 看 b 范围多大；b 范围小则 `(a, b, c)`，b 范围大则 `(a, c, b)` 但 c 难走索引 |

## 深挖追问

### 为什么 `>=` 不停止后续列匹配而 `>` 停止？

`a > 1` 范围内 a 取 2, 3, 4..., 此时 b 在每个 a 值内有序，但跨 a 值时 b 无序。所以无法用 b 做整体二分。`a >= 1` 中 a=1 是等值点，b 在 a=1 内有序，可以利用 b 过滤；a>1 部分仍按 `>` 处理。优化器在 `>=` 时会"等值部分用 b，范围部分不用 b"。

### 索引下推什么时候触发？

当查询条件中包含联合索引的列，但有些列因为最左前缀或范围查询无法用于"索引定位"时，仍可用于"索引过滤"。ICP 把这部分过滤下推到存储引擎层，减少回表。触发条件：

1. 是二级索引（聚簇索引不需要 ICP，本来就有整行数据）。
2. 联合索引的某些列无法用于定位但可用于过滤。
3. `EXPLAIN Extra: Using index condition`。

### 联合索引列顺序错了能改吗？

`ALTER TABLE ... CHANGE INDEX` 或 `DROP INDEX + CREATE INDEX` 可以重建联合索引。但大表重建会锁表很久，建议用 `pt-online-schema-change` 或 8.0 的在线 DDL。

### 联合索引能完全替代多个单列索引吗？

不完全能。联合索引能覆盖"按指定顺序使用列"的查询；但 `WHERE b=?` 这种跳过 a 的查询联合索引帮不上。如果业务上确实有不同顺序的查询，可能需要建多个联合索引（注意权衡写入和空间成本）。

## 易错点

- 误以为"联合索引所有列都能任意组合用"——必须从最左列开始连续匹配。
- 误以为"`>=` 也是范围查询会停止后续列"——`>=`、`<=`、`BETWEEN`、`LIKE 'x%'` 不停止。
- 误以为"ORDER BY 字段独立于联合索引"——ORDER BY 字段加入联合索引末尾可避免 filesort。
- 误以为"索引下推是自动回表"——ICP 是减少回表，不是消除回表。
- 误以为"联合索引列顺序无所谓"——顺序错了，索引可能完全用不上。

## 总结

联合索引按字段顺序在 B+ 树中字典序排列，遵循最左前缀原则：从最左列开始连续匹配。`>`、`<` 会停止后续列匹配，`>=`、`<=`、`BETWEEN`、`LIKE 'x%'` 不停止。5.6+ 的索引下推（ICP）把无法用于定位但能用于过滤的条件下推到存储引擎层，减少回表。设计联合索引时：区分度高放左、等值放左、范围放右、ORDER BY 放末尾。用 `EXPLAIN` 的 `key_len` 和 `Extra` 验证索引使用情况。
