# 使用联合索引需要注意什么？

## 核心概念

使用联合索引需要注意什么？
联合索引基本概念
通过将多个字段组合成一个索引，该索引就被称为联合索引。联合索引的非叶子节点用多个字段的值作为 B+Tree 的 key 值。当在联合索引查询数据时，按照联合索引定义的顺序，从左往右进行比较，前者相同再往下一个比较。因此，使用联合索引时，存在最左匹配原则，也就是按照最左优先的方式进行索引的匹配。在使用联合索引进行查询的时候，如果不遵循「最左匹配原则」，联合索引会失效，这样就无法利用到索引快速查询的特性了。

联合索引范围查询
联合索引有一些特殊情况，并不是查询过程使用了联合索引查询，就代表联合索引中的【所有字段】都用到了联合索引进行索引查询，也就是可能存在部分字段用到联合索引的 B+Tree，部分字段没有用到联合索引的 B+Tree 的情况。这种特殊情况就发生在范围查询。联合索引的最左匹配原则会一直向右匹配直到遇到「范围查询」就会停止匹配。也就是范围查询的字段可以用到联合索引，但是在范围查询字段的后面的字段无法用到联合索引。
以联合索引（a,b）为例：
case1：select * from t_table where a > 1 and b = 2。这种在符合 a > 1 条件的二级索引记录的范围里，b 字段的值是无序的，因此，这条查询语句只有 a 字段用到了联合索引进行索引查询，而 b 字段并没有使用到联合索引。
case2：select * from t_table where a >= 1 and b = 2。这种差别在于范围查询中，a=1这个条件中，b字段的值是有序的，此时b的索引可以生效，因此，这条查询语句 a ，b字段都可以用到了联合索引进行索引查询。
case3：select * from t_table where a between 2 and 8 and b = 2。在 MySQL 中，between 包含了 value1 和 value2 边界值，类似于 >= and =<。有的数据库则不包含 value1 和 value2 边界值。类似case2，这条查询语句 a 和 b 字段都用到了联合索引进行索引查询。
case4：select * from t_user where a like 'j%' and b= 22。虽然在符合前缀为 ‘j’ 的 name 字段的二级索引记录的范围里，age 字段的值是「无序」的，但是对于符合 name = j 的二级索引记录的范围里，age字段的值是「有序」的。因此，Q4 这条查询语句 a 和 b 字段都用到了联合索引进行索引查询。
综上所示，联合索引的最左匹配原则，在遇到范围查询（如 >、<）的时候，就会停止匹配，也就是范围查询的字段可以用到联合索引，但是在范围查询字段的后面的字段无法用到联合索引。注意，对于 >=、<=、BETWEEN、like 前缀匹配的范围查询，并不会停止匹配。
可以通过explain执行计划中的key_len可以知道优化器具体使用了多少个字段的搜索条件来行程扫描区间的边界条件，取值表示实际用于联合索引的字节大小总和，跟字段的定义有关。
索引下推
对于联合索引（a, b），在执行 select * from table where a > 1 and b = 2 语句的时候，只有 a 字段能用到索引，那在联合索引的 B+Tree 找到第一个满足条件的主键值（ID 为 2）后，还需要判断其他条件是否满足（看 b 是否等于 2），那是在联合索引里判断？还是回主键索引去判断呢？
在 MySQL 5.6 之前，只能从 ID2 （主键值）开始一个个回表，到「主键索引」上找出数据行，再对比 b 字段值。
MySQL 5.6 引入的索引下推优化（index condition pushdown)， 可以在联合索引遍历过程中，对联合索引中包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数。
当你的查询语句的执行计划里，出现了 Extra 为 Using index condition，那么说明使用了索引下推的优化。

索引区分度
建立联合索引时的字段顺序，对索引效率也有很大影响。越靠前的字段被用于索引过滤的概率越高，实际开发工作中建立联合索引时，要把区分度大的字段排在前面，这样区分度大的字段越有可能被更多的 SQL 使用到。区分度就是某个字段 column 不同值的个数「除以」表的总行数：

区分度 = disctinct(column) / count(*)

联合索引进行排序
通过建立一个联合索引，可以避免 MySQL 数据库发生文件排序。如：

select * from order where status = 1 order by create_time asc
在不建立联合索引的情况下，在查询时，如果只用到 status 的索引，但是这条语句还要对 create_time 排序，这时就要用文件排序 filesort，也就是在 SQL 执行计划中，Extra 列会出现 Using filesort。所以，要利用索引的有序性，在 status 和 create_time 列建立联合索引，这样根据 status 筛选后的数据就是按照 create_time 排好序的，避免在文件排序，提高了查询效率。

## 面试官想考什么

- 索引能否减少扫描行数、排序、临时表和回表成本。
- B+ 树、聚簇索引、二级索引、联合索引、最左前缀是否理解。
- 能否通过 EXPLAIN 判断命中索引、回表、索引失效。

## 标准回答

MySQL 索引本质是用有序数据结构降低查询扫描成本。InnoDB 常见索引以 B+ 树组织，主键索引叶子节点存整行数据，二级索引叶子节点存主键值；查询不能覆盖索引时通常需要回表。索引设计要围绕高频查询、选择性、排序/分组、范围条件和写入成本综合权衡。

## 深挖追问

1. 为什么联合索引遵循最左前缀？因为 B+ 树按联合键字典序排序。
2. 覆盖索引一定最快吗？不一定，还要看扫描行数、过滤性和维护成本。
3. 索引越多越好吗？不是，写入和更新都要维护索引。

## 实战场景 / SQL 示例

```sql
EXPLAIN SELECT id, name FROM user WHERE status = 1 AND created_at >= ? ORDER BY created_at DESC LIMIT 20;
-- 可评估联合索引：(status, created_at)，若需要避免回表再考虑覆盖字段。
```

## 易错点 / 总结

- 不要认为建了索引就一定会走索引；优化器会按成本选择。
- 联合索引遇到范围条件后，后续列通常难以继续用于精确定位。
- 低选择性字段、频繁更新字段要谨慎建索引。

## 面试版详细讲解

### 核心概念

这道题属于 **MySQL 索引** 的高频考点，核心要抓住：B+ 树、聚簇索引、二级索引、联合索引、覆盖索引。索引用空间和写入维护成本换查询效率。联合索引应结合等值、范围、排序/分组和覆盖查询设计，不是越多越好。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：B+ 树定位过程、最左前缀、回表、索引下推、索引失效和 EXPLAIN；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

索引用空间和写入维护成本换查询效率。联合索引应结合等值、范围、排序/分组和覆盖查询设计，不是越多越好。

答题时建议用“三段式”：

1. 先给结论，明确适用前提；
2. 再解释底层机制或执行过程；
3. 最后补充业务取舍、风险点和排查手段。

### 深挖追问

- 这个结论在高并发或大数据量下是否仍然成立？
- 它依赖哪些版本、配置、索引/编码或业务一致性要求？
- 线上异常时应该看哪些命令、日志、指标或执行计划？

### 示例 / 实战场景

慢 SQL 先看 where/order by/group by/返回列，再用 EXPLAIN 看 type、key、rows、Extra，必要时用覆盖索引或调整列顺序。

```sql
EXPLAIN SELECT * FROM your_table WHERE biz_id = ? ORDER BY created_at DESC LIMIT 20;
-- 关注 type/key/rows/Extra，确认是否命中合适索引
```

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
