# InnoDB 的 B+ 树是如何查询数据的

## 核心概念

InnoDB 的 B+ 树查询本质是"层层二分下行 + 叶子链表扫描"。从根节点开始，每一层在页内通过 Page Directory 的槽做二分查找定位下一层子页，到达叶子节点后再在页内二分定位到具体记录。等值查询走到叶子节点即结束；范围查询定位到起始叶子后顺着双向链表扫描。

整个过程的关键是：每一层都是"页内二分"，而不是"页内线性扫描"。Page Directory（槽目录）是这个二分查找的基础。理解了它，就理解了 B+ 树查询为什么是 O(log N)。

## 标准回答

InnoDB B+ 树查询从根节点开始，每层在页内通过 Page Directory 的槽做二分查找定位子页号，下行到叶子节点；叶子页内同样通过槽二分定位记录所在的分组，再在分组内线性查找具体记录。等值查询走到叶子即结束；范围查询定位到起始叶子后顺着双向链表向右扫描，直到不满足条件。3 层 B+ 树只需 3 次磁盘 IO。

## 等值查询流程

以 `SELECT * FROM user WHERE id = 25` 为例，假设 B+ 树高 3 层，根节点常驻内存：

```
[根节点页 page=10, 常驻 Buffer Pool]
  Page Directory 槽: [10, 20, 30, 40]   ← 每个槽对应一组记录的最大值
  Step 1: 在槽上二分查找 "25"
          → 25 落在 (20, 30] 这组，下一层走 page=20 对应的子页

[非叶节点页 page=20, 从磁盘读入]
  Page Directory 槽: [21, 22, 25, 28]
  Step 2: 在槽上二分查找 "25"
          → 25 落在 (22, 25] 这组，下一层走 page=25 对应的子页

[叶子节点页 page=25, 从磁盘读入]
  Page Directory 槽: [23, 24, 25, 26]
  Step 3: 在槽上二分查找 "25"
          → 25 落在 (24, 25] 这组（slot 指向记录 25）
  Step 4: 在该组（最多 8 条记录）内线性扫描，找到 id=25 的记录
  Step 5: 返回整行数据
```

3 层 B+ 树最多 3 次 IO（根节点常驻内存时只 2 次）。

## 范围查询流程

以 `SELECT * FROM user WHERE id BETWEEN 25 AND 60` 为例：

```
Step 1~3: 同等值查询，定位到 id=25 在叶子页 page=25
Step 4: 在 page=25 内从 id=25 开始向右扫描
        → 25, 26, 28, 30
        → 到 page=25 末尾，顺着 FIL_PAGE_NEXT 进入 page=26
Step 5: 在 page=26 继续向右扫描
        → 32, 35, 40, 50, 60
        → id=60 仍满足条件，继续
Step 6: id=65 不满足，停止扫描
```

关键点：范围查询不需要回到根节点重走 B+ 树。叶子节点之间的双向链表让"向右扫描"成为单次顺序 IO，性能远高于"对每个值重新走一遍 B+ 树"。

## 二级索引查询流程

`SELECT * FROM user WHERE name = 'bob'`，假设有二级索引 `idx_name`：

```
Step 1: 在 idx_name 的 B+ 树走"等值查询流程"，定位到 name='bob' 的叶子记录
        → 叶子节点返回 (name='bob', id=42)   ← 二级索引叶子存主键值

Step 2: 拿着 id=42 回到聚簇索引的 B+ 树
        → 走"等值查询流程"找到 id=42 的整行数据   ← 这一步叫"回表"

Step 3: 返回整行数据
```

回表是两次 B+ 树查找：二级索引 B+ 树 + 聚簇索引 B+ 树。如果查询字段都在二级索引里（覆盖索引），Step 2 省略。

## Page Directory 与二分查找

页内记录按主键顺序组成单向链表，从头遍历是 O(N)。为了支持二分查找，InnoDB 把页内记录分组：

- 每组 4~8 条记录（第一组 1 条 Infimum，最后一组 1~8 条）。
- 每组最后一条记录的偏移量存到 Page Directory 的一个槽（slot）里。
- 槽按记录大小有序排列。

二分查找过程：

```
查找 key=25:
  槽: [slot0→10, slot1→20, slot2→30, slot3→40]
  Step 1: 比较中间槽 slot1=20，25 > 20 → 右半边
  Step 2: 比较中间槽 slot2=30，25 ≤ 30 → 25 在 (slot1, slot2] 这组
  Step 3: 该组对应记录范围 (20, 30]，线性扫描组内 4~8 条找到 25
```

页内二分把 O(N) 降为 O(log N)，再叠加组的"4~8 条线性扫描"开销极小。

## 代码示例

观察 B+ 树查询行为：

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM user WHERE id = 25;
-- type: const          ← 主键等值查询，最优
-- rows: 1              ← 预估扫描 1 行

EXPLAIN SELECT * FROM user WHERE id BETWEEN 25 AND 60;
-- type: range          ← 范围查询
-- rows: 估算行数

EXPLAIN SELECT * FROM user WHERE name = 'bob';
-- type: ref            ← 二级索引等值
-- Extra: NULL          ← 需要回表

EXPLAIN SELECT id, name FROM user WHERE name = 'bob';
-- type: ref
-- Extra: Using index   ← 覆盖索引，不回表
```

查看 B+ 树高度（8.0）：

```sql
SELECT stat_value AS tree_height
FROM mysql.innodb_index_stats
WHERE database_name = 'mydb'
  AND table_name = 'user'
  AND index_name = 'PRIMARY'
  AND stat_name = 'tree_height';
```

## 实战场景

| 场景 | B+ 树行为 | 优化要点 |
|------|-----------|----------|
| 主键点查 | 3 层 = 3 次 IO（根常驻只 2 次） | 主键设计短小，根节点常驻 Buffer Pool |
| 主键范围查 | 起始定位 + 叶子链表扫描 | 主键自增，新数据集中在尾部，热页少 |
| 二级索引等值 + 回表 | 两次 B+ 树查找 | 高频查询考虑覆盖索引 |
| 联合索引最左前缀 | 按联合键字典序定位 | 把高频等值列放左边，范围列放右边 |
| 深度分页 `LIMIT 100000, 10` | 扫描前 100000 行 + 回表 100000 次 | 用游标分页 `WHERE id > last_id` |

## 深挖追问

### 为什么根节点常驻 Buffer Pool？

根节点是 B+ 树的入口，每次查询都要走它。InnoDB 把根节点固定在 Buffer Pool 中不淘汰，避免每次查询都从磁盘读根节点。所以 3 层 B+ 树实际只产生 2 次磁盘 IO（中间层 + 叶子层）。

### 页内二分查找是怎么找到槽的？

Page Directory 是一个数组，每个元素是一个槽（slot），存的是组内最大记录在页内的偏移量。槽按记录大小有序，所以可以在槽数组上做二分查找。找到目标槽后，组内通常只有 4~8 条记录，线性扫描即可。

### 范围查询为什么不需要回根节点？

叶子节点之间通过 `FIL_PAGE_PREV` / `FIL_PAGE_NEXT` 双向链表串联。范围查询定位到起始叶子后，顺着 `NEXT` 指针向右扫描即可，每扫一页是一次顺序 IO，比"对每个值重新走 B+ 树"高效得多。这是 B+ 树相比 B 树的关键优势之一。

### 二级索引查询为什么是两次 B+ 树查找？

因为 InnoDB 二级索引叶子节点存的是主键值，不是数据行本身。找到主键值后，要拿到整行数据必须回到聚簇索引的 B+ 树再查一次。这就是"回表"。覆盖索引能省掉这一次，是因为查询字段已经在二级索引叶子节点里。

### INSERT 时 B+ 树会发生什么？

1. 定位到目标叶子页（同查询流程）。
2. 页内有空闲空间 → 直接插入到正确位置（页内记录重排）。
3. 页满 → 触发页分裂：把页分成两半，中间键上推到父节点；如果父节点也满，递归分裂，最坏情况树高 +1。
4. 自增主键插入时，新行总追加在最后，几乎不触发页分裂。

## 易错点

- 误以为"B+ 树查询每次都从磁盘读 3 页"——根节点常驻 Buffer Pool，实际通常 2 次 IO。
- 误以为"页内查找是线性扫描"——通过 Page Directory 二分，O(log N)。
- 误以为"范围查询每个值都重新走 B+ 树"——叶子链表顺序扫描。
- 误以为"二级索引查到主键就结束了"——还要回表到聚簇索引拿整行（除非覆盖索引）。
- 误以为"B+ 树叶子是单向链表"——InnoDB 是双向链表。

## 总结

InnoDB B+ 树查询是"层层二分下行 + 叶子链表扫描"：每层在页内通过 Page Directory 二分定位子页，叶子页内同样二分定位记录分组。等值查询走到叶子即结束，3 层 B+ 树只需 2~3 次 IO；范围查询定位到起始叶子后顺着双向链表扫描。二级索引查询找到主键值后需要回表到聚簇索引，覆盖索引可省略回表。理解查询流程是理解索引优化、回表、覆盖索引、最左前缀的基础。
