# 索引失效的场景有哪些？

索引失效的场景有哪些？

索引的存储结构
MySQL默认的存储引擎是InnoDB引擎，它默认采用的B+树作为索引的数据结构。在创建表的时候，InnoDB存储引擎默认会创建一个主键索引，也就是聚簇索引，其他索引都属于二级索引。MySQL的MyISAM存储引擎也支持多种索引结构，如B+树索引、R树索引、FULL-TEXT索引，MyISAM存储引擎在创建表时，默认使用到的也是B+树索引。虽然两者都支持B+树索引，但是它们数据的存储结构实现方式不同，不同之处在于InnoDB存储引擎的B+树索引叶子节点保存的是数据本身，而MyISAM的B+树的叶子节点保存的是数据的物理地址，即用户数据的指针。

索引失效的场景
当使用左或者左右模糊匹配时：
原因是索引B+树是按照索引值有序排序存储的，只能根据前缀进行比较，否则无法确定索引开始比较位置。
对查询条件中对索引列使用函数：
因为索引保存的是索引字段的原始值，而不是经过函数计算后的值，自然就没办法走索引了。不过，从MySQL8.0开始，索引特性增加了函数索引，既可以针对函数计算后的值建立一个索引。

alter table t_user add key idx_name_length((length(name)));

对查询条件中对索引列使用表达式：
原因跟对索引使用函数差不多，因为索引保存的是索引字段的原始值，而不是表达式计算后的值，所以无法走索引，只能通过把索引字段的取值都取出来，然后依次进行表达式的计算来进行条件判断，因此采用全表扫描的方式。

select * from t_user where id + 1 = 10 // 无法使用索引
select * from t_user where id = 10 + 1; // 可以使用索引

当索引列发生隐式类型转换时：
MySQL的数据转换规则是MySQL在遇到字符串和数字进行比较的时候，会自动把比较值从字符串转换成数字。因此如果索引字段为字符串，比较值为数字时，就会使用CAST函数将字符串索引转成数字，再进行比较，导致索引失效。

select * from t_user where id = '1'; // 可以使用索引
select * from t_user where phone = 1881923343545; // 无法使用索引

联合索引需要遵循最左匹配原则，否则索引失效；
原因是联合索引的情况下，数据是按照索引第一列进行排序，第一列相同时，才会按照第二列排序，也就是说，如果我们想使用联合索引中尽可能多的列，查询条件中各个列必须是联合索引中从最左开始连续的列。如果我们仅仅按照第二列搜索，肯定无法走索引。
如果在OR前的条件是索引列，而OR后的是非索引列时：
因为OR的含义是两个中只要满足一个即可，因此只有一个条件是索引列是没有意义的，只要有条件列不是索引，就会全表扫描。
