# 索引失效的场景有哪些

## 核心概念

建了索引不代表一定会走索引。MySQL 优化器会基于成本估算选择执行计划，当走索引的成本高于全表扫描时，或者索引本身的结构无法支持查询条件时，索引就会失效。索引失效可分为两类：**结构失效**（查询条件与 B+ 树有序性不兼容，索引根本无法用）和**优化器放弃**（索引可用但成本更高，优化器选全表扫描）。

理解失效原因的关键是"B+ 树按索引列原始值有序"。一旦对索引列做函数运算、表达式计算、隐式类型转换，B+ 树中存的"原始值"就不再是查询条件里的"变换后值"，无法二分定位。

## 标准回答

索引失效的常见场景：1) 对索引列使用函数或表达式（`WHERE YEAR(t)=...`、`WHERE id+1=10`）；2) 隐式类型转换（`WHERE phone=13800` 但 phone 是 VARCHAR）；3) `LIKE '%xxx'` 或 `LIKE '%xxx%'`（前缀不固定）；4) 联合索引不满足最左前缀；5) `OR` 连接的条件中有非索引列；6) `!=`、`<>`、`NOT IN`（选择性差时优化器放弃）；7) 索引列上 `IS NULL` 过多；8) 优化器估算扫描行数过多主动放弃。验证方法：`EXPLAIN` 看 `type` 是否为 `ALL`、`key` 是否为 `NULL`。

## 失效场景详解

### 1. 对索引列使用函数

```sql
-- 索引: idx_created(created_at)
SELECT * FROM orders WHERE YEAR(created_at) = 2024;        -- 索引失效
SELECT * FROM orders WHERE created_at >= '2024-01-01'
  AND created_at < '2025-01-01';                            -- 索引生效
```

原因：B+ 树存的是 `created_at` 原始值（如 `2024-06-15 10:30:00`），不是 `YEAR()` 计算后的值（`2024`）。`YEAR(created_at)` 在索引里是无序的，无法二分。

8.0+ 解决方案：函数索引。

```sql
ALTER TABLE orders ADD KEY idx_year_created ((YEAR(created_at)));
```

### 2. 对索引列使用表达式

```sql
-- 索引: PRIMARY(id)
SELECT * FROM user WHERE id + 1 = 10;     -- 索引失效
SELECT * FROM user WHERE id = 10 - 1;     -- 索引生效（常量折叠后等价于 id=9）
```

原因：B+ 树按 `id` 原始值排序，`id + 1` 在索引里无序。把表达式挪到右边变成常量计算，左侧仍是 `id` 原始值，索引就能用。

### 3. 隐式类型转换

```sql
-- phone 是 VARCHAR(20)，有索引
SELECT * FROM user WHERE phone = 13800138000;     -- 索引失效
SELECT * FROM user WHERE phone = '13800138000';   -- 索引生效

-- id 是 INT，有索引
SELECT * FROM user WHERE id = '1';                -- 索引生效（字符串转数字）
```

原因：MySQL 的转换规则是"字符串和数字比较时，把字符串转成数字"。所以 `phone = 13800138000` 实际等价于 `CAST(phone AS SIGNED) = 13800138000`，对 `phone` 列做了函数运算，索引失效。反过来 `id = '1'` 是把右侧字符串 `'1'` 转成数字 `1`，左侧 `id` 没变，索引生效。

**口诀**：左边的列是什么类型，右边就传什么类型，避免对索引列做隐式转换。

### 4. LIKE 左模糊

```sql
-- 索引: idx_name(name)
SELECT * FROM user WHERE name LIKE '张%';        -- 索引生效（前缀匹配）
SELECT * FROM user WHERE name LIKE '%张';        -- 索引失效
SELECT * FROM user WHERE name LIKE '%张%';       -- 索引失效
```

原因：B+ 树按 name 字典序排列，前缀 `'张%'` 是一个明确的范围 `[张, 录)`，可以二分定位。`'%张'` 前缀不固定，无法确定起始位置，只能全表扫描。

解决方案：用全文索引、覆盖索引、或外部搜索引擎（Elasticsearch）。

### 5. 联合索引不满足最左前缀

```sql
-- 索引: idx_status_user_created(status, user_id, created_at)
SELECT * FROM orders WHERE user_id = 1001;                        -- 索引失效（跳过 status）
SELECT * FROM orders WHERE status = 1 AND created_at > '...';     -- 部分生效（用 status，不用 created_at）
SELECT * FROM orders WHERE status = 1 AND user_id = 1001;         -- 索引生效
```

详见 [使用联合索引需要注意什么](使用联合索引需要注意什么？.md)。

### 6. OR 连接的条件中有非索引列

```sql
-- name 有索引，age 没有索引
SELECT * FROM user WHERE name = '张三' OR age = 25;       -- 索引失效（必须全表扫描满足 age=25）
SELECT * FROM user WHERE name = '张三' OR phone = '138';  -- 索引生效（如果 phone 也有索引，用 Index Merge）

-- name 和 age 都有索引
SELECT * FROM user WHERE name = '张三' OR age = 25;       -- 索引生效（Index Merge）
```

原因：`OR` 表示两个条件满足其一即可，如果其中一个条件列没有索引，必须全表扫描才能找到该条件匹配的记录，整条查询就只能全表扫描。

### 7. !=, <>, NOT IN, NOT EXISTS

```sql
SELECT * FROM user WHERE status != 1;            -- 通常索引失效
SELECT * FROM user WHERE status NOT IN (1, 2);   -- 通常索引失效
```

原因：`!=` 否定条件意味着要匹配绝大部分数据，优化器估算扫描行数接近全表，主动放弃索引。如果 `status != 1` 的行数很少（大部分是 1），仍可能走索引，所以这不是绝对失效，而是看选择性。

### 8. IS NULL / IS NOT NULL

```sql
SELECT * FROM user WHERE email IS NULL;       -- NULL 多时索引失效
SELECT * FROM user WHERE email IS NOT NULL;   -- NULL 少时索引失效
```

原因：当 NULL 占比很高时，扫描行数接近全表，优化器放弃索引。NULL 占比低时索引仍生效。

### 9. 优化器主动放弃

```sql
-- 性别字段，索引 idx_gender(gender)
SELECT * FROM user WHERE gender = '男';     -- 50% 数据匹配，优化器选全表扫描更快
```

原因：索引扫描 50% 行 + 回表 50% 次，比全表扫描（顺序 IO）慢。优化器基于 `rows` 估算主动放弃索引。

### 10. 字符集或校对规则不一致

```sql
-- 表 A 是 utf8mb4，表 B 是 utf8
SELECT * FROM A JOIN B ON A.name = B.name;   -- 索引可能失效（字符集不匹配）
```

JOIN 时如果两边字符集不同，会对一侧做隐式转换，索引可能失效。生产中表字符集要统一。

## 代码示例

```sql
-- 验证索引是否生效
EXPLAIN SELECT * FROM user WHERE phone = 13800138000;
-- type: ALL          ← 全表扫描
-- key: NULL          ← 没用索引
-- rows: 表总行数

EXPLAIN SELECT * FROM user WHERE phone = '13800138000';
-- type: ref          ← 走索引
-- key: idx_phone
-- rows: 1

-- 强制走索引（验证优化器是否选错）
EXPLAIN SELECT * FROM user FORCE INDEX (idx_phone) WHERE gender = '男' AND phone = '138';
-- 如果强制后更慢，说明优化器选全表是正确的
```

8.0 查看优化器为什么放弃索引：

```sql
EXPLAIN FORMAT=TREE SELECT * FROM user WHERE gender = '男';
-- 或
EXPLAIN ANALYZE SELECT * FROM user WHERE gender = '男';
```

## 实战场景

| 失效场景 | 修复方法 |
|----------|----------|
| `WHERE YEAR(created_at) = 2024` | 改为 `WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'` |
| `WHERE phone = 13800`（phone 是 VARCHAR） | 改为 `WHERE phone = '13800'` |
| `WHERE name LIKE '%张'` | 用全文索引或 ES；或在尾部维护一个反转字段 |
| `WHERE id + 1 = 10` | 改为 `WHERE id = 9` |
| `WHERE a=? OR b=?`（b 无索引） | 给 b 加索引，或拆成 UNION |
| `WHERE status != 1` | 改为 `WHERE status IN (0, 2, 3)` 或业务上反过来查 `status = 1` |
| JOIN 字符集不一致 | 统一表字符集为 utf8mb4 |

## 深挖追问

### `IN` 会导致索引失效吗？

不会。`WHERE id IN (1, 2, 3)` 等价于 `id=1 OR id=2 OR id=3`，每個都是等值查询，可以走索引。但如果 `IN` 列表太长（如几千个值），优化器可能选全表扫描，因为索引定位 + 回表的成本累积超过全表扫描。

### `!=` 一定不走索引吗？

不一定。如果 `!= 1` 的行数很少（如 99% 是 1），扫描行数少，索引可能走。`!=` 失效通常是"匹配数据占大部分"的情况。

### 索引失效一定慢吗？

不一定。如果表很小，全表扫描本身就快。失效的真正问题是"大表上预期走索引却走了全表"，导致扫描行数爆炸。

### 如何系统性排查索引失效？

1. 开启慢查询日志，找慢 SQL。
2. 对每条慢 SQL 用 `EXPLAIN` 看 `type`、`key`、`rows`、`Extra`。
3. `type=ALL` 或 `key=NULL` 说明没走索引。
4. 根据上面的失效场景逐一对照检查。
5. 改写 SQL 或加索引后重新 `EXPLAIN` 验证。

### 优化器选错索引怎么办？

- 临时方案：`USE INDEX` / `FORCE INDEX` 强制走某个索引。
- 永久方案：`ANALYZE TABLE` 更新统计信息；删除冗余索引；调整索引设计；用 `OPTIMIZER_TRACE` 排查。

## 易错点

- 误以为"`IN` 会让索引失效"——`IN` 等价于多个等值 OR，仍走索引。
- 误以为"LIKE 都不走索引"——`LIKE 'x%'` 前缀匹配走索引。
- 误以为"对索引列做任何运算都失效"——右侧的常量运算不影响（`id = 10 - 1` 仍走索引）。
- 误以为"!= 一定不走索引"——取决于扫描行数估算。
- 误以为"强制走索引一定更快"——优化器选全表有时是对的，强制反而慢。

## 总结

索引失效分两类：结构失效（函数/表达式/隐式转换/左模糊破坏 B+ 树有序性）和优化器放弃（扫描行数过多）。常见失效场景：对索引列做函数或表达式、隐式类型转换、`LIKE '%xxx'`、不满足最左前缀、`OR` 含非索引列、`!=`/`NOT IN` 选择性差、JOIN 字符集不一致。排查用 `EXPLAIN` 看 `type`、`key`、`rows`、`Extra`，根因通常是"索引列被变换"或"扫描行数过多"。修复优先 SQL 改写（避免对索引列运算、统一类型），其次才考虑加索引或函数索引。
