# 索引结构是怎么样的？

## 核心概念

索引结构是怎么样的？
索引的存储结构跟 MySQL 使用哪种存储引擎有关，因为存储引擎就是负责将数据持久化在磁盘中，而不同的存储引擎采用的索引数据结构也会不相同。MySQL 默认的存储引擎是 InnoDB，它采用 B+Tree 作为索引的数据结构。

在创建表时，InnoDB 存储引擎默认会创建一个主键索引，也就是聚簇索引，其它索引都属于二级索引。MySQL 的 MyISAM 存储引擎支持多种索引数据结构，比如 B+ 树索引、R 树索引、Full-Text 索引。MyISAM 存储引擎在创建表时，创建的主键索引默认使用的是 B+ 树索引。虽然，InnoDB 和 MyISAM 都支持 B+ 树索引，但是它们数据的存储结构实现方式不同。不同之处在于：
InnoDB 存储引擎：B+ 树索引的叶子节点保存数据本身；
MyISAM 存储引擎：B+ 树索引的叶子节点保存数据的物理地址，即用户数据的指针。

## 面试官想考什么

- 索引能否减少扫描行数、排序、临时表和回表成本。
- B+ 树、聚簇索引、二级索引、联合索引、最左前缀是否理解。
- 能否通过 EXPLAIN 判断命中索引、回表、索引失效。

## 标准回答

MySQL 索引本质是用有序数据结构降低查询扫描成本。InnoDB 常见索引以 B+ 树组织，主键索引叶子节点存整行数据，二级索引叶子节点存主键值；查询不能覆盖索引时通常需要回表。索引设计要围绕高频查询、选择性、排序/分组、范围条件和写入成本综合权衡。

## 深挖追问

1. 为什么联合索引遵循最左前缀？因为 B+ 树按联合键字典序排序。
2. 覆盖索引一定最快吗？不一定，还要看扫描行数、过滤性和维护成本。
3. 索引越多越好吗？不是，写入和更新都要维护索引。

## 实战场景 / SQL 示例

```sql
EXPLAIN SELECT id, name FROM user WHERE status = 1 AND created_at >= ? ORDER BY created_at DESC LIMIT 20;
-- 可评估联合索引：(status, created_at)，若需要避免回表再考虑覆盖字段。
```

## 易错点 / 总结

- 不要认为建了索引就一定会走索引；优化器会按成本选择。
- 联合索引遇到范围条件后，后续列通常难以继续用于精确定位。
- 低选择性字段、频繁更新字段要谨慎建索引。
