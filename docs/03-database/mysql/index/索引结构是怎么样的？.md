# 索引结构是怎么样的

## 核心概念

MySQL 的索引结构由存储引擎决定。InnoDB 默认且仅主要使用 B+ 树，MyISAM 也用 B+ 树但叶子节点存的是数据行的物理地址而非数据本身；Memory 引擎默认哈希索引（也支持 B+ 树）；NDB Cluster 用 T-tree 索引。日常讨论 MySQL 索引结构，默认指 InnoDB 的 B+ 树。

InnoDB 的 B+ 树有几个特化设计：每个节点是一个数据页（默认 16KB）、非叶子节点只存目录项、叶子节点存数据并按主键顺序通过双向链表连接。这种结构让 InnoDB 在等值查询、范围查询、排序、分组上都表现良好。

## 标准回答

MySQL InnoDB 的索引结构是 B+ 树。每个节点是一个 16KB 的数据页；非叶子节点只存索引键和子页号指针，用于二分定位；叶子节点存实际数据（聚簇索引）或主键值（二级索引），叶子节点之间通过双向链表串联，支持范围扫描。MyISAM 也用 B+ 树，但叶子节点存数据行的物理地址。Memory 引擎默认哈希索引。

## InnoDB 的 B+ 树结构

```
                     [根节点 (页号=10)]
                     keys: 10 | 20 | 30
                     ptrs: →p1 →p2 →p3 →p4
                    /          |          |          \
              [叶 p1]      [叶 p2]      [叶 p3]      [叶 p4]
              (1,5,8)      (10,12,18)   (20,25,28)   (30,35)
                  ↔            ↔            ↔            ↔
                  └──────── 双向链表 ────────┘

每个节点 = 一个 16KB 数据页
非叶子节点：存 (最小键, 子页号) 目录项，支持页内二分查找
叶子节点：存整行数据（聚簇索引）或主键值（二级索引）
叶子之间：FIL_PAGE_PREV / FIL_PAGE_NEXT 指针，双向链表
```

### 非叶子节点（目录项页）

存放 `(最小键, 子页号)` 形式的目录项，每个目录项指向一个子页。页内目录项按键大小有序，可通过二分查找快速定位到下一层应该走哪个子页。

由于非叶子节点不存数据，单页可以容纳大量目录项（约 1170 个，假设 Bigint 主键 8B + 页号 4B = 12B），所以 B+ 树非常"扁平"，3 层就能覆盖千万级数据。

### 叶子节点（数据页）

- 聚簇索引的叶子节点存完整行数据。
- 二级索引的叶子节点存 `(索引列值, 主键值)`。
- 页内行按主键顺序组成单向链表。
- 页与页之间通过双向链表连接。

### 为什么叶子节点要双向链表

范围查询 `WHERE id BETWEEN 10 AND 28` 时，定位到 `id=10` 的叶子页后，可以顺着链表向右扫描到 `id=28`，无需回到根节点重新走一遍 B+ 树。这是 B+ 树相比 B 树的关键优势。

## MyISAM 的 B+ 树结构

```
MyISAM 索引（主键索引和二级索引结构相同）:
  叶子节点: (key, 数据行物理地址)

       [根节点]
        keys: 10 | 20 | 30
       /        |          \
   [叶]       [叶]         [叶]
  (10,addr1)  (20,addr2)   (30,addr3)
                 ↓
            通过 addr 到 .MYD 文件定位整行
```

MyISAM 数据和索引分离（`.MYD` 存数据，`.MYI` 存索引），所以主键索引和二级索引在结构上没有本质区别，都是 `(key, 行物理地址)`。这也是 MyISAM 主键长度不影响其他索引大小的原因。

## InnoDB vs MyISAM 索引结构差异

| 维度 | InnoDB | MyISAM |
|------|--------|--------|
| 数据存储位置 | 聚簇索引叶子节点 | 独立的 `.MYD` 文件 |
| 主键索引叶子节点 | 整行数据 | 行物理地址 |
| 二级索引叶子节点 | 主键值 | 行物理地址 |
| 二级索引查到后 | 回表到聚簇索引 | 直接通过地址读 `.MYD` |
| 主键长度影响其他索引 | 是（二级索引存主键值） | 否 |
| 一张表有几棵 B+ 树 | 1（聚簇） + N（二级） | N（每个索引一棵，平级） |

## 其他索引数据结构

### 哈希索引

Memory 引擎默认哈希索引，等值查询 O(1)。但**不支持范围查询、排序、前缀匹配**。InnoDB 不直接支持哈希索引，但有"自适应哈希索引"（AHI）：Buffer Pool 中热点页的索引会被自动建哈希，等值查询走哈希更快。

### 全文索引（FULLTEXT）

5.6+ InnoDB 支持全文索引，底层是倒排索引：把文本分词后建 `(词, 文档ID列表)` 的索引，支持 `MATCH() AGAINST()` 全文搜索。中文需 `ngram` 分词器。

### R 树（空间索引）

MyISAM 和 InnoDB（5.7+）支持空间索引，底层 R 树，用于 GIS 数据类型（`POINT`、`POLYGON`）的范围查询。

## 代码示例

查看表的索引结构：

```sql
SHOW INDEX FROM orders;
-- 关注 Key_name、Column_name、Index_type（BTREE/HASH/FULLTEXT）
```

查看索引大小（MB）：

```sql
SELECT
  index_name,
  ROUND(stat_value * @@innodb_page_size / 1024 / 1024, 2) AS size_mb
FROM mysql.innodb_index_stats
WHERE database_name = 'mydb' AND table_name = 'orders'
  AND stat_name = 'size';
```

查看 Buffer Pool 中的自适应哈希索引使用情况：

```sql
SHOW ENGINE INNODB STATUS\G
-- 找到 "INSERT BUFFER AND ADAPTIVE HASH INDEX" 段
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 主键点查、范围查 | 聚簇索引 B+ 树天然支持 | 主键尽量短、自增 |
| 二级索引等值/范围查 | 二级索引 B+ 树 + 回表 | 高频查询考虑覆盖索引避免回表 |
| 全文搜索 | InnoDB FULLTEXT + ngram | 大数据量建议用 Elasticsearch |
| 内存临时表等值查 | Memory 表哈希索引 | 重启丢失，不可作业务表 |
| GIS 范围查 | InnoDB 8.0 空间索引 R 树 | 需要 `SRID` 一致的列类型 |

## 深挖追问

### 为什么 InnoDB 二级索引存主键值而不是行物理地址？

因为 InnoDB 是聚簇索引组织表，行物理位置可能因页分裂而变化。如果二级索引存物理地址，每次主键索引页分裂都要同步更新所有二级索引，代价极高。存主键值虽然查二级索引后要回表，但二级索引不受聚簇索引页分裂影响，维护成本低。

### 自适应哈希索引（AHI）什么时候会启用？

InnoDB 监控热点索引页的访问模式，如果某些索引值被等值查询访问频繁，会自动在内存中建哈希索引。AHI 由 `innodb_adaptive_hash_index` 控制，默认开启。高并发场景下 AHI 本身的锁争用可能成为瓶颈，有时反而需要关掉。

### InnoDB 为什么不支持用户手动建哈希索引？

因为哈希索引不支持范围、排序、前缀匹配，与 InnoDB 的 OLTP 定位不完全契合。InnoDB 通过 AHI 在内存中自动对热点建哈希，用户感知不到。Memory 引擎允许显式哈希索引。

## 易错点

- 误以为"MySQL 索引就是 B+ 树"——Memory 默认哈希，MyISAM 还支持 R 树、全文索引。
- 误以为"主键索引和二级索引结构不同"——InnoDB 中两者都是 B+ 树，只是叶子节点内容不同。
- 误以为"MyISAM 二级索引存主键值"——MyISAM 二级索引存行物理地址。
- 误以为"InnoDB 支持哈希索引"——InnoDB 不支持用户显式建哈希，只有 AHI。
- 误以为"叶子节点是单向链表"——InnoDB 是双向链表，支持向前向后扫描。

## 总结

InnoDB 的索引结构是 B+ 树：每个节点是一个 16KB 数据页，非叶子节点存目录项支持二分定位，叶子节点存数据（聚簇）或主键值（二级）并通过双向链表串联，支持范围扫描和排序。MyISAM 也是 B+ 树但叶子存行物理地址，主键和二级索引平级。Memory 默认哈希索引，全文索引用倒排索引，空间索引用 R 树。理解每种结构的适用场景，才能避免误用。
