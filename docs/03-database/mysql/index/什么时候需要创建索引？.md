# 什么时候需要创建索引

## 核心概念

索引能加速查询，但也有代价：占用磁盘空间、降低写入速度、增加优化器选错索引的概率。所以"什么时候建索引"本质是一个 ROI 评估：查询收益是否大于写入和存储成本。

判断的核心维度有四个：**字段是否高频出现在 WHERE/ORDER BY/GROUP BY/JOIN 中**、**字段的选择性是否足够高**、**查询是否需要回表**、**写入频率是否过高**。前两个判断"该不该建"，后两个判断"怎么建最优"。

## 标准回答

需要建索引的场景：高频 WHERE 条件字段、ORDER BY/GROUP BY 字段、JOIN 关联字段、有唯一性约束的字段、需要覆盖索引避免回表的高频查询。不需要建索引的场景：低选择性字段（如性别）、WHERE/ORDER BY 用不到的字段、数据量很小的表、频繁更新且查询不频繁的字段。判断公式：`区分度 = distinct(column) / count(*)`，越接近 1 越适合建索引。

## 适合建索引的场景

### 1. 高频 WHERE 条件字段

```sql
-- 经常按 user_id 查订单
SELECT * FROM orders WHERE user_id = 1001;
CREATE INDEX idx_user_id ON orders(user_id);
```

如果查询条件有多个字段，考虑联合索引而非多个单列索引。

### 2. ORDER BY / GROUP BY 字段

```sql
-- 经常按 created_at 排序分页
SELECT * FROM orders WHERE user_id = 1001 ORDER BY created_at DESC LIMIT 20;
CREATE INDEX idx_user_created ON orders(user_id, created_at);
-- 索引本身有序，避免 Using filesort
```

如果只建 `idx_user_id`，查询时还要对 `created_at` 做 filesort，性能差。

### 3. JOIN 关联字段

```sql
SELECT * FROM orders o JOIN user u ON o.user_id = u.id WHERE u.status = 1;
-- 在 orders.user_id 上建索引（被驱动表的关联字段必须建索引）
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

JOIN 优化铁律：被驱动表的关联字段必须有索引，否则嵌套循环扫描全表。

### 4. 唯一性约束字段

```sql
-- 业务上要求 email 唯一
CREATE UNIQUE INDEX uk_email ON user(email);
```

唯一索引既是约束也是查询加速，一举两得。

### 5. 需要覆盖索引的高频查询

```sql
-- 高频查询：根据 user_id 查 user 的 name 和 age
SELECT name, age FROM user WHERE user_id = 1001;
CREATE INDEX idx_user_id_name_age ON user(user_id, name, age);
-- 覆盖索引，Using index，避免回表
```

### 6. 区分度高的字段

区分度 = `COUNT(DISTINCT column) / COUNT(*)`，越接近 1 越适合建索引：

| 字段类型 | 区分度 | 是否适合建索引 |
|----------|--------|----------------|
| 性别（男/女） | ~0.5/总行数 | 不适合（接近 0） |
| 状态（1/2/3） | ~3/总行数 | 不适合 |
| 手机号 | 接近 1 | 适合 |
| 身份证号 | 1 | 适合 |
| 用户 ID | 1 | 适合 |

```sql
-- 估算区分度
SELECT
  COUNT(DISTINCT status) / COUNT(*) AS status_selectivity,
  COUNT(DISTINCT phone) / COUNT(*) AS phone_selectivity
FROM user;
```

## 不适合建索引的场景

### 1. 低选择性字段

性别、状态这种字段，无论索引还是全表扫描，扫描行数差不多，索引反而是负担。优化器可能直接选全表扫描，索引白建。

### 2. WHERE/ORDER BY/GROUP BY 用不到的字段

索引的价值是"快速定位"，如果字段不出现在查询条件里，建索引没有任何收益，只有写入开销。

### 3. 数据量很小的表

小表全表扫描本身就快（几个数据页），索引维护的开销可能超过查询收益。经验值：几百行以内的小表不必建索引。

### 4. 频繁更新的字段

每次 UPDATE 都要维护 B+ 树（可能引起页分裂）。比如电商用户的"余额"字段，频繁更新且很少直接以余额为查询条件，不适合建索引。

### 5. 大字段（TEXT/BLOB）

大字段不适合直接做索引键（一个字段就占满一页）。如果需要在大字段上查询，用前缀索引或全文索引。

## 联合索引的设计原则

详见 [使用联合索引需要注意什么](使用联合索引需要注意什么？.md)。要点：

1. **等值条件放左边，范围条件放右边**：`WHERE a=? AND b>? AND c=?` 中只有 a、b 能用上索引，c 用不上。
2. **区分度高的放左边**：让最左前缀过滤效果最大化。
3. **ORDER BY 字段放末尾**：利用索引有序性避免 filesort。
4. **尽量覆盖查询字段**：避免回表。

## 代码示例

判断是否需要建索引：

```sql
-- 1. 看查询频率（慢日志）
SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;

-- 2. 看字段区分度
SELECT
  COUNT(DISTINCT user_id) / COUNT(*) AS user_id_sel,
  COUNT(DISTINCT status) / COUNT(*) AS status_sel
FROM orders;

-- 3. 看现有索引使用情况（8.0）
SELECT * FROM sys.schema_unused_indexes
WHERE object_schema = 'mydb';
```

建联合索引：

```sql
-- 查询: SELECT id, order_no, amount FROM orders
--       WHERE user_id = ? AND status = 1
--       ORDER BY created_at DESC LIMIT 20;

CREATE INDEX idx_user_status_created
ON orders(user_id, status, created_at);
-- 覆盖 id（主键自带）、user_id、status、created_at
-- 还差 order_no 和 amount，如果业务允许可加入索引末尾
```

验证执行计划：

```sql
EXPLAIN SELECT id, order_no, amount FROM orders
WHERE user_id = 1001 AND status = 1
ORDER BY created_at DESC LIMIT 20;
-- type: ref          ← 走联合索引
-- key: idx_user_status_created
-- Extra: Using index ← 覆盖索引（如果索引包含 order_no、amount）
```

## 实战场景

| 场景 | 是否建索引 | 原因 |
|------|-----------|------|
| 订单表 `user_id` | 是 | 高频 WHERE 条件，区分度高 |
| 订单表 `status` 单独 | 否 | 区分度低，单独走索引优化器可能拒绝 |
| 用户表 `phone` | 是（唯一） | 唯一约束 + 高频登录查询 |
| 用户表 `gender` | 否 | 区分度太低 |
| 日志表 `created_at` | 视情况 | 范围查询频繁则建；按时间分区也可代替 |
| JOIN 关联字段 `orders.user_id` | 是 | 被驱动表关联字段必须建索引 |
| 报表 `GROUP BY dept_id` | 是 | 利用索引有序性避免临时表 |

## 深挖追问

### 区分度多少才适合建索引？

经验值：区分度 > 0.3 比较有价值。但这只是参考，最终要看"扫描行数 + 回表成本" vs "全表扫描成本"。比如 1 亿行的表，区分度 0.3 意味着索引扫描 3000w 行 + 回表，可能还不如全表扫描。优化器会自己判断，但建索引前可以用 `EXPLAIN` 估算。

### 主键需要"建索引"吗？

不需要显式建，InnoDB 自动把主键作为聚簇索引。但主键设计本身要符合"短、自增、稳定"的原则，否则影响所有二级索引。

### 已经有联合索引 `(a, b)`，还需要单独建 `idx_a` 吗？

不需要。联合索引 `(a, b)` 的最左前缀已经覆盖了 `WHERE a=?` 的查询。重复建 `idx_a` 是浪费空间和写入开销。

### 索引数量上限？

没有硬上限，但经验上一张 OLTP 表的索引不超过 5 个，单索引字段数不超过 5。索引越多，写入越慢、优化器选错概率越高、空间越大。

### 唯一索引和普通索引性能差多少？

查询：唯一索引找到第一条匹配就停止；普通索引找到第一条后还要继续找下一条（因为可能有重复）。差异很小。
写入：唯一索引需要先检查唯一性（多一次 B+ 树查找），写入比普通索引慢。

## 易错点

- 误以为"建了索引就一定走"——优化器会按成本选择，低选择性字段可能不走。
- 误以为"区分度低就一定不走索引"——配合联合索引仍可能走（如 `(status, user_id)`）。
- 误以为"任何字段都该建索引"——低选择性、频繁更新、不查询的字段建了反而拖累写入。
- 误以为"联合索引等价于多个单列索引"——联合索引是有序的一棵树，更高效。
- 误以为"小表不需要索引"——JOIN 时被驱动表即使是小表也要建索引。

## 总结

建索引的核心判断是 ROI：查询收益是否大于写入和存储成本。需要建索引的场景：高频 WHERE/ORDER BY/GROUP BY/JOIN 字段、唯一约束字段、需要覆盖索引的高频查询、区分度高的字段。不需要建索引的场景：低选择性字段、查询用不到的字段、小表、频繁更新字段。联合索引设计要遵循"等值放左、范围放右、区分度高放左、ORDER BY 放末尾"。建完用 `EXPLAIN` 验证执行计划。
