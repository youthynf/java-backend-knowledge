# 聚簇索引和二级索引是什么

## 核心概念

InnoDB 的索引分为聚簇索引和二级索引两类。**聚簇索引**的叶子节点直接存放整行数据，一张表只能有一个；**二级索引**的叶子节点存放主键值，一张表可以有多个。这种"主键索引就是数据本身"的设计是 InnoDB 区别于 MyISAM 的核心特征，也是 InnoDB 主键设计敏感、二级索引查询需要回表的根本原因。

理解聚簇索引和二级索引的差异，是理解 InnoDB 主键设计、回表、覆盖索引、写入性能等所有问题的前提。

## 标准回答

聚簇索引按主键组织 B+ 树，叶子节点存整行数据，一张表只能有一个，由 InnoDB 按"主键 → 第一个非空唯一索引 → 隐式 DB_ROW_ID"的规则选择。二级索引按索引列组织 B+ 树，叶子节点存主键值，可有多个。查二级索引找非索引字段时需要"回表"到聚簇索引；如果查询字段都在二级索引里，就形成"覆盖索引"，无需回表。

## 聚簇索引

### 选择规则

InnoDB 建表时按以下顺序选择聚簇索引：

1. 显式声明的 `PRIMARY KEY`。
2. 第一个所有列都 `NOT NULL` 的 `UNIQUE` 索引。
3. 都没有则自动生成 6 字节 `DB_ROW_ID` 作为隐式主键。

所以**生产表必须显式定义主键**，否则 InnoDB 用隐式 `DB_ROW_ID`（全局递增，高并发争用），且业务无法通过它定位数据。

### 结构

```
聚簇索引 B+ 树（按主键 id 排序）:

              [根节点]
              keys: 100 | 200 | 300
             /        |          |          \
        [叶 1]    [叶 2]      [叶 3]      [叶 4]
        id<100   100≤id<200  200≤id<300  id≥300
          ↓         ↓           ↓            ↓
      存整行数据  存整行数据   存整行数据    存整行数据

每个叶子节点存放完整行（id, name, age, ...）。
```

### 特点

- 叶子节点 = 完整行数据，主键查询一次 B+ 树即可拿到整行。
- 数据按主键顺序物理组织，主键范围查询可顺叶子链表扫描。
- 一张表只有一个聚簇索引。
- 主键越长，二级索引越大（因为二级索引叶子节点存主键值）。
- 主键非自增时插入会引发页分裂。

## 二级索引

### 结构

```
二级索引 B+ 树（按 name 排序）:

              [根节点]
              keys: 'alice' | 'bob' | 'charlie'
             /          |             |           \
        [叶 1]       [叶 2]         [叶 3]       [叶 4]
        name<alice   ...           ...          ...
          ↓             ↓             ↓             ↓
      (name, id)    (name, id)    (name, id)    (name, id)
      叶子节点存: 索引列值 + 主键值
```

### 查询流程

`SELECT * FROM user WHERE name = 'bob'`：

1. 在 `idx_name` 的 B+ 树定位到 `name='bob'`，得到 `id=42`。
2. 回到聚簇索引的 B+ 树，按 `id=42` 找到整行数据。 ← **回表**

如果查询是 `SELECT id, name FROM user WHERE name = 'bob'`：

1. 在 `idx_name` 的 B+ 树定位到 `name='bob'`，叶子节点已经包含 `id` 和 `name`。
2. 直接返回，不需要回表。 ← **覆盖索引**

### 为什么二级索引存主键值而不是行物理地址

行物理地址会因页分裂而变化。如果二级索引存物理地址，每次聚簇索引页分裂都要同步更新所有二级索引，代价极高。存主键值虽然查二级索引后要回表，但二级索引不受页分裂影响，维护成本低。

代价是：二级索引查询非覆盖字段时需要两次 B+ 树查找（二级索引 + 聚簇索引），但写入性能和空间利用率更高。

## 回表与覆盖索引

### 回表

二级索引查到主键值后，再到聚簇索引查整行数据的过程。回表是随机 IO（聚簇索引叶子节点物理上可能很分散），是查询性能的常见瓶颈。

### 覆盖索引

查询的所有字段都在二级索引的叶子节点里（索引列 + 主键），不需要回表。`EXPLAIN` 的 `Extra` 字段会显示 `Using index`。

```sql
-- 索引: idx_name(name)
SELECT id, name FROM user WHERE name = 'bob';  -- 覆盖索引，Using index
SELECT * FROM user WHERE name = 'bob';          -- 需要回表
```

通过把高频查询字段加入联合索引形成覆盖索引，是优化慢查询的常用手段。

## 代码示例

查看表的索引和聚簇索引：

```sql
SHOW INDEX FROM user;
-- 主键索引的 Key_name = PRIMARY，Seq_in_index = 1
-- 二级索引的 Key_name 是建索引时起的名字
```

通过 EXPLAIN 判断是否回表：

```sql
EXPLAIN SELECT id, name FROM user WHERE name = 'bob';
-- Extra: Using index      ← 覆盖索引，不回表

EXPLAIN SELECT * FROM user WHERE name = 'bob';
-- Extra: NULL 或 Using where ← 需要回表
```

强制走某个索引（调试用，生产慎用）：

```sql
EXPLAIN SELECT * FROM user USE INDEX (idx_name) WHERE name = 'bob';
EXPLAIN SELECT * FROM user FORCE INDEX (idx_name) WHERE name = 'bob';
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 高频点查 | 主键查询最优，聚簇索引一次命中 | 主键设计要短、自增 |
| 高频组合查询 | 联合索引 + 覆盖索引避免回表 | 把 SELECT 字段加入索引末尾 |
| 主键长度选择 | Bigint（8B）足够大且不太长 | 不要用 UUID（36B）作主键，二级索引会膨胀 |
| 写多读少表 | 减少二级索引数量，降低写入维护成本 | 评估每个索引的查询价值 |
| 长字符串搜索 | 二级索引 + 前缀索引 | 前缀索引无法覆盖索引 |

## 深挖追问

### 为什么 InnoDB 主键建议自增？

聚簇索引按主键顺序组织数据。自增主键插入时新行总追加在最后，无需移动已有数据，页满了直接开新页。非自增主键（如 UUID）插入位置随机，会触发页分裂：把已有页的一部分数据复制到新页，产生碎片区和写放大，写入性能下降。详见 [MySQL 索引如何优化](MySQL索引如何优化？.md)。

### 没有主键的表会怎样？

InnoDB 会按"第一个非空唯一索引 → 隐式 DB_ROW_ID"的顺序选聚簇索引。都没有时，生成 6 字节 `DB_ROW_ID` 作隐式主键，存到聚簇索引。隐式主键全局递增（共享一个计数器），高并发插入有争用，且业务无法用它定位数据。生产表必须显式定义主键。

### 二级索引存主键值，那主键变了怎么办？

InnoDB 中主键值是不会"变"的——`UPDATE` 修改主键等价于"删除旧行 + 插入新行"，会重建聚簇索引和所有二级索引中对应项，代价极高。所以**主键设计要稳定不变**，不要选会修改的字段（如手机号、邮箱）作主键。

### 一个表能同时有多个聚簇索引吗？

不能。聚簇索引的本质是"数据按这个键组织"，数据只能按一种顺序物理组织。如果需要多种顺序的查询，通过建二级索引实现，代价是回表。一些其他数据库（如 SQL Server）支持"聚集索引 + 非聚集索引"的概念，但一张表仍然只能有一个聚集索引。

## 易错点

- 误以为"主键索引 = 聚簇索引"——InnoDB 中通常如此，但若没主键，聚簇索引可能是非空唯一索引或隐式 `DB_ROW_ID`。
- 误以为"二级索引存行物理地址"——那是 MyISAM。InnoDB 二级索引存主键值。
- 误以为"覆盖索引是索引类型"——它是一种查询效果，`EXPLAIN Extra: Using index` 才知道。
- 误以为"主键可以随便改"——改主键等于删行+插行，所有索引都要重建，性能极差。
- 误以为"UUID 适合做主键"——UUID 长且随机，二级索引膨胀且插入页分裂严重。

## 总结

聚簇索引按主键组织数据，叶子节点存整行，一张表只有一个；二级索引按索引列组织，叶子节点存主键值，可有多个。二级索引查非覆盖字段需要回表（聚簇索引再查一次），形成覆盖索引时无需回表。InnoDB 主键设计要短、自增、稳定，因为主键决定了聚簇索引的组织方式和所有二级索引的大小。
