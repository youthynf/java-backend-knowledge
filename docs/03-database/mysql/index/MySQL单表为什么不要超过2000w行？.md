# MySQL 单表为什么不要超过 2000w 行

## 核心概念

"单表不超过 2000w 行"是 MySQL 圈的经典经验值，它的本质是**3 层 B+ 树能容纳的行数上限**。InnoDB 的 B+ 树每个节点是一个 16KB 数据页，3 层 B+ 树在典型行大小（约 1KB）下大约能存 2000w 行。超过这个量级，B+ 树会从 3 层增长到 4 层，磁盘 IO 次数从 3 次增加到 4 次，查询性能下降；同时索引页无法全部驻留 Buffer Pool，导致更多磁盘 IO。

需要强调的是：2000w 不是绝对上限，而是"性能拐点"。行大小、主键长度、Buffer Pool 大小、查询模式都会影响实际拐点。但作为经验值，超过 2000w 就应该考虑分表。

## 标准回答

单表不建议超过 2000w 行，本质是 InnoDB 3 层 B+ 树的容量上限。一个 16KB 数据页非叶子节点约能容纳 1170 个目录项（Bigint 主键 8B + 页号 4B），叶子节点约能存 15 行（行大小约 1KB），3 层 B+ 树容量 = 1170 × 1170 × 15 ≈ 2000w 行。超过这个量级 B+ 树会增长到 4 层，磁盘 IO 从 3 次增加到 4 次，且索引无法全部驻留 Buffer Pool，性能下降明显。所以业界把 2000w 作为单表分表的经验拐点。

## B+ 树高度推算

### 假设条件

- `innodb_page_size` = 16KB（默认）
- 主键类型：Bigint，8 字节
- 页号：4 字节
- 行大小：约 1KB（典型业务行，含几个 VARCHAR、几个数字、几个时间字段）

### 非叶子节点容量（x）

一个非叶子节点页（目录项页）存放 `(主键值, 子页号)` 形式的目录项。

- 页大小：16384 字节
- 页头页尾等固定开销：约 150 字节（File Header 38B + Page Header 56B + Infimum/Supremum 26B + File Trailer 8B + Page Directory）
- 每个目录项大小：8B（主键）+ 4B（页号）= 12B
- 单页能容纳的目录项数：`x = (16384 - 150) / 12 ≈ 1170`

### 叶子节点容量（y）

叶子节点页存放完整行数据。

- 页大小：16384 字节
- 固定开销：约 150 字节
- 行大小：约 1KB（1024 字节）
- 单页能容纳的行数：`y = (16384 - 150) / 1024 ≈ 15`

### 树高与容量关系

```
Total = x^(z-1) × y
```

| 树高 z | 容量计算 | 容量 |
|--------|---------|------|
| 2 | 1170^1 × 15 | 约 1.7w 行 |
| 3 | 1170^2 × 15 | 约 2000w 行 |
| 4 | 1170^3 × 15 | 约 240 亿行 |

**3 层 B+ 树约 2000w 行**，这就是经验值的来源。

### 推算的关键意义

- **3 层 B+ 树 = 3 次磁盘 IO**：根节点常驻 Buffer Pool，实际通常 2 次 IO。
- **4 层 B+ 树 = 4 次磁盘 IO**：性能下降 33%。
- **2000w 是 3 层 B+ 树在典型行大小下的极限**：行大小越小，单页能容纳的行越多，3 层能存的总量越大；行大小越大，3 层能存的越少，拐点会提前到来。

## 影响实际拐点的因素

### 1. 行大小

| 行大小 | 单页行数 y | 3 层 B+ 树容量 |
|--------|-----------|----------------|
| 500B | ~30 | ~4000w |
| 1KB | ~15 | ~2000w |
| 2KB | ~7 | ~950w |
| 4KB | ~3 | ~410w |

行越大，拐点越早。如果表里有大量 TEXT/BLOB 字段（溢出到独立页），实际能存的行数更少。

### 2. 主键长度

主键越长，非叶子节点单条目录项越大，x 越小：

| 主键类型 | 主键长度 | x | 3 层容量（行 1KB） |
|----------|----------|-----|-------------------|
| Int | 4B | ~1360 | ~2700w |
| Bigint | 8B | ~1170 | ~2000w |
| UUID(36B) | 36B | ~410 | ~250w |

UUID 作主键不仅插入页分裂严重，还会让 B+ 树容量降到 250w，3 层拐点提前 8 倍。这是"主键要短"的另一个原因。

### 3. Buffer Pool 大小

3 层 B+ 树约 1170 × 1170 ≈ 137w 个叶子页 + 1170 个非叶子页 ≈ 137w × 16KB ≈ 21GB。

- Buffer Pool >= 21GB：整个聚簇索引能驻留内存，查询几乎不产生磁盘 IO。
- Buffer Pool < 21GB：部分叶子页在磁盘，查询可能产生磁盘 IO，性能下降。

所以"2000w"拐点也受 Buffer Pool 大小影响。如果服务器内存大，可以适当放宽；内存小则要更早分表。

### 4. 二级索引

每个二级索引也是一棵独立的 B+ 树，行数同样受 3 层限制。但二级索引叶子节点只存 `(索引列, 主键)`，单页能容纳的行数远多于聚簇索引（行大小 = 索引列 + 主键，远小于完整行）。所以二级索引的 3 层容量通常远大于聚簇索引。

但二级索引查询需要回表，回表是聚簇索引的随机 IO，行数大时回表代价显著。

## 代码示例

查看表的 B+ 树高度（8.0）：

```sql
SELECT
  index_name,
  stat_name,
  stat_value AS tree_height
FROM mysql.innodb_index_stats
WHERE database_name = 'mydb'
  AND table_name = 'orders'
  AND stat_name = 'tree_height';
-- PRIMARY 的 tree_height = 3 表示 3 层
```

查看表的行数和平均行大小：

```sql
SHOW TABLE STATUS LIKE 'orders'\G
-- 关注 Rows、Avg_row_length、Data_length
-- Data_length / innodb_page_size ≈ 叶子页数
```

估算表的理论容量上限：

```sql
-- 假设 Bigint 主键 + 1KB 行
-- 3 层容量 = 1170 × 1170 × (16384 / 1024) ≈ 2000w
-- 4 层容量 = 1170 × 1170 × 1170 × 15 ≈ 240 亿
```

## 实战场景

| 场景 | 处理方式 |
|------|----------|
| 单表 500w 行 | 不必分表，正常运维 |
| 单表 1500w 行 | 接近拐点，开始规划分表方案 |
| 单表 3000w 行 | 性能下降明显，建议分表 |
| 单表 1 亿行 | 必须分表，或归档历史数据 |
| 日志表 1 亿行 | 按时间分区或归档到 Archive 表 |

## 深挖追问

### 2000w 是硬性限制吗？

不是。它是经验拐点，不是绝对限制。如果行很小（如 200B）、Buffer Pool 足够大、查询都走索引，单表 5000w 甚至 1 亿也能跑得不错。但超过 2000w 后，"性能下降风险"显著增加，建议提前规划。

### 分表还是分区？

- **分表**：把一张表拆成多张独立表，应用层或中间件层路由。彻底突破单表限制，但跨表查询复杂。
- **分区**：一张表逻辑上按规则（如时间、HASH）分区，对应用透明。能解决部分管理问题（如快速删除老数据），但单表数据量限制仍在，且分区键必须包含在主键里。

数据量大且查询模式简单时优先分表；数据量大但查询集中在某个维度（如时间）时优先分区。

### 行大小怎么算？

行大小 = 各字段长度之和 + 行头开销（约 20~30 字节，含 DB_TRX_ID、DB_ROLL_PTR 等隐藏列）+ NULL 值列表。VARCHAR 按实际占用算，CHAR 按定义长度算。TEXT/BLOB 超过 768 字节溢出到独立页。

```sql
-- 估算平均行大小
SHOW TABLE STATUS LIKE 'orders'\G
-- Avg_row_length 字段
```

### 为什么是"3 层"而不是"4 层"？

3 层 B+ 树 IO 次数 = 3（根节点常驻 Buffer Pool 时实际 2 次），4 层 = 4 次（实际 3 次）。多一次磁盘 IO 意味着查询延迟翻倍（机械盘 10ms → 20ms）。同时 4 层 B+ 树的索引页数量大幅增加，Buffer Pool 难以全部缓存，更多查询变成磁盘 IO。所以"3 层"是性能与成本的合理平衡点。

### 聚簇索引和二级索引的拐点一样吗？

不一样。二级索引叶子节点存 `(索引列, 主键)`，行大小远小于聚簇索引，单页能容纳的行数多，3 层容量通常远大于 2000w。但二级索引查询需要回表（聚簇索引随机 IO），所以二级索引的"性能拐点"由回表代价决定，不一定按 3 层容量算。

## 易错点

- 误以为"2000w 是绝对上限"——是经验拐点，行大小和 Buffer Pool 都会影响实际拐点。
- 误以为"行数小就一定不需要分表"——高并发写入、热点行争用也可能需要分表。
- 误以为"UUID 主键不影响拐点"——UUID 主键让 3 层容量降到 250w，拐点提前 8 倍。
- 误以为"4 层 B+ 树也能用"——能跑，但每次查询多一次 IO，性能下降明显。
- 误以为"分区 = 分表"——分区对应用透明但单表限制仍在，分表是物理拆分。

## 总结

单表不超过 2000w 行的本质是 3 层 B+ 树的容量上限：16KB 页 × Bigint 主键 × 1KB 行，3 层容量 = 1170 × 1170 × 15 ≈ 2000w。超过这个量级 B+ 树长到 4 层，IO 次数增加，且索引页无法全部驻留 Buffer Pool。行大小、主键长度、Buffer Pool 都会影响实际拐点：行越大拐点越早，主键越长拐点越早，Buffer Pool 越大拐点越晚。2000w 是经验值，实际看性能监控决定是否分表。
