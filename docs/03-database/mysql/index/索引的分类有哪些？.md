# 索引的分类有哪些

## 核心概念

同一份索引可以从不同维度分类，最常见的有四个：按数据结构（B+ 树 / Hash / Full-text）、按物理存储（聚簇 / 二级）、按字段特性（主键 / 唯一 / 普通 / 前缀）、按字段个数（单列 / 联合）。这些分类不是并列关系，而是从不同角度看同一个东西：比如 `idx_email(email)` 既是 B+ 树索引，又是二级索引，又是唯一索引，又是单列索引。

理解分类的意义在于：每类索引解决不同问题，建索引时需要先想清楚自己要的是数据结构层面、物理存储层面、还是约束层面的能力。

## 标准回答

MySQL 索引按数据结构分为 B+ 树、Hash、Full-text；按物理存储分为聚簇索引（主键索引，叶子存整行）和二级索引（叶子存主键值）；按字段特性分为主键索引、唯一索引、普通索引、前缀索引；按字段个数分为单列索引和联合索引。InnoDB 默认且主要使用 B+ 树，聚簇索引一张表只有一个，二级索引可有多个。

## 按数据结构分类

| 类型 | 说明 | 支持的引擎 |
|------|------|-----------|
| B+ 树索引 | 默认，支持等值/范围/排序/前缀 | InnoDB、MyISAM、Memory |
| Hash 索引 | O(1) 等值查询，不支持范围 | Memory（默认）、NDB |
| Full-text 索引 | 倒排索引，全文搜索 | InnoDB（5.6+）、MyISAM |
| R 树索引 | 空间索引，用于 GIS | InnoDB（5.7+）、MyISAM |

InnoDB 不支持用户显式建 Hash 索引，但有自适应哈希索引（AHI）对热点页自动建哈希。

## 按物理存储分类

### 聚簇索引（Clustered Index）

- 叶子节点存**整行数据**。
- 一张表**只能有一个**聚簇索引。
- InnoDB 选择规则：主键 > 第一个非空唯一索引 > 隐式 `DB_ROW_ID`。
- 数据本身按聚簇索引键有序组织，所以"表 = 聚簇索引"。

### 二级索引（Secondary Index，又叫辅助索引）

- 叶子节点存**主键值**。
- 一张表可有多个二级索引。
- 查询非索引字段时，先在二级索引找到主键值，再回聚簇索引找整行（回表）。

详见 [聚簇索引和二级索引是什么](聚簇索引和二级索引是什么？.md)。

## 按字段特性分类

### 主键索引（PRIMARY KEY）

```sql
CREATE TABLE user (
  id BIGINT PRIMARY KEY,  -- 自动创建主键索引（聚簇）
  name VARCHAR(50)
);
```

- 一张表最多一个。
- 索引列值唯一且不为 NULL。
- InnoDB 中即聚簇索引。

### 唯一索引（UNIQUE）

```sql
CREATE UNIQUE INDEX uk_email ON user(email);
```

- 一张表可有多个。
- 索引列值必须唯一，但**允许 NULL**（NULL 不参与唯一性判断，可以有多个 NULL）。
- InnoDB 中是二级索引。

### 普通索引（INDEX / KEY）

```sql
CREATE INDEX idx_name ON user(name);
```

- 无约束，仅加速查询。
- 一张表可有多个。

### 前缀索引

```sql
CREATE INDEX idx_name_prefix ON user(name(10));  -- 只对 name 前 10 个字符建索引
```

- 用于长字符串字段，减少索引空间。
- **不支持覆盖索引**（因为索引里只有前缀，不是完整值）。
- **不能用于 ORDER BY / GROUP BY**（同上原因）。

## 按字段个数分类

### 单列索引

```sql
CREATE INDEX idx_name ON user(name);
```

索引键只有一个字段。

### 联合索引（Composite Index，又叫复合索引）

```sql
CREATE INDEX idx_status_created ON orders(status, created_at);
```

索引键是多个字段的组合，B+ 树按字段顺序排序。遵循**最左前缀原则**：从最左列开始连续匹配。详见 [使用联合索引需要注意什么](使用联合索引需要注意什么？.md)。

## 按功能特性分类（补充）

| 索引 | 说明 |
|------|------|
| 覆盖索引 | 索引包含查询所需全部字段，无需回表 |
| 函数索引（8.0+） | 对函数计算结果建索引，如 `((LOWER(name)))` |
| 不可见索引（8.0+） | `INVISIBLE` 索引，优化器忽略，用于安全测试删除索引 |
| 降序索引（8.0+） | `idx(a DESC, b ASC)` 真正按降序存储，避免反序扫描 |

## 代码示例

```sql
-- 主键索引
CREATE TABLE orders (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  order_no VARCHAR(32) NOT NULL,
  user_id BIGINT NOT NULL,
  status TINYINT NOT NULL,
  created_at DATETIME NOT NULL,
  email VARCHAR(128),

  -- 唯一索引
  UNIQUE INDEX uk_order_no (order_no),
  -- 联合索引
  INDEX idx_user_status_time (user_id, status, created_at),
  -- 前缀索引
  INDEX idx_email_prefix (email(20))
) ENGINE=InnoDB;

-- 查看所有索引
SHOW INDEX FROM orders;
-- 关注 Non_unique（0=唯一，1=普通）、Key_name、Column_name、Index_type

-- 8.0 不可见索引
ALTER TABLE orders ALTER INDEX idx_email_prefix INVISIBLE;
ALTER TABLE orders ALTER INDEX idx_email_prefix VISIBLE;
```

## 实战场景

| 场景 | 选用的索引类型 | 注意点 |
|------|---------------|--------|
| 业务订单号唯一性 | 唯一索引 | 既是约束也是查询加速 |
| 用户列表分页 | 联合索引 `(user_id, created_at)` | 最左前缀 + 覆盖 `id, order_no` 避免回表 |
| 长字符串搜索 | 前缀索引 或 全文索引 | 前缀索引无法覆盖索引和 ORDER BY |
| 测试删除索引是否安全 | 8.0 设为 INVISIBLE | 观察一段时间无影响再正式删除 |
| 函数查询 `WHERE LOWER(name)=` | 8.0 函数索引 | 5.7 及之前需重写 SQL |

## 深挖追问

### 唯一索引和主键索引的区别？

1. 数量：主键一张表只能有一个，唯一索引可有多个。
2. NULL：主键不允许 NULL，唯一索引允许（多个 NULL 不冲突）。
3. 物理存储：InnoDB 中主键即聚簇索引，唯一索引是二级索引。
4. 选择规则：建表时 `PRIMARY KEY` 优先于第一个非空唯一索引作为聚簇索引。

### 覆盖索引是索引分类吗？

严格说"覆盖索引"是一种查询效果，不是独立的索引类型。任何二级索引只要覆盖了查询的所有字段，就形成覆盖索引。它依赖 `EXPLAIN` 的 `Extra: Using index` 来确认。

### 联合索引和多个单列索引等价吗？

不等价。联合索引 `(a, b)` 是按 `(a, b)` 字典序组织的一棵 B+ 树，可以用于 `WHERE a=? AND b=?`、`WHERE a=?`、`ORDER BY a, b` 等多种查询；两个单列索引 `idx_a`、`idx_b` 是两棵独立的树，MySQL 可用索引合并（Index Merge）同时利用，但通常不如联合索引高效，且无法用于 `ORDER BY a, b`。

### 前缀索引为什么不能用于覆盖索引？

前缀索引的 B+ 树叶子节点只存字段的前 N 个字符，不是完整值。查询 `SELECT name FROM user WHERE name='xxx'` 时，前缀匹配后无法确认完整值是否相等，必须回表读完整行再比对。所以前缀索引无法形成覆盖索引。

## 易错点

- 误以为"唯一索引不允许 NULL"——允许，且多个 NULL 不冲突。
- 误以为"前缀索引能用于 ORDER BY"——不能，因为只存了前缀。
- 误以为"联合索引 = 多个单列索引"——联合索引是一棵树，索引合并不等价。
- 误以为"覆盖索引是独立类型"——它是一种查询效果。
- 误以为"InnoDB 主键就是聚簇索引"——如果没有主键也没有非空唯一索引，InnoDB 用隐式 `DB_ROW_ID` 作聚簇索引。

## 总结

MySQL 索引按数据结构分（B+ 树/Hash/Full-text/R 树）、按物理存储分（聚簇/二级）、按字段特性分（主键/唯一/普通/前缀）、按字段个数分（单列/联合）。InnoDB 默认 B+ 树，聚簇索引一张表只有一个，二级索引可有多个。建索引时先想清楚要的是约束（唯一）、加速（普通/联合）、还是空间节省（前缀），再用 `EXPLAIN` 验证。
