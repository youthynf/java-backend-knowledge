# 为什么InnoDB选择B+树作为索引的数据结构？

## 核心概念

为什么InnoDB选择B+树作为索引的数据结构？
B+Tree vs B Tree
B+Tree 只在叶子节点存储数据，而 B 树 的非叶子节点也要存储数据，所以 B+Tree 的单个节点的数据量更小，在相同的磁盘 I/O 次数下，就能查询更多的节点。
B+Tree 叶子节点采用的是双链表连接，适合 MySQL 中常见的基于范围的顺序查找，而 B 树无法做到这一点。
B+树的查询性能更稳定，每次查询都要查找到叶子结点；而B树的查询可能会在非叶子结点找到数据，性能相对不稳定；
B+树有大量冗余的节点（所有非叶子节点都是冗余索引），这些冗余索引让B+树在插入和删除的效率更高，比如删除根节点，不会像B树那样发生复杂的树变化。

B+Tree vs 二叉树
对于有 N 个叶子节点的 B+Tree，其搜索复杂度为O(logdN)，其中 d 表示节点允许的最大子节点个数为 d 个。在实际的应用当中， d 值是大于100的，这样就保证了，即使数据达到千万级别时，B+Tree 的高度依然维持在 3~4 层左右，也就是说一次数据查询操作只需要做 3~4 次的磁盘 I/O 操作就能查询到目标数据。而二叉树的每个父节点的儿子节点个数只能是 2 个，意味着其搜索复杂度为 O(logN)，这已经比 B+Tree 高出不少，因此二叉树检索到目标数据所经历的磁盘 I/O 次数要更多。

B+Tree vs Hash
Hash 在做等值查询的时候效率贼快，搜索复杂度为 O(1)。但是 Hash 表不适合做范围查询，它更适合做等值的查询，这也是 B+Tree 索引要比 Hash 表索引有着更广泛的适用场景的原因。

MySQL中的B+树
MySQL 的存储方式根据存储引擎的不同而不同，我们最常用的就是 Innodb 存储引擎，它就是采用了 B+ 树作为了索引的数据结构。Innodb 使用的 B+ 树有一些特别的点，比如：
B+ 树的叶子节点之间是用「双向链表」进行连接，这样的好处是既能向右遍历，也能向左遍历。
B+ 树节点内容是数据页，数据页里存放了用户的记录以及各种信息，每个数据页默认大小是 16 KB。

Innodb 根据索引类型不同，分为聚簇和二级索引。他们区别在于：
聚簇索引的叶子节点存放的是实际数据，所有完整的用户记录都存放在聚簇索引的叶子节点
而二级索引的叶子节点存放的是主键值，而不是实际数据。
因为表的数据都是存放在聚簇索引的叶子节点里，所以 InnoDB 存储引擎一定会为表创建一个聚簇索引，且由于数据在物理上只会保存一份，所以聚簇索引只能有一个，而二级索引可以创建多个。

## 面试官想考什么

- 索引能否减少扫描行数、排序、临时表和回表成本。
- B+ 树、聚簇索引、二级索引、联合索引、最左前缀是否理解。
- 能否通过 EXPLAIN 判断命中索引、回表、索引失效。

## 标准回答

MySQL 索引本质是用有序数据结构降低查询扫描成本。InnoDB 常见索引以 B+ 树组织，主键索引叶子节点存整行数据，二级索引叶子节点存主键值；查询不能覆盖索引时通常需要回表。索引设计要围绕高频查询、选择性、排序/分组、范围条件和写入成本综合权衡。

## 深挖追问

1. 为什么联合索引遵循最左前缀？因为 B+ 树按联合键字典序排序。
2. 覆盖索引一定最快吗？不一定，还要看扫描行数、过滤性和维护成本。
3. 索引越多越好吗？不是，写入和更新都要维护索引。

## 实战场景 / SQL 示例

```sql
EXPLAIN SELECT id, name FROM user WHERE status = 1 AND created_at >= ? ORDER BY created_at DESC LIMIT 20;
-- 可评估联合索引：(status, created_at)，若需要避免回表再考虑覆盖字段。
```

## 易错点 / 总结

- 不要认为建了索引就一定会走索引；优化器会按成本选择。
- 联合索引遇到范围条件后，后续列通常难以继续用于精确定位。
- 低选择性字段、频繁更新字段要谨慎建索引。
