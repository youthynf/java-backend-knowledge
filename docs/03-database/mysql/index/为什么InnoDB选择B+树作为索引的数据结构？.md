# 为什么 InnoDB 选择 B+ 树作为索引的数据结构

## 核心概念

数据库索引的核心矛盾是"磁盘 IO 慢且贵"。磁盘随机 IO 的延迟是内存访问的 10 万倍（机械盘约 10ms vs 内存 100ns），所以索引数据结构的核心目标不是 CPU 计算复杂度低，而是**让磁盘 IO 次数尽量少**。

B+ 树之所以胜出，是因为它的设计极度契合磁盘特性：非叶子节点只存键和指针（不存数据），单页能容纳上千个目录项，3 层就能覆盖千万行数据，意味着 3 次磁盘 IO 就能找到任意一条记录。同时叶子节点通过链表串联，范围查询可以顺序扫描，排序也能直接利用索引有序性。哈希、B 树、红黑树、跳表都各有优劣，但在 OLTP 这个场景下综合性能都不如 B+ 树。

## 标准回答

InnoDB 选 B+ 树是因为它最适合磁盘存储：非叶子节点只存键+指针，单页容纳上千目录项，3 层覆盖千万行，IO 次数少；叶子节点存数据并通过双向链表串联，范围查询和排序高效；查询性能稳定（每次都要走到叶子节点）。B 树非叶子节点也存数据导致树更高，Hash 不支持范围查询和排序，红黑树/二叉树高度过大，跳表在磁盘场景下不如 B+ 树的"页内集中存储"友好。

## 候选方案对比

### B+ 树 vs B 树

| 维度 | B+ 树 | B 树 |
|------|-------|------|
| 非叶子节点 | 只存键+指针 | 存键+数据+指针 |
| 单页能容纳的键数 | 多（约 1170 个，假设 Bigint+4B 页号） | 少（数据占空间） |
| 树高度 | 矮（3 层覆盖千万级） | 高 |
| 范围查询 | 顺着叶子链表扫描，高效 | 需要中序遍历，每次跳节点都是 IO |
| 查询稳定性 | 都走到叶子，稳定 | 可能在非叶子节点命中，性能不稳定 |
| IO 次数 | 3 层 = 3 次 | 3 层数据规模下，需要更多层 |

B+ 树的关键优势：非叶子节点不存数据 → 单页键数多 → 树更矮 → IO 次数少。这是 B+ 树胜过 B 树的根本原因。

### B+ 树 vs 哈希索引

| 维度 | B+ 树 | 哈希 |
|------|-------|------|
| 等值查询 | O(log N)，约 3 次 IO | O(1)，1 次 IO |
| 范围查询 | 支持，叶子链表扫描 | 不支持 |
| 排序 | 索引本身有序 | 不支持 |
| 模糊匹配 | 支持 `LIKE 'x%'` | 不支持 |
| 哈希冲突 | 不存在 | 存在，需链表解决 |

哈希等值快，但范围、排序、前缀匹配全不行。OLTP 业务大量涉及 `WHERE a > ?`、`ORDER BY b`、`LIKE 'x%'`，哈希索引覆盖不了。所以 InnoDB 不直接支持哈希索引，只在内存里对热点页建自适应哈希（AHI）。

### B+ 树 vs 红黑树 / 二叉搜索树

红黑树是二叉平衡树，每个节点最多 2 个子节点。N 个节点的红黑树高度约 `log2(N)`，1000w 数据需要 `log2(10^7) ≈ 24` 层。每层一次磁盘 IO，意味着 24 次 IO，远超 B+ 树的 3 次。

二叉树的根本问题是"扇出太低"——每页只能存两三个键，导致树过高。B+ 树把节点放大到 16KB 一页，扇出可达上千，把树高从 24 压到 3。

### B+ 树 vs 跳表

跳表（Skip List）是 Redis ZSet 的底层结构，多层链表实现近似 O(log N) 查询。

| 维度 | B+ 树 | 跳表 |
|------|-------|------|
| 节点大小 | 一页 16KB，存大量键 | 一个节点存一个键 |
| 磁盘友好性 | 高（一次 IO 读一页） | 低（节点小，IO 频繁） |
| 实现复杂度 | 复杂（页分裂、合并、平衡） | 简单（随机层数） |
| 并发控制 | 复杂（页锁） | 简单（CAS） |
| 适用场景 | 磁盘存储（InnoDB） | 内存存储（Redis） |

跳表在内存场景下表现优秀（Redis ZSet、LevelDB MemTable 都用跳表），但磁盘场景下"节点小导致 IO 多"是硬伤。B+ 树的"页内集中存储"对磁盘顺序 IO 更友好。

## InnoDB B+ 树的关键设计

### 1. 节点 = 数据页 = 16KB

B+ 树每个节点是一个 16KB 的数据页，与 InnoDB 的 IO 单位一致。一次 IO 读一页，正好读到一个节点的全部内容。

### 2. 非叶子节点只存键 + 子页号

假设 Bigint 主键 8B + 页号 4B = 12B/项，单页可容纳约 `(16384 - ~150 头尾) / 12 ≈ 1170` 个目录项。这意味着每个非叶子节点可以指向 1170 个子页。

### 3. 三层 B+ 树能存多少行

- 第 1 层（根节点）：1 页，1170 个目录项
- 第 2 层：1170 页，每页 1170 目录项 → 1170 × 1170 = 1368900 个目录项
- 第 3 层（叶子）：1368900 个叶子页，假设每页存 15 行（行大小约 1KB）→ 1368900 × 15 ≈ 2000w 行

所以**3 层 B+ 树能存约 2000w 行**，这是单表 2000w 经验值的来源。详见 [MySQL 单表为什么不要超过 2000w 行](MySQL单表为什么不要超过2000w行？.md)。

### 4. 叶子节点双向链表

叶子节点通过 `FIL_PAGE_PREV` / `FIL_PAGE_NEXT` 指针双向连接。范围查询 `WHERE id BETWEEN 100 AND 200` 定位到 `id=100` 后顺着链表向右扫即可，无需回根节点重走 B+ 树。

### 5. 查询性能稳定

B+ 树的所有数据都在叶子节点，每次查询都要从根走到叶子，IO 次数恒定等于树高。B 树可能在非叶子节点命中数据，性能波动大。稳定的查询性能对 OLTP 更重要（避免偶发慢查询）。

## 代码示例

估算一棵 B+ 树的容量：

```sql
-- 查看页大小
SHOW VARIABLES LIKE 'innodb_page_size';
-- 16384 字节（16KB）

-- 查看表的索引层数（8.0）
SELECT
  table_name, index_name,
  stat_name, stat_description, stat_value
FROM mysql.innodb_index_stats
WHERE database_name = 'mydb' AND table_name = 'orders'
  AND stat_name = 'tree_height';
-- tree_height = 3 表示 B+ 树高度为 3
```

观察 B+ 树节点分裂：

```sql
-- 大量插入后看索引统计
ANALYZE TABLE orders;
SHOW INDEX FROM orders;
```

## 实战场景

| 场景 | B+ 树如何发挥优势 |
|------|------------------|
| 主键点查 | 3 层 = 3 次 IO 命中，根节点常驻 Buffer Pool 实际只 1~2 次 IO |
| 范围查询 | 叶子链表顺序扫描 |
| `ORDER BY` | 索引本身有序，避免 `Using filesort` |
| 联合索引最左前缀 | B+ 树按联合键字典序排列，前缀有序 |
| `LIKE 'java%'` | 前缀匹配本质是范围查询 `(java, javb)` |

## 深挖追问

### 既然 B+ 树这么好，为什么 Redis 用跳表不用 B+ 树？

Redis 是纯内存数据库，没有"磁盘 IO 慢"的问题。跳表实现简单、并发友好（CAS 即可）、范围查询高效，且内存场景下跳表的"节点小"不再是劣势。B+ 树的页结构是为磁盘 IO 优化的，在内存里反而是过度设计。

### 为什么 MongoDB 用 B 树而不是 B+ 树？

MongoDB 早期使用 B 树（WiredTiger 引擎实际是 B+ 树变种），原因是文档型数据库单文档较大，B 树在非叶子节点直接存数据可以让某些点查一次命中，省去走到叶子的开销。但范围查询不如 B+ 树。后续 WiredTiger 也采用 B+ 树变种（带行存页）。

### Hash 索引完全没用吗？

不是。等值查询场景下 Hash 比 B+ 树快（O(1) vs O(log N)）。Memory 引擎默认哈希索引用于内存临时表；InnoDB 的自适应哈希索引（AHI）会在内存里对热点页自动建哈希，等值查询走哈希更快。但作为持久化的用户索引，Hash 局限太大。

### 4 层 B+ 树能存多少？

4 层 = 1170 × 1170 × 1170 × 15 ≈ 240 亿行。但 4 层意味着 4 次 IO，性能下降明显，所以经验上"3 层是合理的极限"，对应约 2000w 行。这也是为什么单表超过 2000w 就要考虑分表。

## 易错点

- 误以为"B 树就是 B+ 树"——B 树非叶子节点也存数据，B+ 树不存。
- 误以为"Hash 索引一定比 B+ 树快"——只对等值查询，范围、排序全废。
- 误以为"B+ 树高度永远是 3"——取决于数据量、主键大小、行大小，超过 2000w 通常到 4 层。
- 误以为"叶子节点单向链表"——InnoDB 是双向链表。
- 误以为"跳表磁盘也合适"——跳表节点小，磁盘 IO 频繁，不如 B+ 树页式存储。

## 总结

InnoDB 选 B+ 树是因为它最适合磁盘存储：非叶子节点只存键+指针使树矮（3 层覆盖 2000w 行），叶子节点双向链表支持范围查询和排序，查询性能稳定。B 树非叶子也存数据导致树高，Hash 不支持范围/排序，红黑树扇出太低，跳表节点太小不适合磁盘。理解"为什么是 B+ 树"，关键看"磁盘 IO 次数"。
