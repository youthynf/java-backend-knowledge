# 索引是什么

## 核心概念

索引是数据库为了加速查询而维护的"额外数据结构"，可以类比为书的目录：没有目录时找一个章节要翻整本书，有了目录直接查目录定位到页码。MySQL InnoDB 默认使用 B+ 树作为索引结构，叶子节点按主键顺序组织，每次查询通过几次磁盘 IO 就能命中目标记录，而不是全表扫描。

索引的代价是空间和维护成本：每个索引都是一棵独立的 B+ 树，占用磁盘；表的 INSERT/UPDATE/DELETE 需要同步维护所有索引，索引越多写入越慢。所以索引设计是"读性能 vs 写性能 + 存储"的权衡，不是越多越好。

## 标准回答

索引是用空间换时间的数据结构，本质是在表数据之外维护一份有序的副本，让查询不必全表扫描。InnoDB 默认用 B+ 树，主键索引（聚簇索引）叶子节点直接存整行数据，二级索引叶子节点存主键值，查非索引字段需要"回表"。索引能加速等值、范围、排序、分组，但会增加写入成本和存储空间，要按查询条件、选择性、排序需求综合设计。

## 索引解决什么问题

没有索引时，`SELECT * FROM user WHERE id = 100` 必须从第一行扫到最后一行，复杂度 O(N)。100w 行的表每次查询都扫一遍，磁盘 IO 是不可接受的。

有了 B+ 树索引后，查询变成"从根节点二分下行到叶子节点"，3 层 B+ 树只需 3 次磁盘 IO，复杂度 O(log N)。这是从"线性扫描"到"对数级定位"的本质提升。

## InnoDB 索引的工作模型

```
            聚簇索引（主键索引）          二级索引（如 idx_name）
                  ↓                          ↓
              B+ 树                        B+ 树
                ...                         ...
              [叶子]                      [叶子]
          存整行数据                  存 (name, 主键值)

查询 SELECT * FROM user WHERE name = '张三':
  1. 在 idx_name 的 B+ 树找到 name='张三' → 得到主键 id=42
  2. 回到聚簇索引的 B+ 树，按 id=42 找到整行数据   ← 这一步叫"回表"
```

如果查询字段都在二级索引里（如 `SELECT id, name FROM user WHERE name='张三'`），二级索引叶子节点已经包含 id 和 name，不需要回表，称为"覆盖索引"。

## 索引的代价

| 代价 | 说明 |
|------|------|
| 空间 | 每个索引一棵 B+ 树，叶子节点存索引列 + 主键。索引多占用可达数据本身的几倍 |
| 写入 | INSERT/UPDATE/DELETE 需要维护所有受影响的索引，可能引起页分裂 |
| 优化器选择 | 索引多时优化器选错索引反而变慢，需 `OPTIMIZER_TRACE` 排查 |
| 维护 | 索引碎片区过多需 `OPTIMIZE TABLE` 重建 |

## 代码示例

```sql
-- 创建普通索引
CREATE INDEX idx_name ON user(name);

-- 创建联合索引
CREATE INDEX idx_status_created ON orders(status, created_at);

-- 查看表的索引
SHOW INDEX FROM orders;

-- 用 EXPLAIN 验证是否走索引
EXPLAIN SELECT * FROM user WHERE name = '张三';
-- 关注 type、key、rows、Extra
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 订单列表分页 | `(user_id, created_at)` 联合索引 + 覆盖 `id, order_no` | 范围条件放在联合索引右边 |
| 用户登录 | `email` 唯一索引 | 既是查询条件也是唯一约束 |
| 标签搜索 | 频繁 `LIKE 'java%'` 可走前缀索引；`LIKE '%java%'` 走不了 | 考虑全文索引或搜索引擎 |
| 报表排序 | ORDER BY 字段加入联合索引末尾 | 避免 `Using filesort` |

## 深挖追问

### 为什么 MySQL 常用 B+ 树而不是 B 树、Hash、红黑树？

B+ 树非叶子节点只存键和指针，单页能容纳上千个目录项，3 层就能覆盖千万行；叶子节点通过链表串联，范围查询和排序高效。Hash 等值快但范围查不行；红黑树是二叉树，高度过高；B 树非叶子节点也存数据，单页键数少、树更高。详见 [为什么 InnoDB 选择 B+ 树](为什么InnoDB选择B+树作为索引的数据结构？.md)。

### 索引一定能让查询变快吗？

不一定。优化器是基于成本选择执行计划的，如果索引选择性差（如性别字段），扫描行数接近全表，优化器可能选全表扫描更快。另外索引维护成本会让写入变慢，对写多读少的表反而拖累整体。

### 聚簇索引和二级索引的本质区别是什么？

聚簇索引叶子节点存整行数据，一张表只能有一个；二级索引叶子节点存主键值，可以建多个。查二级索引后要拿到完整行数据需要"回表"到聚簇索引。详见 [聚簇索引和二级索引是什么](聚簇索引和二级索引是什么？.md)。

## 易错点

- 误以为"建了索引就一定走"——优化器会按成本选择，索引选择性差或扫描行数多时可能不走。
- 误以为"索引越多越好"——索引越多写入越慢、空间越大、优化器选错概率越高。
- 误以为"主键就是聚簇索引"——InnoDB 中主键是聚簇索引的依据，但聚簇索引也可以是第一个非空唯一索引，或隐式 `DB_ROW_ID`。
- 误以为"`SELECT *` 性能差不多"——`SELECT *` 几乎一定回表，覆盖索引失效。
- 误以为"LIKE 都走不了索引"——`LIKE 'java%'`（前缀匹配）能走索引，`LIKE '%java'` 才走不了。

## 总结

索引是用空间换时间的有序数据结构，InnoDB 默认 B+ 树，主键索引（聚簇）存整行数据，二级索引存主键值需要回表。索引能大幅降低查询复杂度，但增加写入和存储成本，要按查询条件、选择性、排序需求综合设计。建索引后必须用 `EXPLAIN` 验证执行计划，避免"建了但不走"。
