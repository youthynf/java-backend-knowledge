# 索引是什么？

## 核心概念

索引是数据库为了加速查询而维护的数据结构，类似书的目录。MySQL InnoDB 最常见的是 B+Tree 索引，也支持全文索引、空间索引等。索引可以减少扫描行数、降低排序成本、提升 Join 效率，但会占用存储空间，并增加写入、更新和删除时的维护成本。

InnoDB 主键索引是聚簇索引，叶子节点保存整行数据；普通二级索引叶子节点保存索引列和主键值，查询非索引字段时需要“回表”。如果查询字段都在二级索引中，就是覆盖索引，可以减少一次主键查询。

## 面试官想考什么

- 是否理解索引的作用和代价；
- 是否知道 InnoDB B+Tree、聚簇索引、二级索引、回表；
- 是否能解释最左前缀、覆盖索引、索引下推；
- 是否能结合慢 SQL 分析如何建索引。

## 标准回答

> 索引是用空间换时间的数据结构。InnoDB 主要使用 B+Tree 索引，适合等值、范围、排序和前缀匹配。主键索引叶子节点保存整行数据，二级索引保存主键值，所以查到二级索引后可能还要回表。建立索引要看查询条件、排序、分组、基数和写入频率，不是越多越好。

## 深挖追问

### 为什么 MySQL 常用 B+Tree？

B+Tree 高度低，磁盘 IO 次数少；非叶子节点只存键和指针，能容纳更多索引项；叶子节点有序并通过链表连接，适合范围扫描和排序。

### 什么情况下索引可能失效？

对索引列使用函数或表达式、发生隐式类型转换、`LIKE '%xxx'` 前缀不固定、联合索引不满足最左前缀、低选择性字段单独建索引、统计信息不准等，都可能导致优化器不使用索引。

## 实战场景 / SQL 示例

订单列表常按用户、状态、时间分页：

```sql
SELECT id, order_no, amount, created_at
FROM orders
WHERE user_id = 1001 AND status = 1
ORDER BY created_at DESC
LIMIT 20;
```

可建立联合索引：

```sql
CREATE INDEX idx_user_status_time ON orders(user_id, status, created_at DESC);
```

如果查询字段都包含在索引中，还可以形成覆盖索引，减少回表。

## 易错点 / 总结

- 索引不是越多越好，写入和更新也要维护索引；
- 联合索引顺序要结合等值、范围、排序综合设计；
- 范围条件后的列通常难以继续用于有序定位；
- 建索引后要用 `EXPLAIN` 验证扫描行数和访问类型；
- 大字段、低基数字段不适合盲目单列索引。

## 面试版详细讲解

### 核心概念

这道题属于 **MySQL 索引** 的高频考点，核心要抓住：B+ 树、聚簇索引、二级索引、联合索引、覆盖索引。索引用空间和写入维护成本换查询效率。联合索引应结合等值、范围、排序/分组和覆盖查询设计，不是越多越好。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：B+ 树定位过程、最左前缀、回表、索引下推、索引失效和 EXPLAIN；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

索引用空间和写入维护成本换查询效率。联合索引应结合等值、范围、排序/分组和覆盖查询设计，不是越多越好。

答题时建议用“三段式”：

1. 先给结论，明确适用前提；
2. 再解释底层机制或执行过程；
3. 最后补充业务取舍、风险点和排查手段。

### 深挖追问

- 这个结论在高并发或大数据量下是否仍然成立？
- 它依赖哪些版本、配置、索引/编码或业务一致性要求？
- 线上异常时应该看哪些命令、日志、指标或执行计划？

### 示例 / 实战场景

慢 SQL 先看 where/order by/group by/返回列，再用 EXPLAIN 看 type、key、rows、Extra，必要时用覆盖索引或调整列顺序。

```sql
EXPLAIN SELECT * FROM your_table WHERE biz_id = ? ORDER BY created_at DESC LIMIT 20;
-- 关注 type/key/rows/Extra，确认是否命中合适索引
```

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
