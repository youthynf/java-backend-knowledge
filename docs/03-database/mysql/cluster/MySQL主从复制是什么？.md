# MySQL主从复制是什么？

## 核心概念

MySQL主从复制是什么？
MySQL的主从复制依赖于binlog，也就是记录MySQL上的所有变化，并以二进制形式保存在磁盘上。复制的过程就是将binlog中的数据从主库传输到从库上。这个过程一般是异步的，也就是主库上执行事务操作的线程，不会等待复制binlog的线程同步完成。
image.png

MySQL集群的主从复制过程主要是3个阶段：
写入binlog：MySQL主库收到客户端提交事务请求后，会先写入binlog，再提交事务，更新存储引擎中的数据，事务提交完成后，返回客户端操作成功的响应；
同步binlog：从库会创建一个专门的I/O线程，连接主库的Binlog Dump线程，来接收主库的binlog日志，再把binlog日志信息写入relay log的中继日志中，再返回给主库复制成功的响应；
回放binlog：从库会创建一个用于回收binlog的线程，去读取relay log中继日志，然后回放binlog更新存储引擎中的数据，最终实现主从的数据一致性。

主从延迟都有什么处理方法：
强制走主库方案：对于大失误或者资源密集型操作，直接再主库上执行，避免从库的额外延迟。
选择性读主：智能选择是否强制读主。利用Redis作为缓存，如果修改了数据，缓存该数据为“脏”，并设置较短的超时时间，当读取数据时，优先判断缓存中是否有“脏”标识，如果有则说明数据刚刚修改过，强制读主，否则可以安全路由到从服务器，实现负载均衡。

## 面试官想考什么

- 复制、选主、故障转移、分片、一致性的取舍。
- 主从延迟、脑裂、数据丢失窗口和读写分离风险。
- 能否给出监控、限流、降级和恢复方案。

## 标准回答

MySQL 高可用的核心是复制链路和一致性取舍。主从复制通常依赖 binlog，异步复制存在延迟和故障丢失窗口；读写分离能扩展读能力但要处理读旧数据。分库分表能解决容量和写入瓶颈，但会引入路由、全局 ID、跨库查询和事务复杂度。

## 深挖追问

1. 主从延迟如何处理？关键读走主库、监控延迟、等待位点或降级。
2. 读写分离有什么坑？写后读从库可能读到旧数据。
3. 分库分表最难是什么？跨库查询、事务、全局 ID、扩容迁移。

```sql
SHOW REPLICA STATUS\G
-- 关键写后读：读主库、等待复制位点，或在业务上接受最终一致。
```

## 易错点 / 总结

- 异步复制不等于强一致。
- 读写分离要处理写后读一致性。
- 扩容、迁移、故障切换必须提前演练。

## 面试版详细讲解

### 核心概念

这道题属于 **MySQL 集群** 的高频考点，核心要抓住：主从复制、读写分离、分库分表、高可用。主从复制提升读能力和可用性，但异步复制存在延迟；分库分表解决容量/写入瓶颈，同时带来跨库查询、事务和运维复杂度。

### 面试官想考什么

面试官通常不是只想听定义，而是想确认你能否说明：binlog 复制链路、主从延迟、读己之写、分片路由和全局 ID；还能否把它和真实业务里的性能、可靠性、可维护性联系起来。

### 标准回答

主从复制提升读能力和可用性，但异步复制存在延迟；分库分表解决容量/写入瓶颈，同时带来跨库查询、事务和运维复杂度。

答题时建议用“三段式”：

1. 先给结论，明确适用前提；
2. 再解释底层机制或执行过程；
3. 最后补充业务取舍、风险点和排查手段。

### 深挖追问

- 这个结论在高并发或大数据量下是否仍然成立？
- 它依赖哪些版本、配置、索引/编码或业务一致性要求？
- 线上异常时应该看哪些命令、日志、指标或执行计划？

### 示例 / 实战场景

下单后立即查询要读主库或做会话一致性；报表/列表可读从库，分表前先确认单表瓶颈真实存在。

```sql
EXPLAIN SELECT * FROM your_table WHERE biz_id = ? ORDER BY created_at DESC LIMIT 20;
-- 关注 type/key/rows/Extra，确认是否命中合适索引
```

### 易错点

- 只背概念，不说明适用场景、代价和边界。
- 忽略数据量、并发量、版本差异和线上配置，给出绝对化结论。
- 没有把问题落到可观测手段：执行计划、慢日志、监控指标、客户端超时或错误日志。

### 一句话总结

这类题的面试核心不是“知道名词”，而是能说清 **机制 + 取舍 + 落地排查**。先给稳定结论，再讲底层原因，最后结合业务场景说明如何使用和如何避免坑。
