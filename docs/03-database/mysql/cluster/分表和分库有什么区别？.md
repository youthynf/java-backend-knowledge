# 分表和分库有什么区别

## 核心概念

单库单表在数据量和并发量上来后会遇到瓶颈：单表数据千万级查询变慢、单库连接数打满、单机 IO/CPU 饱和。分表和分库是水平扩展的两种手段，分别解决不同层面的瓶颈。

分表是把一张大表拆成多个结构相同的小表，解决单表数据量过大导致的查询性能下降。分库是把数据按规则分散到多个数据库实例，解决单库并发连接和 IO 瓶颈。两者可以垂直拆分（按字段/业务）或水平拆分（按行/数据）。

拆分能解决容量和性能问题，但引入新复杂度：跨库 JOIN、分布式事务、全局 ID、路由、扩容迁移。是否拆分、怎么拆分要看业务规模和团队能力，过早拆分会过度设计。

## 标准回答

> 分表解决单表数据量大、查询慢，分库解决单库并发连接和 IO 瓶颈。两者都可垂直或水平拆分：垂直分库按业务拆库（专库专用）、垂直分表按字段拆表（热冷字段分离）、水平分库按规则把同表数据分到多库、水平分表按规则把同表数据分到同库多表。拆分后引入跨库 JOIN、分布式事务、全局 ID、路由、扩容迁移等复杂度，通常用 ShardingSphere、MyCat 等中间件管理。生产建议先优化索引和 SQL，单表千万级以上、单库连接打满再考虑拆分。

## 实现原理

### 四种拆分方式

**垂直分库**：按业务维度拆分到不同数据库实例。如电商拆为用户库、订单库、商品库、支付库，专库专用。

```
单库 → 用户库（user_db）
       订单库（order_db）
       商品库（product_db）
       支付库（pay_db）
```

解决单库连接数和资源瓶颈，业务隔离清晰。但不减少单表数据量。

**垂直分表**：把一张宽表按字段拆成多张表，热字段和冷字段分离。

```
用户表（100 字段）→ 用户主表（id, name, phone 等高频字段）
                  → 用户详情表（id, intro, avatar 等低频字段）
```

数据库以行为单位加载页，宽表一行很大，一页加载的行数少。垂直分表后热表行小，一页能装更多行，索引命中率和 IO 效率提升。

**水平分库**：同一张表按规则分散到多个数据库实例。

```
订单表 → order_db_0.orders (user_id % 4 == 0)
       → order_db_1.orders (user_id % 4 == 1)
       → order_db_2.orders (user_id % 4 == 2)
       → order_db_3.orders (user_id % 4 == 3)
```

解决单库数据量和写入瓶颈，分散到多机。但跨库 JOIN、聚合查询困难。

**水平分表**：同一张表在同一个库内拆成多个结构相同的表。

```
订单表 → orders_0 (user_id % 4 == 0)
       → orders_1 (user_id % 4 == 1)
       → orders_2 (user_id % 4 == 2)
       → orders_3 (user_id % 4 == 3)
```

解决单表数据量大，但仍受单库连接和 IO 限制（还在一个实例上）。是水平分库的“前置步骤”或轻量版。

### 分片键与路由

水平拆分要选分片键（sharding key），决定数据落到哪个分片：

| 分片键 | 示例 | 特点 |
|--------|------|------|
| 用户 ID | `user_id % 4` | 适合 toC 业务，按用户隔离 |
| 订单 ID | `order_id % 8` | 适合订单类，需全局 ID |
| 时间 | 按月分表 | 适合日志类，按时序 |
| 取模/范围 | `id % N` 或 `id 范围` | 取模均匀但扩容难，范围易扩容但易热点 |

路由规则决定查询怎么找到分片。带分片键的查询直接路由到对应分片；不带分片键的查询要广播到所有分片（性能差）。

### 全局 ID

分库分表后 MySQL 自增主键失效（多个分片会冲突），需要全局唯一 ID 生成方案：

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| UUID | 本地生成 | 简单，但无序、占空间、索引差 |
| 雪花算法 Snowflake | 时间戳+机器ID+序列号 | 有序、高性能、依赖时钟 |
| 号段模式 | 数据库批量发号 | 简单、性能好、依赖数据库 |
| Redis INCR | Redis 自增 | 简单、依赖 Redis 可用性 |

生产常用雪花算法或美团 Leaf、滴滴 Tinyid 等号段模式。

### 跨库 JOIN 与聚合

分库后跨库 JOIN 失效，处理方法：

- **业务层组装**：分别查各库，应用层 JOIN
- **冗余字段**：常用关联字段冗余到各库
- **数据同步**：用 binlog 同步到 ES/数仓做聚合查询
- **绑定表**：经常 JOIN 的表用相同分片键，路由到同库

聚合查询（COUNT、SUM）要发到所有分片汇总，性能差，常用离线数仓或缓存预计算。

### 扩容迁移

取模分片扩容需要 rehash 全部数据（如 % 4 扩到 % 8，所有数据要重分布），成本高。常用“倍扩”策略：4 扩到 8 时，原分片 0 的数据只需分流到 0 和 4，其他分片不变，迁移量减半。一致性哈希也能减少迁移量。

## 代码示例

ShardingSphere-JDBC 配置（Java 应用层分片）：

```yaml
shardingsphere:
  datasource:
    names: ds0,ds1
    ds0: { type: HikariDataSource, jdbcUrl: jdbc:mysql://host0:3306/order_db, ... }
    ds1: { type: HikariDataSource, jdbcUrl: jdbc:mysql://host1:3306/order_db, ... }
  rules:
    sharding:
      tables:
        orders:
          actualDataNodes: ds${0..1}.orders_${0..3}
          databaseStrategy:
            standard:
              shardingColumn: user_id
              shardingAlgorithmName: db_mod
          tableStrategy:
            standard:
              shardingColumn: user_id
              shardingAlgorithmName: table_mod
      shardingAlgorithms:
        db_mod: { type: MOD, props: { sharding-count: 2 } }
        table_mod: { type: MOD, props: { sharding-count: 4 } }
      keyGenerators:
        snowflake: { type: SNOWFLAKE }
```

```java
// 应用层透明使用，分片由 ShardingSphere 处理
@Repository
public interface OrderMapper {
    @Insert("INSERT INTO orders(user_id, amount) VALUES(#{userId}, #{amount})")
    void insert(Order order);

    @Select("SELECT * FROM orders WHERE user_id = #{userId}")
    Order selectByUserId(Long userId);  // 带分片键，直接路由
}
```

## 实战场景

| 场景 | 拆分方式 | 说明 |
|------|---------|------|
| 单表千万级查询慢 | 水平分表 | 按用户 ID 取模拆表 |
| 单库连接打满 | 垂直分库 | 按业务拆库 |
| 宽表 IO 效率低 | 垂直分表 | 热冷字段分离 |
| 订单量大写入瓶颈 | 水平分库 | 多库分担写入 |
| 日志类时序数据 | 按月分表 | `logs_202506` |
| 跨业务聚合查询 | 不拆分+同步数仓 | ES/ClickHouse 做分析 |

## 深挖追问

### 1. 什么时候该分库分表？

不要过早拆分。常见触发点：单表数据超过 1000 万、单表数据文件超过 50GB、单库 QPS 超过 5000、单库连接数经常打满。先优化索引、SQL、加缓存、读写分离，仍不满足再拆分。拆分是不可逆的架构决策，迁移成本极高。

### 2. 分片键怎么选？

选查询最频繁的字段。toC 业务通常选 `user_id`，保证同一用户的数据在同一分片，用户维度查询直接路由。订单类可选 `order_id` 或 `user_id`。避免选低基数字段（如性别）导致数据倾斜。

### 3. 不带分片键的查询怎么办？

要广播到所有分片汇总，性能差。生产中尽量避免：把不带分片键的查询改造成带分片键（如先查用户 ID 再查订单），或把数据同步到 ES/数仓做查询。完全无法避免的用广播查询并接受性能损耗。

### 4. 分库分表后事务怎么办？

跨库事务用分布式事务方案：XA（性能差）、TCC（业务侵入）、Saga（长事务）、本地消息表（最终一致）。实践中多用“本地消息表 + 异步补偿”实现最终一致，避免强一致分布式事务的性能损耗。

### 5. 分库分表和读写分离怎么选？

读写分离扩展读能力，分库分表扩展写能力和数据容量。读压力大先读写分离（成本低），写压力或数据量大再分库分表。两者可组合：每个分片再做主从读写分离。

## 易错点

- 过早分库分表：优化空间还很大就拆，徒增复杂度。
- 分片键选错：查询频繁不带分片键，全广播。
- 忽略扩容成本：取模分片扩容要 rehash 全部数据。
- 跨库 JOIN 失效没预案：业务层组装或冗余字段要提前设计。
- 全局 ID 用自增：多分片主键冲突。

## 总结

分表解决单表数据量大、分库解决单库并发和 IO 瓶颈。两者都可垂直或水平拆分：垂直分库按业务、垂直分表按字段、水平分库按规则多机、水平分表按规则多表。拆分引入跨库 JOIN、分布式事务、全局 ID、路由、扩容迁移等复杂度，通常用 ShardingSphere 等中间件管理。生产建议先穷尽单库优化（索引、SQL、缓存、读写分离），仍不满足再拆分。分片键选择和扩容预案是设计核心。
