# InnoDB 是如何存储数据的

## 核心概念

InnoDB 把数据组织成一个"分层 + 分页"的结构：表空间（Tablespace）→ 段（Segment）→ 区（Extent）→ 页（Page）→ 行（Row）。磁盘和内存之间以"页"为单位交换数据，默认 16KB。理解这套结构是理解 InnoDB 索引、锁、Buffer Pool 工作方式的前提。

为什么以页为单位而不是以行为单位？因为磁盘的随机 IO 极慢（机械盘约 10ms/次），如果每次只读一行，一行一次 IO，整体效率极差。把多行打包成一页，一次 IO 读取 16KB，利用局部性原理，下一次访问相邻行时已经在内存里了，平均 IO 次数大幅下降。

## 标准回答

InnoDB 数据按"表空间 → 段 → 区 → 页 → 行"五层组织。表空间分系统表空间和独立表空间；段分数据段（叶子节点）、索引段（非叶子节点）、回滚段；区是 1MB（64 个连续页）；页是 InnoDB 磁盘 IO 的最小单位，默认 16KB；页内行按主键顺序组成单向链表，并通过页目录（slot）支持二分查找。所有数据页在文件中逻辑连续，不要求物理连续。

## 存储层级

### 表空间（Tablespace）

- **系统表空间**：`ibdata1` 等共享文件，存放数据字典、双写缓冲、change buffer、undo log（5.7 默认在系统表空间，8.0 默认独立）。
- **独立表空间**（file-per-table，5.6+ 默认）：每张表一个 `.ibd` 文件，存放该表的数据和索引。
- **undo 表空间**（8.0 默认独立）：存放 undo log。
- **临时表空间**：存放临时表和排序中间结果。

### 段（Segment）

段是表空间的逻辑划分，一个索引会占用两个段：

- **数据段（leaf segment）**：B+ 树的叶子节点，存放实际行数据。
- **索引段（non-leaf segment）**：B+ 树的非叶子节点，存放目录项。

InnoDB 不是一上来就分配 1MB，而是先从"碎片区"借 32 个页，表大了再申请完整区。这样小表不浪费空间。

### 区（Extent）

区是 InnoDB 分配空间的单位，**1 个区 = 64 个连续页 = 1MB**（页大小 16KB 时）。分配完整区的好处是物理连续，扫描时磁盘顺序读，性能远高于随机读。

### 页（Page）

页是 InnoDB 磁盘 IO 的最小单位，默认 16KB，可通过 `innodb_page_size` 调整为 4K/8K/16K/32K/64K（5.7+）。常见页类型：

- **数据页（FSP_PAGE_TYPE_INDEX / index page）**：存放 B+ 树节点（叶子或非叶子）。
- **undo 页**：存放 undo log。
- **系统页**：存放表空间元信息。
- **inode 页**：存放段信息。
- **blob 页**：存放溢出的大字段。

### 行（Row）

页内的行按主键顺序组成单向链表。每行除了用户字段还有两个隐藏列：`DB_TRX_ID`（最后修改该行的事务 ID）、`DB_ROLL_PTR`（指向 undo log 的回滚指针）。如果表没有主键也没有非空唯一索引，InnoDB 还会生成一个 6 字节的 `DB_ROW_ID` 作为隐式主键。

## 数据页内部结构

一个 16KB 的数据页分为 7 个部分：

```
+---------------------------+
| File Header (38B)         |  页号、前后页指针、表空间ID、LSN
+---------------------------+
| Page Header (56B)         |  页内记录数、槽位数、上下页指针等
+---------------------------+
| Infimum + Supremum (26B)  |  两个虚拟记录，标记页内最小/最大值
+---------------------------+
| User Records              |  用户行数据，按主键单向链表组织
+---------------------------+
| Free Space                |  空闲空间，插入新行时从这里分配
+---------------------------+
| Page Directory (槽目录)   |  每个槽指向一组记录的最大值，二分查找用
+---------------------------+
| File Trailer (8B)         |  校验和、LSN，用于崩溃校验
+---------------------------+
```

### 页内如何快速定位记录

如果每次查记录都从头遍历单向链表，效率太低。InnoDB 把页内记录分组，每组最大记录的偏移量存到 Page Directory 的一个"槽（slot）"里：

- 第一组固定 1 条（Infimum）。
- 最后一组 1~8 条。
- 中间组 4~8 条。

查找时先在槽目录上做**二分查找**定位到槽，再在槽内 4~8 条记录里做线性查找，复杂度从 O(N) 降到 O(log N)。

### 页与页之间如何关联

数据页通过 File Header 中的 `FIL_PAGE_PREV` 和 `FIL_PAGE_NEXT` 两个指针组成**双向链表**。B+ 树叶子节点之间通过这个链表串联，范围查询时可以直接顺着链表扫描，无需回到根节点。

## B+ 树如何组织这些页

```
                    [根节点页 page_level=2]
                   /         |          \
        [非叶页 page_level=1] [非叶页]   [非叶页]
            /     \             |          /   \
       [叶页 lv=0] [叶页 lv=0] [叶页 lv=0][叶页 lv=0]
            ↔         ↔          ↔          ↔
        (双向链表连接，叶子节点存整行数据)
```

- 非叶子节点存放目录项 `(最小主键, 子页号)`，用于二分定位。
- 叶子节点存放完整行数据，按主键顺序通过双向链表连接。
- 树高度通常 3 层，因为 16KB 的页能容纳大量目录项，3 层就能覆盖千万级行数。

## 代码示例

查看页大小：

```sql
SHOW VARIABLES LIKE 'innodb_page_size';
-- 默认 16384（字节），即 16KB
```

查看表的行格式和大小：

```sql
SHOW TABLE STATUS LIKE 'orders'\G
-- 关注 Row_format、Avg_row_length、Data_length、Max_data_length
```

查看页内结构（需要 MySQL 自带的 `innodb_page_info` 工具或 `innochecksum`）：

```bash
# 解析 .ibd 文件，查看页类型分布
innochecksum -P 5 /var/lib/mysql/mydb/orders.ibd
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 大表 `OPTIMIZE TABLE` | 重建表，回收碎片区和页内碎片 | 大表锁表久，建议用 `pt-online-schema-change` |
| 调整页大小 | 8.0 可设 `innodb_page_size=32K` 或 `64K`，对 SSD 大页有时更友好 | 页大小是实例级参数，初始化时设定，不可中途改 |
| 行溢出（TEXT/BLOB） | 行数据超过页大小一半时，溢出部分存到独立的 blob 页 | 大字段会触发溢出，影响查询性能 |
| 排序临时表 | `tmp_table_size` 小于结果集时会落临时表空间 | 注意 SSD 上临时表空间 IO 压力 |

## 深挖追问

### 为什么页默认是 16KB 而不是更大？

16KB 是 InnoDB 在 IO 次数和页内查找成本之间的折中。机械盘一次 IO 通常 4~16KB，16KB 刚好匹配；页太大时页内二分查找的 slot 数变多，且单页加载到内存的开销变大。InnoDB 允许 4K~64K，但 16KB 是经过长期实践验证的默认值。

### 区为什么是 64 个页？

64 × 16KB = 1MB，对应大多数文件系统的块大小（1MB 对齐），便于预分配和顺序 IO。一次申请 1MB 比一次申请一个页更高效，磁盘顺序写性能远高于随机写。

### 表没有主键会怎样？

InnoDB 会优先选第一个所有列都 NOT NULL 的唯一索引作为聚簇索引；都没有则自动生成 6 字节 `DB_ROW_ID` 作为隐式主键，存到聚簇索引里。隐式主键是全局递增的（共享一个计数器），高并发插入时会有争用，且无法通过它定位业务数据，所以**生产表一定要显式定义主键**。

## 易错点

- 误以为 InnoDB 的 IO 单位是行——实际是页，一次 IO 至少 16KB。
- 误以为页之间物理连续——其实是逻辑连续（通过指针），物理上可能很分散。
- 误以为没有主键就没问题——会生成隐式 `DB_ROW_ID`，且全局争用。
- 误以为页大小可动态调整——只能在初始化时设置。
- 误以为区是 IO 单位——区是空间分配单位，IO 单位仍然是页。

## 总结

InnoDB 用"表空间→段→区→页→行"五层组织数据，页是磁盘 IO 最小单位（默认 16KB），区是空间分配单位（1MB），段对应 B+ 树的叶子/非叶子部分。页内行按主键单向链表，并通过 Page Directory 的槽支持二分查找；页与页之间双向链表串联。理解这套结构之后，再看 B+ 树、Buffer Pool、锁就都有了抓手。
