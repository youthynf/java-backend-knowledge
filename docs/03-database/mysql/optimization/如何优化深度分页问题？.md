# 如何优化深度分页问题

## 核心概念

深度分页指 `LIMIT offset, size` 中 `offset` 很大的场景，典型如 `LIMIT 1000000, 10`。这种查询慢的根因是 MySQL 必须扫描并丢弃前 `offset` 行，offset 越大浪费的 IO 和 CPU 越多。

`LIMIT 0, 10` 只需扫 10 行，`LIMIT 1000000, 10` 需要扫 1000010 行再丢弃前 100 万行。即使有索引，扫描百万行仍要时间。如果走非主键索引，还要回表百万次，性能更差。

主流优化方案有三种：**游标法（Seek Method）**用上一页末尾 ID 做下一页查询条件，彻底消除 offset；**覆盖索引 + 延迟关联**先用子查询拿到主键 ID 再回表，减少回表次数；**业务层改造**改"翻页"为"加载更多"，避免大 offset。生产环境最推荐游标法。

## 标准回答

> 深度分页慢是因为 `LIMIT offset, size` 要扫描并丢弃前 `offset` 行。优化方案三种：游标法（`WHERE id > last_id ORDER BY id LIMIT size`）彻底消除 offset，性能最优但要求按主键有序翻页；覆盖索引 + 延迟关联（先子查询取主键再 JOIN 回表）减少回表；业务改造（"加载更多"代替"跳页"）。如果必须支持跳页，迁移到 ElasticSearch 或限制最大页数。

核心要点：

1. **根因**：`LIMIT offset, size` 扫描并丢弃前 offset 行。
2. **方案 1**：游标法，按主键 ID 翻页，消除 offset。
3. **方案 2**：覆盖索引 + 延迟关联，减少回表。
4. **方案 3**：业务改造，"加载更多"代替跳页。
5. **方案 4**：迁移 ES 或限制最大页数。

## 详细机制

### LIMIT 执行机制

```sql
SELECT * FROM orders ORDER BY id LIMIT 0, 10;
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;
```

**主键索引场景**：

- `LIMIT 0, 10`：在主键索引上扫 10 行，返回。
- `LIMIT 1000000, 10`：在主键索引上扫 1000010 行，丢弃前 100 万，返回 10 行。

**二级索引场景**：

```sql
SELECT * FROM orders ORDER BY user_id LIMIT 1000000, 10;
```

- 在 `idx_user_id` 上扫 1000010 行，每行回表到聚簇索引取完整行，丢弃前 100 万，返回 10 行。
- 即使有索引，回表百万次仍然慢。
- 优化器评估成本后可能直接走全表扫描，更糟。

### 方案 1：游标法（Seek Method）

记录上一页末尾的 ID，下一页用 `WHERE id > last_id` 替代 `OFFSET`：

```sql
-- 第一页
SELECT * FROM orders ORDER BY id LIMIT 10;
-- 假设返回的最后一条 id = 10

-- 第二页
SELECT * FROM orders WHERE id > 10 ORDER BY id LIMIT 10;
-- 假设返回的最后一条 id = 20

-- 第三页
SELECT * FROM orders WHERE id > 20 ORDER BY id LIMIT 10;
```

**优势**：

- 每次只扫 `size` 行，无论翻到第几页性能都一样。
- 走主键索引范围扫描，效率最高。

**劣势**：

- 只能顺序翻页（上一页/下一页），不能跳到第 N 页。
- 业务需要保存上一页末尾 ID（URL 参数或前端缓存）。

如果按非主键字段排序（如 `created_at DESC`），游标改为该字段值 + ID 联合条件：

```sql
-- 第一页
SELECT * FROM orders ORDER BY created_at DESC, id DESC LIMIT 10;
-- 末尾记录 created_at = '2026-06-29 10:00:00', id = 100

-- 下一页
SELECT * FROM orders
WHERE created_at < '2026-06-29 10:00:00'
   OR (created_at = '2026-06-29 10:00:00' AND id < 100)
ORDER BY created_at DESC, id DESC
LIMIT 10;
```

### 方案 2：覆盖索引 + 延迟关联

利用二级索引只取主键 ID，再用 JOIN 回表，减少回表次数：

```sql
-- 优化前：扫百万行，每行回表
SELECT * FROM orders ORDER BY user_id LIMIT 1000000, 10;

-- 优化后：子查询只取主键 ID（覆盖索引），再 JOIN 回表
SELECT t1.*
FROM orders t1
INNER JOIN (
  SELECT id FROM orders ORDER BY user_id LIMIT 1000000, 10
) t2 ON t1.id = t2.id;
```

子查询 `SELECT id FROM orders ORDER BY user_id LIMIT 1000000, 10`：

- `SELECT id` 只取主键，二级索引 `idx_user_id` 上就有（叶子节点存主键）。
- 子查询走覆盖索引（`Using index`），不回表。
- 扫 1000010 个索引项，丢弃前 100 万，返回 10 个主键。
- 外层 JOIN 用 10 个主键回表 10 次，性能大幅提升。

**适用场景**：必须支持跳页，不能改业务。

### 方案 3：业务改造

把"翻页 + 跳页"改成"加载更多 / 无限滚动"：

- 移动端瀑布流天然适合游标法。
- 列表页只显示前 N 页，超过则提示"请使用搜索缩小范围"。
- 搜索场景允许用户输入筛选条件，避免翻太深。

这是从根上解决深度分页问题的方式，比 SQL 优化更彻底。

### 方案 4：迁移 ES 或限制最大页数

- 把分页查询迁移到 ElasticSearch，ES 用 `search_after` 实现高效游标分页。
- 业务上限制最大页数（如最多 100 页），超过提示用户加筛选条件。
- 限制单次返回行数上限。

## 代码示例

订单表分页优化前后对比：

```sql
CREATE TABLE `orders` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `order_no` VARCHAR(32) NOT NULL,
  `user_id` BIGINT UNSIGNED NOT NULL,
  `amount` DECIMAL(10,2) NOT NULL,
  `status` TINYINT NOT NULL,
  `created_at` DATETIME NOT NULL,
  KEY `idx_user_id` (`user_id`),
  KEY `idx_user_created` (`user_id`, `created_at`)
) ENGINE=InnoDB;
```

**优化前**：常规分页

```sql
EXPLAIN SELECT * FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 100000, 10;

-- type: ref
-- key: idx_user_created
-- rows: 100010
-- Extra: Using where
-- 实际执行：扫 100010 行，每行回表
```

**优化 1**：游标法（按 created_at + id 翻页）

```sql
-- 第一页
SELECT id, order_no, amount, created_at
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC, id DESC
LIMIT 10;
-- 记下末尾 (created_at='2026-06-29 10:00:00', id=50000)

-- 下一页
EXPLAIN SELECT id, order_no, amount, created_at
FROM orders
WHERE user_id = 1001
  AND (created_at < '2026-06-29 10:00:00'
       OR (created_at = '2026-06-29 10:00:00' AND id < 50000))
ORDER BY created_at DESC, id DESC
LIMIT 10;

-- type: ref
-- key: idx_user_created
-- rows: 10
-- Extra: Using where; Using index
-- 每次只扫 10 行
```

**优化 2**：延迟关联（必须跳页时）

```sql
EXPLAIN SELECT t1.*
FROM orders t1
INNER JOIN (
  SELECT id FROM orders
  WHERE user_id = 1001
  ORDER BY created_at DESC
  LIMIT 100000, 10
) t2 ON t1.id = t2.id;

-- 子查询 Using index（覆盖索引），不回表
-- 外层 JOIN 只回表 10 次
```

## 实战场景

| 场景 | 推荐方案 |
|------|----------|
| 移动端瀑布流、加载更多 | 游标法 |
| 后台管理列表（顺序翻页） | 游标法 |
| 必须支持跳页的列表 | 延迟关联 + 限制最大页数 |
| 多维度筛选 + 分页 | 迁移 ES 用 `search_after` |
| 主键自增表的翻页 | 游标法 `WHERE id > last_id` |
| 按时间倒序的动态流 | 游标法 `WHERE created_at < last_time` |
| 用户订单列表 | 游标法 + 联合索引 `(user_id, created_at)` |

## 深挖追问

### 游标法为什么不能跳页？

游标法依赖"上一页末尾值"作为下一页起点，没有这个值就无法定位。跳页（如直接跳到第 100 页）需要从头扫到第 100 页末尾，等价于 LIMIT offset。所以游标法只支持"上一页/下一页"，不支持"跳到第 N 页"。

### 延迟关联为什么快？

子查询 `SELECT id FROM ... LIMIT offset, size`：

- `SELECT id` 只取主键，二级索引叶子节点上就有，走覆盖索引不回表。
- 扫描索引比扫描聚簇索引快（叶子节点紧凑，IO 少）。
- 子查询只返回 size 个主键。
- 外层 JOIN 用 size 个主键回表 size 次，远少于原本 offset+size 次回表。

### 复合排序游标怎么写？

按 `(created_at DESC, id DESC)` 排序时，下一页条件是"小于上一页末尾值的元组"：

```sql
WHERE (created_at, id) < ('2026-06-29 10:00:00', 50000)
```

MySQL 5.7+ 支持行表达式比较，写法简洁。低版本要拆成 OR 条件。

### LIMIT 1 能优化吗？

`LIMIT 1` 已经是最优。但要注意 `WHERE` 条件是否走索引：如果走索引 `LIMIT 1` 几乎是 O(1)；如果不走索引，仍要全表扫到第一条匹配。

### 为什么有些场景深度分页反而走全表扫描？

优化器基于成本估算。当 offset 很大时，"扫聚簇索引全表 + 排序"可能比"扫二级索引 offset+size 次 + 回表 offset+size 次"成本更低，所以优化器选了全表扫描。`EXPLAIN` 看 `key=NULL`、`type=ALL` 即可确认。延迟关联能让子查询走覆盖索引，避开这个问题。

## 易错点

- 用游标法时排序字段有重复值，导致漏数据或重复数据。解决：游标加 ID 做唯一性兜底。
- 延迟关联的子查询忘了 `SELECT id`，回表百万次仍慢。
- 业务坚持支持"跳页"却不限制最大页数，深度分页拖垮 DB。
- 用 `LIMIT 1000000, 10` 测试以为没事，生产数据量翻倍后突然慢。
- 排序字段没建联合索引，触发 filesort + 大量回表。
- 游标法翻页条件写错（`<` vs `<=`），导致边界行重复或丢失。

## 总结

深度分页慢的根因是 `LIMIT offset, size` 要扫描并丢弃前 offset 行。三种主流优化：游标法（消除 offset，性能最优但只能顺序翻页）、延迟关联（子查询走覆盖索引减少回表，支持跳页）、业务改造（"加载更多"代替"跳页"，从需求侧解决）。多维度筛选场景迁移到 ES 用 `search_after`。生产环境优先用游标法，必须跳页用延迟关联 + 限制最大页数。
