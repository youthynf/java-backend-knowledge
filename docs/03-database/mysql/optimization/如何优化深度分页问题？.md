# 如何优化深度分页问题？

## 核心概念

如何优化深度分页问题？
limit 的执行过程
MySQL内部分为server层和存储引擎层，一般情况下存储引擎都用innodb。执行器是用于跟存储引擎打交道的组件，执行器可以通过调用存储引擎提供的接口，将一行行数据取出，当这些数据完全符合要求时，则会放到结果集中，最后返回给调用MySQL的客户端。结果集返回给客户端的两种模式：
流式（增量）返回：当结果集较大时（默认），server层分批发送数据，执行器通过调用引擎接口，逐行获取数据，每当一行数据符合筛选条件，执行器就会将其写入网络发送缓存（net_buffer），默认是积累到 net_buffer_length（默认16KB）才发送，减少内存占用；
缓冲返回：当需要排序/分组/临时表时，必须将所有结果存储在内存/磁盘，完成后才返回。

基于主键索引的limit执行过程

select * from t_user order by id limit 0, 10;
这里limit 0,10 底层实现是server层会调用innodb的接口，在innodb的主键索引中获取到第0到10的完整行数据，依次返回给server层，并放到server层的结果集中，返回给客户端。而limit 6000000,10时，server层会调用innodb接口，在innodb里的主键索引获取第0到（6000000+10）条完整行数据，返回给server层之后，根据offset的值挨个抛弃，最后只留下最后面的size条，也就是10条数据，放到server层的结果集中，返回给客户端。

因此，mysql查询中limit 1000,10会limit 10慢，原因是limit 1000,10会取出1000+10条数据，并抛弃前1000条，这部分耗时更大。

优化方式：
核心思想是减少无用数据的获取：在返回server层的过程中，从需要拷贝数据行的所有列，改造成只需要获取数据行的id这一列，当数据量较大时，这部分耗时优化比较明显。

select * from t_user where id >= (select id from page order by id limit 6000000, 1) order by id limit 10

基于非主键索引的limit执行过程

select * from t_user order by user_name limit 0,10
server层会调用innodb的接口，在innodb里的非主键索引中获取到第0条数据对应的主键id后，回表到主键索引中找到对应完整行数据，然后返回给server层，server层将其放入结果集中，返回给客户端。而当offse>0时，且offset的值较小时，逻辑也类似，区别在于offset>0时会丢弃前面的offset条数据。相当于，非主键索引的limit过程，比主键索引的limit过程，多了个回表的消耗。当limit offset过大时，非主键索引查询非常容易（优化器分析决定）变成全表扫描。

优化方式：
核心思想是避免回表操作：通过二级索引只获取主键ID，然后通过逐渐索引进行主键ID过滤，避免回表。

select * from t_user t1, (select id from t_user order by user_name limit 6000000, 10)) t2 where t1.id = t2.id

深度分页问题优化思路
优化返回字段，比如通过子查询返回主键ID或内连接方式，然后通过ID进行部分数据查询：

-- 优化前
SELECT * FROM t_user ORDER BY user_name LIMIT 6000000, 10;
-- 优化后：利用索引覆盖拿到 ID，再内连接
SELECT * FROM t_user t1, 
    (SELECT id FROM t_user ORDER BY user_name LIMIT 6000000, 10) t2
WHERE t1.id = t2.id;
增加主键ID作为条件过滤，比如根据主键ID进行排序，然后分批次获取，将当前批次最大的ID作为下次筛选的条件进行查询；

-- 下一页查询
SELECT * FROM t_user WHERE id >=6000001 ORDERBY id LIMIT 10;
迁移elasticSearch优化分页和搜索速度；
审视需求合理性，更换分页模式为上下翻页（类似瀑布流）。

## 面试官想考什么

- 能否按慢 SQL、执行计划、索引、数据量、系统资源定位。
- 是否会用 EXPLAIN、慢日志、监控指标闭环验证。
- 是否避免盲目加索引、改参数或缓存一切。

## 标准回答

MySQL 优化建议按“发现慢 SQL → EXPLAIN 分析 → 判断扫描行数/回表/排序/临时表 → 调整 SQL 或索引 → 压测验证”闭环。优化前要确认瓶颈是 CPU、IO、锁等待、网络、连接数还是数据模型。

## 深挖追问

1. EXPLAIN 重点看什么？type、key、rows、filtered、Extra。
2. 慢查询只靠加索引吗？还要看 SQL、数据模型、锁、IO 和缓存。
3. 深分页为什么慢？OFFSET 会扫描并丢弃大量前置行。

## 实战场景 / SQL 示例

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC LIMIT 20;
-- 结合 rows、key、Extra 判断是否需要 (user_id, created_at) 索引。
```

## 易错点 / 总结

- 不要只凭经验优化，要用执行计划和监控验证。
- 加索引会影响写入，需评估整体收益。
- 分页、排序、分组和回表是高频瓶颈。
