# 如何排查和优化慢查询

## 核心概念

慢查询指执行时间超过阈值的 SQL。MySQL 通过 `long_query_time`（默认 10 秒，生产建议 1 秒甚至 0.5 秒）界定慢查询，并写入慢查询日志。排查慢查询是 DBA 和后端开发的日常工作，面试必考。

排查慢查询的标准流程：**发现 → 定位 → 分析 → 优化 → 验证**。先用慢日志/监控发现慢 SQL，用 `EXPLAIN` 看执行计划，从 SQL 写法、索引、表结构、数据量、并发、硬件六个维度分析，最后改完用 `EXPLAIN ANALYZE` + 压测验证。

优化不是只加索引。盲目加索引会拖慢写入、占用磁盘、增加优化器选择成本。正确的优化必须先找到瓶颈：是 SQL 写法差、索引设计不合理、表数据量过大、还是硬件资源不足，对症下药。

## 标准回答

> 慢查询排查按"发现 → 定位 → 分析 → 优化 → 验证"闭环。发现：开启慢查询日志 + APM 监控；定位：`mysqldumpslow` 汇总慢日志找到 TOP N SQL；分析：`EXPLAIN` 看 `type`/`key`/`rows`/`Extra`，定位全表扫描、回表、filesort、临时表等问题；优化：从 SQL 写法、索引、表结构、架构、硬件五个层次下手；验证：`EXPLAIN ANALYZE` + 压测对比。优化前先确认瓶颈，避免盲目加索引。

核心要点：

1. **发现**：慢日志 + APM（SkyWalking/Prometheus/ARMS）。
2. **定位**：`mysqldumpslow` 汇总 TOP N。
3. **分析**：`EXPLAIN` 看 `type`/`key`/`rows`/`Extra`。
4. **优化层次**：SQL → 索引 → 表结构 → 架构 → 硬件。
5. **验证**：`EXPLAIN ANALYZE` + 压测对比。

## 详细机制

### 第一阶段：发现与定位

#### 开启慢查询日志

```sql
-- 查看慢日志配置
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';

-- 动态开启（重启失效）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- 记录未走索引的 SQL
SET GLOBAL log_queries_not_using_indexes = ON;
```

生产环境建议 `long_query_time = 1` 甚至 `0.5`，并开启 `log_queries_not_using_indexes` 抓全表扫描。注意后者会产生大量日志，仅排查阶段开启。

#### APM 监控

慢日志只能发现"已经慢"的 SQL。APM 工具能实时捕获接口耗时和对应 SQL，更早发现趋势异常：

- **SkyWalking**：开源 APM，自动埋点 Java 应用。
- **Prometheus + Grafana**：监控 MySQL 指标（QPS、慢查询数、连接数）。
- **阿里云 ARMS / 腾讯云 APM**：商业方案，开箱即用。
- **pt-query-digest**：Percona 工具，比 `mysqldumpslow` 更强大。

#### 分析慢日志

```bash
# mysqldumpslow 汇总慢日志
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
# -s t 按总耗时排序，-t 10 取 TOP 10

# pt-query-digest 更详细的分析
pt-query-digest /var/log/mysql/slow.log > report.txt
```

输出按"平均耗时/总耗时/调用次数"排序，找出"高频慢 SQL"和"单次极慢 SQL"两类重点。

### 第二阶段：执行计划分析

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1001 ORDER BY created_at DESC LIMIT 20;
```

重点字段：

| 字段 | 关注点 |
|------|--------|
| `type` | `ALL`/`index` 通常需优化；目标 `ref`/`range` 以上 |
| `key` | `NULL` 表示索引失效；与预期索引不符需排查 |
| `rows` | 估算扫描行数，过大需优化 |
| `filtered` | WHERE 过滤后剩余比例，过低说明扫描浪费多 |
| `Extra` | `Using filesort`/`Using temporary` 是性能杀手 |

MySQL 8.0+ 用 `EXPLAIN ANALYZE` 看实际耗时：

```sql
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 1001 ORDER BY created_at DESC LIMIT 20;
```

### 第三阶段：分层优化

#### 1. SQL 与索引层（性价比最高）

```sql
-- 反例：SELECT * + 函数包裹索引列
SELECT * FROM orders WHERE DATE(created_at) = '2026-06-29';

-- 正例：只查必要列 + 范围查询
SELECT id, order_no, amount FROM orders
WHERE created_at >= '2026-06-29 00:00:00'
  AND created_at <  '2026-06-30 00:00:00';
```

- 避免 `SELECT *`，只查需要的列，促成覆盖索引。
- 索引列不要包函数、不要做运算、不要隐式类型转换。
- LIKE 前导通配 `%xxx` 会让索引失效。
- 联合索引遵循最左匹配原则。
- JOIN 保证被驱动表关联字段有索引。
- 深度分页用游标法或延迟关联。

#### 2. 表结构层

- 字段类型合理：金额 `DECIMAL`、状态 `TINYINT`、IP `INT UNSIGNED`。
- 大字段拆副表：`TEXT`/`BLOB` 独立副表，主表只放列表字段。
- 适当反范式：读多写少场景冗余字段减少 JOIN。
- 历史数据归档：冷数据迁移到归档表，主表保持小。
- 分表：单表超千万且索引优化无效时考虑分库分表。

#### 3. 架构层

- **缓存**：Redis 缓存热点数据。读用 Cache Aside（先读缓存，未命中读 DB 后写缓存）；写用"先更 DB 再删缓存"。
- **读写分离**：主写从读，分担读压力。
- **搜索引擎**：复杂搜索/多维筛选迁移到 ElasticSearch。
- **消息队列**：异步落库，削峰填谷。
- **CDN/边缘缓存**：静态资源前置。

#### 4. 硬件层（最后手段）

- SSD 替换 HDD，IO 性能提升数量级。
- 增加内存，扩大 Buffer Pool。
- 升级 CPU，提升单线程处理能力。
- 网络升级（10Gbps）。
- 注意：硬件优化只是"治标"，不改 SQL 和索引，瓶颈迟早再次出现。

## 代码示例

慢查询优化前后对比：

```sql
CREATE TABLE `orders` (
  `id` BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  `order_no` VARCHAR(32) NOT NULL,
  `user_id` BIGINT UNSIGNED NOT NULL,
  `amount` DECIMAL(10,2) NOT NULL,
  `status` TINYINT NOT NULL,
  `created_at` DATETIME NOT NULL,
  KEY `idx_user_status_time` (`user_id`, `status`, `created_at`)
) ENGINE=InnoDB;
```

**优化前**：全表扫描 + filesort

```sql
EXPLAIN SELECT * FROM orders
WHERE DATE(created_at) = '2026-06-29'
ORDER BY amount DESC;

-- type: ALL
-- key: NULL
-- rows: 1000000
-- Extra: Using filesort
```

**优化后**：走索引 + 覆盖索引

```sql
-- 改写 SQL：去掉函数，范围查询
EXPLAIN SELECT id, order_no, amount
FROM orders
WHERE user_id = 1001
  AND status = 1
  AND created_at >= '2026-06-29 00:00:00'
  AND created_at <  '2026-06-30 00:00:00'
ORDER BY created_at DESC;

-- type: ref
-- key: idx_user_status_time
-- key_len: 17 (user_id 8 + status 1 + created_at 5 + NULL 3?)
-- rows: 20
-- Extra: Using where; Using index
```

**JOIN 优化**：

```sql
-- 反例：被驱动表无索引
SELECT u.name, o.order_no
FROM user u, orders o
WHERE u.id = o.user_id AND u.status = 1;

-- 正例：给 orders.user_id 建索引，用 INNER JOIN 显式语义
SELECT u.name, o.order_no
FROM user u
INNER JOIN orders o ON o.user_id = u.id
WHERE u.status = 1;
-- orders(user_id) 有索引，走 Index Nested Loop
```

## 实战场景

| 场景 | 优化方向 |
|------|----------|
| 慢日志显示某 SQL 平均 5s | `EXPLAIN` 看是否全表扫描、是否走错索引 |
| 列表页翻深后变慢 | 深度分页优化（游标法/延迟关联） |
| 高频小查询慢 | 缓存（Redis）+ 索引优化 |
| 大表 COUNT(*) 慢 | 计数表 / 估算值 / 缓存 |
| 报表查询慢 | 异步聚合到 ClickHouse/统计表 |
| JOIN 慢 | 被驱动表关联字段加索引，必要时反范式 |
| 写入慢 | 检查索引数量、是否过度事务、`innodb_flush_log_at_trx_commit` 配置 |
| 锁等待多 | 慢 SQL 持锁时间长，优化 SQL 减少锁持有 |

## 深挖追问

### 慢查询只靠加索引吗？

不。优化要分层：

1. SQL 写法：`SELECT *`、函数包索引列、隐式转换、深分页。
2. 索引：缺失索引、索引失效、索引选择错误。
3. 表结构：字段类型、大字段拆分、反范式、归档。
4. 架构：缓存、读写分离、ES、MQ。
5. 硬件：SSD、内存、CPU、网络。

只加索引等于只在第二层做优化，许多慢查询根因在其他层。

### EXPLAIN 的 rows 不准怎么办？

`rows` 是估算值，偏差大时：

1. `ANALYZE TABLE orders` 更新统计信息。
2. MySQL 8.0+ 用 `EXPLAIN ANALYZE` 看实际行数和耗时。
3. 持续偏差时考虑建直方图：`ANALYZE TABLE orders UPDATE HISTOGRAM ON status`。

### 缓存一致性怎么保证？

引入 Redis 缓存后重点关注一致性：

- **Cache Aside**：读时先查缓存，未命中查 DB 后写回缓存；写时先更 DB 再删缓存。
- **延迟双删**：写时先删缓存 → 更 DB → 延迟一段时间再删缓存。应对并发读旧值回填。
- **监听 binlog**：通过 Canal/Debezium 监听 binlog 异步刷新缓存，最终一致。
- 强一致场景考虑分布式锁或串行化，但性能损失大。

### 如何判断是 SQL 慢还是锁等待？

- `SHOW PROCESSLIST` 看连接状态：`Sending data`/`Sorting result` 是 SQL 执行慢；`Waiting for table metadata lock`/`Waiting for lock` 是锁等待。
- `information_schema.innodb_trx` 看当前事务和锁信息。
- 慢日志的 `Lock_time` 字段区分锁等待时间和执行时间。
- 监控指标：InnoDB row lock time、row lock waits。

### 优化后压测验证怎么做？

- 用 `sysbench` 或业务压测工具对比优化前后 QPS、P99 延迟。
- `EXPLAIN ANALYZE` 对比实际耗时。
- 灰度发布：先在从库或小流量验证。
- 监控指标持续观察一周，确认无回归。

## 易错点

- 看到 SQL 慢就加索引，没分析瓶颈。
- `SELECT *` 导致无法走覆盖索引。
- 索引列上包函数、做运算、隐式转换，索引失效。
- LIKE `'%xxx'` 前导通配，索引失效。
- 联合索引不遵循最左匹配。
- JOIN 时被驱动表关联字段无索引。
- 优化后不做压测验证，上线后回归。

## 总结

慢查询排查按"发现 → 定位 → 分析 → 优化 → 验证"闭环。发现用慢日志 + APM 监控；定位用 `mysqldumpslow`/`pt-query-digest`；分析用 `EXPLAIN` 看 `type`/`key`/`rows`/`Extra`；优化从 SQL、索引、表结构、架构、硬件五层下手；验证用 `EXPLAIN ANALYZE` + 压测。优化不是只加索引，要找到真正的瓶颈对症下药。生产环境应建立慢 SQL 定期巡检机制，把治理前置而非等线上爆发。
