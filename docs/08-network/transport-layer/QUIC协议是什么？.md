# QUIC协议是什么？

## 核心概念

UDP 是不可靠传输协议，但基于 UDP 的 QUIC 在用户态实现了可靠传输、拥塞控制、多路复用和 TLS 1.3 加密，HTTP/3 就运行在 QUIC 之上。QUIC 的目标不是“绕过可靠性”，而是在保留 UDP 易部署、用户态可演进特性的同时，修复 TCP + TLS + HTTP/2 在握手时延、队头阻塞和连接迁移上的不足。

连接迁移：QUIC 使用 Connection ID 标识连接，网络从 Wi-Fi 切换到移动数据时，即使客户端 IP/端口变化，也可以保持连接语义。重传机制：QUIC 对丢失的数据进行确认和重传，但重传的是 QUIC 帧数据而不是原始 UDP 包。拥塞控制：QUIC 内置类似 TCP 的拥塞控制算法，可以根据 RTT、丢包和带宽估计调整发送速率。前向纠错并不是现代 QUIC 的必选核心能力，面试中更应强调可靠传输、多路复用、TLS 1.3 和连接迁移。

## 面试总结
### 核心概念

QUIC 基于 UDP，但在应用/用户态实现连接管理、可靠传输、流量控制、拥塞控制、TLS 1.3 加密和多路复用。HTTP/3 使用 QUIC 后，一个连接内可以有多个独立 Stream，单个流丢包不会像 HTTP/2 over TCP 那样阻塞其他流的数据交付。

### 面试官想考什么

面试官常考 HTTP/3 为什么用 UDP、QUIC 如何解决 TCP 队头阻塞、0-RTT/1-RTT 握手的意义、连接迁移为什么适合移动网络，以及 QUIC 部署时会遇到哪些网关和防火墙问题。

### 标准回答

HTTP/2 已经解决了 HTTP/1.1 的应用层队头阻塞，但底层仍是 TCP：一旦 TCP 某个包丢失，内核必须按序交付字节，导致同一连接上的所有 HTTP/2 流都等待。QUIC 把多路复用放到 UDP 之上，每个 Stream 有独立的偏移和重传状态，因此一个 Stream 丢包通常不影响其他 Stream。QUIC 还把 TLS 1.3 集成进握手，减少建连时延，并用 Connection ID 支持 NAT rebinding 和网络切换。

### 深挖追问

- QUIC 为什么能缓解 HTTP/2 的 TCP 队头阻塞，但不能消除单个流内部的有序等待？
- 0-RTT 有什么重放攻击风险，哪些请求不适合使用？
- 企业网络或云负载均衡不放行 UDP 443 时，客户端如何回退？

### 实战场景/代码示例

移动端长连接、实时行情、音视频控制信令适合关注 QUIC。排查时可用浏览器 DevTools 查看协议列是否为 `h3`，或使用：
```bash
curl --http3 -I https://example.com
```
如果失败，要检查 UDP 443、防火墙、CDN/网关是否支持 HTTP/3，以及客户端是否回退到 HTTP/2。

### 易错点/总结

QUIC 不是“UDP 所以不可靠”，而是“基于 UDP 自己实现可靠传输”。它的优势来自用户态演进、多路复用、握手优化和连接迁移；代价是 CPU 开销、运维观测、UDP 网络兼容性和负载均衡支持更复杂。
