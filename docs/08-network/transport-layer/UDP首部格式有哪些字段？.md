# UDP首部格式有哪些字段？
UDP首部格式
UDP协议非常简单，头部只有8个字节，UDP头部格式如下：
源端口号（16位）：发送进程的端口号；
目标端口号（16位）：目标进程的端口号；
包长度（16位）：保存了UDP首部的长度和数据的长度之和；
校验和（16位）：提供可靠的UDP首部和数据而设计，防止收到在网络传输中受损的UDP包。

为什么UDP头部没有「首部长度」字段，而TCP头部则有呢？
原因是TCP头部中存在可变长的「选项」字段，而UDP头部长度是固定8字节，不会变化的，因此无需多一个字段记录UDP的首部长度。
为什么UDP头部有「包长度」字段，而TCP头部则没有呢？
TCP计算负载数据长度公式：

TCP数据长度=IP总长度-IP首部长度-TCP首部长度
其中IP总长度和IP首部长度这两个信息在IP首部格式中已知，而TCP首部长度，则在TCP首部格式中已知，因此可以通过公式求得TCP数据的长度，无需记录。虽然UDP也是基于IP层的，同样可以通过上述公式计算得到包长度，可能还需要额外考虑：
为了网络设备硬件设计和处理方便，首部长度需要是4字节的整数倍，因此为了补全UDP首部长度是4字节的整数倍，才补充了「包长度」字段；
如今的UDP是基于IP协议发展的，但是当年可能并非如此，依赖的可能是别的不提供自身报文长度或者首部长度的网络层协议，因此UDP报文首部需要有长度字段以供计算。

## 面试总结
### 核心概念

传输层负责端到端通信。TCP 面向连接、可靠、有序、字节流，依靠三次握手、序列号、确认、重传、滑动窗口、流量控制和拥塞控制；UDP 无连接、开销小、不保证可靠。

### 面试官想考什么

面试官高频考 TCP 三次握手/四次挥手、TIME_WAIT/CLOSE_WAIT、可靠传输、粘包、拥塞控制、连接异常和性能调优。

### 标准回答

TCP 建连通过 SYN、SYN+ACK、ACK 确认双方收发能力；断连通常四次挥手，因为双方方向的数据流要分别关闭。可靠性来自序列号、ACK、超时/快速重传、滑动窗口和校验。TIME_WAIT 保护旧报文消失并保证对端收到最后 ACK；CLOSE_WAIT 多通常是应用未主动关闭连接。UDP 适合实时音视频、DNS、QUIC 等可自定义可靠性或容忍丢包的场景。

### 实战场景/示例

Java 服务大量 CLOSE_WAIT，优先检查代码是否未关闭 socket/HTTP 响应流，或连接池释放逻辑异常；大量 TIME_WAIT 则看短连接、主动关闭方和连接复用。

### 易错点/总结

TCP 是字节流没有消息边界，所以会有粘包/拆包；应用层必须用长度字段、分隔符或固定长度等协议解决。
