# TCP重传机制是什么？
「重传机制」是TCP实现可靠传输的方式之一，基于序列号与确认机制实现。TCP针对数据包丢失的情况，会通过「重传机制」解决。常见的重传机制有：
超时重传
快速重传
SACK
D-SACK

超时重传
超时重传是重传机制的其中一个方式，就是在发送数据时，设定一个定时器，当超过指定的时间后，没有收到对方的ACK确认应答报文，就会重发该数据，也就是我们常说的「超时重传」。当发生数据包丢失或确认应答丢失时，就会发生超时重传。

RTT指的的数据发送时刻到接收到确认的时刻的差值，也就是包的往返时间。超时重传时间是以RTO（Retransmission Timeout超时重传时间）表示，超时时间设定分析：
当超时时间RTO较大时，重发慢，数据丢失很久才重发，没有效率，性能差；
当超时时间RTO较小时，会导致可能并没有丢失就重发，于是重发就会快，会增加网络拥塞，导致更多的超时，更多的超时导致更多的重发，恶性循环。
因此，我们可以得知，重传超时时间RTO的值应该略大于报文往返RTT的值。但实际上RTT是经常变化的，因为网络环境也时常变化，因此RTO的值也应该是一个动态变化的值。最后经过大量调试试验得出：如果超时重发的数据，再次超时需要重传的时候，TCP的策略是超时间隔翻倍，也就是每当遇到一次超时重传的时候，都会将下一次超时时间间隔设为先前值的两倍，两次超时就说明网络环境差，不宜频繁反复发送。

快速重传
超时重传存在的问题是，超时周期可能相对较长，于是可以使用「快速重传」机制来解决超时重发的时间等待问题。「快速重传」它不以时间为驱动，而是以数据驱动重传。其机制过程是：如果发送端收到了是三个相同的ACK确认报文后，就知道对应的序列号报文发生丢失，就会在重试重传定时器过期之前，重传丢失的报文。
三个重复ACK确认报文产生过程：
假如seq2发生丢失，但是后续的seq3，seq4，seq5正常接收，对于正常接收的报文，每次回复的ACK都是相同的ACK 2，当发送方收到三个重复ACK后，于是快速重传了seq2。

但是快速重传机制只解决了一个问题，就是超时时间的问题，但是它依然面临着另外一个问题，就是重传时，是重传一个，还是重传所有的问题，就是在无法确认只是丢了重复收到的ACK报文对应的序号报文，还是说该序号后的报文都丢失的问题。
只发一个，后面如果还有丢失，则还需要重新收到三个重复的ACK报文才能触发。
重发所有，如果后面没有丢失，则浪费资源；

SACK
SACK（Selective Acknowledge选择性确认）也是一种重传机制方式，用于解决快速重传机制中不知道该重传哪些TCP报文的问题。这种方式需要在TCP头部「选项」字段里加一个SACK的字段，它可以将已经收到的数据的信息发送给「发送方」，这样发送方就知道哪些数据收到了，哪些数据没有收到，因此只需要重传发生丢失的数据即可。

如果要支持SACK，必须双方都要支持。在Linux下，可以通过net.ipv4.tcp_sack参数打开这个功能（Linux2.4默认打开）。
D-SACK
Duplicate SACK又称D-SACK，其主要使用了SACK来告诉「发送方」有哪些数据被重复接收了。SACK是发生在接受方因为收不到数据包而重复给发送方发送三个相同ACK确认包并告诉接收方已经收到哪些数据，让发送方选择性重传；而D-SACK则是接收方收到重复的的报文，就会告诉发送方哪些数据被重复接收了，无需重传。

D-SACK好处：
可以让发送方知道是发送的包丢失了，还是接受方回应的ACK包丢失了；
可以知道是不是发送方的数据包被网络延迟了；
可以知道网络中是不是把发送方的数据包给复制了。

在Linux下可以通过net.ipv4.tcp_dsack参数开启/关闭这个功能（Linux2.4后默认开启）。

## 面试复习补充

### 核心概念

传输层负责端到端通信。TCP 面向连接、可靠、有序、字节流，依靠三次握手、序列号、确认、重传、滑动窗口、流量控制和拥塞控制；UDP 无连接、开销小、不保证可靠。

### 面试官想考什么

面试官高频考 TCP 三次握手/四次挥手、TIME_WAIT/CLOSE_WAIT、可靠传输、粘包、拥塞控制、连接异常和性能调优。

### 标准回答

TCP 建连通过 SYN、SYN+ACK、ACK 确认双方收发能力；断连通常四次挥手，因为双方方向的数据流要分别关闭。可靠性来自序列号、ACK、超时/快速重传、滑动窗口和校验。TIME_WAIT 保护旧报文消失并保证对端收到最后 ACK；CLOSE_WAIT 多通常是应用未主动关闭连接。UDP 适合实时音视频、DNS、QUIC 等可自定义可靠性或容忍丢包的场景。

### 深挖追问

- 这个现象发生在内核 TCP 状态机、网络链路还是应用代码中？
- 如何用 ss/netstat、tcpdump、内核计数器证明丢包、重传、半连接或 TIME_WAIT 问题？
- 哪些 Linux 参数、连接池参数和超时设置会影响生产表现？

### 实战场景/示例

Java 服务大量 CLOSE_WAIT，优先检查代码是否未关闭 socket/HTTP 响应流，或连接池释放逻辑异常；大量 TIME_WAIT 则看短连接、主动关闭方和连接复用。

### 易错点/总结

TCP 是字节流没有消息边界，所以会有粘包/拆包；应用层必须用长度字段、分隔符或固定长度等协议解决。

## 面试复习强化

### 核心概念

从面试角度看，**TCP重传机制是什么？** 可以放在“传输层网络”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 围绕连接状态、序列号、确认、重传、窗口、拥塞控制和内核队列展开。 追问常给异常包和边界状态：丢包、乱序、半连接、TIME_WAIT、CLOSE_WAIT、队列溢出、断网。 对于“TCP重传机制是什么？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：实战要能说出 ss/netstat/tcpdump/Wireshark/内核指标如何定位。
