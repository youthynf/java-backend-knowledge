# TCP 如何保证可靠传输

## 核心概念

TCP 的"可靠"是指：**应用层看到的数据是无丢失、无重复、无乱序、无损坏的字节流**。这背后是六种机制叠加的结果——连接管理、序列号、确认应答、重传、流量控制、拥塞控制。每一种机制单独看都不复杂，但合在一起构成了 TCP 可靠性的全部。

## 标准回答

TCP 可靠性靠六个机制：

1. **连接管理**：三次握手建立状态，四次挥手优雅释放，确保双方准备好再传数据。
2. **序列号**：每个字节编号，接收方按序重组，去重并检测乱序。
3. **确认应答**：接收方收到数据回 ACK，告知发送方下次该从哪个序号发。
4. **重传**：超时重传（RTO）+ 快速重传（3 次重复 ACK），丢包时补发。
5. **流量控制**：接收方通过 Window 字段通告自己的缓冲区剩余，防止发送方淹没接收方。
6. **拥塞控制**：发送方维护 CWND，根据网络状况调节发送速率，防止网络瘫痪。

外加校验和（Checksum）保证数据完整性，SACK 提高重传效率。

## 详细机制

### 序列号与确认应答

TCP 给字节流每个字节编号，序号字段是 32 位。发送方发的每个报文携带"本段第一个字节的序号"，接收方收到后回 ACK，确认号 = 收到的最后字节序号 + 1，表示"我已经收到到这里，期待下一个"。

累计确认（Cumulative ACK）：如果接收方收到 seq=1000 长度 500 的包，又收到 seq=2000 长度 500 的包（中间 1500-2000 丢了），回的 ACK 仍然是 1500。这表示"1500 之前的我都收到了"。发送方据此判断 1500-2000 之间需要重传。

### 超时重传（RTO）

发送方发出数据后启动定时器，超过 RTO 还没收到 ACK 就重传。RTO 通过 RTT 动态估算（SRTT 平滑 + RTTVAR 方差），Linux 中默认下限 200ms，上限 120s。

```bash
# 查看连接的 RTO
$ ss -ti
...
cubic wscale:8 rto:217 rtt:17.5/2.5 mss:1448 pmtu:1500 rcvmss:1328
```

### 快速重传（Fast Retransmit）

不等超时，靠"3 次重复 ACK"触发重传。原理：接收方收到乱序包时立即回 ACK（仍是期待的最小序号），连续 3 次相同 ACK 说明中间包丢了，发送方立即重传丢失段。

```
Sender -> seq=1000 (丢失)
Sender -> seq=1500  Receiver: ACK 1000 (dup 1)
Sender -> seq=2000  Receiver: ACK 1000 (dup 2)
Sender -> seq=2500  Receiver: ACK 1000 (dup 3) -> 触发快速重传 seq=1000
```

快速重传比超时重传快得多（RTT 量级 vs RTO 量级），但要配合 SACK 才能精确知道要重传哪些段。

### SACK（选择性确认）

接收方在 SACK 选项中告知"我已经收到了哪些不连续的段"，发送方只重传缺失段。例：收到的段是 [1000,1500)、[2000,2500)、[3000,3500)，SACK 选项列出 [2000,2500) 和 [3000,3500)，发送方据此只重传 [1500,2000) 和 [2500,3000)。

### 流量控制（滑动窗口）

接收方在 ACK 报文的 Window 字段通告自己剩余缓冲区。发送窗口 = min(CWND, RWND)。详细机制见滑动窗口章节。

### 拥塞控制

发送方维护 CWND，根据是否丢包调节。慢启动→拥塞避免→快重传→快恢复四个算法，详见拥塞控制章节。

### 校验和

TCP 头 + 数据 + 12 字节伪首部（含源/目的 IP、协议号）一起做 16 位反码求和。接收方校验失败直接丢弃，发送方超时后重传。

### 抓包示例

```bash
$ tcpdump -i any -n 'tcp port 80'
10:00:01 IP 10.0.0.1.5000 > 10.0.0.2.80: Flags [.], seq 1000:2000, ack 5000, win 65535, length 1000
10:00:01 IP 10.0.0.2.80 > 10.0.0.1.5000: Flags [.], ack 2000, win 65535, length 0
```

`seq 1000:2000` 表示 1000 到 2000 字节（不含 2000），长度 1000。`ack 2000` 表示"我已经收到 2000 之前的数据，期待 2000"。

## 代码示例

Java 验证 TCP 重传：

```java
import java.net.*;
import java.io.*;

public class ReliabilityDemo {
    public static void main(String[] args) throws IOException {
        Socket socket = new Socket("example.com", 80);
        OutputStream out = socket.getOutputStream();
        // TCP 保证这些字节按序、不丢、不重地到达
        out.write("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n".getBytes());
        out.flush();

        InputStream in = socket.getInputStream();
        byte[] buf = new byte[4096];
        int n = in.read(buf);  // 即使中间链路丢包重传，应用层看到的是完整数据
        System.out.println(new String(buf, 0, n));
    }
}
```

## 实战场景

| 场景 | 现象 | 排查 |
|------|------|------|
| 接收方处理慢 | Window 字段持续变小 | 应用层吞吐瓶颈，扩容或异步化 |
| 链路丢包高 | 大量重传，吞吐低 | 用 `nstat -az TcpRetransSegs` 监控 |
| 跨地域 RTT 高 | 单连接吞吐受限 | 调大 Window Scale，或并发多连接 |
| 大文件传输慢 | 拥塞控制限速 | BBR 算法替代 CUBIC，提升瓶颈带宽探测 |

## 深挖追问

**Q1：TCP 一定不丢数据吗？**
不绝对。极端情况下会"丢"：发送方 RST 关闭连接时未确认的数据会丢；连接长时间断开（如拔网线超过 keepalive 探测次数）后内核会丢弃缓冲区数据。详见"TCP 数据一定不会丢失吗？"。

**Q2：重传次数有限制吗？**
有。`tcp_retries2`（默认 15）控制数据段重传上限，达到后放弃连接；`tcp_retries1`（默认 3）达到后报告网络层异常。15 次大约 924 秒到 16 分钟（指数退避）。

**Q3：SACK 默认开启吗？**
Linux 默认开（`net.ipv4.tcp_sack = 1`）。几乎所有现代操作系统都支持，老设备可能不支持。

**Q4：重传的包和原包的序号一样吗？**
一样。重传就是重新发同样的字节范围，序号字段不变。这就是为什么接收方要去重——根据序号判断"这段我已经收过了"。

**Q5：流量控制和拥塞控制的区别？**
流量控制是接收方驱动（怕淹没接收方），窗口由对端通告；拥塞控制是发送方主动（怕淹没网络），窗口自己估算。两者取最小作为实际发送窗口。

## 易错点

- **"TCP 一定不丢"** — 不绝对，RST 关闭、keepalive 失败、缓冲区满等都可能丢。
- **"重传一定是数据丢"** — 也可能是 ACK 丢，发送方等不到 ACK 触发重传。
- **"流量控制 = 拥塞控制"** — 不是，前者防接收方溢出，后者防网络拥塞。
- **"ACK 号是收到的最后字节序号"** — 是收到的最后字节序号 + 1，表示"期待下一个"。
- **"累计确认就是 ACK 一次确认一段"** — 累计确认指一次 ACK 确认此前的所有段，不只是当前段。

## 总结

TCP 可靠性 = 序列号 + 累计确认 + 超时/快速重传 + 流量控制 + 拥塞控制 + 校验和。其中序列号是基础（标识字节），重传是核心（补丢），流控和拥塞控制是优化（不浪费）。理解每条机制解决什么问题，就能在排查网络故障时定位到具体环节。
