# TCP数据一定不会丢失吗？
一个数据包的完整发送链路：
一个数据包，从聊天框发出，消息从聊天软件所在的用户空间，拷贝到内核的发送缓冲区（send buffer），数据包就这样顺着传输层、网络层，进入到数据链路层，在这里的数据包会经过流控（qdisc），在通过RingBuffer发到物理层的网卡。数据就这样顺着网卡发送到了纷繁复杂的网络世界。这里头数据经过n多个路由器和交换机之间的跳转，最后到达目的机器的网卡处。

此时目的机器的网卡会通知DMA将数据包信息放到RingBuffer中，再触发一个硬中断给CPU，CPU触发软中断让ksofrirqd去RingBuffer收包，于是一个数据包就这样顺着物理层，数据链路层，网络层，传输层，最后从内核空间拷贝到用户空间里的聊天软件里。

常见容易发生丢包的场景：
建立连接是丢包：如果半连接或全连接满时，新来的包就会被丢弃；
流量控制丢包：为了避免应用层数据直接打爆网卡，需要让应用层数据按照一定的规则排队依次处理，这就是所谓的qdisc（Queueing Disciplines，排队规则），也就是常说的流量控制机制。一旦数据发送过快，超出队列长度，则容易出现丢包现象；
网卡丢包：RingBuffer过小导致丢包、网卡性能不足传输速度达到上限都会存在丢包问题；
接收缓冲区丢包：TCP Socket网络编程中，内核都会分配一个发送缓冲区和接收缓冲区。如果发送缓冲区满了，采用阻塞方式send时，会阻塞直到有内存，如果采用非阻塞调用，就会立即返回一个错误信息，这种情况下一般不会发生丢包；如果是接收缓冲区满了，TCP的接收窗口会变成0，这就是所谓的0窗口，并且会通过数据包里的win=0告知发送端，一般情况下发送端就停止发送消息，如果这是确实还有数据发来，就会发生丢包；
两端之间的网络丢包：两端外部网络存在不确定性，也可能存在丢包现象；

用了TCP协议就一定不会丢包吗？
TCP位于传输层，在它上面还有各种应用层协议。TCP保证的可靠性是传输层的可靠性，也就是说，TCP是能保证数据从A机器的传输层可靠地发送到B机器的传输层。至于数据到了接收端的传输层之后，能不能保证到应用层，TCP并不管。
假如我们输入一条消息，从聊天框发出，走到传输层TCP协议的发送缓冲区，不管中间有没有丢包，最后通过重传都保证发送到对方的传输层TCP接收缓冲区，此时接收端回复一个ACK，发送端收到这个ACK后就会将自己发送缓冲区中消息给扔掉，到这里TCP的任务就结束了。但是聊天软件的任务还没有结束，它需要将数据从TCP的接收缓冲区中读出来，如果在读出来那一刻收集由于内存不足或者其他各种原因，导致软件崩溃闪退了。这样，发送端以为自己的消息发送成功了，实则接收端没有收到这条消息，于是消息丢失了。

如何解决此类丢包问题？
通过引入第三端服务器，通过服务器与聊天软件进行最新消息的ID进行对比，就知道两端消息是否一致，就像对账一样。对于发送饭，只要定时跟服务器的内容对账一下，就知道那条消息没有发送成功，直接重发就好了，如果是接收方的聊天软件崩溃了，重启后跟服务器稍微通信一下就知道少了那条数据，同步上来就是了，所以也不存在上面提到的丢包问题。

## 面试总结
### 核心概念

传输层负责端到端通信。TCP 面向连接、可靠、有序、字节流，依靠三次握手、序列号、确认、重传、滑动窗口、流量控制和拥塞控制；UDP 无连接、开销小、不保证可靠。

### 面试官想考什么

面试官高频考 TCP 三次握手/四次挥手、TIME_WAIT/CLOSE_WAIT、可靠传输、粘包、拥塞控制、连接异常和性能调优。

### 标准回答

TCP 建连通过 SYN、SYN+ACK、ACK 确认双方收发能力；断连通常四次挥手，因为双方方向的数据流要分别关闭。可靠性来自序列号、ACK、超时/快速重传、滑动窗口和校验。TIME_WAIT 保护旧报文消失并保证对端收到最后 ACK；CLOSE_WAIT 多通常是应用未主动关闭连接。UDP 适合实时音视频、DNS、QUIC 等可自定义可靠性或容忍丢包的场景。

### 实战场景/示例

Java 服务大量 CLOSE_WAIT，优先检查代码是否未关闭 socket/HTTP 响应流，或连接池释放逻辑异常；大量 TIME_WAIT 则看短连接、主动关闭方和连接复用。

### 易错点/总结

TCP 是字节流没有消息边界，所以会有粘包/拆包；应用层必须用长度字段、分隔符或固定长度等协议解决。
