# TCP 为什么需要三次握手

## 核心概念

"为什么是三次而不是两次或四次"是 TCP 协议设计的根本问题。答案是：**三次是全双工通信双方互相确认收发能力并同步初始序列号的最小次数**。两次无法确认服务端的发送能力，四次冗余。

## 标准回答

三次握手解决三件事：

1. **同步双方的初始序列号（ISN）**：双方都要告知对方自己的 ISN 并得到确认。
2. **确认双方的收发能力正常**：客户端发→服务端收（确认客户端发、服务端收）；服务端发→客户端收（确认服务端发、客户端收）；客户端再发→服务端收（确认客户端收、服务端发）。
3. **防止历史连接复活**：第三次握手让客户端有机会拒绝陈旧的 SYN。

少于三次做不到以上三点；多于三次不增加信息量。

## 详细机制

### 两次握手会出什么问题

假设改成两次：客户端发 SYN，服务端回 SYN+ACK 后立即进入 `ESTABLISHED`。

**问题 1：历史连接复活**

```
Client -> Server: SYN seq=x (旧请求，网络延迟)
Client -> Server: SYN seq=y (新请求，先到达)
Server -> Client: SYN+ACK ack=y
                (此时 Server 已 ESTABLISHED)
旧 SYN 到达 Server
Server -> Client: SYN+ACK ack=x  (Server 又建了一个连接？)
```

两次握手下，服务端收到旧 SYN 就会建立连接，浪费资源。三次握手下，客户端收到 `ack=x` 时发现不是自己当前 ISN，会发 RST 终止连接。

**问题 2：服务端不知道客户端的接收能力**

两次握手下，服务端发完 SYN+ACK 就建连，但客户端可能根本没收到这个包（丢了），服务端却以为连接已建立，白白分配资源。

**问题 3：序列号无法可靠同步**

TCP 可靠性依赖序列号，如果客户端没确认服务端的 ISN，后续数据传输的 ACK 都失去意义。

### 四次为什么是冗余

把 SYN 和 ACK 拆成两次发：

```
C -> S: SYN
C <- S: ACK   (确认了客户端的 SYN)
C <- S: SYN   (服务端发自己的 SYN)
C -> S: ACK   (确认服务端的 SYN)
```

完全可以合并 `ACK + SYN` 成一个 `SYN+ACK`，省一个 RTT。所以四次是浪费。

### 三次握手的等价推导

通信双方各自要确认对方的 ISN，所以：

- A 告知 ISN_a → B 确认（B 知道 A 的 ISN）—— 2 步
- B 告知 ISN_b → A 确认（A 知道 B 的 ISN）—— 2 步

但 B 的"告知 ISN_b"和"确认 ISN_a"可以合并成一个 SYN+ACK，所以总共 3 步。

### 实现细节：第三次 ACK 丢了怎么办

客户端已进入 `ESTABLISHED`，会直接发数据；服务端还在 `SYN_RCVD`，收到数据报文时会同时确认这个 ACK（数据报文自带 ACK 标志），从而进入 `ESTABLISHED`。如果客户端不发数据，服务端会重传 SYN+ACK（默认 5 次，由 `tcp_synack_retries` 控制），超时后放弃。

## 代码示例

模拟"两次握手"的危害（伪代码）：

```python
# 假设的两次握手协议
def server_two_way():
    syn = recv()
    send(syn_ack)  # 直接 ESTABLISHED
    # 如果客户端此时早已崩溃，这条连接会一直占用
```

实际 TCP 用三次握手避免这种情况：

```c
// Linux 内核 net/ipv4/tcp_input.c 简化逻辑
if (state == TCP_SYN_RECV) {
    if (ack == expected_ack) {
        tcp_set_state(sk, TCP_ESTABLISHED);
    } else {
        tcp_send_active_reset(sk);  // 不是预期 ACK，发 RST
    }
}
```

## 实战场景

| 场景 | 应用 |
|------|------|
| 面试解释三次握手 | 从"全双工 + ISN 同步"角度答，避免只说"确认双方收发能力" |
| 设计自定义协议 | 单向数据两次握手够用；双向数据流必须三次 |
| TLS 1.3 设计 | 把握手压缩到 1-RTT，仍遵循"双方互相确认"原则 |
| QUIC 设计 | 传输层握手和加密握手合并，1-RTT 完成建立 |

## 深挖追问

**Q1：第三次握手的 ACK 丢了，连接算建立吗？**
客户端算，服务端不算。客户端进入 `ESTABLISHED` 可以发数据；服务端还在 `SYN_RCVD`，收到数据时利用数据包的 ACK 字段完成确认。

**Q2：两次握手真的完全不可用吗？**
对**单向通信**场景够用。比如简单传感器上报：客户端发数据，服务端只接收不回传，两次握手就够了。但 TCP 设计目标是双向可靠流，所以必须三次。

**Q3：如果网络分区导致旧 SYN 复活，三次握手怎么防？**
客户端收到服务端的 SYN+ACK 后会比较 ack 字段：如果是当前连接的 ISN，正常 ACK；如果是历史 ISN，发 RST。

**Q4：TCP Fast Open 怎么实现 0-RTT？**
TFO 不改变三次握手机制，而是允许客户端在第一次握手时就携带数据；服务端用 Cookie 验证客户端身份。第一次连接仍是 1-RTT，后续连接 0-RTT。

**Q5：ISN 为什么要随机？**
防止**序号预测攻击**。如果 ISN 可预测，攻击者可以伪造源 IP 发包绕过认证。RFC 6528 规定 ISN 应基于时钟和散列函数生成。

## 易错点

- **"两次握手无法确认客户端的发送能力"** — 错。两次握手中，客户端发 SYN 服务端收到，已经确认了客户端发送能力。两次握手的问题是服务端发送能力未被客户端确认。
- **"四次握手比三次更可靠"** — 不，三次已经足够可靠，四次只是浪费。
- **"三次握手是为了防止 SYN Flood"** — 不是。三次握手的设计早于 SYN Flood 攻击出现，防 SYN Flood 是后来的工作（SYN Cookies）。
- **"客户端必须发完 ACK 才能发数据"** — 第三次握手的 ACK 可以和数据合并发送。

## 总结

三次握手是全双工可靠通信的最小代价：双方各自同步 ISN 并确认对方的同步，第三次同时承担"确认对方 ISN"和"激活连接"两件事。两次不够，四次多余。设计自定义协议时可以借鉴这个推导：单向通信两次够，双向通信必须三次。
