# TCP 协议有什么缺陷

## 核心概念

TCP 是 1981 年定稿的协议（RFC 793），设计目标是"在不可靠网络上提供可靠字节流"。这个目标它完成得很好，但几十年来网络环境巨变：移动化、加密化、高带宽化、协议演进需求叠加，TCP 内核固化、字节流有序、四元组标识等设计反而成了缺陷。QUIC/HTTP/3 的出现本质上是绕开这些缺陷重新设计传输层。

## 标准回答

TCP 的四大固有缺陷：

1. **协议升级困难**：TCP 在内核实现，协议改动需要升级内核，迭代周期以年计。
2. **建连耗时**：TCP 三次握手 1 RTT，HTTPS 还要叠加 TLS 握手，首包延迟高。
3. **队头阻塞**：TCP 是字节流协议，必须按序交付，一个包丢失阻塞所有应用层流。
4. **连接迁移困难**：TCP 用四元组（源 IP/端口 + 目的 IP/端口）标识连接，IP 变了连接就断，不适合移动网络。

这四个问题正是 QUIC 要解决的。

## 详细机制

### 缺陷 1：协议升级困难

TCP 实现在操作系统内核中，协议栈代码随内核发布。任何协议改动（如新增选项、修改算法）都需要：

- 修改内核代码，等内核版本发布（数月到数年）
- 用户升级内核（Linux 服务器尚可，移动端依赖系统 OTA）
- 中间设备（防火墙、NAT、LB）兼容性测试

历史上 TCP 想加新特性的尝试都很艰难：

| 特性 | 提案时间 | 普及时间 | 难点 |
|------|---------|---------|------|
| Window Scale (RFC 7323) | 1997 | 2010+ | 老防火墙丢带选项的 SYN |
| SACK (RFC 2018) | 1996 | 2005+ | 老设备不支持 |
| MPTCP (RFC 6824) | 2013 | 至今未普及 | 中间设备不支持 |
| TCP Fast Open (RFC 7413) | 2014 | 推广缓慢 | NAT 丢弃带数据的 SYN |

QUIC 在用户态基于 UDP 实现，升级只更新应用程序，迭代周期以周计。

### 缺陷 2：建连耗时

传统 HTTPS 首次连接需要 3 RTT：

```
RTT 1: TCP 三次握手（1.5 RTT，第三次握手可带数据）
RTT 2: TLS 1.2 握手（1 RTT）
RTT 3: TLS 1.2 完成 + HTTP 请求响应
```

跨地域 RTT 100ms 时，首包延迟 300ms+，移动端体验差。优化方向：

- TLS 1.3 把 TLS 握手压到 1 RTT
- TCP Fast Open 让 SYN 携带数据
- QUIC 把传输握手和 TLS 握手合并，1 RTT 完成

### 缺陷 3：队头阻塞（HOL Blocking）

TCP 保证字节流按序交付，序号低的段未到，序号高的段即使已到达也无法交付给应用层。

```
发送方: [seq=1000][seq=2000][seq=3000][seq=4000]
seq=2000 在网络中丢失
接收方: 收到 1000, 3000, 4000，但只能交付 1000
应用层: 必须等 2000 重传到达才能拿到 3000、4000
```

HTTP/2 在 TCP 之上多路复用多个 Stream，但 TCP 不知道 Stream 边界，一个 Stream 的丢包会阻塞所有 Stream。这是 HTTP/3 改用 QUIC 的核心动机。

### 缺陷 4：连接迁移困难

TCP 用四元组标识连接：

```
连接 = (src_ip, src_port, dst_ip, dst_port)
```

移动设备从 Wi-Fi 切到 4G 时 IP 变化，五元组变化，TCP 连接断开，必须重建。重建过程包含：

- TCP 三次握手（1 RTT）
- TLS 握手（1 RTT）
- TCP 慢启动（吞吐爬坡）

用户感受是网络突然卡一下。QUIC 用 Connection ID 标识连接，与 IP 解耦，网络切换时连接保持。

## 代码示例

观察 TCP 队头阻塞（模拟丢包）：

```bash
# 用 tc 在 eth0 上模拟 10% 丢包
$ tc qdisc add dev eth0 root netem loss 10%

# 大文件传输，单 TCP 连接吞吐显著下降
$ curl -o /dev/null http://example.com/largefile
# 即使 90% 的包到达，TCP 也要等丢失的 10% 重传完才能交付
```

观察 IP 变化导致连接断开：

```bash
# 长连接建立后切换网络
$ nc -lk 8080                # 服务端
$ nc 192.168.1.10 8080        # 客户端连接
# 切换 Wi-Fi → 4G，IP 变化
# TCP 连接立即断开，nc 报 Connection reset by peer
```

## 实战场景

| 缺陷 | 触发场景 | 现网影响 | 替代方案 |
|------|---------|---------|---------|
| 升级困难 | 想用新拥塞算法 | 必须升级内核 | BBR 已合入主线，老内核用不了 |
| 建连耗时 | HTTPS 首次访问 | 移动端首屏慢 | TLS 1.3 + HTTP/2，或 QUIC |
| 队头阻塞 | HTTP/2 多路复用 | 单流丢包阻塞所有流 | HTTP/3 over QUIC |
| 连接迁移 | 移动端网络切换 | 切网断连 | QUIC Connection ID |

## 深挖追问

**Q1：TCP 不能改吗？**
能改，但成本极高。任何改动需要全链路兼容（客户端、服务端、中间设备），引入新选项可能被老防火墙丢弃。所以协议演进慢。

**Q2：为什么不直接修 TCP 而是发明 QUIC？**
TCP 在内核，迭代慢；中间设备对 TCP 头部改动容忍度低；MPTCP 等扩展推广失败证明内核路线难走。QUIC 走 UDP + 用户态绕开这些限制。

**Q3：TCP 队头阻塞能消除吗？**
不能在不破坏 TCP 语义的前提下消除。有序字节流是 TCP 的核心承诺。只能换协议（QUIC 各 Stream 独立）。

**Q4：连接迁移可以用 MPTCP 吗？**
理论上可以，MPTCP 支持多路径。但 MPTCP 部署受中间设备限制，移动端支持差。QUIC 的 Connection ID 更简单且部署友好。

**Q5：TCP 还有其他缺陷吗？**
还有：ISN 可预测（已被 RFC 6528 修复）、SYN Flood（已被 SYN Cookies 缓解）、TIME_WAIT 资源占用（用 `tcp_tw_reuse` 缓解）、Nagle 与延迟确认冲突等。

## 易错点

- **"TCP 队头阻塞是 HTTP/2 的问题"** — 不，是 TCP 字节流有序性的固有特性，HTTP/2 在 TCP 之上无法绕开。
- **"TLS 1.3 解决了 TCP 建连慢"** — 部分解决，TLS 握手快了，但 TCP 三次握手仍要 1 RTT。
- **"TCP 不能升级所以过时了"** — 不，TCP 仍是主流，绝大多数 Web/RPC/DB 流量跑在 TCP 上，QUIC 主要是补充而非替代。
- **"连接迁移是 TCP 的 bug"** — 不是 bug，是设计取舍。1981 年没有移动网络场景。

## 总结

TCP 的四大缺陷（升级难、建连慢、队头阻塞、连接迁移困难）源于 1981 年的设计取舍：内核实现、字节流有序、四元组标识。这些设计在固定网络时代合理，但在移动+加密+多路复用时代变成瓶颈。QUIC/HTTP/3 在 UDP 之上重新设计传输层，绕开这些缺陷，但 TCP 仍是基础设施的主流，短期内不会被替代。
