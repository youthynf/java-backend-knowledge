# TCP连接处于ESTABLISH状态，收到SYN报文会发生什么？
## 核心概念

当客户端重启后，而服务端还处于ESTABLISH状态，客户端会给服务端发送SYN报文，会发生什么，需要分情况讨论。
已建立连接的TCP，收到SYN会发生什么？
如果客户端SYN报文里的端口号和历史的连接不同：
服务端会认为是一次新的连接建立，会经过TCP三次握手建立新的连接。
对于老的连接，如果没有一直没有数据传输，则服务端会保持ESTABLISH状态，直到触发TCP保活机制（如开启）；
对于老的连接，如果有数据传输，服务端发送等待报文给客户端，因为客户端连接已经消失，因此客户端对于服务端的请求，会回复一个RST报文，随后双方连接断开。

如果客户端SYN报文里的端口号和历史的连接相同：
处于ESTABLISH状态的服务端收到客户端的SYN报文后，会回复一个携带了正确序列号和确认号的ACK报文，这个ACK报文也称之为Challenge ACK。接着客户端收到这个Challenge ACK，发现确认号并不是自己期望收到的，就会回复一个RST报文，服务端收到后，就会释放掉该连接。
如何断开TCP连接？
通过kill关闭进程：客户端杀掉进程，影响范围只有这个客户端进程所建立的连接，服务端杀掉进程，则所有客户端的连接都会关掉。不够精细化和优雅。
killcx工具：通过发送SYN报文，获得服务端的Challenge ACK携带的正确序列号和确认序列号，然后通过模拟RST报文来断开连接。支持断开所有的TCP连接，不管是否活跃。
tcpkill工具：通过监听连接通信的序列号，模拟RST报文。前提是连接有数据交互，否则无法关闭。

## 面试总结
### 核心概念

传输层负责端到端通信。TCP 面向连接、可靠、有序、字节流，依靠三次握手、序列号、确认、重传、滑动窗口、流量控制和拥塞控制；UDP 无连接、开销小、不保证可靠。

### 面试官想考什么

面试官高频考 TCP 三次握手/四次挥手、TIME_WAIT/CLOSE_WAIT、可靠传输、粘包、拥塞控制、连接异常和性能调优。

### 标准回答

TCP 建连通过 SYN、SYN+ACK、ACK 确认双方收发能力；断连通常四次挥手，因为双方方向的数据流要分别关闭。可靠性来自序列号、ACK、超时/快速重传、滑动窗口和校验。TIME_WAIT 保护旧报文消失并保证对端收到最后 ACK；CLOSE_WAIT 多通常是应用未主动关闭连接。UDP 适合实时音视频、DNS、QUIC 等可自定义可靠性或容忍丢包的场景。

### 实战场景/示例

### 易错点/总结

TCP 是字节流没有消息边界，所以会有粘包/拆包；应用层必须用长度字段、分隔符或固定长度等协议解决。
