# TCP拥塞控制机制是什么？
## 核心概念

什么是TCP拥塞控制？
流量控制是为了避免发送方的数据填满接收方的缓存，但是计算机网络都处在一个共享的环境，因此有可能会因为其他主机之间的通信使得网络拥堵。在网络出现拥堵时，如果继续发送大量数据包，可能会导致数据包时延、丢失等，这时 TCP 就会重传数据，但是一重传就会导致网络的负担更重，于是会导致更大的延迟以及更多的丢包，这个情况就会进入恶性循环被不断地放大。于是，就有了「拥塞控制」，控制的目的就是避免「发送方」的数据填满整个网络。

什么是拥塞窗口？和发送窗口有什么关系？
为了在「发送方」调节所要发送数据的量，定义了一个叫做「拥塞窗口」的概念。拥塞窗口 cwnd是发送方维护的一个的状态变量，它会根据网络的拥塞程度动态变化的。

前面提到，发送窗口swnd和接收窗口rwnd是约等于的关系，那么由于加入了拥塞窗口的概念后，此时发送窗口的值是swnd=min(cwnd, rwnd)，也就是发送窗口是拥塞窗口和接收窗口中的最小值。拥塞窗口cwnd变化的规则：
只要网络中没有出现拥塞，cwnd就会增大；
但网络中出现了拥塞，cwnd就减少。

那么怎么知道当前网络是否出现了拥塞呢？
其实只要「发送方」没有在规定时间内接收到ACK应笞报文，也就是发生了超时重传，就会认为网络出现了拥塞。
拥塞控制主要是四个算法：
慢启动算法
拥塞避免算法
拥塞发生算法
快速恢复算法

慢启动算法
TCP 在刚建立连接完成后，首先是有个慢启动的过程，一开始按照初始的拥塞窗口cwnd发送数据包，当发送方每收到一个ACK，拥塞窗口cwnd的大小就会加1，如收到1个ACK，下次发2个，然后收到2个ACK，下次发4个，以此类推，因此此时发包的个数是指数性的增长。当有一个叫慢启动门限ssthresh（slow start threshold）状态量，一般来说 ssthresh 的大小是 65535 字节：
当拥塞窗口cwnd ＜ ssthresh：使用慢启动算法；
当拥塞窗口cwnd ≥ ssthresh：就会使用「拥塞避免算法」。

拥塞避免算法
当拥塞窗口cwnd超过慢启动门限ssthresh就会进入拥塞避免算法。拥塞避免算法后的规则是：每当收到一个ACK时，拥塞窗口cwnd增加1/cwnd。相当于收到cwnd个ACK时，cwnd才加1。拥塞避免算法就是将原本慢启动算法的指数增长变成了线性增长，增长速度变慢，但还是处于增长阶段。随着一直增长，网络就会慢慢进入了拥塞的状况了，于是就会出现丢包现象，这时就需要对丢失的数据包进行重传。当触发了重传机制，也就进入了「拥塞发生算法」。

拥塞发生算法
当网络出现拥塞，也就是会发生数据包重传，重传机制主要有「超时重传」和「快速重传」，这两种机制使用的拥塞发生算法是不同的

当发生了「超时重传」，则就会使用拥塞发生算法。这个时候，慢启动门限ssthresh和拥塞窗口cwnd的值会发生变化：
慢启动门限ssthresh设为拥塞窗口cwnd/2；
拥塞窗口cwnd重置为初始化值（Linux针对每个TCP连接的cwnd初始值是10，也就是10个MSS）；
接着，就重新开始慢启动，慢启动是会突然减少数据流的，这种方式太激进了，反应也很强烈，会造成网络卡顿。

当发生「快速重传」，即接收方发现丢了一个中间包的时候，发送三次前一个包的ACK，于是发送端就会快速地重传，不必等待超时再重传。TCP认为这种情况不严重，因为大部分没丢，只丢了一小部分，则慢启动门限ssthresh和拥塞窗口cwnd的值变化如下：
拥塞窗口cwnd =cwnd/2，也就是设置为原来的一半，而不是直接重置为初始值；
慢启动门限ssthresh=cwnd，也就是设置为缩减一半后的拥塞窗口cwnd大小；
进入快速恢复算法；

快速恢复算法
快速重传和快速恢复算法一般同时使用，快速恢复算法是认为，你还能收到3个重复ACK说明网络也不那么糟糕所以没有必要像RTO超时重传那么强烈。

在进入快速回复之前，cwnd和ssthresh已经被更新了：
cwnd = cwnd/2，也就是设置为原来的一半；
ssthresh = cwnd；
然后，进入快速恢复算法如下：
拥塞窗口cwnd = ssthresh+3 = 原来cwnd/2 + 3（3的意思是确认有3个数据包被收到了)；
重传丢失的数据包；
如果再收到重复的ACK，那么cwnd增加 1；
如果收到新数据的ACK后，把cwnd设置为第一步中的ssthresh的值，重新进入「拥塞避免」状态，原因是该ACK确认了新的数据，说明从duplicated ACK时的数据都已收到，该恢复过程已经结束，可以回到恢复之前的状态了，继续进行线性增长；
快速恢复算法没有像「超时重传」一夜回到解放前，而是还在比较高的值，后续呈线性增长。

## 面试总结
### 核心概念

传输层负责端到端通信。TCP 面向连接、可靠、有序、字节流，依靠三次握手、序列号、确认、重传、滑动窗口、流量控制和拥塞控制；UDP 无连接、开销小、不保证可靠。

### 面试官想考什么

面试官高频考 TCP 三次握手/四次挥手、TIME_WAIT/CLOSE_WAIT、可靠传输、粘包、拥塞控制、连接异常和性能调优。

### 标准回答

TCP 建连通过 SYN、SYN+ACK、ACK 确认双方收发能力；断连通常四次挥手，因为双方方向的数据流要分别关闭。可靠性来自序列号、ACK、超时/快速重传、滑动窗口和校验。TIME_WAIT 保护旧报文消失并保证对端收到最后 ACK；CLOSE_WAIT 多通常是应用未主动关闭连接。UDP 适合实时音视频、DNS、QUIC 等可自定义可靠性或容忍丢包的场景。

### 实战场景/示例

Java 服务大量 CLOSE_WAIT，优先检查代码是否未关闭 socket/HTTP 响应流，或连接池释放逻辑异常；大量 TIME_WAIT 则看短连接、主动关闭方和连接复用。

### 易错点/总结

TCP 是字节流没有消息边界，所以会有粘包/拆包；应用层必须用长度字段、分隔符或固定长度等协议解决。
