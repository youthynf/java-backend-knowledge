# TCP 滑动窗口是什么

## 核心概念

滑动窗口是 TCP 实现"流水线发送"的核心机制。没有窗口时每发一个包都要等 ACK 才能发下一个（停等协议），效率极低。引入窗口后，发送方可以连续发多个包而不必等 ACK，窗口大小 = "无需确认就能连续发送的最大数据量"。窗口随着 ACK 到达向前"滑动"，所以叫滑动窗口。

## 标准回答

滑动窗口解决"如何在不等 ACK 的情况下批量发送"的问题。窗口大小由接收方通告（RWND），发送方维护三个指针把发送缓冲区分成四段：已确认、已发未确认（在窗）、可发未发（可用窗）、不可发。每收到一个 ACK，窗口右沿右移，新数据进入可发区。

发送窗口 = min(CWND, RWND)，即接收方通告窗口和网络拥塞窗口中较小者。

## 详细机制

### 发送方四分区

```
已发送且已确认  |  已发送未确认  |  可发送未发送  |  不可发送
                ^                ^               ^
              SND.UNA         SND.NXT        SND.UNA+WND
              (左沿)            (中间)         (右沿)
```

- **SND.UNA**：最早未确认的字节序号（窗口左沿）
- **SND.NXT**：下次要发送的字节序号
- **SND.WND**：窗口大小
- **SND.UNA + SND.WND**：窗口右沿，超出此范围的数据不能发

可用窗口 = SND.WND - (SND.NXT - SND.UNA)。当 SND.NXT 追上右沿时，可用窗口为 0，必须等 ACK 才能继续。

### 接收方三分区

```
已确认收下  |  可接收（在窗）  |  不可接收
           ^                  ^
         RCV.NXT           RCV.NXT + RCV.WND
```

- **RCV.NXT**：期待接收的下一个字节序号
- **RCV.WND**：接收窗口大小（通告给发送方）

接收方收到 RCV.NXT 处的数据后，RCV.NXT 右移，应用层读取缓冲区数据后 RCV.WND 可能增大。

### 累计确认的作用

发送方连续发 5 个包，第 3 个 ACK 丢了，但第 4 个 ACK 到达，发送方一看 ack=5 之后，说明 1-5 都收到了，可以直接滑动到 5。这就是累计确认的优势——不要求每个 ACK 都到达。

### 窗口滑动的具体过程

```
T0: 窗口 [1000, 5000)，发送 1000-2000, 2000-3000, 3000-4000
T1: 收到 ACK 2000 (确认了 1000-2000)，窗口右移到 [2000, 6000)
    可发送 5000-6000
T2: 收到 ACK 3000，窗口右移到 [3000, 7000)
```

### 窗口关闭与探测

接收方通告 Window=0 时，发送方停止发送。这叫窗口关闭。如果接收方之后通告非 0 窗口的 ACK 丢了，双方死锁——发送方等窗口，接收方等数据。所以 TCP 有**持续计时器**（persist timer），发送方收到 Window=0 后定期发窗口探测包（1 字节），强制接收方回应当前窗口。

### 抓包示例

```bash
$ tcpdump -i any -n 'tcp port 80'
10:00:01 IP 10.0.0.1.5000 > 10.0.0.2.80: Flags [.], seq 1000:2000, ack 1, win 65535, length 1000
10:00:01 IP 10.0.0.1.5000 > 10.0.0.2.80: Flags [.], seq 2000:3000, ack 1, win 65535, length 1000
10:00:01 IP 10.0.0.1.5000 > 10.0.0.2.80: Flags [.], seq 3000:4000, ack 1, win 65535, length 1000
# 三个包连续发送，不等 ACK —— 这就是滑动窗口在工作
10:00:01 IP 10.0.0.2.80 > 10.0.0.1.5000: Flags [.], ack 4000, win 65535, length 0
# 一个 ACK 确认了全部 3000 字节
```

## 代码示例

Java 调整 socket 缓冲区影响窗口：

```java
import java.net.*;

Socket socket = new Socket("example.com", 80);
// 增大接收缓冲区 → 接收窗口更大 → 发送方能发更多
socket.setReceiveBufferSize(256 * 1024);  // 256 KB
// 增大发送缓冲区 → 发送窗口更大
socket.setSendBufferSize(256 * 1024);
// 实际窗口还受对端通告和拥塞窗口限制
```

## 实战场景

| 场景 | 现象 | 处理 |
|------|------|------|
| 高 BDP 链路吞吐低 | 窗口不够大 | 启用 Window Scale，调大 socket buffer |
| 接收方处理慢 | Window 持续缩小到 0 | 应用层提速或扩容 |
| 窗口死锁 | 连接卡住不发数据 | 持续计时器探测，确保对端响应 |
| 大文件传输优化 | 单流吞吐低 | 调大 RWND、用 BBR 拥塞算法 |

## 深挖追问

**Q1：发送窗口和接收窗口大小相等吗？**
不一定相等，是"约等于"。接收方通告的窗口变化到达发送方有时延，期间两者不一致。实际发送窗口还受 CWND 限制，可能比 RWND 小。

**Q2：Window 字段只有 16 位，怎么传 1 GB 窗口？**
通过 Window Scale 选项（RFC 7323），握手时协商缩放因子（如 8），实际窗口 = Window 字段值 × 2^因子。最大 65535 × 2^14 ≈ 1 GB。

**Q3：窗口右沿会回退吗？**
不会。TCP 规定窗口右沿不能回退（即使接收方缩小窗口，右沿也只能右移或不动）。如果应用层不读数据导致缓冲区满，接收方通告 Window=0，发送方停发。

**Q4：零窗口探测几次后放弃？**
Linux 默认持续探测约 15 次（`tcp_retries2`），超时后关闭连接。每次探测间隔指数退避，最长约 60 秒一次。

**Q5：糊涂窗口综合症是什么？**
接收方通告小窗口、发送方发小包，导致大量小包浪费带宽（每包 40 字节头部）。需要发送方 Nagle 算法 + 接收方 David Clark 算法配合避免。

## 易错点

- **"窗口大小等于发送缓冲区大小"** — 不等。窗口是协议层概念，受对端通告和 CWND 限制；缓冲区是内核/应用层资源。
- **"窗口只往右滑"** — 窗口右沿只能右移或不动，不能回退；左沿随 ACK 右移。
- **"接收窗口 = 发送窗口"** — 约等于，不严格相等。
- **"Window=0 就断开"** — 不，是暂停发送，等窗口探测。
- **混淆 CWND 和 RWND** — CWND 是发送方估算的网络窗口，RWND 是接收方通告的接收窗口，发送窗口 = min(CWND, RWND)。

## 总结

滑动窗口让 TCP 不必停等，可以流水线发送。窗口由接收方通告（RWND），受网络拥塞限制（CWND），实际发送窗口取两者最小。理解窗口滑动要看清 SND.UNA、SND.NXT、SND.WND 三个指针的关系。窗口关闭、糊涂窗口综合症、零窗口探测是窗口机制下的常见边界问题。
