# TCP每次建立连接的初始序列号为什么要不一样？
## 核心概念

为什么每次建立TCP连接时，初始化的序列号都要求不一样？
防止历史报文被下一个相同的四元组的连接接收（主要原因）
防止黑客伪造的相同序列号的TCP报文被对方接收
如果阻塞在网络中的数据包，刚好在下一个相同的四元组的接收窗口范围内，该数据就会被服务端接收，就会造成数据错乱，每次初始化序号不一样，很大程度上能够避免历史报文被下一个相同四元组的连接接收，但不能完全避免，因为序列号会有回绕的问题，所以需要使用时间戳机制来判断历史报文。
TCP四次挥手中TIME_WAIT状态不是会持续2MSL时长，历史报文不是早就在网络中消失了吗？为什么还需要通过序列号来防止历史报文被下一个相同的四元组的连接接收？
是的，如果能正常四次挥手，由于TIME_WAIT状态会持续2MSL时长，历史报文会在下一个连接之前就会自然消失，但是我们不能保证每次连接都能通过四次挥手来正常关闭连接。

初始序列号ISN是如何随机生成的？
RFC793提到初始化序列号ISN随机生成算法：

ISN=M+F(localhost, localport, remotehost, remoteport)
M是一个计时器，这个计时器每个4微妙加1；
F是一个Hash算法，根据源IP、目的IP、源端口、目的端口生成一个随机数值，保证Hash算法不能被外部轻易推算得出。
随机数会基于时钟计时器递增的，基本不可能随机生成一样的初始化序列号。
客户端和服务端初始化序列号都是随机生成的，能避免连接接收到历史报文了，但是也不是完全避免，原因是：
序列号，TCP的头部的一个字段，用于标识TCP发送端到TCP接收端的数据流的每一个字节，TCP为每个传输方向上的每个字节都赋予了一个编号，以便于传输成功后确认、丢失后重传以及在接收端保证不会乱序。序列号是一个32位的无符号数，因此在达到4G之后在循环回到0；
初始化序列号，在TCP建立连接的时候，客户端和服务端都会各自生成一个初始化序列号，它是基于时钟生成的一个随机数，来保证每个连接都拥有不同的初始序列号。初始序列号可被视为一个32位的计数器，该计数器的数值每4微妙加1，循环一次需要4.55小时。

序列号和初始序列号并不是无限递增的，会发生回绕位初始值的情况，意味着无法通过序列号来区分新老数据。为了解决序列号回绕可能导致的数据错乱问题，TCP引入了TCP时间戳，可以通过tcp_timestamps参数进行开启，默认是开启的。它既能便于精确计算RTT，也可以防止序列号的回绕。其实现原理是，每收到一个新数据包都会读取数据包中的时间戳跟最近一次收到的数据包的时间戳Recten TSval进行比较，如果发现收到的数据包中的时间戳不是递增的，则标识该数据包是过期的，就会直接丢弃这个数据包。

其实32bit的时间戳也会存在回绕问题，解决方案：
增加时间戳的大小，由32位扩大到64位，但会导致新旧协议的兼容问题；
将一个与时钟频率无关的数值作为时间戳，时间频率可以增加，时间戳的增速不变，但随着时间频率的提高，TCP在相同时间内能够收发的包会越来越多，如果时间戳增速不变，将会有越来越多相同时间戳的报文，时间戳就会失去意义。

## 面试总结
### 核心概念

传输层负责端到端通信。TCP 面向连接、可靠、有序、字节流，依靠三次握手、序列号、确认、重传、滑动窗口、流量控制和拥塞控制；UDP 无连接、开销小、不保证可靠。

### 面试官想考什么

面试官高频考 TCP 三次握手/四次挥手、TIME_WAIT/CLOSE_WAIT、可靠传输、粘包、拥塞控制、连接异常和性能调优。

### 标准回答

TCP 建连通过 SYN、SYN+ACK、ACK 确认双方收发能力；断连通常四次挥手，因为双方方向的数据流要分别关闭。可靠性来自序列号、ACK、超时/快速重传、滑动窗口和校验。TIME_WAIT 保护旧报文消失并保证对端收到最后 ACK；CLOSE_WAIT 多通常是应用未主动关闭连接。UDP 适合实时音视频、DNS、QUIC 等可自定义可靠性或容忍丢包的场景。

### 实战场景/示例

Java 服务大量 CLOSE_WAIT，优先检查代码是否未关闭 socket/HTTP 响应流，或连接池释放逻辑异常；大量 TIME_WAIT 则看短连接、主动关闭方和连接复用。

### 易错点/总结

TCP 是字节流没有消息边界，所以会有粘包/拆包；应用层必须用长度字段、分隔符或固定长度等协议解决。
