# HTTP/2是什么？

HTTP/2是什么？
一、HTTP/1.1协议的性能问题
延迟难以下降：网络延迟下降到一定幅度后，达到延迟的下限；
并发连接数有限：谷歌浏览器最大并发连接数是6个，而且每个连接都要经过TCP和TLS握手耗时，以及TCP慢启动过程给流量带来的影响；
对头阻塞问题：使用请求-响应模型，默认不开启pipeline，同一个连接只能在完成一个请求-响应后，才能处理下一个事务；
HTTP头部巨大且重复：由于HTTP协议是无状态的，所以每次请求都需要携带HTTP头部，特别对于有携带Cookie的头部，而Cookie的大小通常很大；
不支持服务器推送消息：当客户端需要获取通知时，只能通过定时轮询方式，造成带宽和服务资源浪费。

二、HTTP/2介绍
1.兼容HTTP/1.1
保持原有的协议名：「http://」表示明文协议，使用「https://」表示加密协议；
保持语义不变，更改语法：语义不变指请求方法、状态码、投资端等规则保持不变；语法层面则基本改变了HTTP报文的传输格式。

2.头部压缩
用HPACK算法替代常见的gzip压缩方式进行Header压缩。HPACK算法主要包含静态字典、动态字典、Huffman编码（压缩算法）这三部分组成。客户端和服务器两端都会建立和维护「字典」，用长度较小的索引号表示重复的字符串，再用Huffman编码压缩数据，可达到50%-90%的压缩率。
静态表编码：为高频出现在头部的字符串和字段建立一张静态表，写入HTTP/2框架中，不会变化。表中Index表示索引，HeaderName表示首部字段名称，HeaderValue表示字段值，每个value都通过Huffman编码后才会发送出去，对于值不固定的HeaderValue不一定有值。
动态编码表：首次发送后，对于非静态表范围内的头部字符串，客户端和服务器双方都会更新自己的动态表，下次发送的时候，不需要重复发送这个字段的数据，只需要发送动态表索引即可。但动态表生效的前提是必须同一个连接上，重复传输完全相同的HTTP头部，存在变化的字段均无法利用。动态表会越来越大，避免过大，Web服务器都会限制单个连接最大请求数，超出数量则关闭HTTP/2连接释放内存。

3.二进制编码
HTTP/2 不再像 HTTP/1.1 里的纯文本形式的报文，而是全面采用了二进制格式，头信息和数据体都是二进制，并且统称为帧(frame)：头信息帧（Headers Frame）和数据帧（Data Frame）。对计算机非常友好，收到报文后，无需再将明文的报文转成二进制，而是直接解析二进制报文，提高了数据传输的效率；
帧的结构包含：
帧长度（3字节）：表示帧数据的长度
帧类型（1字节）：分为数据帧和控制帧两类；
帧标志位（1字节）：用于携带简单的控制信息；
流标识（4字节）：用于表示当前帧属于哪个Stream，接收方用于有序组装信息；
帧数据：存放经过HPACK算法压缩过的HTTP头部和包体。

4.并发传输
多个Stream复用1条TCP连接，达到并发的效果，解决了HTTP/1.1对头阻塞的问题，提高了HTTP传输的吞吐量。Steam、Message、Frame三者关系理解：
Steam：一个独立的，双向的虚拟通道，用于传输多个Message；
Message：一个完整的逻辑数据块，通常对应一个请求或响应，由1个HEADERSz帧和多个DATA帧组成；
Frame：HTTP/2二进制传输的最小单位，如HEADERS帧和DATA帧。
不同的Stream的帧是可以乱序发送的，但是同一个 Stream 里的帧必须严格有序。客户端和服务器双方都可以建立Stream，客户端建立的Stream必须是奇数，服务器是偶数。同一个连接中的Stream ID是不能复用的，只能顺序递增，当Stream ID耗尽时，需要发送一个控制帧GOAWAY，用来关闭TCP连接。另外，可以根据资源的渲染顺序来设置Stream的优先级，从而提高用户体验；

5.服务器支持主动推送资源
服务器推送资源时，会先发送PUSH_PROMISE帧，告诉客户端接下来在哪个Stream发送资源，然后用偶数号Stream发送资源给客户端。

三、HTTP/2存在问题
HTTP/2 通过 Stream 的并发能力，解决了 HTTP/1 队头阻塞的问题，看似很完美了，但是 HTTP/2 还是存在「队头阻塞」的问题，只不过问题不是在 HTTP 这一层面，而是在 TCP 这一层。
HTTP/2 是基于 TCP 协议来传输数据的，TCP 是字节流协议，TCP 层必须保证收到的字节数据是完整且连续的，这样内核才会将缓冲区里的数据返回给 HTTP 应用，那么当「前1个字节数据」没有到达时，后收到的字节数据只能存放在内核缓冲区里，只有等到这1个字节数据到达时，HTTP/2 应用层才能从内核中拿到数据，这就是 HTTP/2 队头阻塞问题。

## 面试复习补充

### 核心概念

HTTP/2 在一个 TCP 连接上通过二进制分帧、多路复用、Header 压缩和服务端推送提升性能。

### 面试官想考什么

面试官想考 HTTP/2 与 HTTP/1.1 的差异、HPACK、队头阻塞发生在哪一层。

### 标准回答

HTTP/2 把请求响应拆成帧并复用到同一连接，减少连接数量；HPACK 压缩重复头部；但底层仍是 TCP，一旦 TCP 丢包，同一连接上的流仍可能被阻塞，这也是 HTTP/3/QUIC 要解决的问题。

### 深挖追问

- 这个行为发生在浏览器、客户端库、代理网关还是后端服务？
- 如果接口偶发超时/失败，如何用 curl、DevTools、网关日志和 tcpdump 分层验证？
- 连接池、缓存、CDN、TLS 或反向代理配置会怎样改变现象？

### 实战场景/示例

网关到浏览器开启 HTTP/2 可降低多资源加载时的连接开销，但后端服务间是否使用仍要看协议栈和治理能力。

### 易错点/总结

不要把 HTTP/2 多路复用理解为完全没有队头阻塞，它消除的是 HTTP/1.1 应用层队头阻塞，不消除 TCP 层丢包阻塞。

## 面试复习强化

### 核心概念

从面试角度看，**HTTP/2是什么？** 可以放在“应用层网络”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 从语义、报文、连接复用、安全性、缓存、浏览器/网关行为组织答案。 追问重点是为什么需要该机制、解决什么问题、代价是什么、和相邻技术边界在哪里。 对于“HTTP/2是什么？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- HTTP 题常追问幂等性、缓存、长连接、连接复用、头部压缩、队头阻塞和明文传输风险。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：区分协议保证、客户端/浏览器实现和业务约定，避免混为一谈。
