# DNS底层使用TCP还是UDP？

DNS底层使用TCP还是UDP？
DNS 基于UDP协议实现，DNS使用UDP协议进行域名解析和数据传输。因为基于UDP实现DNS能够提供低延迟、简单快速、轻量级的特性，更适合DNS这种需要快速响应的域名解析服务。

低延迟：UDP是一种无连接的协议，不需要在数据传输前建立连接，因此可以减少传输时延，适合DNS这种需要快速响应的应用场景；
简单快速：UDP相比于TCP更简单，没有TCP的连接管理和流量控制机制，传输效率更高，适合DNS这种需要快速传输数据的场景；
轻量级：UDP头部较小，占用较少的网络资源，对于小型请求和响应来说更加轻量级，适合DNS这种频繁目短小的数据交换。

尽管 UDP 存在丢包和数据包损坏的风险，但在 DNS 的设计中，这些风险是可以被容忍的。DNS 使用了一些机制来提高可靠性，例如查询超时重传、请求重试、缓存等，以确保数据传输的可靠性和正确性。

## 面试总结
### 核心概念

DNS 负责把域名解析为 IP 地址。解析链路可能经过浏览器/系统缓存、本地 DNS、根域、顶级域、权威 DNS；传输通常优先 UDP，响应过大、区域传送等场景会使用 TCP。

### 面试官想考什么

面试官常考 DNS 解析流程、缓存、UDP/TCP 使用场景、DNS 劫持以及它如何影响接口耗时。

### 标准回答

访问域名前先查本地缓存，未命中则向本地 DNS 递归查询；本地 DNS 可能迭代访问根、TLD、权威 DNS，最终返回记录并缓存。DNS 失败会导致连接还没建立就超时，Java 服务中要关注域名缓存、TTL 和连接池复用。

### 深挖追问

- 这个行为发生在浏览器、客户端库、代理网关还是后端服务？
- 如果接口偶发超时/失败，如何用 curl、DevTools、网关日志和 tcpdump 分层验证？
- 连接池、缓存、CDN、TLS 或反向代理配置会怎样改变现象？

### 实战场景/示例

服务调用突然大量超时但目标服务正常，可能是 DNS 解析慢、域名被劫持或 JVM DNS 缓存导致旧 IP 未刷新。

### 易错点/总结

不要把 DNS 当成只发生一次的动作。缓存、TTL、负载均衡和故障切换都会影响线上行为。
