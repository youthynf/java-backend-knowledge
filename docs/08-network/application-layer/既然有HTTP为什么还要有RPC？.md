# 既然有HTTP为什么还要有RPC？

既然有HTTP为什么还要有RPC？
一、HTTP与RPC协议由来
假如我们需要在两个不同机器进程间传输数据，我们一般会在代码中使用Socket进行编程，我们可以选择UDP或TCP，前者不可靠，后者可靠，一般对可靠性有点要求都会选择TCP。TCP具有面向连接、可靠的、基于字节流的特点。其中字节流，可以理解为一个双向的通道中流淌着的数据，本质是一堆二进制数据。

纯裸的TCP收发的这些二进制01串中没有任何边界，会导致粘包问题，因此纯裸的TCP无法直接使用，需要加入一些自定义的规则用于区分消息边界。比如加入消息头，消息头里写清楚一个完整的包长度是多少，根据这个长度可以截取我们真正需要传输的消息体。而这里的消息头只要上下游都约定好了，互相认就可以，这就是所谓的协议。于是基于TCP，就衍生了非常多的协议，比如HTTP和RPC。

二、HTTP与RPC概述
TCP是传输层的协议，而基于TCP造出来的HTTP和各类RPC协议，它们都只是定义了不同的消息格式的应用层协议而已。
RPC本质上不算是协议，而是一种调用方式，而像gRPC和Thrift这样的具体实现才是协议，它们是实现了RPC调用的协议。目的是希望程序员能像调用本地方法那样去调用远端的服务方法。同时RPC有很多种实现方式，不一定非得基于TCP协议，改用UDP或者HTTP，其实也可以做到类似的功能。
HTTP主要用于B/S架构，而RPC更多用于C/S架构。但现在其实已经没分那么清了，B/S和C/S在慢慢融合。很多软件同时支持多端，所以对外一般用HTTP协议，而内部集群的微服务之间则采用RPC协议进行通讯。
RPC其实比HTTP出现的要早，且比目前主流的HTTP/1.1性能要更好，所以大部分公司内部都还在使用 RPC。HTTP/2.0在HTTP/1.1的基础上做了优化，性能可能比很多RPC协议都要好，但由于是这几年才出来的，所以也不太可能取代掉 RPC。

三、HTTP与RPC区别
服务发现
如果需要向某个服务器发起请求，首先得建立连接，建立连接的前提是知道IP和端口号，而这个找到服务对应的IP和端口号的过程就是服务发现。
HTTP中，知道服务的域名，则可以通过DNS服务去解析得到对应的IP地址，默认80端口；
RPC中，一般会有专门的中间服务去保存服务名和IP信息，比如Consul或者Etcd，甚至是Redis。通过这些中间服务去获得IP和端口信息；

底层连接形式
HTTP/1.1为例，默认在建立底层TCP连接之后会一直保持这个连接（Keep Alive），之后的请求和响应都会复用这条连接；
RPC协议：类似HTTP通过TCP建立长连接进行数据交互，不同之处在于，RPC一般会额外建立一个连接池，实现连接的复用。
由于连接池有利于提升网络请求性能，所以不少编程语言的网络库里都会给HTTP加上连接池，比如Go就是这么干的。

传输内容
HTTP/1.1设计初衷是用于网络文本展示，所以内容以字符串为主，Header和Body都是如此。在Body这块，它使用JSON来序列化结构体数据。HTTP/1.1结构体数据较大，相同的请求头部存在重复发送，性能较差。
RPC采用体积更小的Protobuf或者其他序列化协议区保存结构体数据，同时不需要像HTTP那样考虑各种浏览器行为，比如302重定向跳转。因此性能会更好一些，这也是公司内部微服务中抛弃HTTP，选择RPC的最主要原因。
HTTP/2在HTTP/1.1的基础上做了很多改进，所以性能上比很多RPC协议还要好，甚至连gRPC底层都直接使用HTTP/2实现。

## 面试复习补充

### 核心概念

RPC 让调用远程服务像调用本地方法，通常包含服务发现、序列化、网络传输、负载均衡、超时重试和熔断治理。

### 面试官想考什么

面试官想考 HTTP 与 RPC 的定位差异，以及为什么微服务内部常用 RPC。

### 标准回答

HTTP 是通用应用层协议，生态开放；RPC 更强调服务间调用效率和治理能力，常用二进制协议、强契约 IDL、连接复用和服务发现。二者不是替代关系，外部开放接口常用 HTTP/REST，内部高频调用可用 RPC。

### 深挖追问

- 如果线上出现超时/失败，如何验证是不是这个环节？
- 它和相邻层协议的职责边界是什么？
- 有哪些参数或默认行为会影响生产表现？

### 实战场景/示例

用户服务调用订单服务可通过 Dubbo/gRPC；对外开放给第三方的接口则更适合 HTTP API。

### 易错点/总结

不要说 RPC 一定比 HTTP 快。性能取决于协议、序列化、连接池、治理和业务耗时。
