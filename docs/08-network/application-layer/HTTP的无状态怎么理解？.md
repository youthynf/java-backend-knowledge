# HTTP的无状态怎么理解？

HTTP的无状态怎么理解？
HITP是无状态的，这意味着每个请求都是独立的，服务器不会在多个请求之间保留关于客户端状态的信息。在每个HTTP请求中，服务器不会记住之前的请求或会话状态，因此每个请求都是相互独立的。

虽然HTTP本身是无状态的，但可以通过一些机制来实现状态保持，其中最常见的方式是使用Cookie和Session来跟踪用户状态。通过在客户端存储会话信息或状态信息，服务器可以识别和跟踪特定用户的状态，以提供一定程度的状态保持功能。

## 面试总结
### 核心概念

HTTP 无状态指协议本身不会自动记住两次请求之间的业务上下文。每个请求从协议角度看都是独立的，服务端处理请求时需要依赖请求中携带的信息、服务端存储或外部会话系统来识别用户和业务状态。

### 面试官想考什么

面试官想考你能否区分“协议无状态”和“业务系统不能保存状态”。还会追问 Cookie、Session、Token、JWT、负载均衡 Session 粘滞、分布式会话和无状态服务扩展。

### 标准回答

HTTP 无状态让服务更容易横向扩展和缓存，因为任意服务实例理论上都能处理任意请求。但登录态、购物车、权限、流程进度等业务需要状态，所以常用 Cookie + Session、Token/JWT、Redis 会话、数据库状态或请求参数来补充。无状态服务通常把状态外置到客户端令牌或共享存储中，而不是依赖单个进程内存。

### 深挖追问

- Session 存在单机内存里，服务扩容或重启会有什么问题？
- JWT 无状态认证如何做注销、续期和权限变更立即生效？
- 负载均衡的 Session Sticky 有哪些优缺点？

### 实战场景/代码示例

电商用户登录后，浏览器携带 Cookie 中的 `SESSIONID`，后端用它到 Redis 查询用户会话；如果改成 JWT，客户端在 `Authorization: Bearer <token>` 中携带令牌，后端校验签名和过期时间后获取用户身份。

### 易错点/总结

无状态不是没有连接，也不是服务端不能存数据库；它强调 HTTP 协议不为你维护会话。设计分布式系统时，要明确状态放在哪里、如何过期、如何同步、如何吊销。
