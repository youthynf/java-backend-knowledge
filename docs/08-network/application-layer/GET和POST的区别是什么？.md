# GET和POST的区别是什么？

GET和POST的区别是什么？
根据 RFC 规范，GET 的语义是从服务器获取指定的资源，这个资源可以是静态的文本、页面、图片视频等。GET 请求的参数位置一般是写在 URL 中，URL 规定只能支持 ASCII，所以 GET 请求的参数只允许 ASCII字符，而且浏览器会对URL的长度有限制（HTTP协议本身对 URL长度并没有做任何规走）。

根据 RFC 规范，POST 的语义是根据请求负荷（报文body）对指定的资源做出处理，具体的处理方式视资源类型而不同。POST请求携带数据的位置一般是写在报文 body 中，body 中的数据可以是任意格式的数据，只要客户端与服务端协商好即可，而且浏览器不会对 body 大小做限制。

如果从 RFC 规范定义的语义来看：
GET 方法就是安全且幂等的，因为它是「只读」操作，无论操作多少次，服务器上的数据都是安全的，且每次的结果都是相同的。所以，可以对 GET 请求的数据做缓存，这个缓存可以做到浏览器本身上（彻底避免浏览器发请求），也可以做到代理上（如nginx），而且在浏览器中 GET 请求可以保存为书签。
POST 因为是「新增或提交数据」的操作，会修改服务器上的资源，所以 POST 是不安全的，且多次提交数据就会创建多个资源，所以 POST 不是幂等的。所以，浏览器一般不会缓存 POST 请求，也不能把 POST 请求保存为书签。

但是实际过程中，开发者不一定会按照 RFC 规范定义的语义来实现 GET和 POST 方法。比如:
可以用 GET 方法实现新增或删除数据的请求，这样实现的 GET方法自然就不是安全和幂等；
可以用 POST 方法实现査询数据的请求，这样实现的 POST方法自然就是安全和幂等。

## 面试总结
### 核心概念

该主题属于计算机网络基础知识，复习时应放到“应用如何通过网络完成一次请求”的链路中理解。

### 面试官想考什么

面试官想看你能否解释概念、说明它解决的问题、指出边界条件，并结合 Java 后端线上问题排查。

### 标准回答

回答时先定义概念，再说明工作流程，最后落到工程场景：对性能、可靠性、安全性或可观测性有什么影响。

### 深挖追问

- 这个行为发生在浏览器、客户端库、代理网关还是后端服务？
- 如果接口偶发超时/失败，如何用 curl、DevTools、网关日志和 tcpdump 分层验证？
- 连接池、缓存、CDN、TLS 或反向代理配置会怎样改变现象？

### 实战场景/示例

例如接口超时不一定是业务慢，也可能是 DNS、建连、TLS、连接池、网关或下游网络问题。

### 易错点/总结

不要孤立背结论，要能按网络分层和请求链路定位问题。
