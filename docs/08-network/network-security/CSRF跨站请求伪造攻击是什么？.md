# CSRF跨站请求伪造攻击是什么？

CSRF跨站请求伪造攻击是什么？
CSRF（跨站请求伪造）是一种攻击手段，攻击者通过诱导用户执行恶意操作，从而获取用户数据或执行恶意代码。CSRF攻击通常通过伪造一个合法的HTTP请求来实现，这个请求看起来是合法的，但实际上是为了执行一个攻击者控制的操作。
示例说明：
1.用户在网站A登录认证后，产生网站A的Cookies；
2.用户在未退出登录的情况下访问了网站B（攻击者网站），并在网站B触发了向第三方网站网站A的请求；
3.此时请求网站A会带上步骤1的认证Cookies，网站A不知道是用户发出的请求还是网站B发出的请求，所以会按照用户的权限进行执行
至此，攻击者的目的达成。

解决CSRF攻击的方法主要有以下几种：
验证用户会话：在服务器端对用户会话进行验证，确保请求的会话标识符token与当前会话标识符匹配。这样可以防止攻击者伪造会话标识符；
使用双重验证：除了会话验证，还可以使用其他验证方式，例如验证码、签名验证等。这些验证方式可以增加攻击的难度；
防止跨站请求：通过设置CSP（内容安全策略）来防止跨站请求，限制网页中可执行的脚本源，减少攻击者诱导用户执行恶意操作的可能性；
避免使用自动提交表单：禁用默认的自动提交功能，要求用户在提交表单前确认操作，防止攻击者诱导用户在未经授权的情况下提交表单；
强制Referer头部：在服务器端检査请求的Referer头部，确保请求来自可信来源。

## 面试复习补充

### 核心概念

网络安全关注机密性、完整性、可用性和身份认证。常见攻击包括 XSS、CSRF、SQL 注入、DDoS、DNS 劫持和中间人攻击。

### 面试官想考什么

面试官想看你是否能说明攻击原理、危害和工程防护，而不是只背缩写。

### 标准回答

XSS 是注入脚本在用户浏览器执行，防护靠输出编码、CSP、HttpOnly；CSRF 是借用户登录态伪造请求，防护靠 SameSite、CSRF Token、校验 Origin/Referer；SQL 注入靠拼接 SQL 注入恶意语句，防护靠预编译和参数绑定；DDoS 需要限流、清洗和弹性防护；HTTPS 可降低中间人窃听篡改风险。

### 深挖追问

- 攻击入口、利用条件和影响范围分别是什么？
- 只靠 HTTPS、前端校验或 WAF 能不能完全解决？为什么？
- 如果线上已经发生攻击，如何止血、排查、修复和补审计？

### 实战场景/示例

评论区未转义用户输入导致脚本窃取 Cookie，这是典型存储型 XSS；转账接口只靠 Cookie 鉴权且无 CSRF Token，则可能被跨站表单触发。

### 易错点/总结

前端校验不能替代服务端防护；HTTPS 也不能防止应用层越权和注入漏洞。

## 面试复习强化

### 核心概念

从面试角度看，**CSRF跨站请求伪造攻击是什么？** 可以放在“网络安全”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 先建立威胁模型：攻击者控制什么输入、越过什么信任边界、造成什么影响。 追问会考漏洞成因、攻击链路、检测方式、防御手段和防御局限。 对于“CSRF跨站请求伪造攻击是什么？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- CSRF 利用浏览器自动携带凭证，防护依赖 SameSite、CSRF Token、Origin/Referer 校验和重要操作二次确认。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：默认输入不可信、权限最小化、失败默认拒绝，并配合日志审计和告警。
