# SQL注入攻击是什么？

SQL注入攻击是什么？
SOL注入发生在当应用程序直接使用用户提供的输入作为SQL查询的一部分时。当用户输入被错误地用作数据库查询的一部分，而应用程序没有对其进行适当的验证和转义，就可能会发生SQL注入，攻击者可能会利用这一点来执行他们自己的恶意SQL查询。

解决SQL注入问题的方法主要有以下几种：
输入验证和转义：在将用户输入用作SQL查询的一部分之前，对输入进行验证和转义。确保输入符合预期格式，并防止任何可能导致SQL注入的特殊字符；
使用参数化査询：使用参数化査询可以避免直接将用户输入嵌入到SQL查询中。参数化査询使用预定义的变量来接收用户输入，并将其传递给数据库引擎，而不是直接将其用作査询的一部分。这样可以防止SQL注入攻击；
限制数据库权限：限制数据库用户的权限，只授予他们执行所需操作所需的最低权限。攻击者可能具有比预期更多的权限，这可能会使攻击更加容易；
实施输入过滤：在某些情况下，实施输入过滤可以进一步减少SQL注入的风险。这可能涉及检査和过滤用户输入中的特殊字符和词汇，以排除可能的恶意输入。

## 面试复习补充

### 核心概念

网络安全关注机密性、完整性、可用性和身份认证。常见攻击包括 XSS、CSRF、SQL 注入、DDoS、DNS 劫持和中间人攻击。

### 面试官想考什么

面试官想看你是否能说明攻击原理、危害和工程防护，而不是只背缩写。

### 标准回答

XSS 是注入脚本在用户浏览器执行，防护靠输出编码、CSP、HttpOnly；CSRF 是借用户登录态伪造请求，防护靠 SameSite、CSRF Token、校验 Origin/Referer；SQL 注入靠拼接 SQL 注入恶意语句，防护靠预编译和参数绑定；DDoS 需要限流、清洗和弹性防护；HTTPS 可降低中间人窃听篡改风险。

### 深挖追问

- 攻击入口、利用条件和影响范围分别是什么？
- 只靠 HTTPS、前端校验或 WAF 能不能完全解决？为什么？
- 如果线上已经发生攻击，如何止血、排查、修复和补审计？

### 实战场景/示例

评论区未转义用户输入导致脚本窃取 Cookie，这是典型存储型 XSS；转账接口只靠 Cookie 鉴权且无 CSRF Token，则可能被跨站表单触发。

### 易错点/总结

前端校验不能替代服务端防护；HTTPS 也不能防止应用层越权和注入漏洞。

## 面试复习强化

### 核心概念

从面试角度看，**SQL注入攻击是什么？** 可以放在“网络安全”这一类知识中理解。复习时不要只背定义，要能说清：它解决什么问题、依赖哪些前提、正常流程是什么、异常情况下系统会怎样退化或恢复。

### 面试官想考什么

- 是否理解概念背后的设计目标，而不是只记住名词；
- 是否能把机制和真实工程场景联系起来；
- 是否能分析边界条件、失败场景、性能与安全取舍；
- 是否能给出可落地的排查、实现或优化步骤。

### 标准回答

> 先建立威胁模型：攻击者控制什么输入、越过什么信任边界、造成什么影响。 追问会考漏洞成因、攻击链路、检测方式、防御手段和防御局限。 对于“SQL注入攻击是什么？”，回答时建议先给一句话定义，再按“工作流程/关键机制 → 典型场景 → 风险与优化”展开，最后补充一两个线上实践点。

### 深挖追问

- 如果该机制失效，会出现什么现象？如何定位是配置、代码、资源还是外部依赖导致？
- 它和相邻概念有什么区别？例如语义、适用场景、性能成本、可靠性保证分别是什么？
- 在高并发、网络抖动、服务重启、数据不一致或权限受限时，需要补充哪些保护措施？
- 有哪些指标可以证明方案有效？例如延迟、吞吐、错误率、资源使用率、重试次数或业务成功率。

### 示例 / 实战场景

- 设计方案时：先明确业务目标和约束，再选择对应机制，不要为了使用某个技术而引入复杂度。
- 排查问题时：先确认现象和影响面，再查看日志、监控、配置、版本变更和上下游依赖，最后小步验证修复。
- 复盘沉淀时：补充自动化测试、容量评估、告警阈值、降级预案和文档，避免同类问题再次发生。

### 本题高频补充

- SQL 注入防御优先使用预编译参数绑定；拼接 SQL、黑名单过滤和仅前端校验都不可靠。

### 易错点 / 总结

- 只背结论、不讲原因，是面试扣分点；要主动解释“为什么这样设计”。
- 只讲正常路径、不讲异常路径，会显得缺少生产经验；至少补充超时、重试、降级、回滚或兜底。
- 不要把理论保证无限放大，工程实现通常还受网络、资源、配置、版本和业务语义约束。
- 总结一句：默认输入不可信、权限最小化、失败默认拒绝，并配合日志审计和告警。
