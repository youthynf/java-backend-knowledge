# 对称加密和非对称加密有什么区别？

## 核心概念

对称加密用同一把密钥加解密，速度快但密钥分发是难题；非对称加密用公私钥对，公钥可公开、私钥保密，解决了密钥分发问题但速度慢一个数量级。现代加密体系（如 TLS、JWT 签名、SSH 登录）几乎都是两者混合使用：非对称加密用来协商对称密钥或做签名，对称加密用来加密业务数据。

理解这条主线的钥匙是“性能 vs 分发”的取舍。对称算法可达 1~10 GB/s 量级，非对称算法通常只有几十 KB/s 到几 MB/s；但对称密钥如何安全地交给对方，在没有安全通道时是个先有鸡还是先有蛋的问题，非对称加密用数学难题（大数分解、离散对数、椭圆曲线）解开了这个结。

## 标准回答

一句话结论：**对称加密一把钥匙开一把锁，性能好但密钥要私下传；非对称加密公私钥配对，公钥随便发、私钥不出门，性能差但能解决密钥分发和数字签名问题；生产环境几乎都是混合使用**。

要点展开：

- **密钥形态**：对称只有一把共享密钥；非对称有一对（公钥 + 私钥），公钥加密只能私钥解密，私钥签名只能公钥验证。
- **代表算法**：对称有 AES（主流）、ChaCha20、DES/3DES（已淘汰）；非对称有 RSA、ECDSA、ECC（椭圆曲线）。
- **密钥长度对比**：AES-128 安全强度约等于 RSA-3072、ECC-256；同等安全强度下 ECC 密钥最短。
- **性能对比**：对称加密比非对称快 100~1000 倍，非对称只用于加密少量数据（如会话密钥、摘要）或做签名。
- **典型组合**：TLS 握手用非对称协商出对称密钥，传输阶段用 AES-GCM 加密；JWT 用 RS256（RSA 私钥签名）或 HS256（对称密钥 HMAC）。

## 实现原理

### 对称加密

对称加密按处理方式分**分组密码**（Block Cipher）和**流密码**（Stream Cipher）。

- **分组密码**：把明文切成固定长度块（AES 是 128 bit），逐块加密。需要模式（Mode）处理超过一块的数据：ECB（不安全，相同明文产生相同密文）、CBC（需 IV）、CTR（计数器模式，可并行）、GCM（CTR + MAC，AEAD 认证加密，**目前主流**）。
- **流密码**：通过密钥流 XOR 明文，代表是 ChaCha20，常与 Poly1305 组合成 ChaCha20-Poly1305 AEAD。

**AES 关键参数**：密钥长度 128/192/256 bit，块大小 128 bit，轮数 10/12/14。AES 本身是纯加密，**不提供完整性校验**，必须配合 HMAC 或选 GCM 模式才能防篡改。

### 非对称加密

非对称算法建立在单向函数（求逆困难）之上：

- **RSA**：基于大数分解难题。加密 `c = m^e mod n`，解密 `m = c^d mod n`，公钥 `(e, n)`，私钥 `(d, n)`。常用密钥 2048/3072/4096 bit。RSA 加密有填充要求（PKCS#1 v1.5 或更安全的 OAEP），不填充直接加密是极度危险的。
- **ECC**：基于椭圆曲线离散对数难题。相同安全强度下密钥短得多（256 bit ECC ≈ 3072 bit RSA），计算快、存储省，适合移动端和 IoT。常用曲线：P-256、P-384、Curve25519。
- **DH / ECDH**：密钥协商协议，让双方在公开信道上协商出共享秘密，本身不是加密算法。TLS 1.3 只保留 ECDHE，提供前向安全性。

### 数字签名

非对称加密的另一半用途：**私钥签名，公钥验签**。流程：

```
发送方:  原文 -> Hash -> 摘要 -> 私钥加密 -> 签名
                       (附在原文后一起发送)
接收方:  收到原文+签名 -> 原文 Hash 得摘要 A
                         签名用公钥解密得摘要 B
                         A == B 则验签通过
```

签名不提供机密性，只保证完整性 + 不可抵赖 + 身份认证。

### TLS 中的混合使用

```
Client                          Server
  | --- ClientHello --------------> |   (含随机数 R1、支持的密码套件)
  | <-- ServerHello ---------------- |   (含随机数 R2、选定套件)
  | <-- Certificate ---------------- |   (服务器证书，含公钥)
  | <-- ServerKeyExchange ---------- |   (ECDHE 参数)
  | <-- ServerHelloDone ----------- |
  | --- ClientKeyExchange --------> |   (用公钥加密预主密钥，或 ECDHE 公开数)
  | --- ChangeCipherSpec ---------> |
  | --- Finished -----------------> |   (用协商密钥加密的握手摘要)
  | <-- ChangeCipherSpec ---------- |
  | <-- Finished ------------------ |
  | <=== 对称加密的应用数据 ===> |   (用 AES-GCM 等)
```

双方用 R1、R2、预主密钥派生出主密钥，再用主密钥派生出对称加密密钥和 MAC 密钥。握手用非对称，传输用对称。

## 代码示例

### Java：AES-GCM 对称加密

```java
import javax.crypto.Cipher;
import javax.crypto.KeyGenerator;
import javax.crypto.SecretKey;
import javax.crypto.spec.GCMParameterSpec;
import java.security.SecureRandom;
import java.util.Base64;

public class AesGcmDemo {
    private static final int KEY_SIZE = 256;
    private static final int IV_SIZE = 12;     // GCM 推荐 12 字节
    private static final int TAG_SIZE = 128;   // 认证标签长度

    public static void main(String[] args) throws Exception {
        KeyGenerator kg = KeyGenerator.getInstance("AES");
        kg.init(KEY_SIZE);
        SecretKey key = kg.generateKey();

        byte[] iv = new byte[IV_SIZE];
        new SecureRandom().nextBytes(iv);

        String plain = "转账金额 1000 元，账户 user_001";
        byte[] cipherText = encrypt(plain.getBytes("UTF-8"), key, iv);
        System.out.println("密文: " + Base64.getEncoder().encodeToString(cipherText));

        byte[] decrypted = decrypt(cipherText, key, iv);
        System.out.println("解密: " + new String(decrypted, "UTF-8"));
    }

    static byte[] encrypt(byte[] data, SecretKey key, byte[] iv) throws Exception {
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, key, new GCMParameterSpec(TAG_SIZE, iv));
        return cipher.doFinal(data);
    }

    static byte[] decrypt(byte[] data, SecretKey key, byte[] iv) throws Exception {
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.DECRYPT_MODE, key, new GCMParameterSpec(TAG_SIZE, iv));
        return cipher.doFinal(data);
    }
}
```

注意：每次加密都要用 `SecureRandom` 生成新的 IV，IV 不需要保密但要随密文一起传输。GCM 模式天然带完整性校验，密文被改一个 bit 解密就会抛 `AEADBadTagException`。

### Java：RSA 非对称加密 + 签名

```java
import java.security.*;

public class RsaDemo {
    public static void main(String[] args) throws Exception {
        KeyPairGenerator kpg = KeyPairGenerator.getInstance("RSA");
        kpg.initialize(2048);
        KeyPair kp = kpg.generateKeyPair();

        String msg = "hello server";

        // 签名：私钥签
        Signature sig = Signature.getInstance("SHA256withRSA");
        sig.initSign(kp.getPrivate());
        sig.update(msg.getBytes("UTF-8"));
        byte[] signature = sig.sign();

        // 验签：公钥验
        Signature verifier = Signature.getInstance("SHA256withRSA");
        verifier.initVerify(kp.getPublic());
        verifier.update(msg.getBytes("UTF-8"));
        System.out.println("验签结果: " + verifier.verify(signature));
    }
}
```

### Python：bcrypt 密码加盐哈希（属于单向哈希，配合密钥体系）

```python
import bcrypt

password = b"my_plain_password"
salt = bcrypt.gensalt(rounds=12)
hashed = bcrypt.hashpw(password, salt)
print(hashed.decode())

# 校验
if bcrypt.checkpw(b"my_plain_password", hashed):
    print("密码正确")
```

## 实战场景

| 场景 | 用法 | 注意点 |
|------|------|--------|
| 接口数据加密 | 请求体用 AES-GCM 加密，密钥用 RSA 加密随请求头传 | IV 不能复用；密钥轮换要考虑兼容期 |
| JWT 签名 | 单体服务用 HS256（共享密钥）；多服务用 RS256（私钥签、公钥验） | HS256 密钥泄露即全失守，多服务慎用 |
| 文件加密存储 | AES-256-CBC + HMAC-SHA256 或 AES-256-GCM | 不要用 ECB；密钥要放 KMS，不能写死在代码里 |
| SSH 免密登录 | 生成 ed25519 密钥对，公钥写入 authorized_keys | 弃用 RSA-1024，优先 ed25519 |
| 移动端 HTTPS | 服务端用 ECC 证书，握手快、包小 | 兼容旧客户端时同时配 RSA 证书 |
| 数据库字段加密 | 应用层用 AES 加密敏感字段，密钥放 HSM/KMS | 加密后无法等值查询，需要另存哈希做检索 |

## 深挖追问

**Q1：为什么 HTTPS 不直接用非对称加密传所有数据？**

性能。RSA 加解密速度比 AES 慢 100~1000 倍，且非对称加密对明文长度有上限（RSA-2048 单次最多加密 245 字节，OAEP 填充后）。所以握手阶段用非对称协商出对称密钥，后续数据用 AES-GCM 加密，兼顾安全和性能。

**Q2：AES-GCM 为什么比 AES-CBC 安全？**

GCM 是 AEAD（认证加密），同时提供机密性和完整性；CBC 只提供机密性，必须配合 HMAC 才能防篡改，且实现容易出错（如先解密后校验，导致 Padding Oracle 攻击）。TLS 1.3 直接淘汰了 CBC，只保留 GCM 和 ChaCha20-Poly1305。

**Q3：什么是前向安全性（PFS）？为什么 ECDHE 有而 RSA 没有？**

前向安全指长期私钥泄露后，历史通信仍安全。RSA 密钥交换：客户端用服务器公钥加密预主密钥，如果服务器私钥泄露，所有历史录流量都能被解密。ECDHE：每次握手生成临时 DH 密钥对，协商出的会话密钥用完即弃，私钥泄露也无法解密历史流量。TLS 1.3 强制要求 PFS，删除了 RSA 密钥交换。

**Q4：RSA-2048 还能安全用多久？**

目前 RSA-2048 安全强度约 112 bit，NIST 建议在 2030 年前可用，2030 年后建议 RSA-3072 或迁移到 ECC。如果考虑量子计算威胁（Shor 算法），RSA 和 ECC 都不抗量子，需要迁移到后量子密码（PQC），NIST 已标准化 Kyber、Dilithium 等。

**Q5：JWT 用 HS256 还是 RS256？**

单体应用、密钥能严格保密，用 HS256 性能好；微服务、有多个验签方、签发方需要权威性时，用 RS256 或 EdDSA，私钥只在签发服务持有，公钥可公开。注意 HS256 的密钥要足够长（≥256 bit），且不能用 `secret`、`123456` 这种弱密钥。

## 易错点

- **AES 用 ECB 模式**：相同明文块加密后密文相同，图加密后还能看出图案轮廓，必须用 GCM 或 CBC + IV。
- **IV/Nonce 复用**：GCM 模式同一密钥下 IV 复用会泄露明文 XOR 并破坏认证，是灾难性故障。
- **RSA 不加填充直接加密**：相同明文加密结果相同，且易受选择密文攻击，必须用 OAEP 填充。
- **把非对称加密当对称用**：RSA 加密大数据要分块，性能极差，正确做法是用 RSA 加密 AES 密钥。
- **混淆加密和签名**：加密用对方公钥，签名用自己私钥；给对方加密用自己公钥是没用的。
- **MD5/SHA-1 当签名哈希**：已被攻破，存在碰撞，签名要用 SHA-256 及以上。
- **密钥写死在代码里**：一旦代码泄露即失守，应放 KMS、HSM 或环境变量/配置中心加密存储。
- **ECC 选无名曲线**：椭圆曲线安全性依赖曲线参数，只用 NIST 标准曲线或 Curve25519，不要自己造曲线。

## 总结

对称加密解决“如何快速加解密大数据”的问题，非对称加密解决“如何在公开信道分发密钥和做身份认证”的问题，两者是互补而非替代。生产实践的核心原则：握手用非对称、传输用对称、签名用非对称、密码哈希用专门的慢哈希算法。AES-GCM 和 ECDHE-RSA 证书是当前 HTTPS 的黄金组合，TLS 1.3 进一步精简到只保留 AEAD 和 ECDHE。
