# 常见I/O模型有哪些？

常见I/O模型有哪些？
阻塞I/O模型
描述：应用程序发起I/O操作后，会被阻塞，直到I/O操作完成才返回结果。
适用场景：适用于对实时性要求不高的场景，或者在单线程程序中，不需要同时处理多个I/O请求的场景。
优点：实现简单，易于理解。
缺点：在高并发场景下效率低下，因为I/O操作期间无法执行其他任务，导致CPU资源浪费.

非阻塞I/O模型
描述：应用程序发起I/O操作后立即返回，不会被阻塞，但需要不断轮询或者使用select/poll/epoll等系统调用来检查I/O是否完成。
适用场景：适用于需要多路复用的场景，比如需要同时处理多个socket连接的服务器程序。
优点：提高了应用程序的响应性，不会因等待I/O操作而阻塞.
缺点：需要频繁地轮询I/O操作的状态，可能会导致CPU资源的浪费，因为应用程序需要不断地检查I/O是否完成.

I/O复用模型
描述：通过select/poll/epoll等系统调用，应用程序可以同时等待多个I/O操作，当其中任何一个I/O操作准备就绪时，应用程序会被通知。
适用场景：适用于需要同时处理多个I/O操作的场景，比如高并发的服务端程序。
优点：可以有效地管理多个I/O请求，提高了资源利用率，减少了频繁轮询的开销.
缺点：在某些情况下，如大量I/O通道时，可能会导致性能瓶颈，因为需要管理大量的I/O通道状态.

信号驱动I/O模型
描述：应用程序发起I/O操作后，可以继续做其他事情，当I/O操作完成时，操作系统会向应用程序发送信号来通知其完成.
适用场景：适用于需要异步I/O通知的场景，可以提高系统的并发能力.
优点：允许应用程序在I/O操作完成时立即得到通知，而不需要轮询.
缺点：信号处理的复杂性较高，信号的传递和处理可能会导致竞态条件等问题，且信号的处理机制在不同的操作系统和编程环境中可能有所不同.

异步I/O模型
描述：应用程序发起I/O操作后，应用程序只需要等待通知即可，操作系统会在I/O操作完成时通知应用程序.
适用场景：适用于需要大量并发连接和高性能的场景，能够减少系统调用次数，提高系统效率.
优点：实现了真正的异步操作，应用程序不需要等待I/O操作完成，可以完全并行地进行其他任务，提高了系统的吞吐量和响应速度.
缺点：实现复杂，需要处理异步通知和数据同步问题，且在某些系统和编程环境中，异步I/O的支持可能不如其他模型完善.

总的来说，不同的I/O模型在不同的场景下各有优劣，选择合适的I/O模型需要根据具体的应用需求和系统环境来决定。
