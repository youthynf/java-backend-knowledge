# 全库文章更新改造任务计划

> 目标：对 `docs/` 下约 500 篇 Markdown 文章做系统化更新改造，不只是补目录，而是按面试复习要求重新梳理内容、查漏补缺、统一样式排版，并分批提交。

## 一、改造总目标

每篇知识文章最终应尽量满足以下结构：

```markdown
# 标题

## 核心概念
- 讲清定义、背景、解决的问题，而不是只列关键词。

## 面试官想考什么
- 说明该题背后考察的能力点、底层机制、边界条件。

## 标准回答
- 给出面试中可以直接说出口的完整回答。
- 先结论，再解释机制，再补充应用场景。

## 深挖追问
- 常见追问 3~5 个，并给出回答要点。

## 实战场景 / 代码示例
- 尽量给业务场景、故障场景或可运行/可理解的代码示例。

## 易错点 / 总结
- 梳理常见误区、版本差异、边界条件。

## 参考资料
- 如有来源或延伸阅读，保留链接。
```

README、sidebar、cover、404、归档索引类文件不强制按面试模板改造，只做链接和排版整理。

## 二、执行顺序

### 第 0 阶段：全库盘点与基线建立

- [ ] 统计全部 Markdown 文件数量。
- [ ] 排除非文章文件：`README.md`、`sidebar.md`、`_coverpage.md`、`_404.md`、归档索引等。
- [ ] 对每篇文章打标：
  - 是否有 H1 标题
  - 是否有核心概念
  - 是否有面试官想考什么
  - 是否有标准回答
  - 是否有深挖追问
  - 是否有实战场景 / 代码示例
  - 是否有易错点 / 总结
  - 字数是否过短

### 第 1 阶段：统一排版与基础结构

- [ ] 修复明显标题层级问题。
- [ ] 清理空行、重复标题、残缺列表。
- [ ] 为缺少结构的文章补齐面试复习版框架。
- [ ] 保留原文有效内容，不做无意义覆盖。

### 第 2 阶段：Java 核心模块深改

优先级最高，面试高频。

- [ ] `docs/01-java-core/basics/`
- [ ] `docs/01-java-core/collections/`
- [ ] `docs/01-java-core/concurrency/`
- [ ] `docs/01-java-core/jvm/`
- [ ] 其他 Java 核心子模块

### 第 3 阶段：数据库模块深改

- [ ] `docs/03-database/mysql/`
- [ ] `docs/03-database/redis/`
- [ ] 其他数据库主题

### 第 4 阶段：网络、MQ、架构、分布式、DevOps 等模块

- [ ] `docs/08-network/`
- [ ] `docs/04-mq/`
- [ ] `docs/05-distributed/`
- [ ] `docs/06-microservice/`
- [ ] `docs/07-architecture/`

- [ ] 其他模块

### 第 5 阶段：最终校验

- [ ] 重新运行全库质量盘点。
- [ ] 确认所有文章路径和链接不损坏。
- [ ] 检查 Git diff，避免误改正文或删除有效内容。
- [ ] 分批 Git commit + push。

## 三、批次提交规范

每批控制在一个目录或一个主题下，避免一次性提交 500 篇不可审查。

推荐提交信息：

- `update: 优化Java基础文章面试复习版`
- `update: 补充集合框架文章详解`
- `update: 完善JVM文章结构与案例`
- `update: 优化MySQL文章面试回答`
- `update: 完善Redis高频题详解`
- `style: 统一全库文章排版结构`

## 四、质量标准

一篇文章视为完成，至少满足：

1. 有清晰标题。
2. 不是只有知识点列表，要有解释。
3. 有面试标准回答。
4. 有深挖追问或延伸问题。
5. 有实战/代码/场景之一。
6. 有易错点或总结。
7. 排版稳定，可直接阅读。

## 五、当前执行原则

- 不把 500 篇文章一次性做粗暴替换。
- 先自动盘点，再按模块深改。
- 每批改完必须提交 Git。
- 优先保证 Java 基础、集合、并发、JVM、MySQL、Redis 这些高频模块质量。
